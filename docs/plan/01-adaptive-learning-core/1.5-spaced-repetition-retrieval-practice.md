## 1. Problem Statement

Self-learners who use Lextures expect Anki-class retention behaviour. Currently the system has no mechanism to schedule review of previously learned material, leading to rapid forgetting and learner abandonment. Research consistently shows that spaced repetition and retrieval practice are among the most effective learning interventions, with effect sizes of d ≥ 0.6 in meta-analyses. Without an SRS scheduler, Lextures cannot compete with dedicated retention tools like Anki, Quizlet, or Duolingo for the self-learner market, and cannot substantiate its "adaptive" claim for long-term retention.

## 2. Goals

- Implement an SRS scheduler (SM-2 as the initial algorithm, with an upgrade path to FSRS-4.5) that computes a `next_review_at` timestamp per (learner, item) pair after each response.
- Surface a daily "Review" queue to learners: items due today, sorted by priority.
- Allow course authors to mark items as SRS-eligible and configure default retention parameters.
- Provide learners with a "streak" and review-completion indicator to reinforce habits (hooks for 15.9 gamification).
- Expose the review schedule as an API so the recommendations engine (1.8) can interleave reviews with new material.

## 3. Non-Goals

- Full gamification / streak system (covered in 15.9).
- Push notification reminders for due reviews (depends on 6.3 and 7.5).
- Adaptive algorithm selection per learner (v2 enhancement).
- FSRS parameter calibration from user data (requires ~500 reviews per user; ship SM-2 first, migrate to FSRS later).
- Flashcard-deck creation UI (SRS operates on quiz questions from the question bank, not custom flashcards in v1).

## 4. Personas & User Stories

- **As a self-learner**, I want a daily review queue of concepts I have previously studied, sized to take under 15 minutes, so I can maintain long-term retention without guessing what to review.
- **As a K-12 student**, I want the system to remind me to review vocabulary words before a test, automatically spacing practice sessions optimally.
- **As an instructor**, I want to opt specific question types into the SRS pool so that important facts and concepts are reinforced outside of formal quizzes.
- **As a learner**, I want to rate my recall confidence (Again / Hard / Good / Easy) after each review item so the scheduler can adapt the next interval.
- **As a self-learner**, I want to see a "streak" counter showing how many days in a row I have completed my reviews, to maintain motivation.

## 5. Functional Requirements

- **FR-1.** The system MUST implement the SM-2 algorithm: for each (learner, item) the system tracks `interval` (days), `repetition` count, `easiness_factor` (EF, initial 2.5), and `next_review_at`. Grade input is mapped to SM-2 quality scores (0–5).
- **FR-2.** The system MUST update SRS state after each review response within the same transaction as the response recording.
- **FR-3.** The system MUST expose `GET /api/v1/learners/{user_id}/review-queue` returning items due today (or past due), sorted by overdue-ness, paginated.
- **FR-4.** The system MUST allow course authors to mark individual questions (via question bank, 2.1) as `srs_eligible: true`, and courses to have an SRS opt-in toggle.
- **FR-5.** The system MUST record each review event in `srs_review_events` (append-only) with input grade, resulting interval, and EF for audit and algorithm parameter replay.
- **FR-6.** The system SHOULD implement FSRS-4.5 as an optional scheduler back-end that can be enabled per-learner when sufficient review history (≥ 100 events) is available.
- **FR-7.** The system SHOULD expose a `GET /api/v1/learners/{user_id}/review-stats` endpoint returning: items due today, items due this week, review streak (days), and retention estimate.
- **FR-8.** The system MAY allow learners to temporarily suppress an item from the review queue ("I'll skip this for now") without resetting the SRS state.

## 6. Non-Functional Requirements

- **Performance** — Review queue generation MUST complete in p95 ≤ 100 ms for queues up to 200 items. SRS state update per item MUST complete in p95 ≤ 30 ms.
- **Security** — Review queue and SRS state accessible only by the learner and instructors/admins with enrollment access. SRS parameters are not user-configurable (prevents gaming the scheduler).
- **Privacy & Compliance** — SRS review history is educational record (FERPA). Included in GDPR DSAR export. `next_review_at` timestamps reveal study patterns; treat as PII under GDPR definition of behavioural data.
- **Accessibility** — Review card UI MUST be keyboard-navigable (flip card: Space or Enter; grade selection: 1–4 keys). Screen readers MUST hear the question stem before the grade buttons.
- **Scalability** — Expect up to 1 000 items per learner and 50 k active learners = 50 M SRS state rows; index on `(user_id, next_review_at)` is critical.
- **Reliability** — SM-2 computation is deterministic and side-effect-free; unit-test all boundary conditions. Review-queue index MUST be updated atomically with SRS state.
- **Observability** — Prometheus gauge `srs.items_due_now` updated on each write. Log review events at INFO. Alert if queue generation p95 exceeds 300 ms.
- **Maintainability** — SM-2 and FSRS implementations in `server/src/services/srs_scheduler.rs` as pure functions with no DB dependencies; injected via `SrsScheduler` trait.
- **Internationalization** — Grade labels (Again / Hard / Good / Easy) are i18n keys. Date/time rendering for `next_review_at` respects user locale and timezone.
- **Backward compatibility** — SRS tables are additive. Existing quiz flow unchanged; SRS is an opt-in overlay.

## 7. Acceptance Criteria

- **AC-1.** *Given* a learner rates a new review item "Good" (SM-2 grade 4), *when* the response is submitted, *then* `srs_item_states.interval` is set to 1 day, `next_review_at` is set to tomorrow, and an event row is appended.
- **AC-2.** *Given* a learner rates the same item "Good" for the second time, *when* the response is submitted, *then* `interval` is set to 6 days and EF remains ≥ 2.5.
- **AC-3.** *Given* a learner rates an item "Again" (grade 0), *when* the response is submitted, *then* `interval` resets to 1 and `repetition` resets to 0.
- **AC-4.** *Given* 5 items are due for review today, *when* `GET /api/v1/learners/{id}/review-queue` is called, *then* the response contains exactly 5 items sorted by overdue-ness.
- **AC-5.** *Given* a learner completes all due reviews for 7 consecutive days, *when* `GET /api/v1/learners/{id}/review-stats` is called, *then* `streak` equals 7.
- **AC-6.** *Given* a question is not marked `srs_eligible`, *when* a learner answers it in a quiz, *then* no SRS state row is created or updated for that item.

## 8. Data Model

```sql
-- Migration: server/migrations/0034_srs.sql

CREATE TYPE course.srs_grade AS ENUM ('again', 'hard', 'good', 'easy');
CREATE TYPE course.srs_algorithm AS ENUM ('sm2', 'fsrs45');

CREATE TABLE course.srs_item_states (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES "user".users(id) ON DELETE CASCADE,
    question_id     UUID NOT NULL,   -- FK to question bank (2.1)
    algorithm       course.srs_algorithm NOT NULL DEFAULT 'sm2',
    interval_days   NUMERIC(8,2) NOT NULL DEFAULT 0,
    repetition      INTEGER NOT NULL DEFAULT 0,
    easiness_factor NUMERIC(4,3) NOT NULL DEFAULT 2.5,
    next_review_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    due_count       INTEGER NOT NULL DEFAULT 0,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (user_id, question_id)
);

CREATE INDEX idx_srs_queue ON course.srs_item_states (user_id, next_review_at)
    WHERE next_review_at IS NOT NULL;

CREATE TABLE course.srs_review_events (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL,
    question_id     UUID NOT NULL,
    grade           course.srs_grade NOT NULL,
    interval_before NUMERIC(8,2),
    interval_after  NUMERIC(8,2) NOT NULL,
    ef_before       NUMERIC(4,3),
    ef_after        NUMERIC(4,3) NOT NULL,
    response_ms     INTEGER,           -- time taken to answer
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);
```

## 9. API Surface

```
GET  /api/v1/learners/{user_id}/review-queue
  Auth: own user | instructor | admin
  Query: ?limit=20&offset=0
  Response: { items: ReviewItem[], total_due: number }

POST /api/v1/learners/{user_id}/review
  Auth: own user
  Body: { question_id: string, grade: "again"|"hard"|"good"|"easy", response_ms?: number }
  Response 200: { next_review_at: string, interval_days: number }

GET  /api/v1/learners/{user_id}/review-stats
  Auth: own user | instructor | admin
  Response: { streak: number, due_today: number, due_week: number, retention_estimate: number }
```

## 10. UI / UX

**Review mode page** (`/review`, new route in `clients/web/src/`):
- Flash-card style: question front, click/space to reveal answer.
- Four grade buttons: Again (red) / Hard (orange) / Good (blue) / Easy (green) with keyboard shortcuts 1–4.
- Progress bar showing "X of Y due items reviewed."
- Session summary screen after completing queue: items reviewed, estimated retention %.

**Dashboard widget** (home page):
- "X items due for review today" card with "Start Review" CTA.
- Streak counter (fire icon + day count).

Accessibility: all grade buttons have descriptive `aria-label` including the keyboard shortcut.
Mobile: swipe gestures for grade (left=Again, right=Easy) with fallback button row.

## 11. AI / ML Considerations

SM-2 and FSRS-4.5 are deterministic algorithms, not ML models. FSRS uses a stability-retrievability memory model fit from aggregate review data; the per-learner FSRS calibration requires ~500 reviews and is a background job (requires 17.3 job queue).

No LLM calls in this feature. Future: AI-generated hint on "Again" response (hint scaffolding, 1.9).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| SM-2 perceived as inferior to FSRS by power users | M | M | Ship FSRS as opt-in upgrade; document algorithm choice and upgrade path |
| Review queue grows too large, overwhelming learners | M | M | Cap daily review at configurable max (default 200); prioritise most-overdue |
| Learners game the scheduler by always selecting "Easy" | L | L | Record response time; flag suspiciously fast Easy responses for analysis |
| FSRS calibration background job runs too frequently | L | M | Schedule once per week per user with ≥ 500 events; do not run in request path |

## 15. Rollout Plan

- **Feature flag:** `srs_enabled` (default: `false`, per-course toggle also available).
- **Migration sequencing:** deploy `0034_srs.sql` → deploy SM-2 scheduler → enable for self-learner pilot cohort → enable FSRS upgrade flag for power users → GA.
- **Pilot cohort:** 10 self-learner accounts; target: 1 000 review events in first 2 weeks.
- **GA criteria:** SM-2 interval computation matches reference implementation test vectors; review queue p95 ≤ 100 ms under load.
- **Rollback:** disable feature flag; SRS tables are additive; no data loss.

## 16. Test Plan

- **Unit** — SM-2 interval computation for all grade combinations; EF boundary (min 1.3); FSRS stability function against published test vectors.
- **Integration** — Submit review grades for 5 items → verify SRS state updates; verify mastery also updated in `learner_concept_states`.
- **End-to-end** — Playwright: open review queue, flip card, grade all items, verify session summary shows correct count; verify streak increments after daily completion.
- **Security** — Learner cannot modify their own SRS state directly; review endpoint validates ownership.
- **Accessibility** — axe scan on review page; keyboard shortcut test (1–4 grades); screen-reader reads question stem before grade buttons.
- **Performance / load** — k6: 500 concurrent users each fetching 20-item queue; p95 ≤ 100 ms.
- **Manual exploratory** — Verify overdue items appear above due-today items in queue; verify suppressed item does not appear in queue; verify streak resets after missed day.

## 18. Open Questions

1. Should SRS review be per-question (question bank item) or per-concept (concept-level recall prompt)? Per-question is simpler for v1.
2. What should the daily review cap be — 50, 100, or 200 items? Configurable per learner or per course?
3. Do we show estimated retention percentage and projected forgetting curve to learners, or keep it simple?
4. FSRS-4.5 uses four internal state variables (stability, difficulty, retrievability, last-review-day) — should these be stored in the same `srs_item_states` table or a separate FSRS-specific table?
