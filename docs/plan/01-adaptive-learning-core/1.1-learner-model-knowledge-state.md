## 1. Problem Statement

Lextures claims to be "truly adaptive," but adaptivity today is stateless: every quiz session starts from zero with no memory of what the learner already knows. There is no persistent record of mastery, confidence, or knowledge decay per concept. Without a learner model, personalised sequencing, spaced repetition, and meaningful recommendations are architecturally impossible. Fixing this is the single highest-leverage change in the adaptive stack, directly unblocking six other feature areas and making the product's core differentiator credible to districts and universities.

## 2. Goals

- Persist a per-learner, per-concept mastery estimate updated after every quiz attempt and graded activity.
- Represent knowledge decay over time so mastery scores degrade when concepts are not revisited.
- Expose a read model (REST + internal Rust API) that downstream features (recommendations, adaptive sequencing, SRS, reporting) can query without hitting quiz tables directly.
- Keep the model schema-agnostic enough to support both a simple P(mastery) scalar today and an IRT-based θ parameter in 1.6 later.
- Ensure all PII handling is FERPA- and GDPR-compliant from day one.

## 3. Non-Goals

- Full IRT calibration (covered in 1.6).
- Spaced-repetition scheduling logic (covered in 1.5).
- Front-end mastery dashboard / heatmap (covered in 9.3).
- Standards alignment metadata (covered in 1.3).
- Multi-tenant concept hierarchies (covered in 1.2 and 5.1–5.2).

## 4. Personas & User Stories

- **As a student**, I want my mastery of "quadratic equations" to carry over between quiz sessions so the system stops asking me questions I have already demonstrated proficiency on.
- **As an instructor**, I want to see which concepts each student is weak on so I can focus my office-hours time appropriately.
- **As an admin/institution**, I want to know the class-wide concept mastery distribution so I can identify topics where instruction is failing.
- **As a self-learner**, I want the platform to remember what I already know from my diagnostic assessment so I do not repeat beginner material.
- **As a platform engineer**, I want a stable internal API for reading learner state so new features do not query quiz attempt tables directly and bypass authorization.

## 5. Functional Requirements

- **FR-1.** The system MUST maintain a `learner_concept_states` record for each (user_id, concept_id) pair, storing at minimum: mastery probability (0.0–1.0), attempt count, last-seen timestamp, and a `needs_review_at` timestamp.
- **FR-2.** The system MUST update `learner_concept_states` atomically after every quiz attempt grading event, using the concept tags attached to each question (see 1.2).
- **FR-3.** The system MUST apply a time-decay function to mastery scores so that a concept not visited in N days gradually returns toward the prior; the decay half-life MUST be configurable per concept difficulty tier.
- **FR-4.** The system MUST expose a `GET /api/v1/learners/{user_id}/concepts` endpoint returning the full concept state vector for that learner, restricted to the caller's authorization scope (own record or instructor/admin of an enrolled course).
- **FR-5.** The system MUST record each mastery update event in an append-only `learner_concept_events` table for audit and replay.
- **FR-6.** The system SHOULD provide a bulk query path (`POST /api/v1/learners/concepts/batch`) accepting a list of user IDs for gradebook-style reporting without N+1 round trips.
- **FR-7.** The system SHOULD expose an internal Rust trait `LearnerStateService` so that `adaptive_quiz_ai.rs` and future services depend on an abstraction, not the database table directly.
- **FR-8.** The system MAY integrate a Bayesian Knowledge Tracing (BKT) parameter set per concept to improve mastery estimation beyond simple averaging; initial implementation MAY use a weighted exponential moving average as a simpler substitute.

## 6. Non-Functional Requirements

- **Performance** — `GET /api/v1/learners/{user_id}/concepts` MUST respond in p95 ≤ 50 ms for up to 500 concepts per learner. Batch endpoint MUST handle 200 users in ≤ 200 ms. All reads served from a Postgres index scan; no full-table scans.
- **Security** — Learner state is PII. All endpoints MUST require a valid JWT. Students MUST only read their own state. Instructors MUST only read state for enrolled learners. Admin tokens MUST be course-scoped via existing `authz.rs` middleware.
- **Privacy & Compliance** — Mastery data is FERPA-protected educational record. Must be included in any GDPR DSAR export (see 10.3). Must be erasable on right-to-erasure request. No raw mastery data sent to external LLM APIs without PII redaction (see 10.17).
- **Accessibility** — No direct UI in this feature; dependent UI (9.3) must be WCAG 2.1 AA.
- **Scalability** — Schema must support 100 k learners × 2 000 concepts = 200 M rows without query degradation; partition `learner_concept_states` by `user_id` hash if row counts exceed 50 M in production.
- **Reliability** — Mastery updates MUST be written inside the same Postgres transaction as quiz grading to prevent phantom updates on rollback. Idempotency key on `learner_concept_events` prevents duplicate event replay.
- **Observability** — Emit `mastery.updated` counter metric (Prometheus label: `concept_id`, `delta_direction`). Log at DEBUG level each update with `user_id` hash, `concept_id`, old/new score. Alert if update failure rate exceeds 0.1 % in a 5-minute window.
- **Maintainability** — `LearnerStateService` trait in `server/src/services/learner_state.rs`; implementations injected via Axum `State`. No business logic in route handlers.
- **Internationalization** — All concept labels are stored in `concepts` table (1.2) with locale support; mastery scores are numeric, locale-independent.
- **Backward compatibility** — No existing quiz or course tables are altered in a breaking way. The `questions_json` blob in `module_quizzes` remains valid; concept tagging is additive.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student completes a quiz with three questions tagged to concept `algebra.linear-equations`, *when* the quiz is graded, *then* a `learner_concept_states` row for that (student, concept) exists or is updated within 500 ms of grading completion.
- **AC-2.** *Given* a mastery score of 0.9 for a concept, *when* 30 days pass with no activity on that concept (simulated by advancing `last_seen_at`), *then* the effective mastery returned by the API is less than 0.9 (decay applied).
- **AC-3.** *Given* a student requests `GET /api/v1/learners/{their_user_id}/concepts`, *when* the request is authenticated, *then* the response is HTTP 200 and contains an array of concept-state objects with `concept_id`, `mastery`, `attempt_count`, `last_seen_at`, `needs_review_at`.
- **AC-4.** *Given* a student tries to read another student's concept state via `GET /api/v1/learners/{other_user_id}/concepts`, *when* the request is authenticated as the first student, *then* the response is HTTP 403.
- **AC-5.** *Given* a quiz grading transaction is rolled back due to a DB error, *when* the system recovers, *then* no orphaned `learner_concept_events` row exists for that attempt.
- **AC-6.** *Given* a GDPR erasure request is processed for a user, *when* the erasure runs, *then* all `learner_concept_states` and `learner_concept_events` rows for that user are deleted or anonymised.

## 8. Data Model

```sql
-- Migration: server/migrations/0030_learner_model.sql

-- Belongs to the `course` schema alongside quiz tables
CREATE TABLE course.learner_concept_states (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES "user".users(id) ON DELETE CASCADE,
    concept_id      UUID NOT NULL REFERENCES course.concepts(id) ON DELETE CASCADE,
    mastery         NUMERIC(5,4) NOT NULL DEFAULT 0.0 CHECK (mastery BETWEEN 0 AND 1),
    attempt_count   INTEGER NOT NULL DEFAULT 0,
    last_seen_at    TIMESTAMPTZ,
    needs_review_at TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (user_id, concept_id)
);

CREATE INDEX idx_lcs_user ON course.learner_concept_states (user_id);
CREATE INDEX idx_lcs_concept ON course.learner_concept_states (concept_id);
CREATE INDEX idx_lcs_needs_review ON course.learner_concept_states (needs_review_at)
    WHERE needs_review_at IS NOT NULL;

-- Append-only event log for audit / replay
CREATE TABLE course.learner_concept_events (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL,
    concept_id      UUID NOT NULL,
    attempt_id      UUID,                 -- FK to quiz attempt
    delta           NUMERIC(5,4) NOT NULL,
    mastery_after   NUMERIC(5,4) NOT NULL,
    source          TEXT NOT NULL,        -- 'quiz_grade' | 'manual' | 'decay'
    idempotency_key TEXT UNIQUE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_lce_user_concept ON course.learner_concept_events (user_id, concept_id);
```

Backfill: no backfill needed; historical quiz attempts do not have concept tags yet (requires 1.2 to be shipped first).

## 9. API Surface

```
GET  /api/v1/learners/{user_id}/concepts
  Auth: Bearer JWT (own user, or instructor/admin for enrolled course)
  Response 200: { concepts: ConceptState[] }

POST /api/v1/learners/concepts/batch
  Auth: Bearer JWT (instructor or admin scope)
  Body:   { user_ids: string[], concept_ids?: string[] }
  Response 200: { states: Record<UserId, ConceptState[]> }

GET  /api/v1/learners/{user_id}/concepts/{concept_id}
  Auth: Bearer JWT
  Response 200: ConceptState

interface ConceptState {
  concept_id: string;
  concept_name: string;
  mastery: number;         // 0.0–1.0
  attempt_count: number;
  last_seen_at: string | null;  // ISO 8601
  needs_review_at: string | null;
}
```

All new routes documented in OpenAPI via `utoipa` annotations consistent with existing `server/src/routes/` patterns.

## 10. UI / UX

No direct UI shipped with this feature. UI consumers:

- Skill heatmap dashboard (9.3) reads from the batch endpoint.
- Adaptive quiz selection in `server/src/services/adaptive_quiz_ai.rs` reads from `LearnerStateService`.
- Recommendations engine (1.8) queries learner state.

Empty state: if no concept states exist for a learner, the batch endpoint returns an empty array (not 404).

## 11. AI / ML Considerations

Initial mastery update algorithm: exponential moving average with `α = 0.3` (configurable). Formula: `mastery_new = mastery_old * (1 - α) + score * α` where `score` is the normalised question score (0 or 1 for binary, partial credit in [0,1]).

Time decay: `effective_mastery = mastery * exp(-λ * days_since_seen)` where `λ` is a per-difficulty-tier decay constant (default: `λ = 0.02` ≈ half-life 35 days). Decay applied lazily on read and persisted on next update.

Future upgrade path: replace EMA with Bayesian Knowledge Tracing (BKT) or IRT-3PL parameter estimation from 1.6 without changing the storage schema.

PII: `user_id` in prompts sent to LLMs MUST be replaced with a pseudonymous `session_token` before leaving the system (see 10.17).

## 14. Risks & Mitigations


| Risk                                                     | Likelihood | Impact | Mitigation                                                                                                  |
| -------------------------------------------------------- | ---------- | ------ | ----------------------------------------------------------------------------------------------------------- |
| Concept graph (1.2) delayed, blocking mastery updates    | M          | H      | Build stub concept table early; allow concept_id to be null initially and backfill when 1.2 ships           |
| Mastery model too simplistic for RFP buyers              | M          | M      | Document upgrade path to BKT/IRT; EMA is an accepted baseline in academic literature                        |
| High write volume during exam periods causing contention | L          | H      | Use `INSERT ... ON CONFLICT DO UPDATE` (upsert) with minimal locking; consider write batching via job queue |
| Privacy audit flags mastery data as sensitive            | M          | H      | Isolate in `course` schema, add row-level encryption column option, include in DSAR tooling from day one    |


## 15. Rollout Plan

- **Feature flag:** `adaptive_learner_model_enabled` (default: `false`).
- **Migration sequencing:** deploy migration → deploy code behind flag off → enable flag for internal test org → enable for pilot cohort → GA.
- **Pilot cohort:** 2–3 self-learner accounts and 1 instructor test course.
- **GA criteria:** mastery update latency p95 ≤ 50 ms in staging under load; no regression in existing quiz grading tests.
- **Rollback:** disable feature flag; mastery tables can remain (no data loss); grading path reverts to flag-off code path.

## 16. Test Plan

- **Unit** — `learner_state.rs` EMA computation; decay function; upsert idempotency with duplicate idempotency keys.
- **Integration** — complete quiz attempt → assert `learner_concept_states` row created/updated; rollback transaction → assert no orphaned event row.
- **End-to-end** — Playwright: student completes quiz, navigate to future mastery dashboard stub, assert concept list is non-empty.
- **Security** — authz matrix test: student reads own (200) vs. other student (403) vs. instructor for enrolled student (200) vs. instructor for non-enrolled student (403).
- **Accessibility** — N/A for this backend feature; UI consumers carry the obligation.
- **Performance / load** — k6 script: 500 concurrent learners each completing a 10-question quiz; assert p95 update latency ≤ 100 ms.
- **Manual exploratory** — verify decay after manual timestamp manipulation; verify batch endpoint pagination.

## 18. Open Questions

1. Should mastery decay run as a nightly background job (requires 17.3) or lazily on read? Lazy is simpler for MVP but may cause stale reads in reporting.
2. What is the canonical EMA alpha for the initial release — should it be tunable per-course or global?
3. Do we store raw IRT θ in this table now (NULL until 1.6 ships) or add the column in a later migration?
4. Should `learner_concept_events` be in a separate append-only Postgres schema or a time-series store (TimescaleDB) at scale?
