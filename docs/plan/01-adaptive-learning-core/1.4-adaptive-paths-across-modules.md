## 1. Problem Statement

Course structure in Lextures (`course_structure_items`) is entirely linear: every learner follows the same sequence regardless of prior knowledge. Students who have already mastered early material waste time on it; students struggling do not receive automatic remediation. True adaptive learning requires branching at the module level — skipping mastered content, inserting remedial modules for struggling learners, and dynamically reordering content based on concept mastery state. Without this, the "adaptive" label applies only inside a single quiz session, not across the learning journey.

## 2. Goals

- Introduce a rule-based branching layer on top of `course_structure_items` that routes learners to different next modules based on mastery thresholds.
- Support three core path rules: "skip if mastered," "require if not mastered," and "unlock after concept mastered."
- Make branching rules author-configurable via a visual UI in the course editor.
- Evaluate path rules lazily at the moment a learner requests "next content."
- Preserve instructor ability to override the adaptive path for any individual student.

## 3. Non-Goals

- Full graph-based learning path authoring (xAPI cmi5 Activity Tree, IMS Simple Sequencing) — rules-based approach only in v1.
- Automatic AI-generated path generation (covered in 1.8 recommendations engine).
- Real-time re-sequencing mid-module (only at module boundary transitions).
- Adaptive paths inside a single module's content blocks (within-module micro-adaptation is out of scope here).

## 4. Personas & User Stories

- **As an instructor**, I want to mark a review module as "skip if student has ≥ 80 % mastery of fractions" so advanced students are not bored.
- **As a student**, I want the system to automatically send me to a prerequisite review module when I score below 60 % on a concept quiz, rather than continuing to the next topic I am not ready for.
- **As a self-learner**, I want to complete a placement quiz and have the course start me at the right module rather than forcing me through material I already know.
- **As an instructor**, I want to manually override the adaptive path for one student who needs accommodations, without changing the path logic for the whole class.
- **As a platform developer**, I want the path evaluation engine to be side-effect-free and testable independently of routing logic.

## 5. Functional Requirements

- **FR-1.** The system MUST support attaching adaptive path rules to `course_structure_items`; each rule specifies: rule type (`skip_if_mastered` | `required_if_not_mastered` | `unlock_after`), a list of `concept_id`s, a mastery threshold (0.0–1.0), and an optional target `structure_item_id` to redirect to.
- **FR-2.** The system MUST evaluate path rules when a learner requests the next module (`GET /api/v1/enrollments/{id}/next`), returning the recommended next `course_structure_item` based on current mastery state from 1.1.
- **FR-3.** The system MUST record each path routing decision in a `learner_path_events` audit table (learner, from_item, to_item, rule_applied, timestamp).
- **FR-4.** The system MUST allow instructors to create a manual path override for a specific enrollment (bypass all rules for that learner and lock them to a specified item sequence).
- **FR-5.** The system MUST allow instructors to preview the adaptive path a hypothetical learner with given mastery scores would take, without creating real learner state.
- **FR-6.** The system SHOULD surface the recommended next item in the student course-progress UI with a short explanation (e.g., "Skipping this section — you already demonstrated mastery").
- **FR-7.** The system SHOULD support a global per-course toggle to disable adaptive paths and revert to linear ordering (for instructors who prefer traditional sequencing).
- **FR-8.** The system MAY support a "remediation module" rule type that inserts an extra module before continuing the main path when a learner falls below a threshold.

## 6. Non-Functional Requirements

- **Performance** — Next-item resolution MUST complete in p95 ≤ 100 ms including mastery lookup. Rule evaluation is in-memory after a single DB fetch of the learner's concept state vector.
- **Security** — Path override (manual enrollment lock) is instructor/admin-only. Learners cannot view or modify rules. Path event log is append-only and tamper-resistant.
- **Privacy & Compliance** — Path decisions constitute part of the educational record (FERPA). `learner_path_events` included in DSAR exports. No student identifiers in rule definitions (rules are course-level, not student-level).
- **Accessibility** — The "next" button and path explanation message MUST have ARIA live-region updates so screen-reader users hear when they are redirected.
- **Scalability** — Rule evaluation is O(rules × concepts) per request; at most ~50 rules per course item; fast enough without caching. Path event table partitioned by `created_at` month.
- **Reliability** — If mastery data is unavailable (e.g., 1.1 service error), system MUST fall back to linear ordering rather than blocking the learner.
- **Observability** — Prometheus counter `adaptive_path.skip_applied`, `adaptive_path.remediation_applied`. Alert if path evaluation error rate > 1 %.
- **Maintainability** — Rule evaluation logic in `server/src/services/adaptive_path.rs`; no SQL in the evaluation function (pure Rust, takes mastery map as input).
- **Internationalization** — Rule explanation messages (shown to learner) use i18n keys; mastery thresholds are locale-independent numerics.
- **Backward compatibility** — Courses without any path rules continue to work with linear ordering. Rule tables are additive; existing `course_structure_items` rows have null rule sets by default.

## 7. Acceptance Criteria

- **AC-1.** *Given* a module has a `skip_if_mastered` rule for concept `algebra.linear-equations` at threshold 0.8, *when* a student with mastery 0.85 requests their next item, *then* the API response routes to the module after the skipped one and includes a `skip_reason` field.
- **AC-2.** *Given* a module has a `required_if_not_mastered` rule at threshold 0.6, *when* a student with mastery 0.45 requests their next item, *then* the API response requires completion of the remediation target before continuing.
- **AC-3.** *Given* the mastery service returns an error, *when* a student requests their next item, *then* the system falls back to linear ordering and the response includes a `fallback: true` flag.
- **AC-4.** *Given* an instructor applies a manual path override for a specific enrollment, *when* that student requests their next item, *then* the path rules are bypassed and the override sequence is used.
- **AC-5.** *Given* a course has adaptive paths enabled, *when* `GET /api/v1/courses/{id}/adaptive-path?mastery={"concept_id":0.9}` is called by an instructor for preview, *then* the full predicted path is returned without creating any learner state.
- **AC-6.** *Given* a course-level toggle disables adaptive paths, *when* any student requests their next item, *then* linear ordering is used regardless of mastery state.

## 8. Data Model

```sql
-- Migration: server/migrations/0033_adaptive_paths.sql

CREATE TYPE course.path_rule_type AS ENUM (
    'skip_if_mastered', 'required_if_not_mastered', 'unlock_after', 'remediation_insert'
);

CREATE TABLE course.structure_item_path_rules (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    structure_item_id UUID NOT NULL REFERENCES course.course_structure_items(id) ON DELETE CASCADE,
    rule_type       course.path_rule_type NOT NULL,
    concept_ids     UUID[] NOT NULL,   -- concepts evaluated
    threshold       NUMERIC(4,3) NOT NULL CHECK (threshold BETWEEN 0 AND 1),
    target_item_id  UUID REFERENCES course.course_structure_items(id),
    priority        SMALLINT NOT NULL DEFAULT 0,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_sipr_item ON course.structure_item_path_rules (structure_item_id);

CREATE TABLE course.enrollment_path_overrides (
    enrollment_id   UUID NOT NULL REFERENCES course.enrollments(id) ON DELETE CASCADE,
    item_sequence   UUID[] NOT NULL,   -- ordered list of structure_item_ids
    created_by      UUID NOT NULL REFERENCES "user".users(id),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (enrollment_id)
);

CREATE TABLE course.learner_path_events (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    enrollment_id   UUID NOT NULL,
    from_item_id    UUID,
    to_item_id      UUID NOT NULL,
    rule_id         UUID REFERENCES course.structure_item_path_rules(id),
    was_override    BOOLEAN NOT NULL DEFAULT FALSE,
    was_fallback    BOOLEAN NOT NULL DEFAULT FALSE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);
```

## 9. API Surface

```
GET  /api/v1/enrollments/{id}/next
  Auth: student (own enrollment) | instructor | admin
  Response 200: { item: StructureItem, skip_reason?: string, fallback?: bool }

GET  /api/v1/courses/{id}/structure-items/{item_id}/path-rules
  Auth: instructor|admin
POST /api/v1/courses/{id}/structure-items/{item_id}/path-rules
DELETE /api/v1/courses/{id}/structure-items/{item_id}/path-rules/{rule_id}

GET  /api/v1/courses/{id}/adaptive-path?mastery=<json>
  Auth: instructor|admin  (path preview, no side effects)

PUT  /api/v1/enrollments/{id}/path-override
  Auth: instructor|admin
  Body: { item_sequence: string[] }
DELETE /api/v1/enrollments/{id}/path-override
```

## 10. UI / UX

**Course editor — adaptive path panel** (new side panel in module editor, `clients/web/src/pages/lms/`):
- For each module, a "Path Rules" expander showing current rules.
- Add rule form: pick rule type, select concepts via tag picker (from 1.2), set threshold slider, optionally pick target module.
- Path preview: animated flowchart showing hypothetical learner routes.

**Student course view** (`clients/web/src/pages/lms/CourseModuleQuizPage.tsx`):
- "Next" button tooltip showing skip reason when a module is skipped.
- Progress indicator shows active path (not all modules if some are skipped).
- ARIA live region announces "Skipping to [Module Name] — you have already mastered this topic."

Mobile: path rules editor collapses to a list; flowchart preview replaced by a sequential text list on narrow screens.

## 11. AI / ML Considerations

Rule evaluation is deterministic rule-based logic, not ML. The recommendations engine (1.8) will later layer probabilistic next-best-content on top of this rules layer. No LLM calls in this feature.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Rule misconfiguration creates infinite skip loops | M | H | Rule validator detects cycles in skip targets at save time; max depth guard at evaluation time |
| Learner confused by unexpected module skips | M | M | Always show skip-reason notification; learner can manually navigate to skipped module |
| Rules-based approach too rigid for complex courses | M | M | Design API to be extensible; 1.8 adds probabilistic layer later without schema change |
| Performance regression if mastery lookup is slow | L | M | Cache mastery vector per enrollment in Redis (15-min TTL) once job queue (17.3) ships |

## 15. Rollout Plan

- **Feature flag:** `adaptive_paths_enabled` (default: `false`, per-course toggle also available).
- **Migration sequencing:** deploy migration → ship rule CRUD API → ship student next-item logic behind flag → enable for pilot course → GA.
- **Pilot cohort:** one self-learner course and one K-8 math course with a mastered-vs-remediation split.
- **GA criteria:** path evaluation p95 ≤ 100 ms; no regression on existing course navigation tests; instructor path-rule UI usability tested with 2 instructors.
- **Rollback:** disable feature flag; all learners revert to linear ordering; no data loss.

## 16. Test Plan

- **Unit** — rule evaluation function with synthetic mastery maps: skip fires correctly, required fires correctly, fallback path on empty mastery.
- **Integration** — create rule → student completes quiz → mastery updated → next-item returns redirected item.
- **End-to-end** — Playwright: instructor creates skip rule, student logs in with high mastery, clicks "Next," verifies redirect and skip-reason tooltip.
- **Security** — student cannot modify path rules (403); instructor can; manual override only writable by instructor.
- **Accessibility** — ARIA live region fires on redirect; "Next" button accessible with keyboard; skip-reason tooltip readable by screen reader.
- **Performance / load** — k6: 1000 concurrent next-item requests with rule evaluation; p95 ≤ 100 ms.
- **Manual exploratory** — verify infinite-loop guard fires; verify fallback when mastery service is down.

## 18. Open Questions

1. Should path rules be per-section or per-enrollment-group (e.g., apply different paths to honours vs. standard section)?
2. How should the progress bar render for a course where different students are on different paths — show all modules or show personalised path only?
3. Should instructors be able to A/B test path configurations (random assignment of rule sets)?
4. Is the `UUID[]` array for `enrollment_path_overrides.item_sequence` sufficient, or should it be a separate ordered rows table for editability?
