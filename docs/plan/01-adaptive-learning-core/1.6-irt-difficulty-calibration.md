## 1. Problem Statement

Adaptive question selection in `adaptive_quiz_ai.rs` currently relies on LLM-estimated difficulty rather than empirically measured item parameters. This means high-stakes adaptive assessments (placement tests, formative diagnostics) have no psychometric grounding. Item Response Theory provides a principled framework for characterising item difficulty (b parameter), discrimination (a parameter), and guessing probability (c parameter), and for estimating a learner's ability (θ) on a latent trait. Without IRT, Lextures cannot make accurate ability estimates, cannot calibrate item banks for adaptive testing, and cannot produce defensible assessment reports for higher-education contexts.

## 2. Goals

- Store IRT parameters (2PL initially: a, b; 3PL as optional extension) for each question in the question bank.
- Estimate learner ability θ per concept after each adaptive quiz session using Maximum Likelihood Estimation (MLE) or Expected A Posteriori (EAP).
- Implement a Computerised Adaptive Testing (CAT) item-selection strategy (maximum information / next-item selection) as an alternative question-selection mode in `adaptive_quiz_ai.rs`.
- Provide an item-analysis report (difficulty, discrimination, distractor analysis) consumable by instructors (see 9.4).
- Define a calibration pipeline that updates IRT parameters from accumulated response data (background job, async).

## 3. Non-Goals

- Multidimensional IRT (MIRT) — 1D IRT per concept is sufficient for v1.
- Full CAT platform with stopping rules for licensing exams — the implementation targets formative / placement use, not high-stakes credentialing.
- Real-time IRT calibration (batch nightly calibration is sufficient for v1).
- Rasch model — the 2PL model is chosen as the best balance of simplicity and fit.
- Full psychometric validation report for regulatory purposes.

## 4. Personas & User Stories

- **As a student taking a placement assessment**, I want the question difficulty to adapt to my responses so I reach an accurate ability estimate in fewer questions than a fixed-length test.
- **As an instructor**, I want to see which questions discriminate well between strong and weak students (high a parameter) so I can retire poorly performing items.
- **As an assessment designer (HE)**, I want each question in the item bank to have calibrated difficulty parameters so I can assemble parallel test forms with matched difficulty.
- **As a platform data scientist**, I want a background calibration job that re-estimates IRT parameters nightly as new response data accumulates.
- **As a self-learner**, I want the adaptive quiz to home in on my current ability level quickly rather than asking trivially easy or impossibly hard questions.

## 5. Functional Requirements

- **FR-1.** The system MUST store IRT 2PL parameters (`a` discrimination, `b` difficulty) per question in the question bank table, nullable until calibrated, with a `calibration_status` enum.
- **FR-2.** The system MUST store learner ability `theta` per concept in `learner_concept_states` (additive column alongside mastery probability from 1.1).
- **FR-3.** The system MUST implement EAP-based θ estimation in `server/src/services/irt.rs` taking a vector of (question, response) pairs with IRT parameters and returning a posterior mean θ and standard error.
- **FR-4.** The system MUST integrate CAT item selection into `adaptive_quiz_ai.rs` as an opt-in mode: when `delivery_mode = 'cat'`, the next question is selected by maximising Fisher information at the current θ estimate.
- **FR-5.** The system MUST run an asynchronous calibration job (using the job queue from 17.3) that fits 2PL parameters via marginal maximum likelihood (EM algorithm) for any item with ≥ 200 response observations.
- **FR-6.** The system MUST record each θ estimate update in an append-only `learner_theta_events` table for audit.
- **FR-7.** The system SHOULD provide a per-item diagnostic report endpoint (`GET /api/v1/questions/{id}/irt-stats`) returning a, b, calibration_status, sample_size, and item-characteristic curve data points for visualisation.
- **FR-8.** The system SHOULD fall back to random item selection (within concept) when IRT parameters are not yet calibrated for an item pool.

## 6. Non-Functional Requirements

- **Performance** — θ estimation for a 30-item quiz MUST complete in ≤ 20 ms (pure computation, no DB). CAT next-item selection over a 500-item pool MUST complete in ≤ 50 ms.
- **Security** — IRT parameters are instructor/admin-readable. θ estimates are learner-private (same access rules as mastery in 1.1). Calibration job runs with service-account DB credentials; no JWT path.
- **Privacy & Compliance** — Learner θ values are educational record (FERPA). Included in GDPR DSAR export and erasure. Raw response data used for calibration does not leave the cluster (no external ML API calls).
- **Accessibility** — No direct UI for IRT computation; item analysis report UI (9.4) must be WCAG 2.1 AA compliant.
- **Scalability** — Calibration EM algorithm runs off-peak (nightly); shards by concept to allow parallel jobs. Expect 50 k items × 10 k responses each = 500 M response records over time; response log table must be partitioned by date.
- **Reliability** — Calibration failure for one concept MUST NOT block calibration of others. Each calibration job is idempotent and re-runnable.
- **Observability** — Prometheus gauge `irt.calibrated_item_count`, `irt.uncalibrated_item_count`. Log calibration runs with item count, parameter estimates, and convergence status. Alert if θ estimation error rate > 0.5 %.
- **Maintainability** — IRT math in `server/src/services/irt.rs` as pure functions with no I/O. Calibration orchestration in a separate `server/src/jobs/irt_calibration.rs`.
- **Internationalization** — IRT parameters are numeric; item analysis labels (e.g., "difficulty", "discrimination") use i18n keys in the UI.
- **Backward compatibility** — `a`, `b`, `c` columns added nullable to question bank; existing rows have `calibration_status = 'uncalibrated'`; no change to quiz delivery until `delivery_mode = 'cat'` is explicitly set.

## 7. Acceptance Criteria

- **AC-1.** *Given* 200 response observations for an item with true difficulty b=0.5, *when* the calibration job runs, *then* the estimated b parameter is within ±0.3 of the true value.
- **AC-2.** *Given* a CAT quiz with `delivery_mode = 'cat'` and a 50-item calibrated pool, *when* a learner answers the first item correctly, *then* the second item selected has higher b (difficulty) than the first.
- **AC-3.** *Given* a 30-item adaptive quiz session, *when* θ estimation runs, *then* the returned θ value is a finite float in range [-4, 4] and the standard error is positive.
- **AC-4.** *Given* an item pool where no items are calibrated, *when* CAT mode is requested, *then* the system falls back to concept-tagged random selection and logs a warning.
- **AC-5.** *Given* an instructor requests `GET /api/v1/questions/{id}/irt-stats`, *when* the item is calibrated, *then* the response includes `a`, `b`, `sample_size`, and an array of (theta, p_correct) pairs for drawing the ICC.
- **AC-6.** *Given* a GDPR erasure runs for a learner, *when* complete, *then* all `learner_theta_events` rows for that learner are deleted and the corresponding `learner_concept_states.theta` column is nulled.

## 8. Data Model

```sql
-- Migration: server/migrations/0035_irt.sql

CREATE TYPE course.irt_calibration_status AS ENUM (
    'uncalibrated', 'pilot', 'calibrated', 'retired'
);

-- Additive columns on the question bank table (2.1 ships first)
ALTER TABLE course.questions
    ADD COLUMN irt_a          NUMERIC(6,4),   -- discrimination (2PL)
    ADD COLUMN irt_b          NUMERIC(6,4),   -- difficulty (2PL)
    ADD COLUMN irt_c          NUMERIC(6,4),   -- guessing (3PL, optional)
    ADD COLUMN irt_status     course.irt_calibration_status NOT NULL DEFAULT 'uncalibrated',
    ADD COLUMN irt_sample_n   INTEGER NOT NULL DEFAULT 0,
    ADD COLUMN irt_calibrated_at TIMESTAMPTZ;

CREATE INDEX idx_q_irt_status ON course.questions (irt_status);

-- Learner theta per concept (additive column in 1.1 table)
ALTER TABLE course.learner_concept_states
    ADD COLUMN theta        NUMERIC(6,4),
    ADD COLUMN theta_se     NUMERIC(6,4);   -- standard error of theta estimate

CREATE TABLE course.learner_theta_events (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id      UUID NOT NULL,
    concept_id   UUID NOT NULL,
    attempt_id   UUID,
    theta        NUMERIC(6,4) NOT NULL,
    theta_se     NUMERIC(6,4),
    items_n      INTEGER NOT NULL,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);
```

## 9. API Surface

```
GET  /api/v1/questions/{id}/irt-stats
  Auth: instructor|admin
  Response: { a, b, c, status, sample_n, icc: [{theta, p_correct}] }

GET  /api/v1/learners/{user_id}/concepts/{concept_id}/theta
  Auth: own user | instructor | admin
  Response: { theta, theta_se, last_updated_at }

POST /api/v1/admin/jobs/irt-calibrate
  Auth: admin (triggers manual calibration run)
  Body: { concept_id?: string }
  Response 202: { job_id }
```

**CAT progress indicator** (in quiz-taking UI, `clients/web/src/pages/lms/CourseModuleQuizPage.tsx`):
- Optional θ confidence-interval bar showing learner ability estimate evolving across items.
- No raw θ number shown to learner; only qualitative label ("Getting harder — keep going!").

## 11. AI / ML Considerations

All ML computation is on-cluster using pure Rust (or a compiled C library via FFI if needed for EM). No external ML API calls. The EM calibration algorithm is based on the standard Bock-Aitkin (1981) marginal MLE approach. Initial implementation may use a simplified fixed-point iteration over the 2PL log-likelihood for tractability in Rust.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| EM algorithm fails to converge for sparse response data | M | M | Require ≥ 200 responses before calibrating; use informative priors (N(0,1) on b) |
| Instructors misinterpret IRT parameters | M | L | Show plain-language interpretations alongside parameters; hide raw values behind a toggle |
| CAT exposure control not implemented → some items over-exposed | M | M | Add maximum-exposure-rate constraint to item selection (Sympson-Hetter method, v2) |
| Implementing EM in Rust is complex | H | M | Evaluate wrapping the R `mirt` package via subprocess as interim; ship pure Rust in v2 |

## 15. Rollout Plan

- **Feature flag:** `irt_cat_mode_enabled` (default: `false`; applies per quiz delivery setting).
- **Migration sequencing:** add IRT columns (nullable) → calibration job runs nightly → enable CAT mode per-quiz flag → GA after 30-day calibration window.
- **Pilot cohort:** one HE intro course with 500 + students providing response data for calibration.
- **GA criteria:** ≥ 100 items calibrated with sample_n ≥ 200; CAT selection p95 ≤ 50 ms; calibration job completes without errors.
- **Rollback:** disable `irt_cat_mode_enabled`; IRT columns remain (data preserved); quiz falls back to non-CAT mode.

## 16. Test Plan

- **Unit** — 2PL ICC function; Fisher information function; EAP θ estimation against published reference values (Baker & Kim 2004 worked examples).
- **Integration** — Seed 300 synthetic response records for an item → run calibration → verify a and b within tolerance of synthetic truth.
- **End-to-end** — Playwright: student takes 10-item CAT quiz; verify each next item is fetched; verify θ estimate in learner state after completion.
- **Security** — Instructor reads item IRT stats (200); student cannot (403); admin can trigger manual calibration (202).
- **Accessibility** — ICC chart has accessible alternative text table; CAT progress indicator has ARIA live region.
- **Performance / load** — θ estimation for 30 items: < 5 ms (unit benchmark); CAT item selection over 500-item pool: < 50 ms.
- **Manual exploratory** — Verify fallback to random selection when pool is uncalibrated; verify θ resets correctly on GDPR erasure.
