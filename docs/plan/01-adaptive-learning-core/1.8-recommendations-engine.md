## 1. Problem Statement

The RAG notebook (`student_notebook_rag_ai.rs`) is reactive — it answers questions a learner explicitly asks. There is no proactive "next best action" surface. Learners navigating a large course library or a multi-module course cannot discover the most useful content for their current state. Self-learners especially expect Duolingo/Khan-style "here's what to do next" nudges rather than passive content browsing. Without a recommendations engine, the learner model and concept graph remain unused outside of adaptive quizzes, and the product cannot deliver on its adaptive promise beyond assessment sessions.

## 2. Goals

- Build a rule-and-signal-based recommendations engine that produces a ranked list of "next best content" items for a learner at any point in time.
- Combine signals: concept mastery gaps (1.1), SRS due items (1.5), prerequisite completion (1.2), course progression (1.4), and recency of content engagement.
- Expose a `GET /api/v1/learners/{id}/recommendations` endpoint consumed by dashboard and course-home widgets.
- Support three recommendation surfaces: "continue learning" (resume interrupted module), "strengthen" (review weak concepts), and "challenge" (stretch to next concept).
- Allow instructors to pin or suppress specific recommendations for their course.

## 3. Non-Goals

- Collaborative filtering / matrix factorisation across learners (v2; requires data volume not yet available).
- Recommendations for content not yet enrolled in (discovery / catalog browsing, covered in 15.1).
- Real-time streaming recommendations (batch-computed and cached is sufficient for v1).
- Personalised learning path generation from scratch (this engine reorders and prioritises existing paths, not generates new ones).

## 4. Personas & User Stories

- **As a self-learner**, I want a "What's next?" card on my home dashboard showing the single most impactful thing I can do today, so I don't have to decide.
- **As a student**, I want the system to notice when I have been stuck on a concept for two sessions and suggest supplementary material or a worked example.
- **As an instructor**, I want to pin a particular module as the top recommendation for all students after a midterm, so they focus on exam review.
- **As a self-learner**, I want recommendations to interleave review (spaced repetition) with new content so I both remember and progress.
- **As a developer**, I want the recommendation scorer to be a pure function so I can test it with synthetic learner states without touching the database.

## 5. Functional Requirements

- **FR-1.** The system MUST implement a `RecommendationScorer` that takes a `LearnerState` (from 1.1), a `ConceptGraph` snapshot (from 1.2), a list of `SrsItemsDue` (from 1.5), and a course `ProgressState` (from 1.4), and returns a ranked `Vec<Recommendation>`.
- **FR-2.** The system MUST expose `GET /api/v1/learners/{user_id}/recommendations?course_id=&surface=` returning up to 10 recommendations per surface (continue/strengthen/challenge).
- **FR-3.** The system MUST support `surface` parameter values: `continue` (resume in-progress item), `strengthen` (improve mastery of weak concept), `challenge` (advance to next unmastered concept), `review` (SRS due items).
- **FR-4.** The system MUST cache recommendations per learner in a short-lived store (5-minute TTL) to avoid recomputation on every page load.
- **FR-5.** The system MUST allow instructors to create a `course_recommendation_pin` or `course_recommendation_suppress` record per course that boosts or demotes specific items.
- **FR-6.** The system MUST record recommendation impressions and click-through events in `recommendation_events` for scorer iteration.
- **FR-7.** The system SHOULD incorporate engagement recency: content viewed in the last 3 days is down-ranked in "continue" surface to encourage breadth.
- **FR-8.** The system MAY compute recommendations asynchronously as a background job (17.3) and store results in `recommendation_cache`, returning cached results on read.

## 6. Non-Functional Requirements

- **Performance** — Recommendation read (cache hit) p95 ≤ 50 ms. Scorer computation (cache miss) p95 ≤ 500 ms. Background pre-computation runs within 5 min of mastery update.
- **Security** — Recommendations are learner-private. Instructor override endpoints require course-instructor role. No recommendations cross enrollment boundaries without explicit instructor cross-course pin.
- **Privacy & Compliance** — Recommendation signals derived from learner state are educational record (FERPA). Recommendation click-through logs contain user_id; included in DSAR exports. No learner signals sent to external ML APIs without PII redaction.
- **Accessibility** — Recommendation cards on dashboard MUST have descriptive aria-labels. "What's next" card must not use color alone to distinguish surfaces.
- **Scalability** — Pre-computation job processes all active learners (active = logged in within 7 days). At 50 k active learners, job must complete within 30 min on 4-core background worker.
- **Reliability** — If scorer errors, API returns empty recommendations with `degraded: true` rather than 500. Dashboard widget gracefully hides when recommendations are empty.
- **Observability** — Prometheus gauge `recommendations.cache_hit_ratio`. Counter `recommendations.impression` and `recommendations.click` (labeled by surface). Alert if scorer error rate > 2 %.
- **Maintainability** — `RecommendationScorer` in `server/src/services/recommendations.rs` as a pure struct with no I/O. Cache store injected via trait. Scorer weights configurable via feature config (not hardcoded).
- **Internationalization** — Recommendation label strings (surface names, reason phrases) use i18n keys. Concept names use locale from 1.2.
- **Backward compatibility** — Recommendations are an additive read surface; no existing data modified.

## 7. Acceptance Criteria

- **AC-1.** *Given* a learner has mastery < 0.5 on concept X and the course contains a module tagged to X, *when* recommendations are requested with `surface=strengthen`, *then* that module appears in the response.
- **AC-2.** *Given* a learner has 5 SRS items due today, *when* recommendations are requested with `surface=review`, *then* all 5 items are present in the response ranked by overdue-ness.
- **AC-3.** *Given* recommendations are cached and the learner's mastery updates within the TTL window, *when* recommendations are requested, *then* the stale cache is served and a background recomputation is triggered.
- **AC-4.** *Given* an instructor pins module Y for all students in a course, *when* any student in that course requests recommendations, *then* module Y appears first in the `continue` surface regardless of mastery state.
- **AC-5.** *Given* the scorer throws an error, *when* the recommendation endpoint is called, *then* the response is HTTP 200 with `{ recommendations: [], degraded: true }` rather than a 500.
- **AC-6.** *Given* a learner clicks a "strengthen" recommendation, *when* the click event is recorded, *then* a `recommendation_events` row exists with `event_type = 'click'`, `surface = 'strengthen'`, and the item ID.

## 8. Data Model

```sql
-- Migration: server/migrations/0037_recommendations.sql

CREATE TABLE course.course_recommendation_overrides (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    course_id       UUID NOT NULL REFERENCES course.courses(id) ON DELETE CASCADE,
    structure_item_id UUID NOT NULL REFERENCES course.course_structure_items(id),
    override_type   TEXT NOT NULL CHECK (override_type IN ('pin','suppress')),
    surface         TEXT,    -- NULL = all surfaces
    created_by      UUID NOT NULL REFERENCES "user".users(id),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE course.recommendation_cache (
    user_id         UUID NOT NULL REFERENCES "user".users(id) ON DELETE CASCADE,
    course_id       UUID NOT NULL REFERENCES course.courses(id) ON DELETE CASCADE,
    surface         TEXT NOT NULL,
    recommendations JSONB NOT NULL,
    computed_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    expires_at      TIMESTAMPTZ NOT NULL,
    PRIMARY KEY (user_id, course_id, surface)
);

CREATE TABLE course.recommendation_events (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL,
    course_id       UUID NOT NULL,
    item_id         UUID,
    surface         TEXT NOT NULL,
    event_type      TEXT NOT NULL CHECK (event_type IN ('impression','click','dismiss')),
    rank            SMALLINT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);
```

## 9. API Surface

```
GET  /api/v1/learners/{user_id}/recommendations
  Auth: own user | instructor | admin
  Query: ?course_id=&surface=continue|strengthen|challenge|review&limit=10
  Response: { recommendations: Recommendation[], degraded?: bool }

POST /api/v1/recommendations/event
  Auth: own user
  Body: { item_id, surface, event_type: 'impression'|'click'|'dismiss', rank }

POST /api/v1/courses/{id}/recommendation-overrides   Auth: instructor|admin
DELETE /api/v1/courses/{id}/recommendation-overrides/{id}

interface Recommendation {
  item_id: string;
  item_type: 'module' | 'quiz' | 'review_card';
  title: string;
  surface: string;
  reason: string;   // i18n key + params for "You have not reviewed X in 5 days"
  score: number;
}
```

## 10. UI / UX

**Dashboard "What's next" widget** (home page):
- Single primary recommendation card (highest-scored across all surfaces).
- Secondary row: up to 3 surface-labeled chips ("Review 5 items", "Strengthen: Fractions", "Challenge: Decimals").
- "More recommendations" expander.
- Empty state: "You're all caught up! Check back tomorrow."

**Course home sidebar** (new panel in course view):
- Per-surface recommendation lists.
- Instructor-pinned items marked with a pin icon.

Accessibility: all recommendation cards have `role="article"` with descriptive `aria-label`; reason text is programmatic, not only visual.

## 11. AI / ML Considerations

V1 is rule-and-signal-based (no ML model). The `RecommendationScorer` uses a weighted scoring formula:
- `strengthen`: score = (1 - mastery) × concept_importance_weight
- `challenge`: score = readiness (all prerequisites mastered above threshold)
- `review`: score = days_overdue × 0.2
- `continue`: score = recency_inverse + progress_pct

V2 upgrade path: replace scorer with a lightweight bandit model (Thompson sampling on impression/click data) or collaborative filtering once 10 k click events are available.

No LLM calls in v1. Future: LLM-generated personalised "reason" text ("You haven't practiced long division since Tuesday and your quiz is next week").

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Scorer recommends the same item repeatedly | M | M | Add "already seen" de-duplication and recency down-ranking |
| Recommendations are too narrow (only "strengthen") | M | M | Enforce minimum diversity: at least one item per available surface in top-10 |
| Pre-computation job overwhelms DB during peak | L | M | Rate-limit job to 100 learners/min; use read replica for mastery queries |
| Instructors over-pin items, degrading personalisation | L | L | Cap pins to 3 per course; show warning when all slots used |

## 15. Rollout Plan

- **Feature flag:** `recommendations_enabled` (default: `false`).
- **Migration sequencing:** deploy migration → deploy scorer + cache → ship dashboard widget behind flag → ship course sidebar → GA.
- **Pilot cohort:** 20 self-learner accounts; monitor click-through rate to validate scorer relevance.
- **GA criteria:** CTR on recommendations ≥ 15 %; degraded response rate < 1 %; cache hit ratio ≥ 90 %.
- **Rollback:** disable flag; dashboard widget hides; no data corruption.

## 16. Test Plan

- **Unit** — Scorer function with synthetic learner states: verify strengthen surface ranks weakest concepts first; challenge surface excludes items with unmet prerequisites; review surface ranks most-overdue first.
- **Integration** — Seed mastery data → call recommendations endpoint → verify response contains expected surfaces; instructor pin → verify pin appears first.
- **End-to-end** — Playwright: log in as learner, see "What's next" widget, click recommendation, verify navigation to correct module; verify click event recorded.
- **Security** — Learner cannot read another learner's recommendations; instructor can read course-level recommendations for enrolled student.
- **Accessibility** — axe scan on dashboard widget; recommendation reason text read by screen reader; keyboard focus order through recommendation cards.
- **Performance / load** — k6: 1000 concurrent recommendation reads; p95 ≤ 50 ms (cache hits); scorer computation under 500 ms for cache miss.
- **Manual exploratory** — Verify "degraded" flag when scorer errors; verify empty recommendations show correct empty state; verify cache invalidation after mastery update.

## 18. Open Questions

1. Should the cache be in Postgres (`recommendation_cache` table) or Redis? Postgres is simpler but may have write contention; Redis requires an additional service.
2. Should we expose the `score` field to learners or only to admins (for debugging)?
3. How do we handle recommendations when a learner is enrolled in multiple courses simultaneously — aggregate across courses or per-course only?
4. Should the recommendation event log be used for real-time scorer updates or only periodic (weekly) weight adjustment?
