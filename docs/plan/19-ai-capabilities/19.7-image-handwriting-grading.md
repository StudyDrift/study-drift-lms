## 1. Problem Statement

K-12 mathematics education depends heavily on students showing their work. A student photographing a multi-step algebra solution on paper from their phone has no current path to submitting it in Lextures for graded feedback. Vision-capable large language models (GPT-4o, Claude 3.5 Sonnet, Gemini 1.5 Pro) can now evaluate handwritten math solutions, identify step-level errors, and provide targeted feedback. This is a high-value, low-barrier-to-entry differentiator for K-12 STEM courses.

## 2. Goals

- Allow students to photograph or upload images of handwritten work as assignment submissions.
- Use vision LLMs to interpret handwriting, evaluate mathematical correctness step-by-step, and generate rubric-aligned feedback.
- Feed identified errors into the misconception store (§19.4).
- Keep instructors in the approval loop (§19.3 pattern).
- Support accessibility use cases where typing math is not feasible.

## 3. Non-Goals

- Real-time camera-to-grade feedback during in-class work (too speculative).
- Printed (typed) document image grading (covered by §19.3 PDF extraction).
- Diagram/figure grading beyond mathematical expressions.
- General image analysis unrelated to student work.

## 4. Personas & User Stories

- **As a middle school math student**, I want to photograph my worked solution on paper and submit it from my phone so that I get the same feedback as students who type their answers.
- **As a math teacher**, I want the AI to mark each step in the student's solution as correct/incorrect so that I can give targeted feedback without reading 30 papers by hand.
- **As a student with a writing disability**, I want to submit handwritten work as an IEP accommodation so that I am not disadvantaged by a typing-only interface.
- **As an instructor**, I want to verify the AI's interpretation before the score reaches the student so that misread handwriting doesn't result in an unfair grade.

## 5. Functional Requirements

- **FR-1.** The system MUST support image file uploads (JPEG, PNG, WebP, HEIC up to 20 MB) as a submission type on assignments where the instructor has enabled `handwriting_submission: true`.
- **FR-2.** The system MUST support live camera capture within the web browser (via `<input type="file" capture>` / `getUserMedia`) for mobile users.
- **FR-3.** The system MUST send submitted images to a vision LLM with the assignment rubric and question prompt, requesting: OCR transcription of handwriting, step-by-step evaluation, and an overall rubric score.
- **FR-4.** The AI-generated evaluation MUST be placed in the instructor approval queue (same workflow as §19.3) before any score reaches the student.
- **FR-5.** The system MUST store the original image in object storage (§8.1) and the AI transcript/evaluation as text alongside it.
- **FR-6.** Error-step analysis MUST be forwarded to the misconception store (§19.4).
- **FR-7.** The system MUST apply PII redaction to any recognized text before further LLM calls (§19.11).
- **FR-8.** If the vision LLM cannot reliably read the handwriting (low-confidence indicator), the submission MUST be routed to manual grading with a notification.
- **FR-9.** The system MUST display the AI transcript to the student alongside the original image after instructor approval so students can verify the system read their work correctly.

## 6. Non-Functional Requirements

- **Performance** — Vision LLM call for a single image MUST complete within 20 s; job queue ensures non-blocking submission.
- **Security** — Images stored with org-scoped access control; signed short-lived URLs for display; images processed via PII-aware pipeline.
- **Privacy & Compliance** — Handwritten work is FERPA-protected; images may contain minor's face if photo includes background — pre-processing guidance advised; COPPA consent required for under-13; image data not retained by LLM provider per DPA.
- **Accessibility** — Transcript text alternative provided for every handwritten submission; camera capture keyboard accessible with fallback upload.
- **Scalability** — Vision API calls are inherently slower; job queue prevents blocking; max 50 concurrent vision jobs per instance.
- **Reliability** — Transcription failure → manual queue; image preserved regardless.
- **Observability** — `vision_grading_jobs_total{status}`, `vision_confidence_mean`, `vision_fallback_to_manual_total`.
- **Maintainability** — Service: `server/src/services/vision_grading.rs`; provider adapter for vision LLM swap.
- **Internationalization** — Handwriting recognition language code passed to vision model.
- **Backward compatibility** — Image submission is a new submission type; existing types unchanged.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student photographs a 3-step algebra solution and submits it, *When* the vision job completes, *Then* the instructor approval queue shows the original image, an AI transcript of the handwriting, and per-step evaluation.
- **AC-2.** *Given* the vision LLM returns confidence < 0.5 for a submission, *When* the job completes, *Then* the submission is routed to manual grading and the instructor is notified.
- **AC-3.** *Given* the instructor approves the AI-suggested score, *When* the student views their result, *Then* they see the original image, the AI transcript (so they can verify it was read correctly), and the feedback.
- **AC-4.** *Given* a vision LLM evaluation identifies an incorrect step in a math problem, *When* the evaluation is processed, *Then* a misconception record is created for the student with the relevant concept (§19.4).
- **AC-5.** *Given* an image submitted by a student, *When* any recognized text is present, *Then* the PII proxy is applied to the extracted text before subsequent LLM grading calls.

## 8. Data Model

```sql
-- 0XX_vision_submissions.sql
CREATE TABLE vision_submissions (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  submission_id     UUID NOT NULL REFERENCES assignment_submissions(id),
  storage_key       TEXT NOT NULL,
  mime_type         TEXT NOT NULL,
  ai_transcript     TEXT,
  ai_evaluation     JSONB,        -- { steps: [{step_num, correct, feedback}], score, rubric_breakdown }
  confidence        NUMERIC,
  vision_model      TEXT,
  processing_status TEXT NOT NULL DEFAULT 'pending',
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 10. UI / UX

- Assignment submission page: "Upload handwritten work" section with drag-drop and camera capture button.
- Mobile camera capture: full-screen camera overlay with edge-detection guidance.
- Instructor approval queue: image thumbnail + transcript side-by-side; step-level evaluation with green/red indicators per step.
- Student result: image + transcript + step feedback; "Contact instructor if transcript is incorrect" CTA.

## 11. AI / ML Considerations

- **Model**: Claude 3.5 Sonnet vision capabilities (Anthropic API) or GPT-4o (OpenAI API); configurable per §19.10 governance.
- **Prompt**: Include assignment question, rubric, and instruction to evaluate step-by-step; return structured JSON.
- **Handwriting quality**: Pre-process with image sharpening; request confidence score from model.
- **Eval metric**: Step-correctness accuracy vs. human grader on 50 math-problem test set; target ≥ 80% agreement.
- **Fallback**: Confidence < 0.5 or provider error → manual grading queue.
- **Model card**: `docs/ai/vision-grading-model-card.md`.
- **Cost**: Vision API ~$0.003–0.01 per image; budget per §19.14.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Handwriting illegible → wrong grade | H | H | Confidence gate; mandatory instructor review; student can flag misread |
| Image contains student's face (FERPA/privacy) | M | M | Guidance to photograph only work; processing note in student UI |
| Vision API costs high for image-heavy courses | M | M | Per-org monthly budget and alert (§19.14) |
| Provider model changes degrade accuracy | M | M | Eval harness regression suite (§19.13) |

## 15. Rollout Plan

- Feature flag: `handwriting_grading_enabled` (OpenFeature, default `false`).
- Phase 1: K-12 math pilot (opt-in, instructor-enabled per assignment).
- Phase 2: Eval harness validation; accuracy threshold ≥ 80% before broader rollout.
- Phase 3: GA for K-12; expand to HE STEM.

## 16. Test Plan

- **Unit** — Image upload; vision API response parsing; confidence threshold routing.
- **Integration** — Upload image → vision job → approval queue → student result.
- **AI eval** — 50 math solutions with human grader ground truth; step accuracy ≥ 80%.
- **Security** — Signed URL expiry; cross-student image access blocked.
- **Accessibility** — Camera capture keyboard accessible; transcript text alternative present on all images.
