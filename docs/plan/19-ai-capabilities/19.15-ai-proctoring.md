## 1. Problem Statement

Vendor proctoring (14.9) is expensive (often $5–15/exam) and inaccessible to self-learners, small institutions, and low-stakes formative assessments. Today, Lextures has no first-party way to deter or detect dishonest test-taking; instructors fall back to design-time mitigations (question banks, time limits) that don't address active cheating. A built-in AI proctoring layer — combining browser focus tracking, webcam/microphone anomaly detection, and input-pattern analysis — gives instructors a calibrated *integrity signal* without forcing students through a third-party vendor flow, and gives self-learners a credible way to certify their own practice exams. The output is **decision support for an instructor reviewer**, never an automated cheating verdict.

## 2. Goals

- Provide a configurable, per-assignment proctoring profile that combines client-side telemetry (focus, mouse, keystrokes, fullscreen) with optional media monitoring (webcam, microphone, screen).
- Run all media inference on a tiered pipeline (in-browser models first, server-side LLM/vision only for escalations) to control cost and minimize raw biometric egress.
- Produce a per-attempt **integrity report** — a ranked, timestamped event log with severity and short natural-language rationale — surfaced to instructors as decision support.
- Enforce informed consent, parental gating (COPPA), and biometric-law obligations (BIPA, TDPSA, GDPR Art. 9) before any media stream is captured.
- Ship with strong accessibility safeguards so students with disabilities, assistive tech, or non-standard environments are not flagged for behaviors outside their control.

## 3. Non-Goals

- Automatic disqualification, score adjustment, or disciplinary action from AI signals — instructor review is mandatory.
- Live human proctors (out of scope; 14.9 covers vendor-supplied live proctoring).
- Replacing vendor integrations from 14.9 — both options remain; AI proctoring is the in-house low-cost path.
- Persistent identity verification / ID-document OCR at exam start (separate plan; see Open Question 1).
- Network-level / device-level lockdown beyond what 2.10 (lockdown/kiosk mode) already provides.
- Continuous monitoring outside of an active proctored attempt window.

## 4. Personas & User Stories

- **As an instructor**, I want to enable AI proctoring on a midterm so that I receive a flagged-events timeline per student to review during grading, without paying per-seat for Honorlock.
- **As an instructor**, I want to choose which signals are collected (focus only / + webcam / + microphone / + screen) so that I can match the proctoring intensity to the stakes of the assessment.
- **As a student**, I want a clear pre-exam disclosure of exactly what is recorded, where it is stored, for how long, and who can see it, so that I can give informed consent before I begin.
- **As a student with a documented accommodation** (e.g., assistive tech that triggers tab switches, or a face-detection model that fails on my appearance), I want a documented appeal path so that I am not penalized for legitimate behavior.
- **As a parent of an under-13 student** (K-12), I want a verifiable consent step before any audio or video of my child is captured.
- **As an admin**, I want to disable webcam/microphone proctoring institution-wide so that I can ship a "focus & input signals only" profile when our DPA does not cover biometrics.
- **As a self-learner**, I want to self-proctor a certification practice exam so that the resulting score carries more credibility on my profile.
- **As a privacy / compliance officer**, I want a per-attempt audit record of which signals were collected, who viewed the report, and when raw artifacts were deleted.

## 5. Functional Requirements

- **FR-1.** The system MUST define a `proctoring_profile` resource at the assignment level with toggles for: `focus_tracking`, `input_pattern_analysis`, `webcam`, `microphone`, `screen_share`, and `lockdown_mode_required` (delegates to 2.10).
- **FR-2.** Before any media capture, the system MUST present an explicit, plain-language consent screen listing every collected signal, its retention period, and the consequences of declining. The student MUST explicitly accept; declining MUST route to instructor-defined fallback (e.g., in-person makeup, alternate assignment).
- **FR-3.** For users flagged as under 13, webcam, microphone, and screen-share signals MUST be hard-disabled regardless of instructor configuration unless a verified parental-consent record exists (§10.2). Focus and input-pattern signals MAY proceed under existing district consent.
- **FR-4.** The system MUST capture *client-side focus and input telemetry* during a proctored attempt — at minimum: `visibilitychange`, `blur`/`focus`, `fullscreenchange`, paste events, suspicious keystroke patterns (e.g., autocomplete-shaped bursts), and pointer-leave-viewport events.
- **FR-5.** When webcam is enabled, the system MUST run an **in-browser** face-presence and gaze-zone model (e.g., MediaPipe Face Landmarker via WASM) and emit only derived events (`face_absent`, `multiple_faces`, `gaze_off_screen_>5s`) and low-resolution thumbnails at event boundaries — not the continuous video stream.
- **FR-6.** When microphone is enabled, the system MUST run an in-browser voice-activity-detection (VAD) model and emit only event-level signals (`speech_detected_during_silent_window`, `second_voice_detected`); raw audio MUST NOT be streamed continuously to the server.
- **FR-7.** When screen-share is enabled, the system MUST request `getDisplayMedia` with the **entire screen** scope and emit periodic perceptual-hash snapshots (default: every 15 s) for off-LMS application detection; raw video MUST NOT leave the browser.
- **FR-8.** For each emitted event, the system MUST compute a **severity** (`info` / `low` / `medium` / `high`) using a deterministic rule table; severity computation MUST be reproducible and auditable.
- **FR-9.** The system SHOULD escalate `high`-severity event clusters to a server-side LLM-based **rationale generator** that ingests the event log (text + thumbnails) and produces a short natural-language summary for the instructor; raw thumbnails MUST be passed through the PII proxy (19.11) — student face images MUST be blurred or hashed before LLM submission.
- **FR-10.** The system MUST produce a per-attempt **integrity report** containing: ordered event log, severity counts, optional LLM rationale, link to retained artifacts, and a prominent banner: "AI proctoring signals are decision support. They are not evidence of cheating."
- **FR-11.** The system MUST NOT block, freeze, or auto-submit the student's quiz attempt based on AI signals (lockdown 2.10 may still enforce hard limits — that is independent).
- **FR-12.** The system MUST provide an in-attempt **panic button** for students to flag legitimate interruptions (bathroom, emergency, assistive tech glitch); panic events appear in the integrity report and exempt the surrounding window from severity scoring.
- **FR-13.** The system MUST retain raw media artifacts (thumbnails, hashes, short clips) for a configurable retention window (default 30 days, max 180 days) and automatically purge on expiry.
- **FR-14.** The system MUST emit a tamper-evident audit record for: consent acceptance/decline, every artifact access by staff, retention purges, and any manual override of severity.
- **FR-15.** The system MUST expose an instructor "false positive" feedback loop on every event in the report; feedback MUST feed eval datasets (19.13) for model tuning.
- **FR-16.** Students MUST be able to download their own integrity report and raw collected events (data-subject access; §10.3 GDPR).

## 6. Non-Functional Requirements

- **Performance** — Client-side inference MUST sustain ≥10 fps face detection on a 2018-era laptop without dropping quiz interaction latency below 60 fps; total CPU budget ≤ 25%. Event upload latency p95 < 2 s. LLM rationale generation may run async, but MUST complete within 5 minutes of attempt submission.
- **Security** — All telemetry uploads MUST use authenticated, attempt-scoped tokens. Raw media artifacts MUST be encrypted at rest (§10.13), accessed via signed short-lived URLs (≤ 1 h), and MUST NOT be reachable by other students in the same course.
- **Privacy & Compliance** — FERPA: integrity report and artifacts are education records. **BIPA (IL), TDPSA (TX), GDPR Art. 9 (EU/UK)**: webcam, voice, and any face/voice templates are *biometric data* — explicit affirmative consent required, retention disclosed, written policy required. COPPA: hard gate as in FR-3. Per-org legal review MUST gate first enablement.
- **Accessibility (WCAG 2.1 AA + non-discrimination)** — Consent screen and panic button MUST be keyboard-operable and screen-reader announced. Face-detection models MUST be fairness-evaluated across skin tones (Fitzpatrick I–VI) and head coverings; bias metric thresholds MUST be defined in the model card and gate releases. Students using AT (screen readers, switch input, eye-tracking) MUST be excludable from gaze/input-pattern flagging at the accommodation level (§2.11).
- **Scalability** — 5,000 concurrent proctored attempts per cluster; event throughput 50 events/sec/attempt sustained.
- **Reliability** — If client-side models fail to load, the attempt MUST proceed with focus/input telemetry only and the instructor banner MUST disclose "Media inference unavailable for this attempt." Server-side rationale generation failure MUST NOT block report delivery.
- **Observability** — Metrics: `proctor_attempts_total{profile}`, `proctor_events_total{type,severity}`, `proctor_consent_decline_rate`, `proctor_panic_invocations_total`, `proctor_llm_rationale_cost_usd_total`, `proctor_artifact_retention_purged_total`, `proctor_false_positive_feedback_total`. Alerts on consent decline > 30%, false-positive rate > 15%.
- **Maintainability** — Service in `server/internal/service/proctoring/service.go`; client harness in `clients/web/src/proctoring/`; in-browser models loaded from a versioned, integrity-pinned CDN bundle.
- **Internationalization** — Consent copy externalised and reviewed by legal per locale (different jurisdictions have different mandatory disclosures); VAD and face-detection models tested across regions.
- **Backward compatibility** — Assignments with no `proctoring_profile` row behave identically to today.

## 7. Acceptance Criteria

- **AC-1.** *Given* an instructor enables `focus_tracking + webcam` on a quiz, *When* a student begins the attempt, *Then* the consent screen lists "focus tracking" and "webcam (face presence, gaze direction)" with retention period, and the attempt cannot begin until consent is recorded.
- **AC-2.** *Given* a student declines consent, *When* they submit the decline, *Then* the attempt is not started, the instructor sees "Declined AI proctoring — fallback required," and no media APIs were ever invoked.
- **AC-3.** *Given* webcam proctoring is active, *When* the student covers the camera for 30 seconds, *Then* a `face_absent` event of severity `medium` appears in the integrity report with the exact start/end timestamps.
- **AC-4.** *Given* a student presses the panic button, *When* the surrounding 2-minute window is processed, *Then* events in that window are tagged `panic_exempt` and excluded from severity totals shown to the instructor.
- **AC-5.** *Given* an under-13 user without parental consent on file, *When* an instructor assigns a webcam-proctored quiz, *Then* the student sees a non-proctored variant or a "consent required" notice — webcam APIs are never requested.
- **AC-6.** *Given* a `high`-severity event cluster, *When* the server-side rationale generator runs, *Then* student face thumbnails passed to the LLM are blurred such that they do not pass a face-recognition similarity threshold to the unblurred original (verified by automated test).
- **AC-7.** *Given* retention is set to 30 days, *When* a raw artifact crosses day 31, *Then* the object is deleted from storage, the audit log records the purge, and the integrity report retains only the textual event log.
- **AC-8.** *Given* a student files a data-subject access request, *When* the export is generated, *Then* it includes every event, every artifact still within retention, and the full audit trail of staff access.
- **AC-9.** *Given* a student uses a screen reader (detected via accommodation flag from §2.11), *When* the integrity report is generated, *Then* keyboard-pattern and focus-loss events from AT navigation are auto-suppressed or downgraded per the accommodation rule set.
- **AC-10.** *Given* the in-browser face model fails to load, *When* the attempt proceeds, *Then* the instructor's integrity report displays "Webcam inference unavailable" prominently and no fabricated face events appear.

## 8. Data Model

Migration `0XX_ai_proctoring.sql`:

```sql
CREATE TABLE proctoring_profiles (
  id                       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  assignment_id            UUID NOT NULL UNIQUE REFERENCES assignments(id) ON DELETE CASCADE,
  focus_tracking           BOOLEAN NOT NULL DEFAULT FALSE,
  input_pattern_analysis   BOOLEAN NOT NULL DEFAULT FALSE,
  webcam                   BOOLEAN NOT NULL DEFAULT FALSE,
  microphone               BOOLEAN NOT NULL DEFAULT FALSE,
  screen_share             BOOLEAN NOT NULL DEFAULT FALSE,
  lockdown_required        BOOLEAN NOT NULL DEFAULT FALSE,
  retention_days           INT    NOT NULL DEFAULT 30 CHECK (retention_days BETWEEN 1 AND 180),
  severity_rules_version   TEXT   NOT NULL,
  created_by               UUID REFERENCES users(id),
  created_at               TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at               TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE proctoring_attempts (
  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  attempt_id          UUID NOT NULL UNIQUE REFERENCES quiz_attempts(id) ON DELETE CASCADE,
  profile_id          UUID NOT NULL REFERENCES proctoring_profiles(id),
  consent_status      TEXT NOT NULL,   -- 'accepted' | 'declined' | 'parent_required'
  consent_recorded_at TIMESTAMPTZ,
  consent_evidence    JSONB,           -- consent text hash, locale, ip-class, ua-class
  client_capabilities JSONB,           -- which models loaded; fallback flags
  status              TEXT NOT NULL DEFAULT 'in_progress',
  artifacts_purge_at  TIMESTAMPTZ,
  llm_rationale       TEXT,
  llm_rationale_at    TIMESTAMPTZ,
  created_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE proctoring_events (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  proctor_attempt UUID NOT NULL REFERENCES proctoring_attempts(id) ON DELETE CASCADE,
  ts              TIMESTAMPTZ NOT NULL,
  duration_ms     INT,
  event_type      TEXT NOT NULL,     -- 'face_absent', 'multiple_faces', 'tab_blur', ...
  severity        TEXT NOT NULL,     -- 'info' | 'low' | 'medium' | 'high'
  panic_exempt    BOOLEAN NOT NULL DEFAULT FALSE,
  payload         JSONB,             -- model confidence, bounding boxes (no raw frames)
  artifact_key    TEXT,              -- nullable object-storage key for thumbnail/snippet
  instructor_feedback TEXT           -- 'true_positive' | 'false_positive' | NULL
);
CREATE INDEX idx_proctoring_events_attempt_ts ON proctoring_events(proctor_attempt, ts);

CREATE TABLE proctoring_artifact_audit (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  artifact_key      TEXT NOT NULL,
  action            TEXT NOT NULL,    -- 'created' | 'viewed' | 'purged' | 'exported'
  actor_user_id     UUID REFERENCES users(id),
  ts                TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Backfill: none — net-new tables.

## 9. API Surface

```
POST /api/v1/assignments/:aid/proctoring-profile         (instructor)  -> ProfileDto
GET  /api/v1/assignments/:aid/proctoring-profile         (instructor)  -> ProfileDto

POST /api/v1/attempts/:attemptId/proctoring/consent      (student)
        body: { decision: 'accept' | 'decline', consent_text_hash }
        -> 200 { proctor_attempt_id } | 409 if already recorded

POST /api/v1/attempts/:attemptId/proctoring/events       (student)
        body: { events: [ { ts, type, payload, artifact_blob? } ] }   -- batched, ≤ 5 s cadence
        -> 202

POST /api/v1/attempts/:attemptId/proctoring/panic        (student)    -> 202

GET  /api/v1/attempts/:attemptId/proctoring/report       (instructor) -> IntegrityReportDto
POST /api/v1/proctoring/events/:eventId/feedback         (instructor)
        body: { feedback: 'true_positive' | 'false_positive' }
GET  /api/v1/proctoring/artifacts/:key/signed-url        (instructor) -> { url, expires_at }

GET  /api/v1/users/me/proctoring/export                  (student/DSAR) -> ZIP
```

WebSocket: `/ws/attempts/:attemptId/proctoring` — bidirectional event channel for low-latency upload and server-pushed consent revocations.

Rate limits: 20 event-batch posts/min/attempt; 1 LLM rationale generation/attempt.

OpenAPI: all routes documented; biometric-data warning surfaced in route description.

## 11. AI / ML Considerations

- **In-browser models (preferred path, default)**
  - Face presence + landmarks: **MediaPipe Face Landmarker** (WASM, ~10 MB, runs at ~15 fps on mid-range hardware).
  - Voice activity detection: **Silero VAD** (ONNX runtime web, ~2 MB).
  - Gaze zone classification: lightweight head-pose heuristic on top of MediaPipe landmarks; no separate model.
  - Loaded from a versioned, SRI-pinned CDN bundle; hash recorded per attempt.
- **Server-side rationale (escalation only)**
  - Model: **Claude Sonnet 4.6** via Anthropic SDK (`claude-sonnet-4-6`), with prompt caching of the severity rule set and report template. Fall back to `claude-haiku-4-5-20251001` under cost pressure.
  - Inputs: textual event log + ≤ 8 blurred thumbnails. **All face images blurred client-side before upload** to LLM-bound storage tier; PII proxy applied to any captured screen text (19.11).
  - Output: ≤ 200-word rationale, structured JSON with `summary`, `confidence_phrase` (no numeric probabilities to instructors — they over-anchor), `recommended_next_step`.
- **Bias & fairness**
  - Pre-launch fairness eval mandatory: face-detection true-positive rate gap across Fitzpatrick I–VI ≤ 5pp; ≤ 3pp gap with/without head coverings.
  - Documented in model card `docs/ai/proctoring-model-card.md`.
- **Eval harness** (19.13)
  - Labeled dataset of consented synthetic + opt-in real attempts; metrics: per-event-type precision/recall, false-positive rate at p99, LLM rationale faithfulness (cited-event check).
- **Cost** (19.14)
  - Client-side inference: $0/attempt.
  - LLM rationale: ~$0.01–0.03/attempt with caching; budget cap per org.
- **Opt-out / fallback**
  - User-level `ai_proctoring_opt_out` flag (e.g., for documented accommodations) routes student to instructor-defined alternate path.
  - Org-level toggle to disable webcam/microphone/screen-share entirely while still using focus & input signals.
- **Model versioning**
  - Pinned in `server/internal/service/proctoring/models.go`; rollouts behind 19.10 governance.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Biometric-law liability (BIPA, TDPSA, GDPR Art. 9) | H | H | Explicit affirmative consent + retention policy + per-org legal review gate; biometric features disable-able org-wide |
| Bias against students of color, head coverings, or atypical appearance | H | H | Mandatory fairness eval gate; opt-out; instructor banner "decision support, not evidence"; false-positive feedback loop |
| False-positive cascade leading to wrongful integrity charges | M | H | Severity is decision support only; no automated action; mandatory instructor review; per-event feedback feeding eval |
| Students with assistive tech flagged for legitimate behavior | H | M | Accommodation suppression rules (§2.11); panic button; documented appeal path |
| Webcam coverage / lighting / cheap camera → unreliable signals | H | M | Pre-attempt environment check; graceful degradation banner; never block attempt on model failure |
| Consent fatigue / dark-pattern accusations | M | H | Equal-weight Accept/Decline UI; no nudging; clear fallback path documented |
| Client-side model integrity (CDN compromise, model tampering) | L | H | SRI hash pinning; CSP; per-attempt recorded model hash in audit log |
| LLM hallucinates a cheating accusation | M | H | Rationale constrained to cite event IDs; faithfulness check in eval harness; instructor sees raw event log alongside |
| Retention drift / accidental long-term storage of biometric data | M | H | Tamper-evident purge audit; automated retention enforcement job; quarterly compliance review |
| Discovery / e-discovery requests on stored video | M | M | Default retention 30 days; document data-minimization stance; legal hold workflow defined |

## 15. Rollout Plan

- Feature flag: `ai_proctoring_v1` (OpenFeature, default `off`).
- Org-level sub-flags: `ai_proctoring_focus_input`, `ai_proctoring_webcam`, `ai_proctoring_microphone`, `ai_proctoring_screen`.
- Migration: schema → severity-rule version pin → client bundle → server pipeline → flag flip.
- **Phase 1 (alpha, HE only)**: focus + input pattern signals only, 1–2 pilot HE institutions with completed legal review. No biometric capture.
- **Phase 2 (beta, HE + SL)**: webcam + microphone enabled under explicit org opt-in with fairness eval passing; LLM rationale gated behind a second flag.
- **Phase 3 (GA)**: screen-share signals; broader instructor self-service.
- **K-12**: gated until 19.13 eval data shows < 5% false-positive rate and parental-consent flow is durable; default off even after flag enabled.
- GA criteria: per-event false-positive rate < 10% on eval set, fairness deltas within thresholds, model card + DPIA published, instructor docs translated to top 5 locales.
- Rollback: flag flip immediately stops new attempts entering AI proctoring; in-flight attempts complete in focus-only mode.

## 16. Test Plan

- **Unit** — severity rule engine; consent state machine; accommodation suppression rules; retention purge scheduler; SRI hash verification.
- **Integration** — full attempt lifecycle (consent → events → submission → report); WebSocket reconnect mid-attempt; LLM rationale path with mocked Anthropic SDK; artifact signed-URL expiry.
- **End-to-end (Playwright)** — instructor enables profile → student goes through consent → simulated focus loss / face absence (via stubbed media stream) → instructor sees timeline + feedback buttons → false-positive feedback round-trip.
- **Security** — cross-student artifact access blocked; signed URLs short-lived and single-use; consent record tamper-evident; CSP blocks unsigned model loads.
- **Fairness** — automated face-detection accuracy run across labeled Fitzpatrick I–VI dataset; head-covering subset; head-pose extremes.
- **Accessibility** — axe on consent and report pages; screen-reader walkthrough of consent flow; keyboard-only attempt completion with panic button activation; AT-induced events verified suppressed under accommodation flag.
- **Performance / load** — 5,000 concurrent attempts emitting 2 events/sec; p95 ingest < 2 s; client CPU profile recorded on reference laptop.
- **Privacy / DPIA** — data flow diagram reviewed; retention purge verified end-to-end on a 30-day clock; DSAR export contains the expected fields.
- **Manual exploratory** — webcam covered, two faces in frame, second voice in room, alt-tabbing, paste from clipboard, fullscreen exit, panic button, declined consent, model load failure.

## 18. Open Questions

1. Should we add identity verification (ID-document OCR + face match) at exam start? It is the natural pairing but materially expands biometric scope — likely a separate plan.
2. Should self-learners be able to publish their own integrity reports (signed/verifiable) to a public profile, or is that an integrity risk in itself?
3. What is the right default retention — 30 days minimizes risk but limits instructor review windows for late-term grade disputes; 90 days is typical for HE.
4. Do we offer "AI proctoring lite" (focus + input only, no media) as a free tier, with biometric features behind a paid add-on?
5. Should instructors be allowed to enable AI proctoring *without* informing students prior to the attempt? (Recommendation: no — always require consent screen.)
6. How do we handle late-detected false positives that arrive after a grade is finalised — is there a grade-revision workflow trigger?
7. For non-Anthropic deployments (data-residency orgs), do we ship an on-prem rationale generator path, or disable rationales entirely?
