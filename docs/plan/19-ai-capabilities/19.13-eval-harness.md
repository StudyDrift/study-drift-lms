## 1. Problem Statement

Lextures has multiple AI features in production or near-production, but no systematic way to evaluate their quality before deploying prompt changes or model upgrades. A rubric-generation prompt update could silently degrade alignment quality; a Whisper model version change could increase WER; a tutor prompt change could reduce citation grounding. Without a regression suite and eval harness, every AI change is a roll of the dice. This plan cross-references §17.18 (test coverage target) and establishes the AI-specific eval layer on top of the general test infrastructure.

## 2. Goals

- Build a CI-integrated eval harness that runs golden-set tests against all AI features on every prompt/model change.
- Implement RAGAS (RAG quality), ARES (RAG grounding), and custom rubric-alignment metrics.
- Store eval results in the DB for trend analysis and drift detection.
- Block prompt/model promotion if eval results fall below defined thresholds.
- Provide an eval dashboard in the admin console for AI quality visibility.

## 3. Non-Goals

- A/B testing infrastructure for non-AI features.
- Human evaluation campaigns at scale (the harness automates evaluation; human spot-checks are supplementary).
- Evaluating third-party model providers' internal quality (we evaluate Lextures' usage of them).
- Production traffic interception ("shadow eval") — golden-set offline evaluation is the scope.

## 4. Personas & User Stories

- **As an AI engineer**, I want to run the eval suite before merging a prompt change so that I know whether quality improved or regressed.
- **As a Lextures super-admin**, I want to see which AI features have the highest hallucination rates so that I can prioritize improvements.
- **As a district IT admin**, I want confirmation that AI quality metrics are continuously monitored so that I can include this in our vendor risk assessment.
- **As an automated CI pipeline**, I want to block a PR that degrades quiz-generation relevance below the threshold so that regressions don't reach production.

## 5. Functional Requirements

- **FR-1.** The system MUST maintain golden-set datasets (input → expected output) for each AI feature: quiz generation, rubric generation, tutor Q&A, grading feedback, summarization, lesson plan generation.
- **FR-2.** The eval harness MUST run against each golden set and compute: RAGAS faithfulness, answer relevance, context recall (for RAG features); ROUGE-L, BERTScore (for summarization/translation); custom rubric alignment score (for grading).
- **FR-3.** Eval results MUST be stored in the DB with: `feature_key`, `prompt_version`, `model_id`, `metric_name`, `score`, `run_at`.
- **FR-4.** The CI pipeline (GitHub Actions) MUST run the eval harness on every PR that modifies a prompt or AI service; the pipeline MUST fail if any metric falls below its defined threshold.
- **FR-5.** Thresholds MUST be configurable per feature/metric in the `ai_eval_thresholds` table (not hard-coded).
- **FR-6.** The admin console MUST display a time-series chart of eval metrics per feature, allowing detection of model drift over time.
- **FR-7.** The harness MUST support bias-audit golden sets: diverse inputs representing different demographics, writing styles, and languages; metrics include demographic parity of scores and WER across languages.
- **FR-8.** The harness SHOULD run daily in CI against production model/prompt to detect drift even without code changes.

## 6. Non-Functional Requirements

- **Performance** — Full eval suite (all features, all golden sets) MUST complete in ≤ 30 min in CI; individual feature evals runnable independently in ≤ 5 min.
- **Security** — Golden-set data must not contain real student PII; use synthetic or anonymized test data only.
- **Privacy & Compliance** — Eval dataset governance documented; no FERPA-sensitive data in golden sets.
- **Accessibility** — Eval dashboard meets WCAG 2.1 AA.
- **Scalability** — Eval runs in parallel across features using GitHub Actions matrix; results DB table partitioned by run_date.
- **Reliability** — CI failure due to LLM provider downtime logged as `eval_blocked_provider_unavailable` not a quality failure.
- **Observability** — Eval results stored in DB and emitted as GitHub Actions annotations; alert on metric below threshold.
- **Maintainability** — Eval harness code in `evals/` directory at repo root; Python-based using RAGAS, BERTScore libraries; golden sets in `evals/golden-sets/`.
- **Internationalization** — Multilingual golden sets for translation/summarization features.
- **Backward compatibility** — Eval runs read-only; no production data modified.

## 7. Acceptance Criteria

- **AC-1.** *Given* a PR modifies the quiz-generation prompt, *When* the CI pipeline runs the eval harness, *Then* the quiz quality metrics (relevance, alignment) are computed and displayed as PR check annotations.
- **AC-2.** *Given* the tutor grounding metric falls below 0.85 in an eval run, *When* the CI check completes, *Then* the check status is "failed" and the PR cannot be merged until the threshold is met.
- **AC-3.** *Given* eval results are stored, *When* the super_admin views the AI quality dashboard, *Then* a time-series chart shows RAGAS faithfulness for the tutor feature over the past 30 days.
- **AC-4.** *Given* a bias-audit golden set with inputs of varying demographic backgrounds, *When* the grading eval runs, *Then* the demographic parity metric is computed and stored alongside the main accuracy metrics.
- **AC-5.** *Given* the LLM provider is unavailable during a scheduled daily eval, *When* the run fails, *Then* it is logged as `eval_blocked_provider_unavailable` and no threshold failures are recorded.

## 8. Data Model

```sql
-- 0XX_eval_harness.sql
CREATE TABLE ai_eval_runs (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  feature_key   TEXT NOT NULL,
  prompt_version INT,
  model_id      TEXT NOT NULL,
  golden_set_id TEXT NOT NULL,
  triggered_by  TEXT NOT NULL,   -- 'ci' | 'manual' | 'scheduled'
  status        TEXT NOT NULL DEFAULT 'pending',
  run_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  completed_at  TIMESTAMPTZ
);

CREATE TABLE ai_eval_results (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  run_id     UUID NOT NULL REFERENCES ai_eval_runs(id),
  metric     TEXT NOT NULL,      -- 'ragas_faithfulness', 'rouge_l', 'bert_score', etc.
  score      NUMERIC NOT NULL,
  details    JSONB,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX eval_results_run ON ai_eval_results(run_id);

CREATE TABLE ai_eval_thresholds (
  feature_key  TEXT NOT NULL,
  metric       TEXT NOT NULL,
  min_score    NUMERIC NOT NULL,
  updated_by   UUID REFERENCES users(id),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (feature_key, metric)
);
```

## 9. API Surface

```
GET  /api/admin/ai-evals              -> AiEvalRun[]  (super_admin)
POST /api/admin/ai-evals              { feature_key } -> 202 { run_id }
GET  /api/admin/ai-evals/:run_id      -> AiEvalRun & { results: AiEvalResult[] }
GET  /api/admin/ai-evals/thresholds   -> AiEvalThreshold[]
PUT  /api/admin/ai-evals/thresholds   -> AiEvalThreshold[]
GET  /api/admin/ai-evals/dashboard    -> EvalTrendData (for charts)
```

CI integration: `eval/run.py` script invokes `/api/admin/ai-evals` and polls for completion; exits non-zero if any metric below threshold.

## 11. AI / ML Considerations

- **RAGAS** — RAG evaluation: faithfulness, answer relevance, context precision, context recall.
- **ARES** — Alternative RAG evaluation with statistical testing.
- **BERTScore** — Semantic similarity for summarization and translation.
- **ROUGE-L** — Lexical overlap for summarization.
- **Demographic parity** — Custom metric for bias audit: score variance across input demographic proxies.
- **Custom rubric alignment** — Overlap between AI rubric criteria and human-authored criteria using cosine similarity of criterion embeddings.
- Cross-ref §17.18 (test coverage) — eval harness complements but does not replace unit/integration tests.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Golden sets become stale as curriculum changes | M | M | Review golden sets quarterly; version them alongside prompts |
| LLM provider costs from daily eval runs | M | M | Use smaller model for evals where possible; cache eval results |
| CI time increases beyond 30 min | M | M | Parallel matrix; limit golden set size to 20–50 items per feature |
| Engineers bypass eval CI check | L | H | Branch protection rules; require eval check for main branch PRs |

## 15. Rollout Plan

- Feature flag: `ai_eval_dashboard_enabled` (OpenFeature, default `false`).
- Phase 1: Build harness infrastructure and golden sets for quiz generation (highest priority).
- Phase 2: Add tutor/RAG grounding evals; integrate CI check.
- Phase 3: Add grading, summarization, lesson-plan evals.
- Phase 4: Bias audit golden sets; scheduled daily runs.
- Phase 5: GA dashboard for super_admin.

## 16. Test Plan

- **Unit** — Eval result parsing; threshold comparison logic; CI exit code on failure.
- **Integration** — Run eval on a known-good prompt → scores above threshold; deliberately degrade prompt → scores below threshold → CI fails.
- **Manual** — Validate RAGAS scores on tutor golden set correlate with human quality ratings.
- **Performance** — Full suite completes in ≤ 30 min on CI hardware.

## 17. Documentation & Training
