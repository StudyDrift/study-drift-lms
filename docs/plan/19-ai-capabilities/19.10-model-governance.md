## 1. Problem Statement

All Lextures AI services currently route to OpenRouter without any per-tenant configuration, approval workflow, or prompt versioning. A K-12 district admin cannot specify which models are approved to run on their students' data. Prompt changes are hard-coded in Rust source files, so any prompt update requires a deployment. This is a procurement blocker for privacy-conscious districts and a developer-experience problem. NIST AI RMF 1.0 and ISO/IEC 23894 both require Govern-level controls on AI model selection and prompt content.

## 2. Goals

- Store all AI system prompts in the database with version history and org-level override capability.
- Allow org admins to specify an allowlist of approved AI models for their organization.
- Support BYOK (Bring Your Own Key) so enterprise orgs can provide their own Anthropic, OpenAI, or Azure OpenAI API keys.
- Provide a model catalog UI in the admin console showing available models with their data-handling properties.
- Log all model and prompt selections to the audit trail for compliance.

## 3. Non-Goals

- Building a new AI model router (OpenRouter remains the default aggregator).
- Fine-tuning or training custom models.
- Per-user model selection (this is org-level governance, not personalization).

## 4. Personas & User Stories

- **As a district IT admin**, I want to restrict our instance to only using models that are FERPA-compliant and have signed DPAs with our district so that student data does not flow to unapproved vendors.
- **As a Lextures super-admin**, I want to update the quiz-generation system prompt and have the change versioned, reviewable, and deployable without a code change so that I can iterate quickly.
- **As an enterprise customer**, I want to provide my own OpenAI API key so that our organization is billed directly and data goes through our existing agreement.
- **As a compliance auditor**, I want to see which model was used for each AI interaction so that I can verify policy adherence.

## 5. Functional Requirements

- **FR-1.** The system MUST store all AI system prompts in a `ai_prompts` table with: `id`, `feature_key` (e.g., `quiz_generation`, `tutor_session`), `version`, `content`, `model_id`, `active`, `created_by`, `created_at`.
- **FR-2.** The system MUST resolve the active prompt for a feature at runtime from the DB, not from source code.
- **FR-3.** `super_admin` MUST be able to create new prompt versions, set a version as active, and view full version history per feature_key.
- **FR-4.** The system MUST maintain a `ai_model_catalog` table listing all supported provider/model combinations with: `model_id`, `provider`, `data_region`, `ferpa_dpa_signed` boolean, `coppa_approved` boolean, `max_context_tokens`, `cost_per_1k_tokens`.
- **FR-5.** Each org MUST have an `approved_model_ids` array (default: all FERPA-DPA-signed models); AI calls for that org MUST only use approved models.
- **FR-6.** The system MUST support BYOK: org admins MAY store an encrypted API key per provider; if present, that key MUST be used instead of the platform key.
- **FR-7.** Every AI call MUST log the `feature_key`, `prompt_version`, `model_id`, `org_id`, and `actor_id` to an `ai_call_log` table.
- **FR-8.** The system MUST present a model governance dashboard in the admin console showing: current approved models, BYOK status, recent AI call volume by model.

## 6. Non-Functional Requirements

- **Performance** — Prompt DB lookup MUST be cached (in-memory with 60 s TTL) so runtime lookup adds ≤ 5 ms.
- **Security** — BYOK API keys MUST be encrypted at rest (AES-256-GCM) using a per-org DEK stored in the secrets manager (§17.17); keys MUST never appear in logs.
- **Privacy & Compliance** — Model selection must comply with FERPA (data only to DPA-signed providers) and GDPR data residency; NIST AI RMF 1.0 Govern functions implemented; ISO/IEC 23894 risk documentation for each approved model.
- **Accessibility** — Model governance admin UI meets WCAG 2.1 AA.
- **Scalability** — Prompt cache prevents DB hot reads; `ai_call_log` is append-only, partitioned by month.
- **Reliability** — If prompt DB lookup fails, fall back to a compiled-in default; log the fallback event.
- **Observability** — `ai_calls_by_model{model_id, org_id}` counter; `prompt_cache_misses_total`; alert on use of unapproved model.
- **Maintainability** — Prompt management service: `server/src/services/prompt_registry.rs`; governance service: `server/src/services/model_governance.rs`.
- **Internationalization** — Model catalog data in English; UI labels in i18n keys.
- **Backward compatibility** — Default prompts migrated from source code to DB in migration; runtime behavior unchanged.

## 7. Acceptance Criteria

- **AC-1.** *Given* an org has approved models `["claude-3-5-sonnet", "gpt-4o"]`, *When* the quiz generation service runs for that org, *Then* only those models are used and the selection is logged in `ai_call_log`.
- **AC-2.** *Given* a super_admin creates a new version of the `quiz_generation` prompt and sets it active, *When* the next quiz is generated, *Then* the new prompt version is used and the `ai_call_log.prompt_version` field matches.
- **AC-3.** *Given* an org_admin stores a BYOK OpenAI key, *When* an OpenAI model is called for that org, *Then* the org's key is used (verified by checking account-level spend attribution), not the platform key.
- **AC-4.** *Given* an AI call is made, *When* the call log entry is inspected, *Then* it contains `feature_key`, `prompt_version`, `model_id`, `org_id`, `actor_id`, and `token_counts`.
- **AC-5.** *Given* the prompt DB is unavailable, *When* an AI service attempts a call, *Then* it falls back to the compiled-in default prompt and logs a `prompt_fallback` event.

## 8. Data Model

```sql
-- 0XX_ai_governance.sql
CREATE TABLE ai_prompts (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  feature_key   TEXT NOT NULL,
  version       INT NOT NULL,
  content       TEXT NOT NULL,
  model_id      TEXT,               -- default model for this feature
  is_active     BOOLEAN NOT NULL DEFAULT false,
  created_by    UUID REFERENCES users(id),
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX ai_prompts_active ON ai_prompts(feature_key) WHERE is_active = true;
CREATE INDEX ai_prompts_feature ON ai_prompts(feature_key, version DESC);

CREATE TABLE ai_model_catalog (
  model_id              TEXT PRIMARY KEY,
  provider              TEXT NOT NULL,
  display_name          TEXT NOT NULL,
  data_region           TEXT,
  ferpa_dpa_signed      BOOLEAN NOT NULL DEFAULT false,
  coppa_approved        BOOLEAN NOT NULL DEFAULT false,
  max_context_tokens    INT,
  cost_per_1k_input     NUMERIC,
  cost_per_1k_output    NUMERIC
);

CREATE TABLE org_ai_settings (
  org_id              UUID PRIMARY KEY REFERENCES organizations(id),
  approved_model_ids  TEXT[] NOT NULL DEFAULT '{}',   -- empty = all DPA-signed
  byok_keys_encrypted JSONB                           -- { provider: encrypted_key }
);

CREATE TABLE ai_call_log (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          UUID NOT NULL REFERENCES organizations(id),
  actor_id        UUID,
  feature_key     TEXT NOT NULL,
  prompt_version  INT,
  model_id        TEXT NOT NULL,
  input_tokens    INT,
  output_tokens   INT,
  latency_ms      INT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
) PARTITION BY RANGE (created_at);
```

## 10. UI / UX

New page: `clients/web/src/pages/admin/AdminAIGovernance.tsx`
- Tabs: Model Catalog, Prompt Registry, Org Settings, Call Log.
- Model Catalog: table showing all models; toggle to approve/unapprove for org; FERPA/COPPA badges.
- Prompt Registry (super_admin only): list of feature keys; click to see version history; diff view between versions; "Set Active" button.
- BYOK: masked key input per provider; test connection button.
- Call Log: paginated table filterable by date, feature, model.

## 11. AI / ML Considerations

Governance of AI, not AI itself. Key standards:
- **NIST AI RMF 1.0** — Govern, Map, Measure, Manage lifecycle.
- **ISO/IEC 23894:2023** — AI risk management.
- **OECD AI Principles** — Transparency, accountability, human oversight.
- Prompt versioning aligns with MLOps best practices (prompt as code, version-controlled).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Prompt change breaks downstream AI feature | M | H | Prompt version history; canary deployment (test on 5% of traffic before activate) |
| BYOK key leaked via logs | M | H | Encrypted at rest; never log; scrub from tracing spans |
| Admin approves unapproved model | M | H | FERPA/COPPA filter on model catalog; warn before approving non-DPA models |

## 15. Rollout Plan

- Feature flag: `ai_governance_enabled` (OpenFeature, default `false`).
- Phase 1: Migrate all prompts from source code to DB; no UI yet; behavior unchanged.
- Phase 2: Model catalog and org approval UI.
- Phase 3: BYOK support.
- Phase 4: Prompt registry editor for super_admin.
- GA: Required before any public AI feature launch.

## 16. Test Plan

- **Unit** — Prompt resolution (active version); org model approval enforcement; BYOK key encryption/decryption.
- **Integration** — End-to-end: update prompt → next AI call uses new version → call log entry reflects version.
- **Security** — BYOK key never in logs; cross-org model setting isolation; unapproved model call rejected.
- **Performance** — Prompt cache hit rate ≥ 99%; lookup latency < 5 ms with cache.

## 17. Documentation & Training

- Admin guide: Model Governance — approving models, managing BYOK keys.
- Developer docs: Adding a new AI feature — using prompt_registry.rs and model_governance.rs.
- Internal runbook: Prompt version promotion; rollback procedure.
- NIST AI RMF documentation template for each approved model.

## 18. Open Questions

1. Should prompt changes require a second-approver before activation (4-eyes principle for critical prompts)?
2. Should we support per-feature model overrides at the org level (e.g., org uses GPT-4o for grading but Claude for tutor)?
3. How often should BYOK keys be rotated, and should we enforce rotation?
