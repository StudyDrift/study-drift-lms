## 1. Problem Statement

Every current Lextures AI service passes raw text directly to OpenRouter/LLMs. This raw text frequently contains student PII: names in short-answer submissions, email addresses in forum posts, student ID numbers in uploaded documents. Sending unredacted PII to third-party LLM providers violates FERPA (for K-12/HE), GDPR (for EU users), COPPA (for under-13), and the AI disclosure obligations referenced in §10.17. This is classified as a BLOCKER for any institutional sale. Microsoft Presidio is the leading open-source PII detection and redaction framework and is the recommended implementation path.

Cross-reference: §10.17 (AI usage disclosure) requires that prompts to external models do not contain identifiable student data.

## 2. Goals

- Intercept all outbound LLM prompts and redact detected PII before transmission to any external provider.
- Use Microsoft Presidio for entity recognition (names, emails, phone numbers, SSNs, student IDs, dates of birth, addresses).
- Replace detected PII with consistent placeholder tokens so the LLM response remains coherent.
- De-anonymize (restore tokens) in responses where the output is returned to the student (e.g., tutor response referencing "the student" → replace with actual name for display).
- Log redaction events for compliance auditing.

## 3. Non-Goals

- Redacting PII in stored data at rest (separate §10.14 plan — logs; §10.3 plan — DSAR).
- Content moderation or filtering for harmful content (a separate future plan).
- Fully blocking all prompt transmission to LLMs — redaction enables compliant transmission.
- Real-time redaction of live keystroke input (apply on submit/send, not on each keypress).

## 4. Personas & User Stories

- **As a district privacy officer**, I want to know that no student name or email is ever sent to a third-party AI API so that we remain FERPA-compliant.
- **As a student**, I want to ask the AI tutor about my calculus homework without worrying that my name or student ID is being shared with an AI company.
- **As a Lextures engineer**, I want a single redaction proxy layer so that every AI service benefits without duplicating PII-handling logic.
- **As a compliance auditor**, I want to see logs showing how many PII entities were redacted across all LLM calls so that I can include this in our SOC 2 evidence.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a server-side redaction function that accepts raw prompt text and returns redacted text plus a de-anonymization map `{ token: original_value }`.
- **FR-2.** The redaction function MUST detect and redact at minimum: PERSON (names), EMAIL_ADDRESS, PHONE_NUMBER, US_SSN, US_ITIN, DATE_OF_TIME (DOB), LOCATION, STUDENT_ID (custom recognizer), MEDICAL_RECORD_NUMBER.
- **FR-3.** The system MUST use Microsoft Presidio (Python microservice or WASM-compiled library) as the NLP backend for entity recognition.
- **FR-4.** Each detected entity MUST be replaced with a consistent token of the form `[ENTITY_TYPE_<index>]` (e.g., `[PERSON_1]`, `[EMAIL_1]`); the same value in the same prompt MUST map to the same token.
- **FR-5.** The de-anonymization map MUST be held in memory only for the duration of the request and MUST NOT be persisted to any store.
- **FR-6.** All Lextures AI service functions MUST call the redaction proxy before constructing the LLM prompt.
- **FR-7.** The system MUST log per-request redaction statistics: `{ org_id, feature_key, entity_counts: { PERSON: 2, EMAIL: 1 } }` — not the actual values — to the compliance audit log.
- **FR-8.** The system MUST provide an opt-out path at the org level: if `pii_redaction_bypass: true` is set for a org (only available to super_admin for self-hosted orgs with their own model infrastructure), redaction is skipped but logged.
- **FR-9.** Where the LLM response contains placeholder tokens (because the model repeated input tokens), the system MUST attempt de-anonymization before returning the response to the client.

## 6. Non-Functional Requirements

- **Performance** — Presidio analysis MUST complete in ≤ 100 ms p95 for inputs ≤ 2 000 words; total PII proxy overhead including redaction + de-anonymization MUST add ≤ 150 ms to LLM request latency.
- **Security** — De-anonymization map held in-process RAM only; no disk writes; map zeroed on request completion; Presidio microservice accessible only from the Rust server (not publicly exposed).
- **Privacy & Compliance** — FERPA, GDPR Art. 25 (data minimization by design), COPPA; ISO/IEC 23894 data governance; redaction audit log is compliance evidence; entity detection covers GDPR special-category data (health, ethnicity patterns).
- **Accessibility** — Not a UI feature; n/a.
- **Scalability** — Presidio microservice scales independently; Rust server calls it via local gRPC or HTTP; stateless and horizontally scalable.
- **Reliability** — If Presidio is unavailable, AI calls MUST be blocked (fail-closed) with a clear error to the user; this prevents inadvertent PII transmission.
- **Observability** — `redaction_entities_total{entity_type}`, `redaction_latency_ms`, `redaction_blocked_calls_total` (fail-closed events); alert on Presidio unavailability.
- **Maintainability** — Redaction service: `server/src/services/pii_redaction.rs`; Presidio runs as a Docker sidecar; custom recognizer for student IDs configured in Presidio config YAML.
- **Internationalization** — Presidio supports multilingual NLP models; language of input detected automatically.
- **Backward compatibility** — All existing AI services receive a wrapper; external-facing API unchanged.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student prompt "My name is Alice Smith and my email is alice@school.edu. Help me with integrals.", *When* the redaction proxy processes it, *Then* the prompt sent to the LLM reads "My name is [PERSON_1] and my email is [EMAIL_1]. Help me with integrals."
- **AC-2.** *Given* the same value "Alice Smith" appears twice in a prompt, *When* redaction runs, *Then* both occurrences map to the same token `[PERSON_1]`.
- **AC-3.** *Given* Presidio microservice is down, *When* an AI call is attempted, *Then* the call is blocked with error code `pii_proxy_unavailable` and the user sees "AI features temporarily unavailable."
- **AC-4.** *Given* an LLM response contains `[PERSON_1]` (echoed from the prompt), *When* de-anonymization runs, *Then* the response returned to the client reads "Alice Smith" not "[PERSON_1]".
- **AC-5.** *Given* an AI call is processed, *When* the compliance log is inspected, *Then* the log entry contains `entity_counts: { PERSON: 1, EMAIL: 1 }` and no actual PII values.
- **AC-6.** *Given* an org with `pii_redaction_bypass: true` (super_admin-set), *When* an AI call is made, *Then* the prompt is passed unredacted but the log entry includes `{ redaction_bypassed: true, bypass_reason: "self_hosted_org" }`.

## 8. Data Model

No new primary tables. Add to `ai_call_log` (§19.10):
```sql
ALTER TABLE ai_call_log ADD COLUMN redacted_entity_counts JSONB;
ALTER TABLE ai_call_log ADD COLUMN redaction_bypassed BOOLEAN NOT NULL DEFAULT false;
```

De-anonymization map: never stored; held in `HashMap<String, String>` in request scope only.

## 9. API Surface

Presidio is an internal service — not exposed externally. Rust server calls it via HTTP:
```
POST http://presidio-analyzer:3000/analyze   { text, language }
  -> [{ entity_type, start, end, score }]
```

No new public API endpoints from this plan. The redaction is a transparent middleware layer.

## 10. UI / UX

No direct UI. Indirect: AI error state when Presidio is down ("AI features temporarily unavailable") must be displayed clearly in all AI-touching components. Org admin can see redaction stats in the AI governance dashboard (§19.10).

## 11. AI / ML Considerations

- **Microsoft Presidio**: Python-based NER library using spaCy and Hugging Face models for entity recognition.
- **Custom recognizer**: YAML-configured Presidio custom recognizer for `STUDENT_ID` patterns (e.g., 7–10 digit numeric IDs common in US SIS systems).
- **Language detection**: `langdetect` library for auto-language routing to appropriate Presidio NLP model.
- **Recall vs. precision tradeoff**: Presidio configured for high recall (prefer false positives over missed PII); confidence threshold 0.5.
- **Eval**: Measure false-positive rate on a test corpus of educational text; target FPR ≤ 10%.
- **Model card**: `docs/ai/pii-redaction-model-card.md`.
- **No opt-out for students**: Redaction is a system-level privacy protection; individual students cannot disable it.

## 12. Integration Points

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Presidio false positives redact valid content (e.g., teacher name "John" in a history lesson) | H | M | Confidence threshold tuning; entity-type allowlist per feature (don't redact PERSON in history lesson content) |
| Presidio service latency exceeds 100 ms budget | M | M | Local deployment (sidecar); profile and tune spaCy model size |
| De-anonymization map leaks via response logging | M | H | Ensure map never hits tracing/logging layer; scrubber middleware |
| Fail-closed blocks legitimate AI use during outage | M | M | Presidio high-availability deployment; health check + retry with 500 ms timeout |

## 15. Rollout Plan

- Feature flag: `pii_redaction_enabled` (OpenFeature, default `false` — but this is a BLOCKER so it must be enabled before any public AI feature launches).
- Phase 1: Deploy Presidio sidecar; integrate with one AI service (quiz_generation) in shadow mode (redact but also log original to measure false positives).
- Phase 2: Shadow mode across all AI services; tune confidence threshold.
- Phase 3: Hard enforcement (fail-closed) across all AI services.
- Phase 4: GA — required precondition for any AI feature public launch.

## 16. Test Plan

- **Unit** — Redaction proxy: entity replacement, consistent tokenization, de-anonymization, fail-closed behavior.
- **Integration** — Full round-trip: prompt with PII → Presidio → redacted prompt → LLM → de-anonymized response.
- **Security** — De-anonymization map not in logs; bypass only available to super_admin.
- **Performance** — Presidio analysis p95 ≤ 100 ms on 2 000-word input; benchmark with k6.
- **Privacy** — Compliance log contains entity counts but not values; verified by log inspection.
- **Manual** — 50 sample educational texts with manually labeled PII; verify recall ≥ 90%, FPR ≤ 10%.

## 18. Open Questions

1. Should we use Presidio's anonymizer as well (character masking, hashing, encryption) or simple token replacement? Token replacement is more LLM-coherent.
2. Should we provide a "PII audit report" to org admins showing aggregate entity-type counts over time?
3. How do we handle code submissions (§2.4) that might contain variable names matching names (false positive risk)?
4. Should de-anonymization be optional — i.e., for some features, always return tokenized names to the UI?
