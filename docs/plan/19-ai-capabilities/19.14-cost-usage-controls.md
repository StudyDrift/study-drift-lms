## 1. Problem Statement

Every AI feature in Lextures consumes tokens billed at provider rates. Without per-student, per-org, and per-feature token budgets, a single power user can exhaust the platform's AI spend for an entire org. There is currently no visibility into AI costs by org or user, no alerting at spend thresholds, and no throttling mechanism. Before any AI feature launches publicly, cost controls must be in place.

## 2. Goals

- Implement per-student monthly token budgets with configurable soft caps (alert) and hard caps (throttle).
- Implement per-org monthly token budgets with org-admin visibility.
- Provide a cost dashboard for super_admins showing per-org AI spend.
- Alert org admins at 80% and 95% of their budget.
- Throttle AI calls gracefully (return a user-friendly message, not an error) when hard caps are hit.

## 3. Non-Goals

- Direct billing integration with Stripe (§15.3).
- Real-time cost streaming to students (a "you used X tokens" indicator is a nice-to-have future enhancement).
- Controlling third-party provider costs directly (we track our usage and alert; we cannot control OpenRouter's billing).

## 4. Personas & User Stories

- **As a super admin**, I want to see which orgs are spending the most on AI per month so that I can adjust pricing or throttle before a cost overrun.
- **As an org admin**, I want to be alerted when AI usage approaches our contracted limit so that I can request an increase before students are throttled.
- **As a student**, I want a clear message when my AI usage limit is reached ("You've reached your AI usage limit for this month. Your instructor can request an increase.") rather than a cryptic error.
- **As a Lextures finance team member**, I want a monthly report of per-org AI costs in dollars so that I can validate invoices.

## 5. Functional Requirements

- **FR-1.** The system MUST track token consumption per `(student_id, org_id, feature_key, month)` using the `ai_call_log` table (§19.10).
- **FR-2.** The system MUST enforce per-student monthly token budgets: `soft_cap` (alert at threshold, continue) and `hard_cap` (block further AI calls until next month).
- **FR-3.** Default budgets MUST be configurable per org in the `org_ai_budgets` table; defaults: `student_soft_cap = 50_000 tokens/month`, `student_hard_cap = 100_000 tokens/month`, `org_monthly_budget_usd = null` (unlimited unless set).
- **FR-4.** The system MUST send an email alert to the student's instructor and org_admin when any student reaches 80% of their monthly token budget.
- **FR-5.** When a student's hard cap is reached, all AI calls for that student MUST return HTTP 429 with `{ error: "ai_budget_exceeded", reset_at: "<first-of-next-month>" }`.
- **FR-6.** The system MUST provide an org-level AI cost dashboard (in USD, based on configured cost_per_1k_tokens from the model catalog §19.10) visible to org_admin.
- **FR-7.** Super_admin MUST be able to view per-org AI cost summary for any billing period.
- **FR-8.** Super_admin MUST be able to override a student's or org's budget limit (reset, increase) via the admin API.
- **FR-9.** The system SHOULD support per-feature budget allocations (e.g., 30 k tokens/month for tutor, 10 k for summarization) to prevent any single feature from consuming the entire budget.

## 6. Non-Functional Requirements

- **Performance** — Budget check MUST complete in ≤ 20 ms (denormalized counter in Redis or Postgres); this check is on the hot path of every AI call.
- **Security** — Only org_admin and super_admin can modify budget limits; students cannot self-modify limits.
- **Privacy & Compliance** — Token counts are not PII; cost data is business metadata; FERPA not triggered.
- **Accessibility** — Budget dashboard meets WCAG 2.1 AA.
- **Scalability** — Monthly token counters in Redis (TTL-based bucket) or Postgres with UPSERT; high-concurrency safe using atomic increment.
- **Reliability** — If counter store is unavailable, AI calls MUST proceed with logged warning (fail-open for reliability, not security); alert on counter store unavailability.
- **Observability** — `ai_budget_soft_cap_hits_total{org_id}`, `ai_budget_hard_cap_blocks_total{org_id}`, `ai_monthly_cost_usd{org_id}` gauge.
- **Maintainability** — Budget service: `server/src/services/ai_budget.rs`; middleware check inserted into every AI call path.
- **Internationalization** — Budget exceeded message in i18n keys.
- **Backward compatibility** — New middleware; all existing AI calls continue to function until budget is reached.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student has consumed 80 000 of their 100 000-token hard cap, *When* they send a tutor message consuming 1 000 tokens, *Then* the call succeeds and the soft-cap alert email is sent to their instructor.
- **AC-2.** *Given* a student has consumed 100 000 tokens (hard cap), *When* they attempt any AI feature call, *Then* the API returns HTTP 429 with `{ error: "ai_budget_exceeded", reset_at: "2026-05-01T00:00:00Z" }`.
- **AC-3.** *Given* a super_admin overrides a student's budget to 200 000 tokens, *When* the student makes an AI call, *Then* the call succeeds (original hard cap is no longer enforced).
- **AC-4.** *Given* an org has incurred AI usage in the current month, *When* the org_admin views the cost dashboard, *Then* they see total tokens consumed, estimated USD cost, and a breakdown by feature.
- **AC-5.** *Given* the Redis counter store is unavailable, *When* an AI call is attempted, *Then* the call proceeds (fail-open) and a `counter_store_unavailable` warning is logged.

## 8. Data Model

```sql
-- 0XX_ai_budgets.sql
CREATE TABLE org_ai_budgets (
  org_id                  UUID PRIMARY KEY REFERENCES organizations(id),
  student_soft_cap        INT NOT NULL DEFAULT 50000,    -- tokens per student per month
  student_hard_cap        INT NOT NULL DEFAULT 100000,
  org_monthly_budget_usd  NUMERIC,                       -- NULL = unlimited
  per_feature_caps        JSONB,                          -- { "tutor": 30000, "summarize": 10000 }
  updated_by              UUID REFERENCES users(id),
  updated_at              TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Monthly budget usage counters (or Redis TTL keys):
CREATE TABLE ai_budget_usage (
  student_id    UUID NOT NULL REFERENCES users(id),
  org_id        UUID NOT NULL REFERENCES organizations(id),
  feature_key   TEXT NOT NULL,
  month         DATE NOT NULL,    -- first day of month
  tokens_used   BIGINT NOT NULL DEFAULT 0,
  PRIMARY KEY (student_id, feature_key, month)
);
CREATE INDEX abu_org_month ON ai_budget_usage(org_id, month);
```

```

POST   /api/admin/ai-budgets/reset/:student_id -> 204  (super_admin)
GET    /api/super-admin/ai-costs          -> Paginated<OrgCostSummary>  ?month=
```

Budget check: internal middleware function called before every LLM provider call. Returns `BudgetStatus::Allow | BudgetStatus::Warn | BudgetStatus::Block`.

## 11. AI / ML Considerations

Not applicable — this plan governs AI cost infrastructure, not model inference.

Cost estimation formula: `estimated_usd = (input_tokens / 1000 * cost_per_1k_input) + (output_tokens / 1000 * cost_per_1k_output)` using rates from `ai_model_catalog` (§19.10).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Counter race condition under high concurrency | M | M | Atomic Redis INCR or Postgres UPSERT with advisory lock |
| Default caps too low → legitimate students throttled | M | M | Monitor hard-cap hit rate; adjust defaults based on beta data |
| Cost estimation inaccurate (model cost changes) | M | L | Scheduled job to update cost_per_1k rates from provider APIs |

## 15. Rollout Plan

- Feature flag: `ai_budget_enforcement_enabled` (OpenFeature, default `false` for monitoring only initially).
- Phase 1: Deploy counting middleware (no enforcement); measure actual token usage.
- Phase 2: Enable soft-cap alerts only.
- Phase 3: Enable hard-cap enforcement with generous defaults.
- Phase 4: Tighten defaults based on usage data; GA.

## 16. Test Plan

- **Unit** — Budget check logic (allow/warn/block); counter increment; USD cost formula.
- **Integration** — Simulate student reaching soft cap → alert emitted; reaching hard cap → 429 returned.
- **Performance** — Budget check latency ≤ 20 ms under 500 concurrent requests.
- **Security** — Student cannot modify own budget; cross-org budget access blocked.

## 17. Documentation & Training

- Help center: "AI Usage Limits" — what limits are, what happens when reached, how to request increase.
- Admin guide: setting budget limits; reading the cost dashboard.
- Internal runbook: responding to cost overruns; emergency budget override.

## 18. Open Questions

1. Should hard-cap enforcement be a hard requirement before GA, or can we launch with monitoring only and enforce after observing real usage patterns?
2. Should per-feature caps be set by org admins or only by super_admin?
3. What is the right default hard cap — 100 k tokens/month seems generous for K-12 but may be insufficient for self-learners using the tutor heavily?
