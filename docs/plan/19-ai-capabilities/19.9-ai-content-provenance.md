## 1. Problem Statement

Lextures generates AI content across multiple surfaces (quiz questions, rubrics, lesson plans, grading feedback, summaries, translations). Without a standardized provenance record, these artifacts are indistinguishable from human-authored content. Academic integrity tools, institutional auditors, and downstream integrations cannot determine whether a rubric was AI-generated or authored by an instructor. The Content Authenticity Initiative (CAI) and C2PA standard (Coalition for Content Provenance and Authenticity) provide the interoperable framework for signing and verifying digital content origins.

## 2. Goals

- Attach structured provenance metadata to every AI-generated artifact at creation time.
- Implement C2PA-compatible manifests for signable content types (text artifacts, images).
- Display provenance badges in the Lextures UI wherever AI-generated content is shown.
- Provide a verification endpoint so external tools can confirm an artifact's provenance.
- Enable W3C Verifiable Credentials (W3C VC) for AI-generated content attestations where institutional use requires it.

## 3. Non-Goals

- Provenance for human-authored content (out of scope; C2PA supports it, but we focus on AI artifacts).
- Real-time blockchain anchoring (too complex for MVP; timestamp + signature is sufficient).
- Retroactively signing content generated before this plan ships.

## 4. Personas & User Stories

- **As an academic integrity officer**, I want to inspect any AI-generated rubric and see which model produced it, when, and under whose authority so that I can audit AI use in assessments.
- **As an instructor**, I want to see a provenance badge on AI-generated content so that I know what I am editing and approving.
- **As a student**, I want to know if the feedback I received was AI-generated (and reviewed by my instructor) so that I can calibrate my trust in it.
- **As an external tool** (plagiarism checker, LMS integration), I want to query Lextures to verify whether a piece of content was AI-generated by this system.

## 5. Functional Requirements

- **FR-1.** The system MUST attach a `provenance` JSONB field to every AI-generated artifact at creation time containing: `generated_by: "lextures-ai"`, `model_id`, `model_version`, `prompt_template_id`, `generation_ts` (ISO 8601), `org_id`, `actor_id` (user who triggered generation).
- **FR-2.** The system MUST generate a SHA-256 content hash of the artifact at generation time and sign it with a platform Ed25519 private key; the signature and public key fingerprint MUST be stored alongside the artifact.
- **FR-3.** The system MUST provide a public verification endpoint `GET /api/provenance/verify` that accepts a content hash and returns the provenance record (without PII) including signature validity.
- **FR-4.** All AI-generated content displayed in the Lextures UI MUST show a "AI-generated" badge with a link to the provenance detail page.
- **FR-5.** The system SHOULD produce C2PA-compatible manifest JSON for text artifacts, stored alongside the artifact in object storage.
- **FR-6.** The system SHOULD implement W3C Verifiable Credential issuance for AI-generated content attestations for HE institutional use.
- **FR-7.** Provenance records MUST be immutable; the original signature MUST be preserved even if the content is subsequently edited by a human (editing creates a new derivative record).

## 6. Non-Functional Requirements

- **Performance** — Provenance signing MUST add ≤ 50 ms to any generation call (Ed25519 signing is computationally trivial).
- **Security** — Ed25519 private key stored in a secrets manager (§17.17); never in environment variables; key rotation supported with versioning.
- **Privacy & Compliance** — Provenance records on student work (AI grading feedback) must not expose student PII via the public verification endpoint; org_id and actor_id omitted from public response.
- **Accessibility** — Provenance badge has accessible tooltip; not conveyed by icon alone.
- **Scalability** — Provenance records are append-only, low write volume; no scaling concern.
- **Reliability** — If signing fails, generation MUST still succeed; artifact stored without signature with a `signing_failed: true` flag for manual remediation.
- **Observability** — `provenance_records_created_total{artifact_type}`, `signing_failures_total`; alert on signing failures.
- **Maintainability** — Provenance service: `server/src/services/provenance.rs`; called from all AI generation services.
- **Internationalization** — Provenance badge label in i18n keys.
- **Backward compatibility** — Provenance is additive; existing artifacts without provenance not broken.

## 7. Acceptance Criteria

- **AC-1.** *Given* an AI-generated quiz question is created, *When* the question is stored, *Then* a `provenance` record exists with `generated_by`, `model_id`, `generation_ts`, and a valid Ed25519 signature of the content hash.
- **AC-2.** *Given* a provenance record exists, *When* `GET /api/provenance/verify?hash=<sha256>` is called, *Then* the response includes `{ valid: true, generated_by, model_id, generation_ts }` without any student PII.
- **AC-3.** *Given* the content of a generated artifact is subsequently modified by an instructor, *When* the modification is saved, *Then* the original provenance record is preserved, and a new derivative record is created with `derived_from: <original_hash>` and `modified_by: "human"`.
- **AC-4.** *Given* a C2PA manifest is generated for an AI-generated lesson plan text, *When* the manifest is validated by a C2PA-compatible tool, *Then* the manifest passes validation and references the Lextures signing key.
- **AC-5.** *Given* AI-generated feedback is displayed to a student, *When* the feedback is rendered, *Then* an "AI-generated" badge is visible with an accessible tooltip "This content was generated by AI."

## 8. Data Model

```sql
-- 0XX_provenance.sql
CREATE TABLE provenance_records (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  artifact_type     TEXT NOT NULL,   -- 'quiz_question' | 'rubric' | 'lesson_plan' | 'grading_feedback' | ...
  artifact_id       UUID NOT NULL,
  content_hash      TEXT NOT NULL,   -- SHA-256 hex
  signature         TEXT,            -- Ed25519 signature base64
  public_key_fp     TEXT,            -- key fingerprint
  signing_failed    BOOLEAN NOT NULL DEFAULT false,
  provenance_data   JSONB NOT NULL,  -- { generated_by, model_id, model_version, prompt_template_id, generation_ts, org_id }
  derived_from_hash TEXT,            -- if this is a derivative
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX provenance_content_hash ON provenance_records(content_hash);
CREATE INDEX provenance_artifact ON provenance_records(artifact_type, artifact_id);
```

## 9. API Surface

```
GET /api/provenance/verify?hash=<sha256>     -> ProvenanceVerifyResponse (public)
GET /api/provenance/:artifact_type/:id       -> ProvenanceRecord (authenticated, org-scoped)
GET /api/provenance/public-key               -> { key_id, public_key_pem, algorithm }
```

C2PA manifest stored as a sidecar file in object storage alongside the artifact.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Signing key compromise → forged provenance | L | H | HSM or secrets manager; key rotation; audit log on key usage |
| C2PA spec evolves, manifests become invalid | M | L | Abstract C2PA generation behind interface; version manifest spec |
| Provenance adds latency to generation calls | L | L | Ed25519 signing is <1 ms; async fire-and-forget for manifest storage |

## 15. Rollout Plan

- Feature flag: `ai_provenance_enabled` (OpenFeature, default `false`).
- Phase 1: Provenance record creation (no UI badge) — validate data completeness.
- Phase 2: UI badges on AI content.
- Phase 3: Public verification endpoint.
- Phase 4: C2PA manifests and W3C VC (HE-specific rollout).

## 16. Test Plan

- **Unit** — Ed25519 signing; content hash computation; C2PA manifest generation; verification logic.
- **Integration** — AI generation call → provenance record created → verify endpoint returns valid.
- **Security** — Tampered content hash → verify returns `{ valid: false }`; unauthorized artifact provenance access blocked.
- **Accessibility** — Provenance badge tooltip and link keyboard accessible.

## 17. Documentation & Training

- Help center: "AI Content Provenance" — what the badge means, how to verify.
- Developer docs: Verification API reference.
- Internal runbook: signing key rotation procedure.

## 18. Open Questions

1. Should we publish our signing public key as a well-known endpoint for external tools?
2. Should W3C VC issuance be synchronous (adds latency) or asynchronous (badge not immediately available)?
3. Do we need a revocation mechanism for signed artifacts if the model is found to produce biased content?
