## 1. Problem Statement

When students give wrong answers on quizzes or fail rubric criteria, that information currently goes nowhere beyond a numeric score. The adaptive value of knowing *why* a student is wrong — which specific misconception they hold — is entirely lost. Cross-referencing §1.10, there is no "you missed this because..." remediation path. AI-assisted grading (§19.3) produces misconception signals as a byproduct; this plan connects those signals to the gradebook, the learner model, and instructor dashboards.

## 2. Goals

- Aggregate per-student misconception signals from AI grading (§19.3) and quiz analysis into a structured misconception store.
- Roll these up into instructor-facing gradebook indicators and student-facing study recommendations.
- Feed into the learner mastery model (§1.1) so misconception data contributes to knowledge-state updates.
- Provide an instructor dashboard showing class-wide misconception prevalence to guide re-teaching.

## 3. Non-Goals

- Building the full remediation content library (§1.10) — this plan creates the data substrate; remediation links to content are a future step.
- Real-time misconception detection during live quiz (too high latency); detection runs post-submission.
- Parent-facing misconception reports (future phase).

## 4. Personas & User Stories

- **As a student**, I want to see "You may be confusing photosynthesis with cellular respiration" after a wrong answer so that I know what to study.
- **As an instructor**, I want to see that 35% of my class has the same misconception about balancing equations so that I can re-teach that concept next class.
- **As a student**, I want misconception feedback linked to the exact lesson section where the concept is explained so that I can self-remediate.
- **As an adaptive engine** (system), I want misconception records to update the learner's concept mastery state so that future adaptive path decisions account for known errors.

## 5. Functional Requirements

- **FR-1.** The system MUST store per-student misconception records with: `student_id`, `concept_id` (from skill graph §1.2), `description` (natural language), `source` (`ai_grading` | `quiz_attempt`), `severity` (`minor` | `moderate` | `major`), `created_at`.
- **FR-2.** AI grading jobs (§19.3) MUST write misconception records for any identified conceptual error in a graded submission.
- **FR-3.** Quiz grading MUST analyze incorrect answers for common misconceptions using the existing `adaptive_quiz_ai.rs` heuristic and write misconception records.
- **FR-4.** The system MUST aggregate class-level misconception statistics per concept per course for instructor view.
- **FR-5.** The gradebook MUST display a misconception indicator icon on cells where the student has an open misconception record for the graded concept.
- **FR-6.** The student-facing quiz/assignment result page MUST display identified misconceptions with natural-language descriptions and, where available, links to course content chunks (RAG-retrieved).
- **FR-7.** Misconception records MUST be marked as `resolved` when the student subsequently demonstrates mastery of the affected concept (quiz score ≥ threshold on next attempt).
- **FR-8.** The system MUST update the learner mastery model (§1.1) when a misconception is created or resolved.

## 6. Non-Functional Requirements

- **Performance** — Misconception writes MUST be asynchronous (fire-and-forget from grading job); instructor aggregation query MUST return in ≤ 500 ms for a class of 200 students.
- **Security** — Student misconception records visible only to the student, their course instructor, and org_admin; cross-student access blocked.
- **Privacy & Compliance** — Misconception records are FERPA-protected education records; retained per §19.1 tutor retention policy; included in DSAR export (§10.3).
- **Accessibility** — Misconception indicators in gradebook have descriptive tooltip and aria-label; not conveyed by color alone.
- **Scalability** — Indexed by `(student_id, concept_id, created_at)`; concept aggregation uses a materialized view refreshed on a schedule.
- **Reliability** — If misconception write fails, grading job continues; misconception is non-critical path.
- **Observability** — `misconceptions_created_total{source}`, `misconceptions_resolved_total`; alert if a concept has >50% prevalence in a class.
- **Maintainability** — Service: `server/src/services/misconception_store.rs`.
- **Internationalization** — Misconception descriptions localized per student locale.
- **Backward compatibility** — Additive; existing grading flow unchanged.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student submits a quiz answer that confuses mitosis and meiosis, *When* AI grading completes, *Then* a misconception record `{ concept: "meiosis", description: "Confused meiosis with mitosis", severity: "moderate" }` exists for the student.
- **AC-2.** *Given* a class of 30 students where 12 have the same misconception on "balancing chemical equations", *When* an instructor views the concept misconception dashboard, *Then* the concept shows "40% of class" prevalence.
- **AC-3.** *Given* a misconception record exists for a student on concept C, *When* the student scores ≥ 80% on the next quiz assessing concept C, *Then* the misconception record status changes to `resolved`.
- **AC-4.** *Given* a gradebook cell for an assignment, *When* the student has an open misconception linked to that assignment's concept, *Then* a misconception icon is visible in the cell with an accessible tooltip describing the misconception.
- **AC-5.** *Given* a student views their quiz result, *When* misconceptions are detected, *Then* each misconception shows the description and a link to the relevant course content section.

## 8. Data Model

```sql
-- 0XX_misconceptions.sql
CREATE TYPE misconception_severity AS ENUM ('minor', 'moderate', 'major');
CREATE TYPE misconception_source   AS ENUM ('ai_grading', 'quiz_attempt', 'tutor_session');
CREATE TYPE misconception_status   AS ENUM ('open', 'resolved');

CREATE TABLE student_misconceptions (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  student_id    UUID NOT NULL REFERENCES users(id),
  course_id     UUID NOT NULL REFERENCES courses(id),
  concept_id    UUID REFERENCES concepts(id),       -- §1.2 skill graph
  description   TEXT NOT NULL,
  source        misconception_source NOT NULL,
  source_ref_id UUID,                               -- submission_id or quiz_attempt_id
  severity      misconception_severity NOT NULL DEFAULT 'moderate',
  status        misconception_status NOT NULL DEFAULT 'open',
  resolved_at   TIMESTAMPTZ,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX sm_student_course ON student_misconceptions(student_id, course_id, status);
CREATE INDEX sm_course_concept ON student_misconceptions(course_id, concept_id, status);
```

Materialized view for class-level aggregation refreshed hourly:
```sql
CREATE MATERIALIZED VIEW misconception_class_summary AS
  SELECT course_id, concept_id, COUNT(DISTINCT student_id) as student_count, ...
```

## 9. API Surface

```
GET /api/courses/:id/misconceptions/class      -> ClassMisconceptionSummary[]  (instructor)
GET /api/courses/:id/misconceptions/me         -> StudentMisconception[]  (student)
GET /api/courses/:id/misconceptions/student/:sid -> StudentMisconception[]  (instructor)
PATCH /api/misconceptions/:id                  { status: 'resolved' }  (instructor manual resolve)
```

## 10. UI / UX

- Gradebook: misconception badge (yellow triangle icon) on cells; hover/focus shows tooltip with top misconception.
- Student result page: "Learning Notes" section below score showing misconception cards with descriptions and content links.
- Instructor concept dashboard: `clients/web/src/pages/lms/MisconceptionDashboard.tsx` — class heatmap of concepts vs. student count.

## 11. AI / ML Considerations

- **Source**: Misconception descriptions generated by AI grading service (§19.3) and `adaptive_quiz_ai.rs`.
- **Concept tagging**: LLM-extracted misconception text mapped to concept IDs via cosine similarity against concept graph embeddings (§1.2).
- **Remediation links**: RAG retrieval (§19.1 tutor infrastructure) finds course content chunks most relevant to misconception description.
- **Model card**: Covered by §19.3 grading model card.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| False-positive misconceptions annoy students | M | M | Severity threshold: only surface `moderate`/`major`; allow student dismissal |
| Concept graph not available at launch | H | M | Allow free-text concept labels as fallback without concept_id FK |
| FERPA exposure of misconception data cross-student | L | H | ACL enforcement + integration tests |

## 15. Rollout Plan

- Feature flag: `misconception_detection_enabled` (OpenFeature, default `false`).
- Phase 1: Misconception storage only (no UI); validate data quality.
- Phase 2: Student result page indicators.
- Phase 3: Instructor dashboard and gradebook integration.

## 16. Test Plan

- **Unit** — Misconception write from grading job; resolution logic on re-quiz; aggregation query.
- **Integration** — AI grading → misconception created → student views result → resolves on next quiz.
- **Security** — Cross-student access blocked; instructor-only class dashboard access.
- **Accessibility** — axe-core on gradebook badge and student result page indicators.
