## 1. Problem Statement

Lextures has a student notebook RAG service (`server/src/services/student_notebook_rag_ai.rs`) that is reactive and per-session only. When a student returns the next day, the tutor has no memory of previous conversations, misunderstood concepts, or progress. Without persistent memory, the AI cannot provide the longitudinal, personalized guidance that distinguishes an adaptive LMS from a simple chatbot. This is a competitive differentiator gap versus both traditional LMS incumbents and AI-native startups.

## 2. Goals

- Provide every enrolled student with a persistent AI tutor that retains conversation history, tracks identified knowledge gaps, and cites course materials.
- Ground all tutor responses in course content via RAG, with mandatory citation (cross-ref §19.12).
- Apply PII redaction before any prompt leaves the system (cross-ref §19.11).
- Comply with FERPA requirements for AI-tutor data retention and student data use disclosure.
- Provide an opt-out path for students (and district admins on behalf of students) who do not wish their work sent to an LLM.

## 3. Non-Goals

- A general-purpose AI assistant not grounded in course materials.
- Replacing human instructors or automated grading (§19.3).
- Real-time voice tutor (§19.6).
- Parent or instructor view of full tutor transcripts (privacy concern; summary view is a future enhancement).

## 4. Personas & User Stories

- **As a student**, I want the AI tutor to remember that I struggled with derivatives last week so that it can check my understanding when I work on integrals today.
- **As a self-learner**, I want the tutor to cite the specific lesson section it is drawing from so that I can verify the answer myself.
- **As a student whose district has an AI opt-out**, I want the tutor feature to be unavailable to me (not just hidden) so that my work is never sent to an LLM.
- **As an instructor**, I want to see an aggregate summary of what concepts the AI tutor identified as confusing across my class so that I can address them in the next lecture.

## 5. Functional Requirements

- **FR-1.** The system MUST persist each AI tutor conversation as a named session associated with the student's enrollment in a specific course; sessions MUST survive browser close and re-login.
- **FR-2.** At the start of each conversation turn, the system MUST retrieve the 10 most recent messages from the student's tutor history for that course and inject them into the context window.
- **FR-3.** Every tutor response MUST include at least one citation to a course content chunk (lesson, module, or uploaded resource) in a structured `citations[]` field (cross-ref §19.12).
- **FR-4.** All user messages MUST pass through the PII redaction proxy (cross-ref §19.11) before being sent to the LLM provider.
- **FR-5.** The system MUST respect a per-student `ai_tutor_opt_out` flag; if set, the tutor endpoint MUST return HTTP 403 with a human-readable explanation.
- **FR-6.** The system MUST respect per-tenant model governance settings (cross-ref §19.10) when selecting the LLM provider and model.
- **FR-7.** The system MUST store tutor interactions with a retention policy configurable by the org admin (default 1 year, minimum required for FERPA compliance); purge on expiry.
- **FR-8.** The system SHOULD detect when a student's message indicates confusion about a concept and tag the interaction with the relevant concept IDs from the skill graph (cross-ref §1.2).
- **FR-9.** The system SHOULD surface a weekly "concepts your students found confusing" digest to instructors (aggregate, not individual student data).
- **FR-10.** The tutor MUST include a system-level disclosure in its first message: "I am an AI tutor. I can make mistakes — please verify important information with your instructor."

## 6. Non-Functional Requirements

- **Performance** — First token latency ≤ 1.5 s p95 for streaming responses; context retrieval from RAG ≤ 300 ms p95.
- **Security** — Tutor sessions are private to the student; instructors cannot read individual transcripts without impersonation (§18.3); model API keys managed per §19.10.
- **Privacy & Compliance** — FERPA: tutor transcripts are education records; access restricted to student and authorized school officials; COPPA: under-13 users must have parental consent before tutor is enabled (cross-ref §10.2); GDPR: right to erasure must delete tutor history; AI usage disclosed per §10.17.
- **Accessibility** — Chat UI MUST meet WCAG 2.1 AA; messages must have proper heading structure; citations rendered as navigable links.
- **Scalability** — Session context retrieval via indexed query on `(student_id, course_id, created_at DESC LIMIT 10)`; horizontal scaling via stateless API layer.
- **Reliability** — If LLM provider returns an error, the tutor MUST display a graceful error and offer a retry; conversation history MUST not be lost on provider failure.
- **Observability** — `tutor_requests_total`, `tutor_latency_ms` histogram, `tutor_citations_per_response` gauge; token usage logged per interaction for §19.14 cost controls.
- **Maintainability** — Build on top of `server/src/services/student_notebook_rag_ai.rs`; extract session persistence to `server/src/services/tutor_session.rs`.
- **Internationalization** — System prompts localized per org/user locale.
- **Backward compatibility** — Existing RAG notebook endpoint preserved; tutor is an additive enhancement.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student has a previous tutor session where they discussed polynomial factoring, *When* they start a new conversation the next day, *Then* the tutor references the earlier discussion when the student asks about FOIL method.
- **AC-2.** *Given* a student's `ai_tutor_opt_out = true`, *When* they access the tutor endpoint, *Then* HTTP 403 is returned with message "AI tutor is disabled for your account."
- **AC-3.** *Given* a tutor response is generated, *When* the response is returned, *Then* `citations[0].source_id` references a real content chunk in the course's RAG index.
- **AC-4.** *Given* a student message containing a real name and email, *When* the PII proxy processes the message, *Then* the forwarded prompt replaces the name and email with placeholder tokens before reaching the LLM.
- **AC-5.** *Given* tutor sessions older than the org's configured retention period, *When* the nightly purge job runs, *Then* those sessions and their messages are deleted from the database.
- **AC-6.** *Given* the LLM provider returns a 503 error, *When* the tutor request fails, *Then* the UI displays "The tutor is temporarily unavailable. Your conversation history is saved." and no data is lost.

## 8. Data Model

```sql
-- 0XX_tutor_sessions.sql
CREATE TABLE tutor_sessions (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  student_id    UUID NOT NULL REFERENCES users(id),
  course_id     UUID NOT NULL REFERENCES courses(id),
  title         TEXT,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  last_active   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX tutor_sessions_student_course ON tutor_sessions(student_id, course_id, last_active DESC);

CREATE TABLE tutor_messages (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id    UUID NOT NULL REFERENCES tutor_sessions(id) ON DELETE CASCADE,
  role          TEXT NOT NULL CHECK (role IN ('user','assistant','system')),
  content       TEXT NOT NULL,
  citations     JSONB,        -- [{ source_id, chunk_id, excerpt }]
  concept_tags  UUID[],       -- concept IDs from skill graph
  token_count   INT,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX tutor_messages_session_created ON tutor_messages(session_id, created_at);
```

## 9. API Surface

```
GET    /api/courses/:id/tutor/sessions           -> TutorSession[]
POST   /api/courses/:id/tutor/sessions           -> TutorSession
GET    /api/courses/:id/tutor/sessions/:sid      -> TutorSession & { messages: TutorMessage[] }
POST   /api/courses/:id/tutor/sessions/:sid/messages  { content: string }
                                                  -> TutorMessage (SSE stream)
DELETE /api/courses/:id/tutor/sessions/:sid      -> 204
GET    /api/courses/:id/tutor/concept-confusion  -> ConceptConfusionSummary[]  (instructor only)
```

Streaming: SSE on POST message endpoint for progressive token delivery.

## 10. UI / UX

- Chat panel: `clients/web/src/components/TutorChat.tsx` — integrated into course sidebar.
- Session list: dropdown or left-panel list for previous sessions.
- Citation chips below each assistant message; clicking opens the cited content chunk in a modal.
- Disclosure banner on first use: "This AI tutor uses your course materials and conversation history. [Manage AI settings]."
- Opt-out: link to student privacy settings from the disclosure banner.

## 11. AI / ML Considerations

- **Model**: OpenRouter-routed; default to `claude-3-5-sonnet` for balanced cost/quality; configurable per §19.10.
- **Prompt**: System prompt in `server/src/services/tutor_session.rs`; version-controlled in DB (cross-ref §19.10).
- **RAG**: Inherit from `student_notebook_rag_ai.rs`; chunk retrieval via pgvector cosine similarity.
- **Eval metric**: Citation accuracy (% responses with valid citations), hallucination rate (golden-answer RAGAS eval), student satisfaction (thumbs up/down).
- **Fallback**: If RAG retrieval fails, respond from LLM general knowledge with prominent disclaimer.
- **PII redaction**: Microsoft Presidio proxy (§19.11) applied to all user messages.
- **Cost budget**: Per-student token budget enforced by §19.14; default 50 k tokens/student/month.
- **Opt-out**: `ai_tutor_opt_out` flag on user record; set by student or admin.
- **Model card**: Published at `/docs/ai/tutor-model-card.md`.
- **Eval harness**: Cross-ref §19.13.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| LLM produces hallucinated citations | H | H | Citation validation: verify cited chunk_id exists in RAG index before responding |
| FERPA violation from instructor accessing student transcripts | M | H | Hard ACL: instructor can only see aggregated concept summaries, not individual messages |
| Token costs spiral for active students | M | M | Per-student monthly token budget (§19.14) with soft cap and alert |
| Student shares sensitive info (abuse reports, mental health) in tutor | M | H | Keyword detection + mandatory disclosure to school counselor; documented in model card |

## 15. Rollout Plan

- Feature flag: `persistent_tutor_enabled` (OpenFeature, default `false`).
- Phase 1: Internal testing with synthetic student data; eval harness baseline.
- Phase 2: Pilot with 2–3 self-learner courses (no FERPA/COPPA concerns).
- Phase 3: K-12 / HE pilot with parental consent flow and opt-out controls in place.
- Phase 4: GA after FERPA legal review of data retention policy.
- Rollback: Disable flag; existing sessions preserved; no data deleted on rollback.

## 16. Test Plan

- **Unit** — Session retrieval window (10 messages); PII proxy integration; opt-out enforcement; citation validation.
- **Integration** — Multi-turn conversation maintains context; RAG retrieval returns grounded citations; token count logged.
- **End-to-end** — Playwright: student sends 3 messages; closes browser; reopens; verifies history loaded; verifies citation link navigates.
- **Security** — Instructor attempts to read student messages (should fail); cross-student session access.
- **Accessibility** — axe-core on chat UI; keyboard navigation; citation links accessible.
- **Performance** — 100 concurrent tutor requests; p95 first-token ≤ 1.5 s.
- **AI eval** — RAGAS pipeline (§19.13): 50 golden Q&A pairs from course content; citation accuracy ≥ 90%; hallucination rate ≤ 5%.
