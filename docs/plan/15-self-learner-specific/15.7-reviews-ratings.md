## 1. Problem Statement

On self-learner platforms, social proof through reviews and ratings is the primary purchase-decision signal. A course with 4.7 stars and 2,000 reviews sells; a course with no rating does not. Lextures has no rating or review system. Without it, learners cannot evaluate course quality, creators have no feedback loop, and the catalog lacks the trust signals that drive conversion. This is the most visible gap compared to Udemy and Coursera.

## 2. Goals

- Allow enrolled learners who have completed at least one module to leave a star rating (1–5) and optional text review.
- Aggregate ratings into an average displayed on catalog listings and course pages.
- Provide moderation tools for admins to flag or remove inappropriate reviews.
- Prompt learners for a review at the right moment (after first module completion and at course completion).

## 3. Non-Goals

- Instructor reviews of students.
- Anonymous reviews (reviews are attributed to the reviewer by display name).
- Review helpfulness votes ("X people found this helpful") — deferred to v2.
- Review import from third-party platforms.

## 4. Personas & User Stories

- **As a prospective learner**, I want to read reviews from other learners before enrolling so that I can gauge course quality.
- **As a creator**, I want to see ratings and reviews for my course so that I can improve my content.
- **As a learner**, I want to leave a review after completing a course so that I can help others make informed decisions.
- **As an admin**, I want to be able to remove reviews that violate community guidelines.

## 5. Functional Requirements

- **FR-1.** The system MUST allow enrolled learners who have completed at least 10% of a course to submit a 1–5 star rating and an optional text review (max 2,000 characters).
- **FR-2.** The system MUST compute and store an average rating and review count per course, updated on each new review.
- **FR-3.** The system MUST display reviews on the public course landing page (15.1) sorted by recency and by "most helpful" (helpfulness votes deferred; recency only for v1).
- **FR-4.** The system MUST enforce one review per enrolled learner per course (editable within 30 days of submission).
- **FR-5.** The system MUST provide admins a review moderation queue to review flagged content and remove violations.
- **FR-6.** The system SHOULD prompt learners with a review request at 25% course completion and at 100% (completion).
- **FR-7.** The system SHOULD allow creators to publicly respond to reviews.

## 6. Non-Functional Requirements

- **Performance** — Review list on course landing page loads in <500 ms (paginated, 10 per page).
- **Security** — Review submission requires active enrollment; admin moderation endpoints admin-only.
- **Privacy & Compliance** — Reviews attributed to learner's display name (not email); GDPR right-to-erasure: deleting a user must anonymize their reviews.
- **Accessibility** — Star rating input uses `role="radiogroup"` with labelled 1–5 options; WCAG 2.1 AA.
- **Scalability** — Average rating computed as a running aggregate (stored `rating_sum`, `rating_count`) not a query-time average; instant update on new review.
- **Reliability** — Review submission is idempotent (second submit from same user updates existing row).
- **Observability** — Metric `course_reviews_total`; `average_rating_distribution` histogram.
- **Maintainability** — Reviews service in `server/src/services/course_reviews.rs`.
- **Internationalization** — Review UI labels externalised; star labels "1 – Poor, 5 – Excellent."
- **Backward compatibility** — Existing courses have no reviews; `average_rating` starts NULL.

## 7. Acceptance Criteria

- **AC-1.** *Given* a learner has completed 20% of a course, *when* they submit a 4-star review with text, *then* the review appears on the course landing page and the course average rating updates.
- **AC-2.** *Given* a learner tries to submit a second review for the same course, *when* they submit, *then* their existing review is updated, not duplicated.
- **AC-3.** *Given* an admin flags a review as a violation, *when* they remove it, *then* the review is soft-deleted and the course average rating is recomputed.
- **AC-4.** *Given* a course has 0 reviews, *when* the catalog listing renders, *then* no star rating is shown (empty state rather than 0 stars).
- **AC-5.** *Given* a learner deletes their account, *when* the deletion is processed, *then* their review text is replaced with "[Review removed]" and the rating is preserved for aggregate accuracy.

## 8. Data Model

Migration `097_course_reviews.sql`:
```sql
CREATE TABLE course_reviews (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  course_id       UUID NOT NULL REFERENCES courses(id),
  reviewer_id     UUID NOT NULL REFERENCES users(id),
  rating          SMALLINT NOT NULL CHECK (rating BETWEEN 1 AND 5),
  review_text     TEXT,
  creator_response TEXT,
  is_flagged      BOOLEAN NOT NULL DEFAULT FALSE,
  is_removed      BOOLEAN NOT NULL DEFAULT FALSE,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(course_id, reviewer_id)
);
CREATE INDEX idx_reviews_course ON course_reviews(course_id, created_at DESC) WHERE is_removed = FALSE;

-- Running aggregate on courses table
ALTER TABLE courses
  ADD COLUMN rating_sum INT NOT NULL DEFAULT 0,
  ADD COLUMN rating_count INT NOT NULL DEFAULT 0;
-- average_rating derived: rating_sum / NULLIF(rating_count, 0)
```

## 9. API Surface

```
POST  /api/v1/courses/:id/reviews       -- submit or update review (enrolled learner)
GET   /api/v1/courses/:id/reviews       -- list reviews (public, paginated)
PATCH /api/v1/courses/:id/reviews/:rid  -- update own review (learner)
POST  /api/v1/courses/:id/reviews/:rid/flag  -- flag review (any user)
DELETE /api/v1/admin/reviews/:rid       -- remove review (admin)
POST  /api/v1/courses/:id/reviews/:rid/response -- creator response
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Fake/incentivized reviews inflate ratings | M | H | Require completed 10% of course; rate-limit reviews per user; moderation queue |
| Negative reviews harm creator reputation unfairly | M | M | Creator response feature; moderation process for guideline violations |
| GDPR erasure breaks aggregate rating integrity | L | L | Anonymize text but retain rating for aggregate; documented policy |

## 15. Rollout Plan

- Feature flag: `course_reviews` (default: off; SL tier).
- Pilot with 5 courses; monitor for gaming attempts.
- GA alongside 15.1 catalog launch.

## 16. Test Plan

- **Unit** — Rating aggregate update; 10% completion eligibility check; idempotent review update.
- **Integration** — Submit review, verify aggregate updates; flag → admin removes → aggregate recomputes.
- **End-to-end** — Playwright: learner submits review, appears on catalog page, creator responds.
- **Security** — Assert unenrolled user cannot submit review; assert learner cannot remove other's review.
- **Accessibility** — axe on review form and catalog review section.
- **Manual exploratory** — QA: edit review within 30 days, edit after 30 days blocked, zero-review empty state, creator response.
