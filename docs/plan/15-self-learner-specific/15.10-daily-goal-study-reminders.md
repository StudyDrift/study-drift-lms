## 1. Problem Statement

The biggest cause of self-learner course abandonment is forgetting to return to the platform. Duolingo's daily reminder notifications drive an estimated 50% of its daily active users. Lextures has no notification system (6.2 and 6.3 are both MISSING), no daily-goal configuration, and no reminder scheduling. Without this, learners who don't open the app for a week simply never return. Combined with streak mechanics (15.9), personalized daily reminders are the single highest-leverage retention feature.

## 2. Goals

- Allow learners to configure a daily study goal (minutes per day or modules per day).
- Schedule daily push or email reminders at a learner-chosen time (in their time zone).
- Generate streak-at-risk alerts automatically when a learner has not yet studied today and their streak is active.
- Provide weekly progress summary emails showing streak, XP earned, and courses in progress.

## 3. Non-Goals

- AI-generated personalized study schedules (deferred to a future adaptive-path feature).
- SMS reminders (push and email only at GA).
- Reminder scheduling for HE instructor-led courses (those have assignment due date notifications, not daily goals).

## 4. Personas & User Stories

- **As a self-learner**, I want to set a daily study goal of 20 minutes so that I can build a learning habit without overwhelming myself.
- **As a learner**, I want to receive a push notification at 7 PM if I haven't studied yet today so that I don't forget.
- **As a learner**, I want to receive a weekly summary email every Sunday showing how I did that week so that I stay accountable.
- **As a learner**, I want to turn reminders off for a specific course during a busy week without losing my global settings.

## 5. Functional Requirements

- **FR-1.** The system MUST allow learners to configure a daily study goal (minutes), a preferred reminder time, and a reminder channel (email, push, or both).
- **FR-2.** The system MUST send a daily reminder at the configured time (adjusted for learner's time zone) if the learner has not completed any activity that day.
- **FR-3.** The system MUST send a streak-at-risk alert (via 6.2/6.3) when the learner's streak is active and they have not studied by 2 hours before their reminder time.
- **FR-4.** The system MUST send a weekly progress summary email every Sunday (configurable day) with: streak length, XP earned that week, courses in progress with progress percentages.
- **FR-5.** The system SHOULD allow learners to set reminder pause for a configurable number of days.
- **FR-6.** The system MUST respect unsubscribe and opt-out preferences; reminder opt-out must not affect transactional email (receipts, credentials).

## 6. Non-Functional Requirements

- **Performance** — Nightly reminder job processes 100,000 users in <5 min.
- **Security** — Reminder scheduling does not expose user data; email unsubscribe links are signed.
- **Privacy & Compliance** — GDPR: reminder consent collected at onboarding (15.11); CAN-SPAM/CASL compliant one-click unsubscribe; GDPR data export includes reminder preferences.
- **Accessibility** — Weekly summary email template follows email accessibility best practices (table-free layout, adequate font size, alt text).
- **Scalability** — Reminder job fans out via background job queue (17.3) with per-worker concurrency.
- **Reliability** — At-most-once reminder delivery using idempotency keys; reminder not sent if already sent today.
- **Observability** — Metric `study_reminders_sent_total{channel}`; `reminder_opt_out_rate`.
- **Maintainability** — Reminder scheduler in `server/src/services/study_reminders.rs`; reuses email service (6.2) and push service (6.3).
- **Internationalization** — Reminder email templates fully externalised; day names use locale-aware formatting.
- **Backward compatibility** — Reminders default to off; no existing learner is auto-enrolled into reminders.

## 7. Acceptance Criteria

- **AC-1.** *Given* a learner sets a reminder for 7 PM in their time zone, *when* 7 PM arrives and they have not studied today, *then* they receive a push notification within 5 minutes.
- **AC-2.** *Given* a learner has studied today, *when* the reminder job runs at their scheduled time, *then* no reminder is sent.
- **AC-3.** *Given* a learner has a 5-day streak and it is 10 PM with no activity (their reminder is 9 PM), *when* the streak-at-risk job runs, *then* a streak-at-risk notification is sent.
- **AC-4.** *Given* a learner clicks "Unsubscribe" in a reminder email, *when* the link is processed, *then* their reminder preference is set to disabled and no further daily reminders are sent.
- **AC-5.** *Given* a learner opens the weekly summary email on Sunday, *when* they have earned 250 XP this week and maintained an 8-day streak, *then* the email shows both values correctly with their courses' progress percentages.

## 8. Data Model

Migration `100_study_reminders.sql`:
```sql
CREATE TABLE study_reminder_configs (
  user_id             UUID PRIMARY KEY REFERENCES users(id),
  daily_goal_minutes  INT NOT NULL DEFAULT 20,
  reminder_time       TIME NOT NULL DEFAULT '19:00',
  reminder_channels   TEXT[] NOT NULL DEFAULT '{"email"}',
  paused_until        DATE,
  weekly_summary      BOOLEAN NOT NULL DEFAULT TRUE,
  enabled             BOOLEAN NOT NULL DEFAULT FALSE,
  updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE reminder_send_log (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id),
  send_date   DATE NOT NULL,
  reminder_type TEXT NOT NULL, -- 'daily', 'streak_at_risk', 'weekly_summary'
  channel     TEXT NOT NULL,
  idempotency_key TEXT UNIQUE,
  sent_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE UNIQUE INDEX idx_reminder_log_dedup ON reminder_send_log(user_id, send_date, reminder_type, channel);
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Reminder spam increases unsubscribe rate | M | M | Default reminders OFF; respect opt-out; limit to 1 reminder per day |
| Time zone miscalculation causes middle-of-night reminders | M | H | Store TZ in user profile (11.4); double-check TZ in reminder job |
| Email service rate limits during large daily batch | M | M | Fan out reminders over 1-hour window; batch size throttled |

## 15. Rollout Plan

- Feature flag: `study_reminders` (default: off; SL tier).
- Email reminders ship first; push in follow-up after 6.3.
- Pilot with opt-in beta group; measure 7-day retention lift.
- GA alongside 15.11 onboarding (reminder setup in onboarding).

## 16. Test Plan

- **Unit** — Reminder eligibility (no activity today); streak-at-risk threshold; idempotency key generation.
- **Integration** — Reminder job at simulated 7 PM → email sent; job at 7 PM with activity → no email sent.
- **End-to-end** — Playwright: set reminder at 7 PM, simulate no activity, verify email in test inbox.
- **Security** — Assert unsubscribe token is single-use and signed; assert reminder send log not accessible to other users.
- **Accessibility** — axe on reminder settings section; weekly email a11y audit.
- **Performance / load** — 100,000 user reminder job in <5 min.
- **Manual exploratory** — QA: reminder pause, unsubscribe, weekly summary with no activity, midnight time zone boundary.
