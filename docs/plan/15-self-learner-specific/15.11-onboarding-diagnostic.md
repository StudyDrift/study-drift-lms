## 1. Problem Statement

New self-learners arrive at Lextures without context — there is no "where should I start?" flow. They see a course catalog and must guess which course matches their current skill level and goals. This mismatch drives early abandonment. Platforms like Khan Academy and Brilliant use onboarding diagnostic assessments to place learners at the right difficulty level and capture their learning goal (timeline, motivation, prior knowledge). Without this, Lextures cannot deliver on its "truly adaptive" promise for self-learners, and new user activation rates suffer. Cross-reference: 1.7 covers the underlying diagnostic assessment infrastructure; this plan is the SL-specific onboarding UX flow that consumes it.

## 2. Goals

- Present new self-learner registrants with a 5–7 minute onboarding flow that captures: learning goal (topic, timeline), self-assessed prior knowledge, study-habit preferences, and daily-goal configuration.
- Run an optional lightweight diagnostic quiz (from 1.7) to place the learner in the correct starting point within their chosen subject.
- Generate a personalized "start here" course recommendation based on diagnostic results.
- Store the learner's goals in their profile for use by the AI study buddy (15.12) and reminder system (15.10).
- Satisfy GDPR consent and reminder opt-in requirements during onboarding.

## 3. Non-Goals

- Full IRT-based placement assessment (that is 1.7; this plan wraps it in the SL UX flow).
- Onboarding for HE/K-12 (different user journey; they are enrolled by institutions).
- Mandatory onboarding blocking access to the catalog.

## 4. Personas & User Stories

- **As a new self-learner**, I want to tell the platform what I want to learn and by when so that my Dashboard is personalized from day one.
- **As a learner returning after a break**, I want to update my learning goals without going through the full onboarding again.
- **As a learner**, I want to take a quick diagnostic quiz so that I start at the right level rather than content I already know.
- **As a product team member**, I want to see goal completion rates and diagnostic score distributions so that I can optimize the onboarding funnel.

## 5. Functional Requirements

- **FR-1.** The system MUST present a multi-step onboarding wizard to new SL accounts (not HE/K-12) on first login after registration, covering: topic/goal selection, timeline, daily study minutes, reminder opt-in, and optional diagnostic.
- **FR-2.** The system MUST store the learner's captured goals in a `learner_goals` table (topic, goal_text, target_date, daily_minutes, prior_knowledge_level).
- **FR-3.** The system MUST generate a "Start Here" course recommendation card on the Dashboard based on the stated topic and diagnostic result (if taken).
- **FR-4.** The system MUST allow learners to skip the diagnostic without penalty; skipping does not block catalog access.
- **FR-5.** The system MUST collect GDPR marketing consent and reminder opt-in during the onboarding flow (as per 15.10).
- **FR-6.** The system SHOULD integrate with the 1.7 diagnostic assessment infrastructure to present 5–10 calibration questions relevant to the learner's chosen topic.
- **FR-7.** The system SHOULD allow learners to update their goals at any time from their profile/settings page.

## 6. Non-Functional Requirements

- **Performance** — Each onboarding step transitions in <300 ms; diagnostic questions load in <500 ms.
- **Security** — Onboarding is shown only to the authenticated new user; goal data is not shared with other users.
- **Privacy & Compliance** — GDPR: consent collected explicitly during onboarding; reminder opt-in is a separate checkbox from terms of service; prior knowledge level is not considered special-category data.
- **Accessibility** — Multi-step wizard uses `aria-current="step"` for progress; topic selection is keyboard-navigable; WCAG 2.1 AA.
- **Scalability** — Onboarding is a lightweight read/write; goal storage is one row per user.
- **Reliability** — Onboarding state is server-persisted; user can close browser and resume where they left off.
- **Observability** — Metric `onboarding_steps_completed_total{step}`, `onboarding_completions_total`, `onboarding_skips_total`.
- **Maintainability** — Onboarding wizard component in `clients/web/src/pages/onboarding/`; step sequence configurable.
- **Internationalization** — Topic names and goal text externalised; date formatting locale-aware.
- **Backward compatibility** — Existing users with accounts skip onboarding entirely.

## 7. Acceptance Criteria

- **AC-1.** *Given* a new learner registers, *when* they log in for the first time, *then* the onboarding wizard launches automatically with a welcome step.
- **AC-2.** *Given* a learner completes the onboarding and states "I want to learn Python by July," *when* they reach the Dashboard, *then* a "Start Here: Python for Beginners" course recommendation card is visible.
- **AC-3.** *Given* a learner skips the diagnostic, *when* they complete the rest of onboarding, *then* they reach the full Dashboard without any restriction and no mandatory completion screen.
- **AC-4.** *Given* a learner completes the diagnostic, *when* their score indicates intermediate level, *then* the course recommendation links to an intermediate course, not a beginner one.
- **AC-5.** *Given* a learner is in a GDPR-applicable jurisdiction, *when* the consent step appears, *then* reminder opt-in is an unchecked checkbox separate from the mandatory ToS acceptance.

## 8. Data Model

Migration `101_learner_goals.sql`:
```sql
CREATE TABLE learner_goals (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id               UUID NOT NULL REFERENCES users(id) UNIQUE,
  topic                 TEXT NOT NULL,
  goal_text             TEXT,
  target_date           DATE,
  daily_minutes         INT NOT NULL DEFAULT 20,
  prior_knowledge_level TEXT DEFAULT 'beginner', -- 'beginner', 'intermediate', 'advanced'
  diagnostic_score      NUMERIC(5,2),
  onboarding_completed  BOOLEAN NOT NULL DEFAULT FALSE,
  created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## 9. API Surface

```
GET  /api/v1/me/onboarding-status       -- has onboarding been completed?
POST /api/v1/me/onboarding              -- save onboarding step data (incremental)
PATCH /api/v1/me/goals                  -- update goals post-onboarding
GET  /api/v1/me/goals                   -- read current goals
```

Diagnostic questions served from 1.7 diagnostic assessment API.

## 10. UI / UX

New pages under `clients/web/src/pages/onboarding/`:
- `OnboardingWelcome.tsx` — step 0: welcome and "Let's get started."
- `OnboardingGoal.tsx` — step 1: topic picker (category chips) + goal text field + target date.
- `OnboardingPriorKnowledge.tsx` — step 2: self-assessed level selector (Beginner / Some Experience / Advanced).
- `OnboardingDiagnostic.tsx` — step 3 (optional): 5-question diagnostic.
- `OnboardingHabits.tsx` — step 4: daily minutes goal + reminder time setup.
- `OnboardingConsent.tsx` — step 5: GDPR consent + reminders opt-in.
- `OnboardingComplete.tsx` — step 6: "You're all set!" + personalized course card.

Progress indicator: step dots at top with `aria-current="step"` on active step.
Back button resumes from last completed step on return.
Modified: `clients/web/src/pages/lms/Dashboard.tsx` — "Start Here" recommendation card visible post-onboarding.

Accessibility: step wizard uses `role="main"` for content area; each step has a heading; skip-to-content link at top.
Mobile: single-column layout; topic chips wrap; diagnostic questions one per screen.

## 11. AI / ML Considerations

- The diagnostic assessment uses the 1.7 diagnostic infrastructure which leverages the AI quiz generation (`server/src/services/quiz_generation_ai.rs`) for calibration questions.
- Goal text stored as a learner signal for the AI study buddy (15.12) as the initial "I want to learn X" context.
- No model inference during onboarding itself beyond the quiz generation.
- Cross-reference: 1.7 covers the AI model details; this plan covers the UX wrapper.

## 12. Integration Points

- 1.7 diagnostic/placement assessment infrastructure (diagnostic quiz step).
- 15.10 study reminders (reminder opt-in during onboarding).
- 15.12 AI study buddy (goal text as persistent context seed).
- `server/src/routes/me.rs` — goal and onboarding endpoints.
- `clients/web/src/pages/lms/Dashboard.tsx` — "Start Here" card.

## 13. Dependencies & Sequencing

- Must ship after: 1.7 (diagnostic infrastructure for step 3).
- Must ship before: 15.12 (AI study buddy consumes goal data).
- Can ship step 1–4 without 1.7 by skipping the diagnostic step.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Long onboarding reduces conversion | H | H | Make every step skippable; A/B test 3-step vs. 6-step flow |
| Diagnostic accuracy low without 1.7 IRT calibration | M | M | Fallback to self-assessment only; use stated prior_knowledge_level |
| Goal not updated → stale recommendations over time | H | M | "Update your goals?" prompt at 30-day intervals |

## 15. Rollout Plan

- Feature flag: `onboarding_flow` (default: off; SL tier).
- A/B test shorter vs. longer onboarding during beta.
- GA alongside 15.1 catalog and 15.2 enrollment.

## 16. Test Plan

- **Unit** — Onboarding completion flag logic; goal storage and retrieval; diagnostic skip handling.
- **Integration** — POST to each step; verify goal row populated; recommendation card logic.
- **End-to-end** — Playwright: new user registration → full onboarding → Dashboard shows recommendation.
- **Security** — Assert user A cannot read user B's goals.
- **Accessibility** — axe on each onboarding step; step progress indicator.
- **Manual exploratory** — QA: close browser mid-onboarding and resume, skip all optional steps, complete diagnostic.
