## 1. Problem Statement

Lextures currently exposes routes only to its own React client; there is no documented, versioned, or publicly consumable API surface. Third-party developers, institutional IT teams, and automation tooling have no supported integration path. Without a stable public API backed by an OpenAPI 3.1 spec, Lextures cannot participate in district/university ecosystems, cannot support the webhook system (16.3), and cannot attract an integration marketplace (16.9). Delivering a versioned REST API with rate limiting also unblocks OWASP API Top 10 hardening required for the security posture described in `SECURITY.md`.

## 2. Goals

- Publish a versioned (`/api/v1/`) REST API covering the core resource domains (users, courses, enrollments, modules, assignments, grades, quizzes).
- Generate and host an accurate OpenAPI 3.1 specification (Swagger UI + ReDoc) from code annotations.
- Enforce per-token and per-IP rate limits so the public API cannot be weaponised against the shared database.
- Provide pagination, filtering, sparse fieldsets, and error shapes conformant with JSON:API and RFC 7807 Problem Details.
- Lay the groundwork for an optional GraphQL endpoint over the same authorization layer.

## 3. Non-Goals

- Full GraphQL implementation (tracked separately; only schema stubs are in scope here).
- Webhook delivery infrastructure (16.3).
- LTI or IMS-specific endpoints (2.12).
- Admin-only bulk-import endpoints (18.2).

## 4. Personas & User Stories

- **As an institutional developer**, I want to query `/api/v1/courses` with my institutional API token so that I can build a custom roster dashboard without screen-scraping.
- **As an EdTech integration partner**, I want a machine-readable OpenAPI spec so that I can generate a typed SDK in Python or TypeScript in under an hour.
- **As a self-learner app developer**, I want a stable, versioned API so that my mobile app does not break when Lextures ships internal refactors.
- **As a security engineer**, I want rate limits enforced at the API gateway layer so that credential-stuffing attacks on `/api/v1/auth/login` are throttled automatically.
- **As an instructor**, I want my LMS data accessible via API so that I can pipe grades into my own spreadsheet tooling.

## 5. Functional Requirements

- **FR-1.** The system MUST expose all major resource collections under the `/api/v1/` path prefix, with path-based versioning.
- **FR-2.** The system MUST generate an OpenAPI 3.1 document at `/api/v1/openapi.json` derived from code-level annotations (using `utoipa` in Rust), kept in sync by CI.
- **FR-3.** The system MUST serve a Swagger UI at `/api/v1/docs` and a ReDoc UI at `/api/v1/redoc` behind an optional admin feature flag.
- **FR-4.** The system MUST return error responses conformant with RFC 7807 Problem Details (`Content-Type: application/problem+json`) for all 4xx/5xx responses.
- **FR-5.** The system MUST support cursor-based pagination (`?cursor=` + `?limit=`) for collection endpoints, with `Link` header navigation.
- **FR-6.** The system MUST enforce per-token rate limits (see 17.6) and return `429 Too Many Requests` with a `Retry-After` header.
- **FR-7.** The system SHOULD support sparse fieldsets (`?fields[resource]=field1,field2`) on collection endpoints to reduce payload size.
- **FR-8.** The system SHOULD emit a GraphQL schema stub at `/api/v1/graphql` (query-only, no mutations) once the REST layer is stable.
- **FR-9.** The system MAY support JSONP or CORS pre-flight allowlisting for approved partner origins via the tenant settings table.

## 6. Non-Functional Requirements

- **Performance** — p95 latency ≤ 80 ms for single-resource reads under normal load; collection endpoints ≤ 200 ms for pages of 100 items; OpenAPI JSON generation cached, not computed per-request.
- **Security** — All public API routes MUST require a valid Bearer token (RFC 6750); unauthenticated requests return 401. OWASP API Top 10 reviewed: mass assignment prevention, object-level auth checks per request, no internal IDs leaked without authorization. TLS 1.2+ enforced at load-balancer level.
- **Privacy & Compliance** — API responses MUST NOT return PII fields (email, full name) to tokens lacking the `pii:read` scope. Audit log entry written for every authenticated API call (user_id, token_id, method, path, status, latency_ms).
- **Accessibility** — Swagger UI must be keyboard-navigable; no WCAG blocker introduced in `/api/v1/docs`.
- **Scalability** — API layer must be stateless; rate-limit counters stored in Redis (17.2) so counters survive app restarts and scale horizontally.
- **Reliability** — API availability target 99.9%; circuit-breaker for downstream DB calls; graceful degradation returning 503 with `Retry-After` on DB saturation.
- **Observability** — Prometheus counter `api_requests_total{method,path,status}` and histogram `api_request_duration_seconds`; OTel trace per request with `trace-id` in response headers.
- **Maintainability** — `utoipa` annotations co-located with handler functions in `server/src/routes/`; OpenAPI spec diffed in CI to prevent silent breaking changes.
- **Internationalization** — Error message strings keyed, translateable; `Accept-Language` header respected for error messages.
- **Backward compatibility** — Breaking changes require a new `/api/v2/` prefix; `/api/v1/` supported for minimum 12 months after v2 GA.

## 7. Acceptance Criteria

- **AC-1.** *Given* a valid institutional token with `courses:read` scope, *When* `GET /api/v1/courses` is called, *Then* the response is HTTP 200 with a JSON array of course objects and a `Link` header for pagination.
- **AC-2.** *Given* no Authorization header, *When* `GET /api/v1/courses` is called, *Then* the response is HTTP 401 `application/problem+json` with `type` URI and `title` field.
- **AC-3.** *Given* a token that has exceeded its rate-limit quota, *When* any API request is made, *Then* the response is HTTP 429 with a `Retry-After` header indicating seconds until reset.
- **AC-4.** *Given* the server is running, *When* `GET /api/v1/openapi.json` is fetched, *Then* the response validates against the OpenAPI 3.1 JSON Schema with zero errors.
- **AC-5.** *Given* a valid token lacking `pii:read` scope, *When* `GET /api/v1/users/{id}` is called, *Then* the email field is absent from the response body.
- **AC-6.** *Given* CI runs on a pull request that changes a route handler, *When* the generated OpenAPI spec differs from the committed `openapi.json`, *Then* the CI job fails with a diff showing the change.
- **AC-7.** *Given* Swagger UI is loaded at `/api/v1/docs`, *When* a keyboard user navigates with Tab, *Then* all interactive elements are reachable and focusable.

## 8. Data Model

No new tables required for the API layer itself; the token model is defined in 16.2. An `api_request_log` table MAY be added for compliance:

```sql
-- server/migrations/073_api_request_log.sql
CREATE TABLE api.request_log (
    id          BIGSERIAL PRIMARY KEY,
    token_id    UUID REFERENCES auth.api_tokens(id) ON DELETE SET NULL,
    user_id     UUID REFERENCES user.users(id) ON DELETE SET NULL,
    method      TEXT NOT NULL,
    path        TEXT NOT NULL,
    status      SMALLINT NOT NULL,
    latency_ms  INTEGER NOT NULL,
    ip_hash     TEXT,          -- hashed; no raw IP stored (GDPR)
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON api.request_log (token_id, created_at DESC);
CREATE INDEX ON api.request_log (created_at DESC);
```

Retention policy: 90-day rolling delete via scheduled job (17.4).

## 9. API Surface

All routes prefixed `/api/v1/`. Auth: `Authorization: Bearer <token>`.

| Method | Path | Scope | Notes |
|--------|------|-------|-------|
| GET | `/api/v1/courses` | `courses:read` | Paginated |
| GET | `/api/v1/courses/{id}` | `courses:read` | |
| GET | `/api/v1/courses/{id}/enrollments` | `enrollments:read` | |
| GET | `/api/v1/users/{id}` | `users:read` | PII gated on `pii:read` |
| GET | `/api/v1/assignments` | `assignments:read` | |
| GET | `/api/v1/grades` | `grades:read` | |
| GET | `/api/v1/openapi.json` | public | OpenAPI 3.1 spec |
| GET | `/api/v1/docs` | public | Swagger UI |
| GET | `/api/v1/graphql` | `graphql:read` | Query-only stub |

Pagination shape:
```jsonc
{
  "data": [...],
  "meta": { "total": 450, "cursor": "eyJpZCI6IjEyMyJ9" },
  "links": { "next": "/api/v1/courses?cursor=eyJ...&limit=25" }
}
```

Error shape (RFC 7807):
```jsonc
{
  "type": "https://lextures.io/errors/rate-limited",
  "title": "Too Many Requests",
  "status": 429,
  "detail": "Token quota of 1000 req/min exceeded.",
  "instance": "/api/v1/courses"
}
```

## 12. Integration Points

- **Internal:** `server/src/routes/` — new `api/` sub-module added alongside existing routes; `server/src/authz.rs` — scope-check middleware reused.
- **Internal:** `server/src/services/mail.rs` — no direct dependency, but job-queue notifications for rate-limit alerts use mail service.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| OpenAPI spec drifts from implementation | M | H | CI diff check; `cargo test` includes spec snapshot test |
| Rate limits misconfigured, blocking legitimate batch clients | M | M | Per-token override capability; burst allowance in config |
| GraphQL N+1 queries overload DB | M | H | DataLoader pattern; query depth limit enforced at parse time |
| API versioning burden as schema evolves | L | H | Strict semver policy; deprecation period 12 months minimum |

## 15. Rollout Plan

- **Feature flag:** `public_api_enabled` (OpenFeature, default `false`).
- **Phase 1:** Deploy `/api/v1/` routes internally; generate OpenAPI spec; run OWASP checks.
- **Phase 2:** Enable for pilot institutional partners; gather feedback on scope granularity.
- **Phase 3:** GA; publish OpenAPI spec to developer portal; enable Swagger UI.
- **Rollback:** Feature flag flip; v1 routes remain in code but return 503 if flag off.

## 16. Test Plan

- **Unit** — scope-check middleware, pagination cursor encode/decode, RFC 7807 error serializer.
- **Integration** — each route: auth happy path, missing-scope 403, malformed input 422, rate-limit 429.
- **End-to-end** — Playwright: fetch OpenAPI spec, call a course list endpoint with a test token, verify pagination links.
- **Security** — OWASP API Top 10 checklist; fuzz with `cargo-fuzz` on request deserialization.
- **Accessibility** — axe-core run against Swagger UI; keyboard navigation smoke test.
- **Performance / load** — k6 script: 500 VUs sustained 5 min against `/api/v1/courses`; p95 ≤ 80 ms pass criterion.
- **Manual exploratory** — Red-team session: attempt BOLA (Broken Object Level Auth) on cross-tenant course IDs.

## 18. Open Questions

1. Should GraphQL be a first-class parallel track, or a later phase after REST stabilizes?
2. Do we support API key rotation with zero-downtime overlap period, and if so, for how long?
3. Is `/api/v1/graphql` restricted to read-only forever, or will mutations ship in v2?
4. What is the right default rate-limit burst / sustained window (e.g., 100/min per token vs. 1000/min)?
