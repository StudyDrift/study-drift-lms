## 1. Problem Statement

Students and instructors have no way to subscribe to Lextures due dates, assignment deadlines, and quiz windows in their personal calendar applications (Google Calendar, Apple Calendar, Outlook). This forces manual cross-referencing between the LMS and calendar tools, increasing missed-deadline rates and reducing learner engagement. An iCalendar feed export is a low-complexity, high-impact feature that every competing LMS provides and that Lextures currently lacks entirely.

## 2. Goals

- Produce a per-user iCalendar feed (RFC 5545) at a stable, authenticated URL containing all upcoming assignments, quizzes, and course events visible to that user.
- Support CalDAV discovery (RFC 4791) so that CalDAV-capable clients (Apple Calendar, Thunderbird) can subscribe natively.
- Allow per-course feed subscriptions as well as an all-courses aggregate feed.
- Ensure the feed respects course visibility and enrollment authorization at generation time.

## 3. Non-Goals

- Two-way sync / writing events back from an external calendar into Lextures.
- A built-in calendar UI within Lextures (a separate feature).
- Meeting/virtual classroom scheduling (6.7).
- Group or org-level public calendar feeds (institutional calendar integration is a later phase).

## 4. Personas & User Stories

- **As a student**, I want to subscribe to a Lextures calendar feed in Google Calendar so that my assignment due dates appear alongside my personal events automatically.
- **As an instructor**, I want a stable calendar URL for my course so that I can share it with students on the syllabus without them needing to log in to generate a link.
- **As a self-learner**, I want all my enrolled course deadlines in one iCal feed so that I can see my learning schedule in my calendar app of choice.
- **As an IT administrator**, I want CalDAV-compatible feed URLs so that our school's calendar infrastructure can subscribe and surface Lextures events to students through a centralized calendar.

## 5. Functional Requirements

- **FR-1.** The system MUST generate a per-user iCalendar feed (RFC 5545 `text/calendar` format) at `/api/v1/me/calendar.ics?token=<token>` covering all courses the user is enrolled in.
- **FR-2.** The system MUST generate a per-course iCalendar feed at `/api/v1/courses/{id}/calendar.ics?token=<token>` for enrolled users.
- **FR-3.** Each VEVENT MUST include: `SUMMARY` (item title), `DTSTART` / `DTEND` (due date window), `DESCRIPTION` (truncated assignment description), `URL` (deep link into Lextures), `UID` (stable `{item_id}@lextures.io`), `DTSTAMP`.
- **FR-4.** Feed URLs MUST be authenticated via a signed token query parameter (derived from the user's API token, scope `calendar:read`) so that no session cookie is required for calendar client polling.
- **FR-5.** The system MUST support CalDAV discovery: `/.well-known/caldav` MUST redirect to the per-user calendar collection endpoint, and `PROPFIND` on the collection MUST return the feed URL for RFC 4791 compliant clients.
- **FR-6.** The system SHOULD include quiz `available_from` / `available_until` windows as VEVENT `DTSTART` / `DTEND` with a `DESCRIPTION` noting "Quiz window."
- **FR-7.** The system SHOULD allow users to regenerate their calendar token (invalidating the old URL) from the Settings UI.
- **FR-8.** The system MAY filter events by date range via `?start=` and `?end=` query parameters to reduce feed size for clients that request arbitrary ranges.

## 6. Non-Functional Requirements

- **Performance** — Feed generation p95 latency ≤ 200 ms for a user with 10 active courses; feeds cached in Redis with 15-minute TTL, invalidated on assignment/quiz update.
- **Security** — Token embedded in URL is a short-lived (30-day) HMAC-signed capability token, not a full API token, to limit blast radius if URL is shared. Token scoped to `calendar:read` only.
- **Privacy & Compliance** — Feed MUST NOT include student-identifying information in shared course feed URLs. FERPA: per-user feeds are personal and should not be accessible without the user's token.
- **Accessibility** — Feed subscription help page must be WCAG 2.1 AA; step-by-step instructions for Google Calendar, Apple Calendar, Outlook each provided.
- **Scalability** — Feed generation is read-only; stateless; horizontally scalable. Redis cache prevents repeated DB queries from polling calendar clients.
- **Reliability** — If a calendar client polls while the DB is unavailable, return a cached response (possibly stale by ≤15 min) rather than an error; 503 with `Retry-After` if cache also unavailable.
- **Observability** — Counter `calendar_feed_requests_total{type=user|course}`; alert on cache miss rate > 50% sustained (indicates cache invalidation issue).
- **Maintainability** — iCal serializer isolated in `server/src/services/calendar.rs`; reused for both feed types.
- **Internationalization** — `VTIMEZONE` component generated per user's timezone preference; `SUMMARY` uses course language setting.
- **Backward compatibility** — `UID` must be stable: changing assignment titles does not change the UID. Clients rely on UID for event deduplication.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student enrolled in two courses with upcoming assignments, *When* they subscribe to their personal `.ics` feed URL in Google Calendar, *Then* all assignments appear as calendar events with correct due dates and deep-link URLs.
- **AC-2.** *Given* a calendar feed URL with a valid token, *When* it is fetched without a session cookie, *Then* the response is HTTP 200 `text/calendar` with valid RFC 5545 content.
- **AC-3.** *Given* a student regenerates their calendar token, *When* the old URL is fetched, *Then* the response is HTTP 401.
- **AC-4.** *Given* an assignment due date changes, *When* the calendar client polls within 15 minutes, *Then* the updated `DTSTART`/`DTEND` is reflected in the feed.
- **AC-5.** *Given* an Apple Calendar client issues a `PROPFIND` to `/.well-known/caldav`, *When* the response is processed, *Then* the client can discover the user's calendar collection URL without manual URL entry.
- **AC-6.** *Given* the feed is requested with `?start=2026-04-01&end=2026-06-30`, *When* the response is returned, *Then* only events with `DTSTART` within that range are included.

## 8. Data Model

No new tables required. A `calendar_tokens` table is needed for the capability token:

```sql
-- server/migrations/076_calendar_tokens.sql
CREATE TABLE auth.calendar_tokens (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES user.users(id) ON DELETE CASCADE,
    token_hash  TEXT NOT NULL UNIQUE,
    expires_at  TIMESTAMPTZ NOT NULL DEFAULT now() + INTERVAL '30 days',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON auth.calendar_tokens (user_id) WHERE expires_at > now();
```

Token auto-rotated on each visit to Settings → Calendar; old token immediately invalidated.

## 9. API Surface

| Method | Path | Auth | Notes |
|--------|------|------|-------|
| GET | `/api/v1/me/calendar.ics` | `?token=<cap_token>` | Personal iCal feed |
| GET | `/api/v1/courses/{id}/calendar.ics` | `?token=<cap_token>` | Course iCal feed |
| POST | `/api/v1/me/calendar-token` | JWT session | Generate/rotate token |
| GET | `/.well-known/caldav` | — | CalDAV discovery redirect |
| PROPFIND | `/caldav/users/{user_id}/` | `?token=` | CalDAV collection |

Response `Content-Type: text/calendar; charset=utf-8`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Calendar URL shared publicly, exposing assignment metadata | M | M | Document risk; short token expiry (30 days); no PII in feed |
| Timezone handling errors causing wrong due dates | M | M | Use VTIMEZONE component; test with UTC-12 to UTC+14 range |
| CalDAV PROPFIND not supported by all clients | L | L | Fallback direct URL always works; CalDAV is bonus |
| Feed becomes stale during DB maintenance | L | L | Serve Redis-cached version with `X-Cache: stale` header |

## 15. Rollout Plan

- **Feature flag:** `calendar_feeds_enabled` (OpenFeature, default `false`).
- **Phase 1:** Personal and per-course iCal feeds; Settings UI; internal testing.
- **Phase 2:** CalDAV discovery; beta with enrolled users.
- **Phase 3:** GA; help center article published.
- **Rollback:** Disable feature flag; tokens remain in DB but feed endpoints return 404.

## 16. Test Plan

- **Unit** — RFC 5545 serializer: VEVENT output, VTIMEZONE, UID stability, date range filter.
- **Integration** — Token generation/validation; feed generation with mock DB data; cache invalidation on assignment update.
- **End-to-end** — Playwright: visit Settings; copy feed URL; fetch URL directly; validate iCal content.
- **Security** — Expired token returns 401; token for user A cannot access user B's feed; CalDAV endpoint unauthenticated access denied.
- **Accessibility** — axe-core on Settings → Calendar page; keyboard navigation.
- **Performance / load** — k6: 1,000 simultaneous feed fetches; p95 ≤ 200 ms; cache hit rate > 80%.
- **Manual exploratory** — QA: subscribe in Google Calendar; verify event appears; change due date; verify feed updates within 15 min.

## 17. Documentation & Training

- Help center: "Subscribing to your Lextures calendar in Google Calendar / Apple Calendar / Outlook."
- Admin guide: "CalDAV discovery setup."
- Internal runbook: token rotation procedure; cache invalidation troubleshooting.

## 18. Open Questions

1. Should course feeds be accessible to instructors without enrolling (they own the course)?
2. Do we include announcement events or only assignment/quiz due dates?
3. Should feed URLs use token query param (simpler) or HTTP Basic Auth (more standard for CalDAV)?
4. Is a 30-day token expiry the right balance between convenience and security?
