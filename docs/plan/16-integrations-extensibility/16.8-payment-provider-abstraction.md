## 1. Problem Statement

Lextures has no payment infrastructure. Self-learner monetization — paid courses, subscription plans, freemium tiers — is entirely absent. Without a payment layer, Lextures cannot charge for access to courses or generate revenue from the self-learner segment, which represents the closest-to-market monetization opportunity. A provider abstraction (rather than hard-coding Stripe) ensures the platform can serve European markets (iDEAL, SEPA) and American users who prefer PayPal without re-architecting each time a new payment method is needed.

## 2. Goals

- Implement a `PaymentProvider` trait with Stripe as the first production backend.
- Support one-time course purchases and recurring subscription plans.
- Integrate PayPal as a second backend for markets where PayPal is preferred.
- Support iDEAL (Netherlands/EU) via Stripe's iDEAL payment method, not as a separate provider, but ensuring the abstraction allows region-specific method surfacing.
- Provide a checkout flow, purchase history, and refund workflow through the UI.
- Emit webhook events from payment providers through the background job queue (17.3) for idempotent fulfillment.

## 3. Non-Goals

- Cryptocurrency payments.
- B2B invoicing and purchase orders for institutional customers (separate billing track).
- In-app marketplace revenue split with plugin developers (16.9 — depends on this).
- Tax computation and remittance (integrate with Stripe Tax as phase 2).
- Building a custom payment form — PCI compliance requires hosted forms (Stripe Elements / PayPal Buttons).

## 4. Personas & User Stories

- **As a self-learner**, I want to purchase a course with my credit card or PayPal so that I can start learning immediately after payment.
- **As a course creator**, I want to set a price for my course and receive payouts so that I am compensated for my content.
- **As a European learner**, I want to pay with iDEAL (my bank's direct payment method) so that I do not need a credit card.
- **As an admin**, I want to see all transactions in a dashboard and issue refunds from within Lextures so that I do not have to open the Stripe dashboard for routine support requests.
- **As a developer**, I want a single `PaymentProvider` interface so that supporting a new payment method requires only a new backend implementation.

## 5. Functional Requirements

- **FR-1.** The system MUST define a `PaymentProvider` Rust trait covering: `create_checkout_session`, `create_subscription`, `cancel_subscription`, `issue_refund`, `get_transaction`.
- **FR-2.** The system MUST implement the trait for Stripe (Checkout Sessions API + Billing for subscriptions) and PayPal (Orders API).
- **FR-3.** The system MUST provide a hosted checkout flow: the learner is redirected to Stripe Checkout or PayPal, completes payment, and is redirected back to Lextures.
- **FR-4.** The system MUST handle provider webhooks (Stripe `checkout.session.completed`, `invoice.payment_failed`; PayPal `PAYMENT.CAPTURE.COMPLETED`) asynchronously via the background job queue (17.3) with idempotency by event ID.
- **FR-5.** The system MUST unlock course access immediately upon successful payment confirmation (webhook received + job processed).
- **FR-6.** The system MUST record every transaction in a `payments.transactions` table with status, amount, currency, provider, and idempotency key.
- **FR-7.** The system MUST support subscription plans: monthly and annual billing cycles per course or bundle.
- **FR-8.** The system SHOULD support iDEAL as a Stripe payment method (surfaced automatically for NL/EU locale).
- **FR-9.** The system SHOULD support coupon codes / discount codes applied at checkout, managed through the Stripe Coupon API.
- **FR-10.** The system MAY support partial refunds from the Lextures admin panel.

## 6. Non-Functional Requirements

- **Performance** — Checkout session creation ≤ 500 ms; webhook job processing ≤ 2 s after receipt.
- **Security** — Provider API keys stored in secrets manager (17.17); never logged. Webhook signatures verified (Stripe: `Stripe-Signature` header; PayPal: OAuth token validation) before any job enqueue. PCI DSS: Lextures never handles raw card data — all payment data captured by hosted provider forms.
- **Privacy & Compliance** — Transaction data subject to financial record retention (7 years in many jurisdictions). GDPR: customer data sent to Stripe is covered by Stripe's DPA (EU Standard Contractual Clauses). Refund requests must be audited.
- **Accessibility** — Checkout pages are provider-hosted; ensure redirect flow is accessible. Lextures purchase confirmation page must be WCAG 2.1 AA.
- **Scalability** — Webhook endpoint must handle bursts (e.g., marketing campaign triggers 10,000 purchases in minutes); backed by job queue with horizontal workers.
- **Reliability** — Idempotent fulfillment: processing the same `checkout.session.completed` event twice must not grant duplicate access or create duplicate transaction records.
- **Observability** — Counter `payment_transactions_total{provider,status,currency}`; alert on `payment_failed` spike; latency histogram for webhook processing.
- **Maintainability** — New payment provider: implement trait + webhook handler + one migration entry. No changes to checkout UI.
- **Internationalization** — Currency display locale-aware; Stripe Checkout natively internationalizes. Lextures receipt emails (via mail service 6.2) use user locale.
- **Backward compatibility** — Free courses unaffected; payment check is an additive gate on enrollment.

## 7. Acceptance Criteria

- **AC-1.** *Given* a paid course with a Stripe price configured, *When* a learner completes Stripe Checkout, *Then* within 5 seconds of the webhook being received, the learner's enrollment is active and they can access course content.
- **AC-2.** *Given* the same Stripe `checkout.session.completed` event is received twice (duplicate delivery), *When* both are processed, *Then* exactly one enrollment record and one transaction record exist.
- **AC-3.** *Given* a subscription learner's payment fails, *When* the `invoice.payment_failed` webhook is processed, *Then* the enrollment is suspended and the learner receives an email notification via the mail service.
- **AC-4.** *Given* a learner in the Netherlands, *When* the checkout page loads, *Then* iDEAL appears as a payment option alongside credit card.
- **AC-5.** *Given* an admin issues a refund from the admin console, *When* the refund is confirmed, *Then* the Stripe refund API is called, the transaction status is updated to `refunded`, and an audit log entry is written.
- **AC-6.** *Given* a webhook arrives with an invalid Stripe signature, *When* the endpoint processes it, *Then* the request is rejected with HTTP 400 and no job is enqueued.

## 8. Data Model

```sql
-- server/migrations/079_payments.sql
CREATE TABLE payments.transactions (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             UUID NOT NULL REFERENCES user.users(id),
    course_id           UUID REFERENCES course.courses(id),
    provider            TEXT NOT NULL,          -- 'stripe', 'paypal'
    provider_txn_id     TEXT NOT NULL UNIQUE,   -- Stripe session ID / PayPal order ID
    idempotency_key     TEXT NOT NULL UNIQUE,
    amount_cents        INTEGER NOT NULL,
    currency            CHAR(3) NOT NULL,
    status              TEXT NOT NULL,          -- pending|completed|failed|refunded
    subscription_id     TEXT,                   -- Stripe subscription ID if recurring
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE payments.subscriptions (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             UUID NOT NULL REFERENCES user.users(id),
    provider_sub_id     TEXT NOT NULL UNIQUE,
    plan_id             TEXT NOT NULL,
    status              TEXT NOT NULL,   -- active|past_due|canceled
    current_period_end  TIMESTAMPTZ,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON payments.transactions (user_id, created_at DESC);
```

## 9. API Surface

| Method | Path | Auth | Notes |
|--------|------|------|-------|
| POST | `/api/v1/checkout` | JWT session | Create checkout session |
| GET | `/api/v1/me/transactions` | JWT session | Purchase history |
| POST | `/webhooks/stripe` | Stripe signature | Stripe event handler |
| POST | `/webhooks/paypal` | PayPal OAuth | PayPal event handler |
| POST | `/admin/transactions/{id}/refund` | admin JWT | Issue refund |

## 10. UI / UX

- **Course detail page:** "Enroll / Purchase" button; shows price, billing cycle (one-time / monthly / annual).
- **Checkout redirect:** brief loading screen → provider-hosted checkout → return to "Purchase complete" page.
- **Settings → Purchases:** transaction history table with date, course, amount, status, receipt download link.
- **Admin → Payments:** transaction table with filter, search, refund button.
- **Receipt email:** sent via `server/src/services/mail.rs` on webhook fulfillment.
- Mobile: checkout flow on mobile browser uses provider's mobile-optimized checkout pages.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Duplicate enrollment on webhook retry | H | H | Idempotency key unique constraint in DB |
| Stripe API breaking change | L | M | Pin Stripe SDK version; monitor Stripe changelog |
| Payment provider downtime blocks enrollment | M | H | Async webhook fulfillment; fallback: manual enrollment by admin |
| PCI scope creep if card data touched | L | H | Use Stripe-hosted checkout only; no card numbers in Lextures backend |

## 15. Rollout Plan

- **Feature flag:** `payments_enabled` (OpenFeature, default `false`).
- **Phase 1:** Stripe checkout + one-time purchase; test mode only; internal QA.
- **Phase 2:** Subscriptions; coupons; beta with self-learner pilot.
- **Phase 3:** PayPal backend; iDEAL method; GA.
- **Rollback:** Disable flag; existing enrollments and transactions unaffected.

## 16. Test Plan

- **Unit** — Trait mock; idempotency key generation; webhook signature verification.
- **Integration** — Stripe test mode: full checkout → webhook → enrollment pipeline; refund flow; duplicate webhook idempotency.
- **End-to-end** — Playwright + Stripe test cards: complete checkout; verify enrollment unlocked; verify receipt email.
- **Security** — Invalid webhook signature rejected; provider key not in logs; PCI: no card data logged.
- **Accessibility** — Checkout redirect page; purchase history page; axe-core.
