## 1. Problem Statement

Lextures has no mechanism for users or institutions to generate long-lived API credentials. Every automated integration must re-authenticate as a human user, with a 72-hour JWT that cannot be scoped or audited separately. This blocks developers building against the public API (16.1), webhook delivery signing (16.3), and all institutional automation. A token model with fine-grained scopes is also the pre-requisite for the Zapier/Make connector (16.10) and marketplace apps (16.9).

## 2. Goals

- Provide personal API tokens (tied to a single user account) and institutional service tokens (tied to an org, not a person).
- Implement an explicit scope system so tokens carry least-privilege access.
- Offer a UI for creating, labeling, rotating, and revoking tokens.
- Store token secrets using Argon2id (already in use) or HMAC-SHA256 prefix + hash, with only a hashed value persisted.
- Emit audit log events for every token creation, use, and revocation.

## 3. Non-Goals

- OAuth 2.0 authorization-code flow for third-party apps (16.9 / marketplace).
- LTI 1.3 launch credentials (2.12).
- SAML/OIDC session tokens (4.1–4.2).
- Token-based MFA (4.6).

## 4. Personas & User Stories

- **As an instructor**, I want to generate a personal API token so that I can pull my gradebook into a spreadsheet macro without sharing my password.
- **As an institutional IT administrator**, I want to create a service token scoped to `enrollments:read` so that our SIS can sync rosters without human credentials.
- **As a developer building a Zapier integration**, I want to see the list of available scopes in the token UI so that I can select the minimum permissions needed.
- **As a security officer**, I want every token to display its last-used timestamp so that I can identify and revoke stale tokens during audits.
- **As an admin**, I want to revoke any token in the organization from the admin console, not just my own.

## 5. Functional Requirements

- **FR-1.** The system MUST allow any authenticated user to create one or more personal API tokens with a user-specified label and expiry date (or no-expiry option).
- **FR-2.** The system MUST allow org admins to create institutional service tokens not tied to a specific user account, with an associated service-account name.
- **FR-3.** Each token MUST carry one or more explicit scopes selected from a fixed enumeration (e.g., `courses:read`, `grades:read`, `grades:write`, `enrollments:read`, `pii:read`, `webhooks:manage`).
- **FR-4.** The system MUST display the full token value exactly once at creation time and MUST NOT store the plaintext; a SHA-256 hash of the token is stored for comparison.
- **FR-5.** The system MUST support token rotation: generating a new token while keeping the old one valid for a configurable overlap window (default: 24 h).
- **FR-6.** The system MUST record `last_used_at` and `last_used_ip_hash` on each token use, updated asynchronously via a background job to avoid per-request write contention.
- **FR-7.** The system MUST allow users to revoke their own tokens; org admins MUST be able to revoke any token in their organization.
- **FR-8.** The system SHOULD warn users in the UI when a token has not been used for 90+ days, suggesting revocation.
- **FR-9.** The system MAY enforce a maximum of 20 personal tokens per user and 50 service tokens per organization.

## 6. Non-Functional Requirements

- **Performance** — Token validation MUST be O(1) lookup: prefix lookup (first 8 chars) indexes into Redis cache (17.2); cache miss falls back to DB. p99 validation latency ≤ 5 ms.
- **Security** — Token format: `ltk_` prefix + 32 random bytes (URL-safe base64) = ~47-char secret. Only SHA-256 hash stored in DB. Tokens transmitted over TLS only. Scope checks enforced in middleware before handler execution, not inside handlers.
- **Privacy & Compliance** — IP addresses stored as HMAC-SHA256 keyed hashes (GDPR: no reversible IP logging). Token labels and service-account names subject to FERPA-equivalent access rules.
- **Accessibility** — Token creation modal MUST be fully keyboard operable; generated token shown in a clearly labeled, copyable text area with ARIA live region announcing "Token generated — copy now".
- **Scalability** — Redis token cache with 15-minute TTL; invalidation on revocation via pub/sub; supports horizontal API replicas without sticky sessions.
- **Reliability** — Token revocation MUST propagate to all API nodes within 30 seconds via Redis pub/sub invalidation. DB is source of truth; cache is advisory.
- **Observability** — Counter `api_token_validations_total{result=hit|miss|invalid}`; alert on spike in `invalid` (potential credential-stuffing). Audit log event written for create/rotate/revoke.
- **Maintainability** — Scope enum defined in a single `server/src/api/scopes.rs` file; adding a new scope requires one enum variant + migration entry, not scattered string literals.
- **Internationalization** — Scope display names and descriptions stored in i18n keys.
- **Backward compatibility** — Existing JWT sessions unaffected; API tokens are a parallel auth path in Axum middleware.

## 7. Acceptance Criteria

- **AC-1.** *Given* an authenticated user, *When* they POST to `/api/v1/me/tokens` with a label and `["courses:read"]` scope, *Then* a 201 response includes the full token string once and the token ID, and a second GET to `/api/v1/me/tokens` shows the token with masked value.
- **AC-2.** *Given* a token with only `courses:read`, *When* it is used to call `GET /api/v1/grades`, *Then* the response is HTTP 403 with a Problem Details body citing missing scope.
- **AC-3.** *Given* a revoked token, *When* it is used within 30 seconds of revocation, *Then* all API nodes return HTTP 401 (cache invalidated via Redis pub/sub).
- **AC-4.** *Given* a token that has not been used for 91 days, *When* the owner views their token list, *Then* a warning badge "Unused 91 days — consider revoking" is displayed.
- **AC-5.** *Given* an institutional service token, *When* an org admin revokes it via the admin console, *Then* the token is immediately invalidated and an audit log entry records the admin's user_id, the token_id, and the timestamp.
- **AC-6.** *Given* a token creation event, *When* the audit log is queried, *Then* a record exists with `event_type=api_token_created`, `actor_id`, `token_id`, and `scopes[]`.

## 8. Data Model

```sql
-- server/migrations/073_api_tokens.sql
CREATE TABLE auth.api_tokens (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    owner_user_id   UUID REFERENCES user.users(id) ON DELETE CASCADE,
    org_id          UUID,                         -- for service tokens
    label           TEXT NOT NULL,
    token_hash      TEXT NOT NULL UNIQUE,         -- SHA-256 hex
    token_prefix    CHAR(8) NOT NULL,             -- first 8 chars for fast lookup
    scopes          TEXT[] NOT NULL DEFAULT '{}',
    expires_at      TIMESTAMPTZ,
    last_used_at    TIMESTAMPTZ,
    last_used_ip_hash TEXT,
    revoked_at      TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT chk_owner CHECK (
        (owner_user_id IS NOT NULL AND org_id IS NULL) OR
        (owner_user_id IS NULL AND org_id IS NOT NULL)
    )
);
CREATE INDEX ON auth.api_tokens (token_prefix) WHERE revoked_at IS NULL;
CREATE INDEX ON auth.api_tokens (owner_user_id) WHERE revoked_at IS NULL;
CREATE INDEX ON auth.api_tokens (org_id) WHERE revoked_at IS NULL;
```

No backfill required; existing JWT sessions remain unchanged.

## 9. API Surface

| Method | Path | Auth | Notes |
|--------|------|------|-------|
| GET | `/api/v1/me/tokens` | JWT session | List personal tokens (masked) |
| POST | `/api/v1/me/tokens` | JWT session | Create personal token |
| DELETE | `/api/v1/me/tokens/{id}` | JWT session | Revoke personal token |
| POST | `/api/v1/me/tokens/{id}/rotate` | JWT session | Rotate with overlap |
| GET | `/admin/tokens` | admin JWT | List all org tokens |
| DELETE | `/admin/tokens/{id}` | admin JWT | Admin revoke any token |

Create request body:
```jsonc
{
  "label": "Grade export macro",
  "scopes": ["courses:read", "grades:read"],
  "expires_at": "2027-01-01T00:00:00Z"  // optional
}
```

Create response (201):
```jsonc
{
  "id": "uuid",
  "token": "ltk_abc123...",   // shown ONCE
  "label": "Grade export macro",
  "scopes": ["courses:read", "grades:read"],
  "expires_at": "2027-01-01T00:00:00Z",
  "created_at": "2026-04-17T12:00:00Z"
}
```

## 10. UI / UX

- **Settings → Integrations → API Tokens** page in `clients/web/src/pages/settings/`.
- Token list shows: label, scopes, created date, last used date, expiry, revoke button.
- Token creation: modal with label input, multi-select scope picker (grouped by resource), optional expiry date picker.
- After creation: dismissible alert with monospace token display, copy button, and explicit "You will not see this again" warning.
- Rotation flow: confirmation dialog showing overlap window; new token display same as creation.
- Mobile: full-width modal, touch-friendly scope toggle chips.
- Accessibility: modal focus trap; scope list navigable with arrow keys; live region announces token generation.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Token secret exposed in logs | M | H | Ensure `token` field excluded from tracing spans; PII log scrubber (10.14) |
| Cache inconsistency after revocation | L | H | Redis pub/sub invalidation + 30 s max lag SLA |
| Scope creep: too many granular scopes confuse users | M | M | Start with ~12 coarse scopes; refine based on partner feedback |
| Service tokens shared across teams, no rotation | M | M | Enforce max age policy; automated expiry reminders via job queue |

## 15. Rollout Plan

- **Feature flag:** `api_tokens_enabled` (OpenFeature, default `false`).
- **Phase 1:** Schema migration, backend handlers, admin token management. Internal testing only.
- **Phase 2:** Enable personal tokens for beta users; gather scope enumeration feedback.
- **Phase 3:** Enable institutional service tokens; GA announcement.
- **Rollback:** Flag off; no destructive migration; existing JWTs unaffected.

## 16. Test Plan

- **Unit** — SHA-256 hashing, prefix extraction, scope validation, cache key derivation.
- **Integration** — Create token → use token → revoke token → verify 401; scope rejection; admin revocation propagation via Redis.
- **End-to-end** — Playwright: navigate to Settings → Tokens; create token; copy value; use in API call; verify success; revoke; verify next call fails.
- **Security** — Verify token not visible in response after creation; attempt to use expired token; attempt cross-user token use; brute-force prefix with no rate limit bypass.
- **Accessibility** — axe-core on token management page; keyboard navigation test; screen reader announcement of "Token generated".
- **Performance / load** — k6: 10,000 token validations/sec against Redis cache; measure p99 < 5 ms.
- **Manual exploratory** — QA checklist: token rotation overlap, admin view of all org tokens, 90-day unused warning.

## 18. Open Questions

1. Should personal tokens inherit the owner's permission level, or be independently scoped below the owner's roles?
2. What is the right expiry maximum for institutional service tokens — 1 year? 2 years? No maximum?
3. Should we support token scope delegation (child token with subset of parent scopes) as a v2 feature?
4. Is a "last 10 uses" audit trail surfaced in the UI, or only accessible to admins?
