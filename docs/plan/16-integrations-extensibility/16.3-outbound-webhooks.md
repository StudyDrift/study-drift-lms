## 1. Problem Statement

Lextures has no mechanism to notify external systems when significant events occur (grade posted, enrollment created, assignment submitted). Institutional partners and third-party EdTech tools require real-time event notifications to keep their own systems in sync. Without outbound webhooks, every integration must poll the API — wasteful, laggy, and subject to rate limits. Webhooks are also the delivery mechanism for the Zapier connector (16.10), the marketplace event bus (16.9), and IMS Caliper emission (9.6).

## 2. Goals

- Deliver a reliable, signed, retry-capable webhook system for key LMS events.
- Sign every payload with HMAC-SHA256 per RFC 8959 so consumers can verify authenticity.
- Provide a management UI for subscribing endpoints to event types, viewing delivery logs, and testing.
- Implement exponential-backoff retry with dead-letter queue using the background job system (17.3).
- Rate-limit outbound delivery so a misbehaving subscriber cannot DDoS the job queue.

## 3. Non-Goals

- Inbound webhooks from third-party systems (covered per-integration in 16.4).
- Real-time WebSocket push to the browser (a separate concern).
- Full event-sourcing / event store (out of scope; these are fire-and-forget notifications).
- LTI Advantage Deep Link payloads (2.12).

## 4. Personas & User Stories

- **As an institutional SIS developer**, I want to receive a webhook when a student's grade is finalized so that my grade-passback service can update our Banner integration without polling.
- **As a third-party EdTech tool**, I want to subscribe to `enrollment.created` events so that I can provision a seat in my system the moment a student joins a course.
- **As an admin**, I want to see a delivery log showing success/failure for each webhook attempt so that I can debug integration failures without contacting Lextures support.
- **As a developer**, I want a "test webhook" button that sends a sample payload immediately so that I can verify my endpoint configuration without waiting for a real event.
- **As a security engineer**, I want every webhook payload to carry a signature header so that my consumer can reject forged requests.

## 5. Functional Requirements

- **FR-1.** The system MUST allow org admins to register webhook endpoints (URL + event types + optional secret) via the admin UI.
- **FR-2.** The system MUST sign each delivery with `X-Lextures-Signature: sha256=<hmac>` using a per-subscription HMAC-SHA256 key, following RFC 8959 conventions.
- **FR-3.** The system MUST enqueue webhook deliveries as background jobs (17.3) immediately after the triggering database transaction commits.
- **FR-4.** The system MUST retry failed deliveries with exponential backoff: attempts at 1 min, 5 min, 30 min, 2 h, 8 h, 24 h; after 6 failures, move to dead-letter queue.
- **FR-5.** The system MUST record each delivery attempt: timestamp, HTTP status, response body (truncated to 1 KB), latency, retry count.
- **FR-6.** The system MUST deliver a canonical event envelope: `event_type`, `event_id`, `created_at`, `api_version`, `data` object.
- **FR-7.** The system SHOULD support per-subscription event filtering (subscribe to `grade.posted` only, not all grade events).
- **FR-8.** The system SHOULD enforce a per-subscription delivery timeout of 10 seconds and treat timeouts as failures.
- **FR-9.** The system MAY pause a subscription automatically after 3 consecutive days of failures and notify the subscriber email.
- **FR-10.** The system MUST provide a "send test event" capability that fires a synthetic payload to the configured URL immediately.

## 6. Non-Functional Requirements

- **Performance** — Webhook enqueue latency (event → job queue) ≤ 500 ms. Individual HTTP delivery attempt times out after 10 s. Throughput: system must sustain 5,000 deliveries/min across all subscriptions.
- **Security** — Signing key stored encrypted at rest (17.17); never returned in API responses after initial creation. TLS-only outbound delivery; reject self-signed certificates unless `tls_skip_verify` flag explicitly set by admin (with audit log warning). SSRF mitigation: block delivery to RFC 1918 / loopback addresses.
- **Privacy & Compliance** — Webhook payloads MUST NOT include raw PII (names, emails) unless the subscription has `pii:webhook` scope. Delivery log entries subject to 90-day retention policy.
- **Accessibility** — Webhook management UI fully keyboard-navigable; delivery status table uses proper `<th scope>` markup.
- **Scalability** — Delivery workers horizontally scalable via job queue (17.3); no per-worker state. Redis-backed job queue with Postgres dead-letter persistence.
- **Reliability** — At-least-once delivery guarantee; idempotency key (`event_id` UUID) in envelope allows consumers to deduplicate. Dead-letter queue monitored and alerting configured (17.7).
- **Observability** — Prometheus counters: `webhook_deliveries_total{event_type,status}`, `webhook_delivery_latency_seconds`. Alert on dead-letter queue depth > 100.
- **Maintainability** — Event type registry in `server/src/webhooks/events.rs`; adding a new event type requires one enum entry + one emitter call. No scattered string literals.
- **Internationalization** — Event type display names in i18n keys in admin UI.
- **Backward compatibility** — `api_version` field in envelope allows consumers to detect schema changes; old subscriptions continue receiving v1 payloads until they opt in to v2.

## 7. Acceptance Criteria

- **AC-1.** *Given* a registered webhook subscription for `grade.posted`, *When* an instructor posts a grade, *Then* the subscriber endpoint receives an HTTP POST within 2 seconds containing a valid HMAC-SHA256 signature.
- **AC-2.** *Given* a subscriber endpoint returns HTTP 500, *When* the delivery attempt fails, *Then* a retry is scheduled after 1 minute and the delivery log shows status `failed` with the response code.
- **AC-3.** *Given* 6 consecutive delivery failures over 24 h, *When* the dead-letter threshold is reached, *Then* the subscription is paused and a notification email is sent to the subscription owner.
- **AC-4.** *Given* a webhook payload, *When* the consumer verifies the `X-Lextures-Signature` using the shared secret, *Then* the HMAC matches and the payload is accepted as authentic.
- **AC-5.** *Given* an admin clicks "Send test event", *When* the endpoint is reachable and returns 200, *Then* the delivery log shows one successful synthetic delivery with `test: true` in the payload.
- **AC-6.** *Given* a delivery URL targeting a private RFC 1918 address, *When* the subscription is created, *Then* the system rejects the URL with a validation error citing SSRF policy.

## 8. Data Model

```sql
-- server/migrations/074_webhook_subscriptions.sql
CREATE TABLE integrations.webhook_subscriptions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    org_id          UUID NOT NULL,
    label           TEXT NOT NULL,
    endpoint_url    TEXT NOT NULL,
    signing_key_enc TEXT NOT NULL,    -- AES-256 encrypted signing key
    event_types     TEXT[] NOT NULL,
    active          BOOLEAN NOT NULL DEFAULT true,
    paused_at       TIMESTAMPTZ,
    created_by      UUID REFERENCES user.users(id),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE integrations.webhook_deliveries (
    id              BIGSERIAL PRIMARY KEY,
    subscription_id UUID NOT NULL REFERENCES integrations.webhook_subscriptions(id),
    event_type      TEXT NOT NULL,
    event_id        UUID NOT NULL,
    payload_hash    TEXT NOT NULL,   -- SHA-256 of payload for dedup
    attempt_count   SMALLINT NOT NULL DEFAULT 0,
    status          TEXT NOT NULL,   -- pending|delivered|failed|dead_lettered
    last_http_status SMALLINT,
    last_response   TEXT,
    next_retry_at   TIMESTAMPTZ,
    delivered_at    TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON integrations.webhook_deliveries (subscription_id, created_at DESC);
CREATE INDEX ON integrations.webhook_deliveries (status, next_retry_at)
    WHERE status IN ('pending', 'failed');
```

```jsonc
  "test": false,
  "data": { ... }
}
```

## 10. UI / UX

- **Admin → Integrations → Webhooks** page: list of subscriptions with status indicators (active/paused/failing).
- Create/edit form: URL input, event type multi-select checkboxes grouped by domain, label field.
- Subscription detail: delivery log table with timestamp, event type, HTTP status, retry count, expandable response body.
- "Send test event" button with event type selector and response preview panel.
- Failure badges on subscriptions with consecutive failures; "Reactivate" button for paused.
- Mobile: responsive table with horizontal scroll; delivery log collapsible rows.

## 12. Integration Points

- **Internal:** `server/src/services/mail.rs` — email notification for paused subscriptions via job queue (17.3).
- **Internal:** `server/src/routes/` — event emitters added to grade, enrollment, assignment, quiz handlers.

## 13. Dependencies & Sequencing

- Must ship after: 16.2 (signing keys tied to API token infrastructure), 17.3 (job queue required for async delivery).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| SSRF via webhook URL to internal services | H | H | Allowlist/blocklist on URL validation; no RFC 1918 delivery |
| Delivery storm if many subscriptions on high-frequency event | M | H | Per-subscription concurrency cap; circuit breaker in delivery worker |
| Signing key exposed in DB breach | M | H | Encrypt signing key at rest with app-level key (17.17) |
| Consumer processes same event twice | M | M | Include `event_id`; document idempotency expectation; dedup window in consumer guidance |

## 15. Rollout Plan

- **Feature flag:** `webhooks_enabled` (OpenFeature, default `false`).
- **Phase 1:** Schema + backend delivery infrastructure + delivery log. Internal QA only.
- **Phase 2:** Enable for pilot institutional partners; collect feedback on event schema.
- **Phase 3:** GA with documented event catalog; marketplace integration.
- **Rollback:** Disable feature flag; undelivered jobs remain in queue and can be drained.

## 16. Test Plan

- **Unit** — HMAC-SHA256 signature generation/verification, SSRF URL validation, retry backoff schedule calculation.
- **Integration** — Delivery pipeline: mock HTTP server; verify signature; test retry after 500; verify dead-letter after 6 failures.
- **End-to-end** — Playwright: admin creates subscription; triggers a grade post; verifies delivery log shows success.
- **Security** — SSRF test: attempt delivery to 127.0.0.1, 10.0.0.1, 169.254.169.254; verify all blocked. Signature tampering: modify payload after signing; verify consumer rejects.
- **Accessibility** — axe-core on webhook management pages; delivery log table structure audit.
- **Performance / load** — k6: simulate 1,000 concurrent grade-post events; verify all delivery jobs enqueued within 500 ms; sustained 5,000 deliveries/min.
- **Manual exploratory** — QA: test event button; paused subscription reactivation; event type filter behavior.
