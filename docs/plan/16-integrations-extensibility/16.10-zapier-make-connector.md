## 1. Problem Statement

Many educators and administrators use Zapier or Make.com (formerly Integromat) to automate workflows between apps without writing code. Without Lextures connectors on these platforms, any Lextures data integration requires custom development. A Zapier/Make connector exposes Lextures triggers (new assignment, grade posted) and actions (create course, enroll user) to thousands of existing automations, enabling Lextures customers to self-serve integrations with Google Sheets, Airtable, Mailchimp, Salesforce, and hundreds of other tools.

## 2. Goals

- Publish a Zapier integration (at minimum a beta/invite-only app) covering key triggers and actions.
- Publish a Make.com (Integromat) module set with equivalent coverage.
- Implement all Zapier triggers as webhooks (16.3) and all actions via the public REST API (16.1).
- Provide API token–based authentication (16.2) as the connector auth mechanism, leveraging OAuth 2.0 flow supported by both platforms.
- Write and maintain connector definitions (Zapier CLI / Make.com SDK) in a dedicated `integrations/zapier/` and `integrations/make/` directory.

## 3. Non-Goals

- n8n or other self-hosted automation platforms (community-maintained, not first-party).
- Deep bi-directional sync beyond what the public API supports.
- Custom connector features beyond what the Zapier/Make platforms support natively.

## 4. Personas & User Stories

- **As a K-12 admin**, I want a Zap that sends a Slack DM to a teacher when a student submits an overdue assignment so that the teacher is notified immediately without checking Lextures.
- **As a university instructor**, I want to automatically add a row to my Google Sheet gradebook when Lextures posts a grade so that my department tracking spreadsheet stays current.
- **As a self-learner platform operator**, I want a Make.com scenario that enrolls a user in a Lextures course when they complete a payment in a separate checkout tool I already use.
- **As an admin**, I want to use Zapier to sync new Lextures enrollments to our CRM (Salesforce) so that our marketing team can track learner acquisition.
- **As a developer**, I want to test the Zapier connector locally using the Zapier CLI so that I can iterate on trigger/action schemas without waiting for platform review.

## 5. Functional Requirements

- **FR-1.** The system MUST support OAuth 2.0 authentication in the Zapier connector using the authorization-code flow (16.9 marketplace OAuth or direct API token auth as fallback).
- **FR-2.** The Zapier connector MUST implement the following triggers: `new_enrollment`, `grade_posted`, `assignment_created`, `assignment_submitted`, `quiz_completed`.
- **FR-3.** The Zapier connector MUST implement the following actions: `enroll_user`, `create_announcement`, `get_course`, `get_user`, `post_grade`.
- **FR-4.** Zapier triggers MUST use the REST-hook pattern (webhook subscription created on Zap enable, deleted on Zap disable) backed by the 16.3 webhook infrastructure.
- **FR-5.** The Make.com connector MUST implement equivalent triggers and actions as the Zapier connector.
- **FR-6.** The system MUST include a `GET /api/v1/me` endpoint returning basic user info (name, email, org) for Zapier's "test connection" step.
- **FR-7.** The system SHOULD implement polling-based triggers as a fallback for Zapier users who cannot configure webhook endpoints (use `?since=` cursor on collection endpoints).
- **FR-8.** The connector MUST provide human-readable field labels and help text for all trigger/action fields, following Zapier's UX guidelines.
- **FR-9.** The system MAY provide sample data for each trigger so Zapier users can configure downstream actions without needing real events.

## 6. Non-Functional Requirements

- **Performance** — Trigger webhook delivery via 16.3 infrastructure (≤ 5 s latency); polling endpoint for fallback triggers: p95 ≤ 200 ms.
- **Security** — Connector auth via OAuth 2.0 with PKCE; API tokens scoped to the minimum needed for the user's Zap. Zapier/Make platforms receive only the data fields included in the trigger payload; no bulk data export.
- **Privacy & Compliance** — Trigger payloads forwarded to Zapier should not include raw PII unless the user's token has `pii:read` scope. Users should be informed in the connector description that data flows to Zapier's infrastructure.
- **Accessibility** — Connector UI within Zapier/Make is rendered by those platforms; Lextures OAuth consent screen must be WCAG 2.1 AA.
- **Scalability** — Webhook trigger delivery scales with 16.3 infrastructure; no Lextures-specific bottleneck. Polling fallback rate-limited per token (16.1 / 17.6).
- **Reliability** — Zap-triggered webhook subscriptions managed by 16.3; retry/dead-letter applies. Connector must handle Zapier webhook subscription creation failures gracefully.
- **Observability** — Tag webhook deliveries with `source=zapier` / `source=make`; monitor delivery success rate per platform.
- **Maintainability** — Connector definition in `integrations/zapier/index.js` (Zapier CLI project) and `integrations/make/` (Make.com SDK bundle); CI runs `zapier validate` and `make validate` on every PR.
- **Internationalization** — Connector field labels in English only (Zapier/Make do not support localization of third-party connectors).
- **Backward compatibility** — Connector schema changes must be backward-compatible; breaking field removal requires a version bump in the Zapier app.

## 7. Acceptance Criteria

- **AC-1.** *Given* a Zap is set up with the Lextures trigger "New Enrollment" and Slack action, *When* an admin enrolls a student, *Then* the Zap fires within 10 seconds and the Slack message is sent.
- **AC-2.** *Given* a Zap is disabled by the user, *When* the Zapier platform calls the webhook unsubscribe endpoint, *Then* the corresponding webhook subscription in Lextures is deleted.
- **AC-3.** *Given* a user authenticates the Zapier connector with an API token, *When* Zapier calls the `/api/v1/me` test endpoint, *Then* the user's name and email are returned and Zapier shows "Connected."
- **AC-4.** *Given* a Make.com scenario action "Enroll User," *When* it is executed with a valid email and course ID, *Then* the user is enrolled and the action returns the new enrollment ID.
- **AC-5.** *Given* `zapier validate` is run in CI, *When* the connector definition contains a schema error, *Then* the CI job fails with a descriptive error message.
- **AC-6.** *Given* a trigger payload contains a student email, *When* the user's token lacks `pii:read` scope, *Then* the email field is omitted from the webhook payload delivered to Zapier.

## 8. Data Model

No new tables. The Zapier/Make connectors use the existing `integrations.webhook_subscriptions` table (16.3) for REST-hook subscriptions, tagged with `source='zapier'` or `source='make'` in the `settings` JSONB column.

The `GET /api/v1/me` endpoint uses the existing user profile data.

## 9. API Surface

No new server routes beyond `/api/v1/me` (trivial extension of the existing session info endpoint).

Zapier REST-hook subscription endpoints are standard 16.3 webhook management endpoints:
- `POST /api/v1/webhooks` (subscribe)
- `DELETE /api/v1/webhooks/{id}` (unsubscribe)

Polling fallback endpoints are standard 16.1 collection endpoints with `?since=<cursor>` parameter added.

## 10. UI / UX

- Lextures does not own any UI for the Zapier/Make connector beyond the OAuth consent screen (16.9) and the API token generation page (16.2).
- A help center article with screenshots of setting up a Zapier Zap with Lextures is required.
- The marketplace listing page (16.9) will feature the Zapier and Make.com connectors as first-party apps.

## 11. AI / ML Considerations

Not applicable.

## 12. Integration Points

- **Internal:** 16.1 (API), 16.2 (tokens), 16.3 (webhook subscriptions), 16.9 (OAuth).
- **External:** Zapier Platform CLI / Developer Platform; Make.com SDK.
- **Connector code location:** `integrations/zapier/` and `integrations/make/` at repo root (not inside `server/` or `clients/web/`).

## 13. Dependencies & Sequencing

- Must ship after: 16.1, 16.2, 16.3 (all required for connector functionality).
- Must ship before: nothing strictly, but GA of 16.9 marketplace benefits from having Zapier as a featured first app.
- Shared infra: webhook delivery (16.3), rate limiting (17.6).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Zapier platform review delays public listing | H | M | Use invite-only beta while awaiting review; share connector directly with pilot users |
| Zap trigger fires but Zapier's webhook delivery to the user's Zap fails | M | L | Zapier handles their own retry; Lextures responsibility ends at webhook delivery |
| PII in trigger payloads forwarded to third-party Zap actions | M | M | Document scope requirement; default to no-PII payloads |
| Connector schema breaks after Lextures API change | M | M | API versioning (16.1); connector pinned to `/api/v1/` |

## 15. Rollout Plan

- **Feature flag:** `zapier_connector_enabled` (controls REST-hook endpoint access; default `false`).
- **Phase 1:** Zapier connector — triggers only; invite-only beta with 10 pilot users.
- **Phase 2:** Add actions; Make.com connector; submit for platform review.
- **Phase 3:** Public listing on Zapier App Directory and Make.com marketplace; GA.
- **Rollback:** Disable flag; existing webhook subscriptions created by Zapier are deleted.

## 16. Test Plan

- **Unit** — REST-hook subscribe/unsubscribe; polling cursor logic; payload scope filtering.
- **Integration** — `zapier test` against a Lextures staging environment; full trigger → action flow with mock Zap.
- **End-to-end** — Zapier live test: create enrollment in Lextures; verify Zap trigger fires; verify Google Sheet row added.
- **Security** — Verify scope-filtered payloads; OAuth PKCE enforcement; webhook signature verification by Zapier.
- **Accessibility** — OAuth consent screen axe-core (Lextures responsibility); connector UI tested in Zapier (platform responsibility).
- **Performance / load** — k6: 500 simultaneous REST-hook subscriptions; polling endpoint under load.
- **Manual exploratory** — QA: enable/disable Zap; verify subscribe/unsubscribe; test with invalid token.
