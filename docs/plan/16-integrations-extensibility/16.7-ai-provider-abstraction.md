## 1. Problem Statement

All AI calls in Lextures are currently routed exclusively through OpenRouter (`server/src/services/ai/`). This creates several blockers: (1) institutional customers with FERPA/GDPR requirements cannot guarantee data residency because OpenRouter routes to multiple providers; (2) customers with existing enterprise Anthropic or Azure OpenAI agreements want to use their own keys for cost and data-processing reasons; (3) Lextures cannot offer Bedrock or Vertex routing for customers in AWS/GCP environments; (4) no audit trail exists for which model answered which query. An abstraction layer fixes all of these while preserving the current OpenRouter path as the default.

## 2. Goals

- Introduce a trait-based AI provider abstraction in Rust that any service can call without knowing which backend is used.
- Support four provider backends: OpenRouter (existing), Anthropic-direct, OpenAI-direct (including Azure OpenAI), AWS Bedrock, and Google Vertex AI.
- Allow per-tenant configuration of which provider and model to use, with per-tenant BYOK (bring-your-own-key) stored encrypted via the secrets manager (17.17).
- Emit per-call cost/token metrics for the cost-control feature (19.14).
- Maintain a provider-agnostic model identifier mapping so that prompts do not need to hard-code model strings.

## 3. Non-Goals

- Fine-tuning or model training (out of scope entirely).
- Local/on-premise LLM hosting (self-hosted Ollama path deferred to later phase).
- PII redaction proxy (19.11 — that wraps this abstraction layer).
- Eval harness (19.13).
- Per-student AI usage toggle (10.17 — depends on this feature).

## 4. Personas & User Stories

- **As a university IT director**, I want to configure Lextures to use our institution's Azure OpenAI subscription so that student data stays within our enterprise data-processing agreement.
- **As a K-12 district admin**, I want to point Lextures at our AWS Bedrock endpoint with our own key so that no student data touches OpenRouter's routing infrastructure.
- **As a self-learner platform operator**, I want to use Anthropic Claude direct for tutoring features because it has the best instruction-following behavior for our use case.
- **As a developer**, I want a single `ai_provider.complete(prompt)` interface in Rust so that I can write an AI feature once and swap the backend via config without touching application code.
- **As a finance officer**, I want per-provider token usage logged so that I can reconcile AI costs against our provider bills.

## 5. Functional Requirements

- **FR-1.** The system MUST define a `AiProvider` Rust trait with at minimum `complete(messages, options) -> Result<AiResponse>` and `embed(text) -> Result<Vec<f32>>` methods.
- **FR-2.** The system MUST implement the trait for: OpenRouter (existing), Anthropic Messages API, OpenAI Chat Completions API (compatible with Azure OpenAI), AWS Bedrock Converse API, Google Vertex AI Gemini.
- **FR-3.** The system MUST allow per-tenant selection of provider and model via a `tenant_ai_settings` configuration (stored in DB or per-tenant config, admin-editable).
- **FR-4.** The system MUST support per-tenant BYOK: admin supplies an API key which is stored encrypted using the secrets manager (17.17) and retrieved at runtime; the key is never logged or returned in API responses.
- **FR-5.** The system MUST emit token usage (prompt_tokens, completion_tokens) and estimated cost per call as structured log fields and Prometheus metrics.
- **FR-6.** The system MUST maintain a provider-agnostic model alias registry (`claude-3-5-sonnet`, `gpt-4o`, `gemini-1.5-pro`) so that tenant config uses stable aliases, not provider-specific model IDs.
- **FR-7.** The system SHOULD implement a fallback chain: if the primary provider returns a 5xx error, retry once on the fallback provider (configurable per tenant).
- **FR-8.** The system SHOULD support streaming responses (SSE) for providers that support it, forwarding the stream to the browser for real-time AI output.
- **FR-9.** The system MAY support a "dry run" mode that returns a synthetic response without calling any provider, for testing and demo purposes.

## 6. Non-Functional Requirements

- **Performance** — Provider switching overhead ≤ 5 ms (trait dispatch + config lookup); streaming response first-token latency not increased by abstraction layer.
- **Security** — BYOK keys stored using AES-256-GCM via secrets manager (17.17); keys never appear in logs, metrics labels, or API responses. Provider API calls over TLS 1.2+ only.
- **Privacy & Compliance** — Tenant BYOK configuration means provider data processing terms are between the tenant and their provider; Lextures documents this in DPA templates. All AI calls logged with tenant_id, provider, model alias, token counts — not prompt content (FERPA).
- **Accessibility** — No direct UI impact; streaming output in the AI tutor/notebook UI must not break screen reader live regions (existing concern in `student_notebook_rag_ai.rs`).
- **Scalability** — Provider instances are stateless singletons per Axum app; per-tenant config loaded lazily and cached with 5-minute TTL in Arc<RwLock>.
- **Reliability** — Fallback chain (FR-7) handles single-provider outages; circuit breaker per provider tracks failure rate. Hard timeout of 60 s on any single provider call.
- **Observability** — Prometheus histogram `ai_provider_latency_seconds{provider,model}`; counter `ai_provider_errors_total{provider,error_type}`; gauge `ai_estimated_cost_dollars_total{provider,tenant}`.
- **Maintainability** — New provider requires: one `impl AiProvider` file + one entry in model alias registry + one config key; no changes to calling code.
- **Internationalization** — Provider/model names in UI use i18n keys; error messages locale-aware.
- **Backward compatibility** — Existing calls through OpenRouter continue unchanged; opt-in per tenant.

## 7. Acceptance Criteria

- **AC-1.** *Given* a tenant configured to use Anthropic-direct with their own key, *When* an AI feature (quiz generation, notebook RAG) is invoked, *Then* the HTTP call goes to `api.anthropic.com`, not OpenRouter, and the response is processed correctly.
- **AC-2.** *Given* a BYOK key is configured for a tenant, *When* the secrets manager is queried for that tenant's AI key, *Then* the key is returned decrypted; it is never present in any log line or API response.
- **AC-3.** *Given* the primary provider returns HTTP 503, *When* the fallback chain is configured, *Then* the system retries on the fallback provider and the user receives a response without an error.
- **AC-4.** *Given* any AI call completes, *When* Prometheus metrics are scraped, *Then* `ai_provider_latency_seconds` contains a sample for the correct `provider` and `model` labels.
- **AC-5.** *Given* a new provider implementation (`impl AiProvider`) is added, *When* existing unit tests for quiz generation are run with the new provider, *Then* all tests pass without modification to the test or feature code.
- **AC-6.** *Given* a tenant uses AWS Bedrock, *When* token usage is logged, *Then* the structured log entry contains `provider=bedrock`, `model=<alias>`, `prompt_tokens=N`, `completion_tokens=N`.

## 8. Data Model

```sql
-- server/migrations/078_tenant_ai_settings.sql
CREATE TABLE settings.tenant_ai_settings (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    org_id          UUID NOT NULL UNIQUE,
    provider        TEXT NOT NULL DEFAULT 'openrouter',
    model_alias     TEXT NOT NULL DEFAULT 'claude-3-5-sonnet',
    fallback_provider TEXT,
    byok_secret_ref TEXT,    -- reference to secrets manager key path; NOT the key itself
    settings        JSONB NOT NULL DEFAULT '{}',
    updated_by      UUID REFERENCES user.users(id),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Existing AI calls use the system-level OpenRouter config (env var) when no tenant setting exists.

## 10. UI / UX

- **Admin → Settings → AI Configuration** panel: provider dropdown (OpenRouter, Anthropic, OpenAI, Azure OpenAI, Bedrock, Vertex), model alias dropdown, BYOK key input (write-only, shows "Configured" after save), fallback provider selector, "Test Connection" button.
- "Test Connection" shows latency and token consumption of a synthetic "Hello" prompt.
- Mobile: settings page responsive.

## 11. AI / ML Considerations

- Model alias registry maintained in `server/src/services/ai/models.rs`; aliases are stable identifiers mapped to provider-specific model IDs (e.g., `claude-3-5-sonnet` → `claude-3-5-sonnet-20241022` on Anthropic).
- Prompt templates in existing services (`quiz_generation_ai.rs`, `adaptive_quiz_ai.rs`, `syllabus_section_ai.rs`, `student_notebook_rag_ai.rs`, `assignment_rubric_ai.rs`) remain unchanged; they call the trait interface.
- Eval harness (19.13) will validate that switching providers does not degrade output quality below threshold.

## 12. Integration Points

- **Internal:** `server/src/services/ai/` — new `provider.rs` trait, `openrouter.rs`, `anthropic.rs`, `openai.rs`, `bedrock.rs`, `vertex.rs` implementations; all existing AI services refactored to use trait.
- **Internal:** `server/src/services/adaptive_quiz_ai.rs`, `quiz_generation_ai.rs`, `syllabus_section_ai.rs`, `student_notebook_rag_ai.rs`, `assignment_rubric_ai.rs` — call site unchanged (polymorphism handles it).
- **External:** Anthropic Messages API, OpenAI Chat Completions, AWS Bedrock Converse API, Google Vertex AI Gemini API.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Provider API incompatibilities (different error codes/shapes) | H | M | Per-provider error normalization in each impl; integration tests with mocked providers |
| BYOK key rotation breaks existing calls mid-flight | L | H | Grace period: old key valid 5 min after rotation; background refresh on next call |
| Cost explosion if fallback chain loops | L | H | Maximum 1 fallback retry; circuit breaker; cost alert (19.14) |
| Tenant mis-configures provider → all AI features break | M | H | "Test Connection" required before save; rollback to OpenRouter default if test fails |

## 15. Rollout Plan

- **Feature flag:** `ai_provider_abstraction` (OpenFeature, default `false` for BYOK; OpenRouter remains default for all tenants).
- **Phase 1:** Trait refactor internally; all existing behavior unchanged.
- **Phase 2:** Anthropic-direct and OpenAI-direct backends; BYOK UI for pilot enterprise customers.
- **Phase 3:** Bedrock and Vertex; GA.
- **Rollback:** Revert to OpenRouter env var; BYOK config remains in DB but ignored.

## 16. Test Plan

- **Unit** — Trait mock: each existing AI service tested with a `MockAiProvider`; response parsing for each provider impl.
- **Integration** — Sandbox API keys for Anthropic and OpenAI; verify end-to-end quiz generation with each provider.
- **End-to-end** — Playwright: admin sets Anthropic BYOK; triggers quiz generation; verifies output; checks Prometheus metric.
- **Security** — BYOK key not present in any log line (log scrubber test); secrets manager encryption at rest.
- **Accessibility** — Admin AI settings page axe-core scan.
- **Performance / load** — Trait dispatch overhead benchmark: < 5 µs per call.
- **Manual exploratory** — QA: switch providers mid-session; fallback chain; "Test Connection" with invalid key.
