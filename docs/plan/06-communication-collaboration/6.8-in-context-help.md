## 1. Problem Statement

Self-learners and K-12 students encountering issues in Lextures have no in-platform way to get help. There is no help widget, no contextual tooltip system, and no live chat. Users who get stuck must leave the platform, find contact information, and compose an email — a friction pattern that increases churn for self-learners and causes students to abandon activities. Even a basic embedded help widget dramatically reduces support burden and improves completion rates.

## 2. Goals

- Embed a contextual help widget in the platform UI that surfaces relevant help articles based on the current page.
- Provide a chat support channel that, for SL customers, routes to a support agent or (for unmonitored hours) to a bot fallback.
- Track help widget interactions for product analytics.
- Allow admins to configure which support provider (Intercom, Crisp, custom) is used per org.

## 3. Non-Goals

- Building a custom live chat engine — integrate with an existing provider (Crisp is open-source; Intercom for premium).
- AI tutor within the help widget — that is 6.9.
- In-depth help authoring — the help center (20.3) owns that content.
- Dedicated mobile SDK for help — the embedded widget works on the mobile web view.

## 4. Personas & User Stories

- **As a self-learner** stuck on uploading an assignment, I want a "Help" button that shows relevant articles so that I can self-serve without leaving the page.
- **As a K-12 student**, I want to chat with a support bot when it is after hours so that I get at least some help even when my teacher is unavailable.
- **As an admin**, I want to configure the support widget with our org's branding and link to our own help articles so that the experience feels district-provided.
- **As a support agent**, I want to see which page the user was on when they opened the help chat so that I have context before responding.

## 5. Functional Requirements

- **FR-1.** The system MUST embed a help widget launcher button (bottom-right of viewport) on all authenticated pages.
- **FR-2.** The system MUST surface contextual help article links based on the current route (e.g., on the quiz page, show quiz-related articles from 20.3).
- **FR-3.** The system MUST pass contextual data to the chat widget: current page URL, user role, org slug, and a non-PII session ID.
- **FR-4.** The system MUST support pluggable chat providers: Crisp (default, open-source SDK), Intercom (premium), or a self-hosted option — configured per deployment via env var `SUPPORT_CHAT_PROVIDER`.
- **FR-5.** The system SHOULD display a "FAQ" section in the widget before escalating to live chat, to reduce agent load.
- **FR-6.** The system SHOULD allow org admins to disable the widget or customize the default help article list from the admin settings UI.
- **FR-7.** The system MAY log widget-open events to the product analytics event stream (page + role + timestamp) for feature discovery metrics.

## 6. Non-Functional Requirements

- **Performance** — the help widget script must load asynchronously (non-blocking); must not increase page TTI by more than 100 ms.
- **Security** — widget must not expose PII to the external provider; pass only non-sensitive identifiers; providers with data-processing agreements only.
- **Privacy & Compliance** — COPPA / FERPA: K-12 students must not have PII sent to third-party chat providers without parental consent and DPA. For K-12 orgs, the admin must confirm DPA before enabling live chat.
- **Accessibility** — widget launcher button has `aria-label="Get help"`; widget dialog accessible; close button within widget has visible focus ring.
- **Scalability** — the widget runs entirely client-side via third-party SDK; no server-side load.
- **Reliability** — if the chat provider script fails to load, no JS error is thrown; the widget simply does not appear.
- **Observability** — widget-open events logged via product analytics (not sent to external provider).
- **Maintainability** — widget integration in `clients/web/src/components/layout/HelpWidget.tsx`; provider config abstracted behind a props interface.
- **Internationalization** — widget supports multilingual display (Crisp has built-in i18n); custom article titles externalized.
- **Backward compatibility** — widget is additive; disabling it has no effect on existing functionality.

## 7. Acceptance Criteria

- **AC-1.** *Given* an authenticated user on the quiz page, *When* they click the help button, *Then* a panel opens showing at least 2 quiz-related help article links.
- **AC-2.** *Given* the Crisp chat provider is configured, *When* the page loads, *Then* the Crisp widget script loads asynchronously and does not block `DOMContentLoaded`.
- **AC-3.** *Given* the chat provider script fails to load (network error), *When* the user clicks the help button, *Then* the button falls back to opening the help center URL in a new tab.
- **AC-4.** *Given* a K-12 org admin who has not confirmed a DPA, *When* they attempt to enable live chat for their org, *Then* a confirmation dialog requesting DPA acknowledgment is shown before activation.
- **AC-5.** *Given* a user opens the help widget, *When* the chat session is started, *Then* the support agent's console shows the user's current page URL and role (not their name or email).
- **AC-6.** *Given* an admin disables the help widget for their org, *When* users in that org view any page, *Then* the help button is not rendered.

## 8. Data Model

No new tables required. Org-level setting stored in `org_settings` JSONB (extend `settings` schema):

```jsonc
// In org settings
{
  "support_widget": {
    "enabled": true,
    "provider": "crisp",
    "crisp_website_id": "...",
    "dpa_confirmed_at": "2026-04-17T00:00:00Z"
  }
}
```

Optional: `help_widget_events` table for analytics if built in-house rather than via external analytics.

## 9. API Surface

| Verb | Path | Auth scope | Description |
|---|---|---|---|
| GET | `/api/orgs/:id/settings/support-widget` | `org_admin` | Get widget config |
| PUT | `/api/orgs/:id/settings/support-widget` | `org_admin` | Update widget config |
| GET | `/api/help/contextual-articles?route=` | authenticated | Get context-relevant article links |

## 10. UI / UX

New component: `clients/web/src/components/layout/HelpWidget.tsx`
- Floating button (bottom-right, `?` icon) visible on all pages.
- Click opens a slide-out panel: search, contextual articles, "Chat with us" CTA.
- On mobile: same behavior; widget panel covers lower 60% of screen.

New admin settings section in org admin UI: "Support & Help" — enable/disable toggle, provider selection, DPA confirmation checkbox.
Accessibility: launcher button `aria-label="Help"`; panel `role="dialog"` with focus trap; close via Escape key.

## 11. AI / ML Considerations

Not applicable for V1. AI bot fallback for after-hours chat may use the conversational AI tutor infrastructure (6.9) in a future iteration.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Third-party chat SDK violates COPPA for K-12 | M | H | DPA confirmation gate; option to disable widget for K-12 orgs |
| Widget script significantly slows page load | M | M | Load asynchronously with `defer`; benchmark TTI before/after |
| Support agent receives PII from widget metadata | M | M | Pass only non-PII fields; document what is and is not sent |

## 15. Rollout Plan

- Feature flag: `help_widget_enabled` (default: `true` for SL orgs, `false` for K12/HE pending DPA).
- Sequence: Crisp SDK integration → contextual articles API → admin config UI → flag.
- Dogfood: enable on internal org; test on quiz and gradebook pages.
- GA criteria: widget loads asynchronously; contextual articles appear; fallback on script failure.
- Rollback: disable flag; no data loss.

## 16. Test Plan

- **Unit** — contextual article route-matching logic.
- **Integration** — org settings save/read for widget config.
- **End-to-end** — Playwright: navigate to quiz page, open help widget, see quiz articles.
- **Security** — confirm no PII fields sent in widget initialization.
- **Accessibility** — widget launcher and panel accessible.
- **Performance / load** — confirm async load does not increase TTI >100 ms (Lighthouse).
- **Manual exploratory** — script failure fallback; K-12 DPA gate.

## 17. Documentation & Training

- Help center: "Getting help within Lextures."

## 18. Open Questions

1. Should we default to Crisp (open-source, self-hostable) or a managed SaaS provider?
2. Should the contextual article mapping be admin-configurable (via a JSON mapping file) or hard-coded by route?
3. For K-12 orgs that reject third-party chat, should we fall back to an internal ticket form?
