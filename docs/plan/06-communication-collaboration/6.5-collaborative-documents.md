## 1. Problem Statement

Group projects and peer learning activities require students to co-author documents and sketch ideas together. Lextures has a solo TipTap editor for content authoring, but there is no real-time multi-user collaborative document editing and no shared whiteboard. Instructors have no in-platform tool for co-creating resources with peers, and students cannot work on group assignments in-platform — they resort to Google Docs or Miro, fragmenting the learning experience outside the LMS.

## 2. Goals

- Enable real-time collaborative rich-text document editing using CRDT-based conflict resolution (Y.js).
- Provide a shared infinite-canvas whiteboard using an embedded open-source tool (Excalidraw or tldraw).
- Allow collaborative documents to be created at the course level or inside group spaces (6.6).
- Surface collaborative docs as module content items (alongside existing content pages).
- Record document edit history for academic integrity review.

## 3. Non-Goals

- Building a custom CRDT engine — use Y.js (or Automerge) as the library.
- Full integration with external editors (Google Docs, Notion) — those are link-embeds only, not native collab.
- Video/audio conferencing inside the document — handled by 6.4.
- Handwriting / drawing recognition — not in scope.
- Document locking / instructor-only editing — V1 all enrolled participants can edit; future: per-role access.

## 4. Personas & User Stories

- **As a group of students**, I want to co-write a project proposal in real time so that we can see each other's edits without emailing files back and forth.
- **As an instructor**, I want to create a shared whiteboard for a brainstorming activity so that students can contribute sticky notes simultaneously during class.
- **As an instructor**, I want to view the edit history of a group document so that I can assess each student's contribution for grading.
- **As a student**, I want to see online presence indicators (who is editing right now) in the document so that I avoid overwriting my teammate's work.

## 5. Functional Requirements

- **FR-1.** The system MUST integrate Y.js for real-time collaborative text editing using a WebSocket-based sync provider (y-websocket) hosted at `wss://app.lextures.io/collab`.
- **FR-2.** The system MUST store Y.js document updates persistently in Postgres (as JSONB binary or a `ydoc_updates` table) so documents survive server restarts.
- **FR-3.** The system MUST provide a collaborative_documents table: `id`, `course_id`, `group_id` (nullable), `title`, `doc_type` (`rich_text`, `whiteboard`), `ydoc_state` (BYTEA), `created_by`, `created_at`, `updated_at`.
- **FR-4.** The system MUST embed Excalidraw (MIT license) for whiteboard documents, synced via Y.js shared state.
- **FR-5.** The system MUST display online presence indicators in the document editor: avatars of currently active collaborators with colored cursors.
- **FR-6.** The system MUST record document-level edit history: snapshots every 5 minutes + on each user's first edit in a session, stored in `collab_doc_snapshots`.
- **FR-7.** The system SHOULD allow a collaborative document to be added as a module item (alongside existing content pages) in `course_structure_items`.
- **FR-8.** The system MAY allow instructors to export a collaborative document to PDF or download as Markdown/HTML.

## 6. Non-Functional Requirements

- **Performance** — Y.js sync latency ≤150 ms p95 for collaborative edits across two concurrent users on different continents (via CDN WebSocket proxy); initial document load ≤1 s.
- **Security** — only enrolled course members (or group members for group docs) can access a collaborative document; WebSocket connection authenticated via JWT passed in connection handshake.
- **Privacy & Compliance** — FERPA: document content is educational record; edit history accessible to instructor/admin; exportable on DSAR request.
- **Accessibility** — TipTap collaborative editor must preserve existing WCAG 2.1 AA keyboard and screen-reader support; presence avatars use `aria-label="[Name] is editing"`.
- **Scalability** — Y.js sync server must support ≥100 concurrent document rooms; connection count monitored.
- **Reliability** — document state persisted to DB after every 30 s of inactivity; no data loss on process restart.
- **Observability** — log `doc_id`, `user_id`, active connection count per room; alert on room crash.
- **Maintainability** — Y.js sync server can be a lightweight Node.js side-car (`y-websocket`) or a Rust-native CRDT server; keep separate from main Axum server.
- **Internationalization** — UI labels externalized; document content stored as-is.
- **Backward compatibility** — existing solo content pages use TipTap without Y.js; collaborative doc is an opt-in new doc type.

## 7. Acceptance Criteria

- **AC-1.** *Given* two students open the same collaborative document, *When* Student A types "Hello", *Then* the text appears in Student B's editor within 150 ms.
- **AC-2.** *Given* a collaborative document is open, *When* a third user joins, *Then* their avatar appears in the presence bar and their cursor is visible with a distinct color.
- **AC-3.** *Given* the Y.js server restarts, *When* a user reopens the document, *Then* all previously saved content is present.
- **AC-4.** *Given* a whiteboard document, *When* a student draws a shape, *Then* the shape appears on all co-editors' canvases in real time.
- **AC-5.** *Given* an instructor requests the edit history of a collaborative document, *When* they call `GET /api/collab-docs/:id/snapshots`, *Then* timestamped snapshots with author attribution are returned.
- **AC-6.** *Given* a user who is not enrolled in the course, *When* they attempt to connect to the document WebSocket, *Then* the connection is rejected with an auth error.

## 8. Data Model

```sql
-- server/migrations/087_collab_docs.sql
CREATE TABLE collaborative_documents (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  course_id   UUID NOT NULL REFERENCES courses(id) ON DELETE CASCADE,
  group_id    UUID,
  title       TEXT NOT NULL,
  doc_type    TEXT NOT NULL DEFAULT 'rich_text'
                CHECK (doc_type IN ('rich_text','whiteboard')),
  ydoc_state  BYTEA,
  created_by  UUID NOT NULL REFERENCES users(id),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE collab_doc_snapshots (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  doc_id     UUID NOT NULL REFERENCES collaborative_documents(id) ON DELETE CASCADE,
  author_id  UUID NOT NULL REFERENCES users(id),
  snapshot   BYTEA NOT NULL,
  taken_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_collab_docs_course ON collaborative_documents(course_id);
CREATE INDEX idx_doc_snapshots_doc  ON collab_doc_snapshots(doc_id, taken_at DESC);
```

## 9. API Surface

| Verb | Path | Auth scope | Description |
|---|---|---|---|
| POST | `/api/courses/:id/collab-docs` | `instructor` / `student` | Create collaborative document |
| GET | `/api/courses/:id/collab-docs` | enrolled | List documents |
| GET | `/api/collab-docs/:id` | enrolled | Get document (initial Y.js state) |
| PATCH | `/api/collab-docs/:id` | enrolled | Update title |
| DELETE | `/api/collab-docs/:id` | `instructor` | Delete document |
| GET | `/api/collab-docs/:id/snapshots` | `instructor` | Get edit history |
| WS | `wss://.../collab/:doc_id?token=` | enrolled | Y.js sync WebSocket |

## 10. UI / UX

New component: `clients/web/src/components/collab/CollabEditor.tsx` — TipTap + Y.js binding.
New component: `clients/web/src/components/collab/CollabWhiteboard.tsx` — Excalidraw + Y.js binding.
Presence bar: row of avatars with names; "3 others editing" overflow indicator.
Document created from: course page "New → Collaborative Document", or module item creation.

Empty state (blank doc): "Start writing…" placeholder.
Offline: document saved to IndexedDB; edits synced when reconnected (Y.js offline support).
Mobile: responsive TipTap toolbar (icon-only on small screens); whiteboard with touch support via Excalidraw.
Accessibility: collaborative editor preserves TipTap ARIA roles; presence avatars use `aria-label`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Y.js sync server is a new operational component | M | M | Use battle-tested `y-websocket`; container-managed; health check |
| Document state corruption on concurrent writes | L | H | Y.js CRDT merges are conflict-free by design; test with high-concurrency scenario |
| Excalidraw license conflict | L | M | Excalidraw is MIT; review license terms before bundling |
| Large document causes slow initial load | M | M | Lazy-load Y.js state; stream updates after initial snapshot |

## 15. Rollout Plan

- Feature flag: `collab_documents_enabled` (default: `false`).
- Sequence: DB migration → Y.js server → collab editor component → whiteboard component → module item integration → flag.
- Dogfood: test collaborative editing with two internal users.
- GA criteria: sub-150 ms sync latency in staging; document persisted after server restart.
- Rollback: disable flag; Y.js server can be stopped; data preserved in DB.

## 16. Test Plan

- **Unit** — Y.js state serialization/deserialization; snapshot scheduling logic.
- **Integration** — WebSocket auth; Y.js two-client sync via test WebSocket clients.
- **End-to-end** — Playwright: two users open same doc, one types, other sees update.
- **Security** — unauthenticated WebSocket rejected; non-enrolled user rejected.
- **Accessibility** — editor preserves keyboard nav; presence avatars labeled.
- **Performance / load** — 100 concurrent document rooms; sync latency under load.
- **Manual exploratory** — network disconnect/reconnect; 5-minute auto-snapshot.

## 18. Open Questions

1. Should the Y.js sync server run as a side-car container within the same pod, or as a separate service?
2. Should we use Redis for Y.js document persistence at scale, or keep it Postgres-only for simplicity?
3. Should we support document import (paste from Google Docs, upload DOCX)?
4. How long should snapshots be retained before pruning (90 days? forever)?
