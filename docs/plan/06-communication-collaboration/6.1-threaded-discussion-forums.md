## 1. Problem Statement

Lextures has a course feed (announcements, activities) and a direct inbox, but no structured asynchronous discussion format. Every LMS from Moodle to Canvas centers on threaded discussion forums as the primary student-to-student and instructor-to-student async communication tool. Without forums, instructors have no structured space to run Socratic discussions, peer reviews, or graded participation activities. The absence of discussion boards is consistently mentioned in LMS RFP evaluations and is a disqualifying gap for HE and many K-12 districts.

## 2. Goals

- Introduce course-level discussion boards with one or more named forums per course.
- Support threaded replies (up to 3 levels deep in V1) with rich-text (TipTap) content.
- Support graded discussions (integrating with the gradebook assignment group framework).
- Allow instructors to create, pin, lock, and delete threads.
- Enable students to upvote / mark "helpful" on replies.

## 3. Non-Goals

- Real-time co-presence in threads (WebSocket "someone is typing") — deferred.
- Group-only discussion spaces — covered in 6.6.
- Anonymous posting in discussions — deferred.
- Full-text search within discussions — deferred to global search feature.
- AI-summarization of discussion threads — deferred to 19.5.

## 4. Personas & User Stories

- **As an instructor**, I want to create a "Week 3 Reflection" discussion prompt so that students can respond asynchronously with their thoughts.
- **As a student**, I want to reply directly to another student's post so that we can have a threaded conversation without it becoming a flat wall of text.
- **As an instructor**, I want to mark a student's discussion post as graded so that participation is reflected in the gradebook.
- **As a student**, I want to upvote helpful replies so that the most useful answers are surfaced.
- **As an instructor**, I want to pin important threads at the top of the forum so that key posts are always visible.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a `discussion_forums` table per course, and a `discussion_threads` table for individual topics/prompts.
- **FR-2.** The system MUST provide a `discussion_posts` table supporting threaded replies with `parent_post_id` (nullable, up to 3 levels of nesting).
- **FR-3.** The system MUST allow instructors to create, edit, pin, lock (prevent new replies), and delete threads.
- **FR-4.** The system MUST support graded discussions: a thread can be linked to a `module_assignment` record; each student's first post triggers grade creation.
- **FR-5.** The system MUST support TipTap rich-text content in posts (images, lists, bold/italic, inline code) stored as ProseMirror JSON.
- **FR-6.** The system MUST provide post upvoting (students can upvote any post once) and display upvote count.
- **FR-7.** The system SHOULD notify participants (via 6.2 email / 6.3 push) when a new reply is posted to a thread they have participated in.
- **FR-8.** The system MAY support a "required to post before seeing others' replies" mode (forced viewpoint).

## 6. Non-Functional Requirements

- **Performance** — thread list must render ≤200 ms; post fetch for a thread with 100 replies ≤100 ms.
- **Security** — students can only edit/delete their own posts; instructors can delete any post in their course; org_admin can delete any post in the org.
- **Privacy & Compliance** — FERPA: discussion posts are educational records; exportable on DSAR request. Moderation tooling needed before public launch.
- **Accessibility** — thread and post components must support keyboard navigation; reply actions accessible via ARIA button labels; rich-text editor accessible (existing TipTap).
- **Scalability** — a popular thread may have 500+ posts; pagination required (cursor-based); index on `(thread_id, created_at)`.
- **Reliability** — post submission must be idempotent (prevent double-post on network retry).
- **Observability** — `forum_id`, `thread_id`, `post_id` on log events; forum activity metrics.
- **Maintainability** — discussion routes in `server/src/routes/communication.rs`; service in `server/src/services/discussions.rs`.
- **Internationalization** — all UI labels externalized; post content stored as-is.
- **Backward compatibility** — discussion schema is additive; no existing tables affected.

## 7. Acceptance Criteria

- **AC-1.** *Given* an instructor, *When* they create a forum thread with the title "Week 3 Reflection", *Then* students enrolled in the course see it in the Discussions tab.
- **AC-2.** *Given* a student posts a reply, *When* another student replies to that post, *Then* the reply is nested under the original at depth 2 and both appear in the thread view.
- **AC-3.** *Given* a graded discussion thread, *When* a student submits their first post, *Then* a pending grade record is created in the gradebook for that student.
- **AC-4.** *Given* a student who has not yet posted, *When* "post before viewing" mode is enabled, *Then* other students' posts are hidden until the student submits their own reply.
- **AC-5.** *Given* a student upvotes a post, *When* they try to upvote the same post again, *Then* the second upvote is rejected (idempotent — upvote count unchanged).
- **AC-6.** *Given* an instructor locks a thread, *When* a student attempts to post a reply, *Then* the reply form is disabled with the message "This discussion has been closed by the instructor."

## 8. Data Model

```sql
-- server/migrations/083_discussion_forums.sql
CREATE TABLE discussion_forums (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  course_id   UUID NOT NULL REFERENCES courses(id) ON DELETE CASCADE,
  name        TEXT NOT NULL,
  description TEXT,
  position    INTEGER NOT NULL DEFAULT 0,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE discussion_threads (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  forum_id        UUID NOT NULL REFERENCES discussion_forums(id) ON DELETE CASCADE,
  assignment_id   UUID REFERENCES module_assignments(id),
  author_id       UUID NOT NULL REFERENCES users(id),
  title           TEXT NOT NULL,
  body            JSONB NOT NULL,
  is_pinned       BOOLEAN NOT NULL DEFAULT false,
  is_locked       BOOLEAN NOT NULL DEFAULT false,
  require_post_first BOOLEAN NOT NULL DEFAULT false,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE discussion_posts (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  thread_id      UUID NOT NULL REFERENCES discussion_threads(id) ON DELETE CASCADE,
  parent_post_id UUID REFERENCES discussion_posts(id),
  author_id      UUID NOT NULL REFERENCES users(id),
  body           JSONB NOT NULL,
  upvote_count   INTEGER NOT NULL DEFAULT 0,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_posts_thread ON discussion_posts(thread_id, created_at);

CREATE TABLE discussion_post_upvotes (
  post_id UUID NOT NULL REFERENCES discussion_posts(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES users(id),
  PRIMARY KEY (post_id, user_id)
);
```

## 9. API Surface

| Verb | Path | Auth scope | Description |
|---|---|---|---|
| GET | `/api/courses/:id/forums` | enrolled | List forums |
| POST | `/api/courses/:id/forums` | instructor | Create forum |
| GET | `/api/forums/:fid/threads` | enrolled | List threads (paginated) |
| POST | `/api/forums/:fid/threads` | enrolled | Create thread |
| GET | `/api/threads/:tid/posts` | enrolled | List posts (cursor paginated) |
| POST | `/api/threads/:tid/posts` | enrolled | Create post |
| PATCH | `/api/threads/:tid` | instructor | Pin/lock/update thread |
| DELETE | `/api/posts/:pid` | author or instructor | Delete post |
| POST | `/api/posts/:pid/upvote` | enrolled student | Upvote post |

## 10. UI / UX

New page: `clients/web/src/pages/lms/CourseDiscussions.tsx`
Thread list: title, author avatar, reply count, last activity, pinned badge.
Thread view: original post, nested replies with indentation, upvote button.
Compose: TipTap editor (reusing `clients/web/src/components/editor/`); submit button.

Empty state: "No discussions yet. Start a conversation!"
Locked thread: compose area replaced by "Discussion closed" notice.
Mobile: thread list is full-width; reply indentation caps at 1 level on mobile for readability.
Accessibility: thread items use `role="article"`; reply nesting conveyed via `aria-level`; upvote button has `aria-label="Upvote (N votes)"`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Thread with 1,000 posts causes slow load | M | M | Cursor pagination; virtual scroll for long threads |
| Student posts inappropriate content | M | M | Instructor moderation tools (hide/delete post); report flag |
| Double-post on network retry | M | L | Idempotency key on post creation endpoint |

## 15. Rollout Plan

- Feature flag: `discussion_forums_enabled` (default: `false`).
- Sequence: DB migration → API → frontend Discussion page → gradebook integration → flag on staging → production.
- Dogfood: create a "Team Feedback" discussion in internal test course.
- GA criteria: threaded replies work; graded discussion creates gradebook entry; notifications sent.
- Rollback: disable flag; schema is additive.

## 16. Test Plan

- **Unit** — nesting depth limit enforcement; upvote idempotency.
- **Integration** — graded discussion → gradebook entry creation; reply notification job enqueue.
- **End-to-end** — Playwright: instructor creates thread, student posts, student replies to peer, instructor grades.
- **Security** — student cannot delete another student's post; locked thread blocks new posts.
- **Accessibility** — axe scan on discussion page; keyboard navigation through threaded posts.
- **Performance / load** — thread fetch with 500 posts ≤100 ms; list with 200 threads ≤200 ms.
- **Manual exploratory** — "post before view" mode; pin thread; delete thread with replies.
