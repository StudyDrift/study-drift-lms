## 1. Problem Statement

Instructors in higher education hold regularly scheduled office hours where students can book time slots. Currently, Lextures has no appointment scheduling feature — instructors must manage office hours via email or external tools (Calendly, Google Calendar). This fragments the student experience, introduces scheduling friction, and means office hour visits are untracked. HE institutions expect this as a basic faculty-student interaction tool.

## 2. Goals

- Allow instructors to define recurring or one-off office hour availability windows.
- Allow students to book a time slot within those windows.
- Optionally link a booked slot to a virtual classroom meeting (6.4).
- Send email/push confirmation and reminders (6.2/6.3).
- Export office hour availability as iCalendar (RFC 5545) for integration with personal calendars.

## 3. Non-Goals

- Full calendar management (instructor's entire schedule) — only office-hour windows.
- Payment for appointment booking — deferred.
- Group office hours (many students, one slot) — V1 is 1:1 only; group slots deferred.
- Walk-in queue management — deferred.

## 4. Personas & User Stories

- **As an instructor**, I want to define "Tuesday 2–4 PM" as weekly office hour slots so that students can book individual 15-minute appointments without emailing me.
- **As a student**, I want to see available office hour slots in my course and book one with one click so that I can discuss my assignment before the deadline.
- **As an instructor**, I want to receive an email when a student books an appointment so that I can add it to my calendar.
- **As a student**, I want a reminder 30 minutes before my appointment so that I do not forget.
- **As an instructor**, I want to optionally attach a virtual meeting link to an appointment so that students can join online.

## 5. Functional Requirements

- **FR-1.** The system MUST provide an `availability_windows` table: `id`, `user_id` (instructor), `course_id`, `day_of_week`, `start_time`, `end_time`, `slot_duration_minutes` (default 15), `is_recurring`, `window_date` (nullable, for one-off), `max_slots_per_window`, `location` (text — room or URL), `status`.
- **FR-2.** The system MUST auto-generate individual `appointment_slots` from `availability_windows` for the upcoming 4 weeks.
- **FR-3.** The system MUST allow a student to book one `appointment_slots` record: `slot_id`, `student_user_id`, `note` (optional), `meeting_id` (nullable, links to virtual meeting), `status` (`booked`, `cancelled`, `completed`).
- **FR-4.** The system MUST prevent double-booking (a slot can have at most one student booking).
- **FR-5.** The system MUST send email confirmations and reminders (6.2) to both instructor and student on booking, cancellation, and 30 minutes before the appointment.
- **FR-6.** The system MUST provide a calendar-week view of an instructor's available and booked slots.
- **FR-7.** The system SHOULD export availability and booked appointments as RFC 5545 iCalendar files.
- **FR-8.** The system MAY auto-create a virtual meeting (6.4) room for each booked appointment if the instructor has virtual office hours configured.

## 6. Non-Functional Requirements

- **Performance** — available slot list must render in ≤200 ms for 4 weeks of availability.
- **Security** — students can only see their own bookings (not other students'); instructors see all bookings for their slots; booking IDs must not be guessable (UUID).
- **Privacy & Compliance** — FERPA: appointment booking is educational record; student note field content must not be exposed to other students.
- **Accessibility** — calendar week view keyboard navigable; slot booking dialog accessible; time slots use semantic time elements.
- **Scalability** — an instructor with 200 students may receive many concurrent booking attempts; slot reservation uses database-level row lock to prevent double-booking.
- **Reliability** — booking must be transactional; on confirmation email failure, booking is still committed.
- **Observability** — log booking creation and cancellation; alert on slot-generation job failure.
- **Maintainability** — scheduling logic in `server/src/services/scheduling.rs`.
- **Internationalization** — times displayed in user timezone; day-of-week names localized.
- **Backward compatibility** — no existing tables affected; purely additive.

## 7. Acceptance Criteria

- **AC-1.** *Given* an instructor defines "Tuesday 2–4 PM" recurring office hours with 15-minute slots, *When* the slot generation job runs, *Then* 8 slots are created for the next 4 Tuesdays.
- **AC-2.** *Given* two students attempt to book the same slot simultaneously, *When* both requests arrive, *Then* only one succeeds with HTTP 200; the other receives HTTP 409 "Slot already booked."
- **AC-3.** *Given* a student books a slot, *When* the booking is confirmed, *Then* both instructor and student receive a confirmation email with the appointment time, location, and a calendar attachment (.ics).
- **AC-4.** *Given* an appointment in 30 minutes, *When* the reminder job runs, *Then* both parties receive a reminder notification.
- **AC-5.** *Given* a student cancels a booking, *When* the cancellation is submitted at least 1 hour before the slot, *Then* the slot becomes available again for other students.
- **AC-6.** *Given* virtual office hours mode, *When* a student books a slot, *Then* a BBB/Jitsi meeting is auto-created and the join URL is included in the confirmation email.

## 8. Data Model

```sql
-- server/migrations/089_office_hours.sql
CREATE TABLE availability_windows (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  instructor_id         UUID NOT NULL REFERENCES users(id),
  course_id             UUID REFERENCES courses(id),
  day_of_week           SMALLINT,  -- 0=Sun, 6=Sat; null for one-off
  window_date           DATE,      -- for one-off windows
  start_time            TIME NOT NULL,
  end_time              TIME NOT NULL,
  slot_duration_minutes INTEGER NOT NULL DEFAULT 15,
  location              TEXT,
  is_virtual            BOOLEAN NOT NULL DEFAULT false,
  status                TEXT NOT NULL DEFAULT 'active',
  created_at            TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE appointment_slots (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  window_id      UUID NOT NULL REFERENCES availability_windows(id),
  slot_start     TIMESTAMPTZ NOT NULL,
  slot_end       TIMESTAMPTZ NOT NULL,
  student_id     UUID REFERENCES users(id),
  student_note   TEXT,
  meeting_id     UUID REFERENCES virtual_meetings(id),
  status         TEXT NOT NULL DEFAULT 'available'
                   CHECK (status IN ('available','booked','cancelled','completed')),
  booked_at      TIMESTAMPTZ,
  UNIQUE (window_id, slot_start)
);

CREATE INDEX idx_slots_window ON appointment_slots(window_id, slot_start);
CREATE INDEX idx_slots_student ON appointment_slots(student_id);
```

## 9. API Surface

## 10. UI / UX

New page: `clients/web/src/pages/lms/CourseOfficeHours.tsx`
- Week calendar grid showing available (green), booked (gray), my booking (blue) slots.
- Click available slot → booking modal with optional note field.
- Instructor view: booked slots show student name and note.

Mobile: day-list view instead of week grid; book via tappable slot card.
Accessibility: calendar grid uses `role="grid"` with `role="gridcell"` slots; booking dialog traps focus; time slots have `aria-label="Tuesday 2:00 PM – available"`.
i18n keys: `office_hours.book`, `office_hours.cancel`, `office_hours.my_appointment`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Double-booking race condition | M | H | DB row-level lock on slot during booking transaction |
| Slot generation job misses a week | M | M | Job runs daily; idempotent; alert on failure |
| Time zone confusion for cross-timezone instructors | H | M | Store all times as UTC; display in viewer's local timezone |

## 15. Rollout Plan

- Feature flag: `office_hours_enabled` (default: `false`).
- Sequence: DB migration → slot generation job → booking API → email confirmations → UI → flag.
- Dogfood: create one instructor's office hours in dev environment.
- GA criteria: booking with concurrent access passes double-booking test; confirmation email sent.
- Rollback: disable flag; tables are additive.

## 16. Test Plan

- **Unit** — slot generation from recurrence rule; timezone conversion.
- **Integration** — concurrent booking race condition test; booking cancellation with re-availability.
- **End-to-end** — Playwright: instructor sets up office hours, student books, both receive email.
- **Security** — student cannot see other students' booking notes.
- **Accessibility** — calendar grid keyboard navigable.
- **Performance / load** — 100 students booking simultaneously → only one succeeds per slot.
- **Manual exploratory** — virtual meeting auto-creation on booking; iCal download.

## 18. Open Questions

1. Should instructors be able to set a per-course cancellation policy (e.g., must cancel ≥1h in advance)?
2. Should there be a waitlist when all slots in a window are booked?
3. Should non-course-linked global office hours (faculty advising hours) be supported?
4. Should the week calendar sync with the instructor's Google/Outlook calendar via CalDAV/Google Calendar API?
