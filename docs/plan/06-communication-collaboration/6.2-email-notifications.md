## 1. Problem Statement

A stub mail service exists at `server/src/services/mail.rs` (used only for password reset), but there is no systematic event-driven email notification system. Users receive no emails when a grade is posted, an assignment is created, a discussion reply is made, or a course announcement is published. In competitive LMS evaluations, email notifications are table-stakes: without them instructors must rely on word-of-mouth and students miss critical academic deadlines. This is a BLOCKER for all three market segments.

## 2. Goals

- Implement an event-driven email notification pipeline: system events → job queue → email delivery.
- Support per-user, per-event-type notification preferences (opt-in/opt-out per channel).
- Provide transactional email templates for the 10 most critical notification types.
- Integrate org-level branding (logo, colors, custom display name) from 5.7.
- Support SMTP and a managed provider (SendGrid / Resend / Postmark) configurable via env vars.

## 3. Non-Goals

- Marketing / bulk email campaigns — only transactional notifications.
- Custom email template editor for admins — deferred to 18.5.
- SMS notifications — deferred.
- Push notifications (web/mobile) — covered in 6.3.
- Auto-translate of notification content — covered in 6.10.

## 4. Personas & User Stories

- **As a student**, I want to receive an email when a grade is posted on my assignment so that I know to check my feedback without logging in repeatedly.
- **As an instructor**, I want to receive an email digest of new discussion replies and student submissions so that I stay on top of my course without constant monitoring.
- **As a parent**, I want to receive an email when my child's grade is updated so that I can discuss it with them.
- **As a user**, I want to control which notification types send me emails so that I am not overwhelmed by messages.
- **As a self-learner**, I want an email reminder when a due date is approaching so that I stay on track with my self-paced course.

## 5. Functional Requirements

- **FR-1.** The system MUST enqueue an email job (via the background job queue from 17.3) for each notification event: grade_posted, assignment_created, discussion_reply, course_announcement, submission_received, assignment_due_reminder (24h), password_reset (already done), welcome/invite.
- **FR-2.** The system MUST provide a `notification_preferences` table: `user_id`, `event_type`, `email_enabled` (boolean), `push_enabled` (boolean), `digest_mode` (`instant` / `daily` / `off`).
- **FR-3.** The system MUST check notification preferences before enqueuing an email — if `email_enabled = false` for the event type, no job is enqueued.
- **FR-4.** The system MUST render HTML email templates using a templating engine (e.g., Tera or Handlebars in Rust); templates stored in `server/src/templates/email/`.
- **FR-5.** The system MUST apply org branding (logo, primary color, display name from 5.7) to all outgoing emails for org-scoped users.
- **FR-6.** The system MUST support SMTP delivery configuration (host, port, TLS, username/password) via environment variables, with SendGrid/Resend as a drop-in alternative.
- **FR-7.** The system SHOULD implement a daily digest mode: instead of instant emails, queue up events and send one summary email per day per user.
- **FR-8.** The system SHOULD provide an unsubscribe link in every email footer that, when clicked, disables the notification type for that user.

## 6. Non-Functional Requirements

- **Performance** — email jobs must be processed within 30 seconds of enqueue under normal load; daily digest job runs at 07:00 user-local-timezone.
- **Security** — unsubscribe tokens must be cryptographically signed and one-time-use; email content must not contain raw SQL or sensitive system internals.
- **Privacy & Compliance** — CAN-SPAM: every email must include sender address, subject, and unsubscribe mechanism. GDPR: users have the right to opt out of all non-essential emails. FERPA: grade notifications must not include other students' data.
- **Accessibility** — HTML emails must use semantic markup; link text descriptive; minimum 14 pt font; alt text on images.
- **Scalability** — email throughput: ≥1,000 emails/minute via job workers; job queue back-pressure if provider rate-limits.
- **Reliability** — email job must retry up to 3 times with exponential back-off on provider errors; dead-letter queue for permanently failed jobs.
- **Observability** — log `email_job_id`, `event_type`, `recipient_user_id`; track delivery success/bounce/failure metrics; alert on sustained delivery failure rate >5%.
- **Maintainability** — notification event emission in a central `server/src/services/notifications.rs`; templates version-controlled in `server/src/templates/email/`.
- **Internationalization** — template strings externalized for future translation (6.10); date/time formatted to user locale.
- **Backward compatibility** — password_reset email moves to the new pipeline; no behavior change for users.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student has `grade_posted` email enabled, *When* an instructor posts a grade, *Then* the student receives an email within 30 seconds containing the course name and assignment name.
- **AC-2.** *Given* a user clicks the unsubscribe link in an email, *When* they confirm, *Then* `notification_preferences.email_enabled = false` for that event type and no further emails of that type are sent.
- **AC-3.** *Given* a user's `digest_mode = 'daily'`, *When* 5 notification events occur during the day, *Then* a single digest email is sent at 07:00 summarizing all 5 events.
- **AC-4.** *Given* a provider SMTP error, *When* the email job fails, *Then* it is retried after 30 s, 2 min, and 10 min; after 3 failures it moves to the dead-letter queue.
- **AC-5.** *Given* an org with custom branding (logo, primary color), *When* a grade notification email is sent to a user in that org, *Then* the email contains the org logo and primary color in the header.
- **AC-6.** *Given* a user with `email_enabled = false` for `assignment_created`, *When* an assignment is created in their course, *Then* no email job is enqueued for that user.

## 8. Data Model

```sql
-- server/migrations/084_notification_preferences.sql
CREATE TABLE notification_preferences (
  user_id      UUID NOT NULL REFERENCES users(id),
  event_type   TEXT NOT NULL,
  email_enabled BOOLEAN NOT NULL DEFAULT true,
  push_enabled  BOOLEAN NOT NULL DEFAULT true,
  digest_mode  TEXT NOT NULL DEFAULT 'instant'
                CHECK (digest_mode IN ('instant','daily','off')),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, event_type)
);

CREATE TABLE email_jobs (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  recipient_id  UUID NOT NULL REFERENCES users(id),
  event_type    TEXT NOT NULL,
  subject       TEXT NOT NULL,
  template      TEXT NOT NULL,
  template_vars JSONB NOT NULL DEFAULT '{}',
  status        TEXT NOT NULL DEFAULT 'pending'
                  CHECK (status IN ('pending','sent','failed','dead')),
  attempts      INTEGER NOT NULL DEFAULT 0,
  next_retry_at TIMESTAMPTZ,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  sent_at       TIMESTAMPTZ
);

CREATE INDEX idx_email_jobs_status ON email_jobs(status, next_retry_at)
  WHERE status IN ('pending','failed');
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Email provider rate limiting causes delayed delivery | M | M | Back-pressure in job queue; provider tier upgrade path |
| Notification storm on course creation (100 students get emails) | H | M | Fan-out in job queue, not in-request; deduplicate within 5-minute window |
| Unsubscribe token brute-forced | L | M | 256-bit HMAC-signed token; single-use |
| Email marked as spam due to shared IP | M | M | Use reputable managed provider; set up SPF/DKIM/DMARC |

## 15. Rollout Plan

- Feature flag: `email_notifications_enabled` (default: `false`).
- Sequence: job queue (17.3 prerequisite) → notification service → templates → preferences UI → preference API → flag.
- Dogfood: enable on dev org; confirm delivery via test SMTP (MailHog in dev docker-compose).
- GA criteria: grade_posted email delivered within 30 s in staging; unsubscribe link works; bounce handling confirmed.
- Rollback: disable flag; no emails sent; preferences table data preserved.
