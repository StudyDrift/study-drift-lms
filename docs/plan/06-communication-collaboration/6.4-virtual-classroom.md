## 1. Problem Statement

Every post-pandemic LMS is expected to provide an in-platform way to launch synchronous video sessions. Instructors currently must use an external Zoom or Meet link pasted into a content page — there is no LTI deep-link integration, no calendar-aware scheduling, no attendance recording, and no session recording captured alongside the course. The absence of a native virtual classroom integration is a BLOCKER for K-12 districts (which often mandate BBB or Zoom) and for HE institutions that require click-to-join from within the LMS.

## 2. Goals

- Integrate at least one open-source video provider (BigBlueButton) natively and provide a pluggable adapter for Jitsi Meet, Zoom, and Google Meet.
- Allow instructors to schedule virtual classroom sessions linked to a course or section with automatic calendar entries.
- Provide a join-meeting button inside the course UI that launches the session in a new tab or embedded iframe.
- Record session attendance (who joined and when) and store it alongside the course.
- Support LTI 1.3 deep-link launch for provider integrations that support it (Zoom Education, BBB LTI).

## 3. Non-Goals

- Building a custom video conferencing engine — we integrate with existing providers only.
- Recording storage and transcription — deferred to 8.1 (object storage) and 8.4 (auto-captioning).
- Breakout room management within the LMS UI — handled in provider's own interface.
- Live streaming to external audiences — deferred.
- Real-time whiteboard inside the LMS (uses provider's own tools) — collaborative whiteboard as a standalone feature is 6.5.

## 4. Personas & User Stories

- **As an instructor**, I want to schedule a "Monday 9 AM Live Lecture" session from within my course so that students see it in their calendar and can join with one click.
- **As a student**, I want a "Join" button on the course page when a session is active so that I do not have to search for a link.
- **As an admin**, I want to configure which video provider the org uses (BBB, Jitsi, Zoom) so that I can use our existing institutional licenses.
- **As an instructor**, I want to see attendance records from a BigBlueButton session so that I know who was present.
- **As a student**, I want an email reminder 10 minutes before a scheduled session so that I am not late.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a `virtual_meetings` table: `id`, `course_id`, `section_id` (nullable), `provider` (`bbb`, `jitsi`, `zoom`, `meet`, `custom`), `title`, `scheduled_start`, `scheduled_end`, `join_url`, `host_url`, `external_meeting_id`, `status`, `created_by`, `created_at`.
- **FR-2.** The system MUST implement a BigBlueButton API adapter in `server/src/services/video/bbb.rs` using the BBB REST API (create, join, end, getAttendees).
- **FR-3.** The system MUST implement a Jitsi Meet adapter (`server/src/services/video/jitsi.rs`) using signed JWT room tokens per the Jitsi Meet API.
- **FR-4.** The system SHOULD implement a Zoom adapter (`server/src/services/video/zoom.rs`) using the Zoom Meetings API v2 with OAuth app credentials.
- **FR-5.** The system MUST support LTI 1.3 deep-link launch for providers that expose an LTI tool (BBB, Zoom Education) per IMS Global LTI 1.3 deep linking specification.
- **FR-6.** The system MUST capture meeting attendance: on session end, poll the provider API for attendee list and store it in `meeting_attendance` table.
- **FR-7.** The system MUST show a "Live" indicator on the course page when a meeting is active and a countdown timer when a future meeting is within 30 minutes.
- **FR-8.** The system SHOULD send a push/email reminder notification (via 6.2/6.3) 10 minutes before a scheduled meeting to enrolled students.
- **FR-9.** The system MAY emit an iCalendar event (RFC 5545 `VEVENT`) for each scheduled meeting so students can add it to their personal calendar.

## 6. Non-Functional Requirements

- **Performance** — meeting creation API must respond in ≤2 s (provider API call included); join-URL generation ≤500 ms.
- **Security** — host join URLs must require instructor authentication; student join URLs are time-limited tokens (1h expiry); provider API credentials stored as org-level secrets (never in DB plaintext).
- **Privacy & Compliance** — FERPA: attendance records are educational records. Inform students when sessions may be recorded. Provider data-processing agreements required for FERPA-covered orgs (institution must sign their own DPA with Zoom/BBB).
- **Accessibility** — "Join" button is keyboard accessible with clear `aria-label`; meeting countdown uses `role="timer"` and `aria-live="polite"`.
- **Scalability** — support ≥100 concurrent active meetings per org by routing through provider infrastructure; no local video infrastructure in Lextures.
- **Reliability** — if provider API is unavailable at scheduled start, fall back to displaying the direct join URL (if pre-generated); alert admin.
- **Observability** — log `meeting_id`, `provider`, `course_id` on all meeting events; track meeting creation failure rate.
- **Maintainability** — video provider adapters behind a common Rust trait (`VideoProvider`); adding a new provider requires implementing the trait only.
- **Internationalization** — meeting UI strings externalized; dates/times in user timezone.
- **Backward compatibility** — no existing data affected; `virtual_meetings` is a new table.

## 7. Acceptance Criteria

- **AC-1.** *Given* BBB is configured for the org, *When* an instructor creates a virtual meeting via `POST /api/courses/:id/meetings`, *Then* a BBB room is created and a join URL is returned within 2 seconds.
- **AC-2.** *Given* a scheduled meeting in 10 minutes, *When* the reminder job runs, *Then* enrolled students receive a push notification "Your [Course] live session starts in 10 minutes."
- **AC-3.** *Given* a meeting is marked active, *When* a student views the course page, *Then* a green "Live — Join Now" button is displayed.
- **AC-4.** *Given* a completed BBB session, *When* the attendance poller runs, *Then* attendee records are stored in `meeting_attendance` with join/leave timestamps.
- **AC-5.** *Given* a Zoom adapter configured, *When* an instructor creates a Zoom meeting, *Then* the meeting appears on the course page and a valid Zoom join URL is generated.
- **AC-6.** *Given* a scheduled meeting, *When* `GET /api/meetings/:id/ical` is called, *Then* the response is a valid RFC 5545 `VCALENDAR` file with a `VEVENT` for the meeting time.

## 8. Data Model

```sql
-- server/migrations/086_virtual_meetings.sql
CREATE TABLE virtual_meetings (
  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  course_id           UUID NOT NULL REFERENCES courses(id) ON DELETE CASCADE,
  section_id          UUID REFERENCES course_sections(id),
  provider            TEXT NOT NULL CHECK (provider IN ('bbb','jitsi','zoom','meet','lti','custom')),
  title               TEXT NOT NULL,
  scheduled_start     TIMESTAMPTZ,
  scheduled_end       TIMESTAMPTZ,
  join_url            TEXT,
  host_url            TEXT,
  external_meeting_id TEXT,
  status              TEXT NOT NULL DEFAULT 'scheduled'
                        CHECK (status IN ('scheduled','live','ended','cancelled')),
  created_by          UUID NOT NULL REFERENCES users(id),
  created_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_meetings_course_status ON virtual_meetings(course_id, status);

CREATE TABLE meeting_attendance (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  meeting_id  UUID NOT NULL REFERENCES virtual_meetings(id),
  user_id     UUID NOT NULL REFERENCES users(id),
  joined_at   TIMESTAMPTZ NOT NULL,
  left_at     TIMESTAMPTZ,
  duration_s  INTEGER
);
```

## 9. API Surface

| Verb | Path | Auth scope | Description |
|---|---|---|---|
| POST | `/api/courses/:id/meetings` | `instructor` | Create meeting |
| GET | `/api/courses/:id/meetings` | enrolled | List course meetings |
| GET | `/api/meetings/:id/join` | enrolled | Get time-limited join URL |
| PATCH | `/api/meetings/:id` | `instructor` | Update/cancel meeting |
| GET | `/api/meetings/:id/attendance` | `instructor` | Get attendance records |
| GET | `/api/meetings/:id/ical` | enrolled | Download iCal event |

Per-org config: `POST /api/orgs/:id/video-config` (org_admin) — set provider + API credentials.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Provider API change breaks integration | M | H | Abstract behind `VideoProvider` trait; pin API versions; integration tests against mock |
| BBB server capacity exceeded | M | H | Implement provider failover; recommend scalable BBB hosting (scalelite) |
| LTI 1.3 deep-link flow complex to implement | H | M | Start with direct API integration; add LTI path in V2 |
| Attendance polling fails after session ends | M | M | Retry polling up to 3 times; flag "attendance unavailable" if all retries fail |

## 15. Rollout Plan

- Feature flag: `virtual_classroom_enabled` (default: `false`).
- Sequence: BBB adapter → meeting CRUD API → course UI integration → Jitsi adapter → Zoom adapter → attendance poller → flag.
- Dogfood: run a test BBB session in dev environment using test BBB server.
- GA criteria: BBB join works end-to-end in staging; attendance captured; reminder notification sent.
- Rollback: disable flag; virtual_meetings table preserved.

## 16. Test Plan
