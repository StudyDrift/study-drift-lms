## 1. Problem Statement

In diverse K-12 districts and international university courses, students and parents may not be fluent in the language used by their instructor. Announcements, discussion posts, and inbox messages arrive in a language the recipient cannot read. Without auto-translation, non-English-speaking students are systematically disadvantaged and parents cannot participate in their child's education. This feature adds AI-powered auto-translation of in-platform messages (feed, discussions, inbox) with a user-controlled per-message translate button.

## 2. Goals

- Allow users to request translation of any message (feed post, discussion post, inbox message) to their preferred language with a single click.
- Store translated content in a cache to avoid re-translating the same content.
- Allow admins to configure the translation provider (OpenRouter LLM, DeepL API, Google Translate API) per org.
- Respect per-user language preferences (from 11.1 i18n settings).
- Never auto-translate without user action — translation is always opt-in per message.

## 3. Non-Goals

- Real-time simultaneous interpretation of live meetings — deferred.
- Translation of file attachments (PDFs, DOCx) — deferred.
- Two-way translation of reply chains (auto-translating the reply back) — V1 is one-way per-message only.
- Replacing the i18n UI framework (11.1) — this feature covers user-generated content only, not UI strings.

## 4. Personas & User Stories

- **As a Spanish-speaking parent**, I want to click "Translate" on my child's teacher's announcement so that I can understand it without using a separate translator app.
- **As an instructor in an international course**, I want discussion posts from non-English students to be translatable to English so that I can read and grade them.
- **As a student**, I want my preferred language saved so that translated posts appear in my language automatically rather than my having to specify it each time.
- **As an admin**, I want to choose DeepL as the translation provider because our institution has an enterprise contract with them.

## 5. Functional Requirements

- **FR-1.** The system MUST add a "Translate" button to feed posts, discussion posts, and inbox messages; clicking it triggers a translation API call if no cached translation exists.
- **FR-2.** The system MUST cache translations in a `content_translations` table keyed by `(content_type, content_id, target_language)` to avoid redundant API calls.
- **FR-3.** The system MUST read the user's preferred language from their profile (`users.locale` from 11.1) as the default translation target.
- **FR-4.** The system MUST support pluggable translation providers: OpenRouter LLM (default), DeepL API, Google Cloud Translation API — configured per org via `TRANSLATION_PROVIDER` setting.
- **FR-5.** The system MUST display the translated text inline below the original (not replacing it), with a clear "Translated from [detected language]" label and a "Show original" toggle.
- **FR-6.** The system SHOULD detect the source language automatically (via the provider's language detection) rather than requiring the author to specify.
- **FR-7.** The system SHOULD track translation usage metrics (number of translations per org per month) for cost management.
- **FR-8.** The system MAY allow an org admin to pre-translate all announcements to a list of configured org languages immediately on post.

## 6. Non-Functional Requirements

- **Performance** — translation response must return within 3 s for messages under 500 words; cached translations return in ≤50 ms.
- **Security** — translation API calls must go through the server (not client-side API key exposure); PII redaction from 19.11 applied before sending content to external translation providers.
- **Privacy & Compliance** — content sent to external translation APIs is educational content; orgs must confirm a DPA with the chosen provider. FERPA: translations are cached server-side; not sent to ad networks.
- **Accessibility** — "Translate" button has `aria-label="Translate this message"`; translated text has `lang` attribute set to the target language code; "Show original" toggle is keyboard accessible.
- **Scalability** — translation cache reduces repeated API calls; large courses with popular announcements will hit cache after first translation.
- **Reliability** — if the translation provider returns an error, display "Translation unavailable" rather than an empty or error page.
- **Observability** — log `translation_requested`, `provider`, `source_lang`, `target_lang`, `cached` (bool) per event; track monthly cost.
- **Maintainability** — translation service in `server/src/services/translation.rs`; provider abstracted behind a trait.
- **Internationalization** — this feature IS the i18n content layer; target language list driven by provider support (DeepL supports 30+, Google 130+).
- **Backward compatibility** — translation is entirely additive; no changes to existing content storage.

## 7. Acceptance Criteria

- **AC-1.** *Given* a feed post in Spanish and a user with English as their preferred language, *When* the user clicks "Translate", *Then* the English translation appears below the original within 3 s.
- **AC-2.** *Given* a translation has already been cached, *When* a second user requests the same translation, *Then* the cached result is returned in ≤50 ms and no external API call is made.
- **AC-3.** *Given* a translation provider returns an error, *When* the user clicks "Translate", *Then* the message "Translation temporarily unavailable" is shown; the original text remains visible.
- **AC-4.** *Given* a translation is displayed, *When* the user clicks "Show original", *Then* the original text is shown and the translation is hidden.
- **AC-5.** *Given* the translation text, *When* the HTML is inspected, *Then* the translated paragraph has a `lang="en"` attribute (or appropriate target language code).
- **AC-6.** *Given* DeepL is configured as the provider for an org, *When* a user requests translation, *Then* the server calls the DeepL API with the org's API key (not a generic key).

## 8. Data Model

```sql
-- server/migrations/091_content_translations.sql
CREATE TABLE content_translations (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  content_type  TEXT NOT NULL CHECK (content_type IN ('feed_post','discussion_post','inbox_message','announcement')),
  content_id    UUID NOT NULL,
  source_lang   TEXT NOT NULL,
  target_lang   TEXT NOT NULL,
  translated    TEXT NOT NULL,
  provider      TEXT NOT NULL,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (content_type, content_id, target_lang)
);

CREATE INDEX idx_translations_lookup ON content_translations(content_type, content_id, target_lang);
```

## 11. AI / ML Considerations

- **Models used:** OpenRouter LLM (default) for translation if no dedicated provider configured; DeepL API or Google Cloud Translation as first-class alternatives.
- **Prompt (LLM path):** "Translate the following text to [language]. Output only the translation, no commentary."
- **Eval metric:** BLEU score on a test set of 100 course-context sentences (manual review for academic vocabulary quality).
- **Fallback:** on provider timeout, return error; show original.
- **PII redaction:** apply 19.11 before sending to external provider; particularly important for inbox messages.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Translation quality poor for technical/academic content | M | M | Use DeepL (higher quality) for orgs with contract; show BLEU eval to org admin |
| PII leaks in translated content sent to external API | M | H | Apply PII redactor before sending; log all redacted fields |
| Translation cost spikes with heavily active courses | M | M | Cache aggressively; monthly per-org usage limit with alert |
| Content translation used for academic dishonesty (translate assignment prompt to another language for submission) | L | M | Translations are read-only display; student assignment submission goes through plagiarism check (3.5) |

## 15. Rollout Plan

- Feature flag: `multilingual_messaging_enabled` (default: `false`).
- Sequence: translation service → cache table → `POST /api/translate` endpoint → TranslateButton component → flag.
- Dogfood: test with a Spanish-language test post in dev environment; verify English translation.
- GA criteria: translation returned within 3 s for 500-word post; cache hit confirmed on second request.
- Rollback: disable flag; cache table remains but is inert.

## 16. Test Plan
