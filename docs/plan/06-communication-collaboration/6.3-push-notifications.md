## 1. Problem Statement

Email notifications (6.2) cover async communication, but students on mobile devices expect real-time push alerts: a grade posted while riding the bus, a new discussion reply from a peer, an assignment reminder. Without web push and mobile push, Lextures has no way to reach users who are not actively looking at the application. Self-learners especially rely on push reminders to maintain study streaks. This feature adds Web Push (RFC 8030 + VAPID) for browser notifications and lays the groundwork for APNs/FCM via 7.5.

## 2. Goals

- Implement Web Push (RFC 8030 / VAPID) for browser-based push notifications in the Vite/React frontend.
- Store push subscriptions per user-device in the database.
- Re-use the `notification_preferences` table from 6.2 to honor per-user, per-channel, per-event-type settings.
- Deliver web push notifications for the same 8 critical events as email (6.2).
- Provide a notification bell UI in the app header with an unread count and notification list.

## 3. Non-Goals

- APNs (Apple Push Notification service) or FCM (Firebase Cloud Messaging) for native mobile â€” those are deferred to 7.5 which depends on this plan.
- In-app real-time WebSocket notification (different from push) â€” can be addressed alongside but is a separate concern.
- Push notification for marketing/promotional messages â€” transactional only.

## 4. Personas & User Stories

- **As a student**, I want a browser notification when a grade is posted so that I know immediately even if the Lextures tab is in the background.
- **As an instructor**, I want a push notification when a student submits an assignment so that I can start grading without periodically refreshing the page.
- **As a student**, I want to see a notification bell count in the app so that I can see how many new events have occurred since I last checked.
- **As a user**, I want to opt out of push notifications per event type so that I am not interrupted by low-priority events.
- **As a self-learner**, I want a daily study reminder push notification so that I stay consistent with my learning schedule.

## 5. Functional Requirements

- **FR-1.** The system MUST implement a Web Push subscription flow: request browser permission â†’ generate VAPID key pair â†’ store `PushSubscription` JSON in `push_subscriptions` table.
- **FR-2.** The system MUST generate a VAPID key pair at application startup and expose the public key via `GET /api/push/vapid-public-key` for the service worker to use.
- **FR-3.** The system MUST register a Service Worker (`clients/web/public/sw.js`) that handles incoming push events and displays browser notifications via the Notifications API.
- **FR-4.** The system MUST enqueue a push notification job (via 17.3 job queue) for each qualifying event, checking `notification_preferences.push_enabled` before enqueue.
- **FR-5.** The system MUST implement in-app notification inbox: `notifications` table storing each notification event per user; `GET /api/me/notifications` returns paginated list; `POST /api/me/notifications/:id/read` marks as read.
- **FR-6.** The system MUST display a notification bell icon in the app header with unread count badge; clicking opens a dropdown notification list.
- **FR-7.** The system SHOULD use WebSocket or Server-Sent Events to update the bell count in real time when a new notification arrives.
- **FR-8.** The system SHOULD allow the user to dismiss all notifications in one click.

## 6. Non-Functional Requirements

- **Performance** â€” push job must be delivered within 5 seconds of event under normal load; in-app bell count update via SSE latency â‰¤2 s.
- **Security** â€” VAPID authentication required per RFC 8030; push subscription endpoints require authenticated user; subscriptions are per-user-device, not shareable.
- **Privacy & Compliance** â€” browser push permission must be explicitly requested with a clear opt-in prompt (not auto-prompted on load per browser UX guidelines). Push payloads must not include raw student grade data â€” use generic messages linking back to the app.
- **Accessibility** â€” notification bell has `aria-label="Notifications (N unread)"`; notification dropdown is keyboard navigable; focus returns to bell after close.
- **Scalability** â€” a user may have up to 5 active push subscriptions (multiple devices/browsers); fan-out job sends to all active subscriptions.
- **Reliability** â€” push jobs retry up to 3 times; if subscription returns HTTP 410 (Gone), it is auto-deleted.
- **Observability** â€” log `push_job_id`, `user_id`, `event_type`, delivery status; track subscription churn rate; alert on >10% delivery failure rate.
- **Maintainability** â€” push delivery service in `server/src/services/push_notifications.rs`.
- **Internationalization** â€” push notification message strings must be localizable; currently use English only with i18n keys ready.
- **Backward compatibility** â€” push is opt-in; no change to existing users who do not subscribe.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student has granted browser push permission, *When* an instructor posts a grade, *Then* a browser push notification appears within 5 seconds with the text "[Course Name]: Your grade has been posted."
- **AC-2.** *Given* a push subscription endpoint returns HTTP 410, *When* the push job worker processes it, *Then* the subscription is deleted from `push_subscriptions` and no further pushes are attempted to it.
- **AC-3.** *Given* a user has `push_enabled = false` for `grade_posted`, *When* a grade is posted, *Then* no push job is enqueued for that user.
- **AC-4.** *Given* 3 unread notifications, *When* the user opens the notification bell, *Then* they see a list of 3 notifications; after clicking "Mark all read", the badge count returns to 0.
- **AC-5.** *Given* VAPID keys are configured, *When* `GET /api/push/vapid-public-key` is called, *Then* the base64url-encoded public key is returned without authentication.
- **AC-6.** *Given* a user has 3 active subscriptions on different devices, *When* a notification event occurs, *Then* push jobs are enqueued for all 3 subscriptions.

## 8. Data Model

```sql
-- server/migrations/085_push_notifications.sql
CREATE TABLE push_subscriptions (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      UUID NOT NULL REFERENCES users(id),
  endpoint     TEXT NOT NULL,
  p256dh_key   TEXT NOT NULL,
  auth_secret  TEXT NOT NULL,
  user_agent   TEXT,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  last_used_at TIMESTAMPTZ,
  UNIQUE (user_id, endpoint)
);

CREATE INDEX idx_push_subs_user ON push_subscriptions(user_id);

CREATE TABLE notifications (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id    UUID NOT NULL REFERENCES users(id),
  event_type TEXT NOT NULL,
  title      TEXT NOT NULL,
  body       TEXT NOT NULL,
  action_url TEXT,
  is_read    BOOLEAN NOT NULL DEFAULT false,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_notifications_user ON notifications(user_id, is_read, created_at DESC);
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Browser blocks auto-permission prompt | H | M | Only request permission on explicit user action (settings or bell click) |
| Push subscription becomes stale (410 Gone) | H | L | Auto-delete subscription on 410 response; document cleanup job |
| SSE connection pool exhausted | M | M | Cap SSE connections per server process; use Redis pub/sub for fan-out at scale |
| VAPID keys rotated causing all subscriptions to fail | L | H | Rotate keys only with advance notice; re-subscription prompt to users |
