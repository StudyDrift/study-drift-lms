## 1. Problem Statement

The `assignment_groups` table (seeded in migration 054) supports flat percentage weighting per group. A nearly universal grading policy in HE and K-12 allows instructors to drop the lowest N quiz/homework scores — or replace a low exam score with a later score — before computing the group average. Without this, final course grades computed in Lextures diverge from what instructors advertise in their syllabi, destroying trust in the gradebook and forcing instructors to maintain parallel spreadsheets.

## 2. Goals

- Add `drop_lowest`, `drop_highest`, and `never_drop` policies to each assignment group.
- Compute the effective group average excluding dropped assignments when calculating the course total.
- Show students which of their scores were dropped in the My Grades view.
- Support "rule of replacement": optionally replace a lower early-exam score with the final exam score if the final is higher.

## 3. Non-Goals

- Complex conditional dropping rules beyond drop-N lowest/highest and replace-one.
- Dropping based on standards alignment or mastery level.

## 4. Personas & User Stories

- **As an instructor**, I want to configure "drop lowest 1 quiz" in the Quizzes assignment group so that missing or low-scoring quizzes do not unfairly penalize students.
- **As a student**, I want to see in My Grades that Quiz 3 was dropped and explains my final quiz average so that I trust the calculated grade.
- **As an instructor**, I want to set "never drop" on specific critical assignments so that they are always included regardless of a group drop rule.

## 5. Functional Requirements

- **FR-1.** The system MUST add `drop_lowest INT DEFAULT 0`, `drop_highest INT DEFAULT 0` fields to the `assignment_groups` table.
- **FR-2.** When computing the group average for a student, the system MUST sort the student's scores within the group and exclude the lowest N and/or highest N scores per the configured drop rules.
- **FR-3.** The system MUST support an `never_drop` flag per individual assignment that prevents it from being excluded regardless of group drop rules.
- **FR-4.** The computed group average (after dropping) MUST be used in all final course grade calculations and displayed in the gradebook.
- **FR-5.** The system MUST indicate to the student in the My Grades view which individual scores were dropped.
- **FR-6.** The system MUST handle the case where drop_lowest > number of submitted assignments (drop as many as available, not more).
- **FR-7.** The system SHOULD support a "replace lowest with final exam" variant: if `replace_with_final = true` on a group and the student's lowest score is below the score of a designated "final" assignment, replace it.

## 6. Non-Functional Requirements

- **Performance** — Drop calculation is in-memory after fetching the group's scores (O(n log n) sort); adds < 1 ms per group per student.
- **Security** — Group configuration changes require instructor/owner role.
- **Privacy & Compliance** — Dropped scores are still FERPA records; dropping affects display only, not storage. Original scores retained.
- **Accessibility** — "Dropped" indicator in My Grades has `aria-label="Score dropped by course policy"`. WCAG 2.1 AA.
- **Scalability** — Pure computation; no additional DB queries per student.
- **Reliability** — Re-computation is idempotent; drop selection re-evaluated each time final grade is requested.
- **Observability** — Log `assignment_group_drop_policy_updated` with group_id and new values.
- **Maintainability** — Drop logic in `server/src/services/grading/assignment_groups.rs`; isolated pure function.
- **Internationalization** — "Dropped" label i18n-keyed.
- **Backward compatibility** — Default `drop_lowest = 0`, `drop_highest = 0`; all existing groups compute as today.

## 7. Acceptance Criteria

- **AC-1.** *Given* a Quizzes group with `drop_lowest = 1` and a student scoring [60, 70, 80, 90], *when* the group average is computed, *then* the dropped score is 60 and the average is (70+80+90)/3 = 80.
- **AC-2.** *Given* an assignment with `never_drop = true` in a group with `drop_lowest = 1`, *when* that assignment has the lowest score for a student, *then* it is NOT dropped; instead the next-lowest eligible score is dropped.
- **AC-3.** *Given* a student who has only submitted 2 assignments in a group with `drop_lowest = 3`, *when* the grade is computed, *then* both submitted scores are dropped and the group contributes 0% to the final grade (or shown as no qualifying submissions).
- **AC-4.** *Given* drop_lowest = 1 applied to a student's scores, *when* the student opens My Grades, *then* the dropped assignment row is visually marked as "Dropped" with the score still shown.
- **AC-5.** *Given* the instructor changes `drop_lowest` from 1 to 2, *when* the gradebook refreshes, *then* all students' group averages recalculate immediately.

## 8. Data Model

```sql
-- Migration: server/migrations/081_assignment_group_drop_rules.sql

ALTER TABLE assignment_groups
    ADD COLUMN drop_lowest  INT NOT NULL DEFAULT 0,
    ADD COLUMN drop_highest INT NOT NULL DEFAULT 0;

ALTER TABLE module_assignments
    ADD COLUMN never_drop BOOLEAN NOT NULL DEFAULT FALSE,
    ADD COLUMN replace_with_final BOOLEAN NOT NULL DEFAULT FALSE;
```

No new tables. Drop computation is runtime, not stored.

## 9. API Surface

Modified:
- `GET /api/courses/:cid/grades` — group average now uses drop policy.
- `PATCH /api/courses/:cid/assignment-groups/:gid` — accept `drop_lowest`, `drop_highest`.
- `PATCH /api/courses/:cid/assignments/:aid` — accept `never_drop`, `replace_with_final`.

My Grades response per assignment cell:
```ts
{ score: number; display_grade: string; dropped: boolean }
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Student disputes which score was dropped | M | M | Show dropped indicator clearly; log drop decision in API response |
| Drop count > submitted count causes NaN average | M | H | Guard: cap drops at min(drop_n, submitted_count - 1) so at least 1 score remains |
