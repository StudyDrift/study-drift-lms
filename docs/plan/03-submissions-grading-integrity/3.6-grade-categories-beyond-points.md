## 1. Problem Statement

The `course_grades` table stores a numeric `score` only. Instructors in K-12 and HE routinely grade using letter grades with custom cutoffs, GPA scales, pass/fail criteria, and complete/incomplete states. Without these modes, Lextures cannot serve any course that uses a non-numeric grading scheme — effectively blocking adoption in language arts, music, physical education, competency-based programs, and graduate courses where pass/fail is standard. This is currently listed as THIN because a numeric score exists, but no conversion, display, or entry mechanism for non-point grading modes is present.

## 2. Goals

- Allow courses to configure a grading scheme: points, percentage, letter grade (with custom scale), GPA (4.0, 5.0, etc.), pass/fail, complete/incomplete.
- Convert stored numeric scores to the configured display type for the gradebook and student view.
- Allow direct entry of letter-grade or pass/fail values in the gradebook that map to a numeric score range.
- Support assignment-level grading type override (e.g., most assignments are points but one is pass/fail).
- Store configured grading scales under the course settings.

## 3. Non-Goals

- Standards-based / mastery grading (addressed in 3.7).
- Narrative-only grading mode (no numeric equivalent) — deferred.
- GPA calculation across multiple courses (registrar-level feature).
- Automatic grade conversion to institutional SIS format (14.5).

## 4. Personas & User Stories

- **As an instructor**, I want to configure my course to use a letter grade scale (A ≥ 90%, B ≥ 80%, …) so that my gradebook shows letters rather than raw percentages.
- **As a graduate advisor**, I want to mark a thesis defense as "Pass" or "Fail" rather than entering a point value so that the system reflects how the program actually works.
- **As a K-12 instructor**, I want to enter "Complete" or "Incomplete" for a homework assignment so that parents and students see a status rather than an arbitrary score.
- **As a student**, I want to see my letter grades in My Grades so that I understand my standing in terms meaningful to my program.

## 5. Functional Requirements

- **FR-1.** The system MUST support the following grading display types at the course level: `points`, `percentage`, `letter`, `gpa`, `pass_fail`, `complete_incomplete`.
- **FR-2.** For `letter` and `gpa` types, the system MUST allow instructors to define custom cutoff scales (e.g., A: 93–100, A-: 90–92.99) with configurable plus/minus bands.
- **FR-3.** The system MUST convert stored numeric scores to the configured display type when rendering the gradebook and student My Grades page.
- **FR-4.** The gradebook MUST allow direct letter/pass/fail input that maps to the midpoint of the corresponding numeric range for storage.
- **FR-5.** Each individual assignment MUST support an optional `grading_type` override that takes precedence over the course-level setting.
- **FR-6.** The system MUST recalculate displayed grades when the grading scale is updated, without altering stored numeric scores.
- **FR-7.** The system SHOULD provide a default letter-grade scale (A/B/C/D/F with standard US cutoffs) as a starting template.
- **FR-8.** The system MUST NOT lose the underlying numeric score when the display type is changed (display is a view transformation, not data mutation).

## 6. Non-Functional Requirements

- **Performance** — Grade display conversion is a pure function applied at render time; no additional DB queries. Gradebook load p95 unchanged.
- **Security** — Grading scale configuration requires instructor/owner role. Stored in `course_settings` table under auth controls in `server/src/authz.rs`.
- **Privacy & Compliance** — Letter grades constitute FERPA education records same as numeric scores. FERPA data export (DSAR) must include display type alongside raw score.
- **Accessibility** — Letter grade inputs use `<select>` or combobox; pass/fail uses radio group; all labelled for screen readers. WCAG 2.1 AA.
- **Scalability** — Scale definitions are course-level config fetched once per gradebook session; minimal performance impact.
- **Reliability** — Scale change recalculates displays lazily on read; no bulk job needed. No stored display values to become stale.
- **Observability** — Log `grading_type_updated` event with `course_id`, old type, new type.
- **Maintainability** — Grade conversion logic in `server/src/services/grading/conversion.rs`; a pure function `to_display_grade(score, scale)` easy to unit test.
- **Internationalization** — Grade labels configurable; default labels i18n-keyed (e.g., "Pass"/"Fail" → "Aprobado"/"Reprobado").
- **Backward compatibility** — Existing courses default to `points`; all stored scores preserved exactly.

## 7. Acceptance Criteria

- **AC-1.** *Given* a course configured with letter grades (A ≥ 93), *when* a student's score is 95/100, *then* the gradebook and My Grades display "A."
- **AC-2.** *Given* an assignment configured as pass/fail with cutoff 60%, *when* the instructor enters "Fail" in the gradebook, *then* the stored numeric score is 0 (or the midpoint below cutoff) and the display remains "Fail."
- **AC-3.** *Given* the instructor updates the letter-grade cutoffs (changes B cutoff from 80 to 82), *when* the gradebook is refreshed, *then* all displayed grades recalculate using the new scale without changing stored scores.
- **AC-4.** *Given* an assignment with `grading_type = 'complete_incomplete'` in a course using points, *when* the instructor views that assignment column in the gradebook, *then* the column shows "Complete"/"Incomplete" cells rather than numeric values.
- **AC-5.** *Given* a points-type course, *when* no custom grading scheme is configured, *then* all existing gradebook behaviour is unchanged.
- **AC-6.** *Given* a custom letter scale, *when* the bulk CSV export is run (3.11), *then* both the raw numeric score and the computed letter grade are included as separate columns.

## 8. Data Model

```sql
-- Migration: server/migrations/078_grading_schemes.sql

CREATE TABLE grading_schemes (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    course_id   UUID NOT NULL REFERENCES courses(id) ON DELETE CASCADE,
    name        TEXT NOT NULL DEFAULT 'Default',
    type        TEXT NOT NULL CHECK (type IN
                    ('points','percentage','letter','gpa','pass_fail','complete_incomplete')),
    scale_json  JSONB,   -- [{label: 'A', min_pct: 93.0}, ...]
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

ALTER TABLE courses
    ADD COLUMN grading_scheme_id UUID REFERENCES grading_schemes(id);

ALTER TABLE module_assignments
    ADD COLUMN grading_type TEXT
        CHECK (grading_type IN ('points','percentage','letter','gpa','pass_fail','complete_incomplete'));
-- NULL = inherit from course grading_scheme
```

## 9. API Surface

| Verb | Path | Auth | Description |
|---|---|---|---|
| GET | `/api/courses/:cid/grading-scheme` | enrolled | Get course grading scheme |
| PUT | `/api/courses/:cid/grading-scheme` | instructor/owner | Create or update scheme |
| GET | `/api/courses/:cid/grades` | instructor | Gradebook: includes display_grade per cell |

Assignment endpoints already accept `grading_type` in the update payload.

Response shape for grade cell:
```ts
{ score: number; display_grade: string; grading_type: string }
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Rounding edge cases at grade boundary | H | M | Unit test boundary values; document rounding rule (round half up) |
| Existing integrations expect numeric only | M | M | Always return `score` alongside `display_grade`; API backward compatible |
| Custom scales with gaps or overlaps | M | M | Validate scale_json on save: ranges must be contiguous and non-overlapping |

## 15. Rollout Plan

- Feature flag: `GRADING_SCHEMES_ENABLED` (default: `true` for new courses, `false` for existing to avoid visual change).
- Schema migration → existing courses get `grading_scheme_id = NULL` (display as points).
- Enable UI in course settings; existing gradebook unaffected until instructor opts in.
- GA criteria: unit test coverage of conversion function ≥ 95%; no regression in existing gradebook integration tests.

## 16. Test Plan

- **Unit** — `to_display_grade()` for all types; boundary values; custom scale validation.
- **Integration** — Grading scheme CRUD API; gradebook endpoint returns display_grade.
- **End-to-end** — Playwright: set letter scale → enter score → verify display; change scale → verify recalculation.
- **Accessibility** — axe-core on gradebook with letter-grade select cells.
- **Manual exploratory** — GPA scale with plus/minus; edge score exactly at boundary.

## 18. Open Questions

1. Should we allow multiple named grading schemes per institution (e.g., "Graduate" vs. "Undergraduate")?
2. For GPA type: which scale is the default? 4.0 US, 7.0 AU, 20-point FR?
3. Should `complete_incomplete` honour a configurable cut-off percentage or always require manual instructor entry?
