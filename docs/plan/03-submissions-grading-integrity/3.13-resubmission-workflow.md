## 1. Problem Statement

Once a student submits an assignment in Lextures, they cannot resubmit unless the instructor explicitly reopens the assignment. There is no workflow for an instructor to send a structured "revision request" — with inline feedback and a deadline — back to the student, who can then revise and resubmit. This prevents use of iterative writing workshops, peer-review revision cycles, and graduate-level thesis draft workflows. It is a particularly visible gap for courses centred on writing and project work.

## 2. Goals

- Allow instructors to request a revision from a student with inline feedback and an optional extension deadline.
- Allow students who receive a revision request to resubmit within the deadline.
- Maintain a version history of all submissions (original + resubmissions) linked to the assignment.
- Automatically re-trigger plagiarism/AI detection (3.5) on each resubmission.
- Log revision requests and resubmissions in the grade-change audit trail (3.10).

## 3. Non-Goals

- Fully peer-review / group revision workflows.
- Unlimited free resubmissions without an explicit instructor request.
- Resubmission of quiz attempts (separate policy in 2.9).

## 4. Personas & User Stories

- **As an instructor**, I want to mark a submission with "Request revision" and specify a deadline so that the student knows they have 48 hours to resubmit.
- **As a student**, I want to receive a clear notification that my instructor has requested a revision with their comments so that I know what to fix and when to resubmit.
- **As an instructor**, I want to see all versions of a student's submission side by side so that I can compare the original with the revision.
- **As a student**, I want to see all my submission versions in My Grades so that I understand my revision history.

## 5. Functional Requirements

- **FR-1.** The system MUST add a `resubmission_requested` status to submission records, set by an instructor action.
- **FR-2.** When a revision is requested, the system MUST allow the instructor to specify an optional `revision_due_at` timestamp.
- **FR-3.** The system MUST notify the student (via in-app notification; email if 6.2 is shipped) when a revision is requested, including the instructor's feedback summary and deadline.
- **FR-4.** Students MUST be allowed to resubmit only when a revision is explicitly requested and the `revision_due_at` has not passed.
- **FR-5.** Each submission (original + resubmissions) MUST be stored as a versioned record, all linked to the assignment.
- **FR-6.** The system MUST allow the instructor to grade any version of a submission; the most recent graded version takes precedence for the course grade.
- **FR-7.** The system MUST trigger originality detection (3.5) on each resubmission automatically.
- **FR-8.** The system MUST create a `grade_audit_events` record for each revision request and resubmission.

## 6. Non-Functional Requirements

- **Performance** — Revision request action completes in ≤ 300 ms. Submission version listing returns all versions for an assignment/student within ≤ 200 ms.
- **Security** — Only instructor/owner can request revision; students can only resubmit within the allowed window. File uploads on resubmission go through the same validation as original submissions.
- **Privacy & Compliance** — All submission versions are FERPA records; retained on the same schedule. Revision request feedback is also an education record.
- **Accessibility** — Revision request status shown with text label + icon; notification content readable by screen readers. WCAG 2.1 AA.
- **Scalability** — Versioned submissions stored in object storage (8.1); version metadata in `submission_versions` table.
- **Reliability** — Revision deadline enforced server-side; client timer is advisory only.
- **Observability** — Log `revision_requested`, `resubmission_received` events with `assignment_id`, `student_id`.
- **Maintainability** — Resubmission service in `server/src/services/grading/resubmission.rs`.
- **Internationalization** — Status labels and notification copy i18n-keyed.
- **Backward compatibility** — No change to existing submission behaviour; resubmission only activates on explicit request.

## 7. Acceptance Criteria

- **AC-1.** *Given* the instructor clicks "Request revision" for a student's submission and sets a deadline, *when* the action is saved, *then* the submission status changes to "Revision requested" and the student receives an in-app notification.
- **AC-2.** *Given* a revision has been requested, *when* the student opens the assignment, *then* they see a "Resubmit" button and the revision deadline prominently displayed.
- **AC-3.** *Given* the revision deadline has passed, *when* the student attempts to resubmit via the API, *then* the server returns HTTP 422 "Revision deadline has passed."
- **AC-4.** *Given* a student resubmits, *when* the instructor opens the grader panel, *then* they see both "Version 1" and "Version 2" with timestamps and can toggle between them.
- **AC-5.** *Given* a resubmission, *when* plagiarism detection (3.5) is enabled on the assignment, *then* a new detection job is automatically enqueued for the new version.
- **AC-6.** *Given* a revision request is created, *then* a `grade_audit_events` record with `action = 'revision_requested'` is written.

## 8. Data Model

```sql
-- Migration: server/migrations/084_submission_versions.sql

CREATE TABLE submission_versions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    assignment_id   UUID NOT NULL REFERENCES module_assignments(id) ON DELETE CASCADE,
    student_id      UUID NOT NULL REFERENCES users(id),
    version_number  INT NOT NULL DEFAULT 1,
    storage_key     TEXT,
    text_body       TEXT,
    submitted_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (assignment_id, student_id, version_number)
);

ALTER TABLE module_assignments
    ADD COLUMN resubmission_requested BOOLEAN NOT NULL DEFAULT FALSE,
    ADD COLUMN revision_due_at TIMESTAMPTZ,
    ADD COLUMN revision_feedback TEXT;
-- Per-student revision state tracked via latest submission_version + assignment status
```

## 9. API Surface

| Verb | Path | Auth | Description |
|---|---|---|---|
| POST | `/api/courses/:cid/assignments/:aid/submissions/:sid/request-revision` | instructor | Request revision with optional deadline and feedback |
| GET | `/api/courses/:cid/assignments/:aid/submissions/:sid/versions` | instructor/student-self | List submission versions |
| POST | `/api/courses/:cid/assignments/:aid/resubmit` | student (own) | Submit new version |

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Students exploit resubmission for unlimited retries | M | M | Server-side: only allow resubmit when `resubmission_requested = true`; not self-service |
| Version storage grows large for long iterative courses | L | M | Object storage (8.1) handles at-scale; version count capped at 10 per assignment |

## 16. Test Plan

- **Unit** — Deadline enforcement; version_number incrementing; revision status state machine.
- **Integration** — Request revision API; resubmit within/outside deadline; version list API.
- **End-to-end** — Playwright: instructor requests revision → student sees notification → student resubmits → instructor sees version 2.
- **Security** — Student cannot call request-revision; cannot resubmit past deadline; version endpoint scoped to own records.
- **Accessibility** — axe-core on version switcher and CTA button.
