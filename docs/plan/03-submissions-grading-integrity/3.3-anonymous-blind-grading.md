## 1. Problem Statement

Instructors at universities grading papers, exams, and creative work risk unconscious bias when they can see a student's name. Blind grading — hiding the student's identity from the grader until scores are finalized — is a standard requirement in many HE institutions and is mandated by some academic integrity policies. Lextures currently shows full student names and avatars on every submission in the grader panel. Without a blind grading flag, Lextures fails peer-reviewed journals, university writing programs, and any institution with a published anti-bias grading policy.

## 2. Goals

- Add a per-assignment blind grading toggle that hides student identity (name, avatar, email, student ID) from all graders until grades are posted.
- Preserve the grader's ability to assign a score and annotations without knowing whose work they are reviewing.
- Allow the instructor-of-record to unmask identity after all grades are entered.
- Audit-log all unmask events for FERPA compliance.
- Keep the implementation simple and additive to the current gradebook flow.

## 3. Non-Goals

- Double-blind grading (hiding identity from the instructor-of-record as well) — addressed in 3.4 Moderated Grading.
- Anonymous surveys or peer review (separate features).
- Randomising submission order (enhancement, not required for MVP).

## 4. Personas & User Stories

- **As an instructor**, I want to enable blind grading on my final essay assignment so that I score submissions based purely on content without knowing the author.
- **As an instructor**, I want to click "Reveal identities" after I have finished grading all submissions so that I can confirm scores and return feedback with student names.
- **As a department chair**, I want to know that blind grading was enabled and when identities were revealed so that I can satisfy our academic integrity audit.
- **As a student**, I want my identity hidden during the grading process so that I have confidence my grade reflects the quality of my work alone.

## 5. Functional Requirements

- **FR-1.** The system MUST add a `blind_grading` boolean flag to the assignment settings (default: `false`).
- **FR-2.** When blind grading is enabled, the system MUST replace the student's name and avatar with an anonymised label (e.g., "Student 1," "Student 2") in the grader panel, submission navigator, and gradebook until the instructor reveals identities.
- **FR-3.** The system MUST ensure the anonymised labels are consistent across grader sessions (same student always maps to the same label for a given assignment).
- **FR-4.** The system MUST NOT expose the student identifier in any API response visible to a grader role while blind grading is active.
- **FR-5.** The system MUST provide an instructor-only "Reveal all identities" action that unmasks names after all submissions are graded (or at instructor discretion).
- **FR-6.** The system MUST record an audit event whenever identities are revealed, capturing `assignment_id`, `instructor_id`, and timestamp.
- **FR-7.** The system SHOULD prevent the instructor from revealing identities while any submission is still ungraded, with a confirmation override.
- **FR-8.** The system MAY allow blind grading to be set as a course-level default inherited by all new assignments.

## 6. Non-Functional Requirements

- **Performance** — Anonymised label lookup is a deterministic hash/rank; no additional DB queries beyond existing submission fetch. p95 grader load ≤ 200 ms unchanged.
- **Security** — Server-side enforcement: grader API endpoint for submission list MUST redact `student_name`, `student_email`, `student_id` fields when `blind_grading = true` and grades not yet posted. Redaction happens in the route handler in `server/src/routes/courses.rs`, not only in the frontend.
- **Privacy & Compliance** — Blind grading audit events are FERPA-significant. Reveal events stored in the grade-change audit trail (3.10). FERPA compliance requires that only authorized school officials can unmask.
- **Accessibility** — Anonymised labels follow the same visual design as student names; screen readers announce "Student 1" not icon placeholders. WCAG 2.1 AA.
- **Scalability** — Stateless label computation (rank by `submission.created_at` within assignment); no new hot-path tables.
- **Reliability** — If the label computation is changed, existing revealed-state audit logs remain unchanged; labels are derived at query time, not stored.
- **Observability** — Log `blind_grading_revealed` event with `assignment_id` and `instructor_id`.
- **Maintainability** — Redaction logic in a helper function `redact_submission_for_blind_grading()` in `server/src/routes/` for testability.
- **Internationalization** — "Student N" label format i18n-keyed for localisation.
- **Backward compatibility** — Existing assignments unaffected; `blind_grading` column defaults to `false`.

## 7. Acceptance Criteria

- **AC-1.** *Given* blind grading is enabled on assignment A, *when* a grader opens the submission list via the API, *then* all `student_name`, `student_email`, and `student_id` fields are absent from the response.
- **AC-2.** *Given* blind grading is active, *when* the grader opens submission #3 in the navigator, *then* the UI displays "Student 3" (or equivalent label) — never the actual student name.
- **AC-3.** *Given* the instructor clicks "Reveal identities," *when* the action completes, *then* all student names appear in the grader panel and an audit log entry is created.
- **AC-4.** *Given* two different grader sessions open the same assignment while blind grading is active, *when* both view submission for the same student, *then* both see the same anonymised label.
- **AC-5.** *Given* blind grading is not enabled on an assignment, *when* a grader opens the submission list, *then* student names display normally.
- **AC-6.** *Given* blind grading is enabled, *when* a grader attempts to call the submissions API with a JWT that has grader scope, *then* no student PII is returned regardless of client-side state.

## 8. Data Model

```sql
-- Migration: server/migrations/075_assignment_blind_grading.sql

ALTER TABLE module_assignments
    ADD COLUMN blind_grading BOOLEAN NOT NULL DEFAULT FALSE;

-- Audit events stored in existing user_audit table (029_user_audit.sql)
-- with action = 'blind_grading_revealed', resource_id = assignment_id
```

No new tables required. Anonymised labels derived by ranking `submission.created_at ASC` within the assignment at query time.

## 9. API Surface

Modified endpoints (add redaction when `blind_grading = true`):
- `GET /api/courses/:cid/assignments/:aid/submissions` — omit PII fields.
- `GET /api/courses/:cid/assignments/:aid/submissions/:sid` — omit PII fields.

New endpoint:
- `POST /api/courses/:cid/assignments/:aid/reveal-identities` — instructor/owner only; sets a `identities_revealed_at` timestamp on the assignment; emits audit event.

Assignment create/update endpoint:
- `PATCH /api/courses/:cid/assignments/:aid` — accept `blind_grading: boolean`.

## 10. UI / UX

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Client-side bypass (name visible in network tab) | H | H | Enforce server-side redaction in route handler; never rely on UI alone |
| Instructor reveals identities prematurely | M | M | Confirmation modal + audit log; warn if ungraded submissions remain |
| Label non-determinism across sessions | L | M | Derive label from stable sort (submission created_at); add integration test |

## 15. Rollout Plan

- Feature flag: `BLIND_GRADING_ENABLED` (default: `true` — low risk, additive toggle).
- Schema migration → backfill `blind_grading = false` on existing assignments.
- Code behind flag; test with internal course.
- GA criteria: server-side redaction confirmed by authz integration test; no PII leak in blind mode audit.
- Rollback: disable flag; column remains but UI/API redaction bypassed.

## 16. Test Plan

- **Unit** — `redact_submission_for_blind_grading()` helper.
- **Integration** — Submission list API: assert PII absent when blind mode on, present when off; verify label determinism.
- **End-to-end** — Playwright: enable blind grading → open grader → verify no name shown → reveal → verify names shown.
- **Security** — Authz matrix: student cannot call reveal-identities; non-enrolled user gets 403.
- **Accessibility** — axe-core; anonymous label announced by screen reader.
- **Manual exploratory** — QA: verify labels consistent across browser sessions and devices.
