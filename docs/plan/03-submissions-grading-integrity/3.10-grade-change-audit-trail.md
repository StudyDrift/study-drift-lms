## 1. Problem Statement

When a student's grade changes — whether entered initially, corrected, overridden after moderation, or excused — there is no record of who made the change, what the previous value was, or when it happened. This is a FERPA-significant gap: institutions are required to provide students (and parents in K-12) with a meaningful record of how their grades were assigned. Grade disputes, academic integrity investigations, and accreditation audits all depend on an immutable grade history. The existing `user_audit` table (migration 029) tracks navigation events but not data mutations on education records.

## 2. Goals

- Record an immutable audit event for every grade creation, update, excusal, and deletion in `course_grades`.
- Include: who changed the grade, from what value, to what value, when, and the reason (if provided).
- Expose grade history to instructors in the gradebook and to students in My Grades (with appropriate visibility controls).
- Store audit events in append-only fashion (no update or delete on audit rows).
- Integrate with all other grading features (3.3 reveal, 3.4 reconcile, 3.8 post, 3.12 excused) that touch `course_grades`.

## 3. Non-Goals

- Full admin audit log for all system actions (addressed in 10.11).
- Real-time streaming of audit events to external SIEM (future work).
- Automated grade-dispute workflow (manual process referencing audit trail).

## 4. Personas & User Stories

- **As a student**, I want to see a timeline of my grade history on an assignment so that I can understand when and why my score changed.
- **As an instructor**, I want to see who last edited a grade so that I can verify my TA entered grades correctly.
- **As a registrar**, I want a full grade-change history for a student to satisfy a formal grade appeal so that I can respond with documented evidence.
- **As an auditor (accreditation)**, I want evidence that grade changes are logged and immutable so that the institution satisfies its data governance requirements.

## 5. Functional Requirements

- **FR-1.** The system MUST create a `grade_audit_events` record whenever a row in `course_grades` is inserted, updated (score, excused state, posted state), or soft-deleted.
- **FR-2.** Each audit record MUST capture: `grade_id`, `changed_by_user_id`, `action` (created/updated/excused/posted/retracted), `previous_score`, `new_score`, `previous_status`, `new_status`, `reason` (free text, optional), `changed_at`.
- **FR-3.** Audit records MUST be append-only; no `UPDATE` or `DELETE` SQL is permitted on `grade_audit_events`.
- **FR-4.** The system MUST expose a grade history endpoint accessible to instructors (full history) and students (self, with status changes visible but scorer identity hidden until posted).
- **FR-5.** The system MUST capture audit events for all grading actions in 3.3 (identity reveal), 3.4 (reconcile), 3.8 (post/retract), and 3.12 (excused/exempt).
- **FR-6.** The system SHOULD allow instructors to provide a `reason` when overriding a score (stored in the audit record).
- **FR-7.** The system MAY surface a grade history panel in the instructor gradebook on cell click.

## 6. Non-Functional Requirements

- **Performance** — Audit insert is a fire-and-forget INSERT after the grade update; adds < 5 ms to grade mutation p95.
- **Security** — `grade_audit_events` table: INSERT-only for the app role; no application-level UPDATE/DELETE path. Read access enforced by `server/src/authz.rs`. Students see own records only.
- **Privacy & Compliance** — Grade change history is a FERPA education record. All audit events included in FERPA DSAR exports. GDPR: must be includable in data-subject access requests; right-to-erasure does NOT apply to audit logs (legitimate interest / legal obligation basis).
- **Accessibility** — Grade history panel is a timeline list; keyboard navigable; screen-reader friendly. WCAG 2.1 AA.
- **Scalability** — Append-only; no hot-path writes beyond grade mutations. Index on `(grade_id, changed_at)`.
- **Reliability** — Audit insert runs within the same DB transaction as the grade mutation; audit record either commits with the grade change or neither does.
- **Observability** — Metric: `grade_audit.events_written_total`. Alert if audit insert failure rate > 0%.
- **Maintainability** — Audit logic in `server/src/services/grading/audit.rs`; called from all grade mutation service methods.
- **Internationalization** — Action labels and reason field i18n-keyed for display.
- **Backward compatibility** — Existing `course_grades` rows have no audit history; they display "No history before [migration date]."

## 7. Acceptance Criteria

- **AC-1.** *Given* an instructor enters a score for the first time, *when* the grade is saved, *then* a `grade_audit_events` row with `action = 'created'`, `previous_score = null`, `new_score` = entered value is created in the same transaction.
- **AC-2.** *Given* an instructor corrects a grade from 70 to 85, *when* the update is saved, *then* a new audit row records `previous_score = 70`, `new_score = 85`, `changed_by`, and timestamp.
- **AC-3.** *Given* an audit event is written, *when* any code path attempts an UPDATE or DELETE on `grade_audit_events`, *then* the DB constraint rejects it.
- **AC-4.** *Given* a student requests grade history for assignment A, *when* the API is called, *then* only events for their own grade are returned; other students' events are not visible.
- **AC-5.** *Given* a grade override with a reason "Late submission policy applied," *when* the audit event is queried, *then* the `reason` field contains that text.
- **AC-6.** *Given* grades were posted via 3.8, *when* the audit log is queried for that assignment, *then* a `grades_posted` event with `changed_by` = posting instructor is present.

## 8. Data Model

```sql
-- Migration: server/migrations/082_grade_audit_events.sql

CREATE TABLE grade_audit_events (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    grade_id            UUID NOT NULL,         -- references course_grades(id) — no FK to allow orphan audits
    course_id           UUID NOT NULL,
    assignment_id       UUID NOT NULL,
    student_id          UUID NOT NULL,
    changed_by_user_id  UUID,                  -- NULL for system actions
    action              TEXT NOT NULL CHECK (action IN
                            ('created','updated','excused','unexcused','posted','retracted','deleted')),
    previous_score      NUMERIC(8,2),
    new_score           NUMERIC(8,2),
    previous_status     TEXT,
    new_status          TEXT,
    reason              TEXT,
    changed_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Append-only enforcement
CREATE RULE no_update_grade_audit AS ON UPDATE TO grade_audit_events DO INSTEAD NOTHING;
CREATE RULE no_delete_grade_audit AS ON DELETE TO grade_audit_events DO INSTEAD NOTHING;

CREATE INDEX idx_grade_audit_grade_id ON grade_audit_events(grade_id, changed_at);
CREATE INDEX idx_grade_audit_student ON grade_audit_events(student_id, changed_at);
```

## 9. API Surface

| Verb | Path | Auth | Description |
|---|---|---|---|
| GET | `/api/courses/:cid/assignments/:aid/grades/:gid/history` | instructor (full) / student-self (limited) | Grade change history |
| GET | `/api/courses/:cid/students/:uid/grade-history` | instructor | All grade changes for student |

Response shape:
```ts
{ events: Array<{ id, action, previousScore, newScore, changedBy, changedAt, reason }> }
```
Student view omits `changedBy` for held grades.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Audit insert fails, silent loss | L | H | Wrap in same transaction; emit alert on any failure |
| No-FK orphan audit row if grade deleted | M | L | Keep orphan rows for FERPA completeness; grade_id retained even if course_grades row deleted |
| Table grows very large | L | M | Partition by year if needed; index on (student_id, changed_at) |

## 15. Rollout Plan

- Feature flag: none — audit capture is always-on once shipped.
- Schema migration; backfill: synthetic "created" event for all existing `course_grades` rows with `changed_at = course_grades.created_at`, `changed_by = NULL`.
- Wire audit calls into all grade mutation services.
- GA criteria: 100% of grade mutations in integration tests produce corresponding audit rows; append-only rule confirmed.

## 16. Test Plan

- **Unit** — `record_grade_change()` with all action types; verify fields.
- **Integration** — Grade update transaction: audit row present; UPDATE/DELETE on audit table rejected.
- **End-to-end** — Playwright: enter grade → change grade → open history panel → verify 2 events shown.
- **Security** — Student cannot see other students' audit events; instructor can.
- **Manual exploratory** — Audit trail for bulk import (3.11); reconcile event (3.4); post event (3.8).
