## 1. Problem Statement

The `module_quizzes` schema has no `time_limit_seconds` column and the quiz delivery UI has no timer. Timed assessments are fundamental to HE midterms and finals, standardised K-12 assessments, and any exam where pacing control is required. Without a server-authoritative timer, a quiz cannot be trusted: a student could simply leave the browser tab open indefinitely. The auto-submit on timeout feature is also required to support accommodations (2.11): extended-time accommodation overrides rely on the base time limit being correctly recorded and enforced.

## 2. Goals

- Add a `time_limit_seconds` field to quiz configuration, enforceable by the server.
- Implement a server-authoritative timer: the server records `started_at` and enforces submission deadline at `started_at + time_limit_seconds`, regardless of client behaviour.
- Display a countdown timer in the student quiz UI.
- Auto-submit the in-progress attempt when the deadline is reached, using whatever answers have been saved so far.
- Support per-student extended-time overrides (hook for accommodations, 2.11).

## 3. Non-Goals

- Per-question time limits (entire quiz has one time limit in v1).
- Countdown sound/alarm effects (accessibility concern deferred to UX review).
- Time-unlimited quizzes that pause and resume across sessions (async quizzes are a separate feature).
- Proctoring / screen recording integration (covered in 14.9).

## 4. Personas & User Stories

- **As an HE instructor**, I want to set a 90-minute time limit on my midterm so students cannot take unlimited time.
- **As a student**, I want to see a countdown timer so I can manage my pacing across questions.
- **As a student with extended-time accommodation**, I want the system to automatically apply my 1.5× time extension without the instructor having to manually adjust the quiz for me.
- **As an instructor**, I want the server to auto-submit a student's attempt when time expires so I do not have to manually close unfinished attempts.
- **As a student**, I want to be warned when 10 minutes remain so I can save my work before auto-submit.

## 5. Functional Requirements

- **FR-1.** The system MUST add `time_limit_seconds` (nullable integer) to the quiz configuration; null means no time limit.
- **FR-2.** The system MUST record `started_at` (timestamptz) on the attempt row when the attempt begins and compute `deadline_at = started_at + time_limit_seconds` (adjusted by any accommodation multiplier from 2.11).
- **FR-3.** The system MUST reject quiz answer submissions that arrive after `deadline_at` with HTTP 403 and error `"attempt_time_expired"`.
- **FR-4.** The system MUST auto-submit (grade with current saved responses) any attempt whose `deadline_at` has passed and `submitted_at` is null; this runs as a background job polling every 30 seconds.
- **FR-5.** The system MUST expose the remaining time in the `GET /api/v1/quiz-attempts/{id}` response as `remaining_seconds` (computed server-side from `deadline_at - NOW()`).
- **FR-6.** The system MUST emit a WebSocket event `time_warning` at 10 minutes and 1 minute remaining for attempts with a time limit.
- **FR-7.** The system MUST display a countdown timer in the quiz-take UI, sourced from `remaining_seconds` in the attempt response and updated by a client-side `setInterval` (re-synced every 60 s from the server to prevent client clock drift).
- **FR-8.** The system SHOULD allow instructors to grant an individual time extension on an in-progress attempt (instructor can extend by N minutes via `POST /api/v1/quiz-attempts/{id}/extend-time`).

## 6. Non-Functional Requirements

- **Performance** — `remaining_seconds` computation is a simple arithmetic expression in Postgres; no overhead. Auto-submit background job scans `attempt_deadline_index` with a WHERE `deadline_at < NOW() AND submitted_at IS NULL`; runs in ≤ 1 s for up to 10 k active attempts.
- **Security** — Server-side enforcement of deadline_at is authoritative; client-side timer is display-only. A student manipulating their clock or delaying the POST does not bypass the server check. Extension grant is instructor/admin-only.
- **Privacy & Compliance** — `started_at` and `deadline_at` are part of the attempt record (FERPA). Accommodation multiplier is stored on the enrollment override (2.11) and never exposed directly to the student.
- **Accessibility** — Timer MUST use an ARIA live region (`aria-live="assertive"` for the 1-minute warning; `aria-live="polite"` for routine countdown updates). Timer MUST also be rendered as text (not just a graphic). Time warning MUST trigger both visual and auditory (optional, per user preference) alerts. WCAG 2.1 SC 2.2.1 (Timing Adjustable) — time limit must be extendable or it must be a real-time event (exam scenario exception applies).
- **Scalability** — Background auto-submit job: at 10 k simultaneous active timed exams, the scan and submit batch MUST complete within the 30-second polling interval.
- **Reliability** — If auto-submit job crashes mid-batch, unsubmitted attempts remain open; job is re-runnable and idempotent (checks `submitted_at IS NULL`). Network loss does not cause premature auto-submit — server-side deadline is the authority.
- **Observability** — Counter `quiz_attempts.auto_submitted`. Log each auto-submit event with attempt_id, quiz_id, submission_delay (deadline_at vs. actual submission). Alert if auto-submit queue lag > 60 s.
- **Maintainability** — Auto-submit logic in `server/src/jobs/quiz_auto_submit.rs`. Timer remaining-seconds query in `server/src/services/quiz_attempt.rs`. No timer logic in route handlers.
- **Internationalization** — Timer display format respects locale (HH:MM:SS or M:SS depending on duration). Time warning message is an i18n key.
- **Backward compatibility** — `time_limit_seconds = NULL` means no time limit (existing behaviour preserved). `deadline_at` column added nullable; existing in-progress attempts are unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a quiz with `time_limit_seconds = 3600`, *when* a student starts an attempt, *then* the attempt row has `deadline_at = started_at + 3600s` and `remaining_seconds` in the response is approximately 3600.
- **AC-2.** *Given* a student has 5 minutes remaining and submits their answers, *when* the submission arrives before `deadline_at`, *then* the submission is accepted normally.
- **AC-3.** *Given* `deadline_at` has passed and the attempt is unsubmitted, *when* the auto-submit job runs, *then* the attempt is submitted with whatever answers were saved, and `auto_submitted: true` is set.
- **AC-4.** *Given* a student tries to submit after `deadline_at`, *when* the POST arrives, *then* the response is HTTP 403 with `error: "attempt_time_expired"`.
- **AC-5.** *Given* a student with a 1.5× extended-time accommodation starts a 60-minute quiz, *when* the attempt is created, *then* `deadline_at = started_at + 90 minutes`.
- **AC-6.** *Given* 10 minutes remain, *when* the server emits a `time_warning` WebSocket event, *then* the student's UI shows a prominent warning banner and the ARIA live region announces "10 minutes remaining."

## 8. Data Model

```sql
-- Migration: server/migrations/0045_time_limits.sql

ALTER TABLE course.module_quizzes
    ADD COLUMN time_limit_seconds INTEGER CHECK (time_limit_seconds > 0);

ALTER TABLE course.quiz_attempts
    ADD COLUMN deadline_at    TIMESTAMPTZ,
    ADD COLUMN auto_submitted BOOLEAN NOT NULL DEFAULT FALSE;

CREATE INDEX idx_attempts_deadline ON course.quiz_attempts (deadline_at)
    WHERE deadline_at IS NOT NULL AND submitted_at IS NULL;
```

No new tables needed; the auto-submit job uses the existing grading path.

## 9. API Surface

```
-- Existing attempt response gains:
GET /api/v1/quiz-attempts/{id}
  Response addition: { deadline_at?: string, remaining_seconds?: number, auto_submitted?: bool }

-- Instructor: extend time on active attempt
POST /api/v1/quiz-attempts/{id}/extend-time
  Auth: instructor|admin
  Body: { extra_seconds: number }
  Response 200: { new_deadline_at: string }

WebSocket event (to student):
  { type: "time_warning", remaining_seconds: 600 }   -- at 10 min
  { type: "time_warning", remaining_seconds: 60 }    -- at 1 min
  { type: "time_expired" }                            -- at 0
```

## 10. UI / UX

**Quiz timer** (persistent header in quiz-take UI, `clients/web/src/pages/lms/CourseModuleQuizPage.tsx`):
- Clock icon + HH:MM:SS or MM:SS countdown.
- Green → yellow (< 15 min) → red (< 5 min) colour transition.
- **10-minute warning**: toast notification + timer colour change to yellow.
- **1-minute warning**: toast notification + timer colour change to red + `aria-live="assertive"` announcement.
- **Time expired**: full-page overlay "Time's up — your answers have been submitted."

Auto-answer-save: every 30 s, the UI silently saves current answers (existing functionality) so auto-submit has the latest state.

Accessibility: timer text is always readable; colour changes are supplementary, never the only indicator (WCAG SC 1.4.1). Screen readers receive periodic "X minutes remaining" announcements every 5 minutes.

Mobile: timer pinned to bottom toolbar; warning toasts full-width.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Client clock drift causes premature timer expiry in UI | M | M | Re-sync remaining_seconds from server every 60 s; UI timer is display-only, not authoritative |
| Auto-submit job delays (job queue overload) | L | H | Prioritise auto-submit jobs; alert if queue lag > 60 s |
| Instructor forgets to enable time limit and students take unlimited time | M | M | Show prominent "No time limit" badge in instructor quiz settings |
| WCAG SC 2.2.1 violation — time limit without mechanism to extend | M | H | Ensure instructor can grant extensions; link to accommodation feature (2.11) in docs |

## 15. Rollout Plan

- **Feature flag:** `quiz_time_limits_enabled` (default: `false`; time limit field hidden in quiz settings until enabled).
- **Migration sequencing:** add nullable columns → deploy auto-submit job → deploy timer UI → enable flag.
- **Pilot cohort:** 2 HE midterm exams with 30+ students.
- **GA criteria:** auto-submit tested end-to-end; timer UI verified with NVDA; no false auto-submits in pilot.
- **Rollback:** disable flag; time_limit_seconds ignored; deadline_at stays null for new attempts; existing open attempts not affected.

## 16. Test Plan

- **Unit** — Deadline computation with accommodation multiplier; remaining_seconds negative clamp to 0; auto-submit idempotency.
- **Integration** — Create timed quiz → start attempt → advance clock past deadline → run auto-submit job → assert attempt submitted; late POST rejected with 403.
- **End-to-end** — Playwright: start 2-minute quiz, observe countdown, let time expire, verify auto-submitted overlay shown.
- **Security** — POST after deadline → 403; student cannot call extend-time (403); instructor can extend.
- **Accessibility** — ARIA live region fires at 10-min and 1-min warnings; timer text readable at all colour states (contrast ratio ≥ 4.5:1).
- **Performance** — Auto-submit job with 5 000 expired attempts completes in ≤ 30 s.
- **Manual exploratory** — Verify server-side enforcement when browser clock is set forward; verify instructor extension updates remaining_seconds in real time.

## 18. Open Questions

1. Should auto-submit run every 30 seconds (database poll) or be triggered by a per-attempt scheduled job from a scheduler service? Poll is simpler; per-attempt schedule is more precise.
2. Should the 10-minute warning be configurable by the instructor (some prefer 5 minutes or 15 minutes)?
3. Should students be warned that their timer runs even when the browser tab is closed (background time consumption)?
4. Should pausing the timer be allowed for network disconnections, or is unpaused time the standard (as in most proctored exams)?
