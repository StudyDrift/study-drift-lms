## 1. Problem Statement

The TipTap rich-text editor in `clients/web/src/components/editor/` has no math rendering or input support. Any course content, question, or assignment involving mathematical notation — algebra, calculus, statistics, chemistry — must use plain text approximations (`x^2`, `sqrt(x)`) that are unreadable for K-12 and university STEM students. This is a hard blocker for every STEM course on the platform. Instructors importing from LaTeX-based systems or Word have no path to rendering their existing content. The fix is well-established: add KaTeX rendering (fast, client-side) and a math input toolbar/keyboard to both the authoring and student-response surfaces.

## 2. Goals

- Render LaTeX math expressions in all content surfaces: TipTap editor content, quiz question stems, answer options, feedback, and assignment descriptions, using KaTeX.
- Provide an inline math editor (TipTap extension + toolbar button) for instructors authoring content.
- Provide a student-facing math input keyboard for formula-type questions (2.2) and short-answer STEM questions.
- Support both inline math (`$...$`) and display/block math (`$$...$$`).
- Fall back gracefully (show raw LaTeX) when KaTeX cannot render an expression.

## 3. Non-Goals

- MathML authoring (KaTeX outputs MathML as its accessible format; no separate MathML input surface needed).
- Commercial math authoring tools (Wiris, Desmos equation editor) — open-source KaTeX + a simple toolbar is sufficient for v1.
- Handwriting recognition for math input (covered in 19.7).
- Chemistry structure diagrams (ChemDraw / SMILES rendering) — reserved for v2.
- Server-side LaTeX rendering (client-side KaTeX is sufficient and avoids server load).

## 4. Personas & User Stories

- **As a math instructor**, I want to type `\frac{d}{dx}e^x = e^x` in the course editor and have it display as a properly typeset fraction.
- **As a K-12 student**, I want to click a "√" button in the quiz input field to insert a square-root symbol so I can enter my numeric answer using proper notation.
- **As an instructor**, I want to preview my LaTeX equation as I type it, so I catch errors before saving.
- **As a screen-reader user**, I want math equations to be announced as their mathematical meaning (via MathML) rather than as raw LaTeX strings.
- **As a chemistry instructor**, I want to use `\text{H}_2\text{O}` to write chemical formulas in quiz questions.

## 5. Functional Requirements

- **FR-1.** The system MUST render all LaTeX content delimited by `$...$` (inline) and `$$...$$` (display) in quiz question stems, options, explanations, and course content using KaTeX v0.16+.
- **FR-2.** The system MUST add a TipTap extension (`MathInlineNode` and `MathBlockNode`) that stores math as LaTeX source and renders via KaTeX on display.
- **FR-3.** The system MUST provide a toolbar button "Σ" in the TipTap editor that opens an inline math input popover with: raw LaTeX text field + live KaTeX preview + "Insert" confirmation.
- **FR-4.** The system MUST provide a student-facing math input keyboard component (`MathKeyboard`) with symbol groups: basic operators, fractions, exponents, roots, Greek letters, trigonometric functions.
- **FR-5.** The system MUST output MathML alongside the rendered SVG/HTML so screen-readers can interpret equations (KaTeX's `output: 'mathml'` option or dual-output mode).
- **FR-6.** The system MUST fall back to rendering the raw LaTeX string wrapped in `<code>` when KaTeX throws a parse error, rather than crashing the component.
- **FR-7.** The system SHOULD support `\begin{align}...\end{align}` and other multi-line LaTeX environments in display math mode.
- **FR-8.** The system MAY provide a "copy LaTeX" button on rendered equations for instructors who want to reuse expressions.

## 6. Non-Functional Requirements

- **Performance** — KaTeX renders inline expressions in < 5 ms each on a mid-tier device. Page with 30 math expressions MUST render in < 200 ms total. KaTeX is loaded as a lazy chunk, not in the critical render path.
- **Security** — LaTeX content is rendered client-side; KaTeX is sandboxed and does not execute arbitrary code. User-supplied LaTeX MUST be stored as-is (no server-side eval); XSS risk is from KaTeX's HTML output, which is safe by design (KaTeX does not insert `<script>`).
- **Privacy & Compliance** — Math content is part of course content; standard content privacy rules apply. No special compliance obligations beyond existing content handling.
- **Accessibility** — KaTeX MUST emit MathML output for screen readers (NVDA+Firefox, VoiceOver+Safari, JAWS). WCAG 2.1 AA SC 1.3.1. Math keyboard MUST be keyboard-navigable (Tab + Enter, Escape to dismiss).
- **Scalability** — Client-side rendering; no server load. Font files served from CDN (KaTeX's ~300 KB font set). Lazy-load KaTeX on first math encounter.
- **Reliability** — KaTeX parse errors MUST be caught per-expression; one malformed expression does not break the entire page. Error boundary around math render components.
- **Observability** — Client-side error log for KaTeX render failures (logged to existing Sentry/error reporter when implemented). No server-side observability needed.
- **Maintainability** — KaTeX rendering logic in `clients/web/src/lib/math.ts`; TipTap extension in `clients/web/src/components/editor/extensions/MathExtension.ts`; math keyboard in `clients/web/src/components/quiz/MathKeyboard.tsx`.
- **Internationalization** — Math notation is locale-independent. Math keyboard symbol groups labeled with i18n keys. RTL layout for the keyboard (Arabic/Hebrew) tested.
- **Backward compatibility** — Existing TipTap content stored as HTML/JSON is unaffected; math nodes are a new node type. Content without math continues to work identically.

## 7. Acceptance Criteria

- **AC-1.** *Given* a question stem contains `$\frac{a}{b}$`, *when* the question is rendered in the student quiz view, *then* a typeset fraction is displayed (not raw LaTeX).
- **AC-2.** *Given* an instructor types `$$\int_0^\infty e^{-x^2} dx = \frac{\sqrt{\pi}}{2}$$` in the editor, *when* they click outside the math block, *then* the expression is rendered as a display-mode equation.
- **AC-3.** *Given* a malformed LaTeX string `$\frac{a}{$`, *when* the question renders, *then* the raw string is shown in a `<code>` block and no JavaScript error is thrown.
- **AC-4.** *Given* a screen reader is active, *when* a math expression is focused, *then* the screen reader reads the MathML representation (e.g., "a fraction with numerator a and denominator b") not the raw LaTeX.
- **AC-5.** *Given* a student is on the formula question input, *when* they click the fraction button on the math keyboard, *then* `\frac{}{}` is inserted into the LaTeX input field with cursor positioned inside the first argument.
- **AC-6.** *Given* KaTeX is not yet loaded, *when* a page with math content renders, *then* math expressions are shown as raw LaTeX until KaTeX loads (progressive enhancement, no layout shift).

## 8. Data Model

No new database tables. LaTeX is stored as a string in the TipTap JSON node `{ type: "math_inline", attrs: { latex: "\\frac{a}{b}" } }`. Existing `content` JSONB columns in course and quiz tables support this transparently.

Migration: none required — math nodes are new node types, backward-compatible with existing content.

## 9. API Surface

No new REST endpoints. KaTeX is a client-side library. Optionally, a `POST /api/v1/render/latex` endpoint could provide server-side KaTeX rendering for email/PDF exports — deferred to v2.

## 10. UI / UX

**TipTap editor toolbar** (`clients/web/src/components/editor/`):
- New "Σ" button (formula icon) in the toolbar.
- Click opens a floating popover: LaTeX input textarea + live KaTeX preview pane.
- "Insert inline" / "Insert block" toggle.
- "Recently used" section showing last 5 expressions (localStorage).
- Keyboard: Tab between input and preview; Enter to insert; Escape to dismiss.

**Student math keyboard** (shown below formula-type and numeric inputs in quiz):
- 6 groups: basic, fractions, exponents, roots, Greek, trig.
- Each symbol button has a tooltip with the symbol name.
- Keyboard: Tab navigates buttons; Enter inserts; keyboard toggle button shows/hides.

**Rendered math** (all content surfaces):
- `display: block` for `$$...$$`; `display: inline` for `$...$`.
- Click-to-copy LaTeX on hover (instructor mode only).
- Error state: light-red background + raw LaTeX in `<code>`.

Mobile: math keyboard renders as a bottom sheet; toolbar popover becomes full-width modal.

## 11. AI / ML Considerations

Not applicable for math rendering. AI math generation (e.g., generating problem stems with LaTeX) happens in `quiz_generation_ai.rs` and already produces LaTeX strings. This feature enables those strings to render correctly.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| KaTeX does not support some advanced LaTeX packages (e.g., TikZ, pgf) | M | M | Document supported subset; for unsupported expressions, fall back to graceful error state |
| Math keyboard is too small on mobile | M | M | Design with 44px minimum touch targets; test on smallest viewport (375px) |
| Screen-reader MathML support varies by browser/AT combination | M | H | Test with NVDA+Firefox and VoiceOver+Safari; document known gaps; provide "copy LaTeX" as fallback |
| KaTeX bundle size affects page load | L | L | Lazy-load KaTeX; use code splitting; KaTeX is ~300 KB gzipped (acceptable) |

## 15. Rollout Plan

- **Feature flag:** `math_rendering_enabled` (default: `true` once merged — this is a pure enhancement).
- **Rollout:** Deploy KaTeX rendering first (view-only) → then ship TipTap extension (authoring) → then ship math keyboard (student input).
- **Pilot cohort:** 2 STEM courses; instructors update existing questions with LaTeX.
- **GA criteria:** 50+ questions with LaTeX rendered without errors; MathML output verified with NVDA.
- **Rollback:** remove KaTeX import; math nodes render as raw LaTeX (safe fallback).

## 16. Test Plan

- **Unit** — KaTeX error boundary (malformed LaTeX returns raw string, no throw); TipTap extension serialisation/deserialisation of math nodes.
- **Integration** — Create question with LaTeX stem via API → render in quiz view → assert DOM contains rendered SVG/MathML.
- **End-to-end** — Playwright: instructor opens editor, clicks Σ, types LaTeX, inserts, saves; student views quiz, verifies math renders.
- **Accessibility** — axe audit on quiz page with math; manual NVDA + Firefox test to verify MathML announced correctly.
- **Performance** — Lighthouse audit: page with 20 math expressions; CLS < 0.1; FID < 100 ms.
- **Manual exploratory** — Test 10 common LaTeX patterns; test malformed LaTeX; test math keyboard symbol insertion; test RTL layout.

## 18. Open Questions

1. Should the inline math delimiter be `$...$` (common in LaTeX but conflicts with dollar-sign literal in text) or `\(...\)` (safer but less familiar)?
2. Should we embed a Desmos graphing calculator widget for graph-plotting questions, or is that out of scope for v1?
3. Do we need server-side math rendering for PDF export of gradebooks/reports (covered in 9.8)? If so, should we use a headless Chrome approach or a server-side KaTeX renderer?
