## 1. Problem Statement

Every course needs two types of non-graded questionnaires: course-feedback surveys (end-of-term evaluations where student anonymity is critical) and ungraded check-in questionnaires (entrance tickets, opinion polls, icebreakers). Currently, every activity in Lextures is graded and connected to the gradebook. Instructors have no way to collect student feedback without it affecting grades, and no mechanism for anonymous data collection. For HE institutions, end-of-term course evaluations are often required by accreditation bodies. For K-12, entrance/exit tickets are a standard formative practice. The gap also blocks Lextures from serving as a campus survey tool that many LMSs bundle.

## 2. Goals

- Add a `survey` activity type that is independent of the gradebook and supports anonymous responses.
- Allow instructors to include Likert scales, rating scales, free-text, single-select, and multi-select question types in surveys.
- Provide response aggregation reports viewable by instructors (and optionally shareable with students for anonymous aggregate-only views).
- Support configurable anonymity: fully anonymous (no user_id stored), identified (normal), or pseudo-anonymous (instructor can see identity, students see aggregate only).
- Integrate surveys as course module items (like quizzes and assignments) so they appear in the course timeline.

## 3. Non-Goals

- Enterprise survey platform features (branching/skip logic, complex quotas, panels) — keep scope simple in v1.
- External survey tool integrations (SurveyMonkey, Google Forms) — covered in 16.4.
- LMS-wide surveys outside of course context (requires admin portal, 18.1).
- Real-time poll / live voting during synchronous class (covered in 6.4).

## 4. Personas & User Stories

- **As an instructor**, I want to create an end-of-term evaluation survey with anonymous responses so students feel safe giving honest feedback.
- **As a student**, I want to see that a survey is anonymous before I respond so I can give honest feedback without fear of grade retaliation.
- **As an instructor**, I want to see aggregate survey results (bar charts, word clouds) without being able to drill down to individual student responses in anonymous mode.
- **As a K-12 teacher**, I want to create a 3-question exit ticket at the end of a lesson to gauge comprehension — without it appearing in the gradebook.
- **As an admin**, I want to run a course-experience survey across all sections of a course at the end of term.

## 5. Functional Requirements

- **FR-1.** The system MUST support a `survey` content type as a first-class course module item, distinct from `quiz` and `assignment`.
- **FR-2.** The system MUST support the following survey question subtypes: `likert` (1–5 or 1–7 scale with configurable labels), `rating` (star rating), `single_select`, `multi_select`, `free_text`, `net_promoter_score`.
- **FR-3.** The system MUST support three anonymity modes configurable per survey: `identified` (user_id stored), `anonymous` (no user_id stored, only course enrollment verified), `pseudo_anonymous` (user_id stored server-side but not exposed in reports or exports).
- **FR-4.** The system MUST enforce anonymity on the API: in `anonymous` mode, the response endpoint MUST NOT require or store a `user_id`; authentication still required to verify enrollment.
- **FR-5.** The system MUST enforce a single-response-per-enrollment constraint (even in anonymous mode, using a hashed token to prevent double-submission without storing identity).
- **FR-6.** The system MUST provide `GET /api/v1/surveys/{id}/results` returning aggregated results (counts, averages, distributions) for each question; raw individual responses accessible only in `identified` mode.
- **FR-7.** The system MUST display a survey completion status (not a grade) in the student's course progress view: "Completed" or "Not yet submitted."
- **FR-8.** The system SHOULD allow instructors to set a survey open/close date window, after which responses are no longer accepted.
- **FR-9.** The system MAY allow instructors to share an aggregate-only results link with students after the survey closes.

## 6. Non-Functional Requirements

- **Performance** — Survey submission MUST complete in p95 ≤ 200 ms. Results aggregation for up to 500 responses MUST complete in ≤ 300 ms.
- **Security** — Anonymity guarantees MUST be enforced server-side; client cannot override the mode. Response data encrypted at rest. Even admins MUST NOT be able to link anonymous responses to users (this is a privacy guarantee, not just a UI restriction — no user_id column in `survey_responses` in anonymous mode).
- **Privacy & Compliance** — Free-text responses may contain PII; treat as educational record (FERPA). In GDPR regions, anonymous-mode responses technically cannot be associated to data subjects and therefore may be exempt from erasure (document this). Pseudo-anonymous responses included in DSAR exports. COPPA: surveys for under-13 students must not collect opinion data that could be used for profiling.
- **Accessibility** — Likert and rating scales MUST have `fieldset` + `legend` with accessible question text; radio buttons must be keyboard navigable. Free-text must have a visible label. WCAG 2.1 AA required.
- **Scalability** — End-of-term evaluation surges: expect 5 000 responses in 30 minutes for a large institution. Async response write via job queue (17.3) if needed; direct DB insert is acceptable for v1.
- **Reliability** — Survey submission MUST be idempotent: submitting twice returns 200 with "already submitted" message. Results aggregation uses database-level aggregation (SQL `GROUP BY`), not application-level, for correctness.
- **Observability** — Counter `survey.response_submitted` (labeled by anonymity_mode). Log survey creation and close events. Alert if response error rate > 1 %.
- **Maintainability** — Survey service in `server/src/services/survey.rs`; routes in `server/src/routes/surveys.rs`. Survey question types are a separate, simpler enum from quiz question types.
- **Internationalization** — Likert scale labels use i18n keys (Strongly Agree / Agree / etc.). NPS question text is i18n'd. Free-text responses stored as UTF-8.
- **Backward compatibility** — Surveys are a new module item type; no existing quizzes or assignments affected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a survey is configured as `anonymous`, *when* a student submits their response, *then* no `user_id` is stored in the database for that response row.
- **AC-2.** *Given* a student has already submitted a survey, *when* they attempt to submit again, *then* the response is HTTP 200 with `{ already_submitted: true }` and no duplicate row is created.
- **AC-3.** *Given* a survey closes, *when* a student attempts to submit after the close date, *then* the API returns HTTP 403 with "Survey is closed."
- **AC-4.** *Given* a Likert question with 30 student responses, *when* the instructor views the results, *then* they see a distribution bar chart with counts per scale value and a mean score.
- **AC-5.** *Given* a survey is `anonymous`, *when* the instructor opens the results page, *then* there is no "view individual response" option and no student names are visible.
- **AC-6.** *Given* a free-text survey question, *when* a student uses keyboard-only navigation to focus and type in the text area, *then* the input is accessible (label associated, focus ring visible).

## 8. Data Model

```sql
-- Migration: server/migrations/0042_surveys.sql

CREATE TYPE course.survey_anonymity AS ENUM ('identified', 'anonymous', 'pseudo_anonymous');

CREATE TABLE course.surveys (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    course_id       UUID NOT NULL REFERENCES course.courses(id) ON DELETE CASCADE,
    title           TEXT NOT NULL,
    description     TEXT,
    anonymity_mode  course.survey_anonymity NOT NULL DEFAULT 'identified',
    opens_at        TIMESTAMPTZ,
    closes_at       TIMESTAMPTZ,
    allow_retake    BOOLEAN NOT NULL DEFAULT FALSE,
    created_by      UUID NOT NULL REFERENCES "user".users(id),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE course.survey_questions (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    survey_id   UUID NOT NULL REFERENCES course.surveys(id) ON DELETE CASCADE,
    position    SMALLINT NOT NULL,
    subtype     TEXT NOT NULL,   -- 'likert'|'rating'|'single_select'|'multi_select'|'free_text'|'nps'
    stem        TEXT NOT NULL,
    config      JSONB NOT NULL DEFAULT '{}',  -- scale labels, options, etc.
    required    BOOLEAN NOT NULL DEFAULT TRUE
);

CREATE TABLE course.survey_responses (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    survey_id       UUID NOT NULL REFERENCES course.surveys(id),
    user_id         UUID REFERENCES "user".users(id),    -- NULL in anonymous mode
    submission_hash TEXT,    -- HMAC of (enrollment_id, survey_id) for dedup in anonymous mode
    answers         JSONB NOT NULL,  -- { question_id: value }
    submitted_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (survey_id, submission_hash)   -- prevents double-submission
);
```

## 9. API Surface

```
GET    /api/v1/courses/{id}/surveys              Auth: instructor|admin
POST   /api/v1/courses/{id}/surveys              Auth: instructor|admin
GET    /api/v1/surveys/{id}                      Auth: enrolled student | instructor | admin
PUT    /api/v1/surveys/{id}                      Auth: instructor|admin
DELETE /api/v1/surveys/{id}                      Auth: instructor|admin

POST   /api/v1/surveys/{id}/respond
  Auth: enrolled student
  Body: { answers: { [question_id]: value } }
  Response 200: { submitted: bool, already_submitted?: bool }

GET    /api/v1/surveys/{id}/results              Auth: instructor|admin
  Response: { questions: AggregatedQuestion[], response_count: number }

GET    /api/v1/surveys/{id}/results/aggregate-share   Auth: instructor|admin
  Response: { share_url }   -- read-only aggregate link
```

## 10. UI / UX

**Survey authoring page** (new page `/courses/{id}/surveys/create`):
- Add question button with subtype picker (Likert, Rating, NPS, etc.).
- Drag to reorder questions.
- Anonymity mode selector with explanatory tooltip.
- Open/close date pickers.

**Student survey page** (`/surveys/{id}/take`):
- Anonymity badge prominently shown: "This survey is anonymous — your instructor cannot see your identity."
- Per-question inputs styled by type (radio rows for single-select, star row for rating, scale row for Likert).
- Submit button with confirmation.
- Post-submission: "Thank you — your response was recorded."

**Results page** (`/surveys/{id}/results`):
- Bar chart per Likert/scale question.
- Word-cloud or list for free-text responses.
- NPS gauge.
- Response count, response rate (if identified mode).

Accessibility: all scale questions use `fieldset` + `legend`; radio groups keyboard navigable; no color-only pass/fail indicators.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Anonymous mode privacy guarantee is undermined by audit logs | M | H | Ensure `user_audit` does not log the content of anonymous survey responses; log only the event type |
| HMAC dedup token breaks if enrollment_id changes (re-enrollment) | L | M | Compute HMAC over (user_id, survey_id) but don't store user_id; use a per-survey secret |
| Instructors use "pseudo_anonymous" as "effectively identified" | M | M | Clear in-UI labelling of each mode; legal note in admin settings |

## 15. Rollout Plan

- **Feature flag:** `surveys_enabled` (default: `false`).
- **Migration sequencing:** deploy migration → deploy API → deploy survey UI → enable for pilot.
- **Pilot cohort:** 2 HE courses for end-of-term evaluations; 1 K-12 course for exit tickets.
- **GA criteria:** anonymous mode verified not to store user_id in DB; aggregate results display correct.
- **Rollback:** disable flag; survey endpoints return 404; no course data affected.

## 16. Test Plan

- **Unit** — HMAC dedup token generation and uniqueness; anonymity mode enforcement (no user_id saved in anonymous mode); double-submission prevention.
- **Integration** — Create anonymous survey → student submits → assert no user_id in DB; second submit → assert 200 + already_submitted; fetch results → assert distribution correct.
- **End-to-end** — Playwright: instructor creates Likert survey, student takes it, instructor views aggregate chart.
- **Security** — Admin cannot link anonymous responses to users; pseudo-anonymous user_id not in API response for results.
- **Accessibility** — axe scan on survey take page; Likert fieldset/legend; keyboard navigation through radio groups.
- **Performance** — 500 concurrent survey submissions; p95 ≤ 200 ms.
- **Manual exploratory** — Verify closed survey returns 403; verify share link shows aggregate only; verify NPS score computation.

## 18. Open Questions

1. Should surveys be a subtype of the existing quiz model (reusing delivery infrastructure) or a fully separate entity? Separate entity is cleaner for anonymity guarantees.
2. Should instructors be able to import survey questions from a question bank, or are surveys always standalone?
3. Should there be a "course evaluation" system-template that creates a standard set of questions for all instructors?
4. Is `pseudo_anonymous` mode legally compliant for FERPA purposes — does the server-side user_id storage still constitute "educational record access by school official"?
