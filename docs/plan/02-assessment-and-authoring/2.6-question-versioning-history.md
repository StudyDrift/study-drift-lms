## 1. Problem Statement

Editing a question in Lextures silently changes it for all quizzes and past attempts. An instructor who corrects a typo in a question stem effectively changes what every student historically saw. An accreditation auditor reviewing past exam integrity would find the stored question differs from what was delivered. This is especially critical in HE, where exam records must be immutable for academic integrity appeals. Building question versioning ensures that each attempt records the exact question state at delivery time, supports post-hoc corrections with audit visibility, and allows instructors to maintain a change history without losing prior data.

## 2. Goals

- Make questions in the question bank (2.1) immutable once published: edits create new versions.
- Record which version of a question was delivered in each quiz attempt.
- Provide an instructor-facing version history view per question.
- Allow instructors to revert to a prior version or set a specific version as "current."
- Keep past attempts pointing to their originally delivered version, never silently updating.

## 3. Non-Goals

- Branching / forking of question versions (linear version chain only in v1).
- Versioning of quiz-level metadata (not just question-level).
- Content versioning for assignments or course pages (separate concern).
- Automatic re-grading when a question is corrected (covered as an explicit instructor action, not automatic).

## 4. Personas & User Stories

- **As an HE instructor**, I want to fix a typo in a published question without changing what past students saw, so the exam record remains accurate.
- **As an academic integrity officer**, I want to see the exact question text delivered to a student in a specific attempt, even if the question has since been edited.
- **As an instructor**, I want to see a changelog for a question showing who changed what and when, so I can trace the evolution of a question over multiple semesters.
- **As a course author**, I want to create a new version of a question for the current semester while keeping the old version active in archived quizzes.
- **As a developer**, I want attempt records to reference a `(question_id, version_number)` pair so there is no ambiguity about what was delivered.

## 5. Functional Requirements

- **FR-1.** The system MUST store a `version_number` on each question in the question bank, starting at 1 for newly created questions.
- **FR-2.** The system MUST create a new version when any of the following fields are edited on a published question: `stem`, `options`, `correct_answer`, `explanation`, `points`.
- **FR-3.** The system MUST store version history in an append-only `question_versions` table, capturing the full question state (as JSONB snapshot) for each version.
- **FR-4.** The system MUST record `(question_id, version_number)` on each entry in `attempt_question_selections` (from 2.1) at the moment of attempt start, so grading always uses the version delivered.
- **FR-5.** The system MUST expose `GET /api/v1/questions/{id}/versions` returning the list of versions with timestamp, author, and change summary.
- **FR-6.** The system MUST expose `GET /api/v1/questions/{id}/versions/{v}` returning the full question state at version v.
- **FR-7.** The system MUST expose `POST /api/v1/questions/{id}/versions/{v}/restore` allowing instructors to set a prior version as the new current version (creates a new version number rather than mutating history).
- **FR-8.** The system SHOULD generate a diff summary between consecutive versions (field-level: "stem changed", "option B changed", "correct answer changed").
- **FR-9.** The system SHOULD require an optional `change_note` (free-text) on each edit operation for the version history log.

## 6. Non-Functional Requirements

- **Performance** — Version history list for a question (up to 50 versions) MUST respond in p95 ≤ 100 ms. Attempt start version snapshot write MUST add ≤ 10 ms to the attempt-start transaction.
- **Security** — Version history is instructor/admin-only read. Students cannot access version history, even for their own attempts. Restoring a version is instructor/admin-only. No version can be deleted (append-only integrity).
- **Privacy & Compliance** — Question text may contain named examples referencing students; FERPA applies. Version history records `created_by` (instructor user_id); this is NOT a student record. GDPR erasure of an instructor's data: anonymise `created_by` in version history.
- **Accessibility** — Version history table MUST use `role="table"` with `scope="col"` headers. Diff view MUST use `<ins>` and `<del>` elements with accessible colour contrast.
- **Scalability** — Maximum 50 versions per question is a soft cap (warn at 50; do not hard-block). `question_versions` table indexed by `(question_id, version_number)`.
- **Reliability** — Version write MUST be atomic with the question update in the same transaction. If the version write fails, the question update is rolled back.
- **Observability** — Counter `question_versions.created`. Log version restore events. Alert if version write failure rate > 0.1 %.
- **Maintainability** — Versioning logic in `server/src/services/question_versioning.rs`. Version snapshot is a full JSONB dump of the question row (not just deltas) for simplicity and queryability.
- **Internationalization** — Change notes are free-text UTF-8. Version timestamp displayed in user locale and timezone.
- **Backward compatibility** — Existing questions created before this feature have `version_number = 1` and a synthetic version-1 snapshot created by the migration. Existing attempts reference `version_number = 1` via backfill.

## 7. Acceptance Criteria

- **AC-1.** *Given* an instructor edits the stem of a published question, *when* the edit is saved, *then* a new version row is created in `question_versions` and `question.version_number` increments by 1.
- **AC-2.** *Given* a student starts a quiz, *when* the attempt is created, *then* `attempt_question_selections` records the `version_number` in effect at that moment for each question.
- **AC-3.** *Given* a question is at version 3, *when* an instructor restores version 1, *then* a new version 4 is created with the content of version 1, and the question now returns version 4 on read.
- **AC-4.** *Given* an academic integrity officer views an attempt, *when* they inspect a specific question, *then* the system shows the exact question text from the version delivered (even if the question is now at a higher version).
- **AC-5.** *Given* a question has 3 versions, *when* `GET /api/v1/questions/{id}/versions` is called, *then* the response lists 3 entries with timestamps, authors, and change summaries.
- **AC-6.** *Given* a student is reviewing their completed attempt, *when* they view a question, *then* they see the version delivered to them (not the latest version), and the UI shows "Version delivered: 2 (current: 4)" if applicable.

## 8. Data Model

```sql
-- Migration: server/migrations/0044_question_versioning.sql

ALTER TABLE course.questions
    ADD COLUMN version_number INTEGER NOT NULL DEFAULT 1,
    ADD COLUMN is_published   BOOLEAN NOT NULL DEFAULT FALSE;

CREATE TABLE course.question_versions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    question_id     UUID NOT NULL REFERENCES course.questions(id) ON DELETE CASCADE,
    version_number  INTEGER NOT NULL,
    snapshot        JSONB NOT NULL,    -- full copy of question row at this version
    change_note     TEXT,
    change_summary  JSONB,             -- { field: [old_value, new_value] }
    created_by      UUID REFERENCES "user".users(id),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (question_id, version_number)
);

CREATE INDEX idx_qv_question ON course.question_versions (question_id, version_number DESC);

-- Add version_number to attempt_question_selections (from 2.1)
ALTER TABLE course.attempt_question_selections
    ADD COLUMN version_number INTEGER NOT NULL DEFAULT 1;
```

Backfill: for all existing questions, insert a `question_versions` row with `version_number = 1` and a JSONB snapshot from the current row. Set `attempt_question_selections.version_number = 1` for all existing rows.

## 9. API Surface

```
GET  /api/v1/questions/{id}/versions              Auth: instructor|admin
  Response: { versions: VersionSummary[] }

GET  /api/v1/questions/{id}/versions/{v}          Auth: instructor|admin
  Response: QuestionSnapshot

GET  /api/v1/questions/{id}/versions/{v}/diff     Auth: instructor|admin
  Response: { diffs: [{field, old, new}] }

POST /api/v1/questions/{id}/versions/{v}/restore  Auth: instructor|admin
  Body: { change_note?: string }
  Response 200: { new_version_number: number }

-- Existing question update endpoint gains optional change_note:
PUT  /api/v1/courses/{cid}/questions/{qid}
  Body: { ...fields, change_note?: string }
```

## 10. UI / UX

**Question bank — version history panel** (side drawer on question row action):
- Timeline of versions: date, author, change summary pills (e.g., "stem changed", "correct answer changed").
- "View this version" to open a read-only preview.
- "Restore this version" button (instructor/admin only).
- Version diff modal: side-by-side old/new with `<ins>`/`<del>` highlighting.

**Quiz attempt review — question header**:
- Small badge: "Version 2 (latest: 4)" when a student views a past attempt and the question has been updated.
- Tooltip: "This question was updated after you completed this attempt. Your grade is based on the version you saw."

Accessibility: diff view uses `<ins>` (underline) and `<del>` (strikethrough) with colour that meets AA contrast; not colour-only.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Snapshot JSONB grows too large over many versions | L | M | Soft cap at 50 versions per question; archive older versions to cold storage at cap |
| Instructors confused about why edits don't show in past attempts | M | M | In-UI explanation banner: "Edits create a new version and don't affect past submissions." |
| Backfill migration on large question tables is slow | M | M | Run backfill in batches of 1 000 rows; can be run offline before flag flip |
| Restore operation creates confusion about "current" version | L | M | Always create a new version on restore (never mutate history); label restored version in history |

## 15. Rollout Plan

- **Feature flag:** `question_versioning_enabled` (default: `false`).
- **Migration sequencing:** add schema columns → backfill version-1 snapshots → deploy versioning service → deploy UI → enable for pilot.
- **Pilot cohort:** 2 HE courses; instructors edit 10 questions and verify version history.
- **GA criteria:** backfill complete; version increment verified on edit; attempt delivery records version_number.
- **Rollback:** disable flag; questions revert to mutable-in-place behaviour; version tables remain (additive).

## 16. Test Plan

- **Unit** — Version increment on edit; restore creates new version with old content; change-summary diff generation.
- **Integration** — Create question → publish → edit stem → assert version 2 created; start attempt → assert version_number recorded; edit again → restore version 1 → assert version 3 with version-1 content.
- **End-to-end** — Playwright: instructor edits question, views version history drawer with 2 entries, restores version 1, verifies version 3 now current.
- **Security** — Student cannot access version history endpoint (403); student sees correct version in attempt review.
- **Accessibility** — Diff view `<ins>`/`<del>` with accessible contrast; version history timeline keyboard navigable.
- **Performance** — Version list for 50 versions: p95 ≤ 100 ms; attempt-start transaction overhead ≤ 10 ms.
- **Manual exploratory** — Verify backfill created version-1 for all existing questions; verify restored version shows correct content; verify attempt review badge.

## 18. Open Questions

1. Should `change_note` be required on publish (not just on edit) to force instructors to explain changes?
2. Should students be able to see the version delivered to them in the attempt review, or should version numbers be hidden from students entirely?
3. What is the retention policy for old version snapshots — should versions older than 5 years be archived to cold storage?
4. Should version history be included in QTI export (2.13), or only the current version?
