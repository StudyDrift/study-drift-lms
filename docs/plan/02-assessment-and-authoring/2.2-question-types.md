## 1. Problem Statement

Lextures supports only multiple-choice, true/false, and short-answer question types with AI grading. This is insufficient for every STEM, language, and vocational course that requires matching, ordering, drag-and-drop, hotspot region selection, numeric answers with tolerance and units, formula/LaTeX input, code execution, file upload, and audio/video responses. Instructors attempting to migrate from Canvas, Schoology, or Moodle encounter immediate blockers when their existing item banks require these types. Without a full question-type palette, the platform is not viable for the HE and K-12 markets it targets.

## 2. Goals

- Implement 8 new question types in the question bank (2.1) and quiz delivery UI: matching, ordering, hotspot/image-region, numeric-with-tolerance, formula (LaTeX input), code-execution (stub for 2.4), file-upload, and audio/video response.
- Provide a consistent author-side editor UI for each type, integrated into the question bank editor.
- Provide a consistent student-side rendering and interaction layer for each type in the quiz-take UI.
- Implement automatic grading logic for types where objective grading is possible; route others to instructor/AI grading queue.

## 3. Non-Goals

- Code execution sandbox (covered in 2.4 — code type here only stubs the delivery and stores the submission).
- Math input keyboard beyond LaTeX text entry (covered in 2.3).
- AI grading of file/audio/video submissions (covered in 19.6 and 19.7).
- Drag-and-drop on touch with full accessibility parity in v1 (deferred to accessibility audit; keyboard alternative required).

## 4. Personas & User Stories

- **As a biology instructor**, I want a hotspot question asking students to click on the mitochondrion in a cell diagram so I can assess visual identification.
- **As a math instructor**, I want students to enter numeric answers with a tolerance of ±0.01 so minor rounding differences don't mark correct answers wrong.
- **As a foreign-language instructor**, I want a matching question pairing vocabulary words to definitions, with items drawn in random order per student.
- **As a CS student**, I want to submit code that is run against test cases, not just scored for syntax (stub for 2.4; in this plan the submission is stored).
- **As an instructor**, I want students to upload a lab report as a PDF and mark it for manual review.
- **As a language learner**, I want to record a 30-second spoken response to an oral prompt.

## 5. Functional Requirements

- **FR-1.** The system MUST implement `matching` type: two ordered columns of pairs; student drags/selects left items to match right items; auto-graded by exact pairing.
- **FR-2.** The system MUST implement `ordering` type: a list of items the student places in the correct sequence; graded by position match or partial credit (Kendall τ-based scoring).
- **FR-3.** The system MUST implement `hotspot` type: an image with one or more correct clickable regions defined by author as polygons or rectangles; graded by click inside region.
- **FR-4.** The system MUST implement `numeric` type: a single numerical answer with author-configurable absolute tolerance (±value), relative tolerance (±%), and optional unit string matching.
- **FR-5.** The system MUST implement `formula` type: a LaTeX string input rendered via KaTeX (2.3); graded by symbolic equivalence check (exact string or configurable equivalence list).
- **FR-6.** The system MUST implement `code` type: a code editor with language selection; submission stored as `code_submission` for execution by 2.4; pending grading state.
- **FR-7.** The system MUST implement `file_upload` type: accepts files up to 50 MB (configurable); stores on object storage (8.1); instructor manually grades or AI grades (19.3).
- **FR-8.** The system MUST implement `audio_response` type: browser MediaRecorder API; max 5-min recording; stored as audio file; manual or AI grading.
- **FR-9.** The system MUST implement `video_response` type: max 10-min recording or file upload; stored; manual grading.
- **FR-10.** The system MUST store type-specific authoring metadata in a structured JSONB `type_config` column per question in the question bank.

## 6. Non-Functional Requirements

- **Performance** — Standard question types (matching, ordering, numeric) MUST render in ≤ 200 ms. Hotspot image MUST be served from CDN with < 500 ms TTFB. File upload progress shown for uploads > 1 MB.
- **Security** — File, audio, and video uploads MUST be virus-scanned (ClamAV, 8.6) before storage. File type validation server-side (MIME + magic bytes), not client-only. Hotspot region definitions not exposed to students before submission.
- **Privacy & Compliance** — Audio/video responses are student biometric-adjacent data; FERPA-protected. GDPR erasure MUST delete stored media files from object storage. COPPA: under-13 audio/video recording requires verifiable parental consent (see 10.2).
- **Accessibility** — Matching and ordering MUST have keyboard-alternative interaction (not drag-only). Hotspot MUST have a keyboard-navigable alternative (numbered region selector). WCAG 2.1 AA required for all new question-type UIs. Audio/video MUST have a text-alternative response path for ADA accommodations.
- **Scalability** — File and media uploads routed to object storage (8.1), not local disk. Up to 10 k concurrent file submissions during finals week.
- **Reliability** — Partial uploads MUST resume (tus protocol, 8.2). Grading for auto-graded types MUST be idempotent.
- **Observability** — Counter `quiz.question_type_submitted` labeled by type. Log file upload events with size and MIME type. Alert if virus scan failure rate > 0 %.
- **Maintainability** — Each question type is a Rust enum variant in `server/src/models/question_types.rs` and a React component in `clients/web/src/components/quiz/types/`. New types added by adding an enum variant + component pair.
- **Internationalization** — All UI labels (column headers in matching, position labels in ordering) use i18n keys. Unit strings in numeric type are stored as authored (locale-dependent).
- **Backward compatibility** — Existing MC/TF/short-answer delivery unchanged. New types are additive question_type enum variants.

## 7. Acceptance Criteria

- **AC-1.** *Given* a matching question with 4 pairs, *when* a student correctly matches all 4 pairs, *then* the score is 100 %; partial credit (3/4 correct) returns 75 %.
- **AC-2.** *Given* a numeric question with correct answer 3.14 and tolerance ±0.01, *when* a student enters 3.145, *then* the answer is marked correct; 3.16 is marked incorrect.
- **AC-3.** *Given* a hotspot question with a rectangular region defined, *when* a student clicks inside the region, *then* the answer is marked correct; a click outside is marked incorrect.
- **AC-4.** *Given* a file-upload question, *when* a student uploads a 10 MB PDF, *then* the file is stored, a submission record created, and the question is placed in "pending grading" state.
- **AC-5.** *Given* a matching question rendered in the browser, *when* a student uses only keyboard (Tab + Enter) to complete all matches, *then* the question is completable without a mouse.
- **AC-6.** *Given* an audio response question, *when* the student records and submits audio, *then* the submission is stored and the instructor sees a playback control in the grading view.

## 8. Data Model

```sql
-- Migration: server/migrations/0041_question_types.sql
-- Adds type_config JSONB column and new enum variants to course.question_type

-- type_config JSONB examples per type:
-- matching:  { pairs: [{left, right}], shuffle: bool }
-- ordering:  { items: [string], partial_credit: bool }

-- numeric:   { correct, tolerance_abs?, tolerance_pct?, unit? }
-- formula:   { latex_answer, equivalences: [string] }
-- code:      { language, starter_code?, test_cases: [{input, expected_output}] }
-- file_upload: { max_mb, allowed_mime_types: [string] }
-- audio_response: { max_duration_s }
-- video_response: { max_duration_s, max_mb }

ALTER TABLE course.questions ADD COLUMN type_config JSONB NOT NULL DEFAULT '{}';
```

Submission storage: file/audio/video submissions stored as object storage keys in `attempt_responses.file_key` (new column).

## 9. API Surface

All new types use the existing quiz-attempt and grading endpoints with the question_type discriminating the grading logic. No new top-level routes needed beyond those in 2.1.

New field in attempt-response submission:
```typescript
interface SubmitResponse {
  question_id: string;
  // One of:
  mc_selection?: string;
  matching_pairs?: {left_id: string, right_id: string}[];
  ordering_sequence?: string[];
  hotspot_click?: {x: number, y: number};
  numeric_value?: number;
  formula_latex?: string;
  code_submission?: {language: string, code: string};
  file_key?: string;          // pre-signed upload key returned by presign endpoint
  audio_key?: string;
  video_key?: string;
}
```

Presigned upload: `POST /api/v1/quiz-attempts/{id}/questions/{qid}/presign-upload` returns S3/R2 presigned PUT URL.

## 10. UI / UX

New React components in `clients/web/src/components/quiz/types/`:
- `MatchingQuestion.tsx` — two-column drag-or-select matching UI.
- `OrderingQuestion.tsx` — draggable list with keyboard alternative (arrow keys + confirm).
- `HotspotQuestion.tsx` — `<canvas>` overlay on image; click to mark; keyboard: Tab to cycle regions + Space to select.
- `NumericQuestion.tsx` — number input with unit text field.
- `FormulaQuestion.tsx` — LaTeX text input with live KaTeX preview (2.3).
- `CodeQuestion.tsx` — Monaco editor with language selector.
- `FileUploadQuestion.tsx` — dropzone with progress bar.
- `AudioResponseQuestion.tsx` — record/stop/playback controls.
- `VideoResponseQuestion.tsx` — record or upload, preview.

Author-side components mirror student-side with an additional "correct answer" configuration panel.

## 11. AI / ML Considerations

Formula grading: symbolic equivalence check is rule-based in v1 (exact LaTeX string or equivalence list). V2: use symbolic math evaluation (e.g., `sympy` via subprocess or Rust `mathexpr` crate) for proper equivalence.

Code type grading: deferred to 2.4. Audio/video grading: deferred to 19.6/19.7.

File upload: AI grading route available as a post-submission background job (19.3).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Drag-and-drop inaccessible on touch/keyboard | H | H | Ship keyboard alternative before drag-and-drop; test with screen reader before GA |
| File uploads fill local disk before S3 migration | H | H | Gate file-upload type on object storage feature flag (8.1) |
| Formula equivalence check too strict (LaTeX whitespace) | M | M | Normalise LaTeX before comparison; allow equivalence list per question |
| Audio/video recording unavailable on some browsers | M | M | Show file-upload fallback when MediaRecorder API is unavailable |

## 15. Rollout Plan

- **Feature flag:** `extended_question_types_enabled` (per-type sub-flags: `numeric`, `matching`, `hotspot`, `file_upload`, `audio_video`).
- Ship matching and numeric in week 1; ordering and hotspot in week 2; file/audio/video gated on 8.1 (object storage).
- **Pilot cohort:** two HE STEM courses for numeric/formula; one K-12 science course for hotspot.
- **GA criteria:** all auto-graded types have unit test coverage ≥ 90 %; keyboard accessibility audited; object storage integration working.
- **Rollback:** per-type feature flags; disable individual types without affecting others.

## 16. Test Plan

- **Unit** — Matching grader (exact, partial credit); numeric grader (tolerance absolute, relative, unit string); hotspot point-in-polygon; ordering Kendall τ scorer.
- **Integration** — Create matching question → student submits → verify auto-grade; file upload → verify virus-scan pipeline → verify pending state.
- **End-to-end** — Playwright: numeric question entry + submit; matching drag-and-drop + keyboard alternative.
- **Security** — File MIME validation test (upload a PHP file disguised as PDF); hotspot region coordinates not exposed in HTML before submission.
- **Accessibility** — axe + manual screen-reader test for matching (keyboard), hotspot (keyboard alternative), audio recording (fallback).
- **Performance / load** — 200 concurrent file uploads (10 MB each); p95 upload-completion time ≤ 3 s.
- **Manual exploratory** — Verify partial credit on ordering; verify audio playback in instructor grading view; verify hotspot on mobile touch.

## 18. Open Questions

1. Should partial credit for ordering be Kendall τ (penalises inversions) or simple positional match (simpler)? Kendall τ is more defensible but harder to explain to instructors.
2. Should formula grading perform symbolic simplification (e.g., `x^2 + 2x + 1 = (x+1)^2`) or require exact LaTeX equivalence in v1?
3. What file types should be allowlisted for file-upload questions by default?
4. Should audio/video response questions require verifiable parental consent for under-13 students in the platform UI, or is that handled entirely by the platform's COPPA mode (10.2)?
