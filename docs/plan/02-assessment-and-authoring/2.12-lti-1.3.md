## 1. Problem Statement

Learning Tools Interoperability (LTI) 1.3 (IMS Global standard) is the integration backbone of every university LMS ecosystem. Publishers (Cengage, Pearson, Macmillan, Wiley) deliver digital textbooks and adaptive courseware via LTI 1.3 deep linking. Proctoring services (Honorlock, Respondus) launch via LTI. Library reserves tools, accessibility tools, and institutional systems all use LTI 1.3. Without LTI consumer support, Lextures cannot embed any of these tools. Without LTI provider support, Lextures cannot be embedded into a parent LMS (e.g., a district running Canvas as SIS but wanting Lextures for adaptive features). Both roles are required for HE sales, and LTI 1.3 consumer is a BLOCKER that appears in virtually every HE RFP.

## 2. Goals

- Implement LTI 1.3 Advantage consumer: allow Lextures to launch external tools in course content using Deep Linking 2.0.
- Implement LTI 1.3 Advantage provider (Tool Provider): allow Lextures to be launched from a parent LMS, passing user identity and returning grade-passback via Assignments and Grades Services (AGS).
- Implement LTI Names and Role Provisioning Services (NRPS) for roster sync with parent LMS.
- Implement the platform registration flow so administrators can register external tools without developer intervention.
- Comply with IMS Global LTI 1.3 Core Specification and LTI Advantage feature set.

## 3. Non-Goals

- LTI 1.1 / 1.2 (OAuth 1.0 signed) — only LTI 1.3 (OAuth 2.0 / JWKS) in scope.
- LTI Resource Search / Advantage extension services beyond Deep Linking, AGS, NRPS.
- LTI certificate validation for institutional membership programmes (IMS certification testing in v2).
- Full publisher content marketplace integration (covered in 14.10, 14.11).

## 4. Personas & User Stories

- **As an HE instructor**, I want to embed a Pearson MyLab math tool into my course module so students access it from within Lextures without re-authenticating.
- **As an HE instructor**, I want Lextures to send the grade from a Pearson assessment back to the Lextures gradebook automatically (AGS grade-passback).
- **As a district IT admin**, I want to register Lextures as an LTI 1.3 tool in our Canvas instance so students can access Lextures from their existing LMS without a separate login.
- **As a developer**, I want to register an external LTI tool in Lextures by providing the tool's JWKS URL and redirect URIs without touching environment variables.
- **As a compliance officer**, I want all LTI ID tokens to be validated against the tool's JWKS and replay attacks prevented by checking `nonce` and `iat` claims.

## 5. Functional Requirements

- **FR-1.** The system MUST implement LTI 1.3 Core as both Platform (consumer) and Tool (provider), compliant with IMS Global LTI 1.3 Core Specification v1.3.
- **FR-2.** As **Consumer (Platform)**: the system MUST support LTI Deep Linking 2.0 to allow instructors to search and embed external tool content (resource links) into course modules.
- **FR-3.** As **Consumer (Platform)**: the system MUST implement AGS (Assignments and Grades Services) to receive grade-passback from external tools and write scores to the Lextures gradebook.
- **FR-4.** As **Consumer (Platform)**: the system MUST generate a signed LTI 1.3 ID token (RS256 JWT) for each launch, including standard claims: `sub`, `iss`, `aud`, `iat`, `exp`, `nonce`, `roles`, `context`, `resource_link`, `custom`.
- **FR-5.** As **Provider (Tool)**: the system MUST accept an LTI 1.3 ID token from a Platform, validate the RS256 signature against the Platform's JWKS, and authenticate/provision the user.
- **FR-6.** As **Provider (Tool)**: the system MUST return grades to the Platform via AGS line-item and score resources when a Lextures activity is launched from an external LMS.
- **FR-7.** The system MUST implement NRPS (Names and Role Provisioning Services) as both Platform (roster export) and Tool (roster import from parent LMS).
- **FR-8.** The system MUST provide an admin UI for registering LTI tools (Platform registration: tool's `client_id`, JWKS URL, auth endpoint, redirect URIs; Tool registration: Platform's issuer, JWKS URL, client credentials).
- **FR-9.** The system MUST prevent replay attacks: validate `nonce` is single-use (stored in cache for `exp - iat` duration) and reject tokens where `iat` is more than 10 minutes old.

## 6. Non-Functional Requirements

- **Performance** — LTI launch flow MUST complete in ≤ 3 s end-to-end (OIDC login initiation + launch redirect). JWKS key fetch MUST be cached with a 15-minute TTL. AGS grade-passback MUST complete in ≤ 2 s.
- **Security** — All LTI messages MUST be RS256-signed JWTs. Private signing keys stored in secrets manager (17.17). JWKS endpoints serve public keys only. `state` parameter MUST be validated in the OIDC login flow to prevent CSRF. Nonce stored in Redis or Postgres with TTL to prevent replay.
- **Privacy & Compliance** — LTI ID tokens contain user identifiers; FERPA governs data shared with external tools. Privacy notice MUST disclose which data fields are sent to each registered LTI tool. GDPR: data transfer to external tools may cross EU borders; require DPA with each tool vendor.
- **Accessibility** — LTI launch results in an iframe or a redirect; the iframe MUST have a `title` attribute. If the tool is inaccessible, instructors MUST be able to provide an alternative access path.
- **Scalability** — Expect 10 k simultaneous LTI launches during peak. Nonce validation uses Redis; JWKS cache reduces external JWKS fetches to near-zero during steady state.
- **Reliability** — JWKS fetch failure MUST fail the launch with a clear error ("Tool configuration error") rather than silently ignoring signature validation. AGS grade-passback failures MUST be retried via job queue (17.3) and logged.
- **Observability** — Counter `lti.launches` (labeled by tool_id, role), `lti.grade_passback_success`, `lti.grade_passback_failure`. Alert if passback failure rate > 5 %.
- **Maintainability** — LTI platform logic in `server/src/services/lti.rs`; OIDC/JWT utilities in `server/src/services/lti_jwt.rs`. No LTI-specific logic in route handlers.
- **Internationalization** — LTI launch parameters carry locale (`launch_presentation.locale`); Lextures MUST forward the user's locale in all LTI launches.
- **Backward compatibility** — LTI is fully additive; no existing routes or data affected.

## 7. Acceptance Criteria

- **AC-1.** *Given* an LTI tool is registered with a valid JWKS URL, *when* an instructor launches it from a course module, *then* the user is authenticated in the external tool with the Lextures user's `sub` claim visible to the tool.
- **AC-2.** *Given* an LTI tool sends an AGS grade-passback, *when* the score arrives at the `/api/v1/lti/scores` endpoint with a valid signature, *then* the gradebook for the corresponding student and activity is updated.
- **AC-3.** *Given* a Platform sends an LTI 1.3 ID token with a used `nonce`, *when* Lextures validates the token, *then* the response is HTTP 400 "nonce_already_used."
- **AC-4.** *Given* Lextures is launched as a Tool from Canvas, *when* the LTI handshake completes, *then* the user is logged into Lextures (or provisioned if new) and redirected to the correct course context.
- **AC-5.** *Given* NRPS is invoked by an external Platform, *when* the roster endpoint is called with a valid Bearer token, *then* the course roster is returned in IMS NRPS JSON format.
- **AC-6.** *Given* a JWKS fetch times out during a launch, *when* the launch is attempted, *then* the response is a user-facing error "Tool configuration error — please contact your administrator" and the error is logged.

## 8. Data Model

```sql
-- Migration: server/migrations/0050_lti.sql

CREATE TABLE settings.lti_registrations (
    id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name              TEXT NOT NULL,
    client_id         TEXT NOT NULL,
    platform_iss      TEXT NOT NULL,
    platform_jwks_url TEXT NOT NULL,
    platform_auth_url TEXT NOT NULL,
    platform_token_url TEXT NOT NULL,
    tool_redirect_uris TEXT[] NOT NULL,
    public_key_id     TEXT NOT NULL,    -- reference to key in secrets store
    deployment_ids    TEXT[] NOT NULL DEFAULT '{}',
    active            BOOLEAN NOT NULL DEFAULT TRUE,
    created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (platform_iss, client_id)
);

CREATE TABLE settings.lti_nonces (
    nonce       TEXT NOT NULL,
    expires_at  TIMESTAMPTZ NOT NULL,
    PRIMARY KEY (nonce)
);
CREATE INDEX idx_lti_nonces_exp ON settings.lti_nonces (expires_at);

CREATE TABLE course.lti_resource_links (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    course_id           UUID NOT NULL REFERENCES course.courses(id) ON DELETE CASCADE,
    registration_id     UUID NOT NULL REFERENCES settings.lti_registrations(id),
    resource_link_id    TEXT NOT NULL,
    title               TEXT,
    custom_params       JSONB NOT NULL DEFAULT '{}',
    line_item_url       TEXT,    -- AGS line item for grade-passback
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## 9. API Surface

```
-- Platform-side (Lextures as LTI Platform / Consumer):
GET  /.well-known/jwks.json                    Public — serves Lextures platform public keys
POST /api/v1/lti/launch/{registration_id}      Initiates OIDC login
GET  /api/v1/lti/callback                      OIDC auth_response callback
POST /api/v1/lti/deep-link                     Deep Link content-item response handler
POST /api/v1/lti/scores                        AGS score service (inbound grade-passback)

-- Tool-side (Lextures as LTI Tool / Provider):
GET  /api/v1/lti/provider/jwks                 Public — serves Lextures tool public keys
POST /api/v1/lti/provider/login                OIDC login initiation (from Platform)
POST /api/v1/lti/provider/launch               ID token validation + user provisioning
GET  /api/v1/lti/provider/nrps/memberships     NRPS roster endpoint

-- Admin registration:
GET    /api/v1/admin/lti/registrations         Auth: admin
POST   /api/v1/admin/lti/registrations         Auth: admin
PUT    /api/v1/admin/lti/registrations/{id}    Auth: admin
DELETE /api/v1/admin/lti/registrations/{id}    Auth: admin
```

## 10. UI / UX

**LTI tool registration page** (`/admin/lti-tools`, new admin page):
- Registration form: tool name, client_id, JWKS URL, auth endpoint, redirect URIs.
- "Test connection" button — attempts JWKS fetch and validates signature.
- Active/inactive toggle.
- List of registered tools with launch count and last-used date.

**Course module — embed LTI tool** (in course editor):
- "Add external tool" option in the module item type picker.
- Deep-link flow: click → tool's content selector opens in iframe → instructor selects content → item added to module.

**LTI launch iframe** (student view):
- Embedded iframe with tool's title as `title` attribute.
- Fallback message for iframe-blocking tools: "Open [Tool Name] in a new tab."

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| RSA key rotation breaks in-flight LTI launches | M | H | Serve both old and new keys in JWKS for 24 h during rotation |
| Tool sends malformed ID token, causing unhandled panic | M | H | Wrap all JWT parsing in error boundaries; return structured error to platform |
| AGS grade-passback fails silently | M | M | Async with retry; alert on repeated failures; instructor dashboard shows passback status |
| JWKS fetch rate-limited by tool vendor | L | M | Cache JWKS aggressively (15-min TTL); back off on 429 responses |

## 15. Rollout Plan

- **Feature flag:** `lti_enabled` (default: `false`; admin must configure at least one registration).
- **Migration sequencing:** deploy migration → generate RSA key pair → store in secrets manager → deploy LTI routes → register a test tool → verify launch → enable for pilot institution.
- **Pilot cohort:** one HE institution; register Pearson or OER publisher tool; test deep-link + grade-passback.
- **GA criteria:** end-to-end LTI 1.3 launch verified; AGS grade-passback writes to gradebook; nonce replay rejected; IMS conformance test suite results reviewed.
- **Rollback:** disable flag; LTI routes return 404; no data loss.

## 16. Test Plan

- **Unit** — RS256 sign/verify; nonce uniqueness check; AGS score mapping to gradebook entry; NRPS roster serialisation.
- **Integration** — Full LTI 1.3 launch flow with a real (or mock) external tool; AGS passback end-to-end; nonce replay rejection.
- **End-to-end** — Playwright: instructor embeds a mock LTI tool in a course module, student clicks and launches, grade appears in gradebook.
- **Security** — Invalid JWT signature → rejected; expired token (> 10 min old) → rejected; used nonce → rejected; CSRF state mismatch → rejected.
- **Accessibility** — LTI iframe has `title`; fallback message accessible for iframe-blocked tools.
- **Performance / load** — 1 000 concurrent LTI launches; p95 ≤ 3 s; JWKS cache hit ratio ≥ 99 %.
- **Manual exploratory** — Test with real Canvas as Platform; test with Pearson as Tool; test NRPS roster sync.

## 18. Open Questions

1. Should Lextures support LTI 1.3 Dynamic Registration (automatic registration via a single URL) in v1, or manual registration only?
2. Should we pursue IMS Global LTI 1.3 conformance certification? Required by some enterprise buyers.
3. For the provider role, should Lextures create a new user account when a Platform-provisioned user is seen for the first time, or require them to link to an existing account?
4. How should we handle multi-tenant deployments — does each organisation get its own LTI client_id/JWKS key pair?
