## 1. Problem Statement

Section 504 of the Rehabilitation Act, the Individuals with Disabilities Education Act (IDEA), and ADA Title II collectively require that K-12 schools and HE institutions provide testing accommodations to eligible students. Common accommodations include: extended time (typically 1.5× or 2×), extra attempts, screen-reader compatibility, text-to-speech, reduced-distraction environment (separate testing room), and alternate-format delivery. Lextures has no accommodation infrastructure whatsoever. This is a BLOCKER for K-12 and HE sales: any institutional procurement process checks for accommodation support, and any school that enrolls a student with an IEP or 504 plan is legally required to implement these accommodations. Without this feature, Lextures exposes institutions to ADA and IDEA compliance risk.

## 2. Goals

- Build an accommodation record system: per-student, per-course (or global) accommodation profiles with configurable override fields.
- Automatically apply time-limit multipliers, extra-attempt grants, and hint-availability overrides at quiz delivery time from the student's accommodation profile.
- Provide an accessible "reduced-distraction" quiz mode that strips non-essential UI chrome.
- Allow accessibility services coordinators (or admins) to manage accommodation records; instructors read accommodation flags but not the underlying disability documentation.
- Maintain FERPA-compliant separation between disability documentation (stored by accessibility office, not Lextures) and the operational accommodation settings.

## 3. Non-Goals

- Storage of disability documentation or IEP/504 plan documents (Lextures stores only accommodation settings, not the underlying diagnosis or legal plan document).
- Text-to-speech engine (covered in 12.8).
- Speech-to-text input (covered in 12.9).
- Alternate-format content generation (AI reading level adaptation, 11.6).
- Custom rubric adjustments per student (covered in assignment grading rules).

## 4. Personas & User Stories

- **As an accessibility services coordinator (HE)**, I want to create an accommodation profile for a student with ADHD specifying 1.5× time on all exams and an extra attempt on quizzes, so every instructor automatically applies the correct settings.
- **As an instructor**, I want to see that a student has accommodations (without seeing the underlying disability) when setting up a quiz, so I know the system will handle it automatically.
- **As a student with an accommodation**, I want my extended time to apply automatically without having to ask my instructor each time.
- **As a K-12 student with an IEP**, I want to take a quiz in "reduced-distraction mode" that hides animations and sidebars, helping me focus.
- **As a parent/guardian**, I want to verify that my child's accommodation settings are correctly applied to all assessments in the LMS.

## 5. Functional Requirements

- **FR-1.** The system MUST store accommodation records (`student_accommodations` table) per (user_id, course_id or global) with configurable fields: `time_multiplier` (NUMERIC, default 1.0), `extra_attempts` (INTEGER, default 0), `hints_always_enabled` (BOOLEAN), `reduced_distraction_mode` (BOOLEAN), `alternative_format` (TEXT, freeform note), `effective_from`, `effective_until`.
- **FR-2.** The system MUST automatically apply `time_multiplier` at attempt start: `deadline_at = started_at + (time_limit_seconds * time_multiplier)`. This MUST be transparent to instructors (they see "Extended time active" on the attempt row).
- **FR-3.** The system MUST automatically apply `extra_attempts` when checking `max_attempts` enforcement: effective max = `quiz.max_attempts + accommodation.extra_attempts`.
- **FR-4.** The system MUST override `hints_disabled` for students with `hints_always_enabled = true`, even in kiosk mode (2.10).
- **FR-5.** The system MUST provide a `reduced_distraction_mode` quiz delivery flag that, when active, strips animated elements, auto-playing media, decorative imagery, and non-essential navigation from the quiz-take page.
- **FR-6.** The system MUST restrict accommodation record creation/editing to `accessibility_coordinator` and `admin` roles; instructors MUST NOT be able to modify accommodation records.
- **FR-7.** The system MUST expose a read-only flag to instructors: `GET /api/v1/enrollments/{id}/accommodation-summary` returning `{ has_accommodation: bool, flags: ['extended_time', 'extra_attempts', 'reduced_distraction'] }` without revealing specific multiplier values or disability information.
- **FR-8.** The system SHOULD support course-scoped and global (all courses) accommodation records; course-scoped overrides global.
- **FR-9.** The system MAY send a notification to the instructor when a student with accommodations enrolls in their course.

## 6. Non-Functional Requirements

- **Performance** — Accommodation lookup at attempt start MUST add ≤ 5 ms (single row indexed lookup).
- **Security** — Accommodation records are highly sensitive (disability-adjacent) PII. Restricted to `accessibility_coordinator` and `admin` roles. Instructors see only a boolean flag. Students see only their own accommodation summary (not specific multiplier values). Row-level encryption on `alternative_format` field.
- **Privacy & Compliance** — Accommodation data is FERPA-protected (educational record) and ADA-sensitive. GDPR: accommodation records are sensitive health-adjacent data (Article 9 special categories in EU interpretation); require explicit consent for processing or fall under the legitimate-interests basis for disability services. COPPA: under-13 accommodation data requires verifiable parental consent. DSAR export includes accommodation records. Erasure: anonymise rather than hard-delete (legal obligation may require retention for compliance audit).
- **Accessibility** — Reduced-distraction mode MUST remove `prefers-reduced-motion` violations, auto-playing elements, and animated distractors. The accommodation summary page for accessibility coordinators MUST be fully keyboard navigable (WCAG 2.1 AA).
- **Scalability** — Low volume (< 20 % of students have accommodations); no special scaling concern. Accommodation lookup is a single indexed join on the hot path.
- **Reliability** — If accommodation lookup fails, quiz proceeds with default settings (no accommodation applied) and an error is logged for manual review. Do not block the exam.
- **Observability** — Log each accommodation application at INFO level (user_id hash, accommodation type, quiz_id). Prometheus counter `accommodations.applied` (labeled by type). Alert if accommodation lookup error rate > 0.1 %.
- **Maintainability** — Accommodation service in `server/src/services/accommodations.rs`. Applied at attempt-start in `server/src/services/quiz_attempt.rs`. No accommodation logic in route handlers.
- **Internationalization** — `alternative_format` is a freeform note (UTF-8). Accommodation summary flag labels use i18n keys.
- **Backward compatibility** — Accommodation tables are additive; no students have accommodation records until created. Quiz delivery is unchanged for students without records.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student has a `time_multiplier = 1.5` accommodation and a quiz has `time_limit_seconds = 3600`, *when* the student starts the attempt, *then* `deadline_at = started_at + 5400 seconds`.
- **AC-2.** *Given* a student has `extra_attempts = 1` and a quiz has `max_attempts = 2`, *when* the student has completed 2 attempts, *then* they are permitted one additional attempt (3 total).
- **AC-3.** *Given* a student has `hints_always_enabled = true` and the quiz is in kiosk mode, *when* the student requests a hint, *then* the hint endpoint returns the hint (not 403).
- **AC-4.** *Given* an instructor calls `GET /api/v1/enrollments/{id}/accommodation-summary`, *when* the student has a `time_multiplier = 2.0` accommodation, *then* the response shows `{ has_accommodation: true, flags: ['extended_time'] }` but NOT the specific multiplier value.
- **AC-5.** *Given* a student with `reduced_distraction_mode = true` starts a quiz, *when* the quiz page loads, *then* animated elements are hidden and the page uses a minimal UI layout.
- **AC-6.** *Given* an instructor (not an accessibility coordinator) tries to update a student's accommodation record via the API, *when* the request is received, *then* the response is HTTP 403.

## 8. Data Model

```sql
-- Migration: server/migrations/0049_accommodations.sql

CREATE TABLE course.student_accommodations (
    id                        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id                   UUID NOT NULL REFERENCES "user".users(id) ON DELETE CASCADE,
    course_id                 UUID REFERENCES course.courses(id) ON DELETE CASCADE,   -- NULL = global
    time_multiplier           NUMERIC(4,2) NOT NULL DEFAULT 1.0 CHECK (time_multiplier >= 1.0),
    extra_attempts            INTEGER NOT NULL DEFAULT 0,
    hints_always_enabled      BOOLEAN NOT NULL DEFAULT FALSE,
    reduced_distraction_mode  BOOLEAN NOT NULL DEFAULT FALSE,
    alternative_format        TEXT,    -- encrypted at-rest; accessibility office notes only
    effective_from            DATE,
    effective_until           DATE,
    created_by                UUID NOT NULL REFERENCES "user".users(id),
    updated_by                UUID REFERENCES "user".users(id),
    created_at                TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at                TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (user_id, course_id)   -- one record per user per scope
);

CREATE INDEX idx_sa_user ON course.student_accommodations (user_id, course_id);

-- Already included in 2.9 migration.
```

## 9. API Surface

```
-- Accessibility coordinator / admin only:
GET    /api/v1/users/{user_id}/accommodations           Auth: accessibility_coordinator|admin
POST   /api/v1/users/{user_id}/accommodations           Auth: accessibility_coordinator|admin
PUT    /api/v1/users/{user_id}/accommodations/{id}      Auth: accessibility_coordinator|admin
DELETE /api/v1/users/{user_id}/accommodations/{id}      Auth: accessibility_coordinator|admin

-- Instructor read-only summary:
GET    /api/v1/enrollments/{id}/accommodation-summary   Auth: instructor|admin
  Response: { has_accommodation: bool, flags: string[] }

-- Student: own summary only:
GET    /api/v1/me/accommodations                        Auth: authenticated student
  Response: { accommodations: AccommodationSummary[] }
```

## 10. UI / UX

**Accessibility coordinator dashboard** (new page `/admin/accommodations`):
- Search students by name/email.
- Per-student accommodation record form: time multiplier (numeric input with "1.5× = time-and-a-half" label), extra attempts, checkboxes for hints/reduced-distraction, alternative format notes field.
- Effective date range pickers.
- Audit log showing who created/modified the record.

**Instructor quiz view**:
- On the attempt list, students with active accommodations show a small accessibility icon badge.
- Hover/tooltip: "This student has accommodations — extended time and extra attempts have been applied automatically."
- No details beyond that badge.

**Reduced-distraction quiz UI**:
- Minimal layout: question + answer only; no course navigation sidebar; no animations; `prefers-reduced-motion` CSS forced; no autoplaying media; high-contrast option toggle.

Accessibility: accommodation management forms MUST be screen-reader navigable; multiplier input MUST have descriptive labels.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Accommodation data breach exposes disability information | L | H | Encrypt `alternative_format` column at-rest; restrict API to two roles; audit every access |
| Instructor bypasses system and manually adjusts time for non-accommodated student | M | M | Log all manual time extensions (2.7) for audit trail |
| Accommodation record not yet created when student starts exam | M | M | Accommodation lookup error falls back to default settings + instructor alert; never blocks exam |
| Global accommodation record conflicts with course-scoped override | L | M | Course-scoped record takes precedence; document precedence rules clearly |

## 15. Rollout Plan

- **Feature flag:** `accommodations_enabled` (default: `false`; requires `accessibility_coordinator` role provisioning first).
- **Migration sequencing:** add `accessibility_coordinator` role to `authz.rs` → deploy accommodation tables → deploy coordinator UI → enable for pilot institution.
- **Pilot cohort:** one HE disability services office; 20 students with documented accommodations.
- **GA criteria:** time multiplier verified in attempt `deadline_at`; extra attempts verified; no accommodation details visible to instructor.
- **Rollback:** disable flag; accommodation lookup skipped; quiz defaults apply.

## 16. Test Plan

- **Unit** — time_multiplier application formula; effective_from/until date boundary; global vs. course-scoped precedence.
- **Integration** — Create accommodation → student starts timed quiz → assert deadline_at reflects multiplier; kiosk exam + hints_always_enabled → assert hint returned.
- **End-to-end** — Playwright: accessibility coordinator creates accommodation, student starts exam, instructor views badge without seeing specifics.
- **Security** — Instructor cannot access `/users/{id}/accommodations` (403); student can only see own accommodations; `alternative_format` field not returned to instructor.
- **Accessibility** — Reduced-distraction mode: axe audit shows no animated elements; keyboard navigation works; contrast meets AA.
- **Performance** — Accommodation lookup at attempt start: ≤ 5 ms.
- **Manual exploratory** — Verify global vs. course-scoped precedence; verify effective date range enforcement; verify accommodation lookup failure → defaults applied.

## 18. Open Questions

1. Should accommodation records be shared across an institution's courses automatically (global), or must the accessibility coordinator create one per course?
2. Should students be able to view the specific multiplier values in their own accommodation summary, or only a high-level "You have extended time"?
3. Does the `alternative_format` freeform field need to be encrypted with a separate key (application-level encryption beyond Postgres at-rest), given its sensitivity?
