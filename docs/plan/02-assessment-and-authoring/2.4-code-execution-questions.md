## 1. Problem Statement

Computer science courses and coding bootcamps require students to write and run code as part of assessments. Lextures stores code submissions (from 2.2's `code` question type) but has no execution pipeline: submissions sit in "pending grading" indefinitely. Without a sandboxed code runner, Lextures cannot serve the HE CS market (introductory programming courses, algorithms courses) or the self-learner coding bootcamp market — two fast-growing segments where purpose-built platforms (Replit, Codewars, HackerRank) currently dominate. A code execution integration also removes the need for AI-graded code estimation and provides objective, reproducible grading.

## 2. Goals

- Integrate an external sandboxed code execution service (Judge0 CE as primary, Piston as self-hostable alternative) for executing student code submissions against instructor-defined test cases.
- Auto-grade code questions based on test-case pass/fail outcomes.
- Support the 10 most popular languages in HE CS courses: Python, Java, C, C++, JavaScript, TypeScript, Rust, Go, Ruby, SQL.
- Provide instructors a test-case editor with sample input/output, expected output, hidden test cases, and time/memory limits.
- Execute code asynchronously via the background job queue (17.3) and push results via WebSocket.

## 3. Non-Goals

- Full IDE environment (Replit-style persistent workspace) — quiz-embedded single-file execution only in v1.
- Multi-file project execution.
- Custom Docker sandbox management (use Judge0/Piston, not roll-our-own sandbox in v1).
- Anti-plagiarism for code (covered separately in 3.5).
- Continuous integration workflows for student projects.

## 4. Personas & User Stories

- **As a CS student**, I want to write a Python function, click "Run", and see whether my output matches the expected output, so I know before submitting.
- **As an instructor**, I want to define public test cases (shown to students) and hidden test cases (used only for grading) so students cannot hard-code expected outputs.
- **As a student**, I want to see which test cases passed and which failed, with the expected vs. actual output, so I can debug my solution.
- **As an instructor**, I want to set a per-question time limit (e.g., 2 s) and memory limit (e.g., 256 MB) to prevent infinite loops and resource exhaustion.
- **As a platform admin**, I want to configure which execution backend to use (Judge0 hosted vs. Piston self-hosted) so I can control cost and data residency.

## 5. Functional Requirements

- **FR-1.** The system MUST support code-question grading by sending the student's code + test cases to a configured execution backend (Judge0 CE API or Piston API) and scoring based on test-case pass/fail.
- **FR-2.** The system MUST execute code asynchronously via the background job queue (17.3): submission enqueues a job, result returned via WebSocket `quiz-attempt/{id}/code-result`.
- **FR-3.** The system MUST support at minimum the following language IDs in Judge0: Python (71), Java (62), C (50), C++ (54), JavaScript (63), TypeScript (74), Rust (73), Go (60), Ruby (72), SQL (PostgreSQL, 29).
- **FR-4.** The system MUST allow instructors to define test cases per question: `input` string, `expected_output` string, `is_hidden` boolean, `time_limit_ms`, `memory_limit_kb`.
- **FR-5.** The system MUST allow students to run code against public test cases before submission (a "Run" action distinct from "Submit") with a rate limit of 10 runs per minute per user.
- **FR-6.** The system MUST return per-test-case results for grading: pass/fail, actual output, stderr, execution time, memory used; hidden test cases show only pass/fail to students.
- **FR-7.** The system MUST compute a score as (passing test cases / total test cases) × question points, with optional instructor-configurable partial credit disable.
- **FR-8.** The system SHOULD allow instructors to configure a "starter code" template pre-populated in the student's editor.
- **FR-9.** The system MUST sanitise and validate submitted code size (max 64 KB per submission) and reject submissions exceeding the limit before sending to the execution backend.

## 6. Non-Functional Requirements

- **Performance** — Code execution response (typical Python script, 5 test cases) MUST complete within 10 s end-to-end. WebSocket result push MUST arrive within 500 ms of execution completion. Student "Run" response (non-grading) MUST not block the quiz state machine.
- **Security** — Code executed in sandboxed container (Judge0 / Piston guarantee); no network access from execution environment. Submitted code stored encrypted at rest. Code size limit (64 KB) and test-case count limit (max 50) enforced before execution. Rate limiting on Run action (10/min/user). Instructor-owned test cases with hidden status not returned to students in grading payload.
- **Privacy & Compliance** — Code submissions may contain student-identifying logic; FERPA-protected. Submissions to Judge0 cloud service require a DPA review (data leaves the cluster). Piston self-hosted option preferred for institutions with data-residency requirements.
- **Accessibility** — Monaco code editor MUST have screen-reader mode enabled (`editor.accessibilitySupport: 'on'`). Test-case result table MUST be accessible with `role="table"`. Keyboard shortcut to run code (Ctrl+Enter / Cmd+Enter).
- **Scalability** — Expect up to 2 000 concurrent code executions during exams. Judge0 CE cloud plan handles this; self-hosted Piston requires horizontal scaling of execution nodes.
- **Reliability** — Execution backend timeout MUST be caught and returned as a "Time Limit Exceeded" result rather than a hanging job. Job queue retry with backoff for transient execution failures (max 3 retries).
- **Observability** — Prometheus counter `code_execution.submitted`, `code_execution.passed`, `code_execution.timed_out`. Log execution times per language. Alert if execution error rate > 5 %.
- **Maintainability** — Execution backend abstracted behind `CodeExecutionService` trait in `server/src/services/code_execution.rs`; Judge0 and Piston implementations are swappable.
- **Internationalization** — Language names are locale-translated in the dropdown. Error messages (compile error, runtime error, TLE) use i18n keys.
- **Backward compatibility** — Code question type introduced in 2.2 with `pending` grading state; this plan transitions those to auto-graded. Submissions created before this feature remain in `pending` and must be manually graded or retroactively re-graded.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student submits correct Python code that passes all 3 test cases, *when* grading completes, *then* the score is 100 % and all 3 test cases show "Pass."
- **AC-2.** *Given* a student submits code with an infinite loop, *when* the time limit is exceeded, *then* the result shows "Time Limit Exceeded" and the test case is marked failed.
- **AC-3.** *Given* a student clicks "Run" on a public test case, *when* the execution completes, *then* the actual output is shown alongside the expected output within 10 s.
- **AC-4.** *Given* a question has 2 public and 3 hidden test cases, *when* a student views their results, *then* the 3 hidden test cases show only pass/fail without revealing expected output or input.
- **AC-5.** *Given* a student submits 11 Run requests within 1 minute, *when* the 11th request arrives, *then* the response is HTTP 429 with a rate-limit message.
- **AC-6.** *Given* the execution backend is unreachable, *when* a submission is received, *then* the job is retried 3 times with exponential backoff before being marked "Execution failed — please try again."

## 8. Data Model

```sql
-- Migration: server/migrations/0043_code_execution.sql

CREATE TABLE course.question_test_cases (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    question_id     UUID NOT NULL REFERENCES course.questions(id) ON DELETE CASCADE,
    position        SMALLINT NOT NULL,
    input           TEXT,
    expected_output TEXT NOT NULL,
    is_hidden       BOOLEAN NOT NULL DEFAULT FALSE,
    time_limit_ms   INTEGER NOT NULL DEFAULT 2000,
    memory_limit_kb INTEGER NOT NULL DEFAULT 262144,  -- 256 MB
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE course.code_execution_results (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    attempt_id      UUID NOT NULL,
    question_id     UUID NOT NULL,
    test_case_id    UUID REFERENCES course.question_test_cases(id),
    status          TEXT NOT NULL,   -- 'pass'|'fail'|'tle'|'mle'|'re'|'ce'|'pending'
    actual_output   TEXT,
    stderr          TEXT,
    execution_ms    INTEGER,
    memory_kb       INTEGER,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## 9. API Surface

```
POST /api/v1/quiz-attempts/{attempt_id}/questions/{qid}/run
  Auth: own user (rate-limited: 10/min)
  Body: { code: string, language_id: number, test_case_id?: string }
  Response 202: { job_id }

WebSocket: ws://host/ws/quiz-attempts/{attempt_id}
  Event: { type: "code_result", question_id, results: TestCaseResult[], score? }

GET  /api/v1/quiz-attempts/{attempt_id}/questions/{qid}/code-results
  Auth: own user | instructor | admin
  Response: { results: TestCaseResult[] }  -- hidden test cases redacted for students

-- Instructor: test case management
GET    /api/v1/questions/{qid}/test-cases       Auth: instructor|admin
POST   /api/v1/questions/{qid}/test-cases       Auth: instructor|admin
PUT    /api/v1/questions/{qid}/test-cases/{id}  Auth: instructor|admin
DELETE /api/v1/questions/{qid}/test-cases/{id}  Auth: instructor|admin

-- Admin: execution backend config
GET  /api/v1/admin/code-execution/config        Auth: admin
PUT  /api/v1/admin/code-execution/config        Auth: admin
  Body: { backend: "judge0"|"piston", api_url, api_key? }
```

## 10. UI / UX

**Code question student view** (`clients/web/src/components/quiz/types/CodeQuestion.tsx`):
- Monaco editor (read-only language selector showing instructor-chosen language, or dropdown if multiple languages permitted).
- Starter code pre-populated.
- "Run" button → test-case results panel slides in below: each public test case shows expected/actual/status.
- "Submit" button → triggers grading job; shows loading state then results.
- Keyboard: Ctrl+Enter / Cmd+Enter to run; Ctrl+Shift+Enter to submit.

**Test-case results panel**:
- Pass (green check) / Fail (red X) / TLE / MLE / RE badges.
- Expandable row: input, expected output, actual output, execution time.
- Hidden test cases: show count only ("3 hidden tests: 2 passed, 1 failed").

**Instructor test-case editor** (in question bank question editor):
- Table of test cases with input/expected output text areas, hidden toggle, time/memory limits.
- "Add test case" button.
- "Run all against solution" to validate test cases before publishing.

Mobile: Monaco editor in a full-screen modal; reduced font size; Run/Submit CTAs at bottom.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Judge0 cloud DPA not acceptable for HE institutions | M | H | Ship Piston self-hosted option; document both; default to self-hosted for new installations |
| Code execution backend unavailable during exam | M | H | Circuit breaker with graceful degradation to manual grading; instructor notified |
| Malicious code escaping sandbox | L | H | Rely on Judge0/Piston sandboxing guarantees; do not roll own sandbox; review security advisories |
| Monaco editor bundle size (2+ MB) slows page load | M | M | Lazy-load Monaco only when a code question is present on the page |
| Test cases with non-deterministic output (timestamps, randomness) | M | M | Instructor documentation on writing deterministic test cases; add "ignore whitespace" option |

## 15. Rollout Plan

- **Feature flag:** `code_execution_enabled` (default: `false`; admin configures backend before enabling).
- **Migration sequencing:** deploy job queue (17.3) → deploy execution service → configure backend → deploy UI → enable flag for pilot.
- **Pilot cohort:** one HE intro-to-Python course; 30 students; 5 questions with test cases.
- **GA criteria:** 100 code submissions graded; execution error rate < 2 %; TLE correctly caught.
- **Rollback:** disable flag; code questions revert to "pending grading" state (manually gradeable).

## 16. Test Plan

- **Unit** — Judge0 response mapping to internal status enum; test-case score computation (partial credit); code size validation.
- **Integration** — Submit Python "hello world" to Judge0/Piston mock → assert pass result; submit code with syntax error → assert compile-error result; TLE via infinite loop simulation.
- **End-to-end** — Playwright: student types Python solution, clicks Run, sees test-case results; clicks Submit, sees final score.
- **Security** — Rate-limit test (11th request in 1 min → 429); hidden test cases not in student-facing API response; code size > 64 KB rejected.
- **Accessibility** — axe scan on code question; Monaco `accessibilitySupport: 'on'`; keyboard shortcut Ctrl+Enter works.
- **Performance / load** — k6: 200 concurrent code submissions; end-to-end completion p95 ≤ 10 s.
- **Manual exploratory** — Verify TLE shown correctly; verify hidden test case count display; verify starter code pre-population; verify different language selection.

## 18. Open Questions

1. Should the execution backend be Judge0 CE (cloud, easy setup, data leaves cluster) or Piston (self-hosted, harder setup, data stays)? Recommend Piston self-hosted as default for privacy; Judge0 as an opt-in cloud tier.
2. Should students be allowed to choose their language (instructor-configurable per question) or is the language fixed per question?
3. Should test-case input/output be file-based (for large inputs) or text-only in v1?
4. How should SQL questions work — do we spin up a per-submission Postgres instance, or use a restricted session on the application database?
