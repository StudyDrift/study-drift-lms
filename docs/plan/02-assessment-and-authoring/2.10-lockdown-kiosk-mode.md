## 1. Problem Statement

High-stakes assessments (HE midterms, finals, standardised K-12 tests) require delivery controls that prevent students from navigating freely, going back to change answers, or leaving the assessment window. Currently, Lextures allows full back-navigation, tab-switching, and window changes with no enforcement. The absence of lockdown mode is a direct block to higher-education clients whose exam policies require controlled delivery environments, and to K-12 clients administering standardised assessments in school labs.

## 2. Goals

- Implement three lockdown-level delivery modes: `standard` (current behaviour), `one_at_a_time` (one question per page, no back navigation), and `kiosk` (full-screen enforcement + focus-loss detection + hints disabled).
- Enforce one-at-a-time mode server-side (once a question is answered and advanced, the answer is locked and the server enforces forward-only progression).
- Detect and log focus-loss events (tab switch, window blur) in kiosk mode for instructor review.
- Disable hints (2.9) and back-navigation in lockdown modes.
- Provide an instructor-facing focus-loss event log per attempt.

## 3. Non-Goals

- Full software-based proctoring (video monitoring, AI eye-tracking, screen capture) — covered in 14.9 with third-party integration (Honorlock, Respondus).
- Browser lock (prevents user from opening other applications) — requires native app or Chromebook kiosk app, out of scope.
- Facial recognition or biometric authentication.
- LTI-based external proctoring tool launch (covered in 2.12).

## 4. Personas & User Stories

- **As an HE instructor**, I want to deliver a final exam one question at a time so students cannot see all questions simultaneously and share answers.
- **As an instructor**, I want to know if a student switched browser tabs during a kiosk-mode exam so I can flag it for review.
- **As a student**, I want to know before I start an exam that I will be in "one at a time" mode and cannot go back, so I can prepare.
- **As an instructor**, I want to require full-screen mode for my online exam to reduce visual distractions and signal that this is a formal assessment.
- **As an IT admin**, I want kiosk-mode exams to be disabled on Chromebooks running the Lextures web app, unless configured (future: native kiosk integration).

## 5. Functional Requirements

- **FR-1.** The system MUST add a `lockdown_mode` enum field to quiz configuration: `standard` | `one_at_a_time` | `kiosk`.
- **FR-2.** In `one_at_a_time` mode, the system MUST serve one question per request; the server MUST track the current question index in the attempt and MUST NOT serve a prior question once the learner has advanced past it (after submitting that question's answer).
- **FR-3.** In `one_at_a_time` mode, the system MUST lock an answer once the "Next question" action is confirmed (the answer cannot be changed after advancing); this is enforced server-side.
- **FR-4.** In `kiosk` mode, the client MUST request full-screen on attempt start using the Fullscreen API; the server MUST log `focus_loss_events` when the client reports a `visibilitychange` or `blur` event.
- **FR-5.** In any lockdown mode, the hint endpoint (2.9) MUST return HTTP 403 (hints disabled).
- **FR-6.** The system MUST display a clear pre-start warning to students explaining the delivery constraints (e.g., "This exam is delivered one question at a time. You cannot go back.").
- **FR-7.** The system MUST expose `GET /api/v1/quiz-attempts/{id}/focus-loss-events` for instructor review (kiosk mode only).
- **FR-8.** The system SHOULD allow the instructor to set a maximum tolerated focus-loss count (e.g., 3) and automatically flag attempts exceeding it for review.

## 6. Non-Functional Requirements

- **Performance** — Single-question delivery (one_at_a_time) adds no overhead vs. full-quiz delivery; one fewer question in the response payload.
- **Security** — Server-side answer locking in one_at_a_time mode MUST reject any PUT/PATCH on a locked answer. Focus-loss events are client-reported and treated as indicative (not conclusive) evidence.
- **Privacy & Compliance** — Focus-loss event log contains (user_id, timestamp, event_type) — FERPA-protected. Kiosk mode must not capture screen content or keystrokes (which would require explicit COPPA/FERPA consent and data-use disclosures). Students must be informed of focus-loss logging in the pre-start warning.
- **Accessibility** — Full-screen mode MUST be dismissible via Escape (browser enforced). One-at-a-time mode MUST show clear progress indicator ("Question 5 of 20"). Kiosk-mode requirement for full-screen MUST have an accessible exception path for AT users who need AT overlays outside the browser window — document the exception policy.
- **Scalability** — Focus-loss event table partitioned by month; high cardinality during exam periods expected (50 k events/hour).
- **Reliability** — If full-screen API is unavailable (some browsers/OS configurations), kiosk mode MUST degrade to one_at_a_time with a warning rather than blocking the exam entirely.
- **Observability** — Counter `lockdown.focus_loss_events`, `lockdown.flagged_attempts`. Log lockdown mode enabled per quiz create/update. Alert if focus-loss event rate > 5 per student per attempt (potential academic integrity signal).
- **Maintainability** — Lockdown enforcement in `server/src/services/quiz_attempt.rs`; focus-loss handler in `server/src/routes/quiz_attempts.rs`. Client-side enforcement in `clients/web/src/pages/lms/CourseModuleQuizPage.tsx`.
- **Internationalization** — Pre-start warning and focus-loss notification messages use i18n keys.
- **Backward compatibility** — Default `lockdown_mode = 'standard'`; existing quizzes unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* `lockdown_mode = 'one_at_a_time'`, *when* a student advances past question 3, *then* `GET /api/v1/quiz-attempts/{id}/current-question` returns question 4 and any attempt to re-submit question 3 returns HTTP 403.
- **AC-2.** *Given* `lockdown_mode = 'kiosk'`, *when* a student switches browser tabs, *then* a `focus_loss_events` row is created and the student UI shows a warning banner on return.
- **AC-3.** *Given* `lockdown_mode = 'kiosk'`, *when* the hint endpoint is called, *then* the response is HTTP 403.
- **AC-4.** *Given* an attempt has 5 focus-loss events and the instructor threshold is 3, *when* the attempt is submitted, *then* the attempt is flagged with `academic_integrity_flag = true`.
- **AC-5.** *Given* `lockdown_mode = 'one_at_a_time'`, *when* a student views the quiz, *then* they see only one question at a time with "Question X of Y" and a "Next" button; no "Previous" button is shown.
- **AC-6.** *Given* the full-screen API fails (permissions denied), *when* a student starts a kiosk-mode exam, *then* the exam proceeds in one_at_a_time mode with a warning "Full-screen could not be enabled."

## 8. Data Model

```sql
-- Migration: server/migrations/0048_lockdown.sql

CREATE TYPE course.lockdown_mode AS ENUM ('standard', 'one_at_a_time', 'kiosk');

ALTER TABLE course.module_quizzes
    ADD COLUMN lockdown_mode        course.lockdown_mode NOT NULL DEFAULT 'standard',
    ADD COLUMN focus_loss_threshold INTEGER;  -- NULL = no auto-flag

ALTER TABLE course.quiz_attempts
    ADD COLUMN current_question_index INTEGER NOT NULL DEFAULT 0,
    ADD COLUMN academic_integrity_flag BOOLEAN NOT NULL DEFAULT FALSE;

CREATE TABLE course.attempt_focus_loss_events (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    attempt_id  UUID NOT NULL REFERENCES course.quiz_attempts(id) ON DELETE CASCADE,
    event_type  TEXT NOT NULL,    -- 'blur'|'visibility_hidden'|'fullscreen_exit'
    duration_ms INTEGER,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);

CREATE INDEX idx_afle_attempt ON course.attempt_focus_loss_events (attempt_id);

CREATE INDEX idx_locked_answers ON course.attempt_question_selections (attempt_id)
    WHERE locked = TRUE;   -- new boolean column

ALTER TABLE course.attempt_question_selections
    ADD COLUMN locked BOOLEAN NOT NULL DEFAULT FALSE;
```

## 9. API Surface

```
-- Existing attempt start gains lockdown mode in response:
POST /api/v1/quiz-attempts
  Response addition: { lockdown_mode, hints_disabled, back_navigation_allowed }

-- One-at-a-time: current question only
GET  /api/v1/quiz-attempts/{id}/current-question   Auth: own user
  Response: { question: DeliveredQuestion, question_index, total_questions }

POST /api/v1/quiz-attempts/{id}/advance
  Auth: own user
  Body: { answer_for_current_question }
  Response 200: { locked: true, next_question?: DeliveredQuestion, completed?: bool }
  Response 403: { error: "question_already_locked" }

-- Focus-loss event reporting (kiosk mode)
POST /api/v1/quiz-attempts/{id}/focus-loss
  Auth: own user
  Body: { event_type, duration_ms? }

GET  /api/v1/quiz-attempts/{id}/focus-loss-events
  Auth: instructor|admin
```

## 10. UI / UX

**Pre-start lockdown warning** (modal before attempt start):
- Icon + bold title: "Exam delivery mode: One question at a time."
- Bullet list of constraints: "You cannot go back to previous questions." / "Hints are disabled." / "Your focus is monitored — switching tabs will be logged."
- "I understand — begin exam" CTA; "Cancel" link.

**One-at-a-time quiz view**:
- Single question per page (no scroll to other questions).
- Progress bar: "Question 5 of 20."
- "Next" button only (no "Previous").
- After confirming next: "Your answer to this question has been locked."

**Kiosk focus-loss banner** (on tab return):
- Red dismissible banner: "A focus-loss event was recorded at [time]."

**Instructor focus-loss report** (side panel on attempt row in gradebook):
- Table: event type, timestamp, duration, cumulative count.
- Flag icon if threshold exceeded.

Accessibility: one-at-a-time navigation announced by ARIA live region; focus managed to question stem on advance; progress indicator has `aria-label="Question 5 of 20"`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Instructor enables kiosk mode but student doesn't have fullscreen | M | M | Graceful fallback to one_at_a_time + warning; do not block exam |
| False-positive focus-loss from AT tools (screen reader overlays) | M | H | Document AT exemption; allow per-enrollment focus-loss exemption flag |
| Student claims focus-loss was due to system issue, not cheating | M | L | Focus-loss log is evidence, not automatic penalty; instructor reviews |
| Server-side current_question_index race condition | L | H | Use optimistic locking (compare-and-set on current_question_index column) |

## 15. Rollout Plan

- **Feature flag:** `lockdown_mode_enabled` (default: `false`; lockdown mode option hidden in quiz settings).
- **Migration sequencing:** add schema → deploy API → deploy UI → enable flag for pilot.
- **Pilot cohort:** 1 HE final exam; 50 students; instructor and students briefed on focus-loss monitoring disclosure.
- **GA criteria:** answer locking verified server-side; focus-loss events logged; pre-start warning shown.
- **Rollback:** disable flag; lockdown mode field ignored; standard delivery for all quizzes.

## 16. Test Plan

- **Unit** — current_question_index increment; answer locking rejection; focus-loss threshold flag trigger.
- **Integration** — one_at_a_time: advance past question 3 → assert question 3 answer locked → attempt to re-submit question 3 → 403; kiosk: post focus-loss event → assert logged.
- **End-to-end** — Playwright: start one_at_a_time exam, answer Q1, click Next, verify Q2 shown, verify no Previous button, verify Q1 answer cannot be changed.
- **Security** — Student cannot unlock an answer via API; focus-loss log only accessible to instructor.
- **Accessibility** — Pre-start warning modal dismissible with keyboard; one-at-a-time progress announced by screen reader; focus managed on advance.
- **Performance** — Focus-loss event insert: ≤ 20 ms; current-question fetch: ≤ 30 ms.
- **Manual exploratory** — Fullscreen API permission denied path; focus-loss banner dismissal; academic integrity flag in instructor view.

## 18. Open Questions

1. Should kiosk mode require a PIN/password at the start (to prevent students from pre-filling a different student's login)?
2. Should focus-loss events be shown to the student in real time ("Warning: 2 focus-loss events recorded") or only visible to the instructor?
3. How should we handle the case where a student's computer crashes during a one_at_a_time exam — should they resume from the current question?
4. Is a focus-loss exemption per-enrollment (AT accommodation) or globally configured per quiz?
