## 1. Problem Statement

K-12 students — especially younger ones — struggle with password management, and self-learners frequently forget passwords for low-frequency-use tools. A magic link (one-time login URL sent to email) removes credential friction entirely for this demographic. Lextures currently has a password reset flow (`server/migrations/068_password_reset_tokens.sql`) but no magic-link login path. The existing reset token infrastructure provides a natural foundation. Without magic-link login, Lextures is significantly less accessible for K-12 deployments where students use shared Chromebooks or frequently forget passwords.

## 2. Goals

- Allow users to request a one-time login link sent to their email address.
- On link click, authenticate the user directly without entering a password.
- Issue a standard JWT pair (4.8) on successful magic-link consumption.
- Tokens must be single-use, time-limited (15 min), and cryptographically unpredictable.
- Support a "Sign in without a password" flow as an alternative to password entry, not a replacement.

## 3. Non-Goals

- Replacing password login entirely (magic link is additive).
- Email OTP as a second factor (deferred; 4.6 handles MFA).
- WhatsApp / SMS-based magic links.
- Persistent "remember this device" trust from magic-link logins (can be added as a follow-on with 4.9).

## 4. Personas & User Stories

- **As a K-12 student on a Chromebook**, I want to type my school email and receive a login link so that I never need to remember a password.
- **As a self-learner**, I want to log in to Lextures by clicking an email link when I have not visited in months so that I do not need to reset my password.
- **As a teacher setting up a class**, I want students to be able to access their Lextures accounts with just their school email so that the first-day setup is frictionless.

## 5. Functional Requirements

- **FR-1.** The system MUST provide an endpoint to request a magic link given an email address; the response is always HTTP 200 (prevent email enumeration).
- **FR-2.** If the email exists in the system, the system MUST send an email with a single-use, time-limited (15 min) login URL to that address.
- **FR-3.** The magic link token MUST be a cryptographically random 256-bit value (URL-safe base64).
- **FR-4.** On link click (GET or POST to the token endpoint), the system MUST validate the token, mark it consumed, and issue a JWT access + refresh token pair (4.8).
- **FR-5.** The system MUST reject consumed or expired tokens with HTTP 410 Gone and a clear user message.
- **FR-6.** The system MUST rate-limit magic-link requests per email address (max 3 per 5 min) to prevent email flooding.
- **FR-7.** The system SHOULD support a `redirect_to` parameter in the magic link URL so that users land on the page they were trying to access.
- **FR-8.** The system SHOULD allow institutional configuration to restrict magic-link login to enrolled users only (prevent new account creation via magic link).

## 6. Non-Functional Requirements

- **Performance** — Token request endpoint: ≤ 100 ms (excluding email delivery). Token consumption: ≤ 200 ms p95.
- **Security** — Tokens stored as SHA-256 hashes; plaintext token never persisted. 15-min TTL enforced server-side. Rate limit (3/5 min/email) enforced server-side. `redirect_to` URL validated against allowlist to prevent open redirect. Constant-time token comparison to prevent timing attacks.
- **Privacy & Compliance** — Email address used for magic link is PII. Email delivery requires an SMTP/transactional email service (6.2). Log token request with email hash (not plaintext) for audit. GDPR: token data deleted immediately on consumption or expiry.
- **Accessibility** — "Request magic link" button keyboard accessible; email input correctly labelled. Success message readable by screen readers. WCAG 2.1 AA.
- **Scalability** — Token table partitioned by expiry; expired rows cleaned by scheduled job. High volume (1 k requests/hour) handled by existing DB without separate cache.
- **Reliability** — If email delivery fails, user sees "If an account exists, an email has been sent" (no error detail). Token can be re-requested after rate limit window.
- **Observability** — Metrics: `magic_link.requested_total`, `magic_link.consumed_total`, `magic_link.expired_total`, `magic_link.rate_limited_total`.
- **Maintainability** — Magic link service in `server/src/services/auth/magic_link.rs`; reuses `password_reset_tokens` pattern from migration 068.
- **Internationalization** — Email template i18n-keyed (subject and body per locale).
- **Backward compatibility** — Password login completely unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a registered user with email `x@y.com`, *when* they request a magic link, *then* they receive an email within 60 s with a link containing a 256-bit token.
- **AC-2.** *Given* a valid magic link, *when* clicked within 15 min, *then* the user is logged in and a JWT pair is issued.
- **AC-3.** *Given* a magic link already used, *when* it is clicked again, *then* HTTP 410 is returned and the user sees "This link has already been used or has expired."
- **AC-4.** *Given* a magic link older than 15 min, *when* clicked, *then* the same 410 response is returned.
- **AC-5.** *Given* a user requests 4 magic links within 5 min, *when* the 4th request is made, *then* HTTP 429 is returned and no email is sent.
- **AC-6.** *Given* an unregistered email, *when* a magic link is requested, *then* HTTP 200 is returned and no email is sent (email enumeration prevention).
- **AC-7.** *Given* a `redirect_to=/courses/123` parameter, *when* the magic link is consumed, *then* the user is redirected to `/courses/123` after authentication.

## 8. Data Model

```sql
-- Migration: server/migrations/092_magic_link_tokens.sql
-- Similar to 068_password_reset_tokens.sql

CREATE TABLE magic_link_tokens (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash  TEXT NOT NULL UNIQUE,   -- SHA-256(plaintext_token), hex-encoded
    expires_at  TIMESTAMPTZ NOT NULL,
    consumed_at TIMESTAMPTZ,
    redirect_to TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_magic_link_token_hash ON magic_link_tokens(token_hash)
    WHERE consumed_at IS NULL;
```

Rate limit tracked via a lightweight in-memory counter or Redis; or counted by `WHERE created_at > NOW() - INTERVAL '5 minutes'` on the tokens table.

## 9. API Surface

| Verb | Path | Auth | Description |
|---|---|---|---|
| POST | `/auth/magic-link/request` | none | Request magic link for email |
| GET | `/auth/magic-link/consume` | none | Consume token (from email link) |

`/auth/magic-link/request` body: `{ email: string }`
`/auth/magic-link/consume` query params: `token=<base64>` and optional `redirect_to`
Response on consume: JWT access token + refresh token (same shape as `/auth/login`).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Magic link forwarded by student to another person | M | M | Session management (4.9) shows login source; magic-link logins labelled |
| Email delivery delay makes 15-min TTL feel short | M | M | Increase TTL to 30 min; send "link expires in 30 minutes" in email body |
| Open redirect via redirect_to parameter | H | H | Validate redirect_to against same-origin allowlist before use |

## 15. Rollout Plan

- Feature flag: `MAGIC_LINK_ENABLED` (default: `false`; enable per institution or globally).
- Requires 6.2 email service to be operational first.
- GA criteria: token consumption, expiry, and replay prevention integration tests pass; email delivery tested with real SMTP.

## 16. Test Plan

- **Unit** — Token generation entropy; SHA-256 hash storage; TTL enforcement; rate limit logic; open-redirect prevention.
- **Integration** — Request → consume → JWT issued; expired token rejected; used token rejected; rate limit enforced.
- **End-to-end** — Playwright with email inbox mock: request link → extract token → consume → verify authenticated.
- **Security** — Email enumeration: same response for registered and unregistered email. Replay. Open redirect.
- **Accessibility** — axe-core on login page magic-link option.
