## 1. Problem Statement

Lextures currently has no second factor on the password authentication path (`server/src/services/auth.rs`). Instructor and admin accounts holding sensitive student data, gradebooks, and assessment answers are protected only by passwords, which are susceptible to credential stuffing and phishing. HE institutions increasingly mandate MFA for all staff accessing student records (NIST 800-63B, SOC 2 CC6.1). Additionally, modern passkey support (FIDO2 / WebAuthn) provides a phishing-resistant, passwordless experience that aligns with Lextures' forward-looking positioning. Without MFA, Lextures fails IT security reviews at the majority of HE and large K-12 district customers.

## 2. Goals

- Implement TOTP-based MFA (RFC 6238) with authenticator app enrolment (Google Authenticator, Authy, 1Password).
- Implement WebAuthn / FIDO2 (passkeys) for platform authenticators (Touch ID, Face ID, Windows Hello) and hardware keys (YubiKey).
- Allow admins to enforce MFA for all users or specific roles.
- Integrate MFA verification into the login flow before issuing tokens (4.8).
- Provide backup codes for account recovery.
- Store credential metadata per device for session management (4.9).

## 3. Non-Goals

- SMS OTP (insecure per NIST 800-63B; explicitly not included).
- Email OTP as an MFA factor (covered separately in 4.7 magic-link).
- Push notification MFA (Duo-style) — deferred.
- Admin-delegated MFA reset without audit trail (out-of-scope without support workflows).

## 4. Personas & User Stories

- **As an instructor**, I want to enrol my authenticator app for TOTP MFA so that my gradebook is protected even if my password is compromised.
- **As a student on a Mac**, I want to register my Touch ID as a passkey so that I can log in with biometrics rather than a password.
- **As a university IT admin**, I want to enforce MFA for all instructor and admin roles so that we meet our SOC 2 CC6.1 control requirement.
- **As a user who loses their phone**, I want to use a printed backup code to regain access so that MFA does not permanently lock me out.

## 5. Functional Requirements

- **FR-1.** The system MUST implement TOTP MFA per RFC 6238 (HMAC-based one-time password, 30 s window, 6 digits), with enrolment via QR code (otpauth:// URI).
- **FR-2.** The system MUST implement WebAuthn / FIDO2 credential registration and assertion per the W3C WebAuthn Level 2 specification, supporting both platform authenticators and roaming (hardware key) authenticators.
- **FR-3.** The system MUST generate 10 single-use backup codes on TOTP/WebAuthn enrolment; codes stored as bcrypt hashes.
- **FR-4.** The login flow MUST check if MFA is enabled for the user; if so, issue a short-lived `mfa_pending` token (60 s TTL) and prompt for the second factor before issuing the full JWT pair (4.8).
- **FR-5.** Admins MUST be able to require MFA for: all users, instructors and admins only, or no users (default).
- **FR-6.** The system MUST allow users to enrol and manage multiple MFA methods (TOTP + passkey simultaneously).
- **FR-7.** The system MUST allow users to view and revoke individual MFA credentials (by device name/type and enrolment date).
- **FR-8.** The system SHOULD allow users to name passkey credentials (e.g., "MacBook Pro 2024").
- **FR-9.** The system MUST log every MFA challenge success, failure, and enrolment/removal in the audit trail.

## 6. Non-Functional Requirements

- **Performance** — TOTP verification (hash comparison) < 5 ms. WebAuthn assertion verification (cryptographic) p95 ≤ 200 ms. Login flow adds one additional screen but no perceptible server-side latency.
- **Security** — TOTP: validate within ±1 time-step to tolerate clock skew; mark used OTPs as consumed to prevent replay (store `last_used_counter` per credential). WebAuthn: validate challenge, rpId, origin, user presence/verification flags. `mfa_pending` token single-use; consumed on successful MFA. Backup codes: bcrypt(12); each code single-use. Secrets stored in secrets management (17.17). NIST 800-63B AAL2 compliance.
- **Privacy & Compliance** — MFA credential data (TOTP secret, WebAuthn public key) is account security data, not PII per se, but must be protected at rest. Backup codes generated and displayed once; not stored plaintext. GDPR: MFA credentials deletable on account closure.
- **Accessibility** — MFA TOTP input: numeric input with paste support; autofocus on challenge screen. WebAuthn prompt: browser-native (inherits platform accessibility). Backup code display: monospace, copyable. WCAG 2.1 AA.
- **Scalability** — TOTP verification and WebAuthn assertion are CPU-bound but fast; no DB queries beyond credential lookup. Indexed by `user_id`.
- **Reliability** — TOTP clock skew tolerance (±30 s). WebAuthn challenges stored in Redis or DB with 60 s TTL. Backup codes provide recovery path if all factors lost.
- **Observability** — Metrics: `mfa.challenge_success_total`, `mfa.challenge_failure_total`, `mfa.enrolment_total` per type. Alert on failure spike (brute-force detection).
- **Maintainability** — TOTP in `server/src/services/auth/totp.rs`; WebAuthn in `server/src/services/auth/webauthn.rs`. Use `webauthn-rs` crate.
- **Internationalization** — MFA prompts and error messages i18n-keyed. Backup codes locale-independent (alphanumeric).
- **Backward compatibility** — MFA is opt-in by default; existing logins unaffected until admin enables enforcement.

## 7. Acceptance Criteria

- **AC-1.** *Given* a user enrols TOTP, *when* they log in with correct password, *then* a TOTP challenge screen appears and the full JWT is issued only after a valid OTP is entered.
- **AC-2.** *Given* a user enters a valid TOTP code, *when* they attempt to use the same code again within the same 30 s window, *then* the second attempt is rejected as a replay.
- **AC-3.** *Given* a user registers a passkey (Touch ID), *when* they return to the MFA challenge screen, *then* they can authenticate using the platform authenticator without entering a code.
- **AC-4.** *Given* a user lost their authenticator app, *when* they enter a valid backup code, *then* they are granted access and the code is marked as used.
- **AC-5.** *Given* an admin has enabled "Require MFA for instructors," *when* an instructor without MFA set up attempts to log in, *then* they are redirected to the MFA enrolment flow before accessing the app.
- **AC-6.** *Given* a TOTP enrolment QR code is generated, *when* scanned by Google Authenticator, *then* it generates valid TOTP codes accepted by the server.

## 8. Data Model

```sql
-- Migration: server/migrations/091_mfa_credentials.sql

CREATE TABLE mfa_totp_credentials (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    secret          TEXT NOT NULL,          -- base32-encoded, encrypted at rest
    label           TEXT,
    verified        BOOLEAN NOT NULL DEFAULT FALSE,
    last_used_at    TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE mfa_webauthn_credentials (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    credential_id   BYTEA NOT NULL UNIQUE,
    public_key_cbor BYTEA NOT NULL,
    sign_count      INT NOT NULL DEFAULT 0,
    display_name    TEXT,
    aaguid          UUID,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE mfa_backup_codes (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    code_hash       TEXT NOT NULL,          -- bcrypt(code)
    used_at         TIMESTAMPTZ
);

ALTER TABLE users
    ADD COLUMN mfa_required BOOLEAN NOT NULL DEFAULT FALSE;
```

## 9. API Surface

| Verb | Path | Auth | Description |
|---|---|---|---|
| POST | `/auth/mfa/totp/enrol` | authenticated (no MFA yet) | Begin TOTP enrolment — returns QR URI |
| POST | `/auth/mfa/totp/verify-enrol` | mfa_pending or authenticated | Confirm enrolment with first OTP |
| POST | `/auth/mfa/totp/challenge` | mfa_pending | Verify OTP during login |
| POST | `/auth/mfa/webauthn/register/begin` | authenticated | Begin passkey registration |
| POST | `/auth/mfa/webauthn/register/complete` | authenticated | Complete passkey registration |
| POST | `/auth/mfa/webauthn/authenticate/begin` | mfa_pending | Begin passkey assertion |
| POST | `/auth/mfa/webauthn/authenticate/complete` | mfa_pending | Complete passkey assertion |
| GET | `/api/me/mfa` | authenticated | List enrolled MFA factors |
| DELETE | `/api/me/mfa/:id` | authenticated | Remove a factor |

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| TOTP clock skew locks users out | M | H | Allow ±1 time-step tolerance; document clock sync requirement |
| WebAuthn not supported on older browsers | M | M | Feature-detect; fall back to TOTP; document browser requirements |
| Admin enforces MFA, users get locked out before enrolment | M | H | Grace period (e.g., 7 days to enrol); notify users by email before enforcement |

## 15. Rollout Plan

- Feature flag: `MFA_ENABLED` (default: `false`).
- Phase 1: TOTP only.
- Phase 2: WebAuthn / passkeys.
- Phase 3: admin-enforced MFA with grace period.
- GA criteria: NIST 800-63B AAL2 review passes; replay prevention integration test passes.

## 16. Test Plan

- **Unit** — TOTP code generation and validation; clock-skew tolerance; replay prevention; backup code hashing.
- **Integration** — Login flow with MFA: password → mfa_pending token → TOTP challenge → full JWT. Replay rejection. Admin enforcement check.
- **End-to-end** — Playwright: enrol TOTP (mocked TOTP library) → log in → MFA challenge → success.
- **Security** — Brute-force: 10 failed TOTP attempts trigger lockout. Replay of used backup code rejected. WebAuthn challenge replay rejected.
- **Accessibility** — axe-core on MFA challenge and enrolment pages; TOTP input paste test.
- **Manual exploratory** — Test with real YubiKey; test with Google Authenticator QR.
