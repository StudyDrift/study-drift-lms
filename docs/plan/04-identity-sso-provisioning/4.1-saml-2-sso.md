## 1. Problem Statement

Every university IT department and most K-12 district procurement checklists require SAML 2.0 SSO as a non-negotiable line item. Today Lextures supports only email + password JWT authentication (`server/src/services/auth.rs`, `server/src/jwt.rs`). Institutions run Shibboleth, ADFS, Okta, or Azure AD as their Identity Provider (IdP) and will not deploy a tool that requires students and staff to maintain separate credentials. Without SAML 2.0, Lextures cannot complete a single enterprise RFP for HE or most K-12 district IT security reviews.

## 2. Goals

- Implement SAML 2.0 SP-initiated and IdP-initiated SSO flows per the SAML 2.0 specification.
- Allow per-institution IdP configuration (entity ID, SSO URL, certificate, attribute mapping).
- Map incoming SAML assertions to Lextures user accounts (JIT provisioning + merge by email).
- Issue Lextures JWTs (4.8) on successful assertion validation.
- Provide a self-service SAML configuration UI for institution admins.
- Publish SP metadata at a well-known URL for IdP registration.

## 3. Non-Goals

- SAML as IdP (Lextures acting as the identity provider to other SPs) — deferred.
- Multi-IdP federation per single institution (enterprise feature).
- Attribute-based access control beyond role mapping in assertion.

## 4. Personas & User Stories

- **As a university IT admin**, I want to register Lextures as a SAML SP in our Shibboleth IdP using a downloadable metadata XML so that single sign-on works without manual certificate exchange.
- **As a student**, I want to click "Log in with University SSO" and be redirected to my institution's login page so that I never create a separate Lextures password.
- **As an instructor**, I want my existing institutional identity mapped to my Lextures account so that my courses and grades are preserved when SSO is enabled.
- **As a security administrator**, I want SAML assertions validated against the IdP's certificate so that forged assertions are rejected.

## 5. Functional Requirements

- **FR-1.** The system MUST implement SAML 2.0 SP-initiated SSO: generate `AuthnRequest`, redirect to IdP SSO URL, validate `Response`, extract `NameID` and configured attributes.
- **FR-2.** The system MUST implement IdP-initiated SSO: accept an unsolicited `Response` POST to the ACS URL and process it identically.
- **FR-3.** The system MUST validate the SAML response: XML signature against IdP certificate, `Conditions` not-before/not-after, `Audience`, `InResponseTo` for SP-initiated.
- **FR-4.** The system MUST support configurable attribute mapping per IdP: map any SAML attribute to `email`, `name`, `roles`, `student_id`.
- **FR-5.** The system MUST perform Just-In-Time provisioning: if a user with the mapped email does not exist, create them; if they exist, update attributes per configuration.
- **FR-6.** The system MUST publish SP metadata at `/auth/saml/metadata` (XML) including EntityID, ACS URL, and SP certificate.
- **FR-7.** The system MUST provide per-institution IdP configuration stored securely: entity ID, SSO URL, SLO URL (optional), IdP certificate (PEM), attribute mappings.
- **FR-8.** The system SHOULD support Single Logout (SLO) both SP-initiated and IdP-initiated.
- **FR-9.** The system SHOULD allow an institution to require SAML for all users (block password login when SAML is configured).

## 6. Non-Functional Requirements

- **Performance** — SAML AuthnRequest generation ≤ 50 ms. ACS POST processing (signature verification, DB upsert, JWT issue) p95 ≤ 500 ms.
- **Security** — XML signature validation using `xmlsec` or `samael` Rust crate; reject responses without valid signatures. Assert `Conditions` time window ≤ 5 min clock skew tolerance. IdP certificates stored encrypted at rest (17.17). Replay prevention via `InResponseTo` cache (Redis or DB with 5-min TTL). SAML 2.0 specification compliance (OASIS sstc-saml-core-errata-2.0).
- **Privacy & Compliance** — SAML assertions may carry student PII (name, email, institutional ID) — treated as FERPA data. JIT-provisioned accounts subject to same data retention. GDPR: user data created via JIT included in DSAR scope.
- **Accessibility** — "Log in with SSO" button keyboard accessible; redirect handled transparently. WCAG 2.1 AA.
- **Scalability** — SP state (AuthnRequest ID → relay state) stored in Redis (or DB) with TTL; stateless app servers. Expect up to 5 k concurrent SSO logins during class start.
- **Reliability** — Fallback to password login if IdP is unavailable (unless institution forces SSO). ACS endpoint idempotent: duplicate `Response` replay detected and rejected.
- **Observability** — Metrics: `saml.authn_requests_total`, `saml.assertions_validated`, `saml.assertions_failed`. Log IdP entity ID, `NameID`, and failure reason on each assertion.
- **Maintainability** — SAML logic isolated in `server/src/services/auth/saml.rs`; IdP config CRUD in `server/src/routes/admin/saml.rs`. Use `samael` crate (Rust SAML library).
- **Internationalization** — Error messages on failed SSO i18n-keyed; SP metadata EntityID locale-independent.
- **Backward compatibility** — Email/password login remains available unless institution explicitly disables it.

## 7. Acceptance Criteria

- **AC-1.** *Given* an institution's IdP is configured, *when* a user clicks "Log in with SSO," *then* they are redirected to the IdP with a valid `AuthnRequest` XML containing `Issuer` = SP EntityID and a signed `ID`.
- **AC-2.** *Given* the IdP posts a valid signed SAML `Response` to the ACS URL, *when* the assertion is processed, *then* the user is logged in and a Lextures JWT is issued.
- **AC-3.** *Given* a SAML `Response` with an invalid signature, *when* it is submitted to the ACS URL, *then* the server returns HTTP 400 and the event is logged as `saml.assertions_failed`.
- **AC-4.** *Given* a new user JIT-provisioned via SAML, *when* they log in for the first time, *then* a Lextures user record is created with attributes from the assertion.
- **AC-5.** *Given* an existing Lextures user whose email matches the SAML assertion, *when* they log in via SSO, *then* they are mapped to the existing account without creating a duplicate.
- **AC-6.** *Given* `/auth/saml/metadata` is requested, *when* the response is returned, *then* it is valid SAML SP metadata XML parseable by standard IdP tooling.
- **AC-7.** *Given* a replayed `Response` with the same `InResponseTo` ID, *when* submitted to ACS, *then* the server returns HTTP 400 "Assertion replay detected."

## 8. Data Model

```sql
-- Migration: server/migrations/086_saml_idp_configurations.sql

CREATE TABLE saml_idp_configurations (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    institution_id  UUID,                  -- FK to org/institution (5.1 — nullable until then)
    entity_id       TEXT NOT NULL UNIQUE,
    sso_url         TEXT NOT NULL,
    slo_url         TEXT,
    idp_cert_pem    TEXT NOT NULL,         -- encrypted at rest via app-layer encryption
    attribute_mapping JSONB NOT NULL DEFAULT '{}',
    force_saml      BOOLEAN NOT NULL DEFAULT FALSE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE saml_authn_request_state (
    request_id      TEXT PRIMARY KEY,
    relay_state     TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
-- Entries expire after 15 min (cleaned by scheduled job or Redis TTL)
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| SAML XML parsing vulnerabilities (XXE) | H | H | Use `samael` which disables external entity processing; never use `libxml2` directly |
| Clock skew between SP and IdP breaks `Conditions` | M | H | Allow configurable tolerance (default 5 min); log skew in error |
| JIT provisioning creates duplicate users | M | H | Match on email (normalised); DB unique constraint on email |
| IdP cert rotation breaks SSO | M | H | Support uploading multiple certs; alert 30 days before cert expiry |

## 15. Rollout Plan

- Feature flag: `SAML_SSO_ENABLED` (default: `false`).
- Phase 1: SP metadata + ACS endpoint (backend only); test with Okta developer tenant.
- Phase 2: JIT provisioning + JWT issuance.
- Phase 3: admin config UI + SLO.
- Pilot: 1 university IT partner.
- GA criteria: SAML conformance test suite (SAMLTest.id) passes; end-to-end login tested with Shibboleth, ADFS, Okta, Azure AD.
