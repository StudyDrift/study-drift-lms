## 1. Problem Statement

Clever and ClassLink are the dominant SSO and rostering middleware platforms used by US K-12 school districts. Over 70% of US public school districts use Clever for single sign-on and application provisioning. Teachers and students in these districts click a Clever app icon (or ClassLink LaunchPad tile) and expect to be instantly logged into any educational app without credentials. Without Clever and ClassLink integration, Lextures cannot appear in district app libraries and is disqualified from the vast majority of US K-12 district procurement processes that use these platforms.

## 2. Goals

- Implement Clever Instant Login (OIDC-based) so students and teachers in Clever districts can SSO into Lextures with one click from the Clever student/teacher portal.
- Implement Clever Secure Sync (roster sync via Clever's API, which wraps OneRoster) to automatically provision and deprovision users and enrollments.
- Implement ClassLink SSO (SAML 2.0 or OIDC, per ClassLink's supported protocols) and ClassLink Roster Server (OneRoster 1.2 compatible).
- Submit Lextures to the Clever App Gallery and ClassLink App Library for district discovery.

## 3. Non-Goals

- Clever Managed Rules (district-defined access control) beyond standard app access.
- ClassLink MFA enforcement (delegated to ClassLink's own IdP).
- Building a full Clever/ClassLink admin portal within Lextures — use their partner dashboards.

## 4. Personas & User Stories

- **As a K-12 student**, I want to click the Lextures icon on my Clever student portal and be immediately logged in without any password so that I can start working in under 5 seconds.
- **As a district IT admin**, I want to enable Lextures in our Clever district app library so that all students and teachers are automatically provisioned with correct course assignments.
- **As a teacher using ClassLink LaunchPad**, I want single-click access to Lextures from ClassLink so that I never switch between platforms manually.
- **As a Lextures admin**, I want to see which students were provisioned via Clever vs. directly, so that I can manage roster hygiene.

## 5. Functional Requirements

- **FR-1.** The system MUST implement Clever Instant Login using OAuth 2.0 Authorization Code with PKCE (Clever's OIDC-compatible flow per Clever developer documentation).
- **FR-2.** The system MUST exchange the Clever authorization code for a Clever access token, then call the Clever `/me` endpoint to retrieve user identity and district affiliation.
- **FR-3.** The system MUST perform JIT provisioning from Clever identity data (id, name, email, role, district) on first login, mapping Clever roles (student/teacher/district_admin) to Lextures roles.
- **FR-4.** The system MUST implement Clever Secure Sync: call Clever's `/v3.0/students`, `/v3.0/teachers`, `/v3.0/sections` endpoints with a district access token to sync roster data on a scheduled basis.
- **FR-5.** The system MUST implement ClassLink SSO (OIDC preferred; SAML as fallback) reusing 4.1/4.2 infrastructure with ClassLink-specific parameter configuration.
- **FR-6.** The system MUST implement ClassLink Roster Server consumption (OneRoster 1.2 REST) reusing 4.3 infrastructure.
- **FR-7.** The system MUST store a `clever_id` and `classlink_id` per user for deduplication across future logins.
- **FR-8.** The system MUST support Clever district-level configuration: district_id, client_id, client_secret, sync schedule (hourly/daily).
- **FR-9.** The system SHOULD display "Connected via Clever" or "Connected via ClassLink" on the user's account profile for provenance visibility.

## 6. Non-Functional Requirements

- **Performance** — Clever Instant Login redirect ≤ 50 ms. Callback (token exchange + `/me` call + DB upsert + JWT issue) p95 ≤ 800 ms (network-dependent). Sync job for 5 k students ≤ 5 min as background job.
- **Security** — `client_secret` for Clever/ClassLink stored in secrets management (17.17). `state` and PKCE verifier validated same as 4.2. Clever webhooks validated by signature header. Data from Clever accessed only with district-consented tokens.
- **Privacy & Compliance** — FERPA: student data from Clever is educational record data. Clever's DPA (district data agreement) covers processing; Lextures must sign Clever's ISV agreement. COPPA: Clever flags under-13 students; respect `is_under_13` field. Data minimisation: store only fields needed (id, name, email, role, district_id).
- **Accessibility** — "Log in with Clever" button meets Clever brand guide; keyboard accessible. WCAG 2.1 AA.
- **Scalability** — Sync job chunked (1 k records/batch) via job queue (17.3). Mapping table indexed by `(district_id, clever_id)`.
- **Reliability** — Sync job retried on Clever API rate-limit (429 with Retry-After header). Instant Login fallback to email/password if Clever is unavailable.
- **Observability** — Metrics: `clever.instant_login_success`, `clever.sync_students_synced`, `clever.sync_errors`. Log district_id, clever_id hash on login.
- **Maintainability** — Clever service in `server/src/services/provisioning/clever.rs`; ClassLink in `server/src/services/provisioning/classlink.rs`. Both reuse OIDC and OneRoster infrastructure.
- **Internationalization** — Button labels i18n-keyed. Clever and ClassLink platforms are US-centric; i18n impact minimal.
- **Backward compatibility** — Lextures accounts not in Clever/ClassLink are unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a Clever district is configured, *when* a student clicks "Log in with Clever" and authorises on Clever's portal, *then* they arrive back at Lextures logged in within 3 s.
- **AC-2.** *Given* a new student authenticating via Clever for the first time, *when* JIT provisioning runs, *then* a Lextures user is created with `role = student`, `clever_id` set, and linked to the correct course sections.
- **AC-3.** *Given* the nightly Clever sync runs for a district with 200 new enrollments, *when* the job completes, *then* all 200 students are enrolled in the correct Lextures courses.
- **AC-4.** *Given* a student removed from a section in Clever, *when* the next sync runs, *then* the student's enrollment in the corresponding Lextures course is deactivated.
- **AC-5.** *Given* a ClassLink district configured with OIDC, *when* a teacher clicks the Lextures tile in ClassLink LaunchPad, *then* they are logged in using ClassLink OIDC.
- **AC-6.** *Given* Clever flags a student as `is_under_13 = true`, *when* JIT provisioning runs, *then* the user record is created with a COPPA minor flag and no marketing communications are sent.

## 8. Data Model

```sql
-- Migration: server/migrations/089_clever_classlink_mapping.sql

ALTER TABLE users
    ADD COLUMN clever_id      TEXT UNIQUE,
    ADD COLUMN classlink_id   TEXT UNIQUE,
    ADD COLUMN is_minor       BOOLEAN NOT NULL DEFAULT FALSE;

CREATE TABLE clever_district_configurations (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    district_id     TEXT NOT NULL UNIQUE,
    client_id       TEXT NOT NULL,
    client_secret   TEXT NOT NULL,          -- encrypted
    sync_schedule   TEXT NOT NULL DEFAULT 'daily',
    last_synced_at  TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Clever App Gallery review takes 4–8 weeks | H | M | Start review process early; use sandbox credentials for development |
| Clever API rate limits block large district sync | M | M | Respect Retry-After; schedule sync during off-peak hours |
| COPPA: minor data handled incorrectly | L | H | Flag `is_minor` on JIT provision; audit COPPA data flows |

## 15. Rollout Plan

- Feature flag: `CLEVER_SSO_ENABLED`, `CLASSLINK_SSO_ENABLED` (both default: `false`).
- Phase 1: Clever Instant Login (auth only).
- Phase 2: Clever Secure Sync (roster).
- Phase 3: ClassLink SSO + Roster.
- Phase 4: App Gallery submission.
- Pilot: 1 Clever-enabled K-12 district partner.
