## 1. Problem Statement

Lextures applies no password strength requirements and does not check whether a chosen password has appeared in a known breach database. The existing bcrypt password storage in `server/src/services/auth.rs` is sound, but any 1-character password is accepted. This fails basic security hygiene benchmarks required by SOC 2 CC6.1, ISO 27001 A.9.4.3, and most institutional IT security policies. Have I Been Pwned (HIBP) k-anonymity API provides a privacy-preserving breach-check at near-zero cost. Both features are small, high-impact, and should ship together.

## 2. Goals

- Enforce configurable password complexity requirements at registration and password change.
- Check the submitted password against the HIBP Pwned Passwords API (k-anonymity prefix model) at registration and password change.
- Block passwords found in breach data; surface a user-friendly message suggesting a different password.
- Allow admins to configure minimum length and complexity rules per institution.
- Log password change events with breach-check outcome in the audit trail.

## 3. Non-Goals

- Periodic re-checking of existing stored passwords against HIBP (not feasible without plaintext access; communicate via security advisory instead).
- Password expiry / forced rotation (controversial and not required by NIST 800-63B unless breach is detected).
- Blocking commonly-guessed passwords beyond HIBP (dictionary list deferred).
- KeePass-style password strength meter animation (simple strength indicator is sufficient).

## 4. Personas & User Stories

- **As a new user registering**, I want to be told immediately if my chosen password has been seen in a data breach so that I choose a stronger one.
- **As a system administrator**, I want to configure a minimum password length of 12 characters for instructor accounts so that our institution meets our security policy.
- **As an instructor changing their password**, I want clear feedback on what the password requirements are so that I do not cycle through invalid attempts.
- **As a security officer**, I want evidence in the audit trail that breach checks ran at registration so that I can demonstrate due diligence for SOC 2 / ISO 27001 audits.

## 5. Functional Requirements

- **FR-1.** The system MUST enforce a configurable minimum password length (default: 8 characters; recommended: 12).
- **FR-2.** The system MUST allow configuring optional complexity rules per institution: require uppercase, lowercase, digit, special character (default: none required, per NIST 800-63B guidance).
- **FR-3.** The system MUST check every new or changed password against the HIBP Pwned Passwords API using the k-anonymity SHA-1 prefix method (send first 5 hex chars only; compare locally).
- **FR-4.** The system MUST reject passwords found in the HIBP database (count ≥ 1) with a user-friendly message: "This password has appeared in a data breach. Please choose a different password."
- **FR-5.** The system MUST NOT send the full password or full SHA-1 hash to any external service; only the 5-char prefix is sent per the HIBP k-anonymity specification.
- **FR-6.** If the HIBP API is unavailable, the system MUST log the failure and allow the password submission to proceed (fail open), not block users.
- **FR-7.** The system MUST log the breach-check result (breach_found: bool, hibp_available: bool) with the password-change audit event.
- **FR-8.** The system SHOULD display a real-time password strength indicator (e.g., weak/fair/strong) on the registration and change-password forms.

## 6. Non-Functional Requirements

- **Performance** — HIBP API call adds ≤ 300 ms to registration/password-change (HIBP p99 ~200 ms). If HIBP call exceeds 500 ms timeout, fail open without blocking the user.
- **Security** — K-anonymity strictly enforced: only 5-char hex prefix sent to HIBP API; full SHA-1 hash never leaves server. HIBP response cached per prefix for 24 h to reduce API calls. HTTPS-only to `api.pwnedpasswords.com`. Password never logged. Minimum length enforced server-side, not client-only.
- **Privacy & Compliance** — K-anonymity ensures HIBP cannot reconstruct the user's password. No PII sent to HIBP. GDPR: no personal data in HIBP request. ISO 27001 A.9.4.3 — evidence of password policy enforcement. SOC 2 CC6.1 — access control policy implementation.
- **Accessibility** — Password strength indicator uses both colour and text label (not colour alone — WCAG 2.1 AA 1.4.1). Error messages on validation failure use `aria-invalid` and `aria-describedby`. WCAG 2.1 AA.
- **Scalability** — HIBP prefix cache (24 h TTL) in Redis or in-memory; reduces external API calls by ~99% for common prefixes. No scaling concern at expected registration volume.
- **Reliability** — Fail-open on HIBP unavailability; log incident; do not block users. Password length/complexity check is local and always available.
- **Observability** — Metrics: `password_policy.violations_total` (min_length, complexity, hibp), `hibp.api_latency_seconds`, `hibp.api_errors_total`. Log breach found (bool), not password or hash.
- **Maintainability** — Password policy in `server/src/services/auth/password_policy.rs`; HIBP check in `server/src/services/auth/hibp.rs`. Called from registration and password-change endpoints.
- **Internationalization** — Validation error messages i18n-keyed. Password strength labels i18n-keyed.
- **Backward compatibility** — Existing passwords not re-validated. Policy applies only on next change. Existing users with short passwords prompted to update via optional banner.

## 7. Acceptance Criteria

- **AC-1.** *Given* a password shorter than the configured minimum, *when* submitted at registration, *then* the server returns HTTP 422 with a message indicating the minimum length requirement.
- **AC-2.** *Given* a user submits the password "password" (HIBP count: ~9 million), *when* the HIBP check runs, *then* registration is rejected with "This password has appeared in a data breach."
- **AC-3.** *Given* the HIBP API is unavailable (timeout), *when* a user submits a password, *then* the registration proceeds (fail open) and a warning is logged.
- **AC-4.** *Given* the HIBP check runs, *when* the request is sent to HIBP, *then* only the first 5 characters of the SHA-1 hash are transmitted — verified by inspecting the outbound request in tests.
- **AC-5.** *Given* a password change, *when* the audit trail is queried, *then* the event includes `breach_check_result: { breach_found: false, hibp_available: true }`.
- **AC-6.** *Given* an institution configured with `min_length = 12`, *when* a user sets an 11-character password, *then* the server rejects it with the institution-specific requirement message.

## 8. Data Model

```sql
-- Migration: server/migrations/095_password_policy.sql

CREATE TABLE password_policies (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    institution_id  UUID,               -- NULL = global default
    min_length      INT NOT NULL DEFAULT 8,
    require_upper   BOOLEAN NOT NULL DEFAULT FALSE,
    require_lower   BOOLEAN NOT NULL DEFAULT FALSE,
    require_digit   BOOLEAN NOT NULL DEFAULT FALSE,
    require_special BOOLEAN NOT NULL DEFAULT FALSE,
    check_hibp      BOOLEAN NOT NULL DEFAULT TRUE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- HIBP prefix cache (optional — can use Redis instead)
CREATE TABLE hibp_prefix_cache (
    prefix      CHAR(5) PRIMARY KEY,
    suffixes    TEXT NOT NULL,          -- raw HIBP response body
    cached_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| HIBP API rate limit on high-volume registration | L | M | Cache prefix responses 24 h; HIBP free tier is generous (millions of req/month) |
| Policy change breaks existing valid passwords | L | M | Policy only applied on next password change; banner prompts re-set |
| Over-aggressive complexity rules hurt UX (NIST 800-63B warns against this) | M | M | Default: length-only; complexity rules opt-in per institution |

## 15. Rollout Plan

- No feature flag — additive validation; default policy is lenient (length ≥ 8 only).
- Schema migration → policy defaults.
- Enable HIBP check globally.
- GA criteria: AC-1 through AC-6 pass; HIBP k-anonymity verified in integration test by inspecting outbound request.

## 16. Test Plan

- **Unit** — Password length/complexity validation; HIBP prefix construction; cache lookup; fail-open on timeout.
- **Integration** — Register with breached password → rejected. Register with strong password → accepted. HIBP API mocked.
- **End-to-end** — Playwright: registration form → type weak password → strength indicator updates → submit → HIBP error.
- **Security** — Verify only 5-char prefix in outbound HIBP request (mock HIBP server, inspect request). Server-side enforcement not bypassable from client.
- **Accessibility** — axe-core on signup form; strength indicator announced.

## 17. Documentation & Training

- User help: "Password requirements" (what is required, why HIBP check matters).
- Admin guide: "Configuring password policy for your institution."
- Security runbook: evidence of HIBP check for SOC 2 audit.

## 18. Open Questions

1. Should we show users the approximate breach count ("your password has been seen in 9 million breaches") or just a generic message?
2. Should `check_hibp` be on by default for all institutions, or opt-in?
3. Password expiry: NIST 800-63B recommends against forced rotation; should we offer it as an institutional option anyway for legacy-policy institutions?
