## 1. Problem Statement

OneRoster 1.2 is the IMS Global standard adopted by the majority of US K-12 districts for roster synchronisation between a Student Information System (SIS) and third-party applications. Districts provision teachers, students, courses, and enrollments through OneRoster and expect the receiving application to consume it. Without OneRoster support, every K-12 district deployment requires manual user and enrollment setup — making Lextures operationally unacceptable for district-scale adoption (hundreds of teachers, thousands of students). This is a hard sales blocker for the K-12 segment.

## 2. Goals

- Implement OneRoster 1.2 CSV import (file-based batch sync) for orgs, classes, users, and enrollments.
- Implement OneRoster 1.2 REST API consumer (SIS pushes diffs via the OneRoster REST protocol).
- Map OneRoster roles (teacher, student, aide) to Lextures RBAC roles.
- Support incremental sync: create, update, and soft-delete (status=tobedeleted) users and enrollments.
- Provide an admin UI to trigger and monitor sync jobs.

## 3. Non-Goals

- Lextures as a OneRoster provider (acting as the data source — deferred).
- OneRoster Grade Passback (grade sync to SIS — deferred to 14.5).
- Support for OneRoster 1.1 (1.2 is the current standard; 1.1 compatibility is backward-compatible for most fields).

## 4. Personas & User Stories

- **As a district IT administrator**, I want to upload a OneRoster 1.2 CSV bundle so that all teachers, students, courses, and enrollments are provisioned automatically.
- **As a district SIS (PowerSchool, Infinite Campus)**, I want to push roster changes via the OneRoster 1.2 REST API so that changes in the SIS propagate to Lextures within minutes.
- **As a Lextures admin**, I want to see a sync log showing how many users and enrollments were created, updated, or deleted in each sync run.
- **As a teacher**, I want my class roster in Lextures to automatically reflect when a student is added or removed in the SIS, without any manual re-enrollment.

## 5. Functional Requirements

- **FR-1.** The system MUST parse and import the OneRoster 1.2 CSV bundle (orgs.csv, users.csv, classes.csv, enrollments.csv, academicSessions.csv) per the IMS Global OneRoster 1.2 CSV specification.
- **FR-2.** The system MUST implement the OneRoster 1.2 REST Service API endpoints (at minimum: GET /orgs, /users, /classes, /enrollments, /gradingPeriods) to act as a consumer when an SIS pushes data.
- **FR-3.** The system MUST map OneRoster roles: `teacher` → instructor, `student` → student, `aide` → TA, `administrator` → admin.
- **FR-4.** The system MUST handle OneRoster `status = tobedeleted` by deactivating (not hard-deleting) the corresponding Lextures user or enrollment.
- **FR-5.** The system MUST be idempotent: running the same CSV import twice produces the same state.
- **FR-6.** The system MUST link OneRoster `sourcedId` to Lextures entity IDs in a mapping table to support future updates by `sourcedId`.
- **FR-7.** The system MUST provide a sync audit log per import run: entity type, operation (create/update/deactivate), `sourcedId`, Lextures ID, error if any.
- **FR-8.** The system SHOULD support delta sync: on REST-based sync, only changed records (based on `dateLastModified`) are processed.
- **FR-9.** The system MAY support OAuth 2.0 bearer token authentication on the REST sync endpoint as specified by OneRoster 1.2.

## 6. Non-Functional Requirements

- **Performance** — CSV import of 10 k users completes in ≤ 60 s as a background job. REST sync delta for 500 records ≤ 5 s.
- **Security** — REST endpoint secured by OAuth 2.0 bearer token issued to the SIS. CSV upload requires admin authentication and file size limit (50 MB). `sourcedId` values are institution-scoped to prevent cross-tenant data leakage.
- **Privacy & Compliance** — Student roster data (name, email, student ID, date-of-birth if included) is FERPA-protected. COPPA: students under 13 flagged in import. OneRoster data processed per IMS Global DPA template. Data retained only as long as enrollment is active.
- **Accessibility** — Admin sync UI: import status, error counts, log table all keyboard accessible. WCAG 2.1 AA.
- **Scalability** — Import job runs via 17.3 background queue; chunked processing in batches of 500 rows. Mapping table indexed by `(institution_id, sourcedId)`.
- **Reliability** — Import job is transactional per batch; partial batch failure rolls back that batch and logs the error. Job retried up to 3× on failure.
- **Observability** — Metrics: `oneroster.sync_run_duration`, `oneroster.entities_created`, `oneroster.entities_failed`. Log sync run ID, institution ID, entity counts.
- **Maintainability** — OneRoster parser in `server/src/services/provisioning/oneroster.rs`; REST endpoints in `server/src/routes/provisioning/oneroster.rs`.
- **Internationalization** — Sync log UI labels i18n-keyed; OneRoster data fields treated as UTF-8.
- **Backward compatibility** — Manually created users and enrollments not affected by sync unless their email matches a OneRoster record (merge on email).

## 7. Acceptance Criteria

- **AC-1.** *Given* a valid OneRoster 1.2 CSV bundle with 100 users and 50 enrollments, *when* an admin uploads it, *then* all 100 users and 50 enrollments are created and a sync log entry records the counts.
- **AC-2.** *Given* a second import of the same CSV with no changes, *when* the import runs, *then* 0 creates and 0 updates are recorded (idempotent).
- **AC-3.** *Given* a user CSV row with `status = tobedeleted`, *when* processed, *then* the corresponding Lextures user's enrollment is deactivated (not deleted) and a log entry records the deactivation.
- **AC-4.** *Given* a REST sync push from an SIS with OAuth 2.0 bearer token, *when* 20 new enrollments arrive, *then* they are created in Lextures within 5 s.
- **AC-5.** *Given* an invalid CSV (missing required `sourcedId` column), *when* uploaded, *then* the import is rejected before processing and an error message names the missing column.
- **AC-6.** *Given* an existing user provisioned via OneRoster, *when* the user's `givenName` changes in a subsequent sync, *then* the Lextures display name is updated and the change is logged.

## 8. Data Model

```sql
-- Migration: server/migrations/088_oneroster_mapping.sql

CREATE TABLE oneroster_entity_mappings (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    institution_id  UUID NOT NULL,
    entity_type     TEXT NOT NULL CHECK (entity_type IN ('user','class','org','enrollment')),
    sourced_id      TEXT NOT NULL,
    lextures_id     UUID NOT NULL,
    last_synced_at  TIMESTAMPTZ,
    UNIQUE (institution_id, entity_type, sourced_id)
);

CREATE TABLE oneroster_sync_runs (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    institution_id  UUID NOT NULL,
    trigger         TEXT NOT NULL CHECK (trigger IN ('csv_upload','rest_push','scheduled')),
    status          TEXT NOT NULL DEFAULT 'running',
    created_count   INT DEFAULT 0,
    updated_count   INT DEFAULT 0,
    deactivated_count INT DEFAULT 0,
    error_count     INT DEFAULT 0,
    started_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at    TIMESTAMPTZ
);
```

## 9. API Surface

Internal admin:
| Verb | Path | Auth | Description |
|---|---|---|---|
| POST | `/api/admin/provisioning/oneroster/upload` | admin | Upload CSV bundle |
| GET | `/api/admin/provisioning/oneroster/sync-runs` | admin | List sync run history |
| GET | `/api/admin/provisioning/oneroster/sync-runs/:id` | admin | Sync run detail log |

OneRoster REST endpoint (SIS pushes here):
| Verb | Path | Auth | Description |
|---|---|---|---|
| GET | `/oneroster/v1p2/users` | OAuth 2.0 bearer | OneRoster REST user list |
| GET | `/oneroster/v1p2/classes` | OAuth 2.0 bearer | Class list |
| GET | `/oneroster/v1p2/enrollments` | OAuth 2.0 bearer | Enrollment list |

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| SIS sends duplicate sourcedIds across districts | M | H | Scope mapping table by institution_id |
| CSV encoding issues (BOM, line endings) | H | M | Normalise encoding on ingest; accept UTF-8 and UTF-8-BOM |
| Large district (50 k users) overwhelms single import job | M | M | Stream CSV rows; process in parallel batches; report partial progress |

## 15. Rollout Plan

- Feature flag: `ONEROSTER_ENABLED` (default: `false`).
- Phase 1: CSV import (simpler to test without SIS cooperation).
- Phase 2: REST endpoints + OAuth issuance.
- Pilot: 1 district with PowerSchool or Infinite Campus.
- GA criteria: IMS Global OneRoster 1.2 conformance tests pass; CSV import tested with real district extract.

## 16. Test Plan

- **Unit** — CSV parser; `tobedeleted` handling; idempotency logic; sourcedId deduplication.
- **Integration** — CSV upload endpoint; entity creation; sync run log; REST endpoint authentication.
- **End-to-end** — Upload test CSV bundle → verify users and enrollments created → re-upload → verify idempotent.
- **Security** — Unauthenticated REST endpoint returns 401; wrong bearer token returns 401.
- **Performance** — Import 10 k users via CSV ≤ 60 s.
- **Manual exploratory** — Test with real PowerSchool CSV extract; test deactivation flow.
