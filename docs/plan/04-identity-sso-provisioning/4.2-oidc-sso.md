## 1. Problem Statement

Google Workspace for Education and Microsoft Entra ID (Azure AD) are the dominant identity systems in K-12 and higher-education respectively. Students and teachers expect a "Sign in with Google" or "Sign in with Microsoft" button on every educational app. Apple Sign-In is required by Apple's App Store policy for any app that offers social login. Without OIDC SSO, Lextures faces friction-driven abandonment in K-12 (Chromebook-first classrooms) and fails institutional IT requirements for federated identity in HE. OIDC Core (OAuth 2.1) is significantly simpler to implement than SAML and should ship first.

## 2. Goals

- Implement OIDC Authorization Code flow (with PKCE) for Google, Microsoft (Azure AD / Entra), and Apple as providers.
- Allow per-institution restriction to a specific Google Workspace domain (HD parameter).
- Perform JIT provisioning and email-based account linking same as 4.1 SAML.
- Issue Lextures JWTs (4.8) on successful ID token validation.
- Provide a self-service OIDC configuration UI for institution admins (custom OIDC provider support beyond the three named providers).

## 3. Non-Goals

- OAuth 2.0 authorization for API access (separate from identity — deferred to 16.2).
- OIDC as an IdP (Lextures issuing ID tokens to third parties) — deferred.
- OIDC for non-SSO social login in a consumer/SL context (consumer Google Sign-In) — same implementation, included.

## 4. Personas & User Stories

- **As a K-12 student**, I want to click "Sign in with Google" and be immediately authenticated with my school Google account so that I never need to remember a Lextures password.
- **As a university instructor**, I want to sign in with my Microsoft 365 institutional account so that multi-factor authentication is enforced by the institution, not by Lextures separately.
- **As a self-learner**, I want to sign up with my Apple ID so that I can use Sign in with Apple on my iPhone without sharing my email address.
- **As a K-12 district IT admin**, I want to restrict Google sign-in to accounts in `@district.edu` so that external accounts cannot access our courses.

## 5. Functional Requirements

- **FR-1.** The system MUST implement OIDC Authorization Code flow with PKCE (RFC 7636) for Google, Microsoft Entra ID, and Apple.
- **FR-2.** The system MUST validate the received ID token: signature (JWKS endpoint), `iss`, `aud`, `exp`, `iat`, and `nonce`.
- **FR-3.** The system MUST perform JIT provisioning: create user if email not found; link to existing account if email matches.
- **FR-4.** The system MUST support `hd` (hosted domain) restriction for Google: reject ID tokens from outside the configured domain.
- **FR-5.** The system MUST support custom OIDC provider configuration: `client_id`, `client_secret`, discovery URL (`/.well-known/openid-configuration`), attribute mapping.
- **FR-6.** After successful OIDC, the system MUST issue a Lextures access token and refresh token per 4.8.
- **FR-7.** The system MUST store the OIDC `sub` claim per provider in a `user_oidc_identities` table to handle future logins from the same identity without relying on email alone.
- **FR-8.** The system SHOULD support "Link account" flow: a logged-in user can connect an OIDC provider to their existing Lextures account.
- **FR-9.** Apple Sign-In MUST support the `private_relay` email address (`@privaterelay.appleid.com`) without requiring a real email from the user.

## 6. Non-Functional Requirements

- **Performance** — Redirect to provider ≤ 50 ms (pure redirect). Callback processing (token exchange, JWKS fetch cached, DB upsert, JWT issue) p95 ≤ 600 ms.
- **Security** — PKCE required for all providers (OAuth 2.1 best practice). `state` parameter validated to prevent CSRF. JWKS cached with TTL; rotated automatically. `client_secret` stored in secrets management (17.17). `nonce` validated against session store. Redirect URIs strictly whitelisted per provider config.
- **Privacy & Compliance** — OIDC claims (email, name, profile picture) are PII. Users consent to data sharing via provider's own consent screen. GDPR: users can disconnect OIDC provider from account. FERPA: JIT-provisioned records are education records.
- **Accessibility** — Provider buttons visually distinct with provider logos; keyboard accessible. WCAG 2.1 AA.
- **Scalability** — PKCE verifier and `state` stored in Redis or DB with 10-min TTL; stateless app servers.
- **Reliability** — JWKS fetch failure falls back to cached keys; if cache empty, return 503 with clear error.
- **Observability** — Metrics: `oidc.login_attempts`, `oidc.login_success`, `oidc.login_failed` per provider. Log provider, `sub` hash (not full sub), failure reason.
- **Maintainability** — OIDC logic in `server/src/services/auth/oidc.rs`; provider-specific configs in `server/src/services/auth/oidc/google.rs`, `microsoft.rs`, `apple.rs`.
- **Internationalization** — Provider button labels i18n-keyed; error messages localised.
- **Backward compatibility** — Email/password login unaffected; OIDC is additive.

## 7. Acceptance Criteria

- **AC-1.** *Given* Google OIDC is configured, *when* a user clicks "Sign in with Google," *then* they are redirected to Google with a valid `state`, `nonce`, `code_challenge`, and `redirect_uri`.
- **AC-2.** *Given* Google returns a valid ID token for `user@school.edu`, *when* the callback is processed, *then* the user is logged in and a Lextures JWT pair is issued.
- **AC-3.** *Given* a Google OIDC config with `hd = school.edu`, *when* a user authenticates with `user@gmail.com`, *then* the callback returns HTTP 400 "Your account is not in the allowed domain."
- **AC-4.** *Given* a new user logging in via Microsoft OIDC, *when* JIT provisioning runs, *then* a `users` row and a `user_oidc_identities` row are created.
- **AC-5.** *Given* an existing user with email `x@y.com`, *when* they log in via Apple Sign-In for the first time with the same email, *then* they are linked to the existing account (not a new account created).
- **AC-6.** *Given* a tampered `state` parameter in the callback, *when* the server validates it, *then* HTTP 400 is returned and the attempt is logged as CSRF.
- **AC-7.** *Given* an Apple user who uses private relay email, *when* they complete sign-in, *then* a user account is created using the relay email; no error occurs.

## 8. Data Model

```sql
-- Migration: server/migrations/087_user_oidc_identities.sql

CREATE TABLE user_oidc_identities (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    provider    TEXT NOT NULL CHECK (provider IN ('google','microsoft','apple','custom')),
    sub         TEXT NOT NULL,          -- provider's unique subject identifier
    email       TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (provider, sub)
);

CREATE TABLE oidc_provider_configurations (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    institution_id  UUID,
    provider        TEXT NOT NULL,
    client_id       TEXT NOT NULL,
    client_secret   TEXT NOT NULL,     -- encrypted at rest
    discovery_url   TEXT NOT NULL,
    hd_restriction  TEXT,
    attribute_mapping JSONB DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE oidc_flow_state (
    state       TEXT PRIMARY KEY,
    nonce       TEXT NOT NULL,
    code_verifier TEXT NOT NULL,
    provider    TEXT NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
-- Entries expire after 10 min
```

## 9. API Surface

| Verb | Path | Auth | Description |
|---|---|---|---|
| GET | `/auth/oidc/:provider/login` | none | Initiate OIDC flow — redirect to provider |
| GET | `/auth/oidc/:provider/callback` | none | Handle callback, exchange code, issue JWT |
| POST | `/auth/oidc/:provider/link` | authenticated | Link provider to existing account |
| DELETE | `/auth/oidc/:provider/link` | authenticated | Unlink provider from account |
| GET/PUT | `/api/admin/oidc/:provider/config` | global admin | Provider configuration CRUD |

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| PKCE verifier or state lost between request and callback | M | M | Store in DB with TTL; stateless app servers |
| Apple private relay email changes between logins | M | M | Match on `sub` first, not email; `user_oidc_identities` uses `sub` |
| Google JWKS key rotation breaks validation | L | H | Cache JWKS with short TTL (1 h); on validation failure re-fetch |
| Provider outage blocks all login for SSO-forced institutions | M | H | Keep password login as fallback unless explicitly force-disabled |

## 15. Rollout Plan

- Feature flag: `OIDC_SSO_ENABLED` (default: `false`).
- Phase 1: Google (K-12 priority).
- Phase 2: Microsoft (HE priority).
- Phase 3: Apple (App Store requirement), custom provider.
- GA criteria: all three providers tested end-to-end; PKCE and state CSRF tested; JIT provisioning confirmed.

## 16. Test Plan

- **Unit** — JWKS validation; nonce/state checks; JIT provisioning; `hd` restriction.
- **Integration** — Mock OAuth server (e.g., `mockito`): full callback flow; duplicate account merge; CSRF rejection.
- **End-to-end** — Playwright + mock OIDC server: login → JWT → authenticated page.
- **Security** — CSRF with wrong state; replay with expired `exp`; missing `nonce`; wrong `aud`.
- **Accessibility** — axe-core on Login page with provider buttons.
- **Performance** — k6: 500 concurrent OIDC callbacks (JWKS cached).
