## 1. Problem Statement

Enterprise and higher-education institutions using Okta, Azure AD, or Ping Identity expect SCIM 2.0 (RFC 7644) as the standard protocol for automated user lifecycle management — creating accounts when someone joins, updating attributes when they change roles, and deprovisioning accounts when they leave. Without SCIM, IT administrators must manually manage Lextures accounts independently from their central identity system. This creates stale accounts (security risk), missed deprovisioning (FERPA risk), and significant administrative burden at scale. SCIM 2.0 is the complement to SAML/OIDC SSO and is typically required together in enterprise IT checklists.

## 2. Goals

- Implement a SCIM 2.0 Service Provider (SP) endpoint per RFC 7644 accepting push from IdPs (Okta, Azure AD, Ping, OneLogin).
- Support SCIM User resource: create, read, update (replace + patch), deactivate.
- Support SCIM Group resource (optional but recommended) for mapping IdP groups to Lextures course enrollment or org roles.
- Authenticate SCIM requests via OAuth 2.0 bearer token.
- Trigger session revocation (4.8) when a user is deprovisioned via SCIM.

## 3. Non-Goals

- SCIM client (Lextures initiating SCIM pushes) — deferred.
- SCIM for non-user resources (courses, sections) — covered by OneRoster (4.3).
- SCIM filtering beyond `filter=userName eq "..."` — basic filter support only for MVP.

## 4. Personas & User Stories

- **As a university IT administrator using Okta**, I want to assign the Lextures app to a user in Okta so that their Lextures account is created automatically with the correct role.
- **As an IT admin**, I want to deactivate a user in Azure AD so that their Lextures account is immediately disabled and their active sessions are terminated.
- **As a Lextures admin**, I want to see a log of SCIM provisioning events so that I can audit account creation and deletion.
- **As a security officer**, I want to confirm that offboarded employees cannot access Lextures after their AD account is disabled, even if they have an active JWT.

## 5. Functional Requirements

- **FR-1.** The system MUST implement SCIM 2.0 RFC 7644 endpoints for the User resource: `GET /Users`, `GET /Users/:id`, `POST /Users`, `PUT /Users/:id`, `PATCH /Users/:id`, `DELETE /Users/:id`.
- **FR-2.** The system MUST implement the SCIM ServiceProviderConfig endpoint (`GET /ServiceProviderConfig`) declaring supported features.
- **FR-3.** The system MUST implement the SCIM Schemas endpoint (`GET /Schemas`) listing supported User attributes.
- **FR-4.** When a SCIM `DELETE` or `PATCH active=false` is received, the system MUST deactivate the user account and revoke all active sessions (4.8).
- **FR-5.** The system MUST authenticate SCIM requests using a per-institution OAuth 2.0 bearer token issued via the admin UI.
- **FR-6.** The system MUST support the SCIM Group resource (`GET /Groups`, `POST /Groups`, `PATCH /Groups/:id`) for mapping groups to Lextures RBAC roles or course enrollments (optional for MVP but listed in ServiceProviderConfig).
- **FR-7.** The system MUST return SCIM-compliant error responses (RFC 7644 §3.12) with `scimType` and `detail` fields.
- **FR-8.** The system SHOULD support SCIM filtering: `filter=userName eq "{email}"` at minimum.
- **FR-9.** The system MUST maintain a `scim_provisioning_events` log per action for audit purposes.

## 6. Non-Functional Requirements

- **Performance** — SCIM endpoint response p95 ≤ 300 ms for single-resource operations. List operations ≤ 500 ms for up to 1 k users.
- **Security** — Bearer token per institution stored in secrets management (17.17); 256-bit entropy. SCIM endpoint only accessible via HTTPS; tokens rotatable from admin UI without service interruption. Deprovisioning triggers immediate JWT revocation (4.8 refresh token revocation).
- **Privacy & Compliance** — SCIM user attributes (email, name, role) are PII. GDPR: SCIM deprovisioning does not erase education records; only deactivates login. FERPA: role changes via SCIM are FERPA-significant; logged in audit trail.
- **Accessibility** — SCIM provisioning events log UI keyboard accessible. WCAG 2.1 AA.
- **Scalability** — SCIM endpoints stateless; bearer token validated on each request (no session state). Expect up to 10 k users per institution.
- **Reliability** — SCIM operations are idempotent: `POST /Users` for an existing `userName` returns HTTP 409 with `scimType=uniqueness`. `PUT` is a full replace; `PATCH` is partial update.
- **Observability** — Metrics: `scim.operations_total` per verb/resource. Log institution_id, operation, scim_id, affected user_id.
- **Maintainability** — SCIM implementation in `server/src/services/provisioning/scim.rs`; routes in `server/src/routes/provisioning/scim.rs`. Adhere to RFC 7644 as the single source of truth.
- **Internationalization** — SCIM error messages in English (spec-defined); UI event log labels i18n-keyed.
- **Backward compatibility** — SCIM is additive; existing user management unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* an Okta SCIM provisioner with a valid bearer token, *when* it sends `POST /scim/v2/Users` with a new user payload, *then* a Lextures user is created and HTTP 201 is returned with a SCIM User resource.
- **AC-2.** *Given* a user with an active JWT, *when* Okta sends `PATCH /scim/v2/Users/:id` with `{ active: false }`, *then* the user is deactivated, all refresh tokens are revoked, and future API calls with their JWT return HTTP 401.
- **AC-3.** *Given* a SCIM `POST /Users` for an email that already exists, *when* the request is processed, *then* HTTP 409 is returned with `scimType: "uniqueness"`.
- **AC-4.** *Given* `GET /scim/v2/ServiceProviderConfig`, *when* requested with a valid bearer token, *then* the response is a valid SCIM ServiceProviderConfig JSON as per RFC 7644.
- **AC-5.** *Given* a SCIM request with an invalid or expired bearer token, *when* processed, *then* HTTP 401 is returned.
- **AC-6.** *Given* a SCIM `PATCH` updating `displayName`, *when* processed, *then* the Lextures user's `name` is updated and a `scim_provisioning_events` log entry is created.

## 8. Data Model

```sql
-- Migration: server/migrations/090_scim_provisioning.sql

CREATE TABLE scim_bearer_tokens (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    institution_id  UUID NOT NULL,
    token_hash      TEXT NOT NULL UNIQUE,   -- SHA-256 of token
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    revoked_at      TIMESTAMPTZ
);

CREATE TABLE scim_provisioning_events (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    institution_id  UUID NOT NULL,
    operation       TEXT NOT NULL CHECK (operation IN ('create','update','deactivate','delete')),
    scim_resource   TEXT NOT NULL,
    affected_user_id UUID REFERENCES users(id),
    payload_json    JSONB,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

ALTER TABLE users
    ADD COLUMN scim_external_id TEXT;  -- IdP's externalId for the user
```

## 9. API Surface

SCIM endpoints (all under `/scim/v2/`):
| Verb | Path | Auth | Description |
|---|---|---|---|
| GET | `/scim/v2/ServiceProviderConfig` | bearer | Config declaration |
| GET | `/scim/v2/Schemas` | bearer | Schema list |
| GET/POST | `/scim/v2/Users` | bearer | List / Create users |
| GET/PUT/PATCH/DELETE | `/scim/v2/Users/:id` | bearer | User CRUD |
| GET/POST | `/scim/v2/Groups` | bearer | Group operations (optional MVP) |

Admin:
- `POST /api/admin/scim/tokens` — generate bearer token.
- `DELETE /api/admin/scim/tokens/:id` — revoke token.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| IdP sends non-standard SCIM payloads (Okta/Azure quirks) | H | M | Test against Okta and Azure SCIM probes; document quirk workarounds |
| Deprovisioning deletes education records prematurely | M | H | SCIM delete = deactivate only; hard delete requires admin action with confirmation |
| Bearer token leaked grants provisioning access | L | H | Tokens are hashed in DB; UI masks token after creation; rotate on suspected compromise |

## 15. Rollout Plan

- Feature flag: `SCIM_ENABLED` (default: `false`).
