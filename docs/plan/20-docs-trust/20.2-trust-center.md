## 1. Problem Statement

Enterprise and institutional buyers conduct security reviews before adopting an LMS. They expect a public Trust Center containing: a list of sub-processors (third-party vendors who process customer data), a security whitepaper or overview, a history of security incidents, and current certification status. Without a Trust Center, Lextures fails the due-diligence step of virtually every K-12 and HE RFP. Compliance frameworks like SOC 2 and ISO 27001 list sub-processor transparency as a control requirement.

## 2. Goals

- Publish a public Trust Center page at `/trust` listing all sub-processors, their data types, locations, and DPA/agreement status.
- Publish a security overview covering: infrastructure, encryption, access controls, pen testing cadence, and vulnerability disclosure.
- Maintain a public incident history with dates, impact, and resolution summaries.
- Provide an email/form for security inquiries and a link to the responsible disclosure policy.
- Follow sub-processor list conventions from Vanta / Drata / SecureFrame templates for consistency with compliance tooling.

## 3. Non-Goals

- Full SOC 2 audit report publication (shared under NDA on request).
- Hosting a live CVE feed.
- Real-time status page (§18.6 outage banner; §17.13 status page integration).
- DPA negotiation portal (customers can request via sales).

## 4. Personas & User Stories

- **As a district IT admin doing vendor due diligence**, I want to see every third-party vendor Lextures shares student data with, their data processing purpose, and their location so that I can complete our district's sub-processor review.
- **As a university CISO**, I want to read the security whitepaper to understand Lextures' encryption, access controls, and incident response policy so that I can assess risk.
- **As a security researcher**, I want to find the responsible disclosure policy without contacting sales so that I can report a vulnerability through the proper channel.
- **As a customer**, I want to see the history of past security incidents so that I can assess the vendor's transparency and response quality.

## 5. Functional Requirements

- **FR-1.** The system MUST publish a Trust Center page at `/trust` accessible without login.
- **FR-2.** The Trust Center MUST include a sub-processor list table with columns: Vendor Name, Service/Purpose, Data Categories Processed, Headquarters Country, Data Region, DPA/Agreement Status (signed/in-review), link to vendor's own privacy policy.
- **FR-3.** Sub-processors MUST include at minimum: LLM providers (Anthropic/OpenAI/OpenRouter), hosting provider, object storage provider, email delivery provider, monitoring/observability tools, authentication providers, and payment processors (when applicable).
- **FR-4.** The Trust Center MUST include a security overview covering: infrastructure (cloud provider, region), network security, encryption at rest and in transit, access controls (MFA, principle of least privilege), vulnerability management, security testing cadence, and incident response overview.
- **FR-5.** The Trust Center MUST include a certifications and audits section listing current status of SOC 2, ISO 27001, FERPA alignment, COPPA alignment, and WCAG 2.1 AA status.
- **FR-6.** The Trust Center MUST include an incident history table: incident date, severity, summary, impact, resolution date, status (resolved/monitoring).
- **FR-7.** The Trust Center MUST include a contact section with: security disclosure email, link to responsible disclosure policy (SECURITY.md), and a request form for full SOC 2 report (NDA-gated).
- **FR-8.** The sub-processor list MUST include an "Effective Date" and be versioned; subscribers who signed up for updates MUST receive an email when the list changes.
- **FR-9.** The Trust Center SHOULD integrate with Statuspage/Status.io (§18.6) to display current system status.

## 6. Non-Functional Requirements

- **Performance** — Static or SSG page; loads in < 1 s; available from CDN.
- **Security** — Page is public read-only; no sensitive data displayed (no API keys, no internal IPs).
- **Privacy & Compliance** — Sub-processor list itself is a GDPR Art. 28 / FERPA compliance requirement; must be kept current; Vanta/Drata/SecureFrame template conventions followed for compliance tooling compatibility.
- **Accessibility** — WCAG 2.1 AA; tables with proper `<caption>` and column headers; printable.
- **Scalability** — Static page; trivially scalable.
- **Reliability** — Available during platform outages (CDN-served).
- **Observability** — Track `/trust` page views for sales-funnel analytics.
- **Maintainability** — Content in Markdown/MDX or a CMS; sub-processor list in a structured YAML/JSON file parsed at build time.
- **Internationalization** — English primary; consider non-binding translations for major markets.
- **Backward compatibility** — New page at `/trust`; no existing routes affected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a district IT admin visits `/trust`, *When* they look for sub-processors, *Then* they find a table listing all LLM providers (Anthropic, OpenAI, OpenRouter) with data categories, region, and DPA status.
- **AC-2.** *Given* an incident occurred, *When* it is added to the incident history, *Then* a table row appears with date, severity, summary, and resolution date.
- **AC-3.** *Given* the sub-processor list changes (new vendor added), *When* the change is published, *Then* customers who subscribed to sub-processor change notifications receive an email within 24 hours.
- **AC-4.** *Given* the Trust Center page is rendered, *When* an axe-core scan runs, *Then* zero critical WCAG 2.1 AA violations are reported.
- **AC-5.** *Given* a security researcher clicks the disclosure link, *When* the link resolves, *Then* it navigates to the responsible disclosure policy (`SECURITY.md` or `/security` page) with clear reporting instructions.

## 8. Data Model

Sub-processor list stored as structured data (not DB):
```yaml
# clients/web/src/content/trust/sub-processors.yml
- name: Anthropic
  service: AI language model (AI tutor, grading, generation)
  data_categories: [course_content, anonymized_student_queries]
  headquarters: US
  data_region: US
  dpa_status: signed
  privacy_url: https://www.anthropic.com/privacy
```

Incident history in MDX:
```yaml
# clients/web/src/content/trust/incidents.yml
- date: 2026-01-15
  severity: low
  summary: "Planned maintenance overran scheduled window by 45 minutes"
  impact: "Read access degraded for ~500 users"
  resolved_date: 2026-01-15
  status: resolved
```

## 9. API Surface

```
POST /api/trust/sub-processor-updates/subscribe  { email } -> 204
```

All other Trust Center content is static; no API.

## 10. UI / UX

New page: `clients/web/src/pages/TrustCenter.tsx` (or static HTML)

Sections:
1. Overview + status badge (Statuspage embed).
2. Security Overview (accordion sections: Infrastructure, Encryption, Access Controls, Testing, Incident Response).
3. Certifications (SOC 2, ISO 27001 — show "In Progress" until complete).
4. Sub-Processors (filterable table with columns: Vendor, Purpose, Data, Region, DPA Status).
5. Incident History (table, most recent first).
6. Contact / Disclosure (email links, request SOC 2 report form).

Mobile: single-column accordion layout; tables scroll horizontally.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Sub-processor list becomes stale | H | M | Quarterly review reminder in team calendar; YAML in repo = change-tracked |
| Incident disclosed prematurely | L | H | Incident history entries reviewed by legal before publication |
| Competitor uses sub-processor list for competitive intelligence | L | L | Public disclosure is legally required; accept the tradeoff |
