## 1. Problem Statement

Lextures has no public API documentation. Even if a REST API existed today (§16.1 is currently MISSING), there is no OpenAPI specification, no versioning strategy, no rate-limit documentation, and no developer portal. Institutions with technical staff expect to integrate Lextures with their own systems (SIS, analytics platforms, custom dashboards). The absence of public API docs prevents any programmatic integration and signals immaturity to enterprise buyers.

This plan is tightly coupled to §16.1 (Public REST API). The API must be designed and implemented before full documentation can be published; however, the documentation infrastructure, developer portal, and OpenAPI tooling should be built in parallel.

## 2. Goals

- Maintain an OpenAPI 3.1 specification as the single source of truth, generated from Rust source code annotations.
- Publish a developer portal at `developers.lextures.com` (or `/api/docs`) with interactive "Try it out" functionality.
- Document all public endpoints with: description, request/response schemas, authentication requirements, rate limits, and example payloads.
- Implement API versioning strategy (URL-based: `/api/v1/`) and document the deprecation policy.
- Provide SDKs or code examples in at least TypeScript and Python.

## 3. Non-Goals

- Building the public REST API itself (§16.1).
- GraphQL API (deferred).
- Internal-only admin API documentation (though admin APIs should be annotated in OpenAPI for internal use).
- Paid developer tier / API marketplace.

## 4. Personas & User Stories

- **As a university IT developer**, I want to call the Lextures API to pull grade data into our institutional data warehouse so that we can run our own analytics.
- **As a district integration engineer**, I want to use the Lextures API to trigger enrollment updates from our SIS so that we can supplement the CSV import.
- **As a self-learner building a third-party Lextures widget**, I want clear API docs with code examples so that I can build without support.
- **As an instructor**, I want to understand what data I can export via the API so that I can build my own gradebook dashboard.

## 5. Functional Requirements

- **FR-1.** The system MUST maintain an OpenAPI 3.1 specification (`openapi.yaml`) generated from Rust route annotations (using `utoipa` or `aide` crate).
- **FR-2.** The developer portal MUST render the OpenAPI spec as interactive documentation (Swagger UI or Redoc) with "Try it out" capability using personal API tokens.
- **FR-3.** All public API endpoints MUST be documented with: summary, description, request body schema, response schemas (200, 400, 401, 403, 404, 429), and at least one example request/response.
- **FR-4.** The API reference MUST document: authentication (Bearer JWT, personal access tokens), rate limits per endpoint (requests/min, burst), and pagination conventions (cursor-based).
- **FR-5.** The API MUST use URL-based versioning (`/api/v1/`); the documentation MUST cover the versioning and deprecation policy.
- **FR-6.** The documentation MUST include a "Getting Started" guide: how to obtain an API token, make a first request, handle errors, and handle pagination.
- **FR-7.** The system SHOULD provide generated code snippets (TypeScript SDK via `openapi-typescript`, Python snippets via `httpx`) for every endpoint.
- **FR-8.** An OpenAPI spec validation step MUST run in CI (`openapi-validator`) to catch schema drift.
- **FR-9.** The developer portal MUST be publicly accessible without login; trying API calls requires an API token.

## 6. Non-Functional Requirements

- **Performance** — Developer portal loads in ≤ 2 s; API spec download ≤ 200 KB.
- **Security** — "Try it out" uses user's own personal API token; no platform-level test token that could be abused; tokens scoped per §16.2.
- **Privacy & Compliance** — API reference page does not require login; no PII in documentation examples (use placeholder user IDs like `usr_example_123`).
- **Accessibility** — Redoc or Swagger UI MUST meet WCAG 2.1 AA; interactive elements keyboard navigable.
- **Scalability** — Static documentation site hosted on CDN; no dynamic rendering for spec pages.
- **Reliability** — Developer portal available independently from the main app; CDN-hosted.
- **Observability** — Track API reference page views and "Try it out" usage for developer-adoption metrics.
- **Maintainability** — OpenAPI spec auto-generated from Rust; manual overrides in a YAML overlay file; validated in CI.
- **Internationalization** — English-only for API docs (developer audience).
- **Backward compatibility** — Deprecation policy in docs: endpoints deprecated with 6-month notice before removal; `Deprecation` and `Sunset` response headers added to deprecated endpoints.

## 7. Acceptance Criteria

- **AC-1.** *Given* a developer visits the API reference, *When* they navigate to the "List Courses" endpoint, *Then* they see the full request schema, response schema, authentication requirements, and a working "Try it out" button.
- **AC-2.** *Given* a PR modifies a route in `server/src/routes/`, *When* CI runs, *Then* the OpenAPI validator step checks schema validity and fails the PR if any endpoint is undocumented or schema is malformed.
- **AC-3.** *Given* a developer copies the TypeScript code snippet for "Get Enrollment", *When* they paste it into their project and substitute their API token, *Then* the code compiles and makes a successful API call.
- **AC-4.** *Given* an API endpoint is deprecated, *When* a response is returned from that endpoint, *Then* it includes `Deprecation: true` and `Sunset: <date>` headers documented in the API reference.
- **AC-5.** *Given* the API reference's "Getting Started" guide, *When* a developer follows it from scratch, *Then* they can make a first authenticated API call within 5 minutes.

## 8. Data Model

No new DB tables. OpenAPI spec maintained as `server/openapi.yaml`. Personal API tokens (§16.2):

```sql
-- From §16.2 plan:
CREATE TABLE api_tokens (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id),
  name        TEXT NOT NULL,
  token_hash  TEXT NOT NULL UNIQUE,
  scopes      TEXT[] NOT NULL,
  last_used   TIMESTAMPTZ,
  expires_at  TIMESTAMPTZ,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 9. API Surface

The API reference documents all existing and new public endpoints. New endpoints added by this plan:

```
GET /api/v1/openapi.yaml          -> OpenAPI 3.1 spec  (public)
GET /api/v1/openapi.json          -> OpenAPI 3.1 spec  (public, JSON format)
```

Developer portal at `/api/docs` renders the spec via Redoc or Swagger UI.

## 10. UI / UX

Developer portal options:
- **Redoc** (recommended): clean, three-panel layout; generated from OpenAPI; hosted at `/api/docs`.
- **Mintlify or Readme.io** (alternative): prettier, supports additional guides alongside API reference.

Developer portal structure:
1. Getting Started (API token creation, first request, error handling).
2. Authentication (bearer token, personal access tokens, scopes).
3. Rate Limiting (per-endpoint limits, 429 response handling).
4. Pagination (cursor-based pattern).
5. Versioning & Deprecation policy.
6. API Reference (generated from OpenAPI).
7. Code Samples (TypeScript, Python, cURL).
8. Changelog (cross-ref §20.6).

## 11. AI / ML Considerations

Not applicable. AI endpoints (tutor, grading, generation) MUST be documented with additional notes: token usage response headers, async job pattern, rate limits specific to AI calls.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| OpenAPI spec drifts from actual implementation | H | H | Auto-generation from Rust; CI validation on every PR |
| "Try it out" exposes production data if misconfigured | M | M | "Try it out" requires user's own token; sandbox environment link for testing |
| Breaking API changes after publication | M | H | Versioning policy; 6-month deprecation notice; Sunset headers |

## 15. Rollout Plan

- Phase 1: Add `utoipa` annotations to all existing routes; generate `openapi.yaml`; CI validator.
- Phase 2: Deploy Redoc portal at `/api/docs`; publish Getting Started guide.
- Phase 3: Code snippets; TypeScript types package.
- Phase 4: GA after §16.1 API is production-ready.

## 16. Test Plan

- **Unit** — OpenAPI spec validation (spectral rules).
- **CI** — `openapi-validator` check on every PR modifying routes.
- **Manual** — End-to-end walkthrough of Getting Started guide by a developer unfamiliar with the API.
- **Accessibility** — axe-core on Redoc portal; keyboard navigation through endpoint list.

## 17. Documentation & Training

This plan is itself documentation delivery. Internal:
- Developer guide: annotating new routes with `utoipa`; adding examples to OpenAPI spec.
- Style guide: API naming conventions, response shape conventions, error response format.
