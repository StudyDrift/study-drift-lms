## 1. Problem Statement

Lextures has no help center, knowledge base, or in-product support documentation. Every user question requires direct support contact, which is unscalable beyond a handful of customers. K-12 districts and universities expect self-service documentation as a baseline: instructors need to learn the gradebook, students need submission guides, and IT admins need configuration walkthroughs. The absence of a help center is also a trust signal in procurement evaluations ("does this vendor support their product?").

## 2. Goals

- Launch a public help center at `help.lextures.com` (or `/help`) with a searchable knowledge base organized by user role.
- Create a minimum viable article set covering every shipped feature.
- Produce short training videos (2–5 min each) for the top 10 instructor workflows.
- Integrate contextual help links into the product UI (every major page links to its relevant article).
- Enable community Q&A or contact form for questions not answered by articles.

## 3. Non-Goals

- Full LMS training certification program (future).
- Multilingual knowledge base (English MVP; multilingual is a future phase).
- AI chatbot for help center (future enhancement).
- In-product tours (§20.4 — separate plan).

## 4. Personas & User Stories

- **As a first-time instructor**, I want to find a "Getting Started" guide that walks me through creating my first course so that I can begin teaching on day one without support.
- **As a student**, I want to find answers to "How do I submit an assignment?" without emailing support.
- **As a district IT admin**, I want a configuration guide for SSO and CSV import so that I can set up the platform for my district without hand-holding.
- **As a support agent**, I want searchable articles I can link students and instructors to when answering tickets.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a searchable knowledge base with articles organized by user role (Student, Instructor, Admin, IT Admin) and topic area.
- **FR-2.** The help center MUST include articles for every shipped feature referenced in other plan documents; a minimum set includes: course creation, assignments, quizzes, gradebook, enrollment, AI tools, admin console, CSV import, privacy settings.
- **FR-3.** Each article MUST have: title, breadcrumb, last-updated date, role tag, and related articles section.
- **FR-4.** Every major page in the Lextures app MUST include a `?` help icon that links to the contextual article for that page.
- **FR-5.** The help center MUST be publicly accessible (no login required).
- **FR-6.** The system MUST provide a "Was this article helpful? Yes/No" feedback widget on each article; feedback stored for content prioritization.
- **FR-7.** The help center MUST include a contact form or email link for questions not covered by articles.
- **FR-8.** Training videos MUST be produced for at minimum: creating a course, building a quiz, grading submissions, using the gradebook, and setting up as an admin.
- **FR-9.** Videos MUST have captions (WCAG 1.2.2) and a text transcript available.

## 6. Non-Functional Requirements

- **Performance** — Help center MUST load in ≤ 1.5 s on 3G; search results MUST appear in ≤ 500 ms.
- **Security** — Articles are public read-only; no login required; contact form prevents spam via reCAPTCHA or similar.
- **Privacy & Compliance** — Contact form submissions handled per Privacy Policy (§20.1); video transcripts stored for accessibility compliance.
- **Accessibility** — WCAG 2.1 AA for all help center pages; video players with caption support (§8.4).
- **Scalability** — Headless CMS (Notion + Astro export, Mintlify, or Gitbook) handles any traffic scale.
- **Reliability** — Help center hosted separately from the main app; available during platform outages.
- **Observability** — Article view counts; search query analytics (identify gaps); feedback score per article.
- **Maintainability** — Articles in Markdown/MDX in `docs/help/` directory or external CMS; versioned alongside the codebase.
- **Internationalization** — English MVP; Spanish and French in next phase.
- **Backward compatibility** — New standalone site; no existing help infrastructure to migrate.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student searches "how to submit assignment", *When* search results appear, *Then* a relevant article titled "Submitting an Assignment" is in the top 3 results.
- **AC-2.** *Given* an instructor is on the gradebook page in the app, *When* they click the `?` help icon, *Then* a new tab opens to the "Using the Gradebook" article.
- **AC-3.** *Given* a training video is played, *When* captions are enabled, *Then* accurate captions are displayed for the entire video duration.
- **AC-4.** *Given* an article's "Was this helpful? No" feedback is submitted, *When* the feedback is received, *Then* it is stored with the article ID and timestamp for content review.
- **AC-5.** *Given* the help center search is queried for a term with no matching article, *When* the search returns no results, *Then* a "Contact support" CTA is displayed prominently.

## 8. Data Model

```sql
-- 0XX_help_feedback.sql
CREATE TABLE help_article_feedback (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  article_id TEXT NOT NULL,
  helpful    BOOLEAN NOT NULL,
  comment    TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Articles stored in CMS or `docs/help/**/*.md` (not DB).

## 9. API Surface

```
POST /api/help/feedback  { article_id, helpful, comment? } -> 204
```

Search provided by help center platform (Mintlify, Gitbook, Algolia DocSearch, or similar) — not a custom implementation.

## 10. UI / UX

Help center platform: Mintlify (recommended — built for technical products, Algolia search built-in, WCAG-friendly, MDX support) or Gitbook.

In-product contextual links: `clients/web/src/components/HelpLink.tsx` — small `?` icon button that opens help article URL in a new tab; each page imports its article slug constant.

Homepage: hero search bar + role-based quick links (Student, Instructor, Admin). Article layout: sidebar TOC, main content, related articles, feedback widget.

## 11. AI / ML Considerations

Not applicable for MVP. Future: AI-powered help search using content embeddings.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Help articles become stale after feature updates | H | M | Assign article ownership per feature; update checklist in feature PRs |
| Video production takes too long | M | M | Screen recordings with Loom/Descript are faster than scripted productions |
| Search returns irrelevant results | M | M | Use Algolia DocSearch with tuning; weekly search-query review |

## 15. Rollout Plan

- Phase 1: Choose and configure help center platform; publish 10 essential articles (getting started, courses, assignments, gradebook, admin).
- Phase 2: Produce 5 training videos; add in-product `HelpLink` components.
- Phase 3: Expand to full feature coverage; add contact form; feedback widget.
- Phase 4: Spanish/French translations.

## 16. Test Plan

- **Accessibility** — axe-core on help center pages; caption verification on all videos.
- **Search** — Manual test of 20 common user queries; verify top result relevance.
- **Broken links** — Link checker CI against all help articles.
- **Performance** — Lighthouse ≥ 90 on help center homepage.

## 17. Documentation & Training

The help center is itself the documentation deliverable. Internal:
- Article authoring guide: tone, structure, screenshot standards.
- Video production SOP: recording setup, Loom workflow, caption review.
