## 1. Problem Statement

New users arriving at Lextures — whether self-learners, instructors, or district-onboarded students — have no in-product guidance. Without onboarding tours, new instructor activation rates are low, and support tickets about basic features ("where is the gradebook?") are high. Every SaaS product above a certain complexity threshold requires guided onboarding. Shepherd.js and Driver.js are the leading lightweight JavaScript libraries for step-by-step product tours that are accessibility-friendly.

## 2. Goals

- Deliver role-aware first-login onboarding tours for students, instructors, and admins.
- Allow users to dismiss, pause, and restart tours from their profile settings.
- Link each tour step to the relevant help center article (§20.3).
- Track tour completion rates to identify drop-off steps.
- Implement using Shepherd.js (preferred for accessibility support and active maintenance).

## 3. Non-Goals

- Feature announcement modals for every update (that is a changelog popover — §20.6).
- Full interactive product simulations.
- Video walkthroughs (§20.3).
- Replacing the help center — tours supplement, not replace, documentation.

## 4. Personas & User Stories

- **As a first-time instructor**, I want a tour that shows me how to create a course, add a module, and publish it so that I can start teaching without reading documentation.
- **As a student**, I want a brief tour showing me where to find my courses, assignments, and grades so that I feel oriented on my first day.
- **As an admin**, I want a tour of the admin console's main sections so that I can find user management and settings quickly.
- **As a returning user after a major product update**, I want a "What's new" mini-tour highlighting changed UI so that I can adapt quickly.

## 5. Functional Requirements

- **FR-1.** The system MUST present a role-specific onboarding tour to users on their first login (`has_completed_onboarding = false`).
- **FR-2.** The system MUST implement separate tours for: student (5–7 steps), instructor (7–10 steps), and admin (5–7 steps).
- **FR-3.** Each tour step MUST include: a title, 1–3 sentences of explanatory text, a highlighted UI element, and an optional link to the related help article.
- **FR-4.** The tour MUST be dismissible at any step via an "Exit tour" button; dismissed tours MUST be re-accessible from the user's profile settings.
- **FR-5.** The system MUST persist tour completion state (`has_completed_onboarding`, `dismissed_tours[]`) per user in the database.
- **FR-6.** Tour completion rate per step MUST be tracked as analytics events.
- **FR-7.** The system SHOULD support "feature spotlight" mini-tours (2–3 steps) triggered by first use of a specific feature (e.g., first time an instructor opens the AI tools panel).
- **FR-8.** The system MUST allow super_admin to trigger a new tour for all users of a specific role (e.g., after a major UI redesign).

## 6. Non-Functional Requirements

- **Performance** — Shepherd.js bundle ≤ 50 KB gzipped; tours MUST NOT affect Time to Interactive on first load (lazy-loaded after main content).
- **Security** — No PII in tour analytics events; tour trigger API requires authentication.
- **Privacy & Compliance** — Tour analytics are product usage data (not education records); GDPR: tour completion state is user preference data, erasable on account deletion.
- **Accessibility** — Shepherd.js provides ARIA dialog roles, focus trapping, and keyboard navigation (Tab/Shift+Tab through steps, Escape to dismiss); all tour steps MUST pass axe-core.
- **Scalability** — Tour state stored as a JSON column on the user record; minimal DB impact.
- **Reliability** — Tour MUST degrade gracefully if a target element is not found (skip the step with a fallback message).
- **Observability** — `onboarding_tour_started{role}`, `onboarding_tour_completed{role}`, `onboarding_step_dismissed{step_id}` events.
- **Maintainability** — Tour definitions in `clients/web/src/tours/` as TypeScript files; one file per tour; steps reference element selectors.
- **Internationalization** — Tour step text in i18n keys.
- **Backward compatibility** — Tours shown only to users with `has_completed_onboarding = false`; existing users unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a new student logs in for the first time, *When* the dashboard loads, *Then* the student onboarding tour starts automatically after a 500 ms delay.
- **AC-2.** *Given* a user is mid-tour on step 3 of 7, *When* they click "Exit tour", *Then* the tour closes immediately, and the user is NOT shown the tour again on next login.
- **AC-3.** *Given* a user dismissed the tour, *When* they navigate to Profile Settings → "Restart Onboarding Tour", *Then* the tour restarts from step 1.
- **AC-4.** *Given* a tour step targets a UI element that no longer exists in the DOM, *When* the tour renders that step, *Then* it falls back to a centered modal (no target highlight) without throwing a JavaScript error.
- **AC-5.** *Given* an axe-core scan runs on the tour dialog, *When* the scan completes, *Then* zero critical WCAG 2.1 AA violations are found.
- **AC-6.** *Given* a tour completes all steps, *When* the user dismisses the final step, *Then* `has_completed_onboarding = true` is persisted in the user record.

## 8. Data Model

```sql
-- Add to users table:
ALTER TABLE users ADD COLUMN IF NOT EXISTS
  onboarding_state JSONB NOT NULL DEFAULT '{
    "has_completed_onboarding": false,
    "dismissed_tours": [],
    "completed_tours": []
  }';
```

## 9. API Surface

```
PATCH /api/me/onboarding  { has_completed_onboarding?, dismissed_tour?, completed_tour? }
                           -> 204
GET   /api/me/onboarding  -> OnboardingStateDto
POST  /api/admin/onboarding/reset { role?, user_ids? }  -> 202  (super_admin — reset for cohort)
```

Tour analytics events emitted client-side via the product analytics pipeline (not a custom API).

## 10. UI / UX

Library: **Shepherd.js** (v12+; actively maintained; ARIA-compliant).

Tour files:
- `clients/web/src/tours/student-onboarding.ts`
- `clients/web/src/tours/instructor-onboarding.ts`
- `clients/web/src/tours/admin-onboarding.ts`

Each step definition:
```typescript
{
  id: 'course-list',
  title: 'Your Courses',
  text: 'All your enrolled courses appear here. Click a course to get started.',
  attachTo: { element: '[data-tour="course-list"]', on: 'bottom' },
  helpUrl: 'https://help.lextures.com/student/finding-your-courses'
}
```

All interactive elements get `data-tour="<id>"` attributes (avoids brittle CSS selector coupling).

Profile settings page: "Restart Onboarding Tour" button visible if `has_completed_onboarding = true` or `dismissed_tours` is non-empty.

## 13. Dependencies & Sequencing

- Must ship after: stable feature UI (tours must not break after every redesign).
- Must ship before: first public beta launch.
- Can ship in parallel with: 20.3 (help center content cross-linked by tours).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Tour steps break when UI is redesigned | H | M | Use data-tour attributes (not CSS selectors); graceful skip on missing element |
| Tours shown to users who already know the product (post-SSO rollout) | M | L | super_admin reset API; users can always dismiss |
| Tour analytics blocked by ad blockers | M | L | Use first-party analytics endpoint, not third-party tracker |

## 15. Rollout Plan

- Feature flag: `onboarding_tours_enabled` (OpenFeature, default `false`).
- Phase 1: Build and test instructor tour internally.
- Phase 2: Student tour; admin tour; enable for new signups only.
- Phase 3: GA for all users (suppress for existing users with `created_at` before launch date).

## 16. Test Plan

- **Unit** — Tour step registration; onboarding state persistence; missing-element graceful skip.
- **End-to-end** — Playwright: new student login → tour starts → all steps navigable → completion persisted.
- **Accessibility** — axe-core on Shepherd dialog; keyboard-only navigation through all tour steps.
- **Manual** — Test tour on mobile (touch navigation); test dismiss and restart flow.

## 17. Documentation & Training

- Internal: Tour authoring guide — how to add a new tour step; data-tour attribute convention.
- Help center: "Restarting the Onboarding Tour" article.

## 18. Open Questions

1. Should existing users (onboarded before this feature) see the tour once, or opt-in from settings only?
2. Should we show a "Skip" option on the first step and a "Start" option to confirm before beginning?
3. Should the tour be triggered on mobile, or only on desktop (mobile tour UX with Shepherd can be awkward on small screens)?
