## 1. Problem Statement

US K-12 school districts are legally required to sign Data Processing Agreements (DPAs) — often called "Student Data Privacy Agreements" — with every EdTech vendor that processes student data. The Student Data Privacy Consortium (SDPC) provides the National Data Privacy Agreement (NDPA), now adopted by nearly 40 states, as a standardized contract template that districts can use with minimal legal review. Lextures currently offers no signable DPA artifact, no self-service DPA portal for districts, and has not completed the SDPC program registration. Every district that evaluates Lextures will ask for a signed NDPA or equivalent before issuing a purchase order, making this a sales-cycle blocker for the K-12 market.

## 2. Goals

- Register Lextures as a vendor in the SDPC National Data Privacy Agreement program.
- Generate a pre-populated NDPA/DPA document that districts can review and countersign.
- Build a self-service DPA portal for district admins to access, sign, and download the agreement.
- Maintain a complete data inventory (data elements collected, purposes, sub-processors) to populate DPA exhibits.
- Log all DPA acceptance events for compliance evidence.

## 3. Non-Goals

- GDPR Data Processing Agreement for EU institutions (see 10.3) — separate document template.
- Legal counsel review — this plan delivers the engineering and process scaffolding; legal sign-off is required separately.
- Automated contract signature (DocuSign/HelloSign integration) — v1 uses PDF download + email submission; e-sign is a v2 enhancement.

## 4. Personas & User Stories

- **As a district IT director**, I want to download a pre-populated NDPA that my legal team can review and countersign so that we can onboard Lextures within our procurement cycle.
- **As a district privacy officer**, I want to see the complete list of student data elements Lextures collects so that I can evaluate compliance with our state privacy laws.
- **As a Lextures admin**, I want to track which districts have accepted the DPA so that we have evidence for regulatory audits.
- **As a district superintendent**, I want to know which sub-processors (e.g., AI providers) have access to student data so that we can meet our board's approval requirements.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a downloadable NDPA template (PDF and DOCX) pre-populated with Lextures entity name, registered address, data inventory summary, and sub-processor list.
- **FR-2.** The system MUST maintain a machine-readable data inventory listing all student data elements collected, the purpose for each, the legal basis, retention period, and whether the element is shared with third parties.
- **FR-3.** The system MUST provide an SDPC-compatible data inventory export in the SDPC CSV format so that district privacy officers can import it into their privacy management tools.
- **FR-4.** The system MUST record DPA acceptance events when a district admin checks an acceptance acknowledgment, including: admin user ID, org ID, DPA version, timestamp, and IP address.
- **FR-5.** The system SHOULD send an email notification to the district admin and to the Lextures compliance team when a DPA acceptance is recorded.
- **FR-6.** The system MUST version DPA templates; when the template is updated (e.g., due to new sub-processors), existing signatories MUST be notified and asked to re-acknowledge within 30 days.
- **FR-7.** The system SHOULD display a "DPA signed" badge in the admin console for districts that have accepted the current DPA version.

## 6. Non-Functional Requirements

- **Performance** — DPA PDF generation MUST complete in < 10 seconds.
- **Security** — DPA acceptance records MUST be append-only (no UPDATE or DELETE permitted); stored in `compliance.dpa_acceptances`. District admin must be authenticated and hold `org:admin` permission.
- **Privacy & Compliance** — SDPC NDPA v3 (current version) clauses: data use limitation, prohibition on sale, data security, breach notification, data return/destruction. Conformance with FERPA 34 CFR Part 99 and relevant state privacy laws (CA SOPIPA, NY Ed Law 2-d, IL SOPPA — referenced in 10.6). Data inventory MUST be accurate and current; inaccurate inventories constitute a deceptive trade practice under FTC Act § 5.
- **Accessibility** — DPA portal UI MUST meet WCAG 2.1 AA (10.7).
- **Reliability** — DPA acceptance records MUST be backed up as part of the backup/restore plan (10.15).
- **Observability** — Log all DPA acceptance and download events to `admin_audit_log` (10.11).
- **Maintainability** — DPA template content MUST be maintained in version-controlled Markdown/Jinja2 templates rendered to PDF; not hardcoded.
- **Internationalization** — English only for v1; Spanish translation considered for v2.
- **Backward compatibility** — Existing district tenants should be prompted to accept DPA on next admin login if no prior acceptance recorded.

## 7. Acceptance Criteria

- **AC-1.** *Given* a district admin logs into the admin console, *when* no DPA acceptance is on record for the current DPA version, *then* a DPA acknowledgment banner appears with a link to review and accept.
- **AC-2.** *Given* a district admin clicks "Download DPA," *when* the PDF is generated, *then* it contains Lextures' entity name, current sub-processor list, data inventory summary, and the current DPA version string.
- **AC-3.** *Given* a district admin accepts the DPA, *when* the acceptance is recorded, *then* the `compliance.dpa_acceptances` row is created, an email is sent to both parties, and the "DPA signed" badge appears in the admin console.
- **AC-4.** *Given* a new sub-processor is added and the DPA version is incremented, *when* the update is published, *then* all districts with prior acceptances receive an email notification requesting re-acknowledgment within 30 days.
- **AC-5.** *Given* a compliance admin exports the data inventory in SDPC CSV format, *when* the export is downloaded, *then* all data elements are present with their purposes, legal bases, and sub-processor flags.

## 8. Data Model

```sql
-- Migration: 077_dpa.sql

CREATE TABLE IF NOT EXISTS compliance.dpa_versions (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  version_str  TEXT NOT NULL UNIQUE,  -- e.g. '2026-04-01'
  template_url TEXT NOT NULL,         -- S3/R2 path to PDF
  effective_at TIMESTAMPTZ NOT NULL,
  notes        TEXT,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE IF NOT EXISTS compliance.dpa_acceptances (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id         UUID NOT NULL REFERENCES org.organizations(id),
  dpa_version_id UUID NOT NULL REFERENCES compliance.dpa_versions(id),
  accepted_by    UUID NOT NULL REFERENCES user.users(id),
  accepted_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  ip_address     INET,
  UNIQUE(org_id, dpa_version_id)
);

CREATE TABLE IF NOT EXISTS compliance.data_inventory (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  element_name    TEXT NOT NULL,
  category        TEXT NOT NULL,  -- e.g. 'identity','academic','behavioral'
  purpose         TEXT NOT NULL,
  legal_basis     TEXT NOT NULL,
  retention_days  INTEGER,
  shared_with_sub_processors BOOLEAN NOT NULL DEFAULT FALSE,
  sub_processor_names TEXT[],
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## 10. UI / UX

- **DPA portal page** (`/admin/dpa`) — DPA version history, current version download, acceptance status, accept button.
- **Data inventory page** (`/admin/data-inventory`) — table of data elements with filter; SDPC CSV download.
- **Admin console banner** — DPA acceptance prompt for unsigned orgs.
- Empty state: "DPA accepted — next review due [date]."
- WCAG 2.1 AA; i18n keys prefixed `dpa.*`.

## 11. AI / ML Considerations

The data inventory MUST include a row for each AI processing activity (e.g., "quiz question generation", "rubric generation") identifying OpenRouter as a sub-processor and the data elements passed (assignment text, student response). This is required SDPC exhibit content.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| SDPC NDPA template version changes | M | M | Parameterized template with version pinning; annual review cycle |
| District legal team requests non-standard DPA modifications | H | M | Offer negotiation process; log custom agreements separately |
| Data inventory becomes stale as features are added | H | M | Require data inventory update as part of feature-launch checklist |

## 15. Rollout Plan
