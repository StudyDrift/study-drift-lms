## 1. Problem Statement

Encryption of data at rest is a foundational security control required by GDPR Article 32, SOC 2 TSC CC6.7, ISO 27001:2022 Annex A.8.24, and NIST 800-53 SC-28. The current Lextures deployment uses a DigitalOcean droplet running Docker Compose; neither Postgres TDE (transparent data encryption) nor full-disk encryption is documented or codified in `iac/`. If a server is seized, a disk is removed, or a managed DB snapshot is obtained without authorization, all student and instructor data is accessible in plaintext. This is a material security gap that blocks SOC 2 and ISO 27001 certification.

## 2. Goals

- Enable and document full-disk encryption for all production server volumes.
- Enable Postgres-level encryption for backups and managed database snapshots.
- Configure AES-256 encryption for all object storage buckets (DSAR archives, file uploads, backups).
- Implement application-level column encryption for the most sensitive PII fields (as defense-in-depth).
- Document key management and rotation procedures.

## 3. Non-Goals

- Homomorphic encryption or encrypted queries — not practical at this stage.
- Client-side encryption of content authored by instructors — out of scope.
- Hardware Security Module (HSM) integration — v2 once SOC 2 scope matures.

## 4. Personas & User Stories

- **As a SOC 2 auditor**, I want evidence that all storage media containing customer data is encrypted at rest so that I can close CC6.7.
- **As a GDPR DPO**, I want to declare that technical safeguards meeting Article 32 are implemented so that we can demonstrate appropriate security measures.
- **As an ops engineer**, I want encryption enabled and managed through Terraform so that it cannot be accidentally disabled during infrastructure changes.
- **As a security engineer**, I want column-level encryption on SSN, DOB, and contact fields so that a database dump is not immediately exploitable.

## 5. Functional Requirements

- **FR-1.** The system MUST configure all production virtual machine volumes with OS-level full-disk encryption (LUKS on Linux, or cloud-provider managed encryption, e.g., DigitalOcean encrypted volumes / AWS EBS with KMS).
- **FR-2.** The system MUST enable encryption for all managed Postgres backups and snapshots (cloud-provider: DigitalOcean Managed Database encryption at rest is AES-256; or self-managed: WAL-G with AES-256-CTR encryption to object storage).
- **FR-3.** The system MUST configure server-side encryption (SSE-S3 or SSE-C with AES-256) for all object storage buckets used for file uploads, DSAR archives, and database backups.
- **FR-4.** The system SHOULD implement application-level column encryption (using `pgcrypto` or a Rust-side AES-GCM wrapper) for the following PII columns: `users.parent_email`, `users.date_of_birth`, `compliance.*` archive paths.
- **FR-5.** The system MUST store encryption keys in a dedicated secrets manager (AWS KMS, HashiCorp Vault, or DigitalOcean managed secrets); keys MUST NOT be stored in environment variables or code repositories.
- **FR-6.** The system MUST enforce an annual key-rotation policy; key rotation MUST be automated or calendar-reminded.
- **FR-7.** All encryption configurations MUST be codified in Terraform/IaC (`iac/production/`) and reviewed in pull requests; ad-hoc manual configuration is prohibited.

## 6. Non-Functional Requirements

- **Performance** — AES-256 hardware acceleration (AES-NI) is available on all modern x86/ARM instances; throughput impact < 2% for Postgres operations. Column encryption adds < 1 ms per read/write for affected columns.
- **Security** — Key material MUST use AES-256-GCM for application-level encryption; key hierarchy: master key in KMS → data encryption key (DEK) encrypted by master key → DEK stored in `secrets.encryption_keys`. Threat model: protects against stolen disk/snapshot; does NOT protect against compromised application runtime (in-memory). See `docs/SECURITY.md` for threat model context.
- **Privacy & Compliance** — GDPR Article 32(1)(a) (pseudonymisation and encryption); SOC 2 TSC CC6.7 (encryption of data at rest); ISO/IEC 27001:2022 Annex A.8.24 (use of cryptography); NIST SP 800-53 SC-28 (Protection of Information at Rest); NIST SP 800-57 (key management). FIPS 140-2 Level 1 compliance preferred for US government-adjacent customers (AES-256-GCM satisfies this).
- **Scalability** — Cloud-provider managed encryption (DigitalOcean, AWS) scales automatically.
- **Reliability** — Key unavailability MUST NOT cause application outage; DEK is cached in memory after first KMS call; refresh interval: 1 hour.
- **Observability** — Alert if KMS health check fails for > 5 minutes. Log key rotation events to `admin_audit_log` (10.11). Cross-link: 17.7.
- **Maintainability** — All encryption configuration in `iac/production/encryption.tf`; application-level crypto in `server/src/crypto/aes.rs`.
- **Internationalization** — N/A (infrastructure concern).
- **Backward compatibility** — Column encryption added to nullable columns with a backfill job; existing rows encrypted in batch.

## 7. Acceptance Criteria

- **AC-1.** *Given* a production VM disk is removed and attached to another system, *when* an unauthorized party attempts to read the filesystem, *then* the data is unreadable without the volume encryption key (verified by cloud-provider encryption documentation and IaC review).
- **AC-2.** *Given* a Postgres managed database snapshot is obtained, *when* an unauthorized party attempts to restore it without the cloud account credentials, *then* the snapshot is encrypted and inaccessible (verified by DigitalOcean/AWS encryption-at-rest documentation for managed databases).
- **AC-3.** *Given* an object storage bucket is accessed by a party with bucket read permissions but no SSE key, *when* they attempt to read a file, *then* the file is returned encrypted and not directly readable.
- **AC-4.** *Given* column encryption is enabled for `users.date_of_birth`, *when* a developer queries the raw Postgres table as the application DB user, *then* the column value is an encrypted ciphertext, not a plaintext date.
- **AC-5.** *Given* the annual key rotation is due, *when* the rotation script runs, *then* a new DEK is generated, all affected ciphertext is re-encrypted, and the event is logged in `admin_audit_log` with `event_type = 'key_rotation'`.

## 8. Data Model

```sql
-- Migration: 083_encryption.sql

CREATE TABLE IF NOT EXISTS secrets.encryption_keys (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  key_id          TEXT NOT NULL UNIQUE,    -- KMS key alias
  key_version     INTEGER NOT NULL DEFAULT 1,
  algorithm       TEXT NOT NULL DEFAULT 'AES-256-GCM',
  status          TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active','rotating','retired')),
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  rotated_at      TIMESTAMPTZ
);
```

## 11. AI / ML Considerations

AI prompt content (student assignment text sent to OpenRouter) is processed in-memory and not stored encrypted at application level; the PII redaction proxy (19.11) is the primary control. Encryption at rest covers stored data (DB, S3) only.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Column encryption breaks existing queries | M | M | Encrypt nullable columns; encrypt in a new column, migrate reads, drop old; test in staging |
| KMS unavailability blocks application startup | L | H | Cache DEK in memory; circuit breaker with grace period |
| Key rotation job fails silently | M | M | Alert on non-zero exit code; log to audit log |

## 15. Rollout Plan

- Phase 1: IaC — verify cloud-provider volume and DB encryption is enabled in `iac/production/`.
- Phase 2: Object storage SSE configuration.
- Phase 3: Application-level column encryption for top-5 PII fields; backfill job.
- Phase 4: Key rotation automation + annual reminder.

## 16. Test Plan

- **Unit** — AES-256-GCM encrypt/decrypt round-trip; key derivation.
- **Integration** — Column encryption: write encrypted value; read back and decrypt correctly.
- **Security** — Raw DB query returns ciphertext for encrypted columns; key rotation produces new ciphertext.
- **Ops** — IaC plan shows encryption enabled on all resources; `terraform plan` diff-free after rotation.
