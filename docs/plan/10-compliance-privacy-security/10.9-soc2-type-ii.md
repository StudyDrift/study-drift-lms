## 1. Problem Statement

Service Organization Control 2 (SOC 2) Type II is the primary IT-security certification required by US higher-education institutions when approving a new SaaS vendor. A SOC 2 Type II report, issued by an independent AICPA-licensed CPA firm, attests that the service organization's controls covering the Trust Services Criteria (TSC) — Security, Availability, Confidentiality, Processing Integrity, and Privacy — operated effectively over a defined observation period (typically 6–12 months). Lextures has no audit, no formal control policies, and no evidence-collection infrastructure. Every university IT review will ask for the SOC 2 report in the procurement questionnaire; absence causes immediate disqualification in the HE segment.

## 2. Goals

- Define the SOC 2 system description and in-scope Trust Services Criteria.
- Implement or document all controls required for the Security and Availability TSC (baseline) and the Privacy TSC (for student-data processing).
- Establish continuous evidence collection for control operation (audit logs, change management, access reviews).
- Engage a qualified CPA firm for the Type I readiness assessment (6 months) and Type II audit (12-month observation).
- Achieve a clean SOC 2 Type II report with no exceptions for the Security TSC within 18 months.

## 3. Non-Goals

- ISO 27001 certification (see 10.10) — separate standard, shares some evidence.
- Achieving all five TSC in the initial engagement — Security + Availability + Privacy is the v1 scope; Confidentiality and Processing Integrity are v2.
- Building a full GRC platform — lightweight policy documents and evidence collection scripts are sufficient for the initial audit cycle.

## 4. Personas & User Stories

- **As a university CISO**, I want to review Lextures' SOC 2 Type II report so that I can approve the vendor for use with student data.
- **As a Lextures compliance officer**, I want an evidence-collection pipeline so that I can respond to auditor requests without manual scrambling.
- **As a developer**, I want to know which security controls my code changes must not break so that I don't inadvertently cause an audit exception.
- **As an ops engineer**, I want infrastructure change controls documented and enforced so that IaC changes go through an approved review process.

## 5. Functional Requirements

- **FR-1.** The system MUST implement and document controls for all required SOC 2 TSC Security criteria (CC1–CC9): risk assessment (CC3), logical and physical access (CC6), change management (CC8), operations (CC7), communications (CC2).
- **FR-2.** The system MUST maintain an access-review cadence: privileged access reviewed quarterly; all production access reviewed semi-annually; results stored in `compliance.access_reviews`.
- **FR-3.** The system MUST enforce a change-management policy: all production code and IaC changes require a reviewed pull request; direct commits to `main` are blocked by branch protection rules.
- **FR-4.** The system MUST maintain an incident-response plan with defined roles, escalation paths, and post-mortem requirements; the plan MUST be tested via tabletop exercise at least annually.
- **FR-5.** The system MUST collect and retain the following evidence categories for audit: authentication logs (180 days), access reviews, change tickets / PRs, vulnerability scan results, backup verification logs, training completion records.
- **FR-6.** The system SHOULD implement a vendor-risk management process: third-party sub-processors reviewed annually against SOC 2 or equivalent reports.
- **FR-7.** The system MUST publish a System Description document (per SSAE 18) describing the service boundary, infrastructure components, and control environment.

## 6. Non-Functional Requirements

- **Performance** — Evidence-collection scripts MUST run in < 5 minutes per daily job.
- **Security** — Audit log integrity: `admin_audit_log` entries MUST be append-only; hash chaining or write-once storage preferred (CC7.2). Access to evidence bucket requires MFA and separate IAM role.
- **Privacy & Compliance** — SOC 2 Trust Services Criteria 2017 (Security: CC1–CC9; Availability: A1; Privacy: P1–P8). SSAE 18 (AT-C section 320) audit standard. AICPA SOC 2 Guide. NIST SP 800-53 Rev. 5 controls underpin specific TSC criteria: AC (Access Control), AU (Audit and Accountability), CA (Assessment and Authorization), CM (Configuration Management), CP (Contingency Planning), IR (Incident Response), RA (Risk Assessment), SA (System and Services Acquisition), SC (System and Communications Protection), SI (System and Information Integrity).
- **Availability** — CC9.2: system availability target of 99.9% uptime measured monthly; status page required (see 17.13).
- **Scalability** — Evidence collection scales with CI/CD pipeline; no special scaling requirements.
- **Reliability** — Evidence retention: 3 years minimum (typical SOC 2 lookback).
- **Observability** — All security-relevant events (logins, privilege escalation, config changes, policy violations) MUST flow through the `admin_audit_log` (10.11) and be queryable by auditors. Cross-link: plan 17.7 (Observability).
- **Maintainability** — Control policies stored as Markdown in `docs/soc2/`; evidence collection scripts in `scripts/compliance/`.
- **Internationalization** — English only; SOC 2 is a US standard.
- **Backward compatibility** — Audit log schema is append-only; no migration of existing data required.

## 7. Acceptance Criteria

- **AC-1.** *Given* a developer pushes directly to the `main` branch, *when* GitHub branch protection is active, *then* the push is rejected and the developer is directed to open a pull request.
- **AC-2.** *Given* the quarterly access review is due, *when* the compliance team runs the evidence-collection script, *then* a report is generated listing all users with production access, their roles, and the date of last review.
- **AC-3.** *Given* a security incident is declared, *when* the incident-response runbook is activated, *then* the incident is logged in `compliance.incidents`, and the incident post-mortem is completed within 5 business days of resolution.
- **AC-4.** *Given* the Type I readiness assessment is performed, *when* the CPA firm reviews evidence, *then* fewer than 5 control deficiencies are identified and none are in the CC6 (Logical Access) criteria family.
- **AC-5.** *Given* the Type II audit observation period concludes, *when* the report is issued, *then* the Security TSC has no exceptions and the report is made available to prospective HE customers under NDA.
- **AC-6.** *Given* the annual vendor-risk review runs, *when* OpenRouter's SOC 2 status is checked, *then* a current report (< 12 months old) is on file in the vendor risk register.

## 8. Data Model

```sql
-- Migration: 079_soc2_compliance.sql

CREATE TABLE IF NOT EXISTS compliance.access_reviews (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  reviewer_id     UUID NOT NULL REFERENCES user.users(id),
  review_type     TEXT NOT NULL CHECK (review_type IN ('privileged','all_production','third_party')),
  reviewed_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  findings        JSONB,
  next_review_due TIMESTAMPTZ
);

CREATE TABLE IF NOT EXISTS compliance.incidents (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title           TEXT NOT NULL,
  severity        TEXT NOT NULL CHECK (severity IN ('P0','P1','P2','P3')),
  status          TEXT NOT NULL DEFAULT 'open' CHECK (status IN ('open','contained','resolved','closed')),
  opened_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  resolved_at     TIMESTAMPTZ,
  post_mortem_url TEXT,
  tsc_criteria    TEXT[]   -- affected TSC criteria, e.g. {'CC6.1','A1.2'}
);

CREATE TABLE IF NOT EXISTS compliance.vendor_risk (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  vendor_name     TEXT NOT NULL UNIQUE,
  soc2_report_url TEXT,
  report_date     DATE,
  risk_tier       TEXT NOT NULL CHECK (risk_tier IN ('critical','high','medium','low')),
  next_review_due DATE,
  notes           TEXT
);
```
