## 1. Problem Statement

A Voluntary Product Accessibility Template (VPAT) — formally the ACR (Accessibility Conformance Report) — is the industry-standard document that US federal and state agencies, school districts, and universities require vendors to provide during IT procurement. It declares conformance status against WCAG 2.1 AA, Section 508 (36 CFR Part 1194), and optionally EN 301 549. Lextures has not produced a VPAT, and without one, K-12 district procurement officers and university IT accessibility offices will reject the product on first review. This is a same-day sales-cycle blocker that can be unblocked once the WCAG audit (10.7) establishes the honest conformance baseline.

## 2. Goals

- Produce and publish an accurate VPAT 2.5 INT (international edition covering WCAG 2.1, Section 508, and EN 301 549) for the current Lextures release.
- Host the VPAT on a public URL and link it from the product site, privacy policy page, and admin console.
- Establish a process for updating the VPAT with each major release.
- Provide an HTML version of the VPAT alongside the standard Word/PDF format for assistive technology users reviewing the document itself.

## 3. Non-Goals

- Performing the underlying WCAG audit (see 10.7).
- Achieving full conformance — the VPAT honestly reports current status; remediation is tracked in 10.7 and Section 12.
- ATAG 2.0 (Authoring Tool Accessibility Guidelines) conformance for the TipTap editor — future consideration.

## 4. Personas & User Stories

- **As a K-12 district IT director conducting a procurement review**, I want to download Lextures' VPAT so that I can assess accessibility conformance against our district's requirements.
- **As a university accessibility coordinator**, I want the VPAT to be honest and detailed so that I can identify residual risks and plan accommodations.
- **As a procurement specialist**, I want the VPAT to cover Section 508 Table 1 (Electronic and Information Technology Hardware), Table 2 (Software), and Table 3 (Support Documentation and Services) so that the review is complete.
- **As a Lextures compliance manager**, I want a repeatable process for updating the VPAT each release so that the document stays current.

## 5. Functional Requirements

- **FR-1.** The system MUST publish a VPAT 2.5 INT covering: WCAG 2.1 A and AA success criteria, Section 508 Chapter 5 (Software) and Chapter 6 (Support Documentation), and EN 301 549 Clauses 9 and 11.
- **FR-2.** Each WCAG success criterion entry MUST have a conformance level (Supports / Partially Supports / Does Not Support / Not Applicable) and explanatory remarks based on the 10.7 audit findings.
- **FR-3.** The VPAT MUST be available in three formats: Microsoft Word (.docx), PDF, and HTML.
- **FR-4.** The VPAT MUST include the product version and date of the most recent evaluation in its header.
- **FR-5.** The VPAT page at `/accessibility/vpat` MUST be publicly accessible without authentication.
- **FR-6.** The system SHOULD include contact information for users who need accommodation support (ADA/Section 504 contact email or form link).
- **FR-7.** The process MUST include a VPAT update checklist run at each major release, with a GitHub issue template for tracking.

## 6. Non-Functional Requirements

- **Performance** — VPAT HTML page MUST load in < 1 second; PDF/DOCX MUST be < 5 MB.
- **Security** — Public page requires no auth; ensure the page does not expose internal endpoints or configuration.
- **Privacy & Compliance** — VPAT 2.5 template format as published by the Information Technology Industry Council (ITI). Section 508: 36 CFR Part 1194. EN 301 549 v3.2.1. The VPAT document itself MUST meet WCAG 2.1 AA (fonts, color contrast, alt text).
- **Accessibility** — The VPAT HTML page MUST itself conform to WCAG 2.1 AA.
- **Reliability** — VPAT hosted on CDN or static hosting; not dependent on application server uptime.
- **Maintainability** — VPAT source maintained as Markdown or DOCX template in the repo at `docs/vpat/`; rendered to HTML at build time.
- **Internationalization** — English only per ITI template standard; no i18n required.
- **Backward compatibility** — Old VPAT versions archived at versioned URLs (e.g., `/accessibility/vpat/2025-10`).

## 7. Acceptance Criteria

- **AC-1.** *Given* a procurement officer navigates to `/accessibility/vpat`, *when* the page loads without authentication, *then* the current VPAT is displayed in HTML with a download link for Word and PDF versions.
- **AC-2.** *Given* the VPAT covers WCAG 2.1 SC 1.1.1 (Non-text Content), *when* the audit found the TipTap image insertion lacks required alt text enforcement, *then* the VPAT entry reads "Partially Supports" with a specific remark referencing the gap and the remediation timeline.
- **AC-3.** *Given* a major release ships, *when* the release checklist is completed, *then* a GitHub issue is automatically created from the VPAT update template, assigned to the accessibility specialist.
- **AC-4.** *Given* the VPAT is reviewed with a screen reader, *when* the user navigates the conformance table, *then* all table headers and cells are announced correctly and the document is navigable by heading structure.
- **AC-5.** *Given* a user clicks the "Contact Accessibility Support" link, *when* the link is activated, *then* they are routed to the accessibility support request form with pre-populated subject "Accessibility accommodation request."

## 8. Data Model

No database changes required. VPAT is a static document with versioned file storage in `docs/vpat/`.

```
docs/vpat/
  VPAT_2.5_INT_Lextures_YYYY-MM.docx
  VPAT_2.5_INT_Lextures_YYYY-MM.pdf
  VPAT_2.5_INT_Lextures_YYYY-MM.md   # source
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| VPAT is inaccurate / over-claims conformance | M | H | Base every entry strictly on audit evidence; err toward "Partially Supports" |
| Audit findings change between VPAT publication and procurement review | M | M | Publish version date prominently; note remediation timeline in remarks |
| VPAT format version changes (ITI issues VPAT 3.0) | L | L | Template is modular; update on ITI release |

## 15. Rollout Plan
