## 1. Problem Statement

The Lextures server uses the `tracing` crate for structured logging. Current log output is unfiltered: request bodies, SQL query parameters, user IDs, emails, and potentially student response text are logged verbatim. This means that any engineer or ops tool with access to log output (terminal, log aggregation, observability platform) can read student personal information without any access control. GDPR Article 5(1)(f) (integrity and confidentiality) and FERPA's incidental disclosure prohibition require that PII is not exposed through operational logging. SOC 2 TSC CC6.1 requires that personal information not be disclosed to unauthorized parties, which includes ops staff who have log access but no need to read student data.

## 2. Goals

- Implement a `tracing` subscriber layer that redacts configurable PII fields before log events are emitted to any sink.
- Define a PII field registry (email, name, DOB, IP address, student response text, JWT token values).
- Ensure that structured log fields containing PII are replaced with a placeholder or hash.
- Retain sufficient diagnostic information (request ID, error code, HTTP status, user ID hash) for troubleshooting.
- Document the redaction configuration for compliance evidence.

## 3. Non-Goals

- Redacting PII from the `admin_audit_log` (10.11) — audit logs are intentionally retained with before/after values under access control; redaction does not apply there.
- Real-time PII scanning of free-text log messages (NLP-based) — pattern-based redaction of structured fields is sufficient for v1.
- SIEM integration — log pipeline is out of scope; redaction happens before any sink.

## 4. Personas & User Stories

- **As an ops engineer**, I want log output to be useful for debugging without exposing student PII so that I can troubleshoot issues without privacy risk.
- **As a GDPR DPO**, I want to confirm that operational logs do not contain personal data so that I can include log systems in our data inventory without DSAR obligations.
- **As a SOC 2 auditor**, I want evidence that PII is not visible in log aggregation tools accessible to engineering staff so that I can satisfy CC6.1.

## 5. Functional Requirements

- **FR-1.** The system MUST implement a `tracing_subscriber` layer that intercepts all log events and redacts values for registered PII field names before forwarding to any downstream sink.
- **FR-2.** The PII field registry MUST include at minimum: `email`, `user_email`, `parent_email`, `first_name`, `last_name`, `full_name`, `date_of_birth`, `ip_address` (logged as subnet-masked hash), `jwt_token`, `access_token`, `student_response`, `assignment_text`.
- **FR-3.** Redacted values MUST be replaced with the pattern `[REDACTED:{field_name}]` or a stable HMAC-SHA256 hash of the value (configurable per field) so that correlation is possible without exposure.
- **FR-4.** The redaction layer MUST be configurable: fields added to the registry via `REDACT_FIELDS` environment variable or `config.rs` constant; `DISABLE_PII_REDACTION=1` MUST be available for local development only (and MUST be blocked in production environments).
- **FR-5.** HTTP access logs MUST NOT log full request URIs that contain user-identifying path segments (e.g., `/users/{uuid}` MUST be logged as `/users/[REDACTED:user_id]`).
- **FR-6.** SQL query logs (if enabled) MUST NOT log bound parameter values; log only the parameterized query shape.
- **FR-7.** The system SHOULD log a startup warning if PII redaction is disabled and the environment is not `local`.

## 6. Non-Functional Requirements

- **Performance** — Redaction layer overhead MUST be < 100 µs per log event (string pattern matching on structured field names only, not regex scan of message text).
- **Security** — `DISABLE_PII_REDACTION` MUST be blocked by the configuration validator (`server/src/config.rs`) when `APP_ENV = 'production'` or `'staging'`. JWT tokens appearing in logs (e.g., error traces) MUST always be redacted regardless of configuration. See `server/src/config.rs` and `server/src/http_auth.rs`.
- **Privacy & Compliance** — GDPR Article 5(1)(f) (integrity and confidentiality); GDPR Article 25 (data protection by design); FERPA incidental disclosure prohibition; SOC 2 TSC CC6.1 (logical access controls); ISO/IEC 27001:2022 Annex A.8.15 (logging); NIST SP 800-53 AU-3 (Content of Audit Records — appropriate level of detail without PII overexposure), SI-12 (Information Management and Retention).
- **Accessibility** — N/A (backend concern).
- **Scalability** — Redaction layer is synchronous in the logging hot path; stateless, no external calls.
- **Reliability** — If the redaction layer panics, the log event MUST be dropped (not logged in plaintext) as a fail-safe; application continues.
- **Observability** — Counter `pii_redactions_total{field_name}` so we can see which fields are most frequently redacted and verify the layer is active. Cross-link: plan 17.7.
- **Maintainability** — Redaction logic in `server/src/logging/pii_redactor.rs`; field registry in `server/src/logging/pii_fields.rs`.
- **Internationalization** — N/A.
- **Backward compatibility** — Existing log consumers (grepping for `email=`) will see `[REDACTED:email]`; update any log-parsing scripts. Document in ops runbook.

## 7. Acceptance Criteria

- **AC-1.** *Given* a user logs in and the server processes the request, *when* the log output is inspected, *then* no email address, JWT token, or user name appears in plaintext in any log line for that request.
- **AC-2.** *Given* a SQL query includes a bound parameter containing a user email, *when* the query is logged at DEBUG level, *then* the log shows the parameterized query shape only (e.g., `WHERE email = $1`) with no parameter value.
- **AC-3.** *Given* `DISABLE_PII_REDACTION=1` is set in a production environment (`APP_ENV=production`), *when* the server starts, *then* startup fails with a configuration error: "PII redaction cannot be disabled in production."
- **AC-4.** *Given* the redaction layer is active, *when* 10,000 log events are processed, *then* the overhead per event is < 100 µs measured via benchmark.
- **AC-5.** *Given* the `pii_redactions_total` metric is queried, *when* it is checked after a test run that processes 100 requests containing emails, *then* the counter for the `email` label is > 0, confirming the layer is active.
- **AC-6.** *Given* a new PII field (`ssn`) is added to the registry, *when* a log event with `ssn` field is emitted, *then* the value appears as `[REDACTED:ssn]` without any code changes beyond updating the registry.

## 8. Data Model

No database changes. PII field registry is a compile-time or config-time constant.

## 9. API Surface

No new API endpoints. `GET /api/v1/internal/ops/redaction-status` (super-admin) returns active redaction field list and `pii_redactions_total` metric.

## 10. UI / UX

No user-facing UI. Ops-only: redaction status visible in compliance dashboard.

## 11. AI / ML Considerations

The PII redaction proxy for AI prompts (19.11) is a separate, higher-stakes concern — it redacts PII before sending to external models. This plan (10.14) covers logging only. The two systems share the `pii_fields` registry.

## 12. Integration Points

- `server/src/logging/pii_redactor.rs` — new `tracing_subscriber::Layer` implementation.
- `server/src/logging/pii_fields.rs` — field registry.
- `server/src/config.rs` — `DISABLE_PII_REDACTION` config key with production guard.
- `server/src/main.rs` — install redactor layer in subscriber stack.
- `server/src/http_auth.rs` — JWT token logging paths must be covered.
- Prometheus metrics exporter (17.7) — `pii_redactions_total`.

## 13. Dependencies & Sequencing

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| False negatives: new PII fields added in future without registering | H | M | Document in PR template: "Did you add any new user-facing fields? Update PII registry." |
| Over-redaction hides diagnostic information needed for debugging | M | M | Use stable hash for user_id (not full redaction); preserve error codes and request IDs |
| Local dev loses visibility into email values for debugging | M | L | `DISABLE_PII_REDACTION=1` allowed in `local` env; document in README |

## 15. Rollout Plan

- No feature flag needed — ship immediately with production guard.
- Phase 1: Core redactor layer + registry; deploy to staging; validate no plaintext PII in logs.
- Phase 2: Add AI prompt log redaction (shared registry with 19.11).
- Phase 3: Document in compliance evidence pack for SOC 2.

## 16. Test Plan

- **Unit** — Redactor layer: field present in structured event → redacted; field absent → pass-through; unknown field → pass-through.
- **Integration** — Login endpoint: post-request logs contain no email or JWT token in plaintext.
- **Performance** — Benchmark: 100k log events through redactor < 10 ms total.
- **Security** — Attempt to set `DISABLE_PII_REDACTION=1` in production config → startup failure.

## 17. Documentation & Training

- Developer guide: "PII redaction and how to add new fields to the registry."
- Ops runbook: "Log access policy — what is and is not redacted."
- PR template update: PII registry checklist item.

## 18. Open Questions

1. Should user IDs be redacted or logged as stable HMAC hashes for correlation? (Hashing recommended — allows tracing a session without exposing UUID that could be correlated to real identity.)
2. Should IP addresses be fully redacted or subnet-masked (e.g., last octet zeroed) for geographic diagnostics?
3. Should the redaction layer also cover `tracing::span` fields, or only `tracing::event` fields?
