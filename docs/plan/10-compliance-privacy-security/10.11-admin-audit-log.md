## 1. Problem Statement

The existing `user_audit` table (introduced in `server/migrations/`) records navigation events (page views, item opens) but does not capture administrative actions: role assignments, grade overrides, data exports, enrollment changes, or content deletions. FERPA (34 CFR § 99.32) requires a disclosure log for every access to education records. SOC 2 TSC CC7.2 requires logging of privileged-user actions. ISO 27001:2022 Annex A.8.15 requires logging of user activities, exceptions, and information security events. Grade-change audit trails are required by many K-12 district policies and HE academic integrity policies. Without a comprehensive admin audit log, every compliance and security certification is blocked or deficient.

## 2. Goals

- Extend the existing audit infrastructure to capture all admin-action event categories: role changes, grade overrides, data exports, bulk operations, enrollment changes, AI-routing configuration, and security settings.
- Make the audit log tamper-evident (append-only, optionally hash-chained).
- Provide a searchable audit log viewer for compliance admins.
- Expose the audit log as evidence for SOC 2, ISO 27001, FERPA, GDPR, and CCPA compliance reviews.
- Cross-link with plan 17.7 (Observability) for metric/alert integration.

## 3. Non-Goals

- Real-time stream processing of audit events (Kafka/Kinesis) — v1 is synchronous write to Postgres with background archival.
- Student-activity analytics (time-on-task, engagement) — covered by plan 9 (Analytics).
- Full SIEM integration (Splunk/Elastic) — export capability is sufficient for v1; SIEM push is v2.

## 4. Personas & User Stories

- **As a compliance officer**, I want to search all admin actions for a specific user over a 90-day period so that I can respond to a FERPA or GDPR investigation.
- **As a university CISO**, I want to verify that no unauthorized grade changes were made so that I can certify academic integrity.
- **As an auditor (SOC 2 / ISO 27001)**, I want to see evidence that privileged-user actions are logged and retained for 12 months so that I can close the AU control family.
- **As a K-12 district admin**, I want to review who changed a student's grade and when so that I can respond to a parent inquiry.

## 5. Functional Requirements

- **FR-1.** The system MUST log the following admin-action event categories: `role_grant`, `role_revoke`, `grade_override`, `grade_bulk_import`, `data_export` (FERPA package, DSAR, gradebook CSV), `enrollment_create`, `enrollment_delete`, `course_publish`, `course_delete`, `ai_config_change`, `security_config_change`, `user_impersonation`, `password_reset_admin`, `content_delete`.
- **FR-2.** Each audit log entry MUST contain: `event_id` (UUID), `event_type`, `actor_id`, `actor_ip`, `target_type` (user/course/enrollment/etc.), `target_id`, `org_id`, `before_value` (JSONB), `after_value` (JSONB), `timestamp` (TIMESTAMPTZ with microsecond precision), `user_agent`.
- **FR-3.** The audit log MUST be append-only; `UPDATE` and `DELETE` on `admin_audit_log` MUST be blocked by a Postgres `BEFORE` trigger.
- **FR-4.** The system SHOULD implement optional hash chaining: each row stores `SHA-256(prev_hash || event_id || timestamp || actor_id || event_type)` for tamper detection.
- **FR-5.** The system MUST retain audit log entries for a minimum of 12 months and SHOULD retain for 36 months.
- **FR-6.** The system MUST provide a compliance admin UI for searching and filtering audit log entries by: actor, event type, target, date range, org.
- **FR-7.** The system MUST support CSV and JSON export of audit log query results for auditors.
- **FR-8.** The audit log MUST be queryable within 2 seconds for any date-range query spanning up to 90 days.

## 6. Non-Functional Requirements

- **Performance** — Audit log write MUST add < 5 ms to any admin action (synchronous Postgres insert). Queries over the last 90 days MUST return in < 2 seconds with proper indexing.
- **Security** — Append-only trigger prevents tampering; only a superuser Postgres role (not the application role) can drop the trigger. Admin audit log access requires `audit:read` permission (separate from `admin` permission). See `server/src/authz.rs`.
- **Privacy & Compliance** — FERPA 34 CFR § 99.32 (record of disclosures); SOC 2 TSC CC7.2 (monitoring of privileged users); ISO/IEC 27001:2022 Annex A.8.15 (logging); GDPR Article 30 (records of processing activities — data export events provide RoPA evidence); NIST SP 800-53 AU-2 (Event Logging), AU-3 (Content of Audit Records), AU-9 (Protection of Audit Information), AU-11 (Audit Record Retention). `before_value` and `after_value` MUST be treated as PII-sensitive; PII redaction applies to log fields per 10.14 but the audit log itself is an exception to PII deletion (retained for compliance).
- **Accessibility** — Audit log viewer UI MUST meet WCAG 2.1 AA (10.7).
- **Scalability** — Partitioned by month (`PARTITION BY RANGE (timestamp)`) to support multi-year retention without table scan degradation. Index: `(org_id, timestamp DESC)`, `(actor_id, timestamp DESC)`, `(event_type, timestamp DESC)`.
- **Reliability** — Audit log write failure MUST propagate as an error to the calling handler (not silently swallowed); the triggering admin action MUST be rolled back if the audit write fails to prevent unlogged changes.
- **Observability** — Prometheus counter `admin_audit_events_total{event_type}`. Alert on any `grade_override` event > 100 in 1 hour (anomaly detection). Cross-link: plan 17.7.
- **Maintainability** — Centralized `server/src/services/audit.rs` with a typed `AuditEvent` enum; route handlers call `audit::record(event).await?`.
- **Internationalization** — Event type strings are system-defined constants (English only); UI labels use i18n keys (11.1).
- **Backward compatibility** — Existing `user_audit` table retained for navigation events; `admin_audit_log` is a new, separate table.

## 7. Acceptance Criteria

- **AC-1.** *Given* an admin grants a user the `instructor` role, *when* the role change is committed, *then* an `admin_audit_log` row with `event_type = 'role_grant'`, `before_value = null`, `after_value = {"role": "instructor"}`, and the actor's ID and IP is created within the same transaction.
- **AC-2.** *Given* an audit log row is written, *when* a database superuser attempts to UPDATE or DELETE the row, *then* the trigger raises an exception and the modification is rejected.
- **AC-3.** *Given* a compliance admin queries the audit log for all grade overrides for a specific student over 30 days, *when* the query runs, *then* results are returned in < 2 seconds and include `before_value` (old grade) and `after_value` (new grade) for each event.
- **AC-4.** *Given* a compliance admin exports 90 days of audit log events for a specific org as CSV, *when* the export is downloaded, *then* all required fields are present, no entries are missing, and the file is available within 30 seconds.
- **AC-5.** *Given* the hash-chain option is enabled, *when* an auditor computes the hash of a row, *then* the computed hash matches the stored `chain_hash` for every row in the sequence, confirming no tampering.
- **AC-6.** *Given* an admin audit write fails due to a database error, *when* the failure propagates up, *then* the originating admin action (e.g., role grant) is rolled back and the client receives an error response.

## 8. Data Model

```sql
-- Migration: 081_admin_audit_log.sql

CREATE TABLE IF NOT EXISTS compliance.admin_audit_log (
  event_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id        UUID REFERENCES org.organizations(id),
  event_type    TEXT NOT NULL,
  actor_id      UUID NOT NULL REFERENCES user.users(id),
  actor_ip      INET,
  user_agent    TEXT,
  target_type   TEXT,
  target_id     UUID,
  before_value  JSONB,
  after_value   JSONB,
  chain_hash    TEXT,   -- optional SHA-256 hash chain
  "timestamp"   TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE ("timestamp");

-- Create initial monthly partitions
CREATE TABLE compliance.admin_audit_log_2026_04
  PARTITION OF compliance.admin_audit_log
  FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');

-- Append-only trigger
CREATE OR REPLACE FUNCTION compliance.prevent_audit_log_modification()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  RAISE EXCEPTION 'admin_audit_log is append-only; modifications are not permitted.';
END;
$$;

CREATE TRIGGER audit_log_immutable
  BEFORE UPDATE OR DELETE ON compliance.admin_audit_log
  FOR EACH ROW EXECUTE FUNCTION compliance.prevent_audit_log_modification();

-- Indexes
CREATE INDEX idx_admin_audit_log_org_ts   ON compliance.admin_audit_log(org_id, "timestamp" DESC);
CREATE INDEX idx_admin_audit_log_actor_ts ON compliance.admin_audit_log(actor_id, "timestamp" DESC);
CREATE INDEX idx_admin_audit_log_event    ON compliance.admin_audit_log(event_type, "timestamp" DESC);
```

Backfill: no backfill of historic navigation events needed; log starts from migration deployment.

## 9. API Surface

| Method | Path | Auth Scope | Notes |
|--------|------|-----------|-------|
| `GET` | `/api/v1/compliance/audit-log` | `audit:read` | Query with filters: actor, event_type, target_id, from, to, org_id |
| `GET` | `/api/v1/compliance/audit-log/export` | `audit:read` | CSV/JSON export; streamed response |
| `GET` | `/api/v1/compliance/audit-log/:event_id` | `audit:read` | Single event detail |

All endpoints require `audit:read` permission, separate from general `admin` permission, to limit audit access to designated compliance staff.

## 10. UI / UX

- **Audit log viewer** (compliance admin) — searchable, filterable table; event type badges; before/after diff viewer on row expand.
- **Export button** — date range picker + format selector (CSV/JSON); async download with progress indicator for large exports.
- **Grade override history** on gradebook — inline audit trail for individual grade cells (links to audit log viewer).
- WCAG 2.1 AA; keyboard-navigable table; i18n keys prefixed `audit_log.*`.

## 11. AI / ML Considerations

`ai_config_change` events MUST be logged when AI routing configuration is modified (e.g., model selection, prompt template changes). This provides evidence for SOC 2 CC8 (change management) and for the AI usage disclosure (10.17).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Partition management not automated; old partitions fill disk | M | H | Automated partition creation job (17.4); archive partitions > 36 months to cold storage |
| before_value / after_value capture too much PII in the audit log itself | M | M | PII redaction config (10.14) applied to JSONB values before storage; SSN, credit card stripped |
| Hash chain verification is CPU-intensive for large tables | L | M | Hash chain is optional; only enabled when explicitly required by a compliance program |

## 15. Rollout Plan

- Feature flag: `admin_audit_log_enabled` — default `true` (ship immediately; no rollback needed for append-only log).
- Phase 1: schema migration + core event types (role changes, grade overrides).
- Phase 2: remaining event types (exports, deletions, AI config changes).
- Phase 3: admin UI viewer + export.
- GA: all tenants.

## 16. Test Plan

- **Unit** — `audit::record()` correctly serializes before/after values; append-only trigger rejects UPDATE.
- **Integration** — Role grant → audit row created in same transaction; rollback on write failure.
- **End-to-end** — Playwright: admin grants role → audit viewer shows event; grade override → audit trail on gradebook.
- **Security** — `audit:read` required; non-admins cannot query log; trigger tested.
- **Performance** — Audit write < 5 ms added latency; 90-day query < 2 s.
