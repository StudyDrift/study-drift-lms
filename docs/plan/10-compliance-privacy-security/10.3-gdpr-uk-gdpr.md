## 1. Problem Statement

The General Data Protection Regulation (GDPR, EU 2016/679) and the UK GDPR (UK GDPR / Data Protection Act 2018) impose legal obligations on any processor of EU/UK residents' personal data regardless of where the operator is based. Lextures currently has no lawful basis records, no consent management, no Data Subject Access Request (DSAR) tooling, no right-to-erasure implementation, and no Data Processing Agreement (DPA) template. Any international HE institution or EU-based self-learner using the platform creates immediate regulatory exposure. Maximum GDPR fines reach €20 million or 4 % of global annual turnover under Article 83(5).

## 2. Goals

- Implement a complete DSAR (right of access, Article 15) export workflow with 30-day response SLA.
- Implement the right to erasure (Article 17) with configurable retention exceptions.
- Build a consent management module recording lawful basis, consent versions, and withdrawal.
- Produce a signed Data Processing Agreement (DPA) artifact for institutional customers.
- Maintain a Records of Processing Activities (Article 30) register in-system.
- Support the right to data portability (Article 20) with machine-readable export.

## 3. Non-Goals

- CCPA/CPRA compliance (see 10.4) — separate workflow, some shared infrastructure.
- Cookie consent banner implementation (front-end only, handled in a separate UI ticket).
- Cross-border transfer adequacy mechanisms (SCCs, binding corporate rules) — legal document review only, not an engineering deliverable.

## 4. Personas & User Stories

- **As an EU/UK data subject**, I want to request a copy of all personal data held about me so that I can verify what Lextures has collected and exercise my GDPR rights.
- **As an EU/UK data subject**, I want to request erasure of my account and associated data so that my personal information is deleted when I leave the platform.
- **As a data subject**, I want to withdraw consent for AI processing of my content at any time so that my work is no longer sent to LLMs.
- **As a data protection officer (DPO)**, I want an in-system Records of Processing Activities (RoPA) so that I can demonstrate Article 30 compliance to regulators.
- **As a university DPO**, I want a pre-signed DPA template so that procurement sign-off takes days, not months.

## 5. Functional Requirements

- **FR-1.** The system MUST allow any authenticated user to submit a DSAR; the system MUST compile and deliver a portable data export (JSON + CSV) within 30 calendar days (GDPR Article 15 & 12(3)).
- **FR-2.** The system MUST support right-to-erasure requests (GDPR Article 17); upon approval, personal data MUST be deleted or anonymised, except where retention is required by law (e.g., financial records, FERPA audit logs); every retention exception MUST be documented.
- **FR-3.** The system MUST record lawful basis for each processing activity in the RoPA (Article 30); supported bases: contract, legitimate interests, legal obligation, vital interests, consent.
- **FR-4.** The system MUST store granular consent records per user per processing purpose (Article 6(1)(a) & 7); each record MUST include consent version, timestamp, IP hash, and withdrawal timestamp if applicable.
- **FR-5.** The system MUST provide users a consent management centre where they can view active consents and withdraw any consent individually.
- **FR-6.** The system MUST support data portability (Article 20): DSAR export MUST include all user-provided data in a machine-readable format (JSON).
- **FR-7.** The system SHOULD generate a pre-populated DPA template (Article 28) referencing sub-processors, processing purposes, and technical safeguards, downloadable by institutional admins.
- **FR-8.** The system MUST log all data-subject requests and their outcomes in the admin audit log (10.11) with response timestamps and actioning admin.
- **FR-9.** The system MUST notify the DPO (or designated compliance contact) when a DSAR or erasure request is received and when the 30-day deadline is within 5 days of expiry.

## 6. Non-Functional Requirements

- **Performance** — DSAR data compilation MUST complete within 120 seconds for up to 3 years of activity; larger datasets MUST use background jobs with async delivery.
- **Security** — DSAR archives MUST be AES-256 encrypted; download links MUST be time-limited (72 hours) and single-use. Step-up authentication required before submitting erasure requests. See `server/src/http_auth.rs`, `server/src/authz.rs`.
- **Privacy & Compliance** — Full conformance with GDPR (EU) 2016/679: Articles 5 (data minimisation), 6 (lawful basis), 7 (consent conditions), 12 (transparent information), 13/14 (privacy notice), 15 (right of access), 17 (erasure), 20 (portability), 25 (data protection by design), 30 (RoPA), 32 (security of processing), 33 (breach notification 72h), 83 (penalties). UK GDPR (DPA 2018) equivalent obligations. OWASP ASVS 4.0 §8 (Data Protection). NIST 800-53: IP-1 through IP-8 (Individual Participation), SI-12 (Information Management and Retention).
- **Accessibility** — Consent centre and DSAR request form MUST meet WCAG 2.1 AA (see 10.7).
- **Scalability** — RoPA and consent records must handle 500 processing activities and 1M consent records per tenant.
- **Reliability** — DSAR jobs must be idempotent; requeuing on failure must not generate duplicate exports.
- **Observability** — Metrics: `dsar_requests_pending`, `dsar_requests_overdue` (> 30 days), `consent_withdrawals_24h`. Alert when `dsar_requests_overdue > 0`.
- **Maintainability** — GDPR logic in `server/src/services/gdpr.rs`; consent middleware in `server/src/middleware/consent_check.rs`.
- **Internationalization** — Consent notices and DSAR correspondence MUST support all EU/UK official languages via i18n framework (11.1); v1 ships EN, DE, FR.
- **Backward compatibility** — Existing user records treated as `lawful_basis = 'contract'` by default; no retroactive consent prompts for existing institutional users.

## 7. Acceptance Criteria

- **AC-1.** *Given* an authenticated EU user submits a DSAR, *when* the request is processed, *then* a downloadable JSON archive containing all personal data (profile, grades, submissions, AI interaction logs) is delivered within 30 days, and a `dsar_requests` row is created with `completed_at` timestamp.
- **AC-2.** *Given* a user requests erasure of their account, *when* the erasure is approved, *then* all personal data columns are nulled or replaced with opaque hashes within 72 hours; exempted data (financial records, audit logs) retains a tombstone row with erasure timestamp; the user receives a written confirmation.
- **AC-3.** *Given* a user withdraws consent for AI processing, *when* withdrawal is recorded, *then* all subsequent AI feature calls for that user return a "consent withdrawn" error without transmitting any user content to the model, and the withdrawal timestamp is stored in `gdpr_consents`.
- **AC-4.** *Given* a DPO requests the RoPA export, *when* the export is generated, *then* all processing activities appear with lawful basis, data categories, retention periods, and sub-processor names.
- **AC-5.** *Given* the 30-day DSAR deadline is 5 days away and the request is still pending, *when* the system runs its daily SLA check, *then* an escalation email is sent to the compliance contact and an alert fires in the observability stack.
- **AC-6.** *Given* an institutional admin downloads the DPA template, *when* the template is rendered, *then* it is pre-populated with Lextures entity name, sub-processor list, and current privacy policy URL, ready for countersignature.

## 8. Data Model

```sql
-- Migration: 075_gdpr.sql

CREATE TABLE IF NOT EXISTS compliance.gdpr_consents (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES user.users(id) ON DELETE CASCADE,
  purpose         TEXT NOT NULL,   -- e.g. 'ai_processing', 'marketing', 'analytics'
  lawful_basis    TEXT NOT NULL CHECK (lawful_basis IN ('consent','contract','legal_obligation','vital_interests','legitimate_interests')),
  consent_version TEXT NOT NULL,
  granted_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  withdrawn_at    TIMESTAMPTZ,
  ip_hash         TEXT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE IF NOT EXISTS compliance.dsar_requests (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          UUID REFERENCES org.organizations(id),
  user_id         UUID NOT NULL REFERENCES user.users(id),
  request_type    TEXT NOT NULL CHECK (request_type IN ('access','erasure','portability','rectification','restriction','objection')),
  status          TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','in_progress','completed','rejected')),
  archive_path    TEXT,
  rejection_reason TEXT,
  requested_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  due_at          TIMESTAMPTZ GENERATED ALWAYS AS (requested_at + INTERVAL '30 days') STORED,
  completed_at    TIMESTAMPTZ,
  actioned_by     UUID REFERENCES user.users(id)
);

CREATE TABLE IF NOT EXISTS compliance.ropa_entries (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id           UUID NOT NULL REFERENCES org.organizations(id),
  activity_name    TEXT NOT NULL,
  purpose          TEXT NOT NULL,
  lawful_basis     TEXT NOT NULL,
  data_categories  TEXT[] NOT NULL,
  data_subjects    TEXT[] NOT NULL,
  retention_period TEXT NOT NULL,
  sub_processors   TEXT[],
  updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_dsar_requests_due ON compliance.dsar_requests(due_at) WHERE status = 'pending';
CREATE INDEX idx_gdpr_consents_user ON compliance.gdpr_consents(user_id, purpose, withdrawn_at NULLS FIRST);
```

- **Privacy Centre page** (`/privacy-centre`) — consolidated hub: consent list with withdraw buttons, submit DSAR, track request status.
- **Consent management widget** — per-purpose toggles in account settings.
- **DSAR request form** — request type selector, reason field, identity verification notice.

- User flow: Settings → Privacy Centre → "Download my data" → form → "We'll email you within 30 days."
- Error states: "Your request is already in progress. Please wait for the current one to complete."
- WCAG 2.1 AA on all new pages; ARIA live regions for status updates; i18n keys prefixed `gdpr.*`.

## 11. AI / ML Considerations

The `gdpr_consents` table drives consent checks in the AI route middleware. When `purpose = 'ai_processing'` has no active grant (or `withdrawn_at IS NOT NULL`), all AI feature routes MUST return HTTP 403 before any prompt is assembled. The PII redaction proxy (19.11) must also anonymize data in DSAR archives before archiving AI interaction logs.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| 30-day SLA breach triggers regulatory complaint | M | H | Automated escalation alerts; SLA dashboard for compliance team |
| Erasure removes data needed for FERPA audit log | M | H | Maintain tombstone rows; document retention exceptions in RoPA |
| Consent withdrawal mid-session breaks AI features abruptly | L | M | Graceful degradation message; batch recalculation at session boundary |
| DPA template not accepted by specific institution's legal team | M | L | Offer editable DOCX template; note legal review requirement |
