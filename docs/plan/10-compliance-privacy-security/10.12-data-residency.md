## 1. Problem Statement

Multiple regulatory regimes require that student or citizen data remain within defined geographic boundaries: GDPR Article 44 restricts transfers of EU personal data outside the EEA without an adequacy decision or transfer mechanism; some US states (e.g., California EdTech mandates, NY Ed Law 2-d) impose in-state or in-country storage requirements for student data; Canadian PIPEDA and Australian Privacy Act impose similar constraints for their jurisdictions. Lextures currently deploys to a single DigitalOcean droplet with no regional configuration in `iac/`. Every EU university and several US district prospects will require a data residency commitment before signing a contract.

## 2. Goals

- Implement per-tenant `data_residency_region` configuration that constrains where the tenant's data is stored and processed.
- Deploy multi-region infrastructure (at minimum: `us-east`, `eu-west`) backed by region-scoped Postgres instances and object storage buckets.
- Ensure that application-level routing sends tenant requests to the correct regional endpoint.
- Prevent cross-region data leakage (no EU tenant data written to US databases or vice versa).
- Document the regional deployment architecture in the privacy policy and DPA.

## 3. Non-Goals

- Multi-region active-active replication (high-availability) — this is plan 17.12. Data residency and HA are separate concerns; v1 requires data to reside in a region, not that the region is HA.
- Per-field data residency (different data types in different regions) — tenant-level granularity only.
- Sovereign cloud / government cloud (FedRAMP, IL5) — future program.

## 4. Personas & User Stories

- **As an EU university CIO**, I want a contractual and technical guarantee that student data is processed and stored within the EU so that we can comply with GDPR Article 44.
- **As a NY state district admin**, I want to confirm that student data is stored on US servers so that we meet state contract requirements.
- **As a Lextures infrastructure engineer**, I want tenant region pinning enforced at the application layer so that a misconfiguration cannot route EU tenant data to US infrastructure.
- **As a compliance officer**, I want to audit which region each tenant's data is stored in so that I can respond to regulatory inquiries.

## 5. Functional Requirements

- **FR-1.** The system MUST support a `data_residency_region` field on the `org.organizations` table with values: `us-east`, `eu-west`, `au-east`, `ca-central` (v1 ships `us-east` and `eu-west`).
- **FR-2.** The system MUST route all database reads and writes for a tenant to the Postgres instance in the tenant's designated region; cross-region database access MUST be prohibited at the application level.
- **FR-3.** The system MUST store all object storage artifacts (file uploads, DSAR archives, backups) for a tenant in an S3/R2-compatible bucket scoped to the tenant's designated region.
- **FR-4.** The system MUST reject any attempt to change `data_residency_region` after initial provisioning without a migration workflow (see FR-6).
- **FR-5.** The system MUST log every cross-region data-access attempt (which MUST be blocked) to `compliance.admin_audit_log`.
- **FR-6.** The system SHOULD support a tenant region migration workflow (freeze tenant → export data → import to new region → verify → unfreeze) for cases where a tenant needs to move regions.
- **FR-7.** The system MUST display the tenant's data residency region in the admin console and in the DPA document.

## 6. Non-Functional Requirements

- **Performance** — Regional routing adds < 10 ms overhead per request (connection pool per region, selected at request middleware level).
- **Security** — `data_residency_region` is immutable after provisioning (trigger-enforced); only a super-admin Postgres role can override. AI provider calls (OpenRouter) MUST be reviewed: if the EU region is selected, the AI sub-processor contract MUST include EU Standard Contractual Clauses (GDPR Art. 46).
- **Privacy & Compliance** — GDPR Article 44 (transfers to third countries); GDPR Article 46 (transfer mechanisms); GDPR Recital 101 (adequacy decisions); NY Ed Law 2-d § 2-d(2)(c) (data stored in US); PIPEDA (Canada): Principle 4.1.3 (accountability for data transferred to third parties). ISO/IEC 27001:2022 Annex A.5.33 (protection of records) and A.8.10 (information deletion — region-scoped deletion on erasure). NIST 800-53: SC-28 (Protection of Information at Rest), SC-36 (Distributed Processing and Storage).
- **Scalability** — Connection pool manager supports N regional pools; tenant lookup cached in Redis (< 1 ms).
- **Reliability** — Region selection cached; regional Postgres failover handled by managed DB provider (DigitalOcean Managed Postgres, AWS RDS, etc.). See 17.12 for multi-region HA.
- **Observability** — Metric `requests_by_region_total{region}`; alert on any request served from wrong region. Cross-link: plan 17.7.
- **Maintainability** — Region routing logic in `server/src/db/region_pool.rs`; `iac/` per-region Terraform modules in `iac/production/region-us-east/` and `iac/production/region-eu-west/`.
- **Internationalization** — Region labels displayed in UI using i18n keys (11.1) and locale-specific names.
- **Backward compatibility** — Existing tenants assigned `data_residency_region = 'us-east'` as the default during migration.

## 7. Acceptance Criteria

- **AC-1.** *Given* a tenant is provisioned with `data_residency_region = 'eu-west'`, *when* any API request is processed for that tenant, *then* all database queries go to the EU Postgres instance and all file operations go to the EU object storage bucket.
- **AC-2.** *Given* a request arrives at the US-east application server for an EU-region tenant, *when* the tenant lookup resolves the region, *then* the request is forwarded to the EU application server (or the DB connection pool routes to the EU database) without any data being written to the US database.
- **AC-3.** *Given* an admin attempts to change `data_residency_region` via the API after provisioning, *when* the request is processed, *then* the system returns HTTP 422 with message "Region cannot be changed after provisioning without a migration workflow."
- **AC-4.** *Given* the tenant region migration workflow is initiated, *when* the migration completes, *then* all tenant data is present in the new region, the old region data is securely deleted, and the `data_residency_region` field is updated.
- **AC-5.** *Given* the compliance admin queries the audit log, *when* any cross-region access attempt occurred, *then* the event appears with `event_type = 'cross_region_access_blocked'` and the offending request details.

## 8. Data Model

```sql
-- Migration: 082_data_residency.sql

ALTER TABLE org.organizations
  ADD COLUMN data_residency_region TEXT NOT NULL DEFAULT 'us-east'
    CHECK (data_residency_region IN ('us-east','eu-west','au-east','ca-central'));

CREATE OR REPLACE FUNCTION org.prevent_region_change()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  IF OLD.data_residency_region IS DISTINCT FROM NEW.data_residency_region THEN
    RAISE EXCEPTION 'data_residency_region cannot be changed after provisioning without a migration workflow';
  END IF;
  RETURN NEW;
END;
$$;

CREATE TRIGGER org_region_immutable
  BEFORE UPDATE ON org.organizations
  FOR EACH ROW EXECUTE FUNCTION org.prevent_region_change();
```

## 11. AI / ML Considerations

For EU-region tenants, all AI inference calls (OpenRouter) MUST route through an EU endpoint if available, or be covered by GDPR Art. 46 SCCs in the sub-processor agreement. The AI usage disclosure (10.17) must disclose the geographic routing of AI calls.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Regional Postgres latency higher than expected | M | M | Use managed regional DB with read replicas; connection pool tuning |
| AI sub-processor contract does not include EU SCCs | H | H | Review OpenRouter DPA before EU launch; fallback to EU-based model provider |
| Region migration leaves orphaned data | M | H | Verified deletion step in migration workflow; audit log evidence |

## 15. Rollout Plan

- Feature flag: `data_residency_enabled` — default `false`.
- Phase 1: Schema migration; all existing tenants tagged `us-east`.
- Phase 2: EU Postgres + object storage provisioned in `iac/production/region-eu-west/`.
- Phase 3: Regional routing middleware; region pool manager.
- Phase 4: EU app server deployment; end-to-end test with EU tenant.
- GA: enable for EU HE prospects on request.

## 16. Test Plan
