## 1. Problem Statement

Lextures routes student work (assignment text, quiz responses, notebook queries) through OpenRouter to large language models for AI-powered features (adaptive quiz generation, rubric generation, AI grading, RAG notebook). Students and parents have no visibility into which AI models process their data, no opt-out control, and no model-card disclosure. This is an increasingly common RFP question from K-12 districts ("Do you use AI to process student data? Which vendors?") and is required for FERPA compliance (school official disclosure), COPPA (parental consent for AI processing), and emerging state AI-in-education laws. Additionally, no PII redaction proxy sits between user content and the OpenRouter API call.

## 2. Goals

- Implement a per-user and per-tenant AI opt-out control that prevents student data from being sent to any external AI model.
- Publish a machine-readable AI model card / disclosure document listing all models used, their providers, data retention policies, and opt-out paths.
- Implement a per-tenant admin control allowing district/institution admins to restrict which AI features are available and which models are approved.
- Display contextual AI disclosure banners at the point of AI-feature use.
- Log all AI inference calls with metadata (model used, feature, user ID hash, content hash) for audit purposes.

## 3. Non-Goals

- Building a self-hosted LLM inference stack — the disclosure and control layer sits on top of the existing OpenRouter integration.
- Full PII redaction proxy implementation (see 19.11) — this plan implements the disclosure and opt-out controls; 19.11 implements the actual redaction.
- AI content provenance / digital watermarking (see 19.9).

## 4. Personas & User Stories

- **As a student**, I want to see which AI model will process my assignment before I submit it so that I can make an informed choice.
- **As a K-12 student's parent**, I want to opt my child out of AI processing so that their schoolwork is not sent to commercial AI providers.
- **As a district IT admin**, I want to restrict AI features to a pre-approved list of models so that we can meet our board's AI governance requirements.
- **As a compliance officer**, I want a log of every AI inference call (with model ID and content hash) so that I can respond to a FERPA disclosure request asking "was AI used in grading this student?"
- **As a self-learner**, I want to understand how my notebook queries are processed by AI so that I can decide whether to use that feature.

## 5. Functional Requirements

- **FR-1.** The system MUST implement a per-user `ai_processing_opt_out` flag; when set, no user content (assignment text, quiz responses, notebook queries) MUST be sent to any external AI provider.
- **FR-2.** The system MUST display a contextual AI disclosure notice on first use of each AI-powered feature, identifying: the AI provider (OpenRouter + underlying model), the data sent (content type description), the provider's data retention policy, and a link to opt out.
- **FR-3.** The system MUST publish a public AI Disclosure page listing all AI models used in the platform, their providers, sub-processor DPA status, and data retention commitments.
- **FR-4.** The system MUST implement per-tenant AI governance controls: org admins MUST be able to enable/disable individual AI features and restrict allowed model providers for their tenant.
- **FR-5.** The system MUST log every AI inference call to `compliance.ai_inference_log` with fields: `feature_name`, `model_id`, `provider`, `user_id_hash`, `content_hash` (SHA-256 of redacted input), `timestamp`, `org_id`, `opt_in_confirmed`.
- **FR-6.** The system MUST respect COPPA parental AI opt-in (10.2): `coppa_minor` accounts MUST default to `ai_processing_opt_out = true` unless the parent has explicitly enabled AI features.
- **FR-7.** The system MUST respect GDPR consent: when `gdpr_consents` (10.3) does not contain an active grant for `purpose = 'ai_processing'`, AI calls MUST be blocked for that user.
- **FR-8.** The system SHOULD provide a "Why AI was used" explanation link next to any AI-generated content (quiz question, rubric, feedback) identifying the model and feature.

## 6. Non-Functional Requirements

- **Performance** — Opt-out check MUST be cached (Redis or in-memory); < 1 ms overhead per AI route middleware check.
- **Security** — AI inference log entries are append-only (same trigger pattern as admin audit log). Per-tenant AI config changes are logged to `admin_audit_log` (10.11) as `ai_config_change` events. See `server/src/authz.rs`.
- **Privacy & Compliance** — FERPA 34 CFR § 99.31(a)(1) (school official with LEI — AI provider is a sub-processor in this role); COPPA 16 CFR § 312.7 (confidentiality and security of AI-processed data); GDPR Article 13(1)(e) (recipients of personal data including AI processors); GDPR Article 22 (automated decision-making — AI grading triggers this if it produces legal/significant effects); CCPA § 1798.140 (sale/sharing — AI processing may constitute sharing); FERPA prohibits sharing student records with AI providers unless under the "school official" exception with a DPA and LEI. NIST AI RMF (2023) — Map, Measure, Manage, Govern functions relevant to per-tenant AI governance. OWASP LLM Top 10 2025 — LLM02 (Insecure Output Handling), LLM06 (Sensitive Information Disclosure).
- **Accessibility** — Opt-out controls and disclosure pages MUST meet WCAG 2.1 AA (10.7).
- **Scalability** — AI inference log is a high-write table; partitioned by month (same pattern as `admin_audit_log`).
- **Reliability** — If opt-out check service is unavailable, AI calls MUST be blocked by default (fail closed).
- **Observability** — Metrics: `ai_inference_calls_total{feature, model, opted_out}`, `ai_opt_outs_total`. Alert if `opted_out` rate spikes > 20% in 1 hour (possible compliance incident). Cross-link: plan 17.7.
- **Maintainability** — AI gateway middleware in `server/src/middleware/ai_gateway.rs`; disclosure page content in `docs/ai-disclosure/`.
- **Internationalization** — AI disclosure banners and opt-out controls MUST use i18n keys (11.1); disclosure page available in EN, ES, FR for v1.
- **Backward compatibility** — `ai_processing_opt_out` defaults to `false` for non-COPPA users; `coppa_minor` users default to `true`.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student sets `ai_processing_opt_out = true`, *when* any AI-powered feature route is called for that student, *then* the middleware blocks the call with HTTP 403 ("AI processing is disabled for this account") before any prompt is assembled and no content is sent to OpenRouter.
- **AC-2.** *Given* a student uses the AI quiz generation feature for the first time, *when* the feature is invoked, *then* a disclosure banner is shown identifying the model (e.g., "Claude 3.5 Sonnet via OpenRouter"), the data being sent, and a link to the opt-out settings, before the feature executes.
- **AC-3.** *Given* a district admin disables the "AI grading" feature for their tenant, *when* an instructor in that tenant attempts to use AI grading, *then* the feature is hidden and the API returns a tenant-policy-disabled error.
- **AC-4.** *Given* an AI inference call is made, *when* the call completes (or is blocked), *then* a row is written to `compliance.ai_inference_log` with the correct feature name, model ID, user ID hash, and content hash.
- **AC-5.** *Given* a FERPA disclosure request is processed for a student, *when* the DSAR archive is assembled, *then* the AI inference log is queried for all calls involving that student's user ID hash, and a summary is included in the disclosure package.
- **AC-6.** *Given* a COPPA minor account has no parental AI opt-in, *when* the AI gateway middleware checks the account, *then* the call is blocked regardless of the user's own `ai_processing_opt_out` setting.

## 8. Data Model

```sql
-- Migration: 086_ai_disclosure.sql

ALTER TABLE user.users
  ADD COLUMN ai_processing_opt_out BOOLEAN NOT NULL DEFAULT FALSE;

CREATE TABLE IF NOT EXISTS compliance.ai_inference_log (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          UUID REFERENCES org.organizations(id),
  user_id_hash    TEXT NOT NULL,   -- HMAC-SHA256 of user UUID
  feature_name    TEXT NOT NULL,   -- e.g. 'quiz_generation', 'rubric_gen', 'rag_notebook'
  model_id        TEXT NOT NULL,   -- e.g. 'anthropic/claude-3.5-sonnet'
  provider        TEXT NOT NULL,   -- e.g. 'openrouter'
  content_hash    TEXT NOT NULL,   -- SHA-256 of redacted prompt
  opt_in_confirmed BOOLEAN NOT NULL,
  blocked         BOOLEAN NOT NULL DEFAULT FALSE,
  "timestamp"     TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE ("timestamp");

CREATE TABLE IF NOT EXISTS compliance.tenant_ai_config (
  org_id          UUID PRIMARY KEY REFERENCES org.organizations(id),
  features_enabled JSONB NOT NULL DEFAULT '{}',  -- {"quiz_generation": true, "ai_grading": false}
  allowed_models  TEXT[],                         -- NULL = all; list = restricted
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_by      UUID REFERENCES user.users(id)
);

CREATE INDEX idx_ai_inference_log_user ON compliance.ai_inference_log(user_id_hash, "timestamp" DESC);
```

## 11. AI / ML Considerations

This plan IS the AI governance control framework. The AI gateway middleware (`server/src/middleware/ai_gateway.rs`) intercepts all AI feature routes, checks opt-out + consent + COPPA + tenant config, logs to `ai_inference_log`, and passes through or blocks. The PII redaction proxy (19.11) runs as a second middleware layer that sanitizes the assembled prompt before it leaves the system.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| OpenRouter's model list changes; disclosure page becomes stale | H | M | Automate model list from OpenRouter API; weekly refresh job |
| Gateway middleware adds latency to all AI routes | M | M | Cache opt-out status in Redis; < 1 ms per check |
| Opt-out blocks AI features a student needs; friction complaint | M | L | Clear explanation; instructor can note which features require AI; fallback non-AI mode |
| GDPR Art. 22 challenge to AI grading as automated decision-making | M | H | Implement "instructor approval required" gate before AI grade is recorded (see 19.3) |

## 15. Rollout Plan

- Feature flag: `ai_disclosure_enabled` — default `true` (ship immediately; disclosure is always beneficial).
- Phase 1: Opt-out flag + gateway middleware (block calls for opted-out users).
- Phase 2: First-use disclosure banners.
- Phase 3: Public AI disclosure page + tenant governance UI.
- Phase 4: AI inference log + DSAR integration.

## 16. Test Plan

- **Unit** — Gateway middleware: opt-out flag blocks call; COPPA minor blocks call; GDPR no-consent blocks call.
- **Integration** — End-to-end AI call with opt-out: no request sent to OpenRouter; inference log records `blocked = true`.
- **End-to-end** — Playwright: user sets opt-out → attempts quiz generation → blocked with friendly message.
- **Security** — Fail-closed: if gateway service throws, AI call is blocked (not allowed through).
- **Accessibility** — axe-core on disclosure banner and opt-out settings page.
