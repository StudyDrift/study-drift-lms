## 1. Problem Statement

Beyond federal law, three US states with the largest K-12 markets impose additional student-data privacy requirements that go further than FERPA and COPPA. California's SOPIPA (SB 1177, Education Code § 49073.1), New York's Education Law § 2-d and Part 121 regulations, and Illinois' SOPPA (105 ILCS 85/) each mandate specific disclosures, prohibitions, and parent-notification mechanisms. Lextures lacks any policy documentation, parent-disclosure feed, or compliance controls for these laws. Districts in California, New York, and Illinois collectively represent the majority of US K-12 procurement spending; non-compliance is a contract-disqualifying deficiency.

## 2. Goals

- Implement the additional technical controls required by CA SOPIPA, NY Ed Law 2-d, and IL SOPPA.
- Provide per-state parent disclosure feeds showing what student data the platform has used or shared.
- Publish the required prohibition attestations (no targeted advertising, no sale of student data).
- Generate state-specific DPA addenda for districts in these three states.
- Create the compliance documentation artifacts required by each state's law.

## 3. Non-Goals

- Compliance with states not enumerated here (TX, FL, etc.) — covered by a future v2 sweep.
- Legal drafting of final contract language — this plan delivers the engineering scaffold; counsel finalizes wording.
- FERPA/COPPA base controls (see 10.1, 10.2) — assumed shipped.

## 4. Personas & User Stories

- **As a California parent**, I want to see a disclosure of every operator or sub-contractor that has accessed my child's data, as required by SOPIPA § 49073.1(b)(7).
- **As a New York parent**, I want to receive an annual notice describing the student data the platform holds and how it is used, as required by Ed Law 2-d § 2-d(5)(a).
- **As an Illinois parent**, I want the ability to submit a data-deletion request and have it fulfilled within 30 school days, as required by SOPPA 105 ILCS 85/25.
- **As a district data privacy officer in any of these three states**, I want a state-specific DPA addendum pre-populated and ready for signature so that procurement is streamlined.

## 5. Functional Requirements

- **FR-1.** The system MUST refuse to use student data for targeted advertising, building advertising profiles, or selling student data for any purpose — and MUST display an attestation to this effect in the privacy policy and DPA (required by SOPIPA, SOPPA, and NY Ed Law 2-d).
- **FR-2.** The system MUST provide a parent-accessible disclosure page listing all sub-processors and school-official access events for their child's data for the current and prior school year (SOPIPA § 49073.1(b)(7)).
- **FR-3.** The system MUST send an annual notice to parents/guardians in NY state districts summarizing the categories of student data held and their uses (NY Ed Law 2-d § 2-d(5)(a) & Part 121.3).
- **FR-4.** For IL SOPPA districts, the system MUST support parent data-deletion requests fulfillable within 30 school days (105 ILCS 85/25(c)).
- **FR-5.** The system MUST provide state-specific DPA addenda for CA, NY, and IL that incorporate the additional requirements of each state's law into the base NDPA/SDPC agreement (see 10.5).
- **FR-6.** The system MUST tag each tenant's state when configured, and apply the appropriate state-specific restrictions and disclosure obligations automatically.
- **FR-7.** The system SHOULD provide a compliance checklist dashboard for district admins showing outstanding state-law obligations and completion status.

## 6. Non-Functional Requirements

- **Performance** — Parent disclosure page MUST render within 2 seconds for 1 year of access events.
- **Security** — Parent disclosure page gated behind parent/guardian role authentication (5.10). State tag on tenant is admin-only.
- **Privacy & Compliance** — CA SOPIPA (Cal. Ed. Code § 49073.1 et seq.); NY Ed Law 2-d (§ 2-d(1)–(7)) and Part 121 regulations (8 NYCRR Part 121); IL SOPPA (105 ILCS 85/ §§ 5–30). All require prohibition on sale and targeted advertising; all require incident notification. NY additionally requires appointment of a Chief Privacy Officer; IL additionally requires a data security program consistent with NIST CSF. NIST 800-53 controls: AT-2 (Security Training), IR-6 (Incident Reporting), PL-8 (Privacy Architecture).
- **Accessibility** — Parent disclosure pages MUST meet WCAG 2.1 AA (10.7).
- **Scalability** — Annual notice job must handle batches of 50k+ parents without timeout.
- **Reliability** — Annual NY notice emails sent via background job queue (17.3) with retry.
- **Observability** — Log all parent disclosure page views and deletion requests to `admin_audit_log` (10.11).
- **Maintainability** — State-specific logic isolated in `server/src/services/state_privacy.rs`; state configurations in a JSON/YAML config file.
- **Internationalization** — NY requires notices in languages spoken by ≥ 5% of district students; English + Spanish + Chinese v1.
- **Backward compatibility** — State tag defaults to `null` (no state-specific restrictions); activating a state tag is opt-in per tenant.

## 7. Acceptance Criteria

- **AC-1.** *Given* a tenant is tagged as `state = 'CA'`, *when* a sub-processor request for student data is made, *then* the event is logged in the disclosure log and appears on the parent's disclosure page within 24 hours.
- **AC-2.** *Given* a New York district tenant is active and the annual notice job runs, *when* the job executes on the configured date, *then* every parent/guardian email in the district receives the required Ed Law 2-d annual notice.
- **AC-3.** *Given* an Illinois parent submits a data-deletion request, *when* the request is processed, *then* the student's data is deleted within 30 school days and the parent receives a written confirmation per SOPPA § 25(c).
- **AC-4.** *Given* a district admin in any of the three states downloads the state-specific DPA addendum, *when* the document is rendered, *then* it references the correct state statute and includes state-required exhibit content.
- **AC-5.** *Given* the compliance checklist is viewed by a NY district admin, *when* the CPO appointment is not recorded, *then* the checklist shows an outstanding obligation with a link to the NY CPO registration guidance.

## 8. Data Model

```sql
-- Migration: 078_state_privacy.sql

ALTER TABLE org.organizations
  ADD COLUMN state_privacy_jurisdiction TEXT;  -- 'CA','NY','IL', NULL

CREATE TABLE IF NOT EXISTS compliance.state_disclosure_events (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id       UUID NOT NULL REFERENCES org.organizations(id),
  student_id   UUID NOT NULL REFERENCES user.users(id),
  accessor     TEXT NOT NULL,   -- sub-processor name or user role
  purpose      TEXT NOT NULL,
  data_elements TEXT[] NOT NULL,
  occurred_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE IF NOT EXISTS compliance.annual_notice_jobs (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id       UUID NOT NULL REFERENCES org.organizations(id),
  jurisdiction TEXT NOT NULL,
  sent_at      TIMESTAMPTZ,
  year         INTEGER NOT NULL,
  UNIQUE(org_id, jurisdiction, year)
);

CREATE INDEX idx_state_disclosure_student ON compliance.state_disclosure_events(student_id, occurred_at DESC);
```

- **Parent disclosure page** — chronological list of data-access events for child with accessor name, purpose, date.
- **Compliance checklist** — card per state obligation; status: met / outstanding / not applicable.

## 11. AI / ML Considerations

All AI sub-processor access events (OpenRouter model calls with student content) MUST be captured as `state_disclosure_events` when the tenant's state requires it. The PII redaction proxy (19.11) and AI usage disclosure (10.17) controls are prerequisites.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| State statutes amended (e.g., NY Part 121 revisions) | M | M | Annual legal review; modular state config |
| Annual notice email volume triggers spam filters | M | M | SPF/DKIM; transactional email provider; unsubscribe not applicable for statutory notice |
| IL 30-school-day vs calendar-day ambiguity | L | M | Build school-day calendar into deletion SLA calculation; default to 30 calendar days as safe fallback |
