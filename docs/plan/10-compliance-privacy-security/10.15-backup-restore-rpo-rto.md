## 1. Problem Statement

Lextures has no documented backup policy, no automated database snapshots configured in `iac/`, and no tested restore procedure. For any educational institution, loss of grade data, submission history, or course content due to hardware failure, ransomware, or operator error is a catastrophic event with FERPA, COPPA, and contractual implications. SOC 2 Availability TSC (A1.2) and ISO 27001 Annex A.5.30 (ICT readiness for business continuity) require defined and tested recovery procedures. Without this, Lextures cannot credibly sell to any institutional customer that asks the standard "what is your RPO/RTO?" procurement question. Cross-link: plan 17.11 (Backup automation & restore drills).

## 2. Goals

- Define and document Recovery Point Objective (RPO) and Recovery Time Objective (RTO) targets for each data tier.
- Implement automated daily full + continuous WAL streaming backups for Postgres.
- Implement automated daily backups of object storage (user files).
- Implement and test a restore runbook with pass/fail criteria.
- Schedule quarterly restore drills and log results.
- Codify all backup configuration in `iac/production/`.

## 3. Non-Goals

- Multi-region active-active replication (plan 17.12).
- Application-level point-in-time recovery per tenant (v2).
- Backup of Redis / in-memory caches — these are ephemeral and do not require backup.

## 4. Personas & User Stories

- **As a K-12 district IT director**, I want to know Lextures' RPO and RTO so that I can include it in our vendor risk assessment.
- **As an ops engineer**, I want automated backups that I can restore from on demand so that I can recover from accidental data deletion.
- **As a SOC 2 auditor**, I want evidence of backup execution and restore testing so that I can close the Availability TSC control A1.2.
- **As a university CIO**, I want Lextures to have a tested Business Continuity Plan so that we can rely on the service during a critical exam period.

## 5. Functional Requirements

- **FR-1.** The system MUST achieve an RPO of ≤ 1 hour for the Postgres database (achieved via continuous WAL streaming to object storage) and RPO of ≤ 24 hours for object storage (daily snapshot).
- **FR-2.** The system MUST achieve an RTO of ≤ 4 hours for a full Postgres restore from the most recent backup in the same region.
- **FR-3.** The system MUST run automated Postgres backups using WAL-G (or cloud-managed equivalent): continuous WAL archiving to an encrypted S3/R2 bucket, plus a daily base backup.
- **FR-4.** The system MUST configure object storage versioning and cross-region replication (or at minimum, daily snapshot to a separate bucket) for all user-uploaded files.
- **FR-5.** All backup data MUST be encrypted at rest using AES-256 (see 10.13); encryption keys stored in secrets manager.
- **FR-6.** The system MUST retain daily backups for 30 days, weekly backups for 13 weeks, and monthly backups for 12 months.
- **FR-7.** The system MUST run a quarterly automated restore drill: spin up an isolated environment, restore from the most recent backup, run smoke tests, record pass/fail in `compliance.restore_drills`.
- **FR-8.** The system MUST alert if any scheduled backup job fails to complete successfully within its window.

## 6. Non-Functional Requirements

- **Performance** — WAL streaming MUST have < 5-minute lag under normal load. Daily base backup MUST complete within the maintenance window (off-peak hours, typically 02:00–04:00 UTC).
- **Security** — Backup bucket access MUST be restricted to the backup IAM role; no application-runtime credentials can read or delete backup data. Backup bucket MUST have Object Lock (WORM) enabled for 30 days to prevent ransomware deletion. See `docs/SECURITY.md` for threat model.
- **Privacy & Compliance** — SOC 2 Availability TSC A1.2 (environmental protections and recovery planning); ISO/IEC 27001:2022 Annex A.5.29 (information security during disruption), A.5.30 (ICT readiness for business continuity), A.8.10 (information deletion — backup retention and deletion); FERPA incidental disclosure — backup restores must respect FERPA access controls; GDPR Article 32 (appropriate technical measures). NIST SP 800-53: CP-9 (Information System Backup), CP-10 (Information System Recovery and Reconstitution).
- **Scalability** — WAL streaming and base backup scale with managed DB service; object storage backup scales with provider.
- **Reliability** — Backup jobs scheduled via cron (17.4); failures alert on-call; retried once before alerting.
- **Observability** — Metrics: `backup_last_success_timestamp{tier}`, `backup_duration_seconds{tier}`, `wal_lag_seconds`. Alert if `backup_last_success_timestamp` is > 25 hours old (daily backup missed). Cross-link: plan 17.7.
- **Maintainability** — Backup configuration in `iac/production/backup.tf`; WAL-G config in `server/backup/walg.env`.
- **Internationalization** — N/A.
- **Backward compatibility** — Backup is additive; no existing data affected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a daily backup job is scheduled, *when* the job runs, *then* a new base backup appears in the backup bucket within the maintenance window and the `backup_last_success_timestamp{tier='postgres'}` metric is updated.
- **AC-2.** *Given* continuous WAL streaming is active, *when* a commit is made to the database, *then* the WAL segment is archived to the backup bucket within 5 minutes (verified by `wal_lag_seconds` metric).
- **AC-3.** *Given* a quarterly restore drill is scheduled, *when* the drill runs, *then* a fresh Postgres instance is restored from the most recent base backup + WAL segments, the smoke test suite passes, and the result is recorded in `compliance.restore_drills` with `pass = true`.
- **AC-4.** *Given* a backup job fails, *when* the failure is detected, *then* an alert fires in the observability stack and an on-call notification is sent within 15 minutes.
- **AC-5.** *Given* the backup bucket has Object Lock enabled, *when* an attempt is made to delete a backup object within the 30-day retention period, *then* the deletion is rejected by the storage provider.
- **AC-6.** *Given* a full disaster recovery scenario (simulate total DB loss), *when* the restore runbook is followed, *then* the database is restored to within 1 hour of the incident time (RPO ≤ 1h) and the service is operational within 4 hours (RTO ≤ 4h).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| WAL streaming falls behind during peak load | M | M | Monitor `wal_lag_seconds`; alert at 15-min lag; scale WAL archival bandwidth |
| Restore drill environment inconsistency produces false pass | M | M | Automated smoke test suite with coverage of grade reads, quiz attempts, auth |
| Backup bucket credentials compromised | L | H | Object Lock (WORM) prevents deletion; MFA delete on bucket; IAM role separation |
| Ransomware encrypts backup bucket | L | H | Object Lock + versioning + cross-region copy |

## 15. Rollout Plan

- Phase 1: WAL-G configuration + daily backup job; deploy to production.
- Phase 2: Object storage versioning + cross-bucket copy.
- Phase 3: First restore drill; record in `restore_drills`.
- Phase 4: Quarterly drill automation; SOC 2 evidence package.

## 16. Test Plan

- **Integration** — Daily backup job runs in CI against test Postgres; WAL segment archived correctly.
- **Restore drill** — Full restore from backup in staging environment; smoke tests pass.
- **Security** — Object Lock prevents deletion within retention period.
- **Observability** — Backup alert fires when job fails.
