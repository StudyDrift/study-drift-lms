## 1. Problem Statement

The Family Educational Rights and Privacy Act (FERPA), 20 U.S.C. § 1232g and 34 CFR Part 99, requires institutions that receive federal funding to restrict access to student education records, obtain written consent before disclosure, and provide parents (or eligible students 18+) the right to inspect and amend records. Lextures has no directory-information opt-out flag, no "legitimate educational interest" (LEI) gating on record access, and no parent-record-access workflow. Any K-12 district or Title IV-participating university that evaluates the platform will immediately reject it on this basis, blocking the two most commercially significant market segments.

## 2. Goals

- Implement a FERPA-compliant data-access role model distinguishing "school official with LEI" from general administrative access.
- Provide a directory-information opt-out flag per student (34 CFR § 99.37).
- Build a parent/eligible-student record-access request workflow with audit trail.
- Produce a per-tenant FERPA disclosure document template.
- Ensure every FERPA-regulated data access and disclosure is recorded in the admin audit log (see 10.11).

## 3. Non-Goals

- Full SIS integration (see 4.3 OneRoster); this plan covers policy enforcement within Lextures only.
- COPPA parental consent mechanics (see 10.2).
- Grade-passback to external registrar systems (see 14.5).

## 4. Personas & User Stories

- **As a district compliance officer**, I want to configure which data fields are treated as directory information so that we control what is shareable without consent.
- **As a parent of a K-12 student**, I want to request a copy of my child's education records and receive them within the FERPA-mandated 45-day window.
- **As an instructor**, I want to know whether I have legitimate educational interest before accessing a student's full record so that I stay compliant.
- **As an eligible student (18+)**, I want to submit an amendment request for an inaccurate record so that my official transcript is correct.
- **As a compliance admin**, I want to see a complete audit log of all record disclosures so that I can respond to Department of Education inquiries.

## 5. Functional Requirements

- **FR-1.** The system MUST maintain a per-student `ferpa_directory_opt_out` flag (34 CFR § 99.37); when set, the student's name, email, enrollment status, and any other configured directory fields MUST be withheld from any non-essential disclosure API endpoint.
- **FR-2.** The system MUST enforce a "legitimate educational interest" (LEI) check before any school official or admin can access a student's detailed education record; LEI is satisfied by holding a role with the `records:read` permission scoped to the relevant org/course.
- **FR-3.** The system MUST provide a record-access request form allowing a parent (of a minor) or eligible student (age ≥ 18) to request inspection of their education records; the system MUST generate a response package within the 45-calendar-day statutory window.
- **FR-4.** The system MUST support a record amendment request workflow, with the institution able to agree, deny, or initiate a formal hearing; each step MUST be logged.
- **FR-5.** The system MUST log every education-record disclosure (who accessed what data, under what authority) to the `admin_audit_log` table introduced by plan 10.11.
- **FR-6.** The system SHOULD provide a per-tenant configurable list of "directory information" fields, with the default list restricted to the minimal set required by 34 CFR § 99.3.
- **FR-7.** The system MUST support "consent for disclosure" records: when a student or parent grants consent for a third-party disclosure, a signed consent record (timestamp, purpose, recipient) MUST be stored.
- **FR-8.** The system SHOULD send email notification to the requester when their record-access package is ready for download.

## 6. Non-Functional Requirements

- **Performance** — Record-access package generation (compiling grades, submissions, activity log for one student) MUST complete within 60 seconds for a student with up to 5 years of history; for larger histories a background job MUST be used with async notification.
- **Security** — All record download endpoints MUST require re-authentication (step-up auth or fresh JWT challenge) before serving PII-containing archives. Access to `ferpa_directory_opt_out` update endpoint MUST be restricted to `records:admin` permission. See `server/src/http_auth.rs` and `server/src/authz.rs`.
- **Privacy & Compliance** — Full conformance with FERPA 20 U.S.C. § 1232g and 34 CFR Part 99, including: §99.10 (right to inspect), §99.20 (amendment), §99.30 (written consent for disclosure), §99.31 (exceptions), §99.32 (disclosure log), §99.34 (disclosure to school officials), §99.37 (directory information). The NIST 800-53 control families AC-3 (Access Enforcement), AU-2 (Event Logging), and IP-1 (Consent) MUST be addressed. Generated record packages MUST be encrypted at rest (AES-256) until downloaded. Records MUST NOT be sent to external AI/LLM endpoints without explicit consent (see 10.17).
- **Accessibility** — All new UI pages (opt-out toggle, request forms) MUST conform to WCAG 2.1 AA (see 10.7).
- **Scalability** — Must support tens of thousands of students per tenant; background job queue (17.3) MUST be used for bulk record exports.
- **Reliability** — Record-access packages MUST be idempotent: re-requesting a package for the same student within 24 hours returns the cached version.
- **Observability** — All FERPA disclosure events MUST emit a structured log with fields: `user_id`, `target_student_id`, `data_type`, `authority_claim`, `disclosure_recipient`, `timestamp`. A Prometheus counter `ferpa_disclosures_total` MUST be exported (see 17.7).
- **Maintainability** — FERPA enforcement logic MUST live in a dedicated `server/src/services/ferpa.rs` module, not scattered across route handlers.
- **Internationalization** — All UI strings MUST be externalized into i18n keys (see 11.1).
- **Backward compatibility** — The `ferpa_directory_opt_out` column migration MUST default to `false` (opt-in to directory sharing) to avoid retroactively hiding existing data for non-FERPA tenants.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student has set `ferpa_directory_opt_out = true`, *when* an unauthenticated or non-LEI request is made to any directory-information endpoint, *then* the student's directory fields are excluded from the response and a `ferpa_suppressed: true` flag appears in the API metadata.
- **AC-2.** *Given* a parent submits a record-access request, *when* the compliance admin reviews and approves it, *then* a downloadable encrypted archive containing the student's grades, activity log, and submitted work is available within the 45-day window and the event is recorded in `admin_audit_log`.
- **AC-3.** *Given* an instructor with no LEI role tries to access a student's detailed record via a direct API call, *when* the request reaches the FERPA middleware, *then* the server returns HTTP 403 and logs the denied access attempt.
- **AC-4.** *Given* a student (age ≥ 18) submits a record amendment request for an incorrect grade, *when* the institution denies the amendment, *then* the student receives a denial notice with instructions to request a formal hearing, and both steps are stored in `ferpa_amendment_requests`.
- **AC-5.** *Given* a tenant enables third-party disclosure to a scholarship body with student consent, *when* the consent record is created, *then* the disclosure log entry contains the consent timestamp, recipient identity, and purpose string.
- **AC-6.** *Given* the system runs a FERPA compliance audit report, *when* the report covers a 30-day window, *then* every disclosure is listed with authority claim, no disclosure appears without a matching consent or LEI record, and the report exports as a PDF or CSV.

## 8. Data Model

New tables in `server/migrations/`:

```sql
-- Migration: 073_ferpa.sql

ALTER TABLE user.users
  ADD COLUMN ferpa_directory_opt_out BOOLEAN NOT NULL DEFAULT FALSE,
  ADD COLUMN ferpa_eligible_student   BOOLEAN NOT NULL DEFAULT FALSE; -- true when age >= 18

CREATE TABLE IF NOT EXISTS compliance.ferpa_record_requests (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          UUID NOT NULL REFERENCES org.organizations(id),
  student_id      UUID NOT NULL REFERENCES user.users(id),
  requester_id    UUID NOT NULL REFERENCES user.users(id), -- parent or eligible student
  request_type    TEXT NOT NULL CHECK (request_type IN ('inspect','amend','hearing')),
  status          TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','approved','denied','completed')),
  amendment_field TEXT,
  amendment_value TEXT,
  notes           TEXT,
  archive_path    TEXT,          -- encrypted S3/R2 key
  requested_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  due_at          TIMESTAMPTZ,   -- requested_at + 45 days
  completed_at    TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE IF NOT EXISTS compliance.ferpa_consent_records (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          UUID NOT NULL REFERENCES org.organizations(id),
  student_id      UUID NOT NULL REFERENCES user.users(id),
  granted_by      UUID NOT NULL REFERENCES user.users(id),
  recipient       TEXT NOT NULL,
  purpose         TEXT NOT NULL,
  data_fields     TEXT[] NOT NULL,
  consented_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  expires_at      TIMESTAMPTZ,
  revoked_at      TIMESTAMPTZ
);

CREATE TABLE IF NOT EXISTS compliance.ferpa_disclosure_log (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          UUID NOT NULL REFERENCES org.organizations(id),
  accessor_id     UUID NOT NULL REFERENCES user.users(id),
  student_id      UUID NOT NULL REFERENCES user.users(id),
  data_type       TEXT NOT NULL,
  authority_claim TEXT NOT NULL, -- 'lei', 'consent', 'school_official', 'legal_exception'
  recipient       TEXT,
  logged_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_ferpa_disclosure_log_student ON compliance.ferpa_disclosure_log(student_id, logged_at DESC);
CREATE INDEX idx_ferpa_record_requests_student ON compliance.ferpa_record_requests(student_id, status);
```

Backfill: `ferpa_directory_opt_out` defaults to `false`; no backfill required.

## 9. API Surface

| Method | Path | Auth Scope | Notes |
|--------|------|-----------|-------|
| `GET` | `/api/v1/compliance/ferpa/directory-opt-out` | student (self) | Retrieve current flag |
| `PUT` | `/api/v1/compliance/ferpa/directory-opt-out` | student (self) | Toggle flag |
| `POST` | `/api/v1/compliance/ferpa/record-requests` | parent / eligible student | Submit access or amendment request |
| `GET` | `/api/v1/compliance/ferpa/record-requests` | compliance admin | List pending requests |
| `PATCH` | `/api/v1/compliance/ferpa/record-requests/:id` | compliance admin | Approve / deny / complete |
| `GET` | `/api/v1/compliance/ferpa/disclosure-log` | compliance admin | Export disclosure log (CSV/JSON) |
| `POST` | `/api/v1/compliance/ferpa/consent` | student / parent | Grant third-party disclosure consent |
| `DELETE` | `/api/v1/compliance/ferpa/consent/:id` | student / parent | Revoke consent |

All routes require `compliance:ferpa` permission scope. Rate limit: 20 req/min per IP on record-request endpoints.

## 10. UI / UX

- **Student settings page** — new "Privacy & Records" tab with directory opt-out toggle and link to submit record request.
- **Parent dashboard** — record-access request form (depends on 5.10).
- **Compliance admin console** — requests queue with 45-day countdown, approve/deny actions, disclosure log viewer with CSV export.
- **User flows:**
  1. Student toggles directory opt-out → immediate confirmation banner → server-side flag persisted.
  2. Parent submits record-access request → acknowledgment email → admin reviews → download link emailed to parent.
  3. Eligible student submits amendment → institution responds within 45 days → if denied, hearing instructions shown.
- Empty states: "No pending FERPA requests."
- Error states: "You do not have permission to access this student's records."
- All new pages must pass WCAG 2.1 AA; focus order: header → form fields → submit; ARIA roles on status badges.
- i18n keys prefixed `ferpa.*`.

## 11. AI / ML Considerations

Record packages MUST NOT be sent to any AI/LLM model. The PII redaction proxy (10.17 / 19.11) MUST intercept any inadvertent inclusion of FERPA-protected data in AI prompts.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| 45-day deadline SLA is hard to hit without job queue | H | H | Queue jobs immediately; implement SLA countdown and escalation alerts |
| Directory-information opt-out retroactively breaks existing integrations | M | M | Default opt-in; document for tenant admins; provide migration guide |
| Misclassification of "LEI" leads to over-exposure | M | H | Require explicit `records:read` grant per org; log all accesses |
| FERPA scope creep into full SIS replacement | M | M | Strict non-goals; this plan covers Lextures-internal records only |

## 15. Rollout Plan

- Feature flag: `ferpa_workflow_enabled` — default `false`.
- Phase 1 (schema): deploy migration 073 with flag off; backfill `ferpa_eligible_student` via DOB if available.
- Phase 2 (backend): FERPA service + routes behind flag.
