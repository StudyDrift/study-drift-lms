## 1. Problem Statement

`docs/SECURITY.md` contains an internal security audit but there is no `SECURITY.md` at the repository root — the GitHub-standard location for a responsible-disclosure policy. Security researchers who discover vulnerabilities have no defined channel for reporting them, creating legal ambiguity (researchers may fear prosecution under the CFAA) and operational risk (vulnerabilities may be disclosed publicly before Lextures can patch them). SOC 2 CC7.1 (vulnerability management) and ISO 27001 A.8.8 require a process for receiving and addressing vulnerability reports. A published responsible-disclosure policy is also a trust signal in procurement security reviews.

## 2. Goals

- Publish a `SECURITY.md` at the repository root with a responsible-disclosure policy.
- Define a coordinated-disclosure timeline (90-day default, following Google Project Zero / CERT/CC norms).
- Create a `security@lextures.io` contact and a PGP key for encrypted report submission.
- Optionally establish a lightweight bug bounty program via HackerOne Public or a private invite-only program.
- Document the vulnerability-triage and patch process for internal use.

## 3. Non-Goals

- Full penetration testing program — this plan covers the disclosure intake channel only.
- Security incident response (covered under 10.9 SOC 2 incident management).
- Automated vulnerability scanning / DAST (see 17.16 SBOM/Dependabot).

## 4. Personas & User Stories

- **As an independent security researcher**, I want a clear policy telling me how to report a vulnerability so that I can do so responsibly without legal risk.
- **As a Lextures security engineer**, I want a triage process so that I know how to evaluate, prioritize, and acknowledge incoming reports.
- **As a SOC 2 auditor**, I want to see a documented vulnerability-disclosure process with evidence of reports received and remediated so that I can close CC7.1.
- **As a procurement officer at a university**, I want Lextures to have a published security contact so that I can report issues discovered during our own security review.

## 5. Functional Requirements

- **FR-1.** The repository MUST contain a `SECURITY.md` file at the root with: security contact, PGP key, scope definition, out-of-scope activities, safe harbor statement, and coordinated-disclosure timeline.
- **FR-2.** The `SECURITY.md` MUST declare that the vulnerability reporter will not face legal action for good-faith disclosure within the defined scope (safe harbor per ISO/IEC 29147:2018 §6.2).
- **FR-3.** The system MUST maintain a `security@lextures.io` email address monitored by the security team with a maximum 2-business-day initial acknowledgment SLA.
- **FR-4.** The organization MUST maintain an internal vulnerability tracking process: triage → severity classification (CVSS 3.1) → patch SLA (Critical: 7d, High: 30d, Medium: 90d, Low: next-release) → notification to reporter → public disclosure after patch.
- **FR-5.** The system SHOULD establish a private invite-only bug bounty program on HackerOne or Bugcrowd with defined reward ranges for valid critical/high findings.
- **FR-6.** The security contact and policy MUST be linked from: repo root `SECURITY.md`, `/security` page on the product website, and the trust center (20.2).

## 6. Non-Functional Requirements

- **Performance** — N/A (policy document).
- **Security** — PGP key for `security@` MUST be published on keys.openpgp.org. Safe harbor language reviewed by legal counsel.
- **Privacy & Compliance** — ISO/IEC 29147:2018 (vulnerability disclosure) and ISO/IEC 30111:2019 (vulnerability handling processes). SOC 2 CC7.1 (addressing identified vulnerabilities). NIST SP 800-53 SI-5 (Security Alerts, Advisories, and Directives), RA-5 (Vulnerability Monitoring and Scanning).
- **Reliability** — `security@` email alias monitored; auto-responder sends acknowledgment within 1 hour.
- **Observability** — Vulnerability report ticket count tracked in `compliance.security_reports`; patch SLA adherence tracked.
- **Maintainability** — `SECURITY.md` reviewed annually; PGP key rotated every 2 years.

## 7. Acceptance Criteria

- **AC-1.** *Given* a security researcher navigates to the GitHub repository, *when* they look for a vulnerability reporting mechanism, *then* `SECURITY.md` is present at the repo root with all required sections and a security contact email.
- **AC-2.** *Given* a researcher sends an encrypted report to `security@lextures.io`, *when* the email is received, *then* an acknowledgment is sent within 2 business days confirming receipt and providing a ticket reference.
- **AC-3.** *Given* a Critical CVSS 3.1 vulnerability is reported, *when* the triage assigns Critical severity, *then* a patch is deployed within 7 calendar days and the reporter is notified with the patch details.
- **AC-4.** *Given* a SOC 2 auditor requests vulnerability management evidence, *when* the compliance team exports the `compliance.security_reports` table, *then* each report has a severity, triage date, patch date (if patched), and SLA adherence flag.
- **AC-5.** *Given* the responsible-disclosure policy is published, *when* the product website security page is visited, *then* the policy link is present and the page is publicly accessible without authentication.

## 8. Data Model

```sql
-- Migration: 085_security_reports.sql

CREATE TABLE IF NOT EXISTS compliance.security_reports (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  reporter_handle TEXT,
  report_date     DATE NOT NULL,
  cvss_score      NUMERIC(3,1),
  severity        TEXT CHECK (severity IN ('critical','high','medium','low','informational')),
  summary         TEXT NOT NULL,
  status          TEXT NOT NULL DEFAULT 'triaging' CHECK (status IN ('triaging','accepted','patched','disputed','wont_fix')),
  patch_date      DATE,
  sla_met         BOOLEAN,
  bounty_paid     BOOLEAN NOT NULL DEFAULT FALSE,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Flood of low-quality reports | M | L | Clear scope definition; HackerOne triage filter |
| Critical vuln disclosed publicly before patch | L | H | 90-day coordinated disclosure window; expedite critical patches; engage reporter |
| `security@` email not monitored consistently | M | M | PagerDuty/Slack integration; documented on-call rotation |
