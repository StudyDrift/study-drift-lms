## 1. Problem Statement

Section 508 of the Rehabilitation Act (29 U.S.C. § 794d) requires all US federal and state-funded institutions to procure only IT products meeting WCAG 2.1 AA (incorporated by reference via the Revised 508 Standards, 36 CFR Part 1194). The EU equivalent EN 301 549 similarly mandates WCAG 2.1 AA for public-sector procurement. Neither standard has been audited or met by Lextures. The TipTap rich-text editor, the dnd-kit drag-and-drop module reorder, the gradebook grid, and the quiz-taking flow all have unverified keyboard navigation and screen-reader support. Any public-sector procurement — which covers all US K-12 districts and most public universities — will fail without an accessibility conformance statement or VPAT (see 10.8). This is a simultaneous procurement blocker for both target institutional markets.

## 2. Goals

- Establish a systematic WCAG 2.1 AA (and targeted WCAG 2.2 new criteria) conformance program with audits, remediation, and regression prevention.
- Achieve and document AA conformance for the six highest-traffic user journeys: login, course view, quiz-take, module-reorder, gradebook, assignment submission.
- Integrate automated accessibility testing (axe-core / Playwright-axe) into the CI pipeline.
- Implement a component-level accessibility annotation standard for all future UI development.
- Cross-reference with Section 12 (Accessibility) individual feature plans.

## 3. Non-Goals

- AAA conformance — only AA is required for procurement; AAA items are tracked separately.
- Native mobile accessibility audit (iOS/Android VoiceOver/TalkBack) — covered by plan 7.1 when native apps are built.
- WCAG 2.2 full audit — only the 9 new WCAG 2.2 criteria are in scope; 2.1 AA is the primary target.

## 4. Personas & User Stories

- **As a student who uses a screen reader (NVDA/JAWS/VoiceOver)**, I want to navigate and complete quizzes, read course content, and submit assignments without mouse interaction.
- **As a student with a motor disability**, I want to reorder modules, fill out forms, and access the gradebook using keyboard-only navigation.
- **As a student with low vision**, I want all text to meet minimum contrast ratios (4.5:1 for normal text, 3:1 for large text) and all UI elements to be zoomable to 200% without loss of function.
- **As a district accessibility coordinator**, I want a published conformance statement (VPAT) so that I can complete the procurement accessibility review without engaging an external auditor.

## 5. Functional Requirements

- **FR-1.** The system MUST pass WCAG 2.1 AA Success Criteria 1.1–4.1 for all pages included in the audit scope; failures MUST be tracked as P0 or P1 engineering issues.
- **FR-2.** The CI pipeline MUST run axe-core automated checks against every PR touching `clients/web/`; the build MUST fail on any new WCAG 2.1 AA violation introduced by the PR.
- **FR-3.** The TipTap editor MUST expose a fully keyboard-navigable toolbar, support screen-reader announcement of formatting state, and provide an accessible character/word count (WCAG 2.1 SC 4.1.2).
- **FR-4.** The dnd-kit drag-and-drop module reorder interface MUST provide a keyboard alternative allowing reorder via up/down arrow keys with announcements via ARIA live region (WCAG 2.1 SC 2.1.1 + SC 4.1.3).
- **FR-5.** The gradebook grid MUST be navigable as a data table with `role="grid"`, column headers, and row headers; sorting MUST be announced via `aria-sort` (WCAG 2.1 SC 1.3.1).
- **FR-6.** All form error messages MUST be programmatically associated with their inputs via `aria-describedby` (WCAG 2.1 SC 3.3.1).
- **FR-7.** All images MUST have meaningful `alt` text; decorative images MUST have `alt=""` (WCAG 2.1 SC 1.1.1).
- **FR-8.** The system SHOULD achieve compliance with the 9 new WCAG 2.2 criteria including: 2.4.11 (Focus Not Obscured), 2.4.12 (Focus Not Obscured Enhanced), 2.5.7 (Dragging Movements), 2.5.8 (Target Size Minimum 24×24 px).
- **FR-9.** The system MUST maintain a public Accessibility Conformance Statement updated at least annually.

## 6. Non-Functional Requirements

- **Performance** — axe-core CI scan MUST complete in < 60 seconds per page.
- **Security** — No accessibility features introduce XSS vectors (e.g., ARIA label injection from user content must be sanitized).
- **Privacy & Compliance** — Section 508 (29 U.S.C. § 794d; 36 CFR Part 1194 Revised 508 Standards); EN 301 549 v3.2.1; WCAG 2.1 (W3C Recommendation June 2018) Success Criteria 1.1.1–4.1.3; WCAG 2.2 (W3C Recommendation October 2023) new criteria. ADA Title II regulations (28 CFR Part 35) relevant for public-sector HE. OWASP ASVS 4.0 §13 (API and Web Service verification) indirectly relevant for screen-reader trust.
- **Accessibility** — This plan IS the accessibility conformance program; no separate note needed.
- **Scalability** — Accessibility CI gate scales with existing GitHub Actions/CI infrastructure.
- **Reliability** — axe-core checks run on every PR; failures block merge.
- **Observability** — Axe violation counts tracked per component; Storybook accessibility add-on for design-review stage.
- **Maintainability** — Shared `clients/web/src/lib/a11y/` utility for focus management, ARIA announcer, skip-link component.
- **Internationalization** — All ARIA labels and error messages MUST use i18n keys (11.1); RTL layout must maintain logical focus order (11.2).
- **Backward compatibility** — Accessibility fixes are additive (ARIA attributes, semantic HTML); no functional regressions.

## 7. Acceptance Criteria

- **AC-1.** *Given* the CI pipeline runs on any PR touching `clients/web/src/`, *when* an axe-core scan finds a new WCAG 2.1 AA violation, *then* the CI job fails and the PR cannot be merged until the violation is resolved.
- **AC-2.** *Given* a screen-reader user navigates the quiz-take flow, *when* they tab through questions and select answers, *then* all interactive elements have accessible names, focus order is logical, and answer selection is announced by the screen reader.
- **AC-3.** *Given* a keyboard-only user attempts to reorder modules via the drag-and-drop interface, *when* they use the keyboard alternative (e.g., Space to grab + arrow keys to move + Enter to drop), *then* the item moves correctly and an ARIA live announcement confirms the new position.
- **AC-4.** *Given* a user zooms the browser to 200%, *when* they view the gradebook, *then* all content and controls are visible without horizontal scrolling and no content is clipped.
- **AC-5.** *Given* a user submits a form with validation errors, *when* the error state renders, *then* each error message is programmatically associated with its input field and focus is moved to the first error.
- **AC-6.** *Given* the accessibility conformance statement is published, *when* a procurement officer reviews it, *then* it lists each WCAG 2.1 AA success criterion as: Supports / Partially Supports / Does Not Support / Not Applicable, with notes.

## 8. Data Model

No new persistent data model required. Accessibility conformance statement is a static document updated manually and published at `/accessibility`.

## 9. API Surface

No new API endpoints. The conformance statement is a static page. Axe-core results are CI artifacts, not API responses.

## 10. UI / UX

- **Skip navigation link** — visually hidden until focused; appears at top of every page.
- **Focus management** — focus moved to page heading on route change (React Router integration).
- **ARIA announcer** — global live region in `clients/web/src/components/AriaAnnouncer.tsx` for dynamic messages.
- **Color tokens** — all design tokens in `clients/web/src/styles/` validated against WCAG 4.5:1 normal text, 3:1 large text, 3:1 UI component contrast.
- **Keyboard shortcut reference** — accessible modal documenting shortcuts (triggered by `?`).
- **Conformance statement page** (`/accessibility`) — public, updated annually.
- All new components in Storybook MUST have accessibility story with axe check.

## 11. AI / ML Considerations

None. However, the AI-generated alt text feature (12.5) depends on the infrastructure established by this plan (alt text enforcement in the TipTap editor).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| TipTap editor has deep accessibility gaps requiring upstream contributions | H | H | Evaluate Tiptap v2 accessibility roadmap; consider Prosemirror wrapper patches; budget 4w for editor alone |
| axe-core misses manual-only WCAG criteria | H | M | Supplement with manual screen-reader testing (NVDA/Chrome, JAWS/Edge, VoiceOver/Safari) quarterly |
| Design token changes break contrast compliance | M | M | Automated contrast-ratio linting in Storybook and CI |
| Accessibility debt accumulates faster than remediation | M | H | Strict axe gate on new PRs; dedicated 20% sprint capacity for a11y remediation |

## 15. Rollout Plan

- No feature flag needed — accessibility is always enabled.
- Phase 1 (sprint 1–2): axe-core CI integration + skip nav + focus management utilities.
- Phase 2 (sprint 3–6): Systematic audit of 6 priority flows; remediation.
- Phase 3 (sprint 7–8): Color contrast token audit; ARIA grid for gradebook; keyboard drag alternative.
- Phase 4 (sprint 9–10): WCAG 2.2 new criteria; conformance statement published; feeds VPAT (10.8).
- Ongoing: axe-core gate on all future PRs; quarterly manual audit.

## 16. Test Plan

- **Unit** — ARIA utility functions (announcer, focus trap, skip link).
- **Integration** — Axe-core Playwright scans for all 6 priority page flows.
- **End-to-end** — Manual NVDA + Chrome screen-reader walkthrough of quiz-take flow; keyboard-only module reorder.
- **Security** — ARIA label injection from user content sanitized.
- **Accessibility** — This IS the test plan section.
- **Performance / load** — axe-core CI scan < 60s; no regression on page TTI from skip-nav or ARIA additions.
- **Manual exploratory** — QA screen-reader checklist for each release.
