## 1. Problem Statement

Publishers licensing digital textbooks and video content to institutions require that content not be freely redistributable once placed in an LMS. Lextures has no DRM layer and no watermarking capability, making it unsuitable for any publisher content agreement. Instructors who upload licensed PDF textbooks or videos are unknowingly exposing that content to mass distribution via shareable URLs. Visual watermarking (user identity overlay on PDFs and videos) deters unauthorized redistribution even when DRM is technically infeasible.

## 2. Goals

- Implement user-identity watermarks on PDF and video content flagged as licensed.
- Integrate with Widevine and FairPlay DRM for HLS video streams (via a DRM packaging service).
- Provide an admin workflow to mark content as DRM-protected and configure license keys.
- Ensure watermarked/DRM content URLs are non-shareable (pre-signed, user-bound).

## 3. Non-Goals

- Hosting a full DRM key server (delegate to a managed service such as Axinom or BuyDRM).
- CSS-only watermarks easily removed via browser DevTools (must be rendered server-side or canvas-composited).
- Retroactive DRM enforcement on content uploaded before this feature.

## 4. Personas & User Stories

- **As a publisher sales manager**, I want proof that my licensed video content cannot be screen-scraped and redistributed so that I agree to place content in Lextures.
- **As an admin**, I want to mark a course's PDF textbook as licensed so that every page renders with the viewing student's name and date watermarked.
- **As an instructor**, I want to know which course materials are DRM-protected so that I understand any viewing restrictions for my students.
- **As a student**, I want to know why a video requires a browser extension or policy so that I understand the access restriction.

## 5. Functional Requirements

- **FR-1.** The system MUST allow admins to mark a `storage.objects` record with `drm_type` (`watermark_only` | `widevine` | `fairplay` | `none`).
- **FR-2.** For `watermark_only` PDFs, the system MUST dynamically overlay the viewing user's full name and access timestamp on each PDF page using server-side PDF manipulation (e.g., `pdf-lib` or server-side `pdftk`).
- **FR-3.** For `watermark_only` videos, the system MUST overlay a semi-transparent user name + timestamp as a burned-in watermark during the HLS segment signing step (FFmpeg `drawtext` filter on a per-request basis) OR as a CSS-rendered canvas overlay on the player.
- **FR-4.** For `widevine` / `fairplay` content, the system MUST package HLS/DASH streams with CENC/AES-128 encryption during the transcoding pipeline (8.3) and integrate with a DRM license server for key delivery.
- **FR-5.** Pre-signed URLs for DRM content MUST be bound to a single user ID and IP address block (`/24`); sharing the URL MUST result in license denial.
- **FR-6.** The system MUST log every DRM license request and watermark render to `user_audit` for forensic tracking.
- **FR-7.** Admins MUST receive an alert if a watermarked resource is downloaded more than 5 times by the same user within 1 hour (redistribution signal).

## 6. Non-Functional Requirements

- **Performance** — PDF watermark render < 500 ms per page on server-side. DRM license request < 200 ms p95.
- **Security** — DRM keys stored in a secrets manager (17.17), never in DB plaintext. Widevine/FairPlay integration uses HTTPS only. Video CENC encryption per MPEG-CENC spec.
- **Privacy & Compliance** — Watermark embeds user PII; operator must disclose this in privacy policy. FERPA: watermark logs accessible only to admins with legitimate educational interest.
- **Accessibility** — Watermark MUST NOT obscure content readability (semi-transparent, positioned in corner). Confirm text contrast still meets WCAG 2.1 AA after watermark.
- **Scalability** — Watermark rendering is stateless; horizontally scalable. DRM license server managed externally.
- **Reliability** — If watermark rendering fails, fall back to serving unwatermarked content for non-DRM resources with log alert; for hard DRM, deny playback.
- **Observability** — Metrics: `drm_license_requests_total`, `watermark_renders_total`, `drm_license_denied_total`. Alert on anomalous download frequency.
- **Maintainability** — Watermark service in `server/src/services/watermark.rs`; DRM packager integration in `server/src/services/drm.rs`.
- **Internationalization** — Watermark text format locale-aware (date format per user locale).
- **Backward compatibility** — DRM opt-in only; untagged content behaves as before.

## 7. Acceptance Criteria

- **AC-1.** *Given* a PDF is marked `drm_type=watermark_only` *When* student Alice downloads the PDF *Then* every page bears "Alice Johnson — 2026-04-17 14:35 UTC" overlaid semi-transparently in the bottom-right corner.
- **AC-2.** *Given* student A shares their pre-signed PDF URL with student B *When* student B opens the URL *Then* they receive a 403 (URL is user-bound) and the share attempt is logged.
- **AC-3.** *Given* a video is packaged with Widevine encryption *When* a student opens the video in Chrome *Then* the DRM license request succeeds and the video plays; attempting to download the raw segment file yields an encrypted blob.
- **AC-4.** *Given* an admin marks content as DRM-protected *When* the setting is saved *Then* the content page in the course shows a lock icon and "Protected content" label.
- **AC-5.** *Given* a user downloads a watermarked PDF 6 times within 1 hour *When* the anomaly detector runs *Then* an admin alert email is sent listing the user, resource, and download count.

## 8. Data Model

```sql
-- server/migrations/082_drm_settings.sql
CREATE TYPE storage.drm_type AS ENUM ('none', 'watermark_only', 'widevine', 'fairplay');

ALTER TABLE storage.objects
  ADD COLUMN drm_type storage.drm_type NOT NULL DEFAULT 'none',
  ADD COLUMN drm_key_id TEXT,
  ADD COLUMN drm_provider TEXT;

CREATE TABLE storage.drm_license_requests (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  object_id       UUID NOT NULL REFERENCES storage.objects(id),
  user_id         UUID NOT NULL REFERENCES user_account(id),
  ip_address      INET,
  granted         BOOLEAN NOT NULL,
  denial_reason   TEXT,
  requested_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON storage.drm_license_requests (object_id, user_id, requested_at);
```

## 12. Integration Points

- **Widevine** (Google) — CENC encryption, license server proxy.
- **FairPlay** (Apple) — HLS encryption, license server.
- **Axinom DRM / BuyDRM / Castlabs** — managed DRM-as-a-service (reduces key management burden).
- **pdf-lib** (TypeScript) or server-side PDF manipulation for watermark rendering.
- Internal: `server/src/services/watermark.rs`, `server/src/services/drm.rs`, `server/src/workers/transcode.rs` (8.3 — DRM packaging step), `server/src/storage/` (8.1).

## 13. Dependencies & Sequencing

- Must ship after: 8.1 (object storage), 8.3 (video transcoding — DRM packaging is a transcode step).
- Can ship in parallel with: most other section 8 features.
- Shared infra: secrets manager (17.17), DRM license server (external service account needed).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Widevine requires Google certification | M | H | Apply for Widevine L3 certification early; use managed DRM service to reduce friction |
| PDF watermark performance on large PDFs | M | M | Cache watermarked PDFs per user per day; use async render |
| DRM breaks offline/PWA access | H | M | Clearly communicate DRM limitations; exclude DRM content from offline sync |
| Legal complexity of DRM key management | M | H | Use managed DRM-as-a-service; consult legal before storing keys internally |

## 15. Rollout Plan

- **Flag:** `FEATURE_DRM=true` (default false; enterprise-only flag).
- **Sequencing:** watermark-only first (simpler, no Widevine dependency) → Widevine/FairPlay integration → GA.
- **GA criteria:** watermark renders correctly in PDF viewer; DRM video plays in Chrome (Widevine) and Safari (FairPlay).
- **Rollback:** disable flag; all files revert to unprotected access.

## 16. Test Plan

- **Unit** — Watermark coordinate calculation, DRM license request builder, anomaly counter logic.
- **Integration** — Server-side PDF watermark: render test PDF and verify overlay text.
- **End-to-end** — Playwright: open watermarked PDF, screenshot first page, verify watermark text visible.
- **Security** — Confirm user-bound URLs reject cross-user access; confirm DRM key not in API responses.
- **Accessibility** — Confirm watermark opacity does not reduce body text contrast below 4.5:1 ratio.
- **Manual exploratory** — Widevine playback in Chrome; FairPlay in Safari; test anomaly alert delivery.

## 17. Documentation & Training

- Admin docs: setting content protection levels, configuring DRM provider credentials.
- Legal docs: publisher content agreement requirements, watermark disclosure language for privacy policy.
- Internal runbook: managing DRM key rotation, handling license server outages.

## 18. Open Questions

1. Do we build our own DRM license proxy or use a fully managed service (Axinom, BuyDRM)?
2. Should watermarked PDFs be cached per-user to reduce server-side rendering load?
3. Do we support AES-128 (simpler, no Widevine) as an intermediate DRM step for video?
