## 1. Problem Statement

Students and instructors must download a file to their device before viewing it. There is no in-browser thumbnail for images, no PDF.js viewer for PDFs, and no inline preview for file submissions. Instructors grading assignments must download every submission to open it in a local app. This is especially painful for PDF submissions where an inline annotation workflow (3.1) is expected. Providing in-browser previews removes the download friction and is a prerequisite for inline grading tools.

## 2. Goals

- Render PDF files inline using PDF.js without any server-side processing.
- Display image files (JPEG, PNG, GIF, WebP, SVG) in a full-width viewer within the course UI.
- Show file thumbnails in file lists and module cards.
- Keep the implementation client-side only; no server-side thumbnail generation required in initial release.

## 3. Non-Goals

- Server-side thumbnail generation pipeline (potential future optimization for very large PDFs).
- Annotation or markup tools on previews (covered by 3.1).
- Office document preview (DOCX, XLSX, PPTX) — deferred; requires conversion service.
- Preview of videos (covered by 8.3 VideoPlayer component).

## 4. Personas & User Stories

- **As an instructor grading submissions**, I want to view a student's PDF inline without downloading it so that I can grade faster.
- **As a student viewing course materials**, I want to read a PDF embedded in the module page so that I do not lose my browser context.
- **As an instructor browsing course files**, I want image thumbnails in the file list so that I can identify visual assets at a glance.
- **As a student on a managed Chromebook**, I want PDFs to open in the browser because installing PDF software is not allowed.

## 5. Functional Requirements

- **FR-1.** The system MUST render PDF files in an embedded PDF.js viewer (`pdfjs-dist` ≥ 4.x) when a user clicks a PDF file link.
- **FR-2.** The PDF viewer MUST load the PDF via a pre-signed URL (8.1) without exposing the raw storage URL to the page source.
- **FR-3.** The system MUST render image files (JPEG, PNG, GIF, WebP, SVG) in a lightbox overlay with zoom and pan support.
- **FR-4.** The system SHOULD generate client-side image thumbnails (first-frame for GIF, scaled canvas for large images) in file lists.
- **FR-5.** The PDF viewer MUST support text selection (copy-paste) and the PDF.js built-in Find toolbar.
- **FR-6.** File type detection MUST use both the stored MIME type and the file extension; unknown types fall back to a download button.
- **FR-7.** The viewer MUST display a loading indicator and a graceful error state if the pre-signed URL expires before the file loads.

## 6. Non-Functional Requirements

- **Performance** — PDF.js first page render < 2 s on a 5 MB PDF on a typical laptop. Image lightbox opens < 300 ms. Lazy-load pages (render on scroll).
- **Security** — PDF rendered in a sandboxed iframe (`sandbox="allow-scripts"`) to mitigate malicious PDF JS exploits. Pre-signed URLs are short-lived (1 h).
- **Privacy & Compliance** — Files rendered only to enrolled users via pre-signed URLs. No PDF content sent to a server for rendering. FERPA: submission PDFs accessible only to the submitting student and the instructor.
- **Accessibility** — PDF.js text layer enables screen-reader access to selectable text. Lightbox traps focus; Escape closes it. `aria-label` on viewer iframe. WCAG 2.1 AA.
- **Scalability** — Client-side only; no server load per preview. Pre-signed URL rate limit (8.1) applies.
- **Reliability** — Graceful degradation: if PDF.js fails to initialize, fall back to a direct download link.
- **Observability** — Log `file_id`, `mime_type`, `preview_type` on each preview open event to `user_audit`.
- **Maintainability** — `FilePreview` component in `clients/web/src/components/FilePreview.tsx`; PDF viewer wrapper in `clients/web/src/components/PdfViewer.tsx`.
- **Internationalization** — Viewer toolbar labels externalised; PDF.js locale set from app locale.
- **Backward compatibility** — Existing download links still function; preview is layered on top, not replacing the download option.

## 7. Acceptance Criteria

- **AC-1.** *Given* a course file list contains a PDF *When* an instructor clicks the file *Then* a PDF.js viewer opens inline with page navigation, zoom controls, and the first page visible within 2 s.
- **AC-2.** *Given* a student opens a PDF submission in the assignment view *When* the PDF is 50 pages *Then* pages beyond the visible viewport are lazy-loaded on scroll.
- **AC-3.** *Given* a file list contains a PNG image *When* an instructor hovers over it *Then* a thumbnail is shown, and clicking opens a lightbox with zoom controls.
- **AC-4.** *Given* the pre-signed URL expires while the PDF is open *When* the user scrolls to a new page *Then* the viewer automatically fetches a fresh pre-signed URL and continues rendering without a visible error.
- **AC-5.** *Given* a DOCX file is in the file list *When* a user clicks it *Then* a "Download to view" button appears (no inline preview attempt).
- **AC-6.** *Given* a screen-reader user opens the PDF viewer *When* they navigate by line *Then* the PDF.js text layer exposes the correct reading order.

## 8. Data Model

No schema changes required. MIME type already stored in `storage.objects` (8.1).

## 9. API Surface

No new API endpoints. The `GET /courses/:id/files/:file_id` redirect to pre-signed URL (8.1) is the only backend dependency. Optionally add `?inline=true` query param to set `Content-Disposition: inline` on the presigned URL for direct browser rendering.

## 10. UI / UX

- File list items: show a file-type icon or image thumbnail (< 80×80 px canvas). Image files show a scaled preview; PDFs show a PDF icon with page count.
- Clicking a previewable file: opens a modal/drawer overlay. PDF viewer fills 85 % of the viewport height. Image lightbox fills the screen with a dark backdrop.
- PDF viewer controls: page number input, previous/next, zoom in/out/fit-width, Find bar, Download button.
- Image viewer controls: zoom slider, pan (drag), rotate (optional), Download button, close (X / Escape).
- Loading state: skeleton shimmer.
- Error state: "Unable to load file — download instead" with a direct download link.
- Mobile: PDF viewer responsive (single-column, pinch-to-zoom via PDF.js touch events). Lightbox swipe to close.
- Accessibility: modal has `role="dialog"`, `aria-labelledby` = file name. Focus moves to viewer on open. Escape closes. Tab cycles controls.
- i18n keys: `preview.loading`, `preview.error`, `preview.download`, `preview.page`, `preview.of`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| PDF.js bundle size bloat (> 1 MB) | H | M | Lazy-import `pdfjs-dist` only when a PDF file is opened |
| Malicious PDF executing JS in viewer | M | H | Sandbox iframe; disable PDF.js scripting option |
| Pre-signed URL expiry mid-session | M | M | Implement URL refresh hook in PdfViewer before fetch |
| Large PDF (> 100 pages) rendering memory | M | M | Render only visible pages; set PDF.js `maxImageSize` limit |

## 15. Rollout Plan

- **Flag:** `FEATURE_FILE_PREVIEW=true` (default false; ship as opt-in initially).
- **Sequencing:** build components → add to assignment submission view → add to course files list → enable flag for all users.
- **GA criteria:** PDF viewer functional on top-3 browsers; no new JS errors in Sentry from preview sessions.
- **Rollback:** disable flag; file links revert to download-only.

## 16. Test Plan

- **Unit** — MIME-type detection logic, URL refresh logic, page-count display.
- **Integration** — Render test PDFs of 1, 10, 100 pages; render test images of each supported type.
- **End-to-end** — Playwright: open PDF, navigate to page 3, search text, confirm find result highlighted.
- **Security** — Confirm sandboxed iframe prevents PDF JS execution; confirm cross-user pre-signed URL returns 403.
- **Accessibility** — axe scan of PDF viewer modal; NVDA walkthrough for text layer navigation.
- **Performance** — Lighthouse: PDF.js lazy-import does not increase initial page load > 50 ms.
- **Manual exploratory** — Test on iOS Safari (pinch zoom), Android Chrome, Firefox, Edge.

## 17. Documentation & Training

## 18. Open Questions

1. Should we generate server-side PDF thumbnails (first-page PNG) for file list cards, or accept the larger PDF.js bundle for client-side rendering?
2. Do we support DOCX/XLSX preview via a conversion microservice (LibreOffice headless) in a follow-on release?
3. Should the PDF viewer support annotation (highlight, comment) now, or strictly defer to 3.1?
