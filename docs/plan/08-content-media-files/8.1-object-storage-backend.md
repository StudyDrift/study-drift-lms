## 1. Problem Statement

All course files and uploaded media are written to the local filesystem of the application server, making horizontal scaling impossible and risking data loss on any single-server failure or redeploy. There is no CDN delivery path, no signed-URL expiry model, and no per-tenant storage isolation. Any production deployment beyond a single DigitalOcean droplet is blocked until an S3-compatible object storage layer replaces local disk I/O in `server/src/routes/course_files.rs` and `server/src/services/course_image_upload.rs`.

## 2. Goals

- Replace local-disk file writes with an S3-compatible backend (configurable: AWS S3, Cloudflare R2, GCS, MinIO).
- Serve all files via short-lived pre-signed URLs (max 1 h) so the app server never proxies bytes.
- Maintain backward compatibility for files already on disk during migration.
- Expose storage-driver abstraction so switching providers requires only an env change.
- Provide per-bucket / per-prefix isolation per tenant (org) and per course.

## 3. Non-Goals

- Building a user-facing file manager UI beyond the existing upload flow.
- Implementing CDN edge caching rules (that is an IaC concern in `iac/`).
- Chunked / resumable upload protocol (covered by 8.2).
- File encryption at rest beyond what the storage provider offers natively.

## 4. Personas & User Stories

- **As an instructor**, I want uploaded lecture videos to survive a server restart so that students always have access to course materials.
- **As a student**, I want file download links to load quickly via CDN so that I am not bottlenecked by the app server.
- **As a platform admin**, I want to configure any S3-compatible provider with a single env variable so that on-premise MinIO deployments work without code changes.
- **As an infra engineer**, I want the app to scale horizontally without sharing local disk so that I can run multiple app replicas behind a load balancer.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a `StorageDriver` trait (Rust) with `put_object`, `get_presigned_url`, `delete_object`, and `list_objects` methods selectable at startup via `STORAGE_BACKEND` env (`s3`, `r2`, `gcs`, `minio`, `local`).
- **FR-2.** The system MUST generate pre-signed GET URLs with a configurable TTL (default 3600 s) for every file served; direct public URLs MUST NOT be used.
- **FR-3.** The system MUST store files under a key scheme of `{tenant_id}/{course_id}/{resource_type}/{uuid}.{ext}` to enable per-tenant lifecycle policies.
- **FR-4.** The system MUST migrate existing local-disk files to the configured bucket on first startup when `STORAGE_MIGRATE_LOCAL=true` and record migrated keys in a `storage_migrations` table.
- **FR-5.** The system SHOULD support multipart upload initiation (pass-through to 8.2) by returning an upload URL rather than accepting the body directly for files larger than 10 MB.
- **FR-6.** The system MUST validate MIME type and file size against per-course quotas (see 8.5) before issuing a put URL.
- **FR-7.** All object deletes triggered by course or enrollment deletion MUST be queued as background jobs to avoid blocking HTTP responses.
- **FR-8.** The system SHOULD emit a structured `file.uploaded` domain event (course_id, user_id, object_key, bytes, mime_type, timestamp) consumable by the job queue.

## 6. Non-Functional Requirements

- **Performance** — Pre-signed URL generation MUST complete in < 50 ms p95. File metadata reads (without byte transfer) MUST be < 20 ms p95. No file bytes transit the app server.
- **Security** — Pre-signed URLs MUST be scoped to the authenticated user's enrollment; the signing key MUST be rotated without service restart. Bucket MUST have no public ACL.
- **Privacy & Compliance** — FERPA: files accessible only to enrolled users and instructors. GDPR: deletion path implemented. SOC 2: access logged to `user_audit`.
- **Accessibility** — No direct UI impact; downloadable files must have descriptive filenames (enforced in upload validation).
- **Scalability** — Must handle 10 000 concurrent pre-signed URL requests/min across a cluster without shared state.
- **Reliability** — Storage driver MUST retry transient network errors with exponential backoff (max 3 attempts). Availability target: 99.9 % (delegated to storage provider SLA).
- **Observability** — Metrics: `storage_put_duration_ms`, `storage_get_presign_duration_ms`, `storage_error_total`. All errors logged with `object_key` and `tenant_id`.
- **Maintainability** — Driver implementations in `server/src/storage/` with one file per provider. Feature-flagged behind `STORAGE_BACKEND`.
- **Internationalization** — File metadata (display names) stored as UTF-8; no locale-specific logic in the storage layer.
- **Backward compatibility** — `local` driver retained so existing dev setups and demos continue to work without cloud credentials.

## 7. Acceptance Criteria

- **AC-1.** *Given* `STORAGE_BACKEND=minio` is set *When* an instructor uploads a 5 MB PDF *Then* the file is written to MinIO, a pre-signed URL valid for 1 h is returned, and no bytes are written to local disk.
- **AC-2.** *Given* a pre-signed URL has expired *When* a student requests the file *Then* the server issues a fresh pre-signed URL and redirects; the expired URL returns 403.
- **AC-3.** *Given* `STORAGE_MIGRATE_LOCAL=true` and 50 existing local files *When* the server starts *Then* all 50 files are copied to the configured bucket, logged in `storage_migrations`, and the local copies are preserved until explicitly purged.
- **AC-4.** *Given* a course is deleted *When* the background deletion job runs *Then* all objects under `{tenant_id}/{course_id}/` are removed from the bucket and confirmed via `list_objects`.
- **AC-5.** *Given* `STORAGE_BACKEND=local` *When* any upload or download occurs *Then* behaviour is identical to the current implementation and no S3 credentials are required.
- **AC-6.** *Given* the storage provider returns a 503 *When* a put is attempted *Then* the system retries up to 3 times with backoff and returns a 502 to the caller with a structured error body.

## 8. Data Model

```sql
-- server/migrations/073_storage_objects.sql
CREATE TABLE storage.objects (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id     UUID NOT NULL,
  course_id     UUID REFERENCES course.courses(id) ON DELETE SET NULL,
  object_key    TEXT NOT NULL UNIQUE,
  bucket        TEXT NOT NULL,
  mime_type     TEXT NOT NULL,
  size_bytes    BIGINT NOT NULL DEFAULT 0,
  uploaded_by   UUID REFERENCES user_account(id),
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  deleted_at    TIMESTAMPTZ
);
CREATE INDEX ON storage.objects (course_id);
CREATE INDEX ON storage.objects (tenant_id, deleted_at);

-- server/migrations/074_storage_migrations.sql
CREATE TABLE storage.migrations (
  id            SERIAL PRIMARY KEY,
  local_path    TEXT NOT NULL,
  object_key    TEXT NOT NULL,
  migrated_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Backfill: on `STORAGE_MIGRATE_LOCAL=true`, a one-time startup routine reads all `course_files` rows, copies local files, writes `storage.objects` rows, and records in `storage.migrations`.

## 9. API Surface

No new public HTTP endpoints. Existing endpoints change internally:

| Route | Change |
|---|---|
| `POST /courses/:id/files` | Returns `{ object_key, presigned_put_url, expires_at }` instead of uploading directly |
| `GET /courses/:id/files/:file_id` | Returns 302 redirect to fresh pre-signed GET URL |
| `DELETE /courses/:id/files/:file_id` | Enqueues background deletion; returns 202 |

Rate limit: pre-signed URL generation capped at 300 req/min per user via existing middleware.

## 10. UI / UX

- No visible UI change beyond upload progress behavior (handled by 8.2).
- On download, the browser follows a 302 redirect; this is transparent to users.
- Error state: if storage is unreachable, show a toast "File temporarily unavailable — try again in a moment" rather than a raw 500.
- Accessibility: no new interactive elements; existing download links retain `aria-label`.

## 11. AI / ML Considerations

Not applicable. Storage layer is infrastructure only.

## 12. Integration Points

- **AWS S3 API** (aws-sdk-rust `aws-sdk-s3` v1.x) — also used by R2 and MinIO via endpoint override.
- **Google Cloud Storage** — via `google-cloud-storage` crate or presigned URL via HMAC key.
- **Cloudflare R2** — S3-compatible; uses S3 SDK with custom endpoint.
- **MinIO** — S3-compatible; used for local dev via docker-compose.
- Internal: `server/src/routes/course_files.rs`, `server/src/services/course_image_upload.rs`, `server/src/repos/course_files.rs`.
- Job queue (17.3) for async deletions and migration tasks.

- Must ship before: 8.2, 8.3, 8.4, 8.5, 8.6, 8.7, 8.10 — all file-handling features depend on this layer.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Existing local files lost during migration | M | H | Keep local copies until explicit purge flag; write migration log |
| Pre-signed URL leakage via shared link | M | H | Short TTL (1 h), URL bound to user-agent hash in signature metadata |
| Storage provider outage | L | H | Fallback to local driver with circuit-breaker, alerting via 17.7 |
| Large file upload OOM on app server | M | H | Cap direct upload at 10 MB; require pre-signed PUT for larger files |
| Cost overrun on egress | L | M | R2 / MinIO have zero egress; default to R2 in production IaC |

## 15. Rollout Plan

- **Flag:** `STORAGE_BACKEND` env (default `local`; flip to `r2` in staging).
- **Sequencing:** deploy schema migrations → deploy code with `STORAGE_BACKEND=local` → validate → switch env to `r2` → enable `STORAGE_MIGRATE_LOCAL=true` in one-off job.
- **Dogfood:** internal Lextures workspace uploads.
- **GA criteria:** 100 % of new uploads going to object store; zero local-disk writes in logs.
- **Rollback:** set `STORAGE_BACKEND=local`; no schema rollback needed (table is additive).

## 16. Test Plan

- **Unit** — `StorageDriver` trait: mock implementations for each method; test presign TTL calculation, key scheme generation, retry logic.
- **Integration** — MinIO via `testcontainers-rs`: upload, presign, download, delete lifecycle; migration job correctness.
- **End-to-end** — Playwright: upload a PDF in a course, navigate away, return and confirm download resolves via redirect.
- **Security** — Confirm bucket has no public-read ACL; confirm expired URLs return 403; confirm cross-tenant key access denied.
- **Accessibility** — No new UI elements; existing download button aria-labels validated.
- **Performance / load** — k6: 500 concurrent presign requests, target p95 < 50 ms.
- **Manual exploratory** — Test all four provider backends (S3, R2, GCS, MinIO) with real credentials in staging.

## 18. Open Questions

1. Should R2 be the default production provider in `iac/production/` or should we remain provider-agnostic?
2. Do we implement server-side encryption (SSE-S3) or rely on bucket default encryption? Need legal sign-off for FERPA.
3. What is the retention policy for deleted objects — immediate or 30-day soft-delete in the bucket lifecycle?
4. Should `storage.objects` be in its own Postgres schema or co-located in the `course` schema?
