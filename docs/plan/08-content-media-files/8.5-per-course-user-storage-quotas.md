## 1. Problem Statement

There are no storage quotas in the current codebase. Any user can upload unlimited files and any course can consume unbounded disk/object storage, which risks disk exhaustion on local deployments and runaway cloud storage costs in production. Institutional administrators expect to enforce per-course and per-user limits that align with their license tier. Without quotas, a single instructor uploading hundreds of raw lecture videos could consume the entire tenant allocation before other courses get any storage.

## 2. Goals

- Enforce configurable storage quotas at three levels: tenant, course, and user.
- Reject uploads that would exceed any applicable quota before any bytes are transferred.
- Surface real-time quota usage to instructors and administrators in the UI.
- Provide admin tools to adjust quotas and view usage across courses.

## 3. Non-Goals

- Fine-grained per-file-type quotas (e.g., 1 GB for video, 200 MB for PDFs) — initial release uses a single bytes limit.
- Automated tier-based quota assignment from a billing system (deferred to 15.3).
- Storage cost chargebacks to individual departments (enterprise feature).

## 4. Personas & User Stories

- **As a platform admin**, I want to set a 5 GB per-course storage limit so that no single course bankrupts the tenant's object storage budget.
- **As an instructor**, I want to see a storage usage bar in my course settings so that I know how close I am to the limit before uploading a large video.
- **As a student**, I want a clear error message when my submission exceeds the course quota so that I know why my upload failed.
- **As a platform admin**, I want to increase a specific course's quota on demand so that I can accommodate large media-heavy courses.

## 5. Functional Requirements

- **FR-1.** The system MUST maintain a `storage.quota_settings` table with configurable limits (bytes) at tenant, course, and user levels.
- **FR-2.** The system MUST maintain a `storage.usage_counters` table with real-time byte counts per (tenant, course, user) updated atomically on upload completion and file deletion.
- **FR-3.** Before accepting any upload (tus CREATION or direct POST), the system MUST check all three quota levels and return `403 { "error": "quota_exceeded", "quota_type": "course|user|tenant", "used_bytes": N, "limit_bytes": N }` if any would be exceeded.
- **FR-4.** The system SHOULD send an email notification to the instructor when course usage reaches 80 % of the course quota.
- **FR-5.** The system MUST expose a `GET /courses/:id/storage-usage` endpoint returning `{ used_bytes, limit_bytes, percent_used }` for the course storage gauge in the UI.
- **FR-6.** Admin endpoints MUST allow reading and updating quota settings per course and per user without requiring a deployment.
- **FR-7.** Counter updates MUST be atomic: use `UPDATE ... SET used_bytes = used_bytes + $1 WHERE used_bytes + $1 <= limit_bytes RETURNING *` to prevent race conditions between concurrent uploads.

## 6. Non-Functional Requirements

- **Performance** — Quota check MUST complete in < 10 ms p95 (single indexed row read). Usage counter update MUST be < 5 ms p95.
- **Security** — Only admins and course instructors may read course usage; only admins may modify quota limits.
- **Privacy & Compliance** — Usage data does not contain file content or PII. FERPA: usage statistics accessible only to school officials.
- **Accessibility** — Usage bar widget uses `role="progressbar"` with `aria-valuenow`, `aria-valuemin`, `aria-valuemax`, and a text label.
- **Scalability** — Counter table sharded by `(tenant_id, course_id)`; no full-table scans. Up to 100 000 courses per tenant.
- **Reliability** — Counter must be reconcilable: a background job recomputes actual usage from `storage.objects` nightly and patches any drift.
- **Observability** — Metrics: `quota_exceeded_total` (by type), `quota_usage_bytes` (gauge per course). Log quota level hit on every rejection.
- **Maintainability** — Quota logic in `server/src/services/storage_quota.rs` (new); injected as middleware in upload routes.
- **Internationalization** — Error message keys externalised: `quota.exceeded`, `quota.courseNearLimit`, `quota.usageLabel`.
- **Backward compatibility** — Default quota = unlimited (`NULL`) so existing behavior unchanged until admin sets explicit limits.

## 7. Acceptance Criteria

- **AC-1.** *Given* a course has a 1 GB quota and 900 MB used *When* an instructor tries to upload a 200 MB file *Then* the server returns `403 quota_exceeded` before any bytes are received.
- **AC-2.** *Given* an upload completes successfully *When* the `file.uploaded` event is processed *Then* `storage.usage_counters.used_bytes` increments by the file's size within 1 second.
- **AC-3.** *Given* a file is deleted *When* the deletion job completes *Then* `used_bytes` decrements by the file's size.
- **AC-4.** *Given* an admin sets a course quota to 2 GB via the admin API *When* the quota is saved *Then* subsequent uploads validate against 2 GB and the instructor's storage gauge reflects the new limit immediately.
- **AC-5.** *Given* course usage reaches 80 % *When* the threshold-check job runs *Then* an email is sent to the course instructor with current usage and limit.
- **AC-6.** *Given* two concurrent 600 MB uploads to a 1 GB course *When* both quota checks run simultaneously *Then* exactly one succeeds and one receives `403 quota_exceeded`; total usage never exceeds 1 GB.

## 8. Data Model

```sql
-- server/migrations/078_storage_quotas.sql
CREATE TABLE storage.quota_settings (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  scope       TEXT NOT NULL CHECK (scope IN ('tenant', 'course', 'user')),
  scope_id    UUID NOT NULL,
  limit_bytes BIGINT,       -- NULL = unlimited
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (scope, scope_id)
);

CREATE TABLE storage.usage_counters (
  scope       TEXT NOT NULL CHECK (scope IN ('tenant', 'course', 'user')),
  scope_id    UUID NOT NULL,
  used_bytes  BIGINT NOT NULL DEFAULT 0,
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (scope, scope_id)
);
```

Backfill: on first deployment, compute initial `used_bytes` from `storage.objects` grouped by course_id and user_id.

## 9. API Surface

| Method | Path | Auth | Description |
|---|---|---|---|
| `GET` | `/courses/:id/storage-usage` | Instructor JWT | Returns `{ used_bytes, limit_bytes, percent_used }` |
| `GET` | `/admin/storage-quotas` | Admin JWT | List all quota settings with usage |
| `PUT` | `/admin/storage-quotas/:scope/:scope_id` | Admin JWT | Set quota limit |
| `POST` | `/admin/storage-quotas/reconcile` | Admin JWT | Trigger manual counter reconciliation |

## 10. UI / UX

- Course settings page (`CourseSettings.tsx`): add a "Storage" section with a progress bar (`role="progressbar"`) showing `X GB used of Y GB`. Warn icon appears at 80 %.
- Upload failure state: toast "Storage limit reached — contact your instructor or admin" for students; "Course storage limit reached — increase quota in Admin settings" for instructors.
- Admin report page (`Reports.tsx`): new "Storage Usage" tab listing courses sorted by usage descending with an "Edit Quota" button inline.
- Accessibility: progress bar has `aria-label="Course storage usage"`, `aria-valuenow`, percentage shown as text alongside bar.
- i18n keys: `storage.used`, `storage.limitReached`, `storage.warningAt80`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Race condition: two uploads exceed quota simultaneously | M | M | Atomic counter update with constraint check in single SQL statement |
| Counter drift from failed deletion jobs | M | M | Nightly reconciliation job patches drift; alert on divergence > 1 % |
| Admin forgets to set quotas and disk fills | H | H | Default tenant quota derived from `STORAGE_DEFAULT_TENANT_QUOTA_GB` env |

## 15. Rollout Plan

- **Flag:** `FEATURE_STORAGE_QUOTAS=true` (default false).
- **Sequencing:** deploy schema → backfill counters → deploy service and API → deploy UI components → set default quotas in admin → enable flag.
- **Dogfood:** internal tenant with explicit quota set.
- **GA criteria:** quota enforcement operational with zero counter drift after 7-day soak; 80 % alert email delivered correctly.
- **Rollback:** disable flag; upload routes skip quota check.

## 16. Test Plan

- **Unit** — Quota check logic, counter increment/decrement, reconciliation diff calculation.
- **Integration** — DB: atomic increment at limit boundary; concurrent upload race test.
- **End-to-end** — Playwright: set 1 MB quota, upload 900 KB file (success), upload 200 KB file (blocked), verify error toast.
- **Security** — Confirm only admin can modify quotas; student cannot read admin quota API.
- **Accessibility** — axe scan of storage progress bar; verify `aria-valuenow` updates after upload.
- **Performance** — Quota check latency under 10 ms with 100 k courses in DB.
- **Manual exploratory** — Verify 80 % notification email content and timing.

## 17. Documentation & Training

- Admin docs: how to set per-course and per-user quotas, how to run reconciliation, how to interpret usage reports.
- End-user docs: what "Storage limit reached" means and who to contact.
- Internal runbook: manual counter reconciliation steps, clearing stuck counters.

## 18. Open Questions

1. Should default quota be set by tenant plan tier (linked to billing, 15.3) or manually by admin only?
2. Do we count transcoded renditions (8.3) against the quota or only the source file?
3. Should there be a grace period after exceeding quota (e.g., allow 110 % for 24 h) before hard block?
