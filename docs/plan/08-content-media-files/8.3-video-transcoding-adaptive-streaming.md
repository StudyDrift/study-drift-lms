## 1. Problem Statement

Uploaded videos are served as raw MP4 files directly from object storage with no transcoding, no adaptive bitrate streaming, and no format normalization. Students on low-bandwidth connections stall waiting for a 1 GB lecture to buffer; mobile Safari cannot play certain codecs; there is no thumbnail preview. Without HLS or MPEG-DASH, Lextures cannot compete with any modern LMS for video-heavy courses. The transcoding pipeline must be asynchronous, triggered on upload completion, and deliver HLS/DASH manifests served from CDN.

## 2. Goals

- Transcode uploaded video to multi-bitrate HLS (RFC 8216) and optionally MPEG-DASH renditions.
- Generate a poster image (thumbnail) at the 5-second mark.
- Store all output segments and manifests in the object storage bucket under a structured prefix.
- Serve HLS manifests via a CDN-friendly pre-signed or public URL; player integrated into the frontend.
- Provide real-time transcoding status visible to the uploader.

## 3. Non-Goals

- Real-time live streaming (RTMP ingest, SRT) — deferred.
- DRM encryption of transcoded content (covered by 8.10).
- AI-based scene detection or chapter marking (potential future feature).
- Hosting our own transcoding infrastructure vs. delegating to a managed service (decision in Open Questions).

## 4. Personas & User Stories

- **As an instructor**, I want a video I upload to be playable at the student's internet speed so that slow connections do not block learning.
- **As a student on mobile data**, I want the video to auto-select a lower quality so that I do not consume my entire data plan watching a lecture.
- **As an instructor**, I want a thumbnail preview shown on the module page before students click play so that the content looks professional.
- **As an admin**, I want failed transcoding jobs to retry automatically and alert me if they fail permanently so that I am not blindsided by broken videos.

## 5. Functional Requirements

- **FR-1.** The system MUST trigger a transcoding job within 30 seconds of a `file.uploaded` event for video MIME types (`video/mp4`, `video/quicktime`, `video/webm`, `video/x-matroska`).
- **FR-2.** The system MUST produce at least three HLS renditions (360p 500 kbps, 720p 2 Mbps, 1080p 4 Mbps) using FFmpeg with H.264 video and AAC audio.
- **FR-3.** The system MUST generate an HLS master playlist (`master.m3u8`) and per-rendition playlists conforming to RFC 8216 §4.
- **FR-4.** The system SHOULD also produce a MPEG-DASH manifest (`manifest.mpd`) for MSE-capable browsers.
- **FR-5.** The system MUST extract a poster image at the 5-second mark as a JPEG (max 1280×720).
- **FR-6.** The system MUST update a `storage.transcode_jobs` row with status (`queued`, `processing`, `done`, `failed`) and expose status via an API endpoint.
- **FR-7.** The system MUST retry failed jobs up to 3 times with exponential back-off before marking permanent failure and emitting an alert.
- **FR-8.** The system SHOULD delete the original raw upload from object storage after a successful transcode (configurable; default retain for 30 days).
- **FR-9.** The video player component MUST use hls.js (MSE) with fallback to native HLS on Safari/iOS.

## 6. Non-Functional Requirements

- **Performance** — Transcode 1 h of 1080p video in < 20 min wall clock (FFmpeg `-preset fast`, multi-core). Manifest fetch < 100 ms p95 from CDN.
- **Security** — Transcoding worker runs in an isolated container with no network access to production DB. Input files validated before ffprobe. HLS segment URLs may be pre-signed or public based on course visibility setting.
- **Privacy & Compliance** — No PII processed during transcode. ADA: captioning handled in 8.4 (cross-link). FERPA: private courses use pre-signed segment URLs.
- **Accessibility** — Video player MUST expose `<track>` element hookable for WebVTT captions (8.4). Player controls keyboard-navigable. See also 12.4.
- **Scalability** — Transcoding workers scale independently via job queue (17.3). Queue depth metric drives auto-scaling. Up to 50 concurrent jobs.
- **Reliability** — Jobs idempotent: re-processing same source file replaces previous output. Dead-letter queue for permanent failures with alert.
- **Observability** — Metrics: `transcode_job_queued_total`, `transcode_job_duration_seconds`, `transcode_job_failed_total`. Log `job_id`, `source_key`, `duration_s`, `error`.
- **Maintainability** — Transcoding logic in `server/src/workers/transcode.rs`; FFmpeg invoked via subprocess or `ffmpeg-next` crate.
- **Internationalization** — Status messages (`Transcoding…`, `Ready`, `Failed`) externalised as i18n keys.
- **Backward compatibility** — Raw MP4 URLs remain accessible until transcode completes; player falls back to raw MP4 if HLS not ready.

## 7. Acceptance Criteria

- **AC-1.** *Given* an instructor uploads a 100 MB MP4 *When* the upload completes *Then* a transcode job is queued within 30 s and status shows `processing` in the UI.
- **AC-2.** *Given* a transcode job completes successfully *When* a student opens the video player *Then* hls.js loads `master.m3u8`, three quality levels appear in the quality selector, and the poster image is displayed before play.
- **AC-3.** *Given* a student throttles their connection to 500 kbps in browser devtools *When* they play the video *Then* hls.js automatically switches to the 360p rendition without user action.
- **AC-4.** *Given* FFmpeg crashes processing a corrupt input file *When* the worker retries 3 times *Then* the job status becomes `failed`, the instructor sees an error banner, and a Slack/email alert is fired.
- **AC-5.** *Given* a course is private *When* a student fetches an HLS segment URL *Then* the URL is a pre-signed S3 URL with a 1-hour TTL and an unauthenticated request returns 403.
- **AC-6.** *Given* a transcode job succeeds *When* 30 days have elapsed *Then* the original raw MP4 is deleted from object storage (if `TRANSCODE_RETAIN_SOURCE_DAYS=30`).

## 8. Data Model

```sql
-- server/migrations/076_transcode_jobs.sql
CREATE TYPE storage.transcode_status AS ENUM ('queued', 'processing', 'done', 'failed');

CREATE TABLE storage.transcode_jobs (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  source_key      TEXT NOT NULL,
  output_prefix   TEXT,
  master_playlist TEXT,
  dash_manifest   TEXT,
  poster_key      TEXT,
  status          storage.transcode_status NOT NULL DEFAULT 'queued',
  attempts        SMALLINT NOT NULL DEFAULT 0,
  error           TEXT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  started_at      TIMESTAMPTZ,
  completed_at    TIMESTAMPTZ,
  storage_object_id UUID REFERENCES storage.objects(id)
);
CREATE INDEX ON storage.transcode_jobs (status, created_at);
```

## 9. API Surface

| Method | Path | Auth | Description |
|---|---|---|---|
| `GET` | `/files/:object_id/transcode-status` | JWT (enrolled) | Returns `{ status, master_playlist_url, poster_url, renditions[] }` |
| `POST` | `/admin/files/:object_id/retranscode` | Admin JWT | Force re-queue transcode job |

WebSocket: `WS /ws/transcode/:job_id` streams status changes (`queued` → `processing` → `done`) to the uploader's UI in real-time.

## 10. UI / UX

- Module content page (`CourseModuleContentPage.tsx`): replace raw `<video>` tag with `VideoPlayer` component that initializes hls.js, shows poster, and renders a quality picker.
- Transcoding in progress: show a progress indicator "Video is processing — check back in a few minutes" with a spinner; auto-refresh via WebSocket.
- Error state: "Video processing failed — contact support" with retry button visible to instructor only.
- Mobile: hls.js degrades to native HLS on iOS; ensure `playsinline` attribute set.
- Accessibility: player wraps `<video>` with `aria-label`, custom controls accessible by keyboard (Space = play/pause, ←/→ = seek 10 s, F = fullscreen). Screen reader announces quality change.
- i18n keys: `video.processing`, `video.failed`, `video.qualityLabel`, `video.poster`.

## 11. AI / ML Considerations

Not applicable for transcoding itself. Auto-captioning (8.4) runs as a downstream job after transcode completes.

## 12. Integration Points

- **FFmpeg** (≥ 6.0) — invoked as subprocess; flags: `-c:v libx264 -preset fast -c:a aac -hls_time 6 -hls_list_size 0 -f hls`.
- **hls.js** (≥ 1.5, MIT) — browser HLS playback via Media Source Extensions.
- **dash.js** (≥ 4.x, BSD) — optional MPEG-DASH playback.
- **RFC 8216** — HLS specification for manifest format compliance.
- **MPEG-DASH ISO/IEC 23009-1** — DASH manifest format.
- Internal: `server/src/storage/` (8.1), job queue (17.3), `server/src/workers/transcode.rs` (new), `clients/web/src/components/VideoPlayer.tsx` (new).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Transcoding cost explosion from large uploads | M | H | Per-file size cap (4 GB); alert on job > 60 min; consider managed service (Mux, AWS MediaConvert) |
| FFmpeg vulnerability in input parsing | M | H | Run worker in rootless container; sandbox with seccomp; update FFmpeg in CI |
| HLS segment pre-sign overhead at scale | M | M | Cache segment URL tokens in Redis for TTL window |
| Storage cost for multiple renditions | M | M | Delete original after transcode; auto-delete 360p after 90 days for old content |

## 15. Rollout Plan

- **Flag:** `FEATURE_VIDEO_TRANSCODING=true` (default false).
- **Sequencing:** deploy worker image with FFmpeg → deploy schema → deploy API status endpoint → deploy VideoPlayer component → enable flag for internal courses → monitor job success rate → GA.
- **Dogfood:** internal Lextures instructor accounts.
- **GA criteria:** 95 % of video uploads successfully transcoded within 30 min; zero raw-MP4 fallback errors in student sessions.
- **Rollback:** disable flag; raw MP4 playback restored immediately.

## 16. Test Plan

- **Unit** — FFmpeg flag builder, HLS manifest validation, job status state machine, retry logic.
- **Integration** — Worker with MinIO: upload 10 MB test video, assert manifest created, assert 3 renditions accessible, assert poster key populated.
- **End-to-end** — Playwright: upload video, wait for `done` status via WebSocket, confirm hls.js loads and plays first 5 s.
- **Security** — Confirm worker cannot reach production DB; confirm pre-signed segment URLs expire; confirm unauthenticated segment access denied for private courses.
- **Accessibility** — Keyboard navigation of custom player controls; axe scan of player DOM.
- **Performance / load** — Submit 20 concurrent transcode jobs; confirm all complete within SLA; measure queue depth.
- **Manual exploratory** — Test playback on Chrome, Firefox, Safari iOS, Chrome Android; test quality auto-switch under simulated throttling.

## 18. Open Questions

1. Self-hosted FFmpeg workers vs. managed service (AWS Elemental MediaConvert, Mux, Cloudflare Stream)? Managed reduces ops burden but adds cost and vendor lock-in.
2. Should MPEG-DASH be in the initial release or deferred to a follow-on?
3. What is the maximum accepted source video file size (current thinking: 4 GB)?
4. Should 360p rendition be deleted after N days to reduce storage costs on old content?
