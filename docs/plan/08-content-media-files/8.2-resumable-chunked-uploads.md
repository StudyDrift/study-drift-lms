## 1. Problem Statement

Large file uploads — lecture videos, high-resolution images, ZIP packages — routinely fail under the current `multipart/form-data` single-request model in `server/src/routes/course_files.rs`. There is no pause-and-resume capability, no per-chunk retry, and the Axum `DefaultBodyLimit` caps uploads at a hard ceiling. Instructors in unreliable Wi-Fi environments (common in K-12 schools) lose partially uploaded files with no recovery path, forcing them to restart the entire transfer. Implementing the open **tus 1.0 protocol** with an Uppy frontend widget resolves all of these issues without vendor lock-in.

## 2. Goals

- Implement a tus 1.0 protocol server endpoint in Axum backed by S3 multipart uploads.
- Provide an Uppy-powered upload widget in `clients/web/src/components/` that handles pause, resume, and progress display.
- Support files up to 10 GB; default per-upload size limit configurable per course quota (see 8.5).
- Ensure idempotent chunk assembly so network retries do not corrupt file state.
- Clean up incomplete uploads after a configurable TTL (default 48 h).

## 3. Non-Goals

- Implementing a tus server for downloads (download resumption is handled by the storage provider's Range header support).
- Building a dedicated media library browser UI (deferred to future iteration).
- Supporting the tus Concatenation extension for parallel streaming (may revisit for 8.3 pipeline).

## 4. Personas & User Stories

- **As an instructor on a school Wi-Fi**, I want a video upload to resume automatically after a network drop so that I do not have to restart a 500 MB lecture upload from scratch.
- **As a student submitting a video assignment**, I want to see a real-time progress bar so that I know the upload is working and how long to wait.
- **As a platform admin**, I want incomplete uploads older than 48 hours to be automatically deleted so that orphaned chunks do not consume storage quota.
- **As an instructor on mobile**, I want to pause an upload and resume it hours later so that I can manage my data budget.

## 5. Functional Requirements

- **FR-1.** The system MUST implement the tus 1.0 Core protocol at `POST /tus/files`, `PATCH /tus/files/:upload_id`, `HEAD /tus/files/:upload_id`, and `DELETE /tus/files/:upload_id`.
- **FR-2.** The system MUST implement the tus Creation and Termination extensions.
- **FR-3.** The system MUST map each tus upload to an S3 multipart upload, assembling parts as chunks arrive and completing the S3 multipart upload when `Upload-Offset` equals `Upload-Length`.
- **FR-4.** The system MUST persist tus upload state (upload_id, s3_upload_id, offset, length, metadata) in a `storage.tus_uploads` table so that server restarts do not lose progress.
- **FR-5.** The system MUST enforce per-user and per-course quota checks (8.5) at `POST /tus/files` creation time and reject uploads that would exceed the quota.
- **FR-6.** The system SHOULD validate MIME type from the `Upload-Metadata` header against an allowlist before issuing the upload_id.
- **FR-7.** The frontend MUST use Uppy ≥ 4.x with the `@uppy/tus` transport plugin to replace the existing `<input type="file">` upload in all relevant pages.
- **FR-8.** A scheduled job MUST delete tus upload records and corresponding S3 multipart parts for uploads that have been stalled for longer than `TUS_UPLOAD_TTL_HOURS` (default 48).

## 6. Non-Functional Requirements

- **Performance** — Each PATCH request MUST be processed in < 200 ms p95 excluding byte transfer time. Chunk size default 5 MB (S3 minimum part size); configurable.
- **Security** — Upload endpoints MUST require a valid JWT. The `upload_id` MUST be an opaque UUID unguessable by other users. Cross-user access to an upload MUST return 403.
- **Privacy & Compliance** — Chunk metadata logged without PII. FERPA: upload_id tied to authenticated user. GDPR: incomplete uploads purged within 30 days on account deletion.
- **Accessibility** — Uppy widget MUST provide `aria-live` progress announcements and keyboard-operable pause/resume controls. WCAG 2.1 AA.
- **Scalability** — tus state in Postgres (not in-memory) so any app replica can continue a paused upload. Up to 1 000 concurrent active uploads supported.
- **Reliability** — Idempotent PATCH: re-sending a chunk at the same offset MUST be a no-op. S3 part numbers stored to allow re-issue of failed parts.
- **Observability** — Metrics: `tus_upload_created_total`, `tus_upload_completed_total`, `tus_upload_bytes_received_total`, `tus_upload_stalled_cleaned_total`. Log `upload_id`, `course_id`, `user_id` on every event.
- **Maintainability** — tus handler in `server/src/routes/tus.rs`; S3 multipart logic in `server/src/storage/`.
- **Internationalization** — Uppy i18n locale strings externalised via `@uppy/locales`.
- **Backward compatibility** — Existing `POST /courses/:id/files` endpoint retained for small files (≤ 10 MB) to avoid breaking API consumers.

## 7. Acceptance Criteria

- **AC-1.** *Given* an instructor begins uploading a 200 MB video *When* the network is interrupted at 50 % *Then* the Uppy widget shows "Paused" and on reconnect resumes from byte offset 104 857 600 without re-sending the first 100 MB.
- **AC-2.** *Given* an upload is created with `Upload-Length: 1073741824` (1 GB) *When* quota check determines the course is at its storage limit *Then* the server returns `403 Forbidden` with body `{ "error": "quota_exceeded" }` and no S3 multipart upload is initiated.
- **AC-3.** *Given* a tus upload has been stalled for 49 hours *When* the cleanup job runs *Then* the S3 multipart upload is aborted and the `storage.tus_uploads` row is deleted.
- **AC-4.** *Given* two users authenticated separately *When* user B sends a PATCH to user A's `upload_id` *Then* the server returns 403.
- **AC-5.** *Given* a completed upload *When* the S3 `CompleteMultipartUpload` call returns success *Then* a `file.uploaded` domain event is emitted and the `storage.objects` row is created.
- **AC-6.** *Given* a 5 MB file uploaded via the legacy `POST /courses/:id/files` endpoint *When* the file is saved *Then* the response is identical to the pre-8.2 shape (no tus headers required).

## 8. Data Model

```sql
-- server/migrations/075_tus_uploads.sql
CREATE TABLE storage.tus_uploads (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES user_account(id),
  course_id       UUID REFERENCES course.courses(id),
  s3_upload_id    TEXT,
  object_key      TEXT NOT NULL,
  upload_length   BIGINT NOT NULL,
  upload_offset   BIGINT NOT NULL DEFAULT 0,
  mime_type       TEXT,
  filename        TEXT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  completed_at    TIMESTAMPTZ,
  expires_at      TIMESTAMPTZ NOT NULL
);
CREATE INDEX ON storage.tus_uploads (user_id, completed_at);
CREATE INDEX ON storage.tus_uploads (expires_at) WHERE completed_at IS NULL;
```

## 9. API Surface

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/tus/files` | JWT | Create upload; returns `Location` header |
| `PATCH` | `/tus/files/:id` | JWT | Send chunk; `Content-Type: application/offset+octet-stream` |
| `HEAD` | `/tus/files/:id` | JWT | Query offset / length |
| `DELETE` | `/tus/files/:id` | JWT | Terminate upload |

All responses include `Tus-Resumable: 1.0.0` header per spec.

Rate limit: 60 PATCH requests/min/user to mitigate abuse.

## 10. UI / UX

- Replace `<input type="file">` in `clients/web/src/pages/lms/CourseModuleAssignmentPage.tsx` and file-upload modals with Uppy Dashboard component.
- Key flows: (1) Select file → Upload starts automatically. (2) Network drop → "Paused" badge + manual Resume button. (3) Upload complete → thumbnail or filename badge appears inline.
- Empty state: drag-and-drop zone with icon and "Drop files here or click to browse" copy.
- Loading state: per-file progress bar with percentage and ETA.
- Error state: per-file error message with "Retry" button.
- Mobile: Uppy supports camera capture on iOS/Android via `@uppy/webcam`.
- Accessibility: Uppy Dashboard has built-in focus management; validate `aria-live` region for screen-reader announcements. All controls keyboard-operable (Tab, Enter, Space).
- i18n keys: `uppy.dropFilesHere`, `uppy.uploadComplete`, `uppy.uploadFailed`, `uppy.resume`, `uppy.pause`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| S3 part number limit (10 000 parts) exceeded | L | M | Enforce minimum chunk size of 5 MB; warn at creation time |
| tus state table grows unbounded | M | M | TTL index + nightly cleanup job; alert if table > 100 k rows |
| Browser memory pressure from large chunk buffers | M | H | Stream chunks with Fetch API ReadableStream; do not buffer in JS |
| CORS preflight failures for tus headers | M | M | Configure Axum CORS to allow `Tus-Resumable`, `Upload-Offset`, `Upload-Length` |

## 15. Rollout Plan

- **Flag:** `FEATURE_TUS_UPLOADS=true` (default false; enable in staging first).
- **Sequencing:** deploy schema → deploy backend tus handler → deploy Uppy frontend behind flag → validate upload/resume cycle → flip flag GA.
- **Dogfood:** internal instructor accounts uploading lecture materials.
- **GA criteria:** zero upload failures on files > 50 MB in 7-day window; p95 resume latency < 2 s.
- **Rollback:** disable `FEATURE_TUS_UPLOADS`; legacy upload endpoint remains active.

## 16. Test Plan

- **Unit** — tus offset arithmetic, part number calculation, TTL expiry logic, quota enforcement.
- **Integration** — MinIO testcontainer: full tus lifecycle (create → chunk → complete → presign); abort on stall; concurrent resume from two replicas.
- **End-to-end** — Playwright: upload 50 MB file, simulate network drop via `page.route` intercept, confirm resume from correct offset.
- **Security** — Cross-user upload_id access returns 403; unauthenticated PATCH returns 401.
- **Accessibility** — Axe scan on Uppy Dashboard; screen-reader walkthrough of progress announcements.
- **Performance / load** — k6: 200 concurrent uploads at 5 MB/s each; confirm no OOM on app server.
- **Manual exploratory** — Test pause/resume on throttled 3G in Chrome DevTools; test on iOS Safari.
