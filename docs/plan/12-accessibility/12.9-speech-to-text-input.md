## 1. Problem Statement

Students with dysgraphia, motor impairments, or speech-as-primary communication modality are blocked from composing assignment responses, quiz short-answer fields, and discussion posts in Lextures because the platform offers no voice input pathway. These students typically rely on OS dictation tools (Apple Dictation, Windows Speech Recognition, Dragon NaturallySpeaking) which interact poorly with custom rich-text editors like TipTap. A native speech-to-text integration that works within the block editor (`clients/web/src/components/editor/block-editor/`) and short-answer quiz fields provides a direct accommodation pathway for IEP/504-documented needs (cross-ref §2.11, §12.10) and broadens the platform's appeal for self-learners preferring voice-first input on mobile devices.

## 2. Goals

- Implement a microphone dictation button in the TipTap block editor and short-answer quiz input fields.
- Use the Web Speech Recognition API for phase 1 (browser-native, no cost, no PII exposure).
- Insert recognized text at the current cursor position in real time (interim results).
- Provide STT status feedback (listening, processing, error) accessible to both sighted and screen-reader users.
- Allow STT to be activated automatically as an accommodation (§12.10).

## 3. Non-Goals

- Grading of oral spoken responses (§19.6 AI voice grading).
- Continuous ambient dictation outside of explicit input fields.
- Wake-word activation.
- Custom language model fine-tuning for specialized vocabulary.

## 4. Personas & User Stories

- **As a student with dysgraphia**, I want to dictate my essay response by speaking rather than typing so that I can express my knowledge without being limited by my writing impairment.
- **As a student with a motor impairment**, I want to dictate quiz short-answer responses so that I can complete assessments in the same time window as other students without extra keyboard effort.
- **As a self-learner on mobile**, I want to voice-compose my discussion post so that I can interact with the feed hands-free while commuting.
- **As an instructor**, I want to configure speech-to-text as a student's accommodation so that it is always available without the student having to enable it manually.

## 5. Functional Requirements

- **FR-1.** A microphone button MUST appear in the TipTap block editor toolbar and in short-answer (`<textarea>`) quiz question inputs when STT is enabled (WCAG best practice; accommodation support).
- **FR-2.** When the microphone button is activated, the system MUST use the Web Speech Recognition API (`window.SpeechRecognition`) and insert interim results as grey text and final results as committed black text at the current cursor position.
- **FR-3.** The system MUST provide a visible and screen-reader-accessible status indicator: "Listening…" (active), "Processing…" (network result), "Stopped" (idle). Status MUST be announced via `aria-live="polite"` region (WCAG SC 4.1.3).
- **FR-4.** The system MUST support continuous dictation mode (user keeps speaking until explicitly stopping) and single-utterance mode (auto-stops after a pause).
- **FR-5.** The microphone button MUST be keyboard-activatable (Enter/Space toggles recording) and labeled "Start dictation" / "Stop dictation" (WCAG SC 2.1.1, 4.1.2).
- **FR-6.** If the browser does not support `SpeechRecognition`, the microphone button MUST be hidden and a tooltip displayed on hover/focus: "Dictation is not supported on this browser. Use Chrome or Edge." (no degraded state shown; graceful absence).
- **FR-7.** The system MUST request microphone permission once with an accessible permission dialog explaining why the permission is needed, before starting any recording.
- **FR-8.** A server-side STT fallback (via OpenAI Whisper or Google Speech-to-Text API) MUST be available as a POST endpoint for browsers lacking native STT support or for higher accuracy in phase 2.

## 6. Non-Functional Requirements

- **Performance** — Interim results must appear within 200 ms of speech onset. Final results committed within 500 ms of utterance end.
- **Security** — Browser STT sends audio to the browser vendor's speech service; users must be notified of this. Server-side STT fallback must be rate-limited (30 requests/user/hour) and audio not retained after transcription (zero-retention API option required).
- **Privacy & Compliance** — Web Speech API audio processed by browser vendor (Google for Chrome); Lextures does not receive audio. For server-side fallback, audio is PII — apply §19.11 PII proxy, no retention, disclosed in privacy policy (§10.3). COPPA: under-13 students require verifiable parental consent before microphone access (§10.2). FERPA: transcribed text stored as student response; same data controls as other student data.
- **Accessibility** — Microphone button itself must be keyboard-accessible, have a clear visual recording state (pulsing ring animation, paused in reduced-motion mode per §12.7), and aria-live announcements. Screen-reader users who already rely on OS dictation should not experience conflicts — test with VoiceOver + Dragon.
- **Scalability** — Web Speech API is client-side; no server load in phase 1.
- **Reliability** — Network outage during STT produces an error message "Dictation stopped: connection lost"; existing typed text is preserved.
- **Observability** — Metrics: `stt_activations_per_day`, `stt_fallback_api_calls`, `stt_error_rate`.
- **Maintainability** — STT logic isolated in `clients/web/src/lib/stt/useSpeechToText.ts`.
- **Internationalization** — `SpeechRecognition.lang` set to the user's locale; language selector available in STT settings.
- **Backward compatibility** — Feature is opt-in via toolbar button; no existing workflow affected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student clicks the microphone button in the block editor, *When* they speak "Hello world", *Then* "Hello world" appears at the cursor position in the editor within 500 ms of the utterance ending.
- **AC-2.** *Given* interim results mode is active, *When* a student speaks a long sentence, *Then* grey placeholder text updates incrementally as the student speaks, then commits to black text on utterance end.
- **AC-3.** *Given* NVDA is active, *When* the student activates the microphone button with Enter, *Then* NVDA announces "Dictation started. Listening." via the live region.
- **AC-4.** *Given* a browser without `window.SpeechRecognition`, *When* the page loads, *Then* the microphone button does not appear in the toolbar (no broken icon or error).
- **AC-5.** *Given* a student with a documented accommodation (§12.10) for speech-to-text, *When* they open a quiz short-answer field, *Then* the microphone button is auto-visible and a tooltip reads "Speech-to-text enabled by your accommodation plan."
- **AC-6.** *Given* a student's browser denies microphone permission, *When* they click the microphone button, *Then* an accessible error message appears: "Microphone permission denied. Check your browser settings." Focus moves to the message.
- **AC-7.** *Given* axe-core runs on the editor with the microphone button visible, *When* the check executes, *Then* zero violations are reported for the microphone button and status indicator.

## 8. Data Model

No new tables. Dictated text is stored as normal assignment response or quiz answer text. STT activation preference extended from §12.6/§12.8 `user_reading_preferences`:

```sql
-- server/migrations/078_reading_prefs_stt.sql
ALTER TABLE user_reading_preferences
    ADD COLUMN stt_enabled   BOOLEAN NOT NULL DEFAULT false,
    ADD COLUMN stt_language  TEXT NOT NULL DEFAULT 'en-US';
```

## 9. API Surface

- `GET/PATCH /api/me/reading-preferences` — extended with `stt_enabled`, `stt_language`.
- `POST /api/stt/transcribe` — server-side fallback; accepts multipart audio; returns `{ transcript: string }`. Rate-limited.

## 10. UI / UX

- **New components:** `clients/web/src/lib/stt/useSpeechToText.ts`, `clients/web/src/components/editor/block-editor/DictationButton.tsx`.
- **Modified:** `clients/web/src/components/editor/block-editor/MarkdownFormatToolbar.tsx` — add `<DictationButton>`. Short-answer quiz input wrappers.
- **User flows:** 1. Click mic → grant permission once → speak → text inserts. 2. Pause → resumes on re-click. 3. Error → message with recovery instructions.
- **Recording state indicator:** Pulsing red dot (paused if `prefers-reduced-motion`), `aria-live` status text.
- **Mobile:** Tap-to-dictate; browser STT works natively on iOS Chrome and Android Chrome.
- **Accessibility annotations:** `DictationButton` uses `role="button"` with `aria-pressed` and `aria-label` that updates dynamically.

## 11. AI / ML Considerations

- Phase 1: Browser Web Speech API (Google's servers for Chrome; no Lextures server involvement).
- Phase 2: Server-side Whisper for higher accuracy, offline mode, or non-English languages. Audio chunked and streamed to Whisper endpoint; transcription returned as text only; audio discarded immediately.
- PII: Transcribed text may contain PII; stored with same protections as other student responses.

## 12. Integration Points

- **Web Speech API** — browser-native.
- **§12.10 accommodations engine** — enables STT automatically.
- **§19.11 PII proxy** — required for server-side Whisper fallback.
- **Internal:** `clients/web/src/components/editor/block-editor/MarkdownFormatToolbar.tsx`, short-answer quiz components.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Web Speech API accuracy insufficient for STEM vocabulary | H | M | Phase 2 Whisper fallback; document limitation; allow post-dictation editing |
| Chrome-only Web Speech API limits cross-browser support | H | M | Server-side fallback for Firefox/Safari; clearly communicate browser recommendation |
| Student accidentally activates STT in class, broadcasting voice | M | M | Single-tap to start, single-tap to stop; clear visual indicator; no auto-start |
| COPPA violation if under-13 audio sent to Google | H | H | Use server-side Whisper (on-premise) for under-13 users; parental consent flow |

## 16. Test Plan

- **Unit:** `useSpeechToText` state machine (idle → listening → processing → idle).
- **Integration:** Permission denial flow; server-side fallback transcription.
- **End-to-end:** Playwright mocks `SpeechRecognition` event dispatch; verifies text insertion.
- **Security:** Rate limit on `/api/stt/transcribe`; auth required.
- **Accessibility:** axe-core; manual session verifying NVDA + DictationButton interaction.
- **Manual:** QA with actual voice input on Chrome and Edge.

## 17. Documentation & Training

## 18. Open Questions
