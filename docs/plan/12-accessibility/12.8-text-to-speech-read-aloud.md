## 1. Problem Statement

Read-aloud (text-to-speech) is one of the most frequently specified accommodations in K-12 IEPs for students with reading disabilities, dyslexia, visual impairments, and English Language Learner status. Lextures has no built-in TTS capability. Students must rely on OS screen readers or browser extensions — but those tools read navigation chrome alongside content, creating a poor experience. A native, in-context read-aloud that reads only the current content block, highlights each word as it is spoken, and provides simple playback controls (play/pause/speed) is a significant differentiator for the K-12 accessibility market and directly addresses IEP/504 accommodation requirements referenced in §2.11 and §12.10.

## 2. Goals

- Provide a per-page read-aloud button that speaks the main content region of any course content page.
- Highlight each sentence or word as it is spoken (synchronized highlighting).
- Support playback speed adjustment (0.75×, 1×, 1.25×, 1.5×, 2×).
- Allow voice selection (browser TTS voices available on device).
- Persist TTS preferences per user; enable automatic activation via the accommodations engine (§12.10).

## 3. Non-Goals

- Custom AI voice synthesis (ElevenLabs, etc.) — browser TTS is sufficient for phase 1.
- Read-aloud for video or audio content (§12.4 captions handle that).
- TTS in the block editor authoring view (instructor side).
- Real-time caption generation for live video (§6.4 virtual classroom scope).

## 4. Personas & User Stories

- **As a student with dyslexia**, I want the platform to read my assignment instructions aloud while highlighting each sentence so that I can follow along without losing my place.
- **As a student with visual impairment**, I want to listen to quiz questions and answer choices read aloud so that I can take assessments without a separate screen reader that reads all page chrome.
- **As an instructor**, I want to record a TTS preference as part of a student's digital accommodation plan so that read-aloud is always active for that student automatically.
- **As an English Language Learner student**, I want to slow down the reading speed so that I can understand content delivered in my non-primary language.

## 5. Functional Requirements

- **FR-1.** A "Read aloud" button MUST appear in the accessibility toolbar on `CourseModuleContentPage.tsx`, `CourseModuleQuizPage.tsx`, and `CourseSyllabus.tsx` pages (WCAG best practice; supports IEP accommodation).
- **FR-2.** When activated, the TTS engine MUST extract and read only the main content region (identified by `role="main"` or the content article element), excluding navigation, breadcrumbs, and sidebars.
- **FR-3.** The TTS engine MUST highlight each sentence in the content as it is being spoken, using a non-color-only indicator (background highlight + underline) (WCAG SC 1.4.1).
- **FR-4.** Playback controls MUST include: Play/Pause (keyboard shortcut: Alt+P), Restart, and speed selector (0.75×, 1×, 1.25×, 1.5×, 2×).
- **FR-5.** The system MUST use the Web Speech API (`SpeechSynthesis`) in phase 1, with voice selection from available device voices. A fallback to a server-side TTS API (e.g., Google Cloud TTS) MUST be implemented for browsers without `SpeechSynthesis` support.
- **FR-6.** TTS state (enabled by default, preferred speed, preferred voice) MUST be persisted per user via the reading preferences API (§12.6/§12.7 extension).
- **FR-7.** When TTS is reading, pressing Tab or clicking an interactive element MUST pause TTS automatically to avoid interference with navigation.
- **FR-8.** The TTS playback controls MUST be fully keyboard-navigable and screen-reader-announced (WCAG SC 2.1.1, 4.1.2 — controls must not conflict with the screen reader's own speech).

## 6. Non-Functional Requirements

- **Performance** — TTS initialization must complete in < 500 ms for the first utterance. Sentence chunking must be done client-side with no server round-trip for Web Speech API path.
- **Security** — Server-side TTS fallback must not expose content to unauthorized users; use same authz as the content page. No content sent to third-party TTS API without FERPA/COPPA consent checks.
- **Privacy & Compliance** — Web Speech API (browser-native) does not send data to external servers. Server-side fallback path must comply with §10.2 (COPPA), §10.1 (FERPA). WCAG 2.1 SC 1.3.3 (Sensory Characteristics — don't rely on sound alone), SC 1.4.1 (word highlight must supplement audio). IDEA IEP accommodation alignment.
- **Accessibility** — The read-aloud feature must not conflict with screen readers. When a screen reader is detected (via `navigator.userAgent` heuristic or explicit user toggle), TTS should be disabled by default. Playback controls must have `aria-label` and `aria-pressed` states. Running TTS does not remove focus from current element.
- **Scalability** — Web Speech API is client-side; no server load. Server-side fallback batches by paragraph to manage API quota.
- **Reliability** — If Web Speech API or server TTS unavailable, show user-friendly message "Read aloud is not available on this browser. Try Chrome or Edge."
- **Observability** — Metrics: `tts_activations_per_day`, `avg_reading_speed_selected`, `fallback_api_calls`.
- **Maintainability** — TTS logic isolated in `clients/web/src/lib/tts/useTTS.ts`; content extraction in `clients/web/src/lib/tts/extractReadableContent.ts`.
- **Internationalization** — Voice selection should surface voices matching the content's language code. Speed labels localized.
- **Backward compatibility** — Feature is opt-in; no existing user workflow affected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student activates read-aloud on a content page, *When* TTS begins, *Then* the current sentence is highlighted with a yellow background within 100 ms of speech starting, and the highlight advances sentence-by-sentence.
- **AC-2.** *Given* a student presses Alt+P while TTS is playing, *When* the shortcut fires, *Then* TTS pauses immediately and the button state changes to "Play" (aria-pressed="false").
- **AC-3.** *Given* a student changes speed to 1.5×, *When* they reload the content page, *Then* TTS starts at 1.5× without the student re-selecting it.
- **AC-4.** *Given* NVDA is active alongside TTS, *When* the student activates read-aloud, *Then* the playback controls are still reachable by Tab and announced correctly by NVDA, and TTS speech does not overlap NVDA speech for control announcements.
- **AC-5.** *Given* a browser without `window.SpeechSynthesis` (e.g., older Firefox), *When* a student activates read-aloud, *Then* the system falls back to server-side TTS and plays the first paragraph within 2 seconds.
- **AC-6.** *Given* an accommodation plan activates read-aloud by default for a student (§12.10), *When* the student opens a content page, *Then* the read-aloud control shows "Reading enabled by your accommodation plan" and auto-plays after a 3-second delay (configurable).
- **AC-7.** *Given* axe-core runs on a content page with TTS controls visible, *When* the check executes, *Then* zero violations are reported for TTS control elements.

## 8. Data Model

Extends `user_reading_preferences` (§12.6, §12.7):

```sql
-- server/migrations/077_reading_prefs_tts.sql
ALTER TABLE user_reading_preferences
    ADD COLUMN tts_enabled        BOOLEAN NOT NULL DEFAULT false,
    ADD COLUMN tts_speed          NUMERIC(3,2) NOT NULL DEFAULT 1.0,
    ADD COLUMN tts_voice_name     TEXT;   -- null = default browser voice
```

## 9. API Surface

- `GET/PATCH /api/me/reading-preferences` — extended (§12.6 route, new fields).
- `POST /api/tts/synthesize` — server-side fallback; accepts `{ text: string, lang: string, speed: float }`; returns audio/mpeg stream. Rate-limited to 60 req/min/user.

## 10. UI / UX

- **New components:** `clients/web/src/components/a11y/ReadAloudControls.tsx` (floating bottom bar), `clients/web/src/lib/tts/useTTS.ts`, `clients/web/src/lib/tts/extractReadableContent.ts`.
- **Modified:** `clients/web/src/pages/lms/CourseModuleContentPage.tsx`, `CourseModuleQuizPage.tsx`, `CourseSyllabus.tsx` — mount `<ReadAloudControls>`.
- **User flows:** 1. Click "Read aloud" → TTS starts at sentence 1, sentence highlighted. 2. Adjust speed → immediate effect. 3. Pause → resume from same sentence. 4. Page change → TTS stops; resumes on next page if auto-read enabled.
- **Accessibility annotations:** Controls bar is `role="toolbar"` with `aria-label="Read aloud controls"`.

## 11. AI / ML Considerations

- Phase 1: Web Speech API (browser-native, no AI cost). Phase 2: Google Cloud TTS or Amazon Polly for higher-quality voices; SSML for better pronunciation of math/code.
- Fallback path: Server TTS via §17.3 background job for async audio pre-generation of long content.

## 12. Integration Points

- **Web Speech API** — browser-native; no installation.
- **§12.10 accommodations engine** — activates TTS automatically for eligible students.
- **§17.3 background jobs** — optional async pre-generation of audio for long content.
- **Internal:** `clients/web/src/lib/tts/`, `server/src/routes/` (new TTS route).

## 13. Dependencies & Sequencing

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Web Speech API voice quality inconsistent across platforms | H | M | Phase 2 plan for server-side high-quality voice; document limitation in help center |
| TTS and screen reader speech conflict | H | H | Default TTS off when screen reader heuristics detected; explicit user toggle |
| Sentence chunking inaccurate for math or code content | M | M | Skip MathML and code blocks in text extraction; announce "Math equation" / "Code block" |

## 15. Rollout Plan

- Feature flag: `ff_read_aloud` (default `false`).
- Sequencing: Browser TTS path → highlighting → controls UI → preferences API → flip flag.
- Dogfood: Internal team + 2 students with reading disabilities (recruit through SEN coordinator contacts).
- GA: All 7 ACs pass; feedback from user testing incorporated.

## 16. Test Plan

- **Unit:** `extractReadableContent` returns correct text; sentence chunking edge cases.
- **Integration:** TTS preferences round-trip; server-side fallback route.
- **End-to-end:** Playwright — activate TTS → verify highlight class on sentence element.
- **Accessibility:** axe-core on page with TTS controls; manual session with NVDA user.
- **Performance:** Time-to-first-utterance < 500 ms on Chrome.
- **Manual:** Test with VoiceOver on iOS (most common K-12 device).
