## 1. Problem Statement

Approximately 15–20% of the population has dyslexia, and it is one of the most common learning disabilities addressed in K-12 IEPs. Lextures currently offers only one font (Plus Jakarta Sans, set in `clients/web/src/index.css`) with no learner-side controls for font face, letter spacing, line height, or a reading ruler/focus overlay. Research-backed interventions (OpenDyslexic font, increased letter spacing, wider line height, a typographic ruler isolating a reading band) significantly reduce decoding difficulty for dyslexic readers. Without these controls, Lextures fails to meet the spirit of WCAG 2.1 SC 1.4.12 (Text Spacing) and the specific accommodation expectations in K-12 IEP/504 plans (cross-ref 12.10, 2.11). Adding learner-controlled display preferences makes Lextures genuinely usable for a large, underserved student population and is a strong differentiator in the K-12 accessibility market.

## 2. Goals

- Provide a persistent, per-user reading preferences panel accessible from any content page.
- Offer at least three font choices including OpenDyslexic and a high-legibility sans-serif.
- Allow learners to independently adjust letter spacing, word spacing, and line height within WCAG SC 1.4.12 bounds.
- Implement a reading ruler (focus overlay) that highlights the current line and dims surrounding content.
- Ensure all preferences persist across sessions and apply site-wide.

## 3. Non-Goals

- Text-to-speech read-aloud (§12.8 handles that separately).
- High-contrast color themes (§12.7 owns that).
- Content-level Lexile/reading-level adaptation (§11.6 AI rewriting is separate).
- Font customization for the authoring editor (instructor-side TipTap output is fixed HTML).

## 4. Personas & User Stories

- **As a student with dyslexia**, I want to switch to OpenDyslexic font and increase line spacing so that I can read course content more comfortably without needing a separate browser extension.
- **As a student**, I want a reading ruler that follows my scroll position so that I can track the line I am reading and reduce visual crowding.
- **As a parent**, I want my child's dyslexia-friendly settings to be saved on the server so they apply on any device they log in from.
- **As a K-12 accessibility coordinator**, I want to confirm that a student's IEP-specified display accommodations are available in the platform so that I can include Lextures in the accommodation plan.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a Reading Preferences panel accessible from a persistent toolbar button on all content-rendering pages (`CourseModuleContentPage`, `CourseModuleQuizPage`, `CourseSyllabus`) (WCAG SC 1.4.12 support).
- **FR-2.** The font selector MUST offer at minimum: (a) Default (Plus Jakarta Sans), (b) OpenDyslexic 3, (c) Atkinson Hyperlegible, (d) System default (sans-serif).
- **FR-3.** Letter spacing MUST be adjustable in 3 steps (normal, wide +0.12 em, wider +0.35 em) per WCAG SC 1.4.12.
- **FR-4.** Word spacing MUST be adjustable in 3 steps (normal, wide +0.16 em, wider +0.35 em) per WCAG SC 1.4.12.
- **FR-5.** Line height MUST be adjustable in 3 steps (normal 1.5, tall 1.8, taller 2.0) per WCAG SC 1.4.12.
- **FR-6.** The reading ruler MUST render as a semi-transparent band of configurable height and color (minimum 2 presets: yellow tint, grey tint) that follows the cursor's vertical position and stays within the text column (not covering sidebars).
- **FR-7.** All reading preferences MUST be persisted server-side in the `user_reading_preferences` table and loaded on every page render, applying before first paint to avoid layout shift.
- **FR-8.** The system MUST NOT require a page reload when the user changes any reading preference — changes apply in real time via CSS custom property updates (WCAG SC 1.4.12 operability).
- **FR-9.** The Reading Preferences panel MUST itself be fully keyboard-navigable and screen-reader-accessible (WCAG SC 2.1.1, 4.1.2).

## 6. Non-Functional Requirements

- **Performance** — CSS custom property updates on preference change must apply within one animation frame (< 16 ms). OpenDyslexic font served from CDN with `font-display: swap` to avoid FOIT. Server preference load must add < 20 ms to page TTFB (cached in browser after first load).
- **Security** — Reading preferences are user-specific; only the authenticated user can read/write their own preferences. Admin can view (not override) a student's preferences for support purposes.
- **Privacy & Compliance** — Preferences are non-sensitive user data. WCAG 2.1 SC 1.4.12 (Text Spacing). WCAG 2.1 SC 1.4.4 (Resize Text) — text spacing changes must not cause content loss or overlap. IDEA/IEP alignment: system accommodates documented accessibility needs without requiring third-party tools. COPPA: student preferences data storage follows §10.2 data-minimization.
- **Accessibility** — Ironically, the preferences panel itself must be accessible! All sliders/buttons must have `aria-label`, current value announced. Reading ruler must not obscure any interactive element (z-index managed). Font face changes must not reduce contrast below 4.5:1 (axe-core color-contrast rule still passes).
- **Scalability** — One preference row per user in `user_reading_preferences`; lightweight.
- **Reliability** — If preference load fails, fall back silently to default typography; no error state shown.
- **Observability** — Metrics: `reading_preferences_active_users`, `font_choice_distribution`, `ruler_enabled_pct`.
- **Maintainability** — Preferences applied via a single CSS custom property block on `<html>` element, manipulated by `clients/web/src/hooks/useReadingPreferences.ts`.
- **Internationalization** — Font availability and legibility may vary by language (e.g., OpenDyslexic has limited CJK coverage); guard with font-supported locale check.
- **Backward compatibility** — Default values match current typography; users who never change preferences see no change.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student selects OpenDyslexic font in the Reading Preferences panel, *When* the preference is saved, *Then* the font applies immediately to all course content text without page reload, and persists on next login on a different browser.
- **AC-2.** *Given* a student enables the reading ruler, *When* they scroll through a content page, *Then* a highlighted band follows the vertical scroll position, covering approximately 3 lines, and does not obscure navigation elements.
- **AC-3.** *Given* a student sets line height to "Taller", *When* they navigate to the quiz-take page, *Then* all question text and answer options render with line height 2.0 and no text overlaps any adjacent element.
- **AC-4.** *Given* NVDA is active, *When* a student opens the Reading Preferences panel with Tab, *Then* NVDA reads "Reading Preferences panel" and announces each control label and current value as the user navigates.
- **AC-5.** *Given* letter spacing is set to "Wide" (+0.12 em), *When* axe-core runs on the content page, *Then* zero `color-contrast` or layout-overlap violations are introduced vs. the default state.
- **AC-6.** *Given* an accessibility coordinator configures OpenDyslexic font as a student's IEP accommodation (via the accommodations engine, §12.10), *When* the student logs in, *Then* OpenDyslexic is pre-selected and the Reading Preferences panel shows "Set by your accommodation plan".
- **AC-7.** *Given* the reading preferences API is unreachable, *When* a student loads a content page, *Then* the page renders with default typography and no error banner is shown.

## 8. Data Model

```sql
-- server/migrations/075_user_reading_preferences.sql
CREATE TABLE user_reading_preferences (
    user_id           UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    font_face         TEXT NOT NULL DEFAULT 'default',   -- default | open-dyslexic | atkinson | system
    letter_spacing    TEXT NOT NULL DEFAULT 'normal',    -- normal | wide | wider
    word_spacing      TEXT NOT NULL DEFAULT 'normal',
    line_height       TEXT NOT NULL DEFAULT 'normal',    -- normal | tall | taller
    ruler_enabled     BOOLEAN NOT NULL DEFAULT false,
    ruler_color       TEXT NOT NULL DEFAULT 'yellow',    -- yellow | grey
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Backfill: no existing rows; table is additive.

## 9. API Surface

- `GET /api/me/reading-preferences` — returns current preferences; called on app init.
- `PATCH /api/me/reading-preferences` — updates one or more fields; debounced 500 ms on change.

## 10. UI / UX

- **New component:** `clients/web/src/components/a11y/ReadingPreferencesPanel.tsx` — floating panel triggered by a persistent "Aa" button in the top navigation bar (`clients/web/src/components/layout/`).
- **New hook:** `clients/web/src/hooks/useReadingPreferences.ts` — applies CSS custom properties to `<html>`.
- **New CSS variables in `index.css`:** `--reading-font`, `--reading-letter-spacing`, `--reading-word-spacing`, `--reading-line-height`.
- **User flows:** 1. Student clicks "Aa" button → panel opens → selects OpenDyslexic → content updates live. 2. Student enables reading ruler → band appears → adjusts color. 3. Student reloads page → preferences load from server → apply before paint.
- **Empty / loading:** Panel shows skeleton on first load if preferences haven't resolved.
- **Mobile:** Panel collapses to full-screen bottom sheet on mobile viewport.
- **Accessibility annotations:** Panel is `role="dialog"` with `aria-label="Reading Preferences"`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| OpenDyslexic font load increases LCP on slow connections | M | M | `font-display: swap`; preload on content pages only |
| Letter/word spacing changes cause text overflow in fixed-height containers | M | M | Audit all fixed-height text containers; convert to `min-height` |
| Reading ruler z-index conflicts with modals and dropdowns | M | L | Use `z-index` layer system; ruler below interactive overlays |
| IEP integration requires coordination with §12.10 which ships later | M | L | Design preference API to be IEP-writable from day 1; field `set_by_iec` reserved |

## 15. Rollout Plan

- Feature flag: `ff_reading_preferences` (default `false`).
- Sequencing: DB migration → preference API → CSS hooks → panel UI → flip flag.
- Dogfood: Test with internal team members; recruit 2–3 students with dyslexia for usability feedback.
- GA: All 7 ACs pass; usability testing with at least 3 dyslexic users.
- Rollback: Feature flag off; CSS custom properties fall back to defaults.

## 16. Test Plan

- **Unit:** `useReadingPreferences` applies correct CSS variables; preference API PATCH round-trip.
- **Integration:** GET/PATCH reading-preferences routes; accommodations engine override.
- **End-to-end:** Playwright — open panel → change font → reload → verify font persists.
- **Security:** User can only read/write own preferences.
- **Accessibility:** Panel keyboard navigation; axe-core on content page with all preference combinations.
- **Performance:** LCP delta with OpenDyslexic vs. default font < 200 ms on simulated 4G.
- **Manual:** Usability session with dyslexic users (recruit via accessibility advocacy contacts).

## 18. Open Questions

1. Should font choice affect the TipTap editor itself (instructor authoring) or only rendered content?
2. Is there demand for a per-course override (instructor picks a forced font for a specific lesson)?
3. Should the reading ruler be an SVG overlay or a CSS `box-shadow` technique? CSS approach is simpler; SVG allows cursor-following.
