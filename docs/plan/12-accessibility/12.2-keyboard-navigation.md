## 1. Problem Statement

Three key interactive surfaces in Lextures — module reordering via dnd-kit (`clients/web/src/pages/lms/CourseModules.tsx`), the quiz-take flow (`clients/web/src/pages/lms/CourseModuleQuizPage.tsx`), and the course feed composer (`clients/web/src/components/feed/FeedComposer.tsx`) — lack verified keyboard-only operability. Students and instructors who cannot use a pointing device are effectively blocked from these workflows. WCAG 2.1 AA SC 2.1.1 (Keyboard) and SC 2.1.2 (No Keyboard Trap) mandate that all functionality be reachable without a mouse. With K-12 districts requiring Section 508 attestations and IEP-mandated keyboard accommodations for students with motor impairments (cross-ref 12.10, 2.11), this is a procurement and legal compliance blocker.

## 2. Goals

- Ensure 100% of actions in module reorder, quiz-take, and feed surfaces are reachable and operable by keyboard alone.
- Eliminate keyboard traps in all three surfaces.
- Provide visible, high-contrast focus indicators meeting WCAG 2.2 SC 2.4.11 on every interactive element.
- Establish documented keyboard interaction patterns (published in help center) for all three surfaces.
- Integrate keyboard navigation regression tests into the Playwright CI suite.

## 3. Non-Goals

- Keyboard navigation for surfaces not listed (gradebook grid covered in 12.1, TipTap editor covered in 12.1).
- Custom keyboard shortcut designer (out of scope for this iteration).
- Mobile touchscreen equivalence (§7.2 owns responsive/touch work).

## 4. Personas & User Stories

- **As a student with a motor impairment**, I want to answer all quiz questions and submit my attempt using only Tab, Enter, Space, and arrow keys so that I can complete assessments independently.
- **As an instructor using a keyboard-only workflow**, I want to reorder course modules by pressing Space to grab, arrow keys to move, and Enter to drop so that I can organize course content without a mouse.
- **As a student**, I want to compose and post a message to the course feed using only a keyboard so that I can participate in discussions regardless of my input device.
- **As an accessibility coordinator**, I want keyboard interaction documented so I can include it in our assistive-technology compatibility guide.

## 5. Functional Requirements

- **FR-1.** The dnd-kit module list in `CourseModules.tsx` MUST support keyboard-based drag-and-drop using dnd-kit's `KeyboardSensor`: Space/Enter to pick up an item, Arrow Up/Down to move, Space/Enter to drop, Escape to cancel (WCAG SC 2.1.1).
- **FR-2.** During a keyboard drag operation, focus MUST remain on the dragged item and the item's position MUST be announced via a `role="status"` live region (e.g., "Module 'Week 1 Intro' — position 2 of 8") (WCAG SC 4.1.3).
- **FR-3.** The quiz-take flow MUST allow a student to: (a) Tab to each question, (b) select MC/TF answers via Space/Enter on radio/checkbox inputs, (c) Tab to and activate "Next Question" and "Submit" buttons, (d) complete the full attempt without mouse interaction (WCAG SC 2.1.1).
- **FR-4.** No element in the quiz-take, module-reorder, or feed surfaces MUST create a keyboard trap (WCAG SC 2.1.2). All modal dialogs MUST implement the ARIA APG Dialog pattern with Escape-to-close and return-of-focus to trigger.
- **FR-5.** The feed composer (`FeedComposer.tsx`) MUST support keyboard-accessible rich-text entry, emoji picker navigation, file attachment via keyboard, and form submission via keyboard shortcut (Ctrl/Cmd+Enter).
- **FR-6.** Every focusable element in these three surfaces MUST display a visible focus indicator with a contrast ratio of at least 3:1 against adjacent colors (WCAG 2.2 SC 2.4.11).
- **FR-7.** The quiz timer (when present, per §2.7) SHOULD be announced to screen readers as a live region at defined intervals and when critical thresholds are reached (WCAG SC 4.1.3, best practice).
- **FR-8.** The system MUST NOT rely solely on color to indicate focus state; shape, outline, or underline MUST also differentiate the focused element (WCAG SC 1.4.1).

## 6. Non-Functional Requirements

- **Performance** — KeyboardSensor event handlers must not block the main thread for > 16 ms per frame during drag operations.
- **Security** — No new auth surface. Keyboard interaction does not bypass quiz attempt authorization in `server/src/routes/courses.rs`.
- **Privacy & Compliance** — WCAG 2.1 AA SC 2.1.1, 2.1.2, 2.4.3, 2.4.7, 2.4.11 (WCAG 2.2). Section 508 §1194.21(a). EN 301 549 clause 9.2.1.1.
- **Accessibility** — This plan IS the accessibility delivery. All three surfaces must pass axe-core keyboard-navigation rules (aria-required-children, aria-allowed-attr, scrollable-region-focusable). NVDA + Chrome and VoiceOver + Safari manual sessions required per surface. JAWS + Chrome verification for quiz-take (high-stakes testing common in K-12). Focus indicator must be verified visually by at least two QA testers with and without high-contrast OS mode enabled.
- **Scalability** — Keyboard handling is client-side; no server load implications.
- **Reliability** — Keyboard interactions must degrade gracefully if JS partially fails; native HTML form elements for quiz answers preferred over custom widgets where possible.
- **Observability** — Playwright tests report keyboard-navigation pass/fail in CI; Pa11y runs on quiz-take and feed routes.
- **Maintainability** — `KeyboardSensor` configuration extracted into `clients/web/src/lib/dnd/keyboardSensorConfig.ts` for reuse across any future dnd-kit surface.
- **Internationalization** — Live region announcement strings must use i18n keys (e.g., `a11y.dnd.position`, `a11y.dnd.grabbed`, `a11y.dnd.dropped`).
- **Backward compatibility** — Adding `KeyboardSensor` alongside existing `PointerSensor` and `TouchSensor` is additive; no existing drag behavior is broken.

## 7. Acceptance Criteria

- **AC-1.** *Given* a keyboard-only user on `CourseModules.tsx`, *When* they press Tab to reach a module drag handle, press Space, press Arrow Down twice, and press Space, *Then* the module moves two positions down, the DOM order updates immediately, and a live region announces "Module moved to position 3 of 8".
- **AC-2.** *Given* a quiz with 5 multiple-choice questions, *When* a keyboard-only student tabs through the entire quiz and presses Enter on "Submit", *Then* the attempt is recorded server-side and the student sees their score — no mouse interaction required at any step.
- **AC-3.** *Given* NVDA 2024.1 + Chrome is active, *When* a student navigates the quiz-take page by Tab key, *Then* NVDA reads each question text, option label, and current selection state without silence or repetition.
- **AC-4.** *Given* the feed composer is focused, *When* a user presses Tab through all toolbar controls and Shift+Tab back, *Then* focus cycles predictably and no keyboard trap occurs (WCAG SC 2.1.2 verified).
- **AC-5.** *Given* axe-core Playwright runs against `/lms/courses/:id/modules`, `/lms/courses/:id/quizzes/:id/take`, and `/lms/courses/:id/feed`, *When* the CI pipeline executes, *Then* zero keyboard-related violations (scrollable-region-focusable, focus-order-semantics) are reported.
- **AC-6.** *Given* a JAWS 2024 session on the quiz-take page, *When* a student selects an MC answer via the Space key, *Then* JAWS announces the selected option label and "selected" state change.
- **AC-7.** *Given* high-contrast mode is enabled on Windows, *When* a user tabs through module drag handles, *Then* focus indicators remain visible (verified visually by QA tester).

## 8. Data Model

No new database tables. The dnd-kit `order` field update on module reorder is an existing API call; no schema changes needed.

## 9. API Surface

No new routes. Existing PUT `/api/courses/:id/modules/:moduleId` (reorder) and POST `/api/quiz-attempts` (submit) are unchanged.

## 10. UI / UX

- **Modified components:**
  - `clients/web/src/pages/lms/CourseModules.tsx` — add `KeyboardSensor` to dnd-kit, `role="status"` live region.
  - `clients/web/src/pages/lms/CourseModuleQuizPage.tsx` — audit and fix tab order, ensure native `<input type="radio">` or ARIA-compliant custom radio.
  - `clients/web/src/components/feed/FeedComposer.tsx` — add keyboard shortcut for submit, audit toolbar tab order.
- **New component:** `clients/web/src/components/a11y/LiveRegion.tsx` (shared with 12.1 if that ships first).
- **User flows:**
  1. Keyboard module reorder: Tab → focus drag handle → Space to grab → Arrow keys → Space to drop.
  2. Keyboard quiz-take: Tab through questions → Space/Enter to select → Tab to Submit → Enter.
  3. Keyboard feed post: Tab to composer → type → Ctrl+Enter to post.
- **Empty / loading / error states:** Loading spinners must be focusable and announce "Loading" to screen readers. Error messages must receive focus programmatically.
- **Mobile / responsive:** Keyboard sensor is desktop-only; touch equivalents handled by §7.2.
- **Accessibility annotations:** Focus order diagrams for each surface added to design system docs.
- **Copy & i18n keys:** `a11y.dnd.grabbed`, `a11y.dnd.moved`, `a11y.dnd.dropped`, `a11y.dnd.cancelled`, `a11y.quiz.questionN`, `a11y.feed.submitShortcut`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| dnd-kit KeyboardSensor conflicts with existing pointer drag in CourseModules | M | H | Test both sensors in parallel; isolate sensor activation logic |
| Quiz-take uses custom radio widgets that are not keyboard-accessible | H | H | Audit component tree; replace with native `<input type="radio">` wrapped in accessible label |
| Feed composer TipTap toolbar has non-keyboard-accessible dropdowns | M | M | Apply APG Toolbar pattern with roving tabindex |
| Focus indicator CSS overridden by Tailwind `focus:outline-none` utility | H | M | Audit all `outline-none` usages in components; replace with `focus-visible:ring-*` |

## 15. Rollout Plan

- Feature flag: `ff_a11y_keyboard_nav_phase1` (default: `false`).
- Sequencing: Fix dnd-kit → fix quiz-take → fix feed → flip flag.
- Dogfood: Internal QA keyboard-only testing sessions.
- GA criteria: All 7 ACs pass; zero axe keyboard violations in CI.
- Rollback: Feature flag reverts to `false`; changes are purely frontend.

## 16. Test Plan

- **Unit** — Vitest tests for `keyboardSensorConfig.ts`, LiveRegion rendering.
- **Integration** — Playwright keyboard navigation scripts for module reorder, quiz-take, feed composer.
- **End-to-end** — Full quiz attempt (5 questions) via keyboard only.
- **Security** — Standard authz check: keyboard submit cannot bypass quiz attempt ownership.
- **Accessibility** — axe-core + Pa11y CI; manual NVDA, JAWS, VoiceOver sessions.
- **Performance / load** — Keyboard event handler profiling in Chrome DevTools; no regressions > 5 ms.
- **Manual exploratory** — QA checklist: tab order, no traps, visible focus, announcement text.
