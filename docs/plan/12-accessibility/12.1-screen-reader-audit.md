## 1. Problem Statement

No screen-reader audit has ever been performed on Lextures. Four critical interactive surfaces — the TipTap-based block editor (`clients/web/src/components/editor/block-editor/`), the gradebook data grid (`clients/web/src/pages/lms/gradebook/GradebookGrid.tsx`), the dnd-kit drag-and-drop module reorder (`clients/web/src/pages/lms/CourseModules.tsx`), and the command palette (`clients/web/src/components/command-palette/CommandPaletteDialog.tsx`) — are known to have missing ARIA labels, incorrect focus management, and undiscoverable interactions for users of assistive technology. WCAG 2.1 AA conformance (referenced in MISSING_FEATURES.md §10.7) is required for US public-school procurement (Section 508) and EU contracts (EN 301 549); the absence of a VPAT (§10.8) is already a sales blocker. This audit sprint establishes a baseline conformance report, fixes P0 violations, and creates a regression harness so future changes do not silently re-introduce barriers.

## 2. Goals

- Produce a written conformance baseline report against WCAG 2.1 AA SC 1.1.1, 1.3.1, 1.3.2, 1.4.1, 2.1.1, 2.1.2, 2.4.3, 2.4.6, 4.1.2, and 4.1.3 for all four surfaces.
- Remediate all Critical and Serious violations (axe-core severity levels) found on those surfaces.
- Achieve zero axe-core Critical/Serious violations in CI for all four components.
- Verify compliance with VoiceOver (macOS/iOS), NVDA + Chrome (Windows), JAWS + Chrome (Windows), and TalkBack (Android) via scripted manual test protocols.
- Publish an internal VPAT draft referencing audit findings to unblock §10.8.

## 3. Non-Goals

- Full VPAT publication (§10.8 owns that deliverable).
- Audit of pages not listed (inbox, settings, syllabus) — covered by a follow-on sprint.
- Mobile-native app accessibility (§7.1 out of scope).
- Internationalized screen-reader strings (§11.1 owns i18n).

## 4. Personas & User Stories

- **As a blind student**, I want to compose and submit assignment responses in the TipTap block editor using only a keyboard and screen reader so that I can participate equally in course activities.
- **As a low-vision instructor**, I want to navigate the gradebook grid and update grades using NVDA + keyboard so that I do not require sighted assistance to perform my job.
- **As a student with motor impairment**, I want to reorder course modules using the keyboard (dnd-kit) without requiring a mouse so that I can manage my view of course content.
- **As an admin**, I want the command palette to announce search results and selected actions via screen reader so that I can operate the platform without mouse input.
- **As a district accessibility coordinator**, I want a written conformance report so that I can evaluate Lextures against our Section 508 procurement checklist.

## 5. Functional Requirements

- **FR-1.** The TipTap block editor MUST expose each block type (paragraph, heading, image, list, code) as a properly labelled ARIA region or landmark, with `role="textbox"` and `aria-multiline="true"` on the prose editing area (WCAG SC 4.1.2).
- **FR-2.** The gradebook grid MUST be marked up as an ARIA grid (`role="grid"`, `role="row"`, `role="gridcell"`, `role="columnheader"`) with appropriate `aria-sort`, `aria-label`, and `aria-rowcount`/`aria-colcount` attributes (WCAG SC 1.3.1, 4.1.2).
- **FR-3.** The dnd-kit drag-and-drop module reorder MUST provide a keyboard-operable alternative (Space/Enter to pick up, arrow keys to move, Space/Enter to drop) with live-region announcements of current position (WCAG SC 2.1.1, ARIA APG Drag-and-Drop Pattern).
- **FR-4.** The command palette dialog MUST trap focus while open, return focus to the trigger element on close, label the search input with `aria-label`, and announce results count via an `aria-live="polite"` region (WCAG SC 2.4.3, 4.1.3).
- **FR-5.** All interactive elements across these four surfaces MUST have visible focus indicators meeting WCAG 2.2 SC 2.4.11 (focus not obscured) with a minimum 3:1 contrast ratio for the focus ring.
- **FR-6.** All toolbar buttons and icon-only controls MUST have accessible names via `aria-label` or `<title>` elements (WCAG SC 4.1.2, 1.1.1).
- **FR-7.** The system SHOULD emit a `roving tabindex` pattern for the gradebook grid to limit tab-stop count when the grid has more than 10 columns.
- **FR-8.** All audit findings MUST be logged as GitHub Issues with axe-core rule ID, WCAG success criterion, severity, affected component file path, and reproduction steps.

## 6. Non-Functional Requirements

- **Performance** — ARIA attribute additions must not increase component render time by more than 2 ms (measured via React DevTools Profiler).
- **Security** — No new attack surface introduced; ARIA attributes are declarative and do not affect auth/authz.
- **Privacy & Compliance** — Audit report must not include real student PII; use anonymized test fixtures. Section 508 and EN 301 549 compliance assertions feed the VPAT (§10.8).
- **Accessibility** — This plan IS the accessibility delivery. All four surfaces must reach zero axe-core Critical/Serious violations. Manual test protocols must be run with VoiceOver (macOS Ventura + Safari), NVDA 2024.1 + Chrome 124, JAWS 2024 + Chrome 124, and TalkBack (Android 13 + Chrome). Each test session must be recorded (screen + audio) and attached to the audit report. WCAG 2.1 AA success criteria tested: 1.1.1, 1.3.1, 1.3.2, 1.3.3, 1.4.1, 2.1.1, 2.1.2, 2.4.3, 2.4.6, 2.4.7, 2.4.11 (2.2), 4.1.2, 4.1.3. Pa11y CI integration required in the `clients/web` pipeline at the `http://localhost:5173/lms/...` routes.
- **Scalability** — axe-core automated checks run in < 500 ms per page in CI.
- **Reliability** — axe-core Playwright integration must not produce flaky results; configure `waitForTimeout` and stable selectors.
- **Observability** — axe-core violation counts reported as a CI metric and surfaced in the PR check summary.
- **Maintainability** — ARIA fixes applied in co-located component files; no global hacks. New components must pass `eslint-plugin-jsx-a11y` rules enforced in `eslint.config.js`.
- **Internationalization** — All `aria-label` strings added through this work MUST be i18n-ready (string keys, not hard-coded English) to support §11.1.
- **Backward compatibility** — ARIA attribute additions are additive; no breaking API or prop changes to existing components.

## 7. Acceptance Criteria

- **AC-1.** *Given* the TipTap block editor is rendered in a test browser, *When* axe-core 4.x is run against `clients/web/src/components/editor/block-editor/BlockEditorShell.tsx` (rendered via Vitest + jsdom), *Then* zero Critical or Serious violations are reported.
- **AC-2.** *Given* a NVDA + Chrome session on the gradebook page, *When* a tester navigates via arrow keys, *Then* every cell announces column header + row header + cell value (e.g., "John Smith, Assignment 3, 87/100") and sort state changes announce "sorted ascending / descending".
- **AC-3.** *Given* a VoiceOver + Safari session on CourseModules.tsx, *When* a tester presses Space on a module drag handle, moves it with arrow keys, and presses Space to drop, *Then* VoiceOver announces "Grabbed Module 2. Current position 2 of 5. Dropped. Module 2 moved to position 3."
- **AC-4.** *Given* a JAWS 2024 session, *When* the command palette is opened (Cmd+K), *Then* JAWS announces the dialog label "Command Palette", focus lands on the search input, typing updates a live region reading "5 results", and pressing Escape returns focus to the triggering button.
- **AC-5.** *Given* the Pa11y CI task runs against the locally-served app at `/lms/courses/:id/modules`, *When* the CI pipeline executes, *Then* zero WCAG2AA violations are reported and the job passes.
- **AC-6.** *Given* the internal audit report is completed, *When* it is reviewed by the accessibility coordinator, *Then* it maps each of the 13 targeted WCAG 2.1 AA success criteria to "Supports", "Partially Supports", or "Does Not Support" with evidence for each of the four surfaces.
- **AC-7.** *Given* TalkBack (Android 13) is active, *When* a student navigates the gradebook grid by swiping, *Then* each cell value is read without silence or repetition, and interactive cells announce "double-tap to activate".

## 8. Data Model

No new database tables required. Audit findings are tracked in GitHub Issues (labeled `a11y`, `wcag-2.1-aa`). A JSON fixture file `clients/web/src/test/a11y-fixtures/` will hold mock course/gradebook data for repeatable audits.

## 9. API Surface

No new API routes. Existing routes consumed by audited components remain unchanged.

## 10. UI / UX

- **Modified components:**
  - `clients/web/src/components/editor/block-editor/BlockEditorShell.tsx` — add landmark roles, live regions.
  - `clients/web/src/components/editor/block-editor/BlockFloatingToolbar.tsx` — add `aria-label` to icon buttons.
  - `clients/web/src/pages/lms/gradebook/GradebookGrid.tsx` — add `role="grid"`, roving tabindex, sort announcements.
  - `clients/web/src/pages/lms/CourseModules.tsx` — add keyboard drag-and-drop via dnd-kit `KeyboardSensor`, live region.
  - `clients/web/src/components/command-palette/CommandPaletteDialog.tsx` — focus trap, live region, return-focus.
- **New component:** `clients/web/src/components/a11y/LiveRegion.tsx` — reusable `aria-live` wrapper.
- **User flows:**
  1. Instructor opens gradebook → navigates by arrow key → screen reader announces each cell.
  2. Student opens block editor → types in heading block → screen reader announces block type.
  3. Instructor reorders module via keyboard → live region confirms new position.
  4. Admin opens command palette → searches → selects result via Enter.
- **Accessibility annotations:** All focus order documented in a Figma a11y annotation layer (not shipped as code).
- **Copy & i18n:** New `aria-label` values added to `clients/web/src/lib/i18n/` namespace `a11y.*`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| ARIA grid pattern conflicts with existing TanStack Table virtualization in GradebookGrid | H | H | Spike with small virtualized grid prototype before full remediation; consider `aria-rowcount` + `aria-rowindex` on virtual rows |
| dnd-kit keyboard sensor breaks touch drag behavior | M | M | Run separate sensor lists for keyboard vs. pointer; test on mobile |
| Screen-reader test environment availability on Windows | M | M | Provision BrowserStack Automate session with NVDA/JAWS; document session credentials in 1Password |
| ARIA live region chatter overwhelms screen-reader users during fast navigation | M | M | Use `aria-live="polite"` not "assertive" except for errors; batch announcements |
| Remediation time underestimated (gradebook grid is complex) | H | M | Timebox each surface to 3 days; file issues for anything beyond Critical/Serious |

## 15. Rollout Plan

- Feature flag: `ff_a11y_aria_audit_phase1` (default: `false` in dev; `true` after QA sign-off).
- Migration sequencing: No schema changes. Deploy ARIA attribute changes behind flag → QA screen-reader sessions → flip flag to `true` → monitor axe-core CI metrics.
- Dogfood: Internal team uses screen-reader mode for one sprint on staging.
- GA criteria: Zero axe-core Critical/Serious violations in CI; audit report approved by accessibility coordinator.
- Rollback: Revert flag; ARIA additions are purely additive and safe to remove.

## 16. Test Plan

- **Unit** — `jest-axe` / `vitest-axe` checks on `BlockEditorShell`, `GradebookGrid`, `CommandPaletteDialog` render snapshots.
- **Integration** — Playwright `@axe-core/playwright` injectAxe + checkA11y on `/lms/courses/:id/modules`, `/lms/courses/:id/gradebook`, `/lms/courses/:id/feed`.
- **End-to-end** — Playwright happy paths: (a) keyboard-only gradebook navigation; (b) keyboard module reorder; (c) command palette open/close/select; (d) TipTap heading entry.
- **Security** — No new auth surface; standard authz matrix tests cover existing routes.
- **Accessibility** — Automated (axe-core, Pa11y) + manual screen-reader scripts for VoiceOver, NVDA, JAWS, TalkBack recorded sessions.
- **Performance / load** — axe-core must complete per-page in < 500 ms; measured in CI logs.
- **Manual exploratory** — QA checklist covering all 13 WCAG SCs for each of the 4 surfaces; sign-off by two testers.
