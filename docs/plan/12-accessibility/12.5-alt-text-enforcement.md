## 1. Problem Statement

The Lextures TipTap block editor (`clients/web/src/components/editor/block-editor/CourseAwareTipTapImage.tsx`) allows instructors to insert images into course content and assignment descriptions with no mechanism to require or suggest alternative text. Images published without `alt` attributes are invisible to blind and low-vision users, violating WCAG 2.1 AA SC 1.1.1 (Non-text Content). For K-12 districts under IDEA and HE institutions under the ADA, serving inaccessible image-heavy content creates legal exposure. This plan adds (a) inline alt-text enforcement with publish-blocking warnings, and (b) an AI-powered alt-text suggestion that generates a description of the image content, reducing the burden on instructors while improving accessibility quality.

## 2. Goals

- Require every image inserted in the block editor to have an alt attribute before content can be published.
- Provide AI-generated alt-text suggestions so instructors can accept, edit, or replace with one click.
- Allow instructors to mark images as decorative (empty alt), preventing false-positive warnings.
- Surface a course-level accessibility score showing percentage of images with alt text.
- Ensure the alt-text field itself is keyboard and screen-reader accessible.

## 3. Non-Goals

- Alt text for embedded third-party content (YouTube, Vimeo embeds) — deferred.
- Automatic application of AI alt text without instructor review — all AI suggestions require human confirmation.
- Alt text for SVG/icon images in the UI chrome (covered by 12.1).
- Alt text for images in student assignment submissions — out of scope for this sprint.

## 4. Personas & User Stories

- **As a blind student**, I want every image in my course content to have descriptive alt text so that I can understand the full meaning of the material without visual context.
- **As an instructor**, I want the editor to suggest alt text when I insert an image so that I do not have to write descriptions from scratch, especially for scientific diagrams or charts.
- **As an instructor**, I want to mark decorative images (e.g., a horizontal divider graphic) as decorative so that screen readers skip them without my having to write a placeholder.
- **As an accessibility coordinator**, I want to see which courses contain images without alt text so that I can prioritize remediation efforts.

## 5. Functional Requirements

- **FR-1.** When an image is inserted into the block editor, the system MUST prompt the author to enter alt text before the image is accepted into the document (WCAG SC 1.1.1).
- **FR-2.** The alt-text input field MUST be keyboard-accessible and screen-reader-labeled ("Alternative text for image") within the image toolbar/modal (WCAG SC 2.1.1, 4.1.2).
- **FR-3.** The system MUST display a warning banner in the editor when the current document contains images without alt text, listing the images and preventing publication until resolved or explicitly marked decorative.
- **FR-4.** The system MUST offer an AI-generated alt-text suggestion for each image. The suggestion MUST be generated by sending the image to a vision-capable model (e.g., `gpt-4o` via OpenRouter) and must be presented as a suggestion the instructor accepts, edits, or rejects.
- **FR-5.** Instructors MUST be able to mark an image as decorative (sets `alt=""` explicitly), which satisfies the WCAG SC 1.1.1 requirement for decorative images and clears the warning.
- **FR-6.** The AI suggestion MUST pass through the PII redaction proxy (§19.11) before the image is transmitted to the AI service to prevent inadvertent disclosure of student-created images.
- **FR-7.** The system SHOULD track `alt_text_coverage` per course (images with valid alt / total images) and surface it in the course settings accessibility panel.
- **FR-8.** Alt text MUST be stored with the image node in the TipTap document JSON so it is preserved on content export and import.

## 6. Non-Functional Requirements

- **Performance** — AI alt-text suggestion must return within 5 s (p95) to avoid blocking the authoring flow. If not available within 3 s, the editor proceeds without a suggestion and retries asynchronously.
- **Security** — Images sent to AI vision API must be subject to §19.11 PII redaction; image URLs must not expose signed storage tokens to client-visible AI prompts.
- **Privacy & Compliance** — WCAG 2.1 AA SC 1.1.1. IDEA (IEP accommodations for blind students require accessible content). FERPA/COPPA: images in student work must not be sent to AI services without consent (§10.2, §19.11). GDPR: image data is personal data if it contains faces; data-minimization required.
- **Accessibility** — The alt-text modal/toolbar itself must be fully keyboard-navigable (WCAG SC 2.1.1), announce current image label, and not trap focus. The warning banner must be announced by screen readers as a live region. VoiceOver and NVDA sessions required.
- **Scalability** — AI suggestion calls are made per image insertion; no batch needed in phase 1. Rate limit: 100 suggestions/user/hour.
- **Reliability** — Alt text enforcement does not depend on AI service availability; if AI is unavailable, the prompt appears without suggestion and the instructor enters text manually.
- **Observability** — Metrics: `alt_text_coverage_pct` per course, `ai_alt_suggestions_accepted_rate`, `ai_alt_suggestion_latency_ms`.
- **Maintainability** — Alt-text logic centralized in `clients/web/src/components/editor/block-editor/CourseAwareTipTapImage.tsx` and a new `server/src/services/ai/alt_text_ai.rs` service.
- **Internationalization** — Alt text field must accept and store Unicode. AI suggestions should be in the course's configured language.
- **Backward compatibility** — Existing published content with images lacking alt text is not retroactively blocked; the accessibility score surfaces the gap for remediation.

## 7. Acceptance Criteria

- **AC-1.** *Given* an instructor inserts an image into the block editor, *When* no alt text is entered, *Then* the editor shows an inline prompt "Add alt text (required for accessibility)" and the publish button is disabled until resolved.
- **AC-2.** *Given* the AI alt-text service is available, *When* an image is inserted, *Then* an AI-suggested description appears in the alt-text field within 5 seconds, and the instructor can accept, edit, or reject it.
- **AC-3.** *Given* a VoiceOver user on the block editor, *When* the alt-text prompt appears, *Then* VoiceOver announces "Alt text required for image. Enter description or mark as decorative." and the input receives focus.
- **AC-4.** *Given* an instructor marks an image as "decorative", *When* the content is published, *Then* the image has `alt=""` in the rendered HTML and the accessibility warning is cleared.
- **AC-5.** *Given* a course contains 10 images (8 with alt text, 2 without), *When* the instructor views Course Settings → Accessibility, *Then* the alt-text coverage reads "80% (8/10 images)" with links to the 2 uncovered images.
- **AC-6.** *Given* axe-core runs on a published course page, *When* all images have been given alt text through this feature, *Then* zero `image-alt` violations are reported.
- **AC-7.** *Given* the AI vision service is offline, *When* an instructor inserts an image, *Then* the alt-text prompt appears without a suggestion, the editor remains fully functional, and a non-blocking toast reads "AI suggestion unavailable — please enter alt text manually."

## 8. Data Model

Alt text is stored inside TipTap's document JSON (the `alt` attribute of the image node). No new table required. For coverage tracking:

```sql
-- server/migrations/074_image_alt_coverage.sql
-- Materialized view updated on content save:
CREATE MATERIALIZED VIEW course_image_alt_coverage AS
  SELECT course_id,
         COUNT(*) FILTER (WHERE alt_text IS NOT NULL AND alt_text != '') AS with_alt,
         COUNT(*) AS total
  FROM course_content_images   -- populated by content-save trigger/job
  GROUP BY course_id;
```

## 9. API Surface

- `POST /api/ai/alt-text` — sends image URL (or base64 thumbnail), returns `{ suggestion: string, confidence: float }`. Auth: instructor. Rate-limited.
- `GET /api/courses/:id/accessibility` — returns `{ alt_text_coverage: { with_alt: N, total: N } }`. Instructor/admin.

## 10. UI / UX

- **Modified:** `clients/web/src/components/editor/block-editor/CourseAwareTipTapImage.tsx` — add alt-text input panel, decorative toggle, AI suggest button.
- **New:** `clients/web/src/components/editor/block-editor/AltTextPanel.tsx`, `clients/web/src/pages/lms/CourseSettings.tsx` — add accessibility coverage widget.
- **User flows:** 1. Insert image → AI suggestion appears → instructor edits → saves → image published with alt text. 2. Instructor marks decorative → warning clears. 3. Instructor checks Course Settings Accessibility tab → sees coverage score.
- **Empty / loading states:** While AI suggestion loads, show spinner in alt-text field with text "Generating suggestion…".
- **Accessibility annotations:** Alt-text panel has `role="dialog"` with `aria-label="Image alternative text"`. AI suggest button has `aria-label="Generate alt text with AI"`.

## 11. AI / ML Considerations

- Model: GPT-4o vision or equivalent via OpenRouter; prompt: "Describe this image in 1–2 sentences suitable as screen-reader alt text for educational content."
- Eval metric: Human acceptance rate target > 60% without editing; track via `ai_alt_suggestions_accepted_rate`.
- Fallback: If AI unavailable or returns empty, prompt user to enter manually.
- PII: Pass through §19.11 proxy; images with detectable faces should not be sent to external AI.
- Cost budget: ~$0.002 per image; budget alert at $20/month per tenant.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| AI vision model produces inaccurate alt text for domain-specific diagrams (STEM) | H | M | Add "Edit before accepting" affordance; track edit rate; consider fine-tuning |
| Instructors dismiss prompts to meet deadlines, leaving images uncaptioned | H | M | Hard-block publication (soft-block with override is acceptable first step) |
| Images with student faces sent to third-party AI | M | H | Face-detection preflight check; if faces detected, require manual alt text |
| Coverage materialized view becomes stale | L | L | Refresh on every content save; add staleness alert if last refresh > 1 hour |

## 15. Rollout Plan

- Feature flag: `ff_alt_text_enforcement` (default `false`; soft-warn before hard-block).
- Phase 1: AI suggestion + inline warning (soft). Phase 2: Block publication (hard).
- Dogfood: Internal instructors use on staging for 1 sprint.
- GA: AC-1 through AC-7 pass; acceptance rate metric baseline established.
- Rollback: Feature flag off; no data loss.

## 16. Test Plan

- **Unit:** `AltTextPanel` renders with and without AI suggestion; decorative toggle sets `alt=""`.
- **Integration:** `POST /api/ai/alt-text` returns suggestion; coverage query returns correct ratio.
- **End-to-end:** Playwright — insert image → AI suggestion → accept → publish → axe-core zero violations.
- **Security:** Instructor-only can call AI alt-text endpoint; student gets 403.
- **Accessibility:** axe-core on editor page; manual VoiceOver session for alt-text prompt flow.
- **Performance:** AI suggestion < 5 s p95.
- **Manual:** QA verifies decorative, accepted, rejected, manual-entry flows.
