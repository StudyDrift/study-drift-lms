## 1. Problem Statement

Lextures uses Tailwind v4 utility classes referencing `slate`, `neutral`, `indigo`, and other palette tokens hardcoded in `clients/web/src/index.css`. No automated check validates that foreground/background token pairs meet WCAG 2.1 AA SC 1.4.3 (minimum contrast ratio 4.5:1 for normal text, 3:1 for large text) or SC 1.4.11 (3:1 for UI components). The dark-mode theme (`.dark body` in `index.css`) has never been contrast-validated. Until a CI-enforced token-level contract exists, every new component is a potential regression. Districts and universities evaluating WCAG compliance fail on this point at design time, not test time.

## 2. Goals

- Audit all Tailwind color tokens used in Lextures UI components against WCAG 2.1 AA contrast ratios.
- Fix every failing text/background and component/background pair.
- Introduce a CI-enforced design-token contrast validation step that blocks PRs introducing failing pairs.
- Validate both light and dark themes.
- Produce a token contract document mapping approved foreground/background combinations.

## 3. Non-Goals

- Custom theme builder for end users (§12.7 owns high-contrast and reduced-motion).
- Brand color redesign — only fix failing pairs, not aesthetic preferences.
- Mobile-app color validation (§7.1 out of scope).

## 4. Personas & User Stories

- **As a student with low vision**, I want all text on the platform to meet 4.5:1 contrast so that I can read course content without magnification aids.
- **As a color-blind instructor**, I want form validation errors to be identifiable by more than color so that I can understand input states regardless of my color perception.
- **As a district IT administrator**, I want to see a token-level contrast report so that I can verify WCAG compliance during procurement evaluation.
- **As a front-end engineer**, I want a CI check to catch contrast violations before they ship so that I do not introduce regressions unknowingly.

## 5. Functional Requirements

- **FR-1.** Every text-on-background token pair used in production components MUST achieve a contrast ratio ≥ 4.5:1 for body/small text and ≥ 3:1 for large text (≥ 18pt or ≥ 14pt bold) per WCAG SC 1.4.3.
- **FR-2.** Every UI component boundary (input borders, button outlines, focus rings, icon fills) MUST achieve ≥ 3:1 contrast against the adjacent background per WCAG SC 1.4.11.
- **FR-3.** The CI pipeline MUST include a step that runs `@accessible-colors/contrast-checker` or equivalent against the Tailwind v4 token set defined in `clients/web/src/index.css` and fails on any pair below AA threshold.
- **FR-4.** The dark-theme token pairs (`.dark` selector overrides) MUST be validated independently as a separate CI matrix job.
- **FR-5.** All status-indicator colors (success, warning, error, info) MUST NOT use color as the sole differentiator; icons or text labels MUST accompany color (WCAG SC 1.4.1).
- **FR-6.** The system SHOULD generate a human-readable contrast report artifact (HTML or JSON) per CI run, uploaded as a build artifact.
- **FR-7.** A curated list of approved foreground/background token pairs MUST be documented in `docs/design-tokens.md` and referenced in the component library contribution guide.

## 6. Non-Functional Requirements

- **Performance** — Contrast validation script must complete in < 10 s in CI.
- **Security** — No new attack surface; static analysis only.
- **Privacy & Compliance** — WCAG 2.1 AA SC 1.4.1, 1.4.3, 1.4.4, 1.4.11. Section 508 §1194.21(j). EN 301 549 clause 9.1.4.3. Compliance assertion feeds the VPAT (§10.8).
- **Accessibility** — All text contrast ratios verified by automated tool (axe-core color-contrast rule) AND manual spot-check with Colour Contrast Analyser (Paciello Group) on rendered UI. Both light and dark modes verified. Focus-ring contrast validated with forced-colors (Windows High Contrast) simulation in Playwright.
- **Scalability** — Token validation is build-time; no runtime overhead.
- **Reliability** — CI contrast check must be deterministic; avoid dynamic theme values that change at runtime.
- **Observability** — Contrast violation count reported as a CI metric annotation; failing pairs listed in job summary.
- **Maintainability** — A shared `contrast-config.json` file lists all token pairs to test; updated by the design system owner when new tokens are added.
- **Internationalization** — Contrast requirements are locale-independent; no i18n impact.
- **Backward compatibility** — Token value changes may affect component appearance; design review required before merging any palette change.

## 7. Acceptance Criteria

- **AC-1.** *Given* the CI contrast-check job runs against the light theme token set in `clients/web/src/index.css`, *When* it completes, *Then* zero token pairs with contrast ratio below 4.5:1 (normal text) or 3:1 (large text/UI) are reported.
- **AC-2.** *Given* the CI contrast-check job runs against the dark theme token set (`.dark` overrides), *When* it completes, *Then* zero failing pairs are reported for dark mode.
- **AC-3.** *Given* axe-core runs on the gradebook page with both light and dark themes, *When* the `color-contrast` rule executes, *Then* zero violations are reported.
- **AC-4.** *Given* a PR introduces a new Tailwind color utility, *When* CI runs, *Then* if the pair fails 4.5:1 the job fails and blocks merge with a human-readable message identifying the failing pair and its ratio.
- **AC-5.** *Given* a student views a form validation error, *When* the error state is visible, *Then* the error is indicated by both color AND an icon or label (not color alone), verified by removing color perception via browser DevTools "emulate color deficiency".
- **AC-6.** *Given* Windows High Contrast mode is simulated in Playwright via `forced-colors: active`, *When* the modules page is rendered, *Then* no interactive element loses its visible boundary.

## 8. Data Model

No database changes. Token values live in `clients/web/src/index.css` (Tailwind v4 `@theme` block or CSS custom properties).

## 9. API Surface

No new API routes. Static analysis only.

## 10. UI / UX

- **Modified files:** `clients/web/src/index.css` — fix failing token values; update `@theme` color definitions.
- **New files:** `clients/web/contrast-config.json` — exhaustive list of approved token pairs with expected ratios. `scripts/check-contrast.ts` — Node script run in CI.
- **User-facing changes:** Subtle color adjustments to text colors, border colors, and disabled-state colors. No layout changes.
- **Empty / loading / error states:** Error text and loading spinner colors updated to pass 4.5:1.
- **Copy & i18n:** Status labels (success, error, warning) must be present as text, not icon-only.

## 12. Integration Points

- **Tailwind v4 CSS** — `clients/web/src/index.css` is the single source of design tokens.
- **axe-core `color-contrast` rule** — automated checking in Playwright and jsdom tests.
- **Colour Contrast Analyser** — manual spot-check tool (desktop app, Paciello Group).
- **GitHub Actions** — new `contrast-check` job in `.github/workflows/web-ci.yml`.
- **Internal modules:** All components in `clients/web/src/components/` and `clients/web/src/pages/` that use color utilities.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Tailwind v4 JIT generates token values dynamically, making static analysis harder | M | M | Extract used token values at build time via Tailwind's CSS output; parse final CSS |
| Fixing contrast for dark mode changes overall aesthetic and requires design sign-off | M | L | Involve design lead early; prepare before/after screenshots for review |
| axe-core color-contrast rule has known false positives on some gradient backgrounds | L | L | Suppress specific rule IDs on known false positives with documented rationale |
| CI check adds 10+ seconds to build time | L | L | Run contrast check as a parallel job, not on the critical path |

## 15. Rollout Plan

- Feature flag: Not required — CSS token changes ship on merge.
- Sequencing: Audit → fix tokens → add CI check → merge.
- Dogfood: Design team reviews updated palette in Figma and staging.
- GA criteria: Zero axe-core color-contrast violations in CI; CI contrast-check job green.
- Rollback: Revert CSS token PR; CI job can be disabled by removing step.

## 16. Test Plan

- **Unit** — `scripts/check-contrast.ts` unit tests with known passing and failing pairs.
- **Integration** — axe-core color-contrast rule in Playwright on 5 representative pages.
- **End-to-end** — Visual regression snapshots (Playwright) with light and dark modes; diff alerts on color changes.
- **Security** — N/A.
- **Accessibility** — Manual check with Colour Contrast Analyser on rendered pages; forced-colors simulation.
- **Performance / load** — Contrast script runtime < 10 s in CI; no client-side impact.
- **Manual exploratory** — QA tester verifies 10 most-used component states visually.
