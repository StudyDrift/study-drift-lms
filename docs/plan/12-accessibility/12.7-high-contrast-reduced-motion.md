## 1. Problem Statement

Lextures supports only `light` and `dark` themes via a `.dark` CSS class toggle (`clients/web/src/index.css`). The app has a `@media (prefers-reduced-motion: reduce)` partial in `index.css` that suppresses `sidenav-course-items` animations, but no comprehensive audit has been performed, and many Tailwind transition/animation utilities throughout components are unguarded. There is no high-contrast theme for users with photosensitivity, low vision, or Windows High Contrast mode dependency. These gaps violate WCAG 2.1 AA SC 1.4.3 (Contrast), and the absence of `prefers-reduced-motion` support throughout the app creates vestibular risk for users with motion sensitivities (WCAG 2.1 AA SC 2.3.3, AAA, but a practical necessity). For K-12 districts and universities, both issues surface in VPAT evaluations and ADA grievance processes.

## 2. Goals

- Implement a verified high-contrast theme (at least 7:1 contrast ratios) that can be activated manually or automatically from the OS `prefers-contrast: more` media query.
- Audit all CSS transitions and animations in the codebase and guard every one behind `@media (prefers-reduced-motion: reduce)` or the equivalent Tailwind variant.
- Expose a manual in-app toggle for both high-contrast and reduced-motion preferences, persisted per user.
- Verify Windows High Contrast mode compatibility.

## 3. Non-Goals

- Full dyslexia display preferences (§12.6 owns that).
- Color-theme customization for branding (§5.7 out of scope here).
- Animated content within course media (video, GIF) — flashing/animation in uploaded content is a separate §8 concern.

## 4. Personas & User Stories

- **As a student with photosensitivity (vestibular disorder)**, I want all non-essential animations to stop when I enable "Reduce motion" so that navigating the platform does not trigger dizziness or disorientation.
- **As a student with low vision**, I want to switch to a high-contrast mode with white text on black background and no subtle tints so that I can read all text and identify all interactive elements clearly.
- **As a Windows user who relies on High Contrast mode**, I want the app to respect my OS-level setting automatically so that I do not have to configure it separately in every app.
- **As a district IT admin**, I want to confirm that reduced-motion and high-contrast preferences are available so that I can reference them in student accommodation plans.

## 5. Functional Requirements

- **FR-1.** The system MUST automatically apply the reduced-motion setting when the OS reports `prefers-reduced-motion: reduce`, eliminating all CSS transitions, keyframe animations, and scroll-behavior smoothing (WCAG SC 2.3.3 spirit; WCAG 2.1 best practice).
- **FR-2.** The system MUST provide a manual "Reduce motion" toggle in the accessibility preferences panel that overrides the OS detection, persisted server-side.
- **FR-3.** Every CSS animation and transition in `clients/web/src/` (including Tailwind utility classes like `transition-all`, `animate-spin`, `ease-in-out`) MUST be wrapped in a `@media (prefers-reduced-motion: no-preference)` guard or Tailwind `motion-safe:` variant.
- **FR-4.** The system MUST implement a high-contrast theme achieving minimum 7:1 contrast ratio (WCAG AAA SC 1.4.6) for all text and 3:1 for all UI components, activated by `prefers-contrast: more` OR the manual in-app toggle.
- **FR-5.** High-contrast theme MUST also be compatible with Windows Forced Colors / High Contrast mode (`forced-colors: active` media query), using `ButtonText`, `CanvasText`, `LinkText` CSS system colors where appropriate.
- **FR-6.** The manual high-contrast and reduced-motion toggles MUST be accessible from the same Reading Preferences / Accessibility panel introduced in §12.6.
- **FR-7.** Both preferences MUST be persisted server-side (extending `user_reading_preferences` table from §12.6) and apply before first paint to prevent a flash.

## 6. Non-Functional Requirements

- **Performance** — High-contrast theme loaded as a CSS layer; no JS required for activation. Preference loaded with other user preferences (shared API with §12.6).
- **Security** — Preference data is non-sensitive; same access control as §12.6.
- **Privacy & Compliance** — WCAG 2.1 AA SC 1.4.3, 1.4.11, 2.3.3 (AAA but required for vestibular safety). EN 301 549 clause 9.1.4.3, 9.2.3.3. WCAG 2.2 SC 2.4.11. VPAT attestation updated to reference these controls.
- **Accessibility** — Ironic but critical: the accessibility preferences panel itself must render correctly in high-contrast mode and without animations. Transition states (loading, hover, focus) must not rely solely on animation cues.
- **Scalability** — CSS-only; no server rendering changes.
- **Reliability** — If preference load fails, OS media query is used as fallback. App functions identically in all contrast/motion states.
- **Observability** — Metric: `high_contrast_active_pct`, `reduced_motion_active_pct` (privacy-safe; aggregate only).
- **Maintainability** — A single `clients/web/src/lib/motionSafe.ts` utility exports a Tailwind `motion-safe:` variant reminder; ESLint rule added to flag bare `animate-` classes without `motion-safe:`.
- **Internationalization** — No text content changes; CSS-only.
- **Backward compatibility** — Users without preferences set experience no change. High-contrast CSS layer only activates when preference is set.

## 7. Acceptance Criteria

- **AC-1.** *Given* a user has OS-level `prefers-reduced-motion: reduce` set, *When* they load any Lextures page, *Then* all CSS keyframe animations and transitions are absent from the rendered styles (verified by Playwright `page.emulateMedia({ reducedMotion: 'reduce' })` + `getComputedStyle` assertions).
- **AC-2.** *Given* a user manually enables "Reduce motion" in the Accessibility panel, *When* they reload the page, *Then* reduced-motion styles apply before any animation would have fired (no flash of animated content).
- **AC-3.** *Given* a user enables the high-contrast theme, *When* axe-core runs on the gradebook and modules pages, *Then* zero `color-contrast` violations are reported (all text/background pairs ≥ 7:1).
- **AC-4.** *Given* Windows High Contrast (Black) mode is active (simulated via `forced-colors: active` in Playwright), *When* the modules page is rendered, *Then* all interactive elements retain visible boundaries using CSS system colors.
- **AC-5.** *Given* a NVDA user activates the high-contrast toggle via keyboard, *When* the theme switches, *Then* NVDA announces "High contrast enabled" via a live region and focus remains on the toggle.
- **AC-6.** *Given* an accessibility coordinator enables high-contrast as a student's accommodation (§12.10), *When* the student logs in, *Then* high-contrast is pre-activated and the panel shows "Set by your accommodation plan".
- **AC-7.** *Given* the `scripts/audit-animations.ts` CI job runs, *When* it scans all `.tsx` and `.css` files in `clients/web/src/`, *Then* zero bare `animate-`, `transition-`, or `duration-` utilities without `motion-safe:` guard are reported.

Backfill: default values preserve current behavior.

## 9. API Surface

No new routes. `GET /api/me/reading-preferences` and `PATCH /api/me/reading-preferences` (from §12.6) extended with new fields.

## 10. UI / UX

- **Modified:** `clients/web/src/components/a11y/ReadingPreferencesPanel.tsx` (§12.6) — add "Reduce motion" and "High contrast" toggles.
- **New CSS layer:** `clients/web/src/styles/high-contrast.css` — imported conditionally; overrides `@theme` tokens to 7:1 ratios.
- **New CSS audit script:** `scripts/audit-animations.ts` — CI job.
- **User flows:** 1. Student enables high-contrast → immediate CSS layer application → persisted. 2. OS reduced-motion set → auto-applied on next page load. 3. Manual override → persisted, beats OS default.
- **Accessibility annotations:** Toggles are `<button role="switch" aria-checked>`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| High-contrast CSS layer breaks third-party embedded components (charts, video players) | M | M | Scope high-contrast overrides to Lextures-owned selectors; use `@layer` CSS to keep low specificity |
| Forced Colors mode incompatible with custom background gradients | H | M | Replace gradient backgrounds with `forced-colors: active` system color fallbacks |
| Animation audit script produces false positives on third-party libraries | M | L | Scope script to `src/` only; exclude `node_modules/` |

## 15. Rollout Plan

- Feature flag: `ff_high_contrast_reduced_motion` (default `false`).
- Sequencing: CSS layer → audit script → panel UI extension → preference API extension → flip flag.
- Dogfood: Internal team with high-contrast simulations.
- GA: All 7 ACs pass; animation audit script green in CI.
- Rollback: Feature flag off; CSS layer not applied.

## 16. Test Plan

- **Unit:** Toggle switch component renders with correct `aria-checked` state.
- **Integration:** PATCH preferences with `high_contrast: true` → GET returns same.
- **End-to-end:** Playwright `emulateMedia` for reduced-motion; `forcedColors` for HC mode; axe-core in HC.
- **Accessibility:** Manual Windows High Contrast (Black and White) visual inspection.
- **Performance:** No new JS execution; CSS-only change.
- **Manual:** QA verifies no content loss in high-contrast mode on all main pages.
