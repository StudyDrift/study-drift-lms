## 1. Problem Statement

`available_from`, `available_until`, and `due_at` timestamps on module items and assignments are stored as `TIMESTAMPTZ` (UTC), which is correct, but their semantics around time zones are unclear and the UI never asks the user for their time zone. A midnight deadline for a course configured by a US-Eastern instructor will show as 5:00 AM the next day for a UK student and 9:00 AM for an India-based student — without any timezone conversion or display. The `due_at` field is simply rendered as UTC or browser-local time without consistency. There is no per-user time zone setting. This causes missed deadlines, student confusion, and loss of trust in the platform for any internationally distributed course.

## 2. Goals

- Add a per-user `timezone` field (IANA tz database identifier) stored in the user profile.
- Implement auto-detection of the user's timezone from the browser on registration and login.
- Display all deadline-related timestamps in the user's local timezone.
- Allow instructors to specify the timezone intent of a deadline (e.g., "due at 11:59 PM Eastern").
- Ensure all server-side business logic (late submission detection, availability windows) works correctly in UTC regardless of display timezone.

## 3. Non-Goals

- Academic-calendar system with timezone-aware no-class days (see 14.6) — this plan covers per-user tz display.
- Daylight saving time edge cases in server-side scheduling — server always operates in UTC; DST conversion is a display concern.
- Multi-timezone cohort analytics — deferred.

## 4. Personas & User Stories

- **As a student in India taking a US-based course**, I want to see assignment deadlines in IST so that I know exactly when my work is due in my local time.
- **As an instructor in California**, I want to set a deadline of "11:59 PM Pacific Time" and have it displayed in each student's local timezone with a note showing the original timezone.
- **As a student**, I want my timezone to be auto-detected when I sign up so that I don't need to configure it manually.
- **As an admin**, I want course-level timezone to serve as a fallback for students who haven't set their own timezone.

## 5. Functional Requirements

- **FR-1.** The system MUST store a `timezone` field on `user.users` as a valid IANA tz database identifier (e.g., `America/New_York`, `Asia/Kolkata`, `Europe/London`).
- **FR-2.** The system MUST auto-detect the user's timezone from `Intl.DateTimeFormat().resolvedOptions().timeZone` on registration and on first login; the detected value MUST be pre-filled in the timezone selector.
- **FR-3.** The system MUST display all due dates, availability windows, and submission timestamps in the user's stored timezone; if the user has no timezone set, display in the instructor's course timezone, falling back to UTC.
- **FR-4.** When displaying a deadline, the system MUST also show the timezone abbreviation (e.g., "11:59 PM EST" or "04:59 AM GMT+5:30") so that the original intent is transparent.
- **FR-5.** The system MUST allow instructors to set a course-level timezone (IANA identifier) that serves as the "authoritative" timezone for deadline intentions; this is stored as metadata and displayed alongside student-localized times.
- **FR-6.** The system MUST provide a timezone selector component (searchable dropdown of IANA timezone names with UTC offsets) in user account settings.
- **FR-7.** All server-side business logic (late submission checks, availability gates) MUST operate on raw UTC values; the `timezone` field MUST NOT affect backend business logic, only display.

## 6. Non-Functional Requirements

- **Performance** — Timezone conversion is done client-side via `Intl.DateTimeFormat`; no server-side tz conversion library required (the IANA tz data is embedded in V8/SpiderMonkey). Server API responses always return UTC.
- **Security** — `timezone` field validated against the IANA tz database on write (server-side; reject unknown identifiers). Malformed tz strings must not crash `Intl.DateTimeFormat`.
- **Privacy & Compliance** — Timezone is personal data under GDPR (contributes to user location inference); it MUST be included in DSAR exports (10.3) and erasure (set to `null` on erasure). IANA tz database (https://www.iana.org/time-zones) is the authoritative source; use IANA canonical names, not Windows-style or Olson-only names.
- **Accessibility** — Timezone selector MUST be keyboard-navigable with search; ARIA `combobox` pattern. Timezone abbreviations in date display must be interpretable by screen readers (spell out "Eastern Standard Time" in `aria-label`, abbreviation "EST" in visible text).
- **Scalability** — IANA tz validation on the server uses a static lookup table; no external API calls.
- **Reliability** — If the IANA tz ID stored for a user is no longer valid (deprecated alias), fall back to UTC and log a warning.
- **Observability** — Metric `users_with_timezone_set_ratio` — track adoption. Alert if < 50% of active users have tz set 30 days after feature launch.
- **Maintainability** — `clients/web/src/lib/format/timezone.ts` — tz utilities; `server/src/validation/timezone.rs` — IANA validation.
- **Internationalization** — Timezone names in the selector displayed in the user's locale using `Intl.DateTimeFormat` `timeZoneName: 'long'` option. CLDR provides locale-specific timezone names via the browser.
- **Backward compatibility** — `timezone` defaults to `NULL`; existing users display UTC (current behavior) until they set their timezone; auto-detection prompt shown on next login.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student is in India (auto-detected `Asia/Kolkata`), *when* they view an assignment due at `2026-04-15T23:59:00Z`, *then* the displayed due date reads "April 16, 2026 at 5:29 AM IST."
- **AC-2.** *Given* a student registers from a browser in Tokyo, *when* the registration form loads, *then* the timezone selector is pre-filled with `Asia/Tokyo` (detected from `Intl.DateTimeFormat().resolvedOptions().timeZone`).
- **AC-3.** *Given* an instructor sets a course timezone of `America/Los_Angeles` and a deadline of "11:59 PM PDT," *when* a student in London views the deadline, *then* it displays as "07:59 AM BST" with a tooltip showing the original "11:59 PM PDT (instructor timezone)."
- **AC-4.** *Given* a student submits an assignment at `2026-04-15T23:58:00Z` for a deadline at `2026-04-15T23:59:00Z`, *when* the server checks lateness, *then* the submission is marked on-time regardless of the student's display timezone.
- **AC-5.** *Given* a user sets their timezone to an invalid string via a direct API call, *when* the server validates the value, *then* HTTP 422 is returned with a message "Invalid IANA timezone identifier."

## 8. Data Model

```sql
-- Migration: 088_user_timezone.sql

ALTER TABLE user.users
  ADD COLUMN timezone TEXT DEFAULT NULL;  -- IANA tz database ID, e.g. 'America/New_York'

ALTER TABLE course.courses
  ADD COLUMN course_timezone TEXT DEFAULT NULL;  -- Course-level tz fallback

-- Server-side validation: check against pg_timezone_names view
ALTER TABLE user.users
  ADD CONSTRAINT chk_user_timezone
    CHECK (timezone IS NULL OR timezone IN (SELECT name FROM pg_timezone_names));
```

Note: `pg_timezone_names` in Postgres reflects the system IANA tz database; this is updated with OS upgrades.

## 9. API Surface

| Method | Path | Auth Scope | Notes |
|--------|------|-----------|-------|
| `PUT` | `/api/v1/settings/timezone` | authenticated user | Update timezone; validates against IANA list |
| `GET` | `/api/v1/settings/timezone` | authenticated user | Get current timezone |
| `GET` | `/api/v1/timezones` | public | List all valid IANA timezone identifiers with UTC offsets |

API responses for assignment due dates include both `due_at_utc` (ISO 8601) and `due_at_display` (formatted in user's tz) — though display formatting is best done client-side.

## 10. UI / UX

- **Account settings** — Timezone section with searchable IANA timezone selector (grouped by region; shows UTC offset).
- **Auto-detect on registration** — "We detected your timezone as X. Is that correct?" confirmation.
- **Due dates everywhere** — `formatDateTime(dueAt, { timeZone: user.timezone })` from the format library (11.3).
- **Tooltip on deadlines** — hover shows instructor's original timezone alongside user-localized time.
- WCAG 2.1 AA: combobox timezone selector; `aria-label` with full timezone name for abbreviations.

## 11. AI / ML Considerations

None. However, AI-generated assignment descriptions that mention specific times (e.g., "due by Friday evening") should be interpreted as instructor-timezone; this is a content note, not an engineering requirement.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| IANA tz database updates invalidate stored timezone IDs (rare, via alias changes) | L | L | On user login, validate stored tz; if invalid, prompt re-selection |
| Students ignore auto-detection; wrong tz causes missed deadlines | M | M | Show tz confirmation on each deadline display for first 30 days |
| Course timezone vs user timezone conflict confuses students | M | M | Always show both timezones on deadlines |

## 15. Rollout Plan

- No feature flag needed — timezone defaults to NULL (existing behavior) for all current users.
- Phase 1: Schema migration + API endpoint.
- Phase 2: Auto-detection on registration + timezone selector in settings.
- Phase 3: Due-date display uses user timezone everywhere.
- Phase 4: Instructor course-timezone setting.

## 16. Test Plan

- **Security** — Malformed tz string in API → 422 response.
- **Accessibility** — Timezone selector combobox keyboard-navigable; axe-core passes.
