## 1. Problem Statement

Lextures uses naive date and number formatters throughout the frontend — typically `new Date().toLocaleDateString()` without a locale argument or hardcoded US-format strings like `MM/DD/YYYY`. For an EU student, `04/05/2026` is May 4th; for a US student, it is April 5th. Number formatting also differs: European locales use `.` as the thousands separator and `,` as the decimal separator. Currency symbols and positions vary by locale. These formatting errors create confusion, academic-integrity issues (wrong due dates understood by students), and a non-native feel that undermines the product's global ambitions. The `Intl` API (ECMAScript Internationalization API Specification, ECMA-402) provides browser-native locale-aware formatting without any bundle-size cost.

## 2. Goals

- Replace all ad-hoc date, number, and currency formatting with `Intl.DateTimeFormat`, `Intl.NumberFormat`, and `Intl.RelativeTimeFormat` using the active user locale.
- Create shared utility functions and React hooks for consistent locale-aware formatting across the codebase.
- Ensure assignment due dates, quiz submission times, and grade timestamps render correctly for each user's locale and time zone.
- Use Unicode CLDR-backed browser `Intl` APIs; no third-party date-formatting library required for basic cases.

## 3. Non-Goals

- Time zone conversion (see 11.4) — this plan covers formatting only; the time zone input is provided by 11.4.
- Locale-aware sorting of strings (collation) — deferred to v2.
- Server-side date formatting (email templates, PDF reports) — backend uses ISO 8601 everywhere and the client formats.
- Full currency billing (see 15.3) — this plan covers `Intl.NumberFormat` for grade percentages and currency display in hypothetical tuition fields; not payment processing.

## 4. Personas & User Stories

- **As a German student**, I want due dates displayed as `05.04.2026` and numbers as `1.234,56` so that I read them correctly without mental conversion.
- **As a US instructor**, I want grades shown as percentages with a period as the decimal separator (`92.5%`) so that I can read them at a glance.
- **As a French student in Quebec**, I want dates in `5 avril 2026` format and times in 24-hour format.
- **As a developer**, I want a single `useFormatDate()` hook so that I never need to manually format a date again.

## 5. Functional Requirements

- **FR-1.** The system MUST format all dates using `Intl.DateTimeFormat` with the user's active locale (from 11.1) and time zone (from 11.4); bare `Date.toLocaleDateString()` calls without explicit locale MUST be replaced.
- **FR-2.** The system MUST format all numbers (grades, point values, statistics) using `Intl.NumberFormat` with the user's active locale.
- **FR-3.** The system MUST format relative times ("3 hours ago", "due in 2 days") using `Intl.RelativeTimeFormat` with the user's active locale.
- **FR-4.** A shared `clients/web/src/lib/format/` module MUST provide: `formatDate(date, options)`, `formatDateTime(date, options)`, `formatRelativeTime(date)`, `formatNumber(value, options)`, `formatPercent(value)`, `formatCurrency(value, currencyCode)`.
- **FR-5.** The system SHOULD provide a React hook `useLocaleFormat()` that returns all format functions pre-bound to the active locale and time zone.
- **FR-6.** All date display on course cards, assignment due dates, quiz timers, grade history, and audit log timestamps MUST use the locale-aware format utilities.
- **FR-7.** The system MUST NOT store user-locale-formatted dates in the database; all DB storage MUST use ISO 8601 / UTC (enforced by Rust's `chrono` + `TIMESTAMPTZ`).

## 6. Non-Functional Requirements

- **Performance** — `Intl` formatters are expensive to construct; instances MUST be memoized per locale+options combination using `useMemo` or a module-level cache. Format operations MUST add < 0.1 ms per call.
- **Security** — No user-controlled format strings; all format options are code-defined enumerations.
- **Privacy & Compliance** — ISO 8601 storage (TIMESTAMPTZ) ensures no locale bias in backend; ECMA-402 (ECMAScript Internationalization API) standard; Unicode CLDR locale data (embedded in V8/SpiderMonkey) is authoritative for locale-specific patterns.
- **Accessibility** — Machine-readable date values MUST use `datetime` attribute on `<time>` elements (ISO 8601) regardless of displayed locale format (WCAG 2.1 SC 1.3.1). Screen readers read the `datetime` attribute, not the display text.
- **Scalability** — Client-side formatting; no server impact.
- **Reliability** — If the locale's `Intl` data is unavailable (rare), fall back to `en-US` formatting.
- **Observability** — No special metrics; Intl errors logged as warnings in development.
- **Maintainability** — All format logic in `clients/web/src/lib/format/`; prohibited to use raw `Date` methods elsewhere (ESLint rule).
- **Internationalization** — This plan IS the locale formatting concern. Unicode CLDR patterns used via browser-native Intl APIs.
- **Backward compatibility** — Existing English users will see their current format (en-US default unchanged); no functional regression.

## 7. Acceptance Criteria

- **AC-1.** *Given* a user's locale is `de` (German), *when* a due date of 2026-04-15T10:00:00Z is displayed, *then* it renders as "15.04.2026, 10:00 Uhr" (or locale equivalent) rather than "4/15/2026, 10:00 AM".
- **AC-2.** *Given* a user's locale is `fr-CA`, *when* a grade of 92.5% is displayed, *then* it renders as "92,5 %" (French Canadian decimal separator and space before percent sign).
- **AC-3.** *Given* a date element is rendered, *when* the source HTML is inspected, *then* the `<time>` element has a `datetime="2026-04-15T10:00:00Z"` attribute in ISO 8601 format regardless of the displayed locale.
- **AC-4.** *Given* a developer uses a raw `new Date().toLocaleDateString()` call in a new component, *when* the ESLint check runs, *then* a lint error is raised directing the developer to use `useLocaleFormat()` instead.
- **AC-5.** *Given* an `Intl.DateTimeFormat` instance is created for `de` locale, *when* the same locale is requested again within the same React render cycle, *then* the cached instance is returned (verified by mock call count = 1).

## 8. Data Model

No database changes. All DB dates remain as `TIMESTAMPTZ` (UTC); formatting is client-only.

## 9. API Surface

No new API endpoints. All formatting is client-side. API responses always return ISO 8601 timestamps.

## 10. UI / UX

- **Due dates on course cards** — `formatDate(dueAt, { dateStyle: 'medium' })`.
- **Relative timestamps in feed** — `formatRelativeTime(postedAt)` → "2 days ago".
- **Grade percentages in gradebook** — `formatPercent(score / total)`.
- **Quiz timer** — locale-aware `mm:ss` format using `Intl.NumberFormat` with `minimumIntegerDigits: 2`.
- **Audit log timestamps** — full `formatDateTime` with seconds precision.
- All `<time>` elements include `datetime` ISO 8601 attribute.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Memoization cache grows unbounded with many locale variants | L | L | LRU cache with max 20 entries per format type |
| `Intl` behavior differences between browser engines | L | L | Test in Chrome, Firefox, Safari; all support ECMA-402 since 2018 |
| Hardcoded date format strings missed in audit | M | M | ESLint rule + `i18next-parser`-style audit for `toLocaleDateString` calls |

## 15. Rollout Plan

- No feature flag — formatting is always locale-aware once library is deployed.
- Phase 1: Create `format/` library + hook.
- Phase 2: Migrate highest-traffic components (course cards, gradebook, quiz timer).
- Phase 3: ESLint rule + full codebase migration.

## 16. Test Plan

- **Unit** — `formatDate` for de/fr-CA/ar locales; memoization cache hit verified.
- **Integration** — `<time datetime>` attribute present in rendered HTML.
- **End-to-end** — Playwright: switch to German locale → verify due date format.
- **Accessibility** — axe-core: `<time>` elements have `datetime` attribute.

## 17. Documentation & Training

- Developer guide: "Formatting dates and numbers — use `useLocaleFormat()`, never `toLocaleDateString()`."
- API conventions: "All timestamps in API responses use ISO 8601 UTC."

## 18. Open Questions
