## 1. Problem Statement

Arabic, Hebrew, Persian (Farsi), Urdu, and Pashto are right-to-left (RTL) languages with a combined user base of approximately 700 million speakers. Lextures uses Tailwind CSS v4 for styling. The current CSS does not use CSS logical properties (`margin-inline-start` instead of `margin-left`) and does not apply `dir="rtl"` on the HTML root or use Tailwind v4's RTL variants (`rtl:` prefix). The TipTap rich-text editor, the module sidebar, the gradebook grid, and navigation components will all mirror incorrectly for RTL locales without this work. Middle Eastern HE institutions and the growing Arabic-language self-learner market are blocked.

## 2. Goals

- Enable full RTL layout for all existing UI components by converting physical CSS properties to CSS logical properties and enabling Tailwind v4 RTL variants.
- Implement `dir="rtl"` switching on the `<html>` element when an RTL locale is active.
- Audit and fix the TipTap editor for RTL text entry and toolbar mirroring.
- Ensure all icons and directional indicators (arrows, progress bars, breadcrumbs) mirror correctly.
- Ship with Arabic (`ar`) as the v1 RTL locale; Hebrew (`he`) as v1 secondary.
- Cross-link with WCAG 2.1 SC 1.3.4 (Orientation) and SC 3.1.1 (Language of Page) — see 10.7.

## 3. Non-Goals

- Bidirectional text within a single document (bidi mixed content) — complex cases deferred.
- RTL support for native mobile apps (plan 7.1).
- Full Arabic/Hebrew translations of all content — this plan covers layout only; translations are a separate effort under 11.1.

## 4. Personas & User Stories

- **As an Arabic-speaking university student**, I want the Lextures interface to display correctly right-to-left so that reading and navigation feel natural.
- **As a Hebrew-speaking instructor**, I want the course sidebar, gradebook, and text editor to mirror for RTL so that I can author content without layout glitches.
- **As a developer adding a new component**, I want a clear pattern for RTL-safe CSS so that new features are RTL-compatible by default.

## 5. Functional Requirements

- **FR-1.** The system MUST set `dir="rtl"` and `lang="{locale}"` on the `<html>` element when an RTL locale is active; the switch MUST happen synchronously before paint to prevent flash of wrong direction.
- **FR-2.** All physical CSS margin/padding/border properties (`margin-left`, `padding-right`, etc.) in `clients/web/src/` and component stylesheets MUST be replaced with CSS logical properties (`margin-inline-start`, `padding-inline-end`, `border-inline-start`, etc.) or Tailwind v4 logical utility classes (`ms-4`, `pe-2`, etc.).
- **FR-3.** Tailwind v4 `rtl:` variant MUST be used for any directional utility that cannot be expressed as a logical property (e.g., `rtl:rotate-180` for arrow icons).
- **FR-4.** The TipTap editor MUST support RTL text direction: the editor's content node MUST have `dir="auto"` (Unicode bidi algorithm); the toolbar MUST mirror directional buttons (indent/outdent, list alignment) when RTL is active.
- **FR-5.** All SVG icons with directional meaning (arrow-right, chevron-right, back button) MUST be mirrored using `transform: scaleX(-1)` or the Tailwind `rtl:rotate-180` variant when in RTL mode.
- **FR-6.** The gradebook grid, module sidebar, and navigation breadcrumbs MUST render correctly in RTL layout.
- **FR-7.** The system MUST NOT break LTR layout when RTL support is added; Chromatic or Storybook visual regression tests MUST confirm no LTR regressions.

## 6. Non-Functional Requirements

- **Performance** — CSS logical property conversion is zero-runtime-cost; no JS overhead beyond the `dir` attribute flip.
- **Security** — No XSS risk from `dir` attribute; value is set from the locale registry (enum), not user input.
- **Privacy & Compliance** — WCAG 2.1 SC 1.3.4 (Orientation) indirectly relevant; Arabic/Hebrew RTL text direction is a Level A requirement for language conformance (10.7). Unicode Bidirectional Algorithm (UBA, Unicode Standard Annex #9) governs text rendering.
- **Accessibility** — Screen readers (NVDA, JAWS with Arabic/Hebrew support; VoiceOver) must receive correct `dir` and `lang` attributes. ARIA label directions must be logical, not physical. WCAG 2.1 SC 3.1.1 (Language of Page), SC 3.1.2 (Language of Parts). Cross-link: 10.7.
- **Scalability** — CSS logical properties are a compile-time change; no runtime scaling concern.
- **Reliability** — Locale switch must not require page reload; RTL/LTR toggle must be instant.
- **Observability** — No special metrics needed; RTL locale selection tracked by standard `locale_active{locale}` metric from i18n framework (11.1).
- **Maintainability** — ESLint rule (`eslint-plugin-i18next` or custom) flags physical CSS property usage in new components. Storybook RTL story required for all layout components.
- **Internationalization** — This plan IS the RTL i18n concern. Tailwind v4 `rtl:` variant documentation linked in developer guide.
- **Backward compatibility** — LTR locales unchanged; no CSS changes visible to LTR users if logical properties are correctly equivalent.

## 7. Acceptance Criteria

- **AC-1.** *Given* a user selects Arabic as their locale, *when* the page renders, *then* `<html dir="rtl" lang="ar">` is set, all sidebar/nav elements are right-aligned, and the page layout mirrors horizontally.
- **AC-2.** *Given* the TipTap editor is open with RTL locale active, *when* a user types Arabic text, *then* the cursor starts at the right side, text flows right-to-left, and directional toolbar buttons are mirrored.
- **AC-3.** *Given* a back-navigation arrow icon is displayed in LTR mode pointing left, *when* RTL mode is active, *then* the icon points right (correctly indicating "back" in reading direction).
- **AC-4.** *Given* the gradebook grid renders in RTL mode, *when* a screen-reader user navigates it, *then* column headers are announced in the correct right-to-left reading order and cell navigation is logical.
- **AC-5.** *Given* an LTR user switches from English to Arabic and back to English, *when* English is active, *then* all LTR layout properties are fully restored with no visual artifacts.
- **AC-6.** *Given* a developer adds a new component using Tailwind `ml-4` (physical property), *when* the RTL lint check runs in CI, *then* a warning is emitted suggesting `ms-4` (logical property) instead.

## 8. Data Model

No new database schema. RTL direction derived from the user's locale (11.1 `users.locale`). Direction map:

```typescript
// clients/web/src/i18n/rtl-locales.ts
export const RTL_LOCALES = new Set(['ar', 'he', 'fa', 'ur', 'ps']);
export const isRTL = (locale: string) => RTL_LOCALES.has(locale.split('-')[0]);
```

## 9. API Surface

No new API endpoints. `dir` is computed client-side from locale; no server change required.

## 10. UI / UX

- **HTML direction** — `document.documentElement.dir` set on locale change; `document.documentElement.lang` also updated.
- **Tailwind config** — `rtl:` variant enabled in `clients/web/tailwind.config.ts`.
- **Component audit checklist** — all layout components reviewed: Sidebar, NavBar, Breadcrumb, GradebookGrid, TipTap Editor, Modals, Dropdowns, Toasts.
- **Storybook** — All layout stories must have an RTL story variant.
- **Visual regression** — Chromatic or Percy snapshots for RTL + LTR to prevent regressions.
- **Text alignment** — Use `text-start` / `text-end` Tailwind classes instead of `text-left` / `text-right`.

## 11. AI / ML Considerations

None. AI-generated content may include Arabic/Hebrew text; the editor's `dir="auto"` attribute handles bidi rendering via the Unicode Bidirectional Algorithm without additional engineering.

## 12. Integration Points

- `clients/web/tailwind.config.ts` — enable RTL variant.
- `clients/web/src/i18n/rtl-locales.ts` — RTL locale registry.
- `clients/web/src/App.tsx` — set `document.documentElement.dir` on locale change.
- `clients/web/src/components/editor/` — TipTap RTL patches.
- All Tailwind class usage — audit and replace physical → logical.
- Storybook — RTL addons; visual regression integration.
- ESLint — custom rule for physical CSS class detection.

## 13. Dependencies & Sequencing

- Must ship after: 11.1 (locale detection and switching).
- Should ship before: Arabic/Hebrew locale translations are finalized.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| TipTap RTL support is incomplete upstream | H | M | File upstream issue; implement workaround with `dir="auto"` on editor node |
| Physical property audit misses legacy CSS files | M | M | Automated grep for `margin-left/right`, `padding-left/right` in CI |
| LTR regression after logical property conversion | M | M | Chromatic visual regression; 100% Storybook story coverage for layout components |

## 15. Rollout Plan

- Feature flag: `rtl_enabled` — default `false` until audit complete; flip to `true` when AC-5 (LTR non-regression) confirmed.
- Phase 1: Tailwind config + `dir` switching + RTL locale registry.
- Phase 2: Physical → logical CSS audit across all components.
- Phase 3: TipTap RTL editor fix + directional icon mirroring.
- Phase 4: Storybook RTL stories + visual regression baselines.
- GA: Arabic locale translations available + RTL layout verified.

## 16. Test Plan

- **Unit** — `isRTL()` function returns correct value for all RTL locale codes.
- **Integration** — Locale switch to `ar` → `dir="rtl"` set on `<html>`.
- **End-to-end** — Playwright: switch to Arabic → verify sidebar right-aligned → gradebook navigable RTL.
- **Accessibility** — VoiceOver/Arabic (macOS) announces content in correct order; `lang="ar"` triggers Arabic TTS.
- **Visual regression** — Chromatic snapshots: RTL vs LTR for 20 key components; no regressions.
- **Manual** — Native Arabic/Hebrew speaker review of layout for 5 key flows.
