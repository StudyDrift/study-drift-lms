## 1. Problem Statement

Lextures has no internationalization framework. All UI strings are hardcoded in English within React component JSX. This means the product cannot be localized for any non-English market without a full codebase rewrite, and even compliance UI strings (consent notices, FERPA disclosures, COPPA notices) that are legally required in other languages (e.g., Spanish for COPPA and CCPA) cannot be produced without significant manual effort. As a result, the non-US K-12 market, international higher education, and any self-learner not fluent in English are blocked. ICU MessageFormat is the industry standard for handling pluralization, gender, and interpolation across languages; `react-i18next` is the leading React binding.

## 2. Goals

- Install and configure `react-i18next` with ICU MessageFormat parsing via `i18next-icu`.
- Externalize all existing UI strings in `clients/web/src/` into JSON translation files organized by BCP 47 locale codes.
- Establish a translation file structure, key naming convention, and namespace strategy.
- Set up a locale detection and switching mechanism (browser preference → user profile → URL).
- Ship with English (en) as the default locale; Spanish (es) and French (fr) as v1 secondary locales.
- Integrate with a translation management workflow (Crowdin or similar) for future translator handoff.

## 3. Non-Goals

- Translating all strings in the first sprint — this plan establishes the framework and migrates the highest-traffic pages; full coverage is incremental.
- Backend i18n for server-generated content (API error messages, email templates) — Phase 2; v1 ships English-only server messages.
- Machine translation of content authored by instructors (see 19.5) — course content is not in scope.

## 4. Personas & User Stories

- **As a Spanish-speaking K-12 student in a bilingual school**, I want the Lextures UI in Spanish so that I can navigate without language barriers.
- **As a French-speaking university student in Quebec**, I want to complete assignments, take quizzes, and read course content in French.
- **As a self-learner in Germany**, I want dates, numbers, and the UI in German (or at least English with German date formatting) so that the platform feels native.
- **As a developer**, I want a clear key naming convention and ICU format support so that I can add new strings correctly without breaking existing translations.

## 5. Functional Requirements

- **FR-1.** The system MUST install `react-i18next` (≥ 14.x) with `i18next-icu` for ICU MessageFormat support; all UI strings MUST use the `useTranslation` hook or `Trans` component.
- **FR-2.** Translation files MUST be organized by BCP 47 locale code (e.g., `en`, `es`, `fr`) in `clients/web/public/locales/{locale}/` with namespace splitting (e.g., `common.json`, `course.json`, `quiz.json`, `auth.json`, `compliance.json`).
- **FR-3.** The system MUST support ICU MessageFormat for: plural forms (`{count, plural, one {# item} other {# items}}`), select forms (`{gender, select, male {he} female {she} other {they}}`), and date/number formatting placeholders.
- **FR-4.** The system MUST detect the user's preferred locale in this priority order: (1) user profile locale setting (stored in DB), (2) `Accept-Language` browser header, (3) fallback to `en`.
- **FR-5.** The system MUST provide a locale switcher component in the user settings page.
- **FR-6.** The system MUST support lazy-loading of locale files; only the active locale MUST be downloaded on initial page load.
- **FR-7.** All string keys MUST follow the convention `{namespace}.{component}.{element}` (e.g., `course.header.title`); keys MUST NOT contain raw English text.
- **FR-8.** The build pipeline MUST include a check verifying that all keys present in the `en` locale file exist in all other locale files; missing keys MUST fall back to `en` at runtime but MUST fail the translation-completeness CI check at a configurable threshold.

## 6. Non-Functional Requirements

- **Performance** — Locale JSON files MUST be code-split and lazy-loaded; initial JS bundle MUST NOT increase by > 5 kB due to i18n library. Locale file size MUST be < 200 kB per namespace.
- **Security** — Translation values MUST NOT be rendered via `dangerouslySetInnerHTML`; use `Trans` component with explicit JSX interpolation only.
- **Privacy & Compliance** — COPPA consent notices MUST be available in Spanish (16 CFR § 312.4(b) — notice in languages understood by parents); CCPA privacy rights MUST be in Spanish (Cal. Civ. Code § 1798.130(a)(5)); NY Ed Law 2-d annual notices in languages spoken by ≥ 5% of district students. BCP 47 (RFC 5646) for all locale tags. Unicode CLDR for locale-specific display patterns.
- **Accessibility** — All translated strings MUST include ARIA label keys; `lang` attribute on `<html>` MUST be updated on locale switch for screen readers (WCAG 2.1 SC 3.1.1).
- **Scalability** — Framework supports 50+ locales without code changes; namespace splitting keeps individual file sizes manageable.
- **Reliability** — Missing translation key falls back to English without crashing; `console.warn` in development.
- **Observability** — Log missing translation key events in development; production metric `i18n_missing_key_total{locale,namespace}`. Cross-link: 17.7.
- **Maintainability** — `clients/web/src/i18n/` config directory; all namespaces registered in `i18n.ts`; type-safe keys via `i18next-typescript-tools` or equivalent.
- **Internationalization** — This plan IS the i18n framework; all other i18n plans depend on it.
- **Backward compatibility** — All existing hardcoded strings extracted to `en` locale file first; no UI regression on English.

## 7. Acceptance Criteria

- **AC-1.** *Given* a user with browser locale set to `es`, *when* they load Lextures, *then* the UI renders in Spanish and the `<html lang="es">` attribute is set.
- **AC-2.** *Given* a translation key is missing in the Spanish locale file, *when* a Spanish user views the page, *then* the English fallback string is shown (not the key identifier) and a `i18n_missing_key_total` metric is incremented.
- **AC-3.** *Given* a string requires ICU plural form (e.g., "1 assignment / 2 assignments"), *when* the count is 1 or 2, *then* the correct grammatical form is rendered in English, Spanish, and French.
- **AC-4.** *Given* a developer adds a new UI component with hardcoded English text, *when* the i18n lint check runs in CI, *then* the build fails with a message identifying the hardcoded string location.
- **AC-5.** *Given* a user switches locale to French in settings, *when* the page re-renders, *then* all visible text is in French (or falls back to English for untranslated strings), the user's locale preference is saved to their profile, and future sessions load French by default.
- **AC-6.** *Given* the locale file lazy-loading is active, *when* a user loads the application in Spanish, *then* only the Spanish locale files are downloaded (not English or French), verified by network inspection.

## 8. Data Model

```sql
-- Migration: 087_user_locale.sql

ALTER TABLE user.users
  ADD COLUMN locale TEXT NOT NULL DEFAULT 'en'
    CHECK (locale ~ '^[a-z]{2}(-[A-Z]{2})?$');  -- BCP 47 pattern
```

## 9. API Surface

| Method | Path | Auth Scope | Notes |
|--------|------|-----------|-------|
| `PUT` | `/api/v1/settings/locale` | authenticated user | Update user locale preference |
| `GET` | `/api/v1/settings/locale` | authenticated user | Get current locale |

Locale files served as static assets from `/locales/{locale}/{namespace}.json` (CDN-cacheable).

## 10. UI / UX

- **Locale switcher** — dropdown in account settings; flag icon + locale name in native script (e.g., "Español", "Français").
- **`<html lang>` attribute** — updated via `document.documentElement.lang` on locale change.
- **Date/number display** — all dates/numbers route through `Intl.*` APIs (see 11.3); i18n framework does not format dates directly.
- **Right-to-left layout** — triggered by locale direction; see plan 11.2.
- All new i18n-enabled pages use keys prefixed by their namespace; no raw English in JSX.

## 11. AI / ML Considerations

None for the framework itself. AI-generated content (quiz questions, rubric text) is produced by the server in the language of the prompt; i18n framework handles static UI strings only.

## 12. Integration Points

- `clients/web/src/i18n/index.ts` — i18next configuration.
- `clients/web/public/locales/` — translation JSON files.
- `clients/web/src/components/` — all components migrated to use `useTranslation`.
- `clients/web/src/pages/` — all pages migrated.
- `server/migrations/087_user_locale.sql`.
- CI: `i18n-ally` or `i18next-parser` script to detect untranslated strings.
- Crowdin (or alternative TMS) integration for translator workflow.
- All plans in sections 10–11 depend on i18n keys being available.

## 13. Dependencies & Sequencing

- No hard dependencies — foundational; ship first in section 11.
- Must ship before: 11.2, 11.3, 11.4, 11.5, 11.6, and all compliance UI from section 10.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| String extraction effort underestimated (1000+ strings) | H | M | Phased: extract highest-traffic pages first; use `i18next-parser` for automated extraction |
| Translation quality for Spanish/French is poor | M | M | Professional translation for legal/compliance strings; community/machine for lower-risk UI |
| ICU MessageFormat syntax errors in translation files | M | M | CI validation of all JSON files against ICU parser |
| i18n `Trans` component misused with unsafe HTML | M | M | ESLint rule forbidding `dangerouslySetInnerHTML` in translated content |

## 15. Rollout Plan

- Feature flag: `i18n_enabled` — default `true` (always on; English is the default locale, no user-visible change until translations are added).
- Phase 1: Install libraries; configure `i18n.ts`; extract all `en` strings into namespace files.
- Phase 2: Spanish (`es`) translations for compliance strings (legal requirement).
- Phase 3: Spanish + French for all high-traffic pages (auth, course, quiz).
- Phase 4: Crowdin integration; community translation pipeline.
- GA: All pages covered in `en`; `es` and `fr` at > 80% coverage.

## 16. Test Plan

- **Unit** — ICU plural forms render correctly for en/es/fr; missing key fallback behavior.
- **Integration** — Locale stored in DB; reloaded on next session.
- **End-to-end** — Playwright: switch locale to Spanish → all extracted strings show Spanish; ICU plural correct.
- **Accessibility** — `<html lang>` attribute correct per locale; axe-core passes with locale-specific ARIA labels.
- **Performance** — Bundle size increase < 5 kB; locale file lazy-loaded.
- **Manual** — Native speaker review of es/fr translations for high-traffic pages.
