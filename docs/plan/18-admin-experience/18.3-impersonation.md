## 1. Problem Statement

Support staff and instructors currently have no way to see the application from a specific student's perspective without access to that student's credentials. Diagnosing UX issues, verifying grade visibility, or troubleshooting enrollment problems requires either asking the student to share screenshots or accessing the database directly. Both approaches are insecure, time-consuming, and legally problematic under FERPA. An impersonation ("view as") capability is a standard feature in every mature LMS and is required for efficient institutional support.

## 2. Goals

- Allow `org_admin` and `super_admin` users to temporarily assume the identity of any user within their org scope to view the application as that user.
- Ensure all impersonation sessions are clearly branded (persistent banner) and fully audited.
- Prevent the impersonating actor from taking write actions that could damage the impersonated user's data.
- Provide a one-click exit that returns the admin to their own session without any side effects.

## 3. Non-Goals

- Allowing instructors to impersonate other instructors or admins.
- Allowing `org_admin` to impersonate users in a different org.
- Permanent delegation of credentials (impersonation is a bounded, audited session only).
- Exporting data as the impersonated user — read-only view is the scope.

## 4. Personas & User Stories

- **As an org admin**, I want to view the course dashboard as a specific student so that I can verify they see the correct enrolled courses after a roster sync.
- **As a Lextures super-admin**, I want to impersonate a district admin so that I can reproduce a reported UI bug without guessing the org configuration.
- **As an instructor**, I want to preview my course as a student would see it (without impersonation of a real student) — covered by a lightweight "preview as student" course view, distinct from full impersonation.
- **As a student**, I want to be notified (via audit trail) if an admin has accessed my account perspective so that I can exercise transparency rights.

## 5. Functional Requirements

- **FR-1.** The system MUST allow a user with `org_admin` or `super_admin` role to initiate an impersonation session for any user within their org via the admin console.
- **FR-2.** The system MUST issue a short-lived impersonation JWT (TTL ≤ 60 min, non-renewable) that carries both the actor's `admin_id` and the target's `user_id`; the token MUST be distinguishable from a normal session token.
- **FR-3.** During an impersonation session the application MUST display a persistent, dismissal-proof banner: "You are viewing as [user name]. [Exit impersonation]".
- **FR-4.** All write operations (grade changes, submission creation, enrollment modification) MUST be blocked during an impersonation session; the server MUST return HTTP 403 for any non-GET/HEAD request made with an impersonation token.
- **FR-5.** Every impersonation session start and end MUST be recorded in `admin_audit_log` with actor ID, target user ID, and timestamps.
- **FR-6.** The system MUST provide an "Exit impersonation" action that invalidates the impersonation token and restores the admin's original session without requiring re-authentication.
- **FR-7.** The system MUST prevent recursive impersonation (an impersonation token MUST NOT be usable to initiate another impersonation session).
- **FR-8.** The system SHOULD allow an `org_admin` to see the impersonation audit log for their org, filtered by target user.

## 6. Non-Functional Requirements

- **Performance** — Impersonation token issuance MUST complete in ≤ 200 ms p95.
- **Security** — Impersonation tokens MUST be signed separately from regular JWTs and include a `impersonation: true` claim; server middleware MUST validate this claim before allowing the session; tokens MUST be stored in `httpOnly` cookies (not localStorage) to prevent exfiltration; impersonation tokens MUST be revoked server-side on exit.
- **Privacy & Compliance** — FERPA requires that access to student education records be logged with purpose; impersonation sessions constitute record access and MUST be logged; students MUST be able to request the impersonation audit log for their account via DSAR (§10.3).
- **Accessibility** — The impersonation banner MUST meet WCAG 2.1 AA contrast; role `status` aria-live region to announce impersonation state to screen readers.
- **Scalability** — Impersonation token revocation list stored in Redis (or Postgres with short TTL index); low volume feature, no scaling concern.
- **Reliability** — If the revocation store is unavailable, impersonation token issuance MUST fail closed (deny impersonation rather than skip revocation).
- **Observability** — `impersonation_sessions_started_total`, `impersonation_sessions_ended_total` counters; alert on unusually high rate.
- **Maintainability** — Impersonation logic in `server/src/services/impersonation.rs`; banner in `clients/web/src/components/ImpersonationBanner.tsx`.
- **Internationalization** — Banner text in i18n keys.
- **Backward compatibility** — Normal JWT auth flow unchanged.

## 7. Acceptance Criteria

- **AC-1.** *Given* an `org_admin` initiates impersonation of a student, *When* the impersonation session is active, *Then* the application renders the student's dashboard and the impersonation banner is visible on every page.
- **AC-2.** *Given* an impersonation session is active, *When* the impersonating admin attempts a POST request (e.g., submit a quiz answer), *Then* the server returns HTTP 403 and the action is not applied.
- **AC-3.** *Given* an `org_admin` attempts to impersonate a user in a different org, *When* the impersonation request is made, *Then* the server returns HTTP 403 and no token is issued.
- **AC-4.** *Given* an impersonation session is active and the admin clicks "Exit impersonation", *When* the exit action is processed, *Then* the impersonation token is invalidated, the admin is returned to their own session, and both start and end events exist in `admin_audit_log`.
- **AC-5.** *Given* an impersonation JWT with TTL expired, *When* the token is used on any endpoint, *Then* the server returns HTTP 401 and logs a `token_expired` event.
- **AC-6.** *Given* an impersonation token is active, *When* it is used to call the impersonation endpoint to start a nested session, *Then* the server returns HTTP 403.

## 8. Data Model

```sql
-- Add to admin_audit_log (already defined in 18.1):
-- action values 'impersonation.start' and 'impersonation.end' with target_id = impersonated user_id.

-- Short-lived revocation index (or use Redis TTL keys):
CREATE TABLE impersonation_tokens (
  jti         TEXT PRIMARY KEY,      -- JWT ID claim
  admin_id    UUID NOT NULL REFERENCES users(id),
  target_id   UUID NOT NULL REFERENCES users(id),
  issued_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at  TIMESTAMPTZ NOT NULL,
  revoked_at  TIMESTAMPTZ
);
CREATE INDEX impersonation_tokens_exp ON impersonation_tokens(expires_at);
```

A background sweep deletes rows where `expires_at < now() - interval '1 hour'`.

## 9. API Surface

```
POST   /api/admin/impersonate         { target_user_id: UUID }
                                      -> 200 { impersonation_token, expires_at }
DELETE /api/admin/impersonate/session -> 204 (exit; invalidates token)
GET    /api/admin/impersonate/log     -> Paginated<ImpersonationLogEntry> ?target_user_id=
```

All existing routes: middleware checks for `impersonation: true` claim; if present and method is not GET/HEAD, returns 403 with `{ error: "writes_blocked_during_impersonation" }`.

## 10. UI / UX

New components:
- `clients/web/src/components/ImpersonationBanner.tsx` — fixed top bar (red/amber background, high contrast); shows target user name and avatar; "Exit" button with loading state.
- Admin console user table: "View as" button next to each user row.

User flow:
1. Admin clicks "View as" on user row → confirmation dialog ("You are about to view the application as [name]. All writes will be blocked.") → Confirm → impersonation token stored in cookie → page reloads as target user.
2. At any point, admin clicks "Exit impersonation" in the banner → cookie cleared → admin session restored → redirect to `/admin/users`.

Banner: `position: fixed; top: 0; z-index: 9999;` so it is always visible regardless of route. Adds top-padding to main content so banner does not obscure content.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Impersonation token exfiltrated via XSS | M | H | HttpOnly cookie; strict CSP; short 60-min TTL |
| Admin accidentally submits data as student | M | H | Server-side write block on impersonation tokens (FR-4) |
| FERPA audit trail incomplete | L | H | Integration test asserting audit row on every start/end |
| Impersonation banner not visible in iframe embeds (LTI) | M | M | Detect iframe context; show inline alert instead of fixed banner |

## 15. Rollout Plan

- Feature flag: `impersonation_enabled` (OpenFeature, default `false`).
- Phase 1: Enable for `super_admin` only; verify audit logging.
- Phase 2: Enable for `org_admin` roles in pilot orgs.
- Phase 3: GA for all orgs with `org_admin`.
- Rollback: Disable flag; impersonation endpoints return 404; no data effects.

## 16. Test Plan

- **Unit** — Impersonation token generation and claims validation; write-block middleware for all HTTP methods; org-isolation check.
- **Integration** — Full round-trip: POST impersonate → GET /me returns target user → POST blocked → DELETE exits → GET /me returns admin.
- **End-to-end** — Playwright: admin logs in → clicks "View as" on student → verifies banner present → attempts quiz submission (should fail) → exits → verifies back to admin session.
- **Security** — Attempt impersonation without `org_admin` role; attempt cross-org impersonation; attempt nested impersonation; attempt to use expired token.
- **Accessibility** — axe-core on impersonation banner; screen-reader test that aria-live announcement fires when impersonation begins.
- **Manual exploratory** — Verify impersonation banner persistence across page navigations, tab switches, and browser refresh.

## 17. Documentation & Training

- Help center: "Viewing the App as a Student" — when to use, how to exit, privacy notice.
- Admin guide: security responsibilities when using impersonation; FERPA notice.
- Internal runbook: querying impersonation log for support tickets; revoking stuck impersonation sessions.

## 18. Open Questions

1. Should instructors have a lightweight "preview as student" mode (read-only course view without true impersonation) as a separate feature?
2. Should we notify the impersonated user (asynchronously via email or in-app) that their account was accessed?
3. What is the maximum impersonation session TTL — 60 min seems right for support use, but is it too short for debugging long workflows?
4. Should impersonation be available on the mobile app (future §7.1), or restricted to the web admin console?
