## 1. Problem Statement

Lextures has no concept of paid seats, subscription tiers, or license limits. Any user can register and any org can have unlimited users. This means there is no mechanism to enforce contractual seat limits, gate premium features, or provide admins with visibility into their license utilization. This blocks SaaS monetization and prevents enterprise contract compliance.

## 2. Goals

- Introduce a seat-count license model where each org has a `max_seats` limit tracked against active users.
- Alert admins when approaching the seat limit and block new enrollments when the limit is exceeded.
- Provide a license management dashboard in the admin console showing utilization, contract dates, and tier.
- Enable `super_admin` to set, modify, and view license records for any org.

## 3. Non-Goals

- Payment processing or Stripe integration (§15.3).
- Feature flagging by tier (§17.14 OpenFeature handles that).
- Self-service plan upgrades (future; contacts sales for now).

## 4. Personas & User Stories

- **As a super admin**, I want to set a seat limit of 500 for a district so that the district's contract is enforced automatically.
- **As an org admin**, I want to see how many of my 500 licensed seats are occupied so that I can request an upgrade before the school year starts.
- **As an org admin**, I want to receive an email alert when we reach 90% seat utilization so that I have time to purchase additional seats.
- **As a student**, I want to receive a clear message if enrollment is blocked due to seat limits rather than a generic error.

## 5. Functional Requirements

- **FR-1.** The system MUST maintain a `licenses` record per org with: `tier` (enum), `max_seats` (integer, -1 = unlimited), `contract_start`, `contract_end`, and `notes`.
- **FR-2.** The system MUST count "active seats" as the number of users in the org with `is_active = true`.
- **FR-3.** When a new user is activated or created in an org, the system MUST check active seat count against `max_seats`; if the limit is reached, the action MUST be rejected with HTTP 422 and error code `seat_limit_reached`.
- **FR-4.** The system MUST display current seat utilization (`used / max_seats`) on the admin console overview.
- **FR-5.** The system MUST send an email alert to all `org_admin` users when utilization crosses 80% and again at 95%.
- **FR-6.** The system SHOULD display a banner on the admin console when `contract_end` is within 30 days.
- **FR-7.** The super_admin MUST be able to set, view, and modify any org's license record via the admin API.
- **FR-8.** Seat-limit enforcement MUST be bypassed for `org_admin` and `super_admin` role users (admins do not consume learner seats).
- **FR-9.** The system MAY support named license tiers (e.g., `starter`, `growth`, `enterprise`) used by feature-flag rules to gate capabilities.

## 6. Non-Functional Requirements

- **Performance** — Seat count query MUST use a pre-computed counter (maintained by trigger or async job) rather than a full-count scan; counter update MUST complete in ≤ 50 ms.
- **Security** — License records editable only by `super_admin`; org_admin can view but not modify; seat count bypass for admin roles enforced server-side.
- **Privacy & Compliance** — License data (seat counts, contract terms) is business data, not student PII; no FERPA implications.
- **Accessibility** — Seat utilization displayed as both a progress bar and a numeric label for screen readers.
- **Scalability** — Counter table design avoids full-count queries; works at any org size.
- **Reliability** — If the counter becomes inconsistent, a background reconciliation job MUST resync from source-of-truth `COUNT(users WHERE org_id AND is_active)`.
- **Observability** — `org_seat_utilization{org_id}` gauge; alert when any org reaches 95%.
- **Maintainability** — License service in `server/src/services/license.rs`; counter update triggered by user activation/deactivation events.
- **Internationalization** — Seat-limit error message in i18n keys.
- **Backward compatibility** — Orgs without a license record treated as unlimited (default `max_seats = -1`).

## 7. Acceptance Criteria

- **AC-1.** *Given* an org with `max_seats = 10` and 10 active users, *When* an admin attempts to activate an 11th user, *Then* the API returns HTTP 422 with `error: "seat_limit_reached"`.
- **AC-2.** *Given* an org at 80% utilization, *When* the utilization threshold is crossed, *Then* all org_admin users receive an email alert within 5 minutes.
- **AC-3.** *Given* an org with `max_seats = -1` (unlimited), *When* any number of users are added, *Then* no seat-limit error is ever returned.
- **AC-4.** *Given* an `org_admin` user is created (admin role), *When* they are counted, *Then* they do not decrement the learner seat counter.
- **AC-5.** *Given* a super_admin updates `max_seats` from 100 to 200, *When* the change is saved, *Then* the admin console immediately reflects the new limit and previously blocked enrollments can proceed.
- **AC-6.** *Given* `contract_end` is 25 days away, *When* the org admin logs in, *Then* a warning banner is displayed on the admin console dashboard.

## 8. Data Model

```sql
-- 0XX_licenses.sql
CREATE TYPE license_tier AS ENUM ('unlimited', 'starter', 'growth', 'enterprise');

CREATE TABLE licenses (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          UUID NOT NULL UNIQUE REFERENCES organizations(id),
  tier            license_tier NOT NULL DEFAULT 'unlimited',
  max_seats       INT NOT NULL DEFAULT -1,   -- -1 = unlimited
  used_seats      INT NOT NULL DEFAULT 0,    -- denormalized counter
  contract_start  DATE,
  contract_end    DATE,
  notes           TEXT,
  updated_by      UUID REFERENCES users(id),
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 9. API Surface

```
GET    /api/admin/license                   -> LicenseDto  (org_admin scope)
PUT    /api/admin/license                   -> LicenseDto  (super_admin only)
GET    /api/super-admin/licenses            -> Paginated<OrgLicenseDto>  (all orgs, super_admin)
PATCH  /api/super-admin/licenses/:org_id    -> LicenseDto
```

Seat check is internal, not a public endpoint; enforced as middleware on user-activation routes.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Counter drift from trigger failures | M | M | Daily reconciliation job; super_admin "resync" endpoint |
| Orgs unintentionally blocked at limit during school-year rush | M | H | Configurable grace period (e.g., 5 seats over limit); alert lag |
| Admin role bypass exploited to create unlimited admins | L | H | Separate admin seat count; cap admin roles too if needed |

## 15. Rollout Plan

- Feature flag: `seat_management_enabled` (OpenFeature, default `false`).
- Phase 1: Schema migration; all orgs default to `max_seats = -1` (unlimited); no enforcement.
- Phase 2: Enable enforcement for new enterprise contracts only; existing orgs remain unlimited.
- Phase 3: GA enforcement for all tiers.

## 16. Test Plan

- **Unit** — Seat-counter trigger logic; enforcement check (at limit, under limit, unlimited); admin-role bypass.
- **Integration** — Activate user → counter increments; deactivate → decrements; at-limit → 422.
- **End-to-end** — Playwright: admin views seat utilization; approaches limit; blocked enrollment error shown.
- **Security** — Attempt to bypass limit by using admin role; attempt to modify license as org_admin.
- **Performance** — 1000 concurrent user activations in same org; counter consistency check.

## 17. Documentation & Training
