## 1. Problem Statement

Lextures currently has no email delivery system (§6.2 is itself missing), and when email is added, every transactional notification will initially use hard-coded templates. Districts and universities require on-brand, customized email communications — enrollment confirmations, grade postings, assignment reminders, and password-reset emails should carry the institution's branding and voice. Without an admin-controlled template editor, every copy change requires an engineering deployment, which is untenable for self-service customers.

## 2. Goals

- Deliver a WYSIWYG-style email template editor in the admin console that allows org admins to customize the HTML/text body of transactional email templates.
- Support merge-field syntax (e.g., `{{user.first_name}}`, `{{course.title}}`) with a live preview showing resolved values from sample data.
- Version transactional email templates so admins can revert to a previous version if a change breaks deliverability.
- Provide sensible, accessible default templates that orgs can override without starting from scratch.
- Cross-reference §6.2 email notification infrastructure — the template system is the admin-facing authoring layer on top of the delivery layer.

## 3. Non-Goals

- Building the underlying email delivery service (§6.2).
- Marketing campaign emails or bulk newsletters.
- Custom email domains per org (§5.7 branding).
- SMS template editing.

## 4. Personas & User Stories

- **As a district communications coordinator**, I want to edit the "Welcome to Lextures" email so that it uses our district name, logo, and approved language.
- **As an org admin**, I want to preview the password-reset email with a real user's name filled in so that I can verify the merge fields resolve correctly before saving.
- **As a Lextures super-admin**, I want to restore the default template for an org whose custom template is breaking email delivery.
- **As an instructor**, I want the assignment-due-soon email to mention my course name automatically so that students know which assignment is due.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a list of all transactional email template slots (e.g., `welcome`, `password_reset`, `assignment_due_reminder`, `grade_posted`, `enrollment_confirmed`) with their current customization status.
- **FR-2.** The system MUST allow an org admin to edit the HTML and plain-text body of any template slot using an inline rich-text editor supporting basic formatting (bold, italic, links, images via URL).
- **FR-3.** The system MUST document and display all available merge fields for each template slot, with descriptions (e.g., `{{user.first_name}}` = "Recipient's first name").
- **FR-4.** The system MUST provide a live preview panel rendering the template with sample merge-field values as the admin types.
- **FR-5.** The system MUST validate that all merge-field tokens in the saved template are recognized; unrecognized tokens MUST produce a warning (not a hard error) before save.
- **FR-6.** The system MUST store template versions: each save creates a new version; admins can view version history and restore any prior version.
- **FR-7.** The system MUST allow an org admin to "Reset to default" for any template slot, which restores the Lextures system default without deleting the version history.
- **FR-8.** The system SHOULD allow admins to send a test email to themselves with the current template and real merge values resolved against their own user account.
- **FR-9.** The system MUST support both HTML and plain-text versions of each template; if only HTML is provided, a plain-text fallback MUST be auto-generated by stripping HTML tags.
- **FR-10.** The system SHOULD allow an org admin to set a custom `Reply-To` address and a custom sender display name per template slot or org-wide.

## 6. Non-Functional Requirements

- **Performance** — Template save MUST complete in ≤ 500 ms; preview render (server-side merge resolution) MUST complete in ≤ 200 ms.
- **Security** — Template HTML rendered via `mjml` or equivalent; must sanitize stored HTML to prevent XSS when rendering in-browser preview; merge fields resolved server-side only (not client-side eval); `org_admin` role required.
- **Privacy & Compliance** — Template content stored per org; no cross-org leakage; test-email sending must only target the requesting admin's own email address; FERPA-relevant emails (grade notifications) must have appropriate FERPA-compliant footer text enforced by the default template.
- **Accessibility** — Template editor toolbar keyboard navigable; merge-field insertion via keyboard shortcut; preview panel announced by screen reader when updated.
- **Scalability** — Template table indexed by org_id; low-volume feature, no scaling concern.
- **Reliability** — If template rendering fails at send time, fall back to the system default template.
- **Observability** — `email_template_saves_total`, `email_template_test_sends_total` counters; log template render errors.
- **Maintainability** — Template rendering in `server/src/services/email_templates.rs`; Handlebars or Tera used as the merge-field engine (matches Rust ecosystem).
- **Internationalization** — Default templates available in all supported locales; per-org locale applied during merge resolution.
- **Backward compatibility** — If no custom template exists for a slot, system default is used automatically.

## 7. Acceptance Criteria

- **AC-1.** *Given* an org admin opens the `welcome` template slot, *When* they replace `{{user.first_name}}` with a valid merge field token and save, *Then* the next welcome email sent to a real user contains the resolved first name.
- **AC-2.** *Given* an admin types an unrecognized merge field `{{foo.bar}}`, *When* they save, *Then* a yellow warning banner indicates the unknown field but the save is not blocked.
- **AC-3.** *Given* a saved custom template, *When* the admin clicks "Reset to default", *Then* subsequent sends use the system default and the custom version is preserved in version history.
- **AC-4.** *Given* an admin sends a test email, *When* the request is processed, *Then* the email arrives at the admin's own address within 60 s with all merge fields resolved.
- **AC-5.** *Given* a template containing a `<script>` tag is submitted, *When* the server saves the template, *Then* the `<script>` tag is stripped from the stored HTML and the preview renders without executing script.
- **AC-6.** *Given* two versions of the `password_reset` template, *When* the admin views version history, *Then* both versions are listed with timestamps; selecting v1 and clicking "Restore" makes v1 the active template.

## 8. Data Model

```sql
-- 0XX_email_templates.sql
CREATE TABLE email_template_slots (
  id           TEXT PRIMARY KEY,     -- e.g. 'welcome', 'password_reset'
  description  TEXT NOT NULL,
  merge_fields JSONB NOT NULL,       -- { "user.first_name": "Recipient first name", ... }
  default_html TEXT NOT NULL,
  default_text TEXT NOT NULL
);

CREATE TABLE org_email_templates (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id      UUID NOT NULL REFERENCES organizations(id),
  slot_id     TEXT NOT NULL REFERENCES email_template_slots(id),
  html_body   TEXT NOT NULL,
  text_body   TEXT,
  reply_to    TEXT,
  sender_name TEXT,
  created_by  UUID REFERENCES users(id),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  is_active   BOOLEAN NOT NULL DEFAULT true
);
CREATE UNIQUE INDEX org_template_active ON org_email_templates(org_id, slot_id)
  WHERE is_active = true;
CREATE INDEX org_template_history ON org_email_templates(org_id, slot_id, created_at DESC);
```

## 9. API Surface

```
GET    /api/admin/email-templates              -> TemplateSlot[]
GET    /api/admin/email-templates/:slot_id     -> TemplateSlotWithActiveVersion
PUT    /api/admin/email-templates/:slot_id     { html_body, text_body, reply_to, sender_name }
                                               -> OrgEmailTemplate (new version)
GET    /api/admin/email-templates/:slot_id/history -> OrgEmailTemplate[]
POST   /api/admin/email-templates/:slot_id/restore { version_id } -> OrgEmailTemplate
POST   /api/admin/email-templates/:slot_id/reset   -> 204
POST   /api/admin/email-templates/:slot_id/test    -> 202 (sends test email to requesting admin)
GET    /api/admin/email-templates/:slot_id/preview { sample_data } -> { html, text }
```

## 10. UI / UX

New page: `clients/web/src/pages/admin/AdminEmailTemplates.tsx`

Layout: two-panel — left panel is the editor (TipTap-based), right panel is a live preview (iframe sandbox rendering the compiled template). Merge-field picker dropdown accessible via `{{` trigger or toolbar button.

Version history: expandable drawer below the editor showing timeline of saves with restore button per version.

Test email button: triggers confirmation dialog showing recipient (admin's email) before sending.

Mobile: editor full-screen with preview accessible via tab switch.

Cross-reference `clients/web/src/components/editor/EditorWithPrompt.tsx` for editor component patterns.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Admin introduces XSS via HTML template | M | H | Server-side DOMPurify/ammonia sanitization before store and render |
| Broken template causes all emails to fail | M | H | Per-slot fallback to system default on render error; alert on fallback |
| MJML compilation is slow for large templates | L | M | Cache compiled MJML output keyed by template version hash |
| Merge fields with missing data at send time | M | M | Default empty-string for missing fields; log missing field events |

## 15. Rollout Plan

- Feature flag: `email_template_editor_enabled` (OpenFeature, default `false`).
- Phase 1: System default templates locked in (not admin-editable); ship as part of §6.2 email rollout.
- Phase 2: Enable template editor for super_admin and pilot org_admins.
- Phase 3: GA for all orgs once email infrastructure is stable.
- Rollback: Disable flag; templates fall back to system defaults; no data loss.

## 16. Test Plan

- **Unit** — Tera/Handlebars merge resolution with all merge fields; HTML sanitization (ammonia/DOMPurify); plain-text auto-generation from HTML.
- **Integration** — Save template → retrieve → send test email → verify rendered content matches saved template with resolved merge fields.
- **End-to-end** — Playwright: admin edits welcome template → previews with sample data → saves → verifies version history entry.
- **Security** — Submit template with `<script>`, `onerror` attributes, JS URL hrefs; verify all stripped.
- **Accessibility** — axe-core on editor page; test merge-field picker keyboard accessibility.
- **Manual exploratory** — Test MJML rendering across email clients (Gmail, Outlook, Apple Mail) using Litmus or Email on Acid.

## 17. Documentation & Training

- Help center: "Customizing Email Templates" — merge field reference, version history, reset to default.
- Admin guide: MJML rendering limitations; supported HTML tags.
- Internal runbook: Adding new template slots when new notification types are introduced.

## 18. Open Questions

1. Should we use MJML or raw HTML with inline CSS? MJML produces better cross-client output but requires a compilation step.
2. Should image uploads be allowed in email templates, or only external URL references?
3. How long should template version history be retained?
4. Should we provide a library of pre-built template themes (e.g., "modern", "classic", "K-12 friendly")?
