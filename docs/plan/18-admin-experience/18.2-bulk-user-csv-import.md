## 1. Problem Statement

There is currently no way for a district admin or institution IT staff to provision users in bulk. Every user must register individually or be created one at a time through ad-hoc tooling. K-12 districts and universities routinely onboard hundreds to tens of thousands of students and staff at the start of each academic year and cannot afford manual, per-user flows. This gap is a BLOCKER for any institutional sale and is closely related to the OneRoster CSV standard (§4.3), which Lextures must eventually fully implement.

## 2. Goals

- Allow an `org_admin` to upload a CSV file that creates, updates, or deactivates users in bulk within their organization.
- Support both a simple Lextures-native CSV format and the OneRoster 1.2 `users.csv` column schema as a recognized import profile.
- Run the import asynchronously via the background job queue with progress reporting.
- Provide a dry-run / preview mode showing row-level parse errors before any DB writes.
- Emit an admin audit-log event for every user created, updated, or deactivated by the import.

## 3. Non-Goals

- Full OneRoster 1.2 REST API implementation — that is §4.3.
- Clever or ClassLink integration — that is §4.4.
- Real-time roster sync via webhooks — manual CSV upload is the scope here.
- Importing course content or enrollments in this plan (a future extension).

## 4. Personas & User Stories

- **As a district IT admin**, I want to upload the CSV export from our SIS so that all teachers and students are provisioned before the first day of school.
- **As an institution registrar**, I want to merge an updated CSV mid-semester so that newly added students are created and withdrawn students are deactivated without affecting existing accounts.
- **As an org admin**, I want to preview all validation errors before committing the import so that I do not accidentally corrupt user accounts.
- **As a Lextures super-admin**, I want to see import job history and error logs so that I can assist customers with failed imports.

## 5. Functional Requirements

- **FR-1.** The system MUST accept CSV file uploads up to 50 MB containing user records with at minimum: `email`, `first_name`, `last_name`, `role` (teacher/student/admin).
- **FR-2.** The system MUST support an optional `external_id` column for SIS/OneRoster matching so that re-imports do not create duplicates.
- **FR-3.** The system MUST offer a dry-run mode that validates all rows and returns a structured error report without writing to the database.
- **FR-4.** The system MUST process imports asynchronously via the background job queue; the upload endpoint MUST return a job ID immediately and a 202 Accepted status.
- **FR-5.** The system MUST provide a job-status polling endpoint returning `{ status, processed_rows, error_rows, errors: [{row, column, message}] }`.
- **FR-6.** The system MUST support three merge strategies: `create_only`, `upsert` (create + update matching records), and `sync` (create + update + deactivate missing records).
- **FR-7.** The system MUST recognize the OneRoster 1.2 `users.csv` column schema as an import profile and map `sourcedId → external_id`, `givenName → first_name`, `familyName → last_name`, `role → role`.
- **FR-8.** The system MUST reject imports containing users outside the admin's org scope (FR enforcing multi-tenant isolation).
- **FR-9.** The system SHOULD send an email notification to the uploading admin when a large import (>100 rows) completes or fails.
- **FR-10.** The system SHOULD allow downloading a result CSV with per-row outcomes (`created`, `updated`, `deactivated`, `skipped`, `error`) after job completion.

## 6. Non-Functional Requirements

- **Performance** — Import of 10 000 rows MUST complete in ≤ 5 minutes; DB writes MUST be batched (batch size 500) using `INSERT … ON CONFLICT DO UPDATE`.
- **Security** — CSV upload endpoint requires `org_admin` role; uploaded files MUST be virus-scanned (ClamAV §8.6 dependency) before parsing; file contents MUST NOT be logged raw to prevent PII leakage.
- **Privacy & Compliance** — Imported PII (email, name) stored only in the existing `users` table; no secondary copies retained after job completion; FERPA-compliant audit trail created for every account change.
- **Accessibility** — Upload form, progress indicator, and error table MUST be keyboard navigable and screen-reader announced per WCAG 2.1 AA.
- **Scalability** — Background job partitioned per org; multiple orgs can import simultaneously without contention.
- **Reliability** — Import job MUST be idempotent; re-queuing a failed job with the same file produces the same result; rows processed before failure are NOT re-applied on retry (use a job-level cursor).
- **Observability** — `import_jobs_total{status}` Prometheus counter; log structured events `{job_id, org_id, rows_processed, errors_count}` on completion.
- **Maintainability** — CSV parsing logic isolated in `server/src/services/csv_import.rs`; import profiles (Lextures native, OneRoster) as a pluggable enum.
- **Internationalization** — Error messages returned in the org's configured locale.
- **Backward compatibility** — Existing single-user creation API unchanged.

## 7. Acceptance Criteria

- **AC-1.** *Given* a valid 500-row CSV with Lextures-native columns, *When* an org_admin uploads it with `merge_strategy=upsert`, *Then* the API returns HTTP 202 with a `job_id` and all 500 users are created/updated within 60 s.
- **AC-2.** *Given* a CSV with 10 rows where 3 have invalid email addresses, *When* the admin requests a dry-run, *Then* the response lists all 3 invalid rows with column and message; no DB writes occur.
- **AC-3.** *Given* a OneRoster `users.csv` is uploaded with the `oneroster_v1.2` profile, *When* the import completes, *Then* each user's `external_id` matches the `sourcedId` column value.
- **AC-4.** *Given* `merge_strategy=sync` and 50 existing users not in the new CSV, *When* the import completes, *Then* those 50 users have `is_active=false` and an audit-log event `user.deactivated` exists for each.
- **AC-5.** *Given* an admin uploads a CSV containing a user email that belongs to a different org, *When* the import processes that row, *Then* it is skipped with an `org_isolation_violation` error and no changes are made to the foreign user record.
- **AC-6.** *Given* an import job fails mid-way, *When* the admin retries the same job, *Then* already-processed rows are not re-applied and the job resumes from the failure cursor.

## 8. Data Model

```sql
-- 0XX_import_jobs.sql
CREATE TYPE import_job_status AS ENUM ('queued', 'running', 'complete', 'failed');
CREATE TYPE import_merge_strategy AS ENUM ('create_only', 'upsert', 'sync');

CREATE TABLE import_jobs (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id           UUID NOT NULL REFERENCES organizations(id),
  actor_id         UUID NOT NULL REFERENCES users(id),
  status           import_job_status NOT NULL DEFAULT 'queued',
  merge_strategy   import_merge_strategy NOT NULL,
  import_profile   TEXT NOT NULL DEFAULT 'lextures_native',
  total_rows       INT,
  processed_rows   INT NOT NULL DEFAULT 0,
  error_rows       INT NOT NULL DEFAULT 0,
  errors_jsonb     JSONB,
  cursor_row       INT NOT NULL DEFAULT 0,
  result_file_key  TEXT,               -- object-storage key for result CSV
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX import_jobs_org_status ON import_jobs(org_id, status);
```

Add `external_id TEXT` column to `users` if not present (used as SIS upsert key).

## 9. API Surface

```
POST   /api/admin/imports               multipart/form-data { file, merge_strategy, profile, dry_run }
                                        -> 202 { job_id }
GET    /api/admin/imports/:job_id       -> ImportJobStatusDto
GET    /api/admin/imports/:job_id/result -> 302 redirect to signed object-storage URL
GET    /api/admin/imports               -> Paginated<ImportJobSummaryDto>
```

File size limit: 50 MB enforced at the Axum multipart handler. Rate limit: 5 uploads per org per hour.

## 10. UI / UX

New page: `clients/web/src/pages/admin/AdminImport.tsx`

Key user flows:
1. Admin opens Import page → selects import profile (Lextures Native / OneRoster 1.2) → uploads CSV → chooses merge strategy → clicks "Dry Run" → reviews error table → clicks "Import".
2. Progress bar polls job status every 3 s → on completion shows summary card (created/updated/deactivated/errors) → download result CSV link.

Error table: sortable by row number; columns: Row, Column, Error Message. Sticky header. Keyboard navigable.

Empty state: "No imports yet. Upload a CSV to get started." with link to CSV template download. Mobile: full-screen upload flow; progress as percentage text (no wide tables on small screens).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Large CSV (10 k rows) times out HTTP request | H | H | Async job pattern with 202 response eliminates timeout risk |
| Duplicate users created on re-import | M | H | `external_id` upsert key + job idempotency cursor |
| PII in CSV logged to stdout | M | H | Strip file content from tracing spans; log only row counts |
| CSV injection (formula injection) in name fields | M | M | Sanitize values before writing; block `=`, `+`, `-`, `@` lead chars |

## 15. Rollout Plan

- Feature flag: `bulk_csv_import_enabled` (OpenFeature, default `false`).
- Phase 1: Internal + pilot district with <500 users; `create_only` strategy only.
- Phase 2: Enable `upsert` and `sync` strategies after audit in pilot.
- Phase 3: GA for all orgs.
- Rollback: Disable flag; no destructive DB changes; import job table can remain dormant.

## 16. Test Plan

- **Unit** — CSV parser for Lextures-native and OneRoster profiles; merge-strategy logic; org-isolation boundary check.
- **Integration** — Upload → job queue → DB writes → status polling round-trip; dry-run returns errors without DB writes; sync deactivates missing users.
- **End-to-end** — Playwright: upload 50-row CSV → confirm dry-run errors → proceed to import → verify user table updated.
- **Security** — Upload a CSV with SQL-injection payloads in name fields; verify no injection; attempt upload as non-admin role; attempt cross-org user inclusion.
- **Accessibility** — axe-core on upload page and error table; screen-reader test of progress announcement.
- **Performance** — k6: upload 10 000-row CSV; assert job completes in ≤ 5 min; DB batch-write timing profiled.
- **Manual exploratory** — Manually test malformed CSV (wrong delimiters, BOM, Windows CRLF, non-UTF-8 encoding).

## 17. Documentation & Training

## 18. Open Questions

1. Should we support Excel `.xlsx` as an alternative to CSV, given many district IT staff export from SIS to Excel?
2. What is the maximum supported row count per import? (50 MB CSV limit implies ~500 k rows — do we want a lower logical cap, e.g., 50 000?)
3. Should password-reset emails be auto-sent to newly created users, or does the admin control that?
4. Should deactivated users in `sync` mode be hard-deleted or soft-deactivated (is_active = false)?
