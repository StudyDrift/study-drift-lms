## 1. Problem Statement

When Lextures undergoes planned maintenance or experiences an unplanned outage, there is no mechanism to surface a notification inside the application without a code deployment. Admins must communicate outages externally (email, Slack) while users may continue trying to use a degraded system. Industry standard practice is a site-wide dismissable banner that admins can toggle in seconds from the admin console, optionally synchronized with a status page (Statuspage / Status.io).

## 2. Goals

- Allow `org_admin` and `super_admin` to publish, update, and dismiss a site-wide or org-scoped banner message with a severity level and optional link.
- Display the banner to all users on every page until dismissed or expired.
- Persist the dismissed state per user (don't re-show after dismissal unless banner content changes).
- Provide a lightweight integration hook for Statuspage or Status.io webhooks to auto-create banners.

## 3. Non-Goals

- Full incident management or status-page hosting (use Statuspage / Status.io for that).
- Per-course banners (that is a different announcement feature).
- Push notifications for outages (§6.3).

## 4. Personas & User Stories

- **As a super admin**, I want to publish a banner saying "Scheduled maintenance Sunday 2am–4am UTC" so that all users are informed before the window.
- **As an org admin**, I want to display an org-specific notice ("Our district is migrating to a new SSO provider next Monday") to only our users.
- **As a student**, I want to dismiss the banner and have it not reappear so that it does not clutter my view after I have read it.
- **As a DevOps engineer**, I want our Statuspage incident to automatically trigger a banner in the app so that I do not have to update two systems separately.

## 5. Functional Requirements

- **FR-1.** The system MUST allow `super_admin` to create a global banner visible to all users across all orgs.
- **FR-2.** The system MUST allow `org_admin` to create an org-scoped banner visible only to users in their organization.
- **FR-3.** Each banner MUST have: message (plain text, max 500 chars), severity (`info` | `warning` | `error`), optional CTA link, optional start time, optional expiry time.
- **FR-4.** The system MUST serve the active banner via a lightweight endpoint (`GET /api/status/banner`) that returns `null` when no banner is active; this endpoint MUST be unauthenticated to support display on the login page.
- **FR-5.** The system MUST allow users to dismiss the banner; the dismissed state MUST be persisted (localStorage keyed by banner ID, or a server-side preference) so it does not reappear on page refresh.
- **FR-6.** If a new banner is published (or an existing one updated), previously dismissed banners MUST re-appear for users.
- **FR-7.** The system SHOULD auto-expire banners at the configured expiry time without admin intervention.
- **FR-8.** The system SHOULD provide a webhook endpoint (`POST /api/admin/banners/statuspage-webhook`) that accepts a Statuspage incident payload and auto-creates/resolves a banner.

## 6. Non-Functional Requirements

- **Performance** — Banner endpoint MUST be cacheable with a short TTL (10 s) to avoid DB hit on every page load; CDN-cacheable with `Cache-Control: max-age=10, stale-while-revalidate=60`.
- **Security** — Banner creation/update/delete requires `org_admin`/`super_admin`; unauthenticated GET is intentional (login-page display); Statuspage webhook must be HMAC-verified.
- **Privacy & Compliance** — Banner messages MUST NOT contain PII; validation on save rejects email-like patterns as a heuristic.
- **Accessibility** — Banner rendered as `<aside role="status">` with appropriate `aria-live`; dismiss button has descriptive label; color alone does not convey severity (icon + text).
- **Scalability** — Single-row lookup; trivially scalable.
- **Reliability** — If banner DB query fails, the app MUST render without a banner rather than fail to load.
- **Observability** — `banner_active{scope,severity}` gauge metric; log banner create/update/dismiss events.
- **Maintainability** — Banner component: `clients/web/src/components/StatusBanner.tsx`; server route: `server/src/routes/status.rs`.
- **Internationalization** — Banner message stored as authored (admin's language); future multilingual support via §11 i18n pipeline.
- **Backward compatibility** — If no banner exists, endpoint returns `null`; frontend renders nothing. Fully additive.

## 7. Acceptance Criteria

- **AC-1.** *Given* a super_admin creates a `warning` banner with message "Maintenance at midnight", *When* any authenticated user loads any page, *Then* the banner is visible with warning styling.
- **AC-2.** *Given* an org_admin creates an org-scoped banner, *When* a user from a different org loads the app, *Then* they do not see the banner.
- **AC-3.** *Given* an active banner is displayed, *When* a user clicks "Dismiss", *Then* the banner disappears and does not reappear on page refresh within the same banner ID.
- **AC-4.** *Given* a banner with an expiry time of 1 minute is created, *When* 1 minute elapses, *Then* `GET /api/status/banner` returns `null` and the banner no longer renders.
- **AC-5.** *Given* a Statuspage incident webhook is received with correct HMAC signature, *When* the webhook is processed, *Then* a new error-severity banner is created or the existing one is updated to match the incident status.
- **AC-6.** *Given* the banner is rendered, *When* a screen reader focuses on it, *Then* the severity and message are announced; the dismiss button label is "Dismiss maintenance notice" (not just "x").

## 8. Data Model

```sql
-- 0XX_banners.sql
CREATE TYPE banner_severity AS ENUM ('info', 'warning', 'error');
CREATE TYPE banner_scope AS ENUM ('global', 'org');

CREATE TABLE banners (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  scope       banner_scope NOT NULL,
  org_id      UUID REFERENCES organizations(id),  -- NULL for global
  message     TEXT NOT NULL CHECK (length(message) <= 500),
  severity    banner_severity NOT NULL DEFAULT 'info',
  cta_text    TEXT,
  cta_url     TEXT,
  starts_at   TIMESTAMPTZ,
  expires_at  TIMESTAMPTZ,
  is_active   BOOLEAN NOT NULL DEFAULT true,
  created_by  UUID NOT NULL REFERENCES users(id),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX banners_active ON banners(is_active, scope, org_id);
```

## 9. API Surface

```
GET    /api/status/banner                   -> BannerDto | null   (public, cached 10 s)
GET    /api/admin/banners                   -> BannerDto[]
POST   /api/admin/banners                   -> BannerDto
PUT    /api/admin/banners/:id               -> BannerDto
DELETE /api/admin/banners/:id               -> 204
POST   /api/admin/banners/statuspage-webhook -> 204  (HMAC-authenticated)
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Admin publishes banner with PII | L | M | Server-side pattern validation; audit log |
| Banner endpoint becomes hot path bottleneck | L | M | 10 s CDN cache; query a single indexed row |
| Statuspage webhook replay/spoof | M | M | HMAC-SHA256 signature verification; nonce TTL |

## 15. Rollout Plan

- Feature flag: `maintenance_banner_enabled` (OpenFeature, default `true` after first deploy since it is low-risk).
- Ship as a zero-downtime additive change.
- Rollback: Disable flag; banner component returns null; DB table remains.

## 16. Test Plan

- **Unit** — Banner expiry logic; org-scope filter; HMAC webhook verification.
- **Integration** — Create banner → GET /status/banner returns it → DELETE → GET returns null.
- **End-to-end** — Playwright: create banner as admin → verify visible on student dashboard → dismiss → verify gone.
- **Security** — Attempt to create banner without admin role; attempt Statuspage webhook without valid HMAC.
- **Accessibility** — axe-core; screen-reader announcement of aria-live region.

## 17. Documentation & Training

- Help center: "Publishing Maintenance Notices" — how to create, schedule, and clear banners.
- Admin guide: Statuspage integration setup.
- Internal runbook: manual banner creation during incident response.

## 18. Open Questions

1. Should banner dismissal be stored server-side (per user preference) or client-side (localStorage)? Server-side survives cross-device, but adds a table.
2. Should we allow Markdown in banner messages for bold/links, or keep it plain text only?
