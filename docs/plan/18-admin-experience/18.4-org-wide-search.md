## 1. Problem Statement

Current search in Lextures is user-scoped: a student sees only their enrolled courses, and an instructor sees only courses they own. There is no cross-course, cross-user search for administrators. This means a district admin who needs to locate a specific student, find a course by any instructor, or audit all uses of a particular piece of content must resort to direct database queries. For K-12 districts with hundreds of courses and thousands of students, this is untenable.

## 2. Goals

- Provide a unified admin search bar that returns results across users, courses, and content items scoped to the admin's organization.
- Support free-text search by user name/email, course title/description, and content title.
- Return results ranked by relevance with entity-type faceting (users / courses / content).
- Integrate the search into the admin console command palette and as a standalone page.
- Keep the implementation simple and maintainable using Postgres full-text search (pg_trgm + tsvector) before reaching for Elasticsearch.

## 3. Non-Goals

- Search of submission content or quiz answers (private student data — requires careful FERPA scoping beyond this plan).
- Student-facing cross-course search (separate feature).
- Semantic / vector search (a future AI enhancement; basic keyword search is the scope).
- Org-wide content analytics or reporting (§9).

## 4. Personas & User Stories

- **As a district IT admin**, I want to search "Smith" and see all users named Smith across all schools in my district so that I can quickly find and manage their accounts.
- **As a Lextures super-admin**, I want to search for a course title across all orgs so that I can help a customer locate a misconfigured course.
- **As an org admin**, I want to search for "biology" and see all courses with that term in the title or description so that I can ensure naming consistency across departments.
- **As an instructor**, I want to search within my own courses (existing behavior, not changed by this plan).

## 5. Functional Requirements

- **FR-1.** The system MUST provide a `GET /api/admin/search?q=&types=` endpoint returning results across `users`, `courses`, and `content_items` scoped to the authenticated admin's org.
- **FR-2.** The system MUST support entity-type filtering via the `types` query parameter (e.g., `types=users,courses`); omitting the parameter returns all entity types.
- **FR-3.** The system MUST return a maximum of 5 results per entity type in the omnisearch response, with a link to a full-results page for each type.
- **FR-4.** The system MUST use Postgres full-text search (`tsvector`/`tsquery`) with `pg_trgm` for fuzzy matching on user name/email, course title, and content title fields.
- **FR-5.** The system MUST enforce org-scoping so that results never include entities outside the admin's organization.
- **FR-6.** The system MUST return results within 500 ms p95 for queries against an org with 50 000 rows across all indexed entities.
- **FR-7.** The system SHOULD highlight the matching text fragment in each result (return the matched snippet).
- **FR-8.** A `super_admin` MUST be able to optionally pass `org_id` to search within a specific organization.
- **FR-9.** The system SHOULD provide dedicated full-results pages for each entity type (`/admin/search?q=&type=users`) with pagination.
- **FR-10.** Admin search queries SHOULD be logged (query text, result counts, actor) to support analytics on common support searches.

## 6. Non-Functional Requirements

- **Performance** — p95 ≤ 500 ms for omnisearch; p95 ≤ 300 ms for single-type paginated results; GIN indexes on all tsvector columns.
- **Security** — Admin role required; org-scope enforced at query level (WHERE org_id = ?); no search result should expose PII of users in other orgs.
- **Privacy & Compliance** — Search log entries must not store the full query if it contains PII patterns (email addresses); hash or truncate before storing.
- **Accessibility** — Combobox search with ARIA `role="combobox"`, live region for result count announcement; results list keyboard navigable.
- **Scalability** — GIN indexes on tsvector columns scale well; if org grows beyond 500 k rows, add an Elasticsearch index layer behind the same API interface.
- **Reliability** — Search endpoint degrades gracefully if index is stale; return cached results with a staleness indicator.
- **Observability** — `admin_search_duration_ms` histogram; `admin_search_results_total{entity_type}` counter.
- **Maintainability** — Search logic in `server/src/services/admin_search.rs`; SQL queries use parameterized planar_ts_query to prevent injection.
- **Internationalization** — Postgres text-search dictionary configurable per org locale.
- **Backward compatibility** — Existing student/instructor search endpoints unchanged.

## 7. Acceptance Criteria

- **AC-1.** *Given* an org with 200 users including "Alice Johnson" and "Bob Johnson", *When* an org_admin searches `q=johnson`, *Then* both users appear in results within 500 ms with name match highlighted.
- **AC-2.** *Given* a search query that would return results from a different org, *When* the query is executed by an org_admin, *Then* zero cross-org results are included in the response.
- **AC-3.** *Given* a search with `types=courses` only, *When* the query executes, *Then* only course entities are returned (no users or content items).
- **AC-4.** *Given* a typo query `q=biologe`, *When* the search executes with pg_trgm fuzzy matching enabled, *Then* courses titled "Biology" appear in results (trigram similarity ≥ 0.3).
- **AC-5.** *Given* a `super_admin` passes `org_id=<other_org>`, *When* the search executes, *Then* results are scoped to that org only.
- **AC-6.** *Given* the search combobox is open, *When* the user presses arrow keys, *Then* focus moves through results and pressing Enter navigates to the selected entity.

## 8. Data Model

Add tsvector columns and GIN indexes:

```sql
-- 0XX_search_indexes.sql
ALTER TABLE users ADD COLUMN IF NOT EXISTS
  search_vector tsvector
  GENERATED ALWAYS AS (
    to_tsvector('english', coalesce(first_name,'') || ' ' || coalesce(last_name,'') || ' ' || coalesce(email,''))
  ) STORED;
CREATE INDEX users_search_gin ON users USING GIN(search_vector);
CREATE INDEX users_email_trgm ON users USING GIN(email gin_trgm_ops);

ALTER TABLE courses ADD COLUMN IF NOT EXISTS
  search_vector tsvector
  GENERATED ALWAYS AS (
    to_tsvector('english', coalesce(title,'') || ' ' || coalesce(description,''))
  ) STORED;
CREATE INDEX courses_search_gin ON courses USING GIN(search_vector);

-- content_items (modules/lessons) — similar pattern
```

No new tables needed; search is read-only over existing data.

## 9. API Surface

```
GET /api/admin/search
  ?q=<string>          required
  &types=users,courses,content   optional, default all
  &org_id=<uuid>       super_admin only
  -> AdminSearchResponse {
       users: SearchResult[],
       courses: SearchResult[],
       content: SearchResult[],
       took_ms: number
     }

GET /api/admin/search/users    ?q=&page=&per_page=   -> Paginated<AdminUserDto>
GET /api/admin/search/courses  ?q=&page=&per_page=   -> Paginated<AdminCourseDto>
```

Rate limit: 30 req/min per admin to prevent scraping.

## 10. UI / UX

Components:
- `clients/web/src/components/admin/AdminSearchBar.tsx` — global search bar in admin header; opens a Cmdk-style popover with grouped results (Users, Courses, Content).
- `clients/web/src/pages/admin/AdminSearchResults.tsx` — full results page with tabs per entity type.

User flow:
1. Admin presses `Cmd+K` (or clicks search bar) → types query → sees grouped results in popover → clicks result → navigates to user/course detail.
2. Clicks "See all users" link in popover → lands on paginated full-results page with filter sidebar.

Loading state: skeleton result rows. Empty state: "No results for '[query]'" with suggestions. Mobile: full-screen search overlay.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| GIN indexes slow down write throughput on high-insert orgs | M | M | Benchmark INSERT + reindex cost; consider async tsvector refresh |
| Org-scope bypass via crafted query params | L | H | Parameterized queries; integration test asserting cross-org isolation |
| Search returns stale data after bulk import | M | L | GENERATED ALWAYS tsvector columns are always up-to-date |
| pg_trgm not available in production Postgres | L | M | Document extension requirement in deployment runbook |

## 15. Rollout Plan

- Feature flag: `admin_search_enabled` (OpenFeature, default `false`).
- Phase 1: Enable GIN index migration for all orgs (non-breaking, additive).
- Phase 2: Enable search UI for super_admin and pilot org_admin accounts.
- Phase 3: GA.
- Rollback: Disable feature flag; search bar disappears; indexes remain (no harm).

## 16. Test Plan

- **Unit** — search query construction (tsvector, trigram); org-scope WHERE clause presence; super_admin org_id override.
- **Integration** — Seed org with 500 users/courses; run omnisearch; verify result count and org isolation; verify pagination.
- **End-to-end** — Playwright: admin opens search → types partial name → clicks result → verifies navigation.
- **Security** — SQL injection in `q` parameter; cross-org result leakage; unauthorized role access.
- **Accessibility** — axe-core on search popover; keyboard navigation through results; screen-reader result-count announcement.
- **Performance** — pg_bench: 10 concurrent search queries on 50 k-row org; p95 ≤ 500 ms.
