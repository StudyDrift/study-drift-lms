## 1. Problem Statement

Every district and institution has internal metadata fields that do not fit the standard Lextures data model: student ID numbers, grade levels, department codes, disability flags, Title I eligibility, or SIS-specific attributes. Currently there is no way to attach arbitrary key-value data to users, courses, or enrollments without modifying the core schema. This blocks institutional adoption because customers cannot populate, search, or export their own data fields from within Lextures.

## 2. Goals

- Allow org admins to define custom field schemas (name, type, allowed values) on users, courses, and enrollments within their org.
- Allow values to be set via the admin console, the CSV import (§18.2), and the API.
- Expose custom field values in admin tables, CSV exports, and the API.
- Enforce field-level access controls: some fields may be admin-only (e.g., Title I flag), others instructor-visible, others student-visible.

## 3. Non-Goals

- Replacing SIS integration (custom fields are a supplement, not a substitute for §4.3 OneRoster).
- Dynamic form rendering on the student-facing UX (that is a future enhancement).
- Complex conditional logic or inter-field validation rules.

## 4. Personas & User Stories

- **As a district IT admin**, I want to add a "Student ID" field to users and populate it via CSV import so that our internal systems can match records.
- **As an org admin**, I want to add a "Department" field to courses so that I can filter the course list by department in the admin console.
- **As a Lextures super-admin**, I want to see an org's custom field definitions when diagnosing data sync issues.
- **As a compliance officer**, I want to mark certain custom fields as admin-only so that students cannot see their own Title I eligibility flag through the student API.

## 5. Functional Requirements

- **FR-1.** The system MUST allow `org_admin` to define custom field schemas on three entity types: `user`, `course`, `enrollment`.
- **FR-2.** Each custom field definition MUST include: `key` (snake_case slug, unique per org+entity type), `label` (display name), `field_type` (`text` | `number` | `boolean` | `date` | `select`), optional `select_options` array, `is_required` boolean, `visibility` (`admin_only` | `instructor` | `student`).
- **FR-3.** The system MUST validate custom field values against their field_type and select_options on write.
- **FR-4.** The system MUST allow custom field values to be set and retrieved via the admin API alongside the parent entity.
- **FR-5.** The system MUST include custom field columns in CSV export of users and enrollments.
- **FR-6.** The CSV import (§18.2) MUST recognize custom field keys as valid import columns once they are defined for the org.
- **FR-7.** The system MUST enforce visibility: `admin_only` fields MUST NOT appear in API responses to `instructor` or `student` role tokens; `instructor` fields MUST NOT appear in `student` responses.
- **FR-8.** The system SHOULD allow reordering of custom field definitions to control the display order in admin tables.
- **FR-9.** The system SHOULD allow org admins to delete a custom field definition; existing values MUST be soft-deleted (retained in DB, not returned in API) to avoid data loss.

## 6. Non-Functional Requirements

- **Performance** — Custom field values stored as JSONB on the entity row or in a separate EAV table; JSONB path queries MUST complete in ≤ 100 ms for typical org sizes; GIN index on values JSONB.
- **Security** — Visibility enforcement at query layer (not just serialization); field key validation rejects injection-prone characters; `org_admin` only for schema management.
- **Privacy & Compliance** — Admin-only fields may contain sensitive data (FERPA-protected categories); access logged in `admin_audit_log`; DSAR export must include custom field values.
- **Accessibility** — Custom field forms in admin console MUST be keyboard navigable; select type renders as `<select>` not custom widget.
- **Scalability** — JSONB approach avoids schema migrations for each new field; GIN index handles filtering by field value.
- **Reliability** — Schema deletion is soft to prevent accidental data loss.
- **Observability** — `custom_fields_defined_total{entity_type}` gauge; log schema changes in admin audit log.
- **Maintainability** — Custom field logic in `server/src/services/custom_fields.rs`; shared by user, course, and enrollment routes.
- **Internationalization** — Field labels stored as authored; future i18n via §11.
- **Backward compatibility** — No changes to existing entity payloads unless consumer requests custom fields via `?include=custom_fields` query param.

## 7. Acceptance Criteria

- **AC-1.** *Given* an org admin defines a `text` custom field `student_id` on `user`, *When* a user's `student_id` is set via the admin API, *Then* `GET /api/admin/users/:id?include=custom_fields` returns `{ custom_fields: { student_id: "12345" } }`.
- **AC-2.** *Given* a `select` field with options `["Math", "Science"]`, *When* an attempt is made to set the value to `"History"`, *Then* the API returns HTTP 422 with a validation error.
- **AC-3.** *Given* an `admin_only` field `title_one_eligible`, *When* a student token calls `GET /api/me?include=custom_fields`, *Then* the field is absent from the response.
- **AC-4.** *Given* a custom field is defined, *When* a CSV export of users is downloaded by an admin, *Then* the CSV includes a column with the field's label and each user's value.
- **AC-5.** *Given* a CSV import file includes a column matching a custom field key, *When* the import runs, *Then* each user row has the custom field value set correctly.
- **AC-6.** *Given* a custom field definition is soft-deleted, *When* the admin views the entity, *Then* the field is absent from responses; the underlying values remain in the DB and can be queried by super_admin.

## 8. Data Model

```sql
-- 0XX_custom_fields.sql
CREATE TYPE custom_field_entity AS ENUM ('user', 'course', 'enrollment');
CREATE TYPE custom_field_type AS ENUM ('text', 'number', 'boolean', 'date', 'select');
CREATE TYPE custom_field_visibility AS ENUM ('admin_only', 'instructor', 'student');

CREATE TABLE custom_field_definitions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          UUID NOT NULL REFERENCES organizations(id),
  entity_type     custom_field_entity NOT NULL,
  key             TEXT NOT NULL CHECK (key ~ '^[a-z][a-z0-9_]*$'),
  label           TEXT NOT NULL,
  field_type      custom_field_type NOT NULL,
  select_options  TEXT[],
  is_required     BOOLEAN NOT NULL DEFAULT false,
  visibility      custom_field_visibility NOT NULL DEFAULT 'admin_only',
  sort_order      INT NOT NULL DEFAULT 0,
  deleted_at      TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX cfd_org_entity_key ON custom_field_definitions(org_id, entity_type, key)
  WHERE deleted_at IS NULL;

-- Values stored as JSONB on entity tables:
-- ALTER TABLE users        ADD COLUMN IF NOT EXISTS custom_fields JSONB NOT NULL DEFAULT '{}';
-- ALTER TABLE courses      ADD COLUMN IF NOT EXISTS custom_fields JSONB NOT NULL DEFAULT '{}';
-- ALTER TABLE enrollments  ADD COLUMN IF NOT EXISTS custom_fields JSONB NOT NULL DEFAULT '{}';
CREATE INDEX users_custom_fields_gin ON users USING GIN(custom_fields);
```

## 9. API Surface

```
GET    /api/admin/custom-fields?entity_type=user   -> CustomFieldDefinition[]
POST   /api/admin/custom-fields                    -> CustomFieldDefinition
PUT    /api/admin/custom-fields/:id                -> CustomFieldDefinition
DELETE /api/admin/custom-fields/:id                -> 204 (soft delete)

-- Values returned when ?include=custom_fields appended to entity endpoints:
GET    /api/admin/users/:id?include=custom_fields  -> UserDto & { custom_fields: Record<string,unknown> }
PATCH  /api/admin/users/:id                        body may include custom_fields object
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Admin creates fields exposing sensitive SIS data to students | M | H | Visibility enforcement at API layer with integration tests |
| JSONB column bloat on large orgs | M | M | Document max-fields limit (e.g., 50 per entity type per org) |
| Key naming collisions with future core columns | L | M | Reserve namespace; reject keys matching core field names |

## 15. Rollout Plan

- Feature flag: `custom_fields_enabled` (OpenFeature, default `false`).
- Phase 1: Schema migration adding JSONB column to users/courses/enrollments.
- Phase 2: Admin field-definition UI enabled for pilot orgs.
- Phase 3: CSV import column recognition enabled.
- Phase 4: GA.

## 16. Test Plan
