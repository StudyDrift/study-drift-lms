## 1. Problem Statement

Today there is no central administrative console for an organization, district, or institution admin to govern users, courses, terms, integrations, or branding. Every management action must be performed by a global superuser via direct database access or individual course-level controls. Without a dedicated admin surface, Lextures cannot pass procurement evaluations for K-12 districts or higher-education institutions, both of which require delegated administrative capability as a baseline.

## 2. Goals

- Deliver a single `/admin` route group accessible only to users with the `org_admin` or `super_admin` role.
- Provide dashboards for user management, course oversight, term/section hierarchy, integration status, and branding settings.
- Support RBAC-scoped delegation so a district principal can administer only their school's sub-account.
- Establish the UI shell and navigation that all other admin sub-features (18.2–18.8) plug into.
- Emit structured audit-log events for every write action taken from the console.

## 3. Non-Goals

- Building individual admin sub-features (CSV import, impersonation, seat management) — those are separate plans.
- Building the multi-tenancy org/school/term entity model — that is §5.1–5.4, which must land first.
- End-user analytics dashboards — covered in §9.

## 4. Personas & User Stories

- **As a district IT admin**, I want a single dashboard showing all schools, courses, and enrollment counts so that I can assess platform utilization without running SQL queries.
- **As a school principal**, I want to manage teachers and students scoped to my school only so that I do not accidentally modify another school's data.
- **As a Lextures super-admin**, I want to inspect any org's settings, integrations, and billing status so that I can provide effective customer support.
- **As an instructor**, I want to know who my institution's admin contact is so that I can escalate access issues quickly.

## 5. Functional Requirements

- **FR-1.** The system MUST restrict `/admin/*` routes to users with role `org_admin` or `super_admin`; any request from a lower-privilege user MUST return HTTP 403.
- **FR-2.** The system MUST present an overview dashboard showing total users, active courses, pending enrollments, and storage consumption for the admin's scoped organization.
- **FR-3.** The system MUST provide a user-management table with search, filter by role, sort, and pagination (page size 25/50/100).
- **FR-4.** The system MUST provide a course-management table with search by title, instructor, status (active/archived/draft), and term.
- **FR-5.** The system MUST provide a section for org-level settings: name, subdomain slug, logo upload, primary brand color, timezone, locale, and custom email-from display name.
- **FR-6.** The system MUST provide a read-only integration status panel showing connected SSO, OneRoster, SIS, and webhook endpoint health.
- **FR-7.** The system SHOULD allow an `org_admin` to delegate admin rights to another user within the same org without `super_admin` intervention.
- **FR-8.** Every write action in the admin console MUST produce an entry in the `admin_audit_log` table with actor, action type, target entity type/id, and timestamp.
- **FR-9.** The system SHOULD display a summary of recent audit-log events (last 20) on the console home page.
- **FR-10.** The system MAY provide a quick-action command palette (`Cmd/Ctrl+K`) for common admin operations such as "find user," "view course," and "reset password."

## 6. Non-Functional Requirements

- **Performance** — Admin dashboard page MUST load within 1.5 s p95 on a cold request over an org with 10 000 users and 500 courses; paginated table endpoints MUST respond in ≤300 ms p95.
- **Security** — All admin routes protected by JWT + role check in `server/src/routes/rbac.rs`; CSRF tokens required for every mutating form; all queries parameterized against SQL injection.
- **Privacy & Compliance** — Admin actions on student records MUST be logged (FERPA §99.32 audit-log requirement); PII displayed in tables MUST be scoped to the admin's org boundary.
- **Accessibility** — All admin pages MUST achieve WCAG 2.1 AA; data tables MUST include `<caption>`, `scope` headers, and keyboard-navigable pagination.
- **Scalability** — Table queries MUST use keyset/cursor pagination to remain O(1) regardless of total row count.
- **Reliability** — Admin console availability target: 99.5% monthly; read-only degraded mode when DB replica is available but primary is unavailable.
- **Observability** — Emit `admin.page_view`, `admin.user_updated`, `admin.course_archived` events to the platform event bus; expose `admin_requests_total` Prometheus counter.
- **Maintainability** — Admin routes grouped under `server/src/routes/admin/`; React pages under `clients/web/src/pages/admin/`.
- **Internationalization** — All UI strings in `i18n` keys; date/number formatting via `Intl` API.
- **Backward compatibility** — New admin endpoints are additive; existing non-admin routes unchanged.

## 7. Acceptance Criteria

- **AC-1.** *Given* a user with role `student` is authenticated, *When* they request `GET /api/admin/users`, *Then* the server returns HTTP 403 and the React app does not render the admin nav.
- **AC-2.** *Given* an `org_admin` is logged in, *When* they navigate to `/admin`, *Then* they see accurate counts for total users, active courses, and pending enrollments scoped to their org within 1.5 s.
- **AC-3.** *Given* a user-management table with 5 000 users, *When* an admin searches by email fragment, *Then* matching results appear in ≤300 ms and only users within the admin's org are returned.
- **AC-4.** *Given* an admin updates the org's logo and brand color, *When* any user logs in to that org, *Then* the updated branding is reflected in the header without a full redeploy.
- **AC-5.** *Given* an admin deactivates a user account, *When* that user attempts to authenticate, *Then* the login returns HTTP 401 and an `admin_audit_log` row is inserted with the admin's actor ID.
- **AC-6.** *Given* an axe-core accessibility scan runs against the admin dashboard, *When* the scan completes, *Then* zero critical or serious WCAG 2.1 AA violations are reported.
- **AC-7.** *Given* a `super_admin` views an org they do not own, *When* they inspect any table, *Then* data is scoped to that org and cross-org data is not exposed.

## 8. Data Model

New tables:

```sql
-- 0XX_admin_audit_log.sql
CREATE TABLE admin_audit_log (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id      UUID NOT NULL REFERENCES organizations(id),
  actor_id    UUID NOT NULL REFERENCES users(id),
  action      TEXT NOT NULL,          -- e.g. 'user.deactivate', 'course.archive'
  target_type TEXT NOT NULL,          -- 'user' | 'course' | 'enrollment' | 'setting'
  target_id   UUID,
  payload     JSONB,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX admin_audit_log_org_created ON admin_audit_log(org_id, created_at DESC);
CREATE INDEX admin_audit_log_actor ON admin_audit_log(actor_id);
```

Existing tables: add `org_id` FK and `is_active` boolean to `users` if not already present (depends on §5.1 migration).

## 9. API Surface

```
GET    /api/admin/overview          -> AdminOverviewDto
GET    /api/admin/users             -> Paginated<AdminUserDto>  ?q=&role=&page=&per_page=
PATCH  /api/admin/users/:id         -> AdminUserDto
GET    /api/admin/courses           -> Paginated<AdminCourseDto> ?q=&status=&term_id=
PATCH  /api/admin/courses/:id/status -> AdminCourseDto
GET    /api/admin/audit-log         -> Paginated<AuditLogEntry> ?from=&to=&action=
GET    /api/admin/settings          -> OrgSettingsDto
PUT    /api/admin/settings          -> OrgSettingsDto
POST   /api/admin/delegate          -> { user_id, role }
```

All endpoints require `Authorization: Bearer <jwt>` with `org_admin` or `super_admin` claim. Rate limited to 60 req/min per admin.

## 10. UI / UX

New pages under `clients/web/src/pages/admin/`:
- `AdminLayout.tsx` — sidebar nav with sections: Overview, Users, Courses, Terms, Integrations, Settings, Audit Log.
- `AdminOverview.tsx` — KPI cards + recent audit events table.
- `AdminUsers.tsx` — filterable, sortable, paginated data table with inline role-change and deactivate controls.
- `AdminCourses.tsx` — same pattern for courses; link to instructor's course page.
- `AdminSettings.tsx` — branding form with live preview; reuses `clients/web/src/pages/lms/Settings.tsx` pattern.
- `AdminAuditLog.tsx` — read-only log table with date-range filter.

User flow:
1. Admin navigates to `/admin` → sees overview KPIs.
2. Clicks "Users" → sees table → searches by email → selects user → edits role or deactivates.
3. Action confirmation dialog → confirms → audit row created → toast notification.

Empty/loading states: skeleton loaders on all tables; empty-state illustration with CTA when no users/courses exist. Mobile: sidebar collapses to bottom tab bar on screens <768 px; tables scroll horizontally with sticky first column.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Org model (§5.1) slips, blocking admin shell | H | H | Stub `organizations` table with minimal fields to unblock 18.1 UI work in parallel |
| Admin bypasses data isolation between orgs | M | H | Integration tests asserting org boundary on every admin endpoint |
| Admin console becomes performance bottleneck on large orgs | M | M | Keyset pagination + read-replica routing from day one |
| WCAG failures discovered late | M | M | Add axe-core CI check during feature branch development |

## 15. Rollout Plan

- Feature flag: `admin_console_enabled` (OpenFeature, default `false`).
- Phase 1: Deploy schema migrations; enable for internal `super_admin` users only.
- Phase 2: Enable for pilot district admin accounts; gather feedback.
- Phase 3: GA flip — enable for all orgs with `org_admin` role populated.
- Rollback: Disable feature flag; admin routes return 404; no data destructive actions to undo.

## 16. Test Plan

- **Unit** — role-check middleware (all roles × all admin endpoints); org-scoping logic for user/course queries.
- **Integration** — Full CRUD cycles for user, course, settings through admin API; audit-log row creation verified after each write.
- **End-to-end** — Playwright: admin login → overview load → user search → deactivate user → verify audit log entry.
- **Security** — Authz matrix: student/instructor/org_admin/super_admin × every admin endpoint; SQL-injection payloads in search params.
- **Accessibility** — axe-core in CI on every admin page; manual screen-reader test with VoiceOver on Safari.
- **Performance** — k6 load test: 50 concurrent admins running user-search queries; p95 ≤ 300 ms pass criterion.
- **Manual exploratory** — cross-org isolation checklist; mobile layout review on iPhone 15 and Pixel 8.
