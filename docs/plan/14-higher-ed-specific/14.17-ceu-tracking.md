## 1. Problem Statement

Continuing education (CE) divisions at universities offer non-credit professional development courses, certificate programs, and training workshops. These programs typically award Continuing Education Units (CEUs) — a standardized measure where 1 CEU equals 10 contact hours of instruction. Licensing boards (nursing, engineering, teaching, accounting) require seat-time verification for CEU credits. Lextures currently logs `user_audit` navigation events but provides no seat-time aggregation, no CEU computation, and no certificate issuance tied to contact hours. Without this, the platform cannot serve CE divisions or professional-development markets.

## 2. Goals

- Track per-user seat time (time actively engaged with course content) accurately using heartbeat-based session tracking.
- Compute CEU credits from verified seat time against a configurable contact-hours-to-CEU ratio.
- Issue CEU completion certificates when the minimum hours threshold is met.
- Provide a CE transcript per learner listing all earned CEUs with dates and issuing institution.
- Support xAPI/cmi5 seat-time reporting for integrations with external CE management systems.

## 3. Non-Goals

- Credit-bearing academic courses (seat time for those is handled by attendance in 13.2 for K-12; not required for HE credit courses).
- Certification board submission (Lextures generates the record; the learner submits to their board).
- Live session / synchronous class tracking (asynchronous module-based tracking only at GA).
- Billing for CE registrations (see 15.3 / 16.8).

## 4. Personas & User Stories

- **As a nurse completing CE requirements**, I want the LMS to track exactly how many hours I have spent on module content so that I can prove compliance to my licensing board.
- **As a CE program coordinator**, I want to configure each course with a required contact-hours threshold so that only learners who meet it receive a CEU certificate.
- **As a learner**, I want a CE transcript I can download and present to my employer or licensing board showing all CEUs earned in the past year.
- **As an admin**, I want seat-time data to be tamper-evident so that I can defend CEU awards in an audit.

## 5. Functional Requirements

- **FR-1.** The system MUST implement server-side heartbeat-based seat-time tracking: the client sends a heartbeat every 60 seconds while a module content page is in the foreground; the server records accumulated active time per user per content item.
- **FR-2.** The system MUST aggregate seat time per course per user and compute CEU credit using a configurable contact-hours-to-CEU ratio (default: 10 contact hours = 1 CEU).
- **FR-3.** The system MUST automatically issue a CEU completion certificate (distinct from the Open Badges certificate in 15.5) when a learner meets the course's configured minimum contact-hours threshold.
- **FR-4.** The system MUST store an immutable seat-time audit log per session with start time, end time, content item ID, and accumulated minutes.
- **FR-5.** The system SHOULD generate a CE transcript per learner listing all CEU-bearing courses completed, credit awarded, and completion date, downloadable as PDF.
- **FR-6.** The system SHOULD emit xAPI statements (`experienced`, `completed`) with `duration` extensions for each content item session to support external CE management system integrations.
- **FR-7.** The system MAY expose a SCORM-compatible `cmi.core.session_time` reporting endpoint for legacy CE systems.

## 6. Non-Functional Requirements

- **Performance** — Heartbeat endpoint responds in <100 ms; seat-time aggregation query <500 ms.
- **Security** — Heartbeat spoofing mitigated: server validates that heartbeat interval is plausible (≥1/min, ≤4/min); anomalous session data flagged for review.
- **Privacy & Compliance** — Seat-time records are FERPA education records; accessible to the learner and CE administrators only; retained per institution's records-retention schedule.
- **Accessibility** — CE transcript PDF has accessible text; progress indicators WCAG 2.1 AA.
- **Scalability** — Heartbeat endpoint handles 10,000 concurrent learners; writes batched via in-memory buffer flushed every 30 s to reduce DB write amplification.
- **Reliability** — Heartbeat buffer flushed on graceful shutdown; lost heartbeats on crash result in under-counting, not over-counting (conservative by design).
- **Observability** — Metric `seat_time_minutes_total{course_id}` (aggregated, no PII).
- **Maintainability** — Seat-time service in `server/src/services/seat_time.rs`; reuses existing `user_audit` infrastructure for base event recording.
- **Internationalization** — CEU display with locale decimal separator; "hours" label externalised.
- **Backward compatibility** — Seat-time tracking is additive; existing courses without CE configuration are unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a learner spends 10 minutes on a module content page, *when* they navigate away, *then* the `seat_time_sessions` table records a session of approximately 10 minutes for that content item.
- **AC-2.** *Given* a course is configured for 1 CEU at 10 contact hours, *when* a learner accumulates 10 hours of verified seat time, *then* a CEU certificate is automatically generated and the learner is notified by email.
- **AC-3.** *Given* a learner attempts to manipulate the heartbeat by sending 10 heartbeats per second, *when* the server processes them, *then* only 1 heartbeat per minute is counted and an anomaly flag is set on the session.
- **AC-4.** *Given* a learner requests their CE transcript, *when* the PDF downloads, *then* it lists all CEU-bearing courses with credit hours, completion date, and institution name.
- **AC-5.** *Given* a CE coordinator views a course's seat-time report, *when* 20 learners have logged time, *then* the report shows per-learner total minutes and CEU-earned status.

## 8. Data Model

Migration `090_ceu_tracking.sql`:
```sql
CREATE TABLE seat_time_sessions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES users(id),
  content_item_id UUID NOT NULL REFERENCES course_structure_items(id),
  session_start   TIMESTAMPTZ NOT NULL,
  session_end     TIMESTAMPTZ,
  minutes_active  INT NOT NULL DEFAULT 0,
  anomaly_flag    BOOLEAN NOT NULL DEFAULT FALSE
);
CREATE INDEX idx_seat_time_user_course ON seat_time_sessions(user_id, content_item_id);

CREATE TABLE ceu_configurations (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  course_id             UUID NOT NULL UNIQUE REFERENCES courses(id),
  required_hours        NUMERIC(6,2) NOT NULL,
  ceu_credit            NUMERIC(4,2) NOT NULL,
  certificate_template  TEXT,
  enabled               BOOLEAN NOT NULL DEFAULT TRUE
);
```

## 9. API Surface

```
POST /api/v1/seat-time/heartbeat
     Body: { content_item_id: uuid, session_token: string }
     Auth: student (rate-limited to 1/min per item)

GET  /api/v1/me/seat-time?course_id=
     Returns: { total_minutes, ceu_earned, progress_pct }

GET  /api/v1/me/ce-transcript              -- PDF or JSON
GET  /api/v1/courses/:id/seat-time-report  -- per-learner report (admin/coordinator)
POST /api/v1/admin/courses/:id/ceu-config  -- configure CEU threshold
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Heartbeat gaming (automated scripts) | M | H | Session anomaly detection; cap session time at 8 hours/day per course |
| Heartbeat loss on poor network = under-counted CEU | M | M | Client-side buffering with catch-up on reconnect; conservative CEU is better than disputed CEU |
| Licensing boards do not accept LMS-generated transcripts | M | H | Provide PDF with institutional letterhead; allow institution to add accreditation authority info |

## 15. Rollout Plan

- Feature flag: `ceu_tracking` (default: off; CE tier only).
- Pilot with a continuing nursing education program.
- Validate with one licensing board that PDF transcript is acceptable.
- GA after anomaly detection tuned to minimize false positives.

## 16. Test Plan

- **Unit** — Heartbeat rate limiter; seat-time aggregation; CEU computation; anomaly detection.
- **Integration** — Heartbeat API; aggregation job; certificate trigger.
- **End-to-end** — Playwright: learner progresses through course, seat time accumulates, certificate generated.
- **Security** — Assert heartbeat rate limiting prevents gaming; assert learner A cannot view learner B's seat time.
- **Performance / load** — 10,000 concurrent heartbeats; p95 <100 ms.
- **Manual exploratory** — QA: heartbeat with content item switch, session spanning midnight, anomaly flag trigger.
