## 1. Problem Statement

Co-curricular transcripts (CCRs) — also called comprehensive learner records (CLRs) — document student achievements beyond academic grades: badges, certificates, service hours, workshop completions, and leadership roles. Many universities (especially those following the Lumina Foundation / AACRAO Comprehensive Learner Record standard and Suskie HE assessment frameworks) need to generate these for graduate programs, employers, and accreditors. Lextures has no mechanism to generate a verifiable CLR, making it unable to serve institutions with holistic student achievement requirements. Cross-reference: 15.5 (Open Badges 3.0) provides the badge infrastructure this feature consumes.

## 2. Goals

- Aggregate a student's verified achievements (course completions, badges, certificates, portfolio artifacts, extracurricular records) into a CCR document.
- Generate a CCR compliant with IMS Global Comprehensive Learner Record (CLR) Standard v2.0.
- Issue the CCR as a verifiable credential (W3C VC Data Model 1.1) signed with the institution's DID.
- Allow students to share or download the CCR for employment and graduate applications.
- Cross-link with Open Badges 3.0 (15.5) so badges appear in the CCR.

## 3. Non-Goals

- Physical transcript printing or certified-mail delivery.
- Academic grade transcript (that is the registrar's domain; see 14.5).
- K-12 co-curricular records (different scope).
- Replacing the SIS's official transcript.

## 4. Personas & User Stories

- **As a student**, I want to generate my comprehensive learner record so that I can share evidence of all my university achievements — not just grades — with employers.
- **As an employer**, I want to verify a student's CCR is authentic and was issued by the institution so that I can trust the credentials it contains.
- **As an admin**, I want to configure which achievement types appear on the CCR and which institutional DID key is used for signing.
- **As a faculty member**, I want awarded badges and capstone portfolio milestones to automatically appear on a student's CCR without manual data entry.

## 5. Functional Requirements

- **FR-1.** The system MUST aggregate the following achievement types into a CCR: course completions, earned Open Badges (15.5), certificates (15.5 if applicable), portfolio milestones (14.12), and manually added extracurricular records.
- **FR-2.** The system MUST generate a CLR Standard v2.0-compliant JSON document for a student on demand.
- **FR-3.** The system MUST sign the CLR as a W3C Verifiable Credential using the institution's DID (did:web or did:key) and an RS256 or Ed25519 key.
- **FR-4.** The system MUST allow students to download the signed CLR JSON and a PDF rendering.
- **FR-5.** The system MUST provide a shareable verification URL that any third party can use to verify the credential's signature and issuer.
- **FR-6.** The system SHOULD allow admins to add manual extracurricular records (student leadership role, service-learning hours, workshop attendance) to a student's CCR.
- **FR-7.** The system MAY support the Suskie HE assessment framework tags on achievement evidence.

## 6. Non-Functional Requirements

- **Performance** — CCR generation (JSON + PDF) in <5 s for a typical student with 50 achievements.
- **Security** — Signing keys stored in institutional secrets vault; private key never leaves the server; verification URL does not expose the private key.
- **Privacy & Compliance** — Student must explicitly consent to CCR generation and sharing; CCR is a FERPA education record; public verification URL reveals only the signed credential, not other student data.
- **Accessibility** — Student-facing CCR PDF includes accessible text; HTML rendering WCAG 2.1 AA.
- **Scalability** — CCR generation is on-demand; no batch processing needed.
- **Reliability** — PDF generation via headless renderer (Puppeteer or similar); retry on render failure.
- **Observability** — Metric `ccr_generated_total`, `ccr_verifications_total`.
- **Maintainability** — CLR JSON builder in `server/src/services/ccr/`; VC signing in a separate `vc_signing` module.
- **Internationalization** — CCR achievement labels externalised; dates in ISO 8601 in JSON.
- **Backward compatibility** — No impact on existing achievement or grade data.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student has 5 completed courses, 3 Open Badges, and 1 portfolio milestone, *when* they generate their CCR, *then* the CLR v2.0 JSON contains assertions for all 9 achievements.
- **AC-2.** *Given* a CCR is generated, *when* a third party accesses the verification URL, *then* the W3C VC signature is verifiable against the institution's public DID document and the result is "Valid."
- **AC-3.** *Given* a student downloads the CCR PDF, *when* they open it, *then* it includes a QR code linking to the verification URL and lists all achievements with dates.
- **AC-4.** *Given* an admin adds a manual extracurricular record, *when* the student generates their next CCR, *then* the new record appears in the CLR JSON.
- **AC-5.** *Given* a student has not consented to public sharing, *when* a third party attempts to access the student's verification URL, *then* the system returns a 404 unless the student has explicitly generated a shareable link.

## 8. Data Model

Migration `086_ccr.sql`:
```sql
CREATE TABLE ccr_achievements (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES users(id),
  achievement_type TEXT NOT NULL, -- 'course_completion', 'badge', 'certificate', 'portfolio', 'extracurricular'
  source_id       UUID,           -- FK to relevant table depending on type
  title           TEXT NOT NULL,
  description     TEXT,
  issued_at       TIMESTAMPTZ NOT NULL,
  evidence_url    TEXT,
  outcome_tags    TEXT[],
  added_by        UUID REFERENCES users(id)
);

CREATE TABLE ccr_documents (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES users(id),
  generated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  clr_json        JSONB NOT NULL,
  vc_proof        JSONB NOT NULL,
  pdf_key         TEXT,
  share_token     TEXT UNIQUE
);
CREATE INDEX idx_ccr_achievements_user ON ccr_achievements(user_id, issued_at DESC);
```

## 9. API Surface

```
POST /api/v1/me/ccr/generate          -- generate CCR on demand
GET  /api/v1/me/ccr                   -- list generated CCRs
GET  /api/v1/me/ccr/:id/download      -- download JSON or PDF (?format=json|pdf)
GET  /api/v1/verify/:share_token      -- public verification endpoint (no auth)
POST /api/v1/admin/students/:uid/ccr/achievements  -- add manual extracurricular record
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Institution DID management complexity | M | M | Provide managed did:web default; allow BYOK for sophisticated institutions |
| PDF rendering failures | M | M | Async generation; retry; email notification when ready |
| CLR spec evolves during implementation | L | M | Pin to v2.0; design JSON builder as versioned serializer |

## 15. Rollout Plan

- Feature flag: `co_curricular_transcript` (default: off).
- Pilot with one institution after 15.5 (badges) and 14.12 (portfolios) ship.
- GA after legal review of VC signing disclosure.

## 16. Test Plan

- **Unit** — CLR JSON builder; VC signature construction; PDF render trigger.
- **Integration** — Generate CCR for a user with all achievement types; verify signature; public verification endpoint.
- **End-to-end** — Playwright: student generates CCR, downloads PDF, verifies share link.
- **Security** — Assert signing private key not exposed in any API response; assert share token is unguessable (UUID v4).
- **Accessibility** — axe on CCR dashboard and public verification page.
- **Manual exploratory** — QA: student with no achievements, manual extracurricular record, revoked badge, verification with tampered JSON.
