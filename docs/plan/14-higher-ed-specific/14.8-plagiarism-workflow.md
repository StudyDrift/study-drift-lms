## 1. Problem Statement

Academic integrity enforcement is a non-negotiable requirement for every university LMS evaluation. Instructors expect submitted text documents to be automatically checked for plagiarism and AI-generated content. As an AI-first LMS, Lextures is uniquely exposed to the concern that students could use the platform's own AI tools to generate submissions. The absence of any plagiarism or AI-authorship signal means RFP responses fail the academic integrity section, and instructors using Lextures have no visibility into potential misconduct. This feature (cross-referenced as 3.5) is also critical for K-12 adoption where AI-use policies are increasingly district-mandated.

## 2. Goals

- Integrate with Turnitin Similarity (primary) and Copyleaks or Originality.ai (secondary) as external plagiarism-detection providers.
- Build an internal AI-authorship signal using the LLM already available in the platform to provide a lightweight AI-content probability score on all text submissions.
- Surface originality reports inline within the submission/grading view.
- Log reports as immutable artifacts stored with the submission for FERPA audit.
- Provide institution-level configuration: which provider to use, which assignment types trigger checks, and what score thresholds generate instructor alerts.

## 3. Non-Goals

- Acting as an academic integrity authority — reports are advisory, not automated sanctions.
- Code plagiarism detection (MOSS integration) — deferred to a later release.
- Checking non-text submissions (images, spreadsheets, video) — text only at GA.
- Replacing Turnitin's institutional license agreements — Lextures proxies API calls only.

## 4. Personas & User Stories

- **As an instructor**, I want every text submission automatically scanned so that I receive a similarity score without manually copying text into a third-party tool.
- **As a student**, I want to know before I submit that my assignment will be checked for plagiarism so that I understand the expectations.
- **As a student**, I want to self-check my draft against Turnitin before final submission (where institution permits) so that I can correct unintentional citation errors.
- **As a department chair**, I want a summary of originality scores across all submissions in a course so that I can identify trends or concerns.
- **As an admin**, I want to configure which provider the institution uses and set alert thresholds so that all courses benefit from consistent policy.

## 5. Functional Requirements

- **FR-1.** The system MUST integrate with at least one external plagiarism provider (Turnitin Similarity API v2 at GA) and submit text submissions for similarity checking automatically upon final submission.
- **FR-2.** The system MUST compute an internal AI-authorship probability score using the platform's existing LLM service for all text submissions, independent of the external provider.
- **FR-3.** The system MUST store originality reports (score + provider metadata) as immutable `originality_reports` records linked to the submission.
- **FR-4.** The system MUST display the similarity score and AI-authorship score on the grading view, accessible to the instructor and authorized admins.
- **FR-5.** The system MUST display a notice to students on the assignment page that submissions are subject to plagiarism and AI-authorship checks.
- **FR-6.** The system SHOULD allow instructors to configure per-assignment whether checks run (default: on for text submissions) and which provider to use.
- **FR-7.** The system SHOULD surface Turnitin's detailed similarity report in an embedded iframe or linked viewer once the report is generated.
- **FR-8.** The system MAY allow students to submit a draft for self-check (requires institution configuration and Turnitin student-draft API support).
- **FR-9.** The system MUST NOT block submission completion while the originality scan is in progress; scans run asynchronously.

## 6. Non-Functional Requirements

- **Performance** — External provider scan initiated within 5 s of submission; result polled every 30 s; report available within 10 min for typical document.
- **Security** — Submission content sent to external provider only after institution has accepted the provider's DPA; student consent disclosed in assignment instructions; API keys stored encrypted.
- **Privacy & Compliance** — FERPA: student work submitted to third-party only with appropriate disclosure; GDPR: EU institutions must use EU-hosted Turnitin endpoints; internal AI score must not be retained with student PII beyond grading period.
- **Accessibility** — Similarity score displayed as text percentage with color coding supplemented by icon label; WCAG 2.1 AA.
- **Scalability** — Background scan queue handles peak submission load (e.g., 500 submissions at assignment deadline); rate-limited per provider plan.
- **Reliability** — Scan failure (provider unavailable) does not block grading; report shows "Scan failed — retry" with a manual retry button.
- **Observability** — Metric `originality_scans_total{provider, status}`; alert on failure rate >10%.
- **Maintainability** — Provider abstraction trait in `server/src/services/plagiarism/`; Turnitin and Copyleaks each implement the trait.
- **Internationalization** — UI labels externalised; similarity score is a number (no localization needed).
- **Backward compatibility** — Existing submissions without reports show no score; no retroactive scanning.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student submits a text assignment, *when* the submission is saved, *then* a background job submits the text to Turnitin and an internal AI-authorship scan, and both results appear in the grading view within 10 minutes.
- **AC-2.** *Given* the Turnitin API is unavailable, *when* a student submits, *then* the submission succeeds immediately and the grading view shows "Originality scan pending — retry available."
- **AC-3.** *Given* an instructor views a graded submission, *when* the Turnitin similarity score is >40%, *then* a visual indicator highlights the score in yellow/orange and a link to the full Turnitin report is shown.
- **AC-4.** *Given* a student navigates to a plagiarism-checked assignment, *when* the page loads, *then* a clearly visible notice states that submissions are checked for plagiarism and AI-generated content.
- **AC-5.** *Given* an admin queries the `originality_reports` table, *when* 30 days have elapsed, *then* all submitted reports are still intact (immutable), linked to the correct submission ID.
- **AC-6.** *Given* a developer queries the `originality_reports` table, *when* looking for the AI-authorship score, *then* no row contains the student's name or email — only the submission ID (FERPA-safe).

## 8. Data Model

Migration `081_originality_reports.sql`:
```sql
CREATE TABLE originality_reports (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  submission_id     UUID NOT NULL REFERENCES module_assignment_submissions(id),
  provider          TEXT NOT NULL,  -- 'turnitin', 'copyleaks', 'internal_ai'
  similarity_score  NUMERIC(5,2),   -- 0.00–100.00
  ai_score          NUMERIC(5,2),   -- internal AI-authorship probability 0–100
  report_url        TEXT,           -- signed URL to Turnitin viewer
  raw_response      JSONB,
  status            TEXT NOT NULL DEFAULT 'pending', -- 'pending', 'complete', 'failed'
  scanned_at        TIMESTAMPTZ,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_originality_reports_submission ON originality_reports(submission_id);
```

Backfill: no retroactive scans; existing submissions have no report rows.

## 9. API Surface

```
GET  /api/v1/courses/:id/assignments/:aid/submissions/:sid/originality
```

## 11. AI / ML Considerations

- Internal AI-authorship scoring uses the existing LLM service (`server/src/services/ai/`) with a classification prompt.
- Prompt: instructs the model to return a 0–100 probability that the text was AI-generated; evaluated against a golden set of human/AI text pairs.
- Fallback: if LLM call fails, score is marked `null` and displayed as "AI check unavailable."
- Cost budget: ~$0.001 per submission for short-doc classification; token limit 4,000 per check.
- PII: student name/ID not included in prompt; only the submission text.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Turnitin API pricing changes make per-submission cost prohibitive | M | H | Abstract provider; support Copyleaks as cost-effective fallback |
| False positives flagging good-faith student work | H | M | Make scores advisory only; never automatic sanctions; instructor education |
| Student data sent to Turnitin without adequate consent disclosure | M | H | Require DPA acceptance by institution before enabling; surface student notice |
| Internal AI score used improperly in academic proceedings | M | H | Label clearly as "AI-authorship probability — not definitive evidence" |

## 15. Rollout Plan

- Feature flag: `plagiarism_checks` (default: off).
- Pilot with 2–3 HE institutions who have existing Turnitin licenses.
- GA with clear documentation of internal AI score limitations.
- Rollback: disable flag; no submissions are retroactively rescanned.

## 16. Test Plan

- **Unit** — Provider trait mock; AI scoring prompt construction; similarity score parsing.
- **Integration** — Turnitin stub webhook; verify report stored correctly; retry logic on failure.
- **End-to-end** — Playwright: student submits assignment, instructor sees score in grading view, retry works after simulated failure.
- **Security** — Assert API key not logged; assert student cannot view other students' originality reports; verify Turnitin webhook HMAC validation.
- **Accessibility** — axe on grading view with originality panel; verify color is not sole indicator.
- **Performance / load** — 500 submissions queued simultaneously; assert scan jobs complete within 15 min.
- **Manual exploratory** — QA: highly similar submission, AI-generated submission, provider failure, manual retry.
