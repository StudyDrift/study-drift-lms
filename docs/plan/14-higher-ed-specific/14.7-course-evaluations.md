## 1. Problem Statement

End-of-term course evaluations are a non-negotiable accreditation and faculty-review requirement at every accredited university. Students expect a private channel to evaluate their instructors and courses. Currently Lextures has no survey or evaluation system (2.5 is also missing), forcing institutions to use third-party tools like Qualtrics or paper forms. Without built-in evaluations, Lextures cannot be the single LMS for HE institutions, and faculty-development offices have no programmatic data from the LMS.

## 2. Goals

- Deliver a configurable, anonymous end-of-term evaluation system usable without 2.5 shipping first (evaluations are the priority surface for HE).
- Guarantee anonymity: no data path connects a student's identity to their responses, even for admins.
- Support institution-defined question banks with rating scales, multiple-choice, and open-text items.
- Automatically open evaluations based on academic calendar windows (14.6) and close them at term end.
- Surface aggregate evaluation reports to department chairs while keeping individual responses anonymous.

## 3. Non-Goals

- Mid-term "pulse" surveys (nice-to-have; can reuse this infrastructure later).
- Instructor-designed ad-hoc course surveys (covered by 2.5 in a separate plan).
- Integrating with third-party evaluation vendors (EvaluationKIT, etc.) — build-first approach.
- Displaying evaluation comments to the evaluated instructor (institution configurable).

## 4. Personas & User Stories

- **As a student**, I want to submit my course evaluation anonymously so that I can give honest feedback without fear of grade retaliation.
- **As a department chair**, I want aggregate evaluation scores per instructor and course so that I can inform tenure and promotion decisions.
- **As a registrar**, I want evaluations to open automatically two weeks before finals and close at the last day of finals based on the academic calendar.
- **As an instructor**, I want to be notified when my evaluation results are available and see aggregated scores (but not individual responses).
- **As an admin**, I want to build a standard institution-wide question template so that I collect consistent data across all courses.

## 5. Functional Requirements

- **FR-1.** The system MUST allow admins to create evaluation templates with rating-scale (1–5 Likert), multiple-choice, and open-text question types.
- **FR-2.** The system MUST automatically schedule evaluations to open and close based on academic calendar event windows (14.6) configured by the admin.
- **FR-3.** The system MUST guarantee response anonymity: no student identifier may be stored with a response row; k-anonymity threshold (configurable, default ≥5 responses) required before aggregate data is shown.
- **FR-4.** The system MUST prevent instructors from seeing raw individual responses; they see only aggregate statistics after the evaluation closes.
- **FR-5.** The system MUST display a non-dismissible evaluation reminder banner to students during the evaluation window.
- **FR-6.** The system MUST track completion rates (anonymous count of submitted vs. enrolled) per course and surface these to admins and department chairs.
- **FR-7.** The system SHOULD allow department chairs to view cross-course aggregate reports comparing evaluation scores across sections.
- **FR-8.** The system MAY send reminder email notifications to students who have not yet submitted an evaluation at 7 days and 1 day before the window closes.

## 6. Non-Functional Requirements

- **Performance** — Evaluation form submission <300 ms; aggregate report generation <2 s.
- **Security** — Response rows contain no student_id; session/JWT token not logged in response context; admin bypass of anonymity threshold is technically impossible.
- **Privacy & Compliance** — FERPA: evaluation responses are education records but anonymization removes personal identifiability; institutional policy controls instructor access; data retained per institution's records-retention schedule.
- **Accessibility** — Rating scales use `role="radiogroup"` with numeric labels not relying on color; WCAG 2.1 AA.
- **Scalability** — 10,000 concurrent evaluation submissions during finals week; evaluated against background-write queue.
- **Reliability** — Evaluation window open/close is calendar-event-driven (not a cron string); survives server restarts.
- **Observability** — Metric `evaluation_submissions_total{course_id}` (no PII); alert if completion rate <10% with 48h remaining.
- **Maintainability** — Evaluation module in `server/src/services/course_evaluations.rs`; question schema is JSONB for flexibility.
- **Internationalization** — Question text and scale labels stored per locale; student form rendered in user's locale.
- **Backward compatibility** — No impact on existing course grade data.

## 7. Acceptance Criteria

- **AC-1.** *Given* an evaluation is submitted by a student, *when* a developer queries the `evaluation_responses` table, *then* no column contains the student's user ID or any directly identifying field.
- **AC-2.** *Given* a course has fewer than 5 responses, *when* an instructor or department chair views the aggregate report, *then* the system shows "Not enough responses to display results" rather than any data.
- **AC-3.** *Given* the academic calendar window for evaluations opens, *when* a student logs in to Lextures, *then* they see the evaluation reminder banner on every course page for courses with open evaluations.
- **AC-4.** *Given* a department chair views the cross-section report, *when* sorting by average rating, *then* all sections with ≥5 responses are listed sorted correctly.
- **AC-5.** *Given* an evaluation closes at term end, *when* the instructor is notified, *then* they can view aggregate scores and open-text themes but cannot drill down to individual responses.

## 8. Data Model

Migration `080_course_evaluations.sql`:
```sql
CREATE TABLE evaluation_templates (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id      UUID NOT NULL REFERENCES organizations(id),
  name        TEXT NOT NULL,
  questions   JSONB NOT NULL,  -- [{type, text, options, required}]
  created_by  UUID REFERENCES users(id),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE evaluation_windows (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  course_id        UUID NOT NULL REFERENCES courses(id),
  template_id      UUID NOT NULL REFERENCES evaluation_templates(id),
  opens_at         TIMESTAMPTZ NOT NULL,
  closes_at        TIMESTAMPTZ NOT NULL,
  enrolled_count   INT NOT NULL DEFAULT 0,
  response_count   INT NOT NULL DEFAULT 0
);

CREATE TABLE evaluation_responses (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  window_id    UUID NOT NULL REFERENCES evaluation_windows(id),
  -- DELIBERATELY no user_id column
  answers      JSONB NOT NULL,
  submitted_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
-- No index on student identity — by design.
CREATE INDEX idx_eval_responses_window ON evaluation_responses(window_id);
```

## 9. API Surface

```
POST  /api/v1/evaluations/:window_id/submit    -- anonymous submission (authenticated but no ID stored)
GET   /api/v1/courses/:id/evaluations/status   -- is window open + has current user submitted (boolean only)
GET   /api/v1/courses/:id/evaluations/results  -- aggregate results (instructor, post-close, k-anon threshold)
GET   /api/v1/admin/evaluations/report?term_id= -- cross-course aggregate (admin/chair)

-- Admin template management
POST/GET/PATCH/DELETE /api/v1/admin/evaluation-templates
POST /api/v1/admin/courses/:id/evaluation-windows
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Instructor attempts to identify students from open-text | M | H | Technical anonymity (no ID stored); policy warning in UI |
| Low response rates make data useless | H | M | Reminder emails; reminder banner; department reporting of completion rates |
| Template changes mid-window invalidate in-flight responses | L | M | Lock templates once a window opens |

## 15. Rollout Plan

- Feature flag: `course_evaluations` (default: off; on for HE tier).
- Admin creates template; pilot at term end with one department.
- GA after anonymity model is reviewed by legal/privacy team.
- Rollback: disable flag; no grade data affected.

## 16. Test Plan

- **Unit** — Anonymization: assert no student_id stored; k-anon threshold logic.
- **Integration** — Submit as authenticated user; verify no ID in DB; aggregate endpoint with <5 and ≥5 responses.
- **End-to-end** — Playwright: student submits evaluation, instructor views aggregate post-close.
- **Security** — Attempt to correlate submission timestamp to student session; verify no linkage. Assert instructor cannot access raw responses endpoint.
- **Accessibility** — axe on evaluation form; keyboard navigation for Likert scale.
- **Performance / load** — 10,000 concurrent submissions; p95 <300 ms.
- **Manual exploratory** — QA: window opens/closes correctly, reminder banner dismissal not persisted, threshold boundary.
