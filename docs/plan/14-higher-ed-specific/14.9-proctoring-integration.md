## 1. Problem Statement

Remote and hybrid universities require proctored online exams. Without proctoring integrations, instructors cannot verify academic integrity for high-stakes assessments, which blocks adoption by universities with significant online programs. Every major competitor (Canvas, Blackboard, Moodle, Brightspace) supports at least one proctoring vendor via LTI 1.3 deep linking. The absence of this capability causes Lextures to fail procurement evaluations and leaves instructors manually coordinating with proctoring vendors outside the LMS.

## 2. Goals

- Implement LTI 1.3 Deep Linking 2.0 to connect Honorlock, Respondus, ProctorU, and Examity as proctoring tool providers.
- Allow instructors to configure proctoring on individual quizzes/exams from within the LMS.
- Launch the proctoring session seamlessly from the quiz start flow without the student leaving Lextures.
- Receive proctoring result metadata (flagged incidents, session completion) back via AGS or LTI outcomes after the session.
- Cross-reference: depends on LTI 1.3 consumer (2.12) as foundational infrastructure.

## 3. Non-Goals

- Building a first-party proctoring or browser-lockdown solution — vendor integration only.
- Proctoring for K-12 (most K-12 testing is in-person or uses different tools).
- Automated academic integrity sanctions based on proctoring flags — instructor review required.

## 4. Personas & User Stories

- **As an instructor**, I want to attach Honorlock proctoring to a quiz so that the proctoring session launches automatically when a student begins the exam.
- **As a student**, I want a clear pre-exam checklist showing what the proctoring software requires (camera, microphone, ID) before I start so that I am not surprised.
- **As an instructor**, I want to see proctoring incident flags next to the student's quiz submission so that I can review flagged sessions during grading.
- **As an admin**, I want to configure which proctoring vendors are available institution-wide and enter API credentials centrally.

## 5. Functional Requirements

- **FR-1.** The system MUST implement LTI 1.3 Deep Linking 2.0 to allow instructors to attach a proctoring tool to a quiz via the standard LTI content-item selection flow.
- **FR-2.** The system MUST launch the proctoring session via LTI 1.3 Resource Link Launch when the student begins an attached proctored quiz, passing required context (course, section, student) as LTI claims.
- **FR-3.** The system MUST display a pre-exam requirements checklist to the student before launching the proctoring session (camera test, ID verification reminder, browser extension check).
- **FR-4.** The system MUST receive proctoring completion and incident data via AGS score post-back or LTI Advantage Names & Roles Provisioning (NRPS) callbacks and store them linked to the quiz attempt.
- **FR-5.** The system MUST surface proctoring session status (scheduled, in-progress, completed, flagged) on the instructor grading view.
- **FR-6.** The system SHOULD support at least Honorlock and Respondus Monitor at GA; ProctorU and Examity as follow-on providers using the same LTI 1.3 framework.
- **FR-7.** The system MAY prevent quiz submission in the Lextures UI if the proctoring session has not reported completion (configurable per assignment).

## 6. Non-Functional Requirements

- **Performance** — LTI launch redirect <500 ms; session-status webhook processed within 5 s of receipt.
- **Security** — LTI 1.3 message signing (RS256 JWT); tool registration credentials stored encrypted; JWKS endpoint served over HTTPS; LTI state parameter validated to prevent replay attacks.
- **Privacy & Compliance** — FERPA: student biometric/video data is processed by the proctoring vendor, not Lextures; institution must have DPA with vendor; Lextures stores only session metadata, not video.
- **Accessibility** — Pre-exam checklist WCAG 2.1 AA; screen-reader announcement when proctoring session launches.
- **Scalability** — LTI launch generates a unique, short-lived (10 min) state token per student per attempt.
- **Reliability** — Failed proctoring launch does not prevent quiz access if instructor has set "non-blocking" mode; blocking mode shows error and prevents launch.
- **Observability** — Metric `proctoring_launches_total{vendor}`, `proctoring_flags_received_total`; alert on launch failure rate >5%.
- **Maintainability** — Proctoring config stored in `lti_tool_registrations` table (from 2.12); proctoring-specific metadata in `quiz_proctoring_config`.
- **Internationalization** — Pre-exam checklist text externalised; vendor names not translated.
- **Backward compatibility** — Quizzes without proctoring configuration are unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* an instructor attaches Honorlock proctoring to a quiz via deep link, *when* a student begins the quiz, *then* the Honorlock session launches in the same browser tab via LTI 1.3 Resource Link Launch and the quiz is accessible after session initialization.
- **AC-2.** *Given* a student launches a proctored exam, *when* the pre-exam checklist page loads, *then* it shows camera-ready, microphone-ready, and browser-extension-installed checks before proceeding.
- **AC-3.** *Given* Honorlock posts a session-complete callback with incident flags, *when* the instructor views the student's quiz submission, *then* the proctoring status shows "Completed — 2 flags" with a link to the Honorlock review panel.
- **AC-4.** *Given* an admin registers an Honorlock tool with client_id and deployment_id, *when* the LTI Advantage JWKS endpoint is queried, *then* the correct public key is returned and Honorlock can verify platform signatures.
- **AC-5.** *Given* the proctoring vendor is unreachable, *when* a student attempts to launch a proctored exam in non-blocking mode, *then* the quiz opens normally with a banner warning the instructor that proctoring was unavailable.

## 8. Data Model

Migration `082_quiz_proctoring.sql`:
```sql
CREATE TABLE quiz_proctoring_config (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  quiz_id           UUID NOT NULL UNIQUE REFERENCES module_quizzes(id),
  lti_tool_reg_id   UUID NOT NULL,  -- FK to lti_tool_registrations (from 2.12)
  required          BOOLEAN NOT NULL DEFAULT FALSE,
  settings          JSONB NOT NULL DEFAULT '{}',
  created_by        UUID REFERENCES users(id),
  created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE quiz_proctoring_sessions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  attempt_id      UUID NOT NULL REFERENCES quiz_attempts(id),
  vendor          TEXT NOT NULL,
  session_id      TEXT,          -- vendor's session ID
  status          TEXT NOT NULL DEFAULT 'pending', -- 'pending', 'active', 'complete', 'flagged'
  flag_count      INT NOT NULL DEFAULT 0,
  review_url      TEXT,
  started_at      TIMESTAMPTZ,
  completed_at    TIMESTAMPTZ,
  raw_callback    JSONB
);
CREATE INDEX idx_proctoring_sessions_attempt ON quiz_proctoring_sessions(attempt_id);
```

```
     Auth: instructor
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| LTI 1.3 spec variances between proctoring vendors | H | M | Implement to spec; test with each vendor's sandbox before GA |
| Student browser blocks proctoring extension | H | M | Pre-exam checklist detects extension; clear install instructions |
| Proctoring callback arrives after submission — race condition | M | M | Store callback asynchronously; instructor sees "processing" until complete |
| Privacy concerns over biometric data | M | H | Clear disclosure; DPA requirement; Lextures never stores video/audio |

## 15. Rollout Plan

- Feature flag: `proctoring_integration` (default: off; requires 2.12 enabled first).
- Pilot with one HE institution using Honorlock.
- Add Respondus Monitor in follow-up sprint.
- GA alongside 2.12 LTI consumer release.
- Rollback: disable flag; proctoring configs remain in DB but are ignored.

## 16. Test Plan

- **Unit** — LTI 1.3 JWT signing and verification; deep-link response parsing; proctoring config CRUD.
- **Integration** — LTI launch flow with Honorlock sandbox; webhook callback processing; status update.
- **End-to-end** — Playwright: instructor attaches proctoring via deep link, student goes through pre-exam checklist, quiz launches with proctoring session active.
- **Security** — Assert LTI state parameter is single-use; verify HMAC on proctoring callbacks; assert student A cannot view student B's proctoring session.
- **Accessibility** — axe on pre-exam checklist; keyboard navigation through steps.
- **Performance / load** — 200 concurrent LTI launches; p95 <500 ms.
- **Manual exploratory** — QA: proctoring vendor unavailable, extension not installed, required vs. optional proctoring modes.
