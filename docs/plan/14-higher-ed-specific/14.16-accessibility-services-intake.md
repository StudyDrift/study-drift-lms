## 1. Problem Statement

University accessibility services offices (also called disability resource centers) coordinate accommodations between students with documented disabilities and their instructors under the Americans with Disabilities Act (ADA) and Section 504 of the Rehabilitation Act. This process — collecting accommodation letters, notifying instructors, and applying accommodation overrides to assessments — is currently entirely manual and external to the LMS. Without an accessibility services intake workflow, instructors must manually apply extended time and other overrides, creating inconsistency and ADA liability. This feature creates a secure, FERPA-compliant channel between the accessibility office, instructors, and the LMS accommodation engine (2.11).

## 2. Goals

- Allow accessibility services coordinators to create accommodation profiles for students and specify accommodation types (extended time, separate testing room, alternate format, screen reader, speech-to-text).
- Automatically apply configured accommodations to LMS assessments (quiz time limits, alternate submission formats) via the existing assessment accommodation system (2.11).
- Notify instructors of approved accommodations without disclosing the student's underlying disability.
- Provide students visibility into which accommodations are active on which courses.
- Maintain a FERPA-compliant access model: disability documentation visible only to accessibility office staff.

## 3. Non-Goals

- Storing diagnostic or medical disability documentation (stored in SIS/disability-office system, not LMS).
- Processing accommodation requests that require physical resources (separate testing room scheduling).
- Handling K-12 IEP/504 plans (different feature, 12.10).
- Legal adjudication of accommodation disputes.

## 4. Personas & User Stories

- **As an accessibility services coordinator**, I want to create a student's accommodation profile once and have it automatically applied to all their current and future course assessments.
- **As an instructor**, I want to be notified when a student in my course has an active accommodation so that I can plan accordingly without needing to know their disability.
- **As a student**, I want to see which accommodations are active for each of my courses so that I can confirm the LMS is enforcing them correctly.
- **As a student**, I want my disability documentation to remain private — instructors should only see the accommodation, not the reason.

## 5. Functional Requirements

- **FR-1.** The system MUST provide an `accommodation_profiles` model linking a student to a set of accommodation types with configurable parameters (e.g., 1.5x time, 2.0x time, separate testing, alternate format).
- **FR-2.** The system MUST allow accessibility services coordinators (a new `accessibility_coordinator` role) to create, update, and deactivate accommodation profiles.
- **FR-3.** The system MUST automatically propagate active accommodations to the assessment delivery layer (quiz time limits via 2.11) without instructor intervention.
- **FR-4.** The system MUST notify instructors by email when a student in their course has an active accommodation, disclosing only the accommodation type and effective date — not the underlying disability.
- **FR-5.** The system MUST allow students to view their active accommodation profiles and which courses are affected.
- **FR-6.** The system MUST ensure that accommodation profile details are accessible only to the student themselves, accessibility coordinators, and authorized admins — not instructors or other students.
- **FR-7.** The system SHOULD generate an accessibility notification letter template that the coordinator can review before sending to instructors.

## 6. Non-Functional Requirements

- **Performance** — Accommodation propagation to all course assessments on profile creation <2 s.
- **Security** — `accommodation_profiles` rows ACL-controlled; `SECURITY DEFINER` function for coordinator access; instructors query only their own course's accommodation flags (not the profile detail).
- **Privacy & Compliance** — ADA/Section 504: disability information is strictly need-to-know; instructors see accommodation type only; FERPA: accommodation record is an education record with tighter access than typical course records.
- **Accessibility** — The accessibility intake UI itself must be exemplary WCAG 2.1 AA (ironic but required).
- **Scalability** — Propagation of accommodation to 20 courses × 10 assessments per create event: handled synchronously (200 rows max).
- **Reliability** — Accommodation changes propagate atomically; rollback on partial failure.
- **Observability** — Metric `accommodations_applied_total{type}`.
- **Maintainability** — Accommodation service in `server/src/services/accommodations/`; reuses 2.11 accommodation override records.
- **Internationalization** — Accommodation type names and notification letter externalised.
- **Backward compatibility** — Courses without accommodation profiles are unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a coordinator creates a 1.5x extended time accommodation for a student, *when* the student starts any quiz in any of their enrolled courses, *then* the time limit is automatically 1.5x the standard limit without the instructor manually configuring it.
- **AC-2.** *Given* a new accommodation profile is activated, *when* the instructor opens their course, *then* they receive an email notification stating "[Student Name] has an active accommodation: 1.5x Extended Time" without any reference to disability or diagnosis.
- **AC-3.** *Given* a student opens their accessibility profile page, *when* they have two active accommodations, *then* they can see both accommodation types and the list of courses currently affected.
- **AC-4.** *Given* an instructor attempts to access the accommodation profile endpoint for a student, *when* they make the API call, *then* the system returns 403 Forbidden — they can only see a Boolean "accommodation active" flag on the course roster.
- **AC-5.** *Given* a coordinator deactivates an accommodation profile, *when* the next quiz starts, *then* the standard time limit applies and no override is applied.

## 8. Data Model

Migration `089_accessibility_services.sql`:
```sql
CREATE TYPE accommodation_type AS ENUM (
  'extended_time_1_5x', 'extended_time_2x', 'separate_testing',
  'alternate_format', 'screen_reader', 'speech_to_text', 'reduced_distraction', 'other'
);

CREATE TABLE accommodation_profiles (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  student_id      UUID NOT NULL REFERENCES users(id),
  org_id          UUID NOT NULL REFERENCES organizations(id),
  accommodations  accommodation_type[] NOT NULL,
  custom_params   JSONB NOT NULL DEFAULT '{}',  -- {time_multiplier: 1.5}
  effective_from  DATE NOT NULL,
  effective_until DATE,
  created_by      UUID NOT NULL REFERENCES users(id),
  is_active       BOOLEAN NOT NULL DEFAULT TRUE,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
-- Row-level security: visible to student (own), accessibility_coordinator role, admin
CREATE INDEX idx_accommodation_profiles_student ON accommodation_profiles(student_id) WHERE is_active = TRUE;
```

## 9. API Surface

```
POST/GET/PATCH /api/v1/accessibility/profiles         -- coordinator CRUD
GET            /api/v1/me/accommodations               -- student reads own profile
GET            /api/v1/courses/:id/roster/accommodations -- instructor: Boolean flags only
POST           /api/v1/accessibility/profiles/:id/notify-instructors -- send notification email
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Instructor inadvertently learns disability from context | M | H | Notification template reviewed by legal; no disability fields in instructor-facing APIs |
| Accommodation not applied to new assessment after profile creation | M | H | Event-driven propagation: when a new quiz is created, check for active accommodations |
| Coordinator applies wrong time multiplier | M | M | Confirmation step showing "This will apply X time multiplier to all quizzes for [student]" |

## 15. Rollout Plan

## 16. Test Plan

- **Unit** — Accommodation propagation logic; notification letter template rendering; deactivation cleanup.
- **Integration** — Profile create → quiz time-limit override applied; deactivation removes override.
- **End-to-end** — Playwright: coordinator creates profile, instructor sees flag (not detail), student starts quiz with extended time.
- **Security** — Assert instructor API returns only Boolean flag; assert student A cannot view student B's profile.
- **Accessibility** — axe on coordinator admin form; exemplary keyboard navigation.
- **Manual exploratory** — QA: accommodation created after quiz published, deactivation mid-term, custom time multiplier.
