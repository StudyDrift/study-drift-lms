## 1. Problem Statement

Academic advisors and students at universities need tools to plan which courses to take each term in order to satisfy degree requirements. This planning currently lives entirely in the SIS (DegreeWorks, Stellic, EAB Navigate) with no connection to the LMS. Without a degree-planner hook, Lextures is unable to surface advising information in context — students switching between the SIS degree audit and the LMS creates friction, and advisors lack insight into how a student's current course performance is tracking against degree requirements. Light hooks that read degree-audit data and surface it in the LMS would be a meaningful differentiator.

## 2. Goals

- Provide a read-only integration with SIS-hosted degree audit systems (DegreeWorks REST, Stellic API) to surface degree completion status in the LMS.
- Create an advising appointment scheduling link in the student's LMS dashboard.
- Allow advisors to add notes visible to the student from within the LMS.
- Surface "courses recommended for your degree" from the degree planner in the course catalog browse (14.2).

## 3. Non-Goals

- Building a full degree audit engine — read-only integration only.
- Replacing DegreeWorks, Stellic, or EAB Navigate.
- Enforcing prerequisite holds from the LMS.
- Financial aid counseling integration.

## 4. Personas & User Stories

- **As a student**, I want to see how many courses I have left to complete my degree on my LMS dashboard so that I stay on track without switching to a separate system.
- **As an academic advisor**, I want to leave notes in the LMS that the student can see so that follow-up action items from an advising session are visible in context.
- **As a student**, I want the course catalog in Lextures to highlight courses that satisfy my remaining degree requirements so that I register for the right courses.
- **As an advisor**, I want a link to schedule an appointment visible on the student's dashboard so that students can easily reach me.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a configurable advising appointment URL (e.g., EAB Navigate booking link, Calendly, institution scheduler) surfaced on the student Dashboard and profile.
- **FR-2.** The system SHOULD integrate with DegreeWorks REST API (where licensed) to pull a summary of completed, in-progress, and remaining requirements and display them on a student Dashboard widget.
- **FR-3.** The system SHOULD allow advisors to write advising notes visible only to the student and the advisor (and authorized admins), not instructors.
- **FR-4.** The system SHOULD tag catalog sections (14.2) with a "Fulfills Requirement: [requirement group]" badge for the viewing student based on degree audit data.
- **FR-5.** The system MAY surface at-risk flags from EAB Navigate or similar tools as banner alerts on the student Dashboard (advisor-configurable).
- **FR-6.** The system MUST allow the institution to configure which degree-audit integration is active (DegreeWorks, Stellic, or none) from the admin settings.

## 6. Non-Functional Requirements

- **Performance** — Degree audit summary widget loads in <1 s using cached data; cache TTL 4 hours.
- **Security** — Degree audit data is FERPA-protected; accessible only to the student and their assigned advisor; DegreeWorks API credentials stored encrypted.
- **Privacy & Compliance** — Degree progress data is a FERPA education record; institutional DPA with DegreeWorks vendor required.
- **Accessibility** — Degree progress widget WCAG 2.1 AA; progress bars have percentage text labels.
- **Scalability** — Degree audit polling rate-limited to 1 call per student per 4 hours; cached in Redis.
- **Reliability** — If degree audit API unavailable, widget shows stale data with "last updated" timestamp.
- **Observability** — Metric `degree_audit_api_calls_total{provider, status}`.
- **Maintainability** — Advising integration in `server/src/services/advising/`; degree audit adapter pattern parallel to SIS adapter.
- **Internationalization** — Requirement names displayed as returned by degree audit system; currency/credit display in locale.
- **Backward compatibility** — Students without degree audit configured see a simplified "Your Courses" summary instead.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student's DegreeWorks data is configured, *when* they open their Dashboard, *then* the "Degree Progress" widget shows a completion percentage and a count of remaining required courses.
- **AC-2.** *Given* an advisor writes an advising note for a student, *when* the student logs in, *then* a notification badge appears and the note is readable in their advising notes panel.
- **AC-3.** *Given* an admin configures an advising appointment URL, *when* a student opens their Dashboard, *then* a "Schedule Advising Appointment" button links to the configured URL.
- **AC-4.** *Given* a student browses the course catalog, *when* a section fulfills a remaining degree requirement, *then* a "Fulfills: [Requirement Group]" badge appears on the section card.
- **AC-5.** *Given* the DegreeWorks API is unavailable, *when* the Dashboard loads, *then* the degree widget shows the last cached data with a "Data last updated [date]" notice rather than an error crash.

## 8. Data Model

Migration `087_advising.sql`:
```sql
CREATE TABLE advising_notes (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  student_id  UUID NOT NULL REFERENCES users(id),
  advisor_id  UUID NOT NULL REFERENCES users(id),
  content     TEXT NOT NULL,
  visible_to_student BOOLEAN NOT NULL DEFAULT TRUE,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_advising_notes_student ON advising_notes(student_id, created_at DESC);

CREATE TABLE degree_audit_cache (
  user_id     UUID PRIMARY KEY REFERENCES users(id),
  data        JSONB NOT NULL,
  fetched_at  TIMESTAMPTZ NOT NULL,
  source      TEXT NOT NULL -- 'degreeworks', 'stellic'
);
```

## 9. API Surface

```
GET  /api/v1/me/degree-progress         -- student reads own degree audit (cached)
GET  /api/v1/me/advising-notes          -- student reads their advising notes
POST /api/v1/advisor/students/:uid/notes -- advisor creates note
GET  /api/v1/advisor/students/:uid/notes -- advisor reads notes for student
GET  /api/v1/admin/advising/config       -- get advising config
POST /api/v1/admin/advising/config       -- set appointment URL, degree-audit provider
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| DegreeWorks API is not universally licensed | H | M | Feature gracefully absent if provider not configured; appointment link ships independently |
| Degree audit data staleness misleads students | M | M | Prominent "last updated" timestamp; link to SIS for authoritative data |
| Advisor notes misused as grade-related communication | L | M | Notes are not linked to grades; separate system messaging pathway |

## 15. Rollout Plan

- Feature flag: `advising_integration` (default: off).
- Advising appointment link ships first (no external API needed).
- Degree audit widget ships after DegreeWorks REST is validated in pilot.
- GA for advising notes after pilot with 2 advisors.

## 16. Test Plan

- **Unit** — Degree audit cache TTL logic; advising note CRUD; appointment URL config validation.
- **Integration** — DegreeWorks API mock; cache hit and miss scenarios; note visibility rules.
- **End-to-end** — Playwright: student sees degree widget, advisor adds note, student sees notification.
- **Security** — Assert student cannot read another student's advising notes; assert non-advisor cannot write notes.
- **Accessibility** — axe on Dashboard widgets; progress bar labels.
- **Manual exploratory** — QA: degree audit unavailable, student with no advisor assigned, appointment URL config.
