## 1. Problem Statement

No integration exists between Lextures and any university Student Information System (SIS). Institutions running Banner, Workday Student, Colleague, Jenzabar, or PeopleSoft cannot automatically provision courses, sync rosters, or push final grades back to the registrar. This is the single largest adoption blocker for higher-ed accounts: every IT evaluation checklist asks for it, and every sales cycle stalls without it. Shipping a bidirectional SIS connector unlocks roster automation, eliminates manual CSV uploads, and closes the grade-passback loop required by accreditation bodies.

## 2. Goals

- Provide a configurable SIS adapter layer supporting at minimum Ellucian Banner (REST/SOAP), Workday Student REST, and Ellucian Colleague (REST) at GA.
- Automate nightly roster sync: create/update/deactivate enrollments from SIS data.
- Implement IMS LIS 2.0 / LTI Advantage Assignment & Grade Service (AGS) grade passback.
- Allow registrar-triggered final grade export (CSV and AGS) on demand and on schedule.
- Provide per-institution configuration UI and connection health dashboard for admins.

## 3. Non-Goals

- Modifying or writing arbitrary data back to the SIS beyond grades and enrollment-status acknowledgements.
- Real-time (<1 min) sync; nightly or hourly batch is acceptable for GA.
- Custom ERP/finance integrations beyond the student data domain.
- Building SIS connectors for K-12 systems (covered in 13.7).

## 4. Personas & User Stories

- **As a registrar**, I want final grades posted from the LMS to Banner automatically on grade-submission day so that I do not reconcile two systems manually.
- **As a university IT admin**, I want to configure a Banner REST endpoint and credentials once and have rosters sync nightly without manual CSV uploads.
- **As an instructor**, I want my course roster to reflect SIS adds/drops within 24 hours so that I never grade a student who has withdrawn.
- **As a student**, I want my LMS enrollment to appear automatically when I register through the SIS so that I can access course materials on day one.
- **As a system admin**, I want a sync-health dashboard showing last-run status, error counts, and failed records so that I can diagnose integration failures quickly.

## 5. Functional Requirements

- **FR-1.** The system MUST support at least one SIS adapter (Banner REST v9+) at initial release, with Workday Student and Colleague following within 60 days.
- **FR-2.** The system MUST sync course enrollments (create, update role, deactivate) from SIS data on a configurable schedule (default: nightly at 02:00 institution local time).
- **FR-3.** The system MUST implement IMS LIS 2.0 grade passback and LTI Advantage AGS so that final grades can be pushed to any AGS-compliant SIS.
- **FR-4.** The system MUST support per-institution SIS configuration: endpoint URL, authentication credentials (OAuth 2.0 client-credentials or basic auth stored encrypted), sync schedule, and field-mapping overrides.
- **FR-5.** The system MUST log every sync run with record counts, errors, and duration in a queryable audit table.
- **FR-6.** The system MUST allow manual "sync now" triggering from the admin UI for emergency re-syncs.
- **FR-7.** The system SHOULD surface a per-enrollment sync-status field (synced_at, sis_id, sis_section_id) on the course_enrollments table.
- **FR-8.** The system SHOULD support Jenzabar J1/J1 Desktop and PeopleSoft Campus Solutions as configurable adapters (MAJOR priority, post-GA).
- **FR-9.** The system MAY support real-time SIS webhook callbacks for immediate enrollment changes where the SIS supports outbound notifications.

## 6. Non-Functional Requirements

- **Performance** — Sync job for 10,000 enrollments must complete in under 5 minutes; grade passback for a section of 300 must complete in under 30 seconds.
- **Security** — SIS credentials stored encrypted at rest (AES-256); all SIS API calls over TLS 1.2+; OAuth tokens rotated per-session; admin-only access to configuration UI.
- **Privacy & Compliance** — Sync transfers FERPA-covered data; institution must sign a data processing agreement; PII minimised to fields required for LMS function.
- **Accessibility** — Admin configuration and health dashboard WCAG 2.1 AA.
- **Scalability** — Adapter layer supports concurrent syncs for up to 500 institutions without contention; jobs run in background worker pool (see 17.3).
- **Reliability** — Sync failures retry with exponential back-off (3 attempts); partial failures quarantine bad records without halting the batch; 99.5% nightly-run success target.
- **Observability** — Prometheus counters: `sis_sync_records_processed_total`, `sis_sync_errors_total`, `sis_grade_passback_total`; Sentry alerts on consecutive failures.
- **Maintainability** — Adapter pattern in `server/src/services/sis/`; each SIS is a Rust trait implementation; shared sync orchestrator calls trait methods.
- **Internationalization** — Sync scheduler respects institution time zone from org settings; all admin UI strings externalised.
- **Backward compatibility** — Adding a new SIS adapter must not require schema migrations on existing adapters.

## 7. Acceptance Criteria

- **AC-1.** *Given* a Banner REST endpoint is configured for an institution, *when* the nightly sync runs, *then* all active Banner enrollments appear as Lextures course enrollments and any Banner drops set the Lextures enrollment to `withdrawn`.
- **AC-2.** *Given* an instructor submits final grades in the Lextures gradebook, *when* they click "Submit to Registrar," *then* grades are posted to Banner via AGS within 30 seconds and the sync log records a success entry.
- **AC-3.** *Given* the SIS API is temporarily unavailable, *when* the sync job runs, *then* it retries 3 times with exponential back-off, records the failure in the audit log, and sends an admin alert email.
- **AC-4.** *Given* an admin opens the SIS health dashboard, *when* the last sync had errors, *then* the UI shows per-record error details including the SIS person ID and the error reason.
- **AC-5.** *Given* a new SIS adapter trait implementation is added, *when* it is wired into the config enum, *then* existing Banner adapter tests still pass without modification.
- **AC-6.** *Given* SIS credentials are stored, *when* a developer queries the database directly, *then* credentials are not stored in plaintext.

## 8. Data Model

New tables in `server/migrations/073_sis_integration.sql`:

```sql
-- Per-institution SIS configuration
CREATE TABLE sis_configurations (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id        UUID NOT NULL REFERENCES organizations(id),
  adapter_type  TEXT NOT NULL,           -- 'banner', 'workday', 'colleague', 'jenzabar', 'peoplesoft'
  endpoint_url  TEXT NOT NULL,
  auth_config   JSONB NOT NULL,          -- encrypted credential blob
  field_map     JSONB NOT NULL DEFAULT '{}',
  sync_cron     TEXT NOT NULL DEFAULT '0 2 * * *',
  enabled       BOOLEAN NOT NULL DEFAULT TRUE,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Sync run history
CREATE TABLE sis_sync_runs (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  sis_config_id   UUID NOT NULL REFERENCES sis_configurations(id),
  started_at      TIMESTAMPTZ NOT NULL,
  finished_at     TIMESTAMPTZ,
  status          TEXT NOT NULL,  -- 'running', 'success', 'partial', 'failed'
  records_synced  INT NOT NULL DEFAULT 0,
  records_failed  INT NOT NULL DEFAULT 0,
  error_detail    JSONB
);
```

Add columns to existing `course_enrollments` (migration `074_enrollment_sis_fields.sql`):
```sql
ALTER TABLE course_enrollments ADD COLUMN sis_id TEXT;
ALTER TABLE course_enrollments ADD COLUMN sis_section_id TEXT;
ALTER TABLE course_enrollments ADD COLUMN sis_synced_at TIMESTAMPTZ;
CREATE UNIQUE INDEX idx_enrollment_sis_id ON course_enrollments(org_id, sis_id) WHERE sis_id IS NOT NULL;
```

Backfill: existing enrollments leave `sis_id` NULL until first sync.

## 9. API Surface

```
POST   /api/v1/admin/sis/configurations          -- create config (admin only)
GET    /api/v1/admin/sis/configurations           -- list configs
PATCH  /api/v1/admin/sis/configurations/:id       -- update config
POST   /api/v1/admin/sis/configurations/:id/sync  -- trigger manual sync
GET    /api/v1/admin/sis/sync-runs                -- list sync history
GET    /api/v1/admin/sis/sync-runs/:id            -- run detail + error log

-- Grade passback (internal, called from grade submission flow)
POST   /api/v1/internal/sis/grade-passback        -- {course_id, submission_type}
```

All admin endpoints require `org:admin` scope. Grade passback is called server-side only.

Rate limit: manual sync endpoint — 1 request per 5 minutes per config.

## 10. UI / UX

New pages under `clients/web/src/pages/admin/sis/`:
- `SisConfigList.tsx` — table of configured adapters with status badges.
- `SisConfigForm.tsx` — create/edit form with adapter-type selector, credential fields, cron picker.
- `SisSyncHistory.tsx` — paginated run log with error drill-down.

Flow:
1. Admin navigates to Admin → Integrations → Student Information System.
2. Clicks "Add SIS Connection," selects adapter type, fills credentials, sets schedule.
3. Clicks "Test Connection" — system makes a dry-run API call and shows success/failure.
4. Saves; system schedules nightly job.
5. Admin can click "Sync Now" at any time.

States: loading skeleton, empty-state card with setup instructions, error banner with actionable message.

Mobile: form fields stack single-column; sync history uses card layout.

Accessibility: all form fields labelled, error messages associated via `aria-describedby`, focus returns to trigger after modal close.

## 12. Integration Points

- **Ellucian Banner REST API v9** — `/api/v1/persons`, `/api/v1/sections`, `/api/v1/section-registrations`.
- **Workday Student REST API** — Academic Unit, Student Course Registration objects.
- **IMS Global LIS 2.0 / LTI Advantage AGS** — `lineItems`, `scores` endpoints for grade passback.
- Internal: `server/src/repos/enrollment.rs`, `server/src/repos/course.rs`, `server/src/routes/courses.rs`.
- Background job queue (17.3) for scheduled sync tasks.
- Email service (6.2) for sync-failure admin alerts.
- Webhook emission (16.3) for `enrollment.synced` and `grade.posted` events.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Banner API changes between Banner versions | M | H | Version-pin adapter; integration test suite with recorded responses |
| Institution firewall blocks outbound SIS calls | M | H | Document IP ranges; support on-premise agent proxy model in v2 |
| FERPA violation from over-fetched SIS fields | L | H | Strict field-map allowlist; legal review before GA |
| Sync racing with manual instructor enrollments | M | M | Conflict-resolution policy: SIS is source of truth for adds; LMS preserves manual instructor additions |
| Credential leakage in logs | M | H | Scrub credentials from all tracing spans; use secret-redaction middleware |

## 15. Rollout Plan

- Feature flag: `sis_integration_enabled` (default: off).
- Phase 1: Internal dogfood with a test Banner sandbox — confirm sync correctness.
- Phase 2: Invite 2–3 pilot HE customers with Banner; collect feedback on field-mapping UI.
- Phase 3: GA behind flag; enable per institution on contract.
- Rollback: disable flag; enrollments revert to manual management; no data loss.

## 16. Test Plan

- **Unit** — Each SIS adapter trait implementation tested with recorded HTTP fixtures; conflict-resolution logic unit-tested.
- **Integration** — Docker-based Banner/Workday stub server; full sync run against populated stub; assert enrollment state changes.
- **End-to-end** — Playwright: admin configures Banner connection, triggers manual sync, verifies enrollment count changes, submits grades, verifies passback confirmation.
- **Security** — Assert credentials not retrievable in plaintext from API; assert non-admin cannot access `/admin/sis/` routes; test SSRF prevention on endpoint URL field.
- **Accessibility** — axe scan on config form and sync history pages.
- **Performance / load** — Simulate 10,000-enrollment sync; assert <5 min completion.
- **Manual exploratory** — QA checklist: test each SIS adapter type, test invalid credential error handling, test partial-failure quarantine behavior.
