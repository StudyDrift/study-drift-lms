## 1. Problem Statement

Lextures has no connection to the institutional course catalog. Every HE institution manages official course offerings — including CRNs, section numbers, credit hours, meeting patterns, room assignments, and pre-requisite chains — inside their SIS. Without a catalog integration, LMS course shells must be created manually, section metadata is inconsistent, and pre-requisite enforcement is impossible. Read-only catalog sync removes this friction and enables richer course pages, schedule display, and registration-status awareness for students.

## 2. Goals

- Pull course catalog data (sections, CRNs, credit hours, meeting patterns, instructors of record) from the SIS nightly.
- Automatically create or update LMS course shells matched to SIS catalog sections.
- Display official catalog metadata (credits, prereqs, meeting times, room) on the Lextures course page.
- Surface a student's registration status (registered, waitlisted, dropped) from SIS data on their dashboard.
- Provide a catalog-browse UI for students who want to preview available courses before self-enrolling (where the institution permits).

## 3. Non-Goals

- Writing registration transactions back to the SIS (add/drop through Lextures) — read-only only at GA.
- Course catalog for non-HE institutions (K-12 course scheduling is out of scope here).
- Displaying room/building maps.
- Pre-requisite enforcement by the LMS (read and display only).

## 4. Personas & User Stories

- **As a student**, I want to see my official class schedule with meeting times in my Lextures dashboard so that I do not need to switch between the SIS and the LMS.
- **As an instructor**, I want my LMS course shell to be created automatically when the registrar publishes sections so that I can start building content before the term begins.
- **As a registrar administrator**, I want course catalog changes (instructor reassignments, section cancellations) to propagate to the LMS within 24 hours.
- **As a student**, I want to browse available catalog sections in Lextures so that I can preview syllabi before registering in the SIS.

## 5. Functional Requirements

- **FR-1.** The system MUST ingest SIS catalog data (sections, instructors, CRNs, credit hours, meeting patterns) via the configured SIS adapter (14.1) on a nightly schedule.
- **FR-2.** The system MUST create Lextures course shells automatically for each new SIS section that does not yet have one, using a configurable naming template (e.g. `{subject} {number} — {section} — {term}`).
- **FR-3.** The system MUST update existing course shells when the SIS changes instructor of record, credits, or section status (active → cancelled).
- **FR-4.** The system MUST store and display official catalog metadata: CRN, credits, meeting pattern (days/time), room, department, and pre-requisite list.
- **FR-5.** The system MUST surface each student's registration status (registered, waitlisted, auditing, withdrawn) on their Dashboard from SIS data.
- **FR-6.** The system SHOULD provide a catalog-browse page listing available sections for the current term, filterable by department, days, credits, and instructor.
- **FR-7.** The system SHOULD surface a pre-requisite completion indicator for each student on the course page (met / not met / waived) based on transcript data from SIS where available.
- **FR-8.** The system MAY support OneRoster 1.2 `/courses` and `/classes` endpoints as an alternative catalog data source for institutions that expose OneRoster.

## 6. Non-Functional Requirements

- **Performance** — Catalog browse page loads in <2 s for 5,000 sections; search autocomplete responds in <300 ms.
- **Security** — Catalog data marked FERPA-sensitive; unauthenticated users see only public catalog metadata (title, credits, department); enrollment and registration status gated to the enrolled student.
- **Privacy & Compliance** — Instructor names are public directory information; student registration records are not.
- **Accessibility** — Catalog browse table includes sortable column headers with `aria-sort`; filter panel keyboard-navigable; WCAG 2.1 AA.
- **Scalability** — Catalog table supports up to 50,000 sections per institution; full-text search via Postgres `tsvector`.
- **Reliability** — Catalog sync failures do not affect existing course shells; stale data is surfaced with a "last synced" timestamp.
- **Observability** — Metric `catalog_sections_synced_total`; alert if sync lag exceeds 26 hours.
- **Maintainability** — Catalog ingestion is an extension of the SIS adapter trait from 14.1; no separate adapter codebase.
- **Internationalization** — Meeting-pattern display respects locale day names and 12/24h clock preference.
- **Backward compatibility** — Manually created course shells without SIS linkage continue to work unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a Banner catalog section is published for Spring 2027, *when* the nightly sync runs, *then* a Lextures course shell exists with the correct title, CRN, and credit hours.
- **AC-2.** *Given* a student is registered in the SIS for a section, *when* they log in to Lextures, *then* the Dashboard shows that section with their official registration status.
- **AC-3.** *Given* the registrar cancels a section in Banner, *when* the next sync completes, *then* the Lextures course shell is marked "cancelled" and students see a cancellation notice.
- **AC-4.** *Given* a student visits the catalog browse page, *when* they filter by department "CS" and days "MWF," *then* only matching sections are shown and results update without a full page reload.
- **AC-5.** *Given* a catalog section has a pre-requisite, *when* a student views the course page, *then* the prerequisite chain is displayed with met/not-met indicators based on their SIS transcript.

## 8. Data Model

New migration `075_course_catalog.sql`:
```sql
CREATE TABLE catalog_sections (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          UUID NOT NULL REFERENCES organizations(id),
  term_id         UUID NOT NULL REFERENCES terms(id),
  sis_course_id   TEXT NOT NULL,
  sis_section_id  TEXT NOT NULL,
  crn             TEXT,
  subject         TEXT NOT NULL,
  course_number   TEXT NOT NULL,
  section_number  TEXT,
  title           TEXT NOT NULL,
  credits         NUMERIC(4,2),
  meeting_pattern JSONB,
  room            TEXT,
  department      TEXT,
  prerequisites   JSONB,
  status          TEXT NOT NULL DEFAULT 'active',  -- active, cancelled, pending
  lms_course_id   UUID REFERENCES courses(id),
  synced_at       TIMESTAMPTZ,
  search_vector   tsvector GENERATED ALWAYS AS (
    to_tsvector('english', title || ' ' || subject || ' ' || course_number)
  ) STORED
);
CREATE INDEX idx_catalog_sections_search ON catalog_sections USING GIN(search_vector);
CREATE INDEX idx_catalog_sections_org_term ON catalog_sections(org_id, term_id);
```

Backfill: no existing catalog sections; all rows created by sync.

## 9. API Surface

```
GET  /api/v1/catalog/sections          -- browse (public metadata; auth required for reg status)
GET  /api/v1/catalog/sections/:id      -- section detail with prereqs
GET  /api/v1/courses/:id/catalog-info  -- LMS course → catalog metadata

-- Admin
POST /api/v1/admin/catalog/sync        -- trigger manual catalog sync
GET  /api/v1/admin/catalog/sync-status -- last sync timestamp and error count
```

Query params for browse: `term_id`, `department`, `days`, `min_credits`, `max_credits`, `q` (full-text).
Response includes pagination (`cursor`-based).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Catalog data quality varies by institution | H | M | Surface last-sync timestamp; allow manual override of LMS course metadata |
| Auto-created shells have wrong settings defaults | M | M | Configurable template defaults per institution; instructor reviews before publishing |
| Credit-hour display causes confusion with LMS grade weights | L | M | Clear labeling; credits are informational only, not used for LMS grade calculation |
| SIS API rate limits on large catalog pulls | M | M | Paginate requests; cache full catalog in `catalog_sections` table to avoid repeated pulls |

## 15. Rollout Plan

- Feature flag: `catalog_integration_enabled` (default: off).
- Pilot with one Banner institution; verify auto-shell creation accuracy.
- GA when pilot confirms section-matching logic handles edge cases (cross-listed, multi-credit variable courses).
- Rollback: disable flag; catalog_sections table retained for history; manually-created shells unaffected.

## 16. Test Plan

- **Unit** — Catalog ingestion parser for Banner and OneRoster section payloads; shell auto-creation logic; prerequisite serialization.
- **Integration** — Sync against Banner stub: verify section → shell mapping, cancellation propagation.
- **End-to-end** — Playwright: student logs in, sees schedule widget, browses catalog, views section detail.
- **Security** — Assert unauthenticated requests receive only public fields; assert student A cannot view student B's registration status.
- **Accessibility** — axe scan on catalog browse page; keyboard navigation through filter panel.
- **Performance / load** — Catalog browse with 5,000 sections; p95 <2 s.
- **Manual exploratory** — QA checklist: cross-listed sections, variable-credit courses, cancelled sections, sections with no instructor assigned.
