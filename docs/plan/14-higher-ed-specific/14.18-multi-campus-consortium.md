## 1. Problem Statement

University systems (e.g., University of California, SUNY, CSU) and formal consortia (e.g., Five Colleges, Statewide Distance Learning Consortia) frequently share courses across campuses. A student at Campus A may enroll in an online section taught by an instructor at Campus B, with grades recorded at both institutions. Lextures currently has no multi-tenant course-sharing model — courses are siloed within a single organization. Without this, multi-campus systems cannot deploy Lextures for their shared online programs, and consortium partnerships are impossible.

## 2. Goals

- Enable a course created at one institution (the "host") to accept enrollments from students at partner institutions (the "guests").
- Preserve institutional identity: guest students see their home institution's branding; grades are reported back to the guest institution's SIS.
- Allow content sharing agreements to be managed at the consortium level.
- Ensure FERPA isolation: guest institution cannot see host institution's non-shared students.

## 3. Non-Goals

- Single sign-on federation between institutions (covered by 4.1 SAML).
- Financial billing between institutions (consortium contract management is out of scope).
- Real-time synchronization of catalog changes across institutions.
- K-12 district course sharing (different governance model).

## 4. Personas & User Stories

- **As a student at a small consortium college**, I want to enroll in an advanced course offered only at the flagship campus so that I can access specialized instruction.
- **As a host institution IT admin**, I want to control which courses are visible for consortium enrollment so that I do not inadvertently share unpublished or restricted content.
- **As a guest institution registrar**, I want enrolled student grades returned to our SIS as if the student had completed an in-house course.
- **As a student**, I want my LMS experience to look like my home institution even when attending a course at a partner campus.

## 5. Functional Requirements

- **FR-1.** The system MUST support a `consortium_agreements` model linking two or more organizations with a shared-course permission scope.
- **FR-2.** The system MUST allow host institutions to designate specific courses as "consortium-shareable" and visible to partner institutions.
- **FR-3.** The system MUST allow guest institution students to enroll in shared courses, creating an enrollment record linked to both the host course and the guest institution.
- **FR-4.** The system MUST ensure that grades for cross-institutional enrollments are exported via the guest institution's SIS passback configuration (14.1 / 14.5), not only the host's.
- **FR-5.** The system SHOULD present the guest student's home-institution branding (logo, colors) within the shared course interface.
- **FR-6.** The system MUST enforce data isolation: instructors and admins at the host institution see only their own institution's enrollment data; guest institution data is accessible only to that guest institution's admins.
- **FR-7.** The system SHOULD generate consortium enrollment reports showing cross-institution headcounts for capacity planning.

## 6. Non-Functional Requirements

- **Performance** — Consortium course browse for guest institution students loads in <2 s.
- **Security** — Cross-institutional data isolation enforced at the query level via `org_id` filters; no row leakage between institutions.
- **Privacy & Compliance** — FERPA: sharing student enrollment records across institutional boundaries requires a formal consortium agreement and institutional FERPA disclosure.
- **Accessibility** — Consortium course browse WCAG 2.1 AA.
- **Scalability** — Consortium model supports up to 50 partner institutions per host; course sharing supports up to 1,000 cross-enrolled students per shared course.
- **Reliability** — Host institution outage does not affect guest institution's existing enrollments (read-only cache of enrollment status).
- **Observability** — Metric `consortium_enrollments_total{host_org, guest_org}`.
- **Maintainability** — Consortium logic in `server/src/services/consortium.rs`; minimal coupling to core course logic.
- **Internationalization** — Institution name and branding used as-configured; no locale transforms needed.
- **Backward compatibility** — Single-institution courses (99% of cases) are entirely unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a consortium agreement between Institution A (host) and Institution B (guest), *when* Institution A marks a course as shareable, *then* a student from Institution B can enroll and access the course.
- **AC-2.** *Given* a guest student completes a shared course, *when* the host institution runs grade passback to their SIS, *then* the guest student's grade is simultaneously passed back to Institution B's SIS using Institution B's passback configuration.
- **AC-3.** *Given* a host institution admin views the course roster, *when* there are guest students enrolled, *then* they are labeled with their home institution but no other Institution B-specific data is shown.
- **AC-4.** *Given* an Institution B admin attempts to query Institution A's non-shared course list, *when* they call the API, *then* the response contains only Institution A's shareable courses, not their full catalog.
- **AC-5.** *Given* a guest student accesses a shared course, *when* the LMS renders the course page, *then* the header shows Institution B's logo and colors rather than Institution A's.

## 8. Data Model

Migration `091_consortium.sql`:
```sql
CREATE TABLE consortium_agreements (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  host_org_id  UUID NOT NULL REFERENCES organizations(id),
  guest_org_id UUID NOT NULL REFERENCES organizations(id),
  status       TEXT NOT NULL DEFAULT 'active', -- 'pending', 'active', 'terminated'
  signed_at    TIMESTAMPTZ,
  expires_at   TIMESTAMPTZ,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(host_org_id, guest_org_id)
);

ALTER TABLE courses ADD COLUMN consortium_shareable BOOLEAN NOT NULL DEFAULT FALSE;

ALTER TABLE course_enrollments
  ADD COLUMN home_org_id UUID REFERENCES organizations(id);
  -- non-null for cross-institutional enrollments
```

## 9. API Surface

```
POST /api/v1/admin/consortium/agreements       -- create agreement (both orgs must confirm)
GET  /api/v1/consortium/courses?guest_org_id=  -- browse shareable courses
POST /api/v1/consortium/courses/:id/enroll     -- cross-institutional enrollment
GET  /api/v1/admin/consortium/enrollment-report?agreement_id=
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| FERPA violation from cross-institutional data leakage | M | H | Row-level org_id enforcement in all queries; security review before GA |
| Grade passback to two SIS systems causes duplicate records | M | M | Passback configuration is per-enrollment home_org_id; host institution does not passback guest grades to its own SIS |
| Consortium agreement terminates mid-term with enrolled students | L | H | Enrolled students complete their current term regardless of agreement status |

## 15. Rollout Plan

- Feature flag: `consortium_sharing` (default: off).
- Pilot with a two-institution consortium with an existing relationship.
- Legal review of FERPA data-sharing disclosure requirements before GA.
- GA after dual-institution grade passback is validated.

## 16. Test Plan

- **Unit** — Consortium agreement validation; cross-org enrollment creation; branding context switching.
- **Integration** — Shared course enrollment API; dual SIS passback; data isolation query tests.
- **End-to-end** — Playwright: guest student enrolls in shared course, completes it, grade appears in both SIS mocks.
- **Security** — Attempt cross-org data access without consortium agreement; verify 403.
- **Accessibility** — axe on consortium catalog browse and agreement admin page.
- **Manual exploratory** — QA: agreement termination, guest student unenrollment, host institution making course non-shareable while enrolled students exist.
