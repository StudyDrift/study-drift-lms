## 1. Problem Statement

Lextures courses have free-form `available_from` / `available_until` date ranges but no shared institutional calendar. Universities operate on precisely dated terms with legally significant milestones — the last day to add, last day to drop without a W, last day to withdraw, final exam schedule, spring/fall breaks, and holidays. Without an academic calendar model, the LMS cannot enforce enrollment deadlines (14.3), schedule course evaluations to open at the right time (14.7), or display relevant dates to students and instructors. Instructors also manually re-enter term dates on every syllabus.

## 2. Goals

- Provide a per-institution academic calendar object containing named events (add/drop deadline, withdrawal deadline, finals week, no-class days, term start/end).
- Integrate calendar events into deadline enforcement for enrollment state transitions (14.3) and course evaluations (14.7).
- Surface relevant calendar events to students on the Dashboard and in course sidebars.
- Allow SIS sync to populate or update calendar events from Banner/Workday term definitions.
- Export calendar events as iCal/CalDAV feeds (16.5).

## 3. Non-Goals

- Student-personal calendars or per-assignment due-date calendar (covered in 16.5).
- Scheduling room assignments or physical classroom booking.
- K-12 bell schedule management (different feature).
- Synchronous class meeting management (those are meeting patterns from 14.2, not the institutional calendar).

## 4. Personas & User Stories

- **As a registrar**, I want to enter the academic calendar once per term and have all date-dependent behaviors in the LMS (enrollment deadlines, evaluation windows) derive from it automatically.
- **As an instructor**, I want the syllabus to auto-populate the term's no-class days and final exam date so that I do not need to look them up each term.
- **As a student**, I want to see the drop deadline and finals week dates on my LMS dashboard so that I stay informed without consulting the registrar website.
- **As an admin**, I want to configure recurring institutional calendar templates so that common holidays copy forward each year.

## 5. Functional Requirements

- **FR-1.** The system MUST support an `academic_calendar_events` table with event types: `term_start`, `term_end`, `add_drop_deadline`, `withdrawal_deadline`, `finals_start`, `finals_end`, `no_class_day`, `holiday`, `custom`.
- **FR-2.** The system MUST link calendar events to a term (from 5.3) and an institution (org_id).
- **FR-3.** The system MUST expose calendar events to the enrollment state machine (14.3) for deadline enforcement — the drop and withdrawal deadline event types are authoritative.
- **FR-4.** The system MUST expose a calendar widget on the student Dashboard showing upcoming calendar events for all enrolled courses' terms.
- **FR-5.** The system SHOULD allow import of calendar data from the SIS (Banner term definitions, Workday Academic Calendars) via the SIS adapter (14.1).
- **FR-6.** The system SHOULD generate an iCal feed URL per term that students can subscribe to in external calendar apps (16.5).
- **FR-7.** The system MAY display a "Finals Week" banner on course pages during the finals window.
- **FR-8.** The system MUST allow admins to create, edit, and delete calendar events for their institution's terms through an admin UI.

## 6. Non-Functional Requirements

- **Performance** — Calendar widget on Dashboard loads events for up to 5 concurrent terms in <300 ms.
- **Security** — Calendar event CRUD is admin-only; read access is authenticated users in the institution.
- **Privacy & Compliance** — Calendar data is institutional, not personally identifiable; no FERPA concerns.
- **Accessibility** — Calendar widget WCAG 2.1 AA; date displays use `<time>` elements with `datetime` attributes.
- **Scalability** — Up to 10,000 calendar events per institution per year is within Postgres range.
- **Reliability** — Calendar events are read-heavy; cached in Redis for 1 hour.
- **Observability** — No special metrics required; standard API latency tracking.
- **Maintainability** — Calendar service is a clean domain module; no coupling to grading logic beyond the deadline accessor functions.
- **Internationalization** — Event names externalised; dates displayed in institution's locale and time zone.
- **Backward compatibility** — Existing courses without calendar linkage continue to use their manual `available_from` / `available_until` dates.

## 7. Acceptance Criteria

- **AC-1.** *Given* a registrar creates an `add_drop_deadline` event for Spring 2027, *when* an instructor attempts to drop a student after that date, *then* the enrollment state machine rejects the transition with the deadline date in the error message.
- **AC-2.** *Given* a student is enrolled in a Spring 2027 course, *when* they open their Dashboard, *then* they see "Add/Drop Deadline: Jan 20" and "Spring Break: Mar 8–12" in the upcoming events panel.
- **AC-3.** *Given* a no-class day is configured, *when* the system checks whether an assignment due that day should be auto-extended, *then* the no-class day is surfaced to the instructor as a warning in the assignment editor.
- **AC-4.** *Given* an admin imports calendar events from Banner term definitions, *when* the sync completes, *then* term start/end, add/drop, and withdrawal deadlines are populated automatically.
- **AC-5.** *Given* a student subscribes to the iCal feed URL for their current term, *when* they open their calendar app, *then* the term events appear as all-day events with descriptive titles.

## 8. Data Model

Migration `079_academic_calendar.sql`:
```sql
CREATE TYPE calendar_event_type AS ENUM (
  'term_start', 'term_end', 'add_drop_deadline', 'withdrawal_deadline',
  'finals_start', 'finals_end', 'no_class_day', 'holiday', 'custom'
);

CREATE TABLE academic_calendar_events (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id       UUID NOT NULL REFERENCES organizations(id),
  term_id      UUID REFERENCES terms(id),
  event_type   calendar_event_type NOT NULL,
  event_name   TEXT NOT NULL,
  start_date   DATE NOT NULL,
  end_date     DATE,
  all_day      BOOLEAN NOT NULL DEFAULT TRUE,
  notes        TEXT,
  sis_id       TEXT,
  created_by   UUID REFERENCES users(id),
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_calendar_events_org_term ON academic_calendar_events(org_id, term_id, start_date);
```

## 9. API Surface

```
GET    /api/v1/calendar/events?term_id=&org_id=   -- list events (authenticated)
POST   /api/v1/admin/calendar/events              -- create event (admin)
PATCH  /api/v1/admin/calendar/events/:id          -- update event (admin)
DELETE /api/v1/admin/calendar/events/:id          -- delete event (admin)
GET    /api/v1/calendar/terms/:term_id/ical        -- iCal feed (public with token)
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Registrar enters wrong deadline date causing enrollment errors | M | H | Admin confirmation dialog; show affected enrollment count before saving |
| Time zone ambiguity for midnight deadlines | H | M | All deadline events stored at institution's canonical time zone; displayed in user's local TZ |
| SIS calendar import overwrites manually adjusted dates | L | M | Import previews changes before applying; manual overrides flagged |

## 15. Rollout Plan

- Feature flag: `academic_calendar` (default: off).
- Admin UI ships first for manual event entry; SIS import in follow-up.
- Enable for HE institutions; require at least term_start/term_end before 14.3 deadline enforcement activates.
- GA with 14.3 and 14.7.

## 16. Test Plan

- **Unit** — Calendar event type validation; deadline accessor function for enrollment state machine; iCal serialization.
- **Integration** — CRUD API; deadline enforcement integration test with 14.3.
- **End-to-end** — Playwright: admin creates calendar events, student sees Dashboard widget, instructor sees deadline warning in enrollment UI.
- **Security** — Assert non-admin cannot create/edit events; assert iCal feed token cannot be guessed.
- **Accessibility** — axe on calendar admin page and Dashboard widget.
- **Performance / load** — Dashboard widget with 5 concurrent terms; p95 <300 ms.
- **Manual exploratory** — QA: leap years, DST transitions, multi-term student with overlapping events.
