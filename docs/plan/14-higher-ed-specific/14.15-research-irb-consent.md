## 1. Problem Statement

Universities frequently conduct IRB-approved research studies using LMS data — learning analytics, pedagogical experiments, and AI model training all require explicit informed consent from students. Currently Lextures has no consent management system: no mechanism to present an IRB consent form, record consent decisions with timestamps, or restrict data access based on consent. This creates regulatory exposure (45 CFR Part 46, the Common Rule for human-subjects research) and blocks research-active institutions from adopting Lextures for studies that involve student data.

## 2. Goals

- Provide a consent management module allowing researchers to create IRB consent forms and present them to target student populations.
- Record consent decisions (granted/declined/withdrawn) with cryptographic timestamps.
- Gate data access (analytics exports, LLM training data) based on active consent for each research study.
- Allow students to view and withdraw their consent at any time.
- Provide a researcher dashboard showing consent rates and ability to export data for consenting participants only.

## 3. Non-Goals

- Building an IRB submission or approval workflow — Lextures only executes IRB-approved protocols.
- Medical consent or HIPAA-adjacent workflows.
- Consent for routine data processing (that is the privacy policy, handled in 10.3 GDPR).
- K-12 parental consent for research (covered separately under COPPA/10.2).

## 4. Personas & User Stories

- **As a researcher**, I want to upload my IRB-approved consent form and target it to students in specific courses so that I can collect consent without emailing students directly.
- **As a student**, I want to review the research study's consent form and choose whether to participate so that I understand how my data will be used.
- **As a student**, I want to withdraw my consent at any time so that my data is removed from the research dataset.
- **As a privacy officer**, I want a log of every consent decision so that I can demonstrate compliance with 45 CFR 46 in an audit.
- **As a researcher**, I want to export LMS data only for students who have consented so that I comply with my IRB protocol.

## 5. Functional Requirements

- **FR-1.** The system MUST allow authorized researchers to create consent studies with a title, IRB protocol number, consent form text (Markdown), data-use description, and target enrollment group.
- **FR-2.** The system MUST present the consent form to targeted students as an interstitial prompt (on login or Dashboard visit) when the study is active.
- **FR-3.** The system MUST record each consent decision (granted/declined/withdrawn) with user ID, study ID, timestamp, IP address, and user-agent in an immutable `consent_records` table.
- **FR-4.** The system MUST allow students to view all active and historical consent decisions on their privacy settings page and withdraw consent from any active study.
- **FR-5.** The system MUST gate data export for research studies to only include records where `consent = granted` and withdrawal has not occurred before the export date.
- **FR-6.** The system SHOULD notify the researcher when consent rate reaches a configurable threshold or when consent is withdrawn by a participant.
- **FR-7.** The system MAY sign consent records with a server-side HMAC to provide tamper evidence for audit purposes.

## 6. Non-Functional Requirements

- **Performance** — Consent decision record <200 ms; consent form display <500 ms.
- **Security** — Consent records are append-only; no UPDATE/DELETE on `consent_records`; researcher may only access data for their own studies; admin access to all studies.
- **Privacy & Compliance** — 45 CFR Part 46 (Common Rule): voluntary, informed consent required before data collection; right to withdraw; IRB protocol number must be stored; GDPR Article 7 (consent conditions) applies for EU students.
- **Accessibility** — Consent form WCAG 2.1 AA; consent form is full-text readable without JS (PDF fallback link).
- **Scalability** — Consent prompts shown at login do not block the main auth flow; displayed asynchronously.
- **Reliability** — Consent prompt idempotent — shown at most once per study per student login session; state stored in session.
- **Observability** — Metric `consent_decisions_total{study_id, decision}`.
- **Maintainability** — Consent module in `server/src/services/research_consent/`.
- **Internationalization** — Consent form text stored per locale if researcher provides translations.
- **Backward compatibility** — No impact on existing user data or login flow for institutions without active studies.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student targeted by an active consent study logs in, *when* the Dashboard loads, *then* a consent prompt appears with the full consent form text and "I Agree" / "No Thanks" options.
- **AC-2.** *Given* a student declines a study, *when* the researcher attempts to export data for consenting students only, *then* the declined student's records are not included in the export.
- **AC-3.** *Given* a student withdraws consent after previously granting it, *when* the researcher queries the consent records, *then* the withdrawal timestamp is recorded and the student is excluded from future data exports.
- **AC-4.** *Given* a privacy officer queries the consent audit log, *when* they filter by study ID, *then* all grant/decline/withdraw events appear with timestamp and IP address.
- **AC-5.** *Given* a researcher attempts to access a consent study they did not create, *when* they call the admin API, *then* the system returns a 403 Forbidden.

## 8. Data Model

Migration `088_research_consent.sql`:
```sql
CREATE TABLE consent_studies (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          UUID NOT NULL REFERENCES organizations(id),
  researcher_id   UUID NOT NULL REFERENCES users(id),
  title           TEXT NOT NULL,
  irb_protocol    TEXT NOT NULL,
  consent_text    TEXT NOT NULL,
  data_use_desc   TEXT NOT NULL,
  target_criteria JSONB NOT NULL DEFAULT '{}', -- course_ids, term_id, enrollment_state, etc.
  status          TEXT NOT NULL DEFAULT 'draft', -- 'draft', 'active', 'closed'
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE consent_records (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  study_id    UUID NOT NULL REFERENCES consent_studies(id),
  user_id     UUID NOT NULL REFERENCES users(id),
  decision    TEXT NOT NULL CHECK (decision IN ('granted', 'declined', 'withdrawn')),
  ip_address  INET,
  user_agent  TEXT,
  hmac        TEXT,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE UNIQUE INDEX idx_consent_latest ON consent_records(study_id, user_id, created_at DESC);
-- Append-only enforced via DB role permission (no UPDATE/DELETE on consent_records)
```

## 9. API Surface

```
POST /api/v1/admin/consent-studies            -- create study (researcher/admin)
PATCH /api/v1/admin/consent-studies/:id       -- update/activate study
GET  /api/v1/me/consent-studies               -- student: list studies pending consent
POST /api/v1/me/consent-studies/:id/respond   -- student: grant, decline, withdraw
GET  /api/v1/me/consent-studies/history       -- student: consent history
GET  /api/v1/admin/consent-studies/:id/export -- researcher: export consenting participant IDs
```

## 11. AI / ML Considerations

Consent gating is directly relevant to AI features: any LLM fine-tuning or prompt logging using student data MUST check for active research consent. The consent module is foundational for 19.11 PII redaction and 10.17 AI usage disclosure.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Students ignore consent prompts | H | M | Re-surface prompt at next login; cap at 3 prompts then mark as "not responded" |
| Researcher access to non-consenting data via other API | M | H | API-level enforcement: all data exports check consent_records before returning rows |
| IRB protocol changes require re-consent | L | H | Study versioning: new IRB version creates new study requiring fresh consent |

## 15. Rollout Plan

- Feature flag: `research_consent` (default: off).
- Pilot with one research group at a partner institution.

## 16. Test Plan

- **Unit** — Consent targeting criteria evaluation; HMAC record signing; export filter logic.
- **Integration** — API: create study, student responds, researcher exports — verify export filter.
- **End-to-end** — Playwright: student sees prompt, consents, withdraws, researcher sees updated count.
- **Security** — Assert researcher cannot access another researcher's study; assert consent_records cannot be updated or deleted via API.
- **Accessibility** — axe on consent prompt modal; checkbox label association.
- **Manual exploratory** — QA: student who declines then reconsiders (new grant record), prompt timing, multi-study targeting.
