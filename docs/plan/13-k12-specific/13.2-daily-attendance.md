## 1. Problem Statement

Attendance tracking is a core legal and operational requirement for every K-12 school. States receive ADA (Average Daily Attendance) funding based on daily attendance records and require schools to report using state-specific codes (e.g., California CALPADS attendance codes: P-Present, A-Absent, T-Tardy, E-Excused, H-Home Hospital, etc.). Lextures has no attendance schema, no UI for teachers to take roll, no per-period secondary school mode, no daily elementary school mode, and no state-exportable attendance report. Without attendance, Lextures cannot be used as a primary SIS-adjacent system, and parent portals (§13.1) are meaningless. This is a BLOCKER for every K-12 district deployment in any US state.

## 2. Goals

- Implement flexible attendance recording supporting daily (elementary) and per-period (secondary) modes.
- Support configurable attendance codes mapping to state-specific formats (CALPADS as reference implementation; pluggable for other states).
- Provide a teacher roll-taking UI integrated into the course/section interface.
- Generate state-reportable attendance summary exports (daily, weekly, grading-period).
- Expose attendance summaries to the parent portal (§13.1) and report cards (§13.4).

## 3. Non-Goals

- Automated attendance via face recognition or device check-in.
- Truancy workflow and legal notices (administrative process, not LMS scope in v1).
- Attendance for virtual/asynchronous learning (complex "engaged" definition; deferred to v2).
- ADA funding calculation engine (state-specific formula; export data only).

## 4. Personas & User Stories

- **As a secondary teacher**, I want to take roll for my 3rd period class each day in under 2 minutes by clicking a student's name to toggle their status so that I can get back to instruction quickly.
- **As an elementary teacher**, I want to record a single daily attendance status for each student so that the school office has an accurate count by 9 AM.
- **As a school secretary**, I want to see a real-time class-by-class attendance dashboard at the start of the day so that I can quickly identify students absent in some periods but not reported as absent by the office.
- **As a district data coordinator**, I want to export attendance in CALPADS format at the end of each reporting period so that our ADA funding submission is accurate.
- **As a parent**, I want to see my child's attendance record in the parent portal so that I can monitor tardiness.

## 5. Functional Requirements

- **FR-1.** The system MUST support two attendance modes per school (§5.2): `daily` (one record per student per school day) and `period` (one record per student per course section per day) (California CALPADS, IDEA, federal ADA funding requirements).
- **FR-2.** Attendance MUST be recordable by authorized instructors for their course section roster, with a default status of `not_taken` that is distinguishable from `present`.
- **FR-3.** The system MUST support a configurable set of attendance codes per organization (§5.1), including at minimum: Present (P), Absent-Unexcused (AU), Absent-Excused (AE), Tardy-Unexcused (TU), Tardy-Excused (TE). States may add custom codes (e.g., CALPADS H = Home Hospital, R = Independent Study).
- **FR-4.** Teachers MUST be able to record attendance via a grid UI showing all students in the section, with one-click status selection and a bulk "Mark all present" action for speed.
- **FR-5.** Attendance records MUST be editable by the instructor up to 5 school days after the recorded date; office staff (`admin` role) MUST be able to edit any past record.
- **FR-6.** The system MUST provide a school-level attendance dashboard showing per-period and per-day absence rates for the current day.
- **FR-7.** The system MUST generate attendance export files in: (a) CALPADS CalPADS SB format (California), (b) generic CSV with configurable column mapping, for any grading period or date range.
- **FR-8.** The system SHOULD send an automated absence notification to parent (§13.1) when a student is marked absent for any period (configurable per school policy).

## 6. Non-Functional Requirements

- **Performance** — Roll-taking UI for a class of 35 students must load in < 500 ms. Batch attendance save (all 35 students in one request) must complete in < 300 ms p95.
- **Security** — Teachers can only record/edit attendance for their own course section's roster. Admin can edit any school's attendance. Attendance data is FERPA-protected student education record.
- **Privacy & Compliance** — FERPA (20 U.S.C. §1232g): attendance is an education record. CALPADS (California ADA reporting): CalPADS format spec compliance for SB (Student Body) attendance extract. State-specific ADA codes vary; design must be pluggable. COPPA: attendance data for under-13 students subject to data minimization.
- **Accessibility** — Roll-taking grid must be keyboard-navigable (WCAG SC 2.1.1); screen-reader compatible (§12 series); status change announced via live region.
- **Scalability** — A district with 500 sections and 35 students each = 17,500 rows/day; 1.2M rows/school year. Indexed by `(section_id, date)` and `(student_id, date)`.
- **Reliability** — Attendance saves use upsert (idempotent); teachers can re-save without duplicate records. Offline queue (future) allows roll-taking without network.
- **Observability** — Metrics: `attendance_records_saved_per_day`, `export_job_duration`. Alert: `attendance_export_failure`.
- **Maintainability** — Attendance codes defined in a `attendance_codes` table (not hardcoded); new state can add codes via admin UI.
- **Internationalization** — Attendance status labels externalized for i18n; date formatting locale-aware.
- **Backward compatibility** — No existing tables affected. New schema is additive.

## 7. Acceptance Criteria

- **AC-1.** *Given* a teacher with a 30-student section opens the roll-taking UI for today, *When* they click "Mark all present" then change 2 students to "Absent-Excused", *Then* all 30 records are saved in a single batch request in < 300 ms and the UI confirms "Attendance saved."
- **AC-2.** *Given* a school in California has CALPADS codes configured, *When* a district coordinator exports attendance for October, *Then* the downloaded file matches the CALPADS SB extract format with correct code mapping.
- **AC-3.** *Given* a student is marked absent in 1st period, *When* a parent with notification preference "absence alert" enabled receives the notification, *Then* the email arrives within 10 minutes and contains the period, course name, and absence code.
- **AC-4.** *Given* a teacher attempts to edit attendance from 6 days ago, *When* the save is attempted, *Then* the system returns an error "Attendance edits are limited to 5 days; contact your administrator."
- **AC-5.** *Given* a keyboard-only user navigates the roll-taking grid for 35 students, *When* they tab through each row and use Space to toggle status, *Then* they can complete the roll without mouse input and each change is announced by screen reader.
- **AC-6.** *Given* a parent views the portal attendance summary for their child, *When* there are 3 absences in the current term, *Then* the summary shows date, period, and code for each absence.
- **AC-7.** *Given* the attendance export API is called for a 500-student school for one month, *When* the export job completes, *Then* the file is generated in < 30 seconds and contains no records for non-school days (weekends, holidays from the school calendar).

## 8. Data Model

```sql
-- server/migrations/081_attendance.sql
CREATE TABLE attendance_codes (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    org_id      UUID NOT NULL REFERENCES organizations(id),
    code        TEXT NOT NULL,           -- e.g. P, AU, AE, TU, TE, H, R
    label       TEXT NOT NULL,           -- e.g. "Present", "Absent-Excused"
    state_code  TEXT,                    -- CALPADS or other state-specific code
    category    TEXT NOT NULL,           -- present | absent | tardy | other
    UNIQUE(org_id, code)
);

CREATE TABLE attendance_records (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    student_id    UUID NOT NULL REFERENCES users(id),
    section_id    UUID NOT NULL REFERENCES course_sections(id),   -- §5.4
    school_id     UUID NOT NULL REFERENCES schools(id),           -- §5.2
    date          DATE NOT NULL,
    period        TEXT,           -- NULL for daily mode; "1" "2" etc for period mode
    code_id       UUID NOT NULL REFERENCES attendance_codes(id),
    note          TEXT,
    recorded_by   UUID REFERENCES users(id),
    recorded_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX attendance_unique ON attendance_records(student_id, section_id, date, COALESCE(period, ''));
CREATE INDEX attendance_section_date ON attendance_records(section_id, date);
CREATE INDEX attendance_student_date ON attendance_records(student_id, date);
```

## 9. API Surface

- `GET /api/sections/:id/attendance/:date` — get attendance for a section on a date.
- `PUT /api/sections/:id/attendance/:date` — batch upsert all student records for date.
- `GET /api/students/:id/attendance` — student attendance history (parent/admin/instructor).
- `GET /api/schools/:id/attendance/dashboard` — real-time school attendance dashboard.
- `POST /api/admin/attendance/export` — trigger export job; returns job ID.
- `GET /api/admin/attendance/codes` — list/manage attendance codes.
