## 1. Problem Statement

Every K-12 LMS that Lextures competes with — Canvas, Schoology, PowerSchool Learning — provides a parent/guardian portal where parents can monitor their children's grades, attendance, assignments, and announcements. Lextures has no parent role, no child-linkage mechanism, and no parent-facing UI. This is a BLOCKER for K-12 district sales because parents are a key stakeholder constituency in every RFP evaluation, and districts are legally obligated under FERPA to provide parents of minor students access to their child's educational records. Without a parent portal, every K-12 deployment requires the district to maintain a second system solely for parent communication — a non-starter. This plan delivers the parent role, parent-child account linking, and a multi-child dashboard.

## 2. Goals

- Implement a `parent` / `guardian` role with read-only access to linked children's grades, assignments, attendance, and course content.
- Support multi-child households with a switcher in the portal UI.
- Provide parent-specific views of the gradebook, assignment list, and upcoming due dates.
- Allow parents to opt in to email/push notifications for grade updates and missing assignments.
- Enforce FERPA: parent access to records of minor students (under 18) only; student age-majority transition revokes parent access automatically.

## 3. Non-Goals

- Parent messaging system (parental inbox cross-references §6.2 email notifications).
- Direct grade-dispute submission by parents (district workflow).
- Parent access to other students' data (strict child-linkage enforcement).
- Parent portal mobile app (§7.1 native app is separate).

## 4. Personas & User Stories

- **As a parent**, I want to see all three of my children's grades and upcoming assignments on one dashboard so that I can monitor their academic progress without switching between multiple accounts.
- **As a parent**, I want to receive an email when my child receives a low grade so that I can provide support proactively.
- **As a guardian**, I want to see my child's attendance record so that I can confirm the school's records match reality.
- **As a district admin**, I want to provision parent accounts from a CSV roster so that all families are connected to the platform from day one.
- **As a student turning 18**, I want my parent's access to my records to be automatically revoked so that my privacy is protected.

## 5. Functional Requirements

- **FR-1.** The system MUST implement a `parent` role that can be linked to one or more student accounts via a `parent_student_links` table (FERPA §99.31 — school officials with legitimate educational interest).
- **FR-2.** A parent MUST be able to view (read-only): their linked children's current grades per course, individual assignment scores, attendance summary (§13.2), and upcoming due dates.
- **FR-3.** A parent account MUST be able to switch between linked children via a top-nav child switcher when multiple children are enrolled.
- **FR-4.** Parent access to a student's records MUST be automatically revoked when the student's date-of-birth indicates they have reached age 18 (age-majority transfer of rights under FERPA §99.5).
- **FR-5.** Parents MUST be able to subscribe to email notifications for: (a) grade posted, (b) missing assignment (> 24 h past due), (c) low grade threshold (configurable, e.g., < 70%), (d) attendance event.
- **FR-6.** A parent MUST NOT be able to create, edit, or delete any course content, submissions, or grade records. All parent API endpoints MUST be read-only.
- **FR-7.** District admins MUST be able to provision parent-student links in bulk via CSV import (student external ID, parent email).
- **FR-8.** The parent portal MUST display a weekly summary view showing all children's activities for the current week, ordered by due date.

## 6. Non-Functional Requirements

- **Performance** — Parent dashboard (all children, all courses) must load in < 1.5 s p95 with up to 6 linked children each in 8 courses.
- **Security** — Parent can only access data for their explicitly linked children; `server/src/authz.rs` enforces this with a join through `parent_student_links`. Attempting to access another student's data returns HTTP 403. Parent tokens carry a `role: parent` claim and are scoped to linked child IDs.
- **Privacy & Compliance** — FERPA (20 U.S.C. §1232g): parents of minor students have legal right to access education records; age-majority transition transfers rights to student. COPPA (§10.2): parent account itself is adult; linked student records subject to COPPA if student is under 13. State laws vary (e.g., California Education Code §49076). Parent access logs maintained in `user_audit` for FERPA compliance.
- **Accessibility** — Parent portal must meet WCAG 2.1 AA (§12 series); axe-core clean; multi-child switcher keyboard-accessible.
- **Scalability** — Large districts may have 5,000+ parent accounts; parent dashboard query optimized with indexed child-link join.
- **Reliability** — Read-only portal; no write transactions; degradation to empty state on DB slowness without portal crash.
- **Observability** — Metrics: `parent_portal_dau`, `notification_delivery_rate`. Alert: `parent_auth_error_rate > 1%`.
- **Maintainability** — Parent-specific routes namespaced under `/api/parent/` in a new `server/src/routes/parent.rs` module.
- **Internationalization** — Parent portal fully i18n-ready (§11.1); many K-12 parents are non-English speakers.
- **Backward compatibility** — Existing student and instructor accounts unaffected. Parent role is additive.

## 7. Acceptance Criteria

- **AC-1.** *Given* a parent account is linked to two students, *When* they log in, *Then* the dashboard shows a child switcher with both children's names and the first child's grades and upcoming assignments are displayed.
- **AC-2.** *Given* a parent attempts to call `PATCH /api/courses/:id/assignments/:id`, *When* the request is received, *Then* the server returns HTTP 403 and no modification occurs.
- **AC-3.** *Given* a student turns 18 (simulated by setting DOB to 18 years ago), *When* the nightly age-majority job runs, *Then* the parent's `parent_student_links` record for that student is soft-deleted and the parent's next login does not show the student.
- **AC-4.** *Given* a parent is subscribed to "grade posted" notifications, *When* an instructor posts a grade for their child, *Then* the parent receives an email within 5 minutes containing the assignment name and score (no other student data).
- **AC-5.** *Given* a district admin uploads a CSV with 100 parent-student link rows, *When* the import completes, *Then* all 100 links are created and a summary email is sent to the admin with success/failure counts.
- **AC-6.** *Given* VoiceOver is active, *When* a parent navigates the multi-child switcher, *Then* VoiceOver announces each child's name and "selected" state for the active child.
- **AC-7.** *Given* a parent views the weekly summary, *When* there are 5 assignments due across 2 children, *Then* the summary lists all 5 ordered by due date, each labeled with the child's name.

## 8. Data Model

```sql
-- server/migrations/080_parent_portal.sql
CREATE TABLE parent_student_links (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    parent_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    student_id    UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    relationship  TEXT NOT NULL DEFAULT 'guardian',  -- parent | guardian | foster | emergency
    active        BOOLEAN NOT NULL DEFAULT true,
    linked_by     UUID REFERENCES users(id),         -- admin who created the link
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    revoked_at    TIMESTAMPTZ,
    UNIQUE(parent_id, student_id)
);
CREATE INDEX psl_parent_idx ON parent_student_links(parent_id) WHERE active = true;
CREATE INDEX psl_student_idx ON parent_student_links(student_id) WHERE active = true;

CREATE TABLE parent_notification_prefs (
    parent_id            UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    grade_posted         BOOLEAN NOT NULL DEFAULT true,
    missing_assignment   BOOLEAN NOT NULL DEFAULT true,
    low_grade_threshold  INTEGER DEFAULT 70,  -- null = disabled
    attendance_event     BOOLEAN NOT NULL DEFAULT false,
    updated_at           TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 9. API Surface

- `GET /api/parent/children` — list linked children with basic profile.
- `GET /api/parent/children/:studentId/grades` — child's gradebook (read-only).
- `GET /api/parent/children/:studentId/assignments` — upcoming/past assignments.
- `GET /api/parent/children/:studentId/attendance` — attendance summary (§13.2).
- `GET /api/parent/weekly-summary` — all children, current week.
- `GET/PATCH /api/parent/notification-prefs` — notification preferences.
- `POST /api/admin/parent-links/import` — bulk CSV import; admin only.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Parent accesses another student's data due to authz bug | L | H | Comprehensive role-matrix integration tests; security review |
| Age-majority job fails silently; parent retains access past 18 | M | H | Alert on job failure; fallback check on every parent API call |
| Large multi-child districts with 5 children each cause slow dashboard | M | M | Parallelized child queries; dashboard caching with 1-min TTL |

## 15. Rollout Plan

- Feature flag: `ff_parent_portal` (default `false`).
- Sequencing: DB migration → authz → API → UI → notification wiring → flip flag.
