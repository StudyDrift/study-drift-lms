## 1. Problem Statement

Title I of the Elementary and Secondary Education Act (ESEA/ESSA) requires schools to identify and support educationally disadvantaged students, particularly those from low-income families (proxied by free/reduced-price lunch eligibility under the National School Lunch Program). Schools and districts must report aggregate demographic data (race/ethnicity, ELL status, disability status, economic disadvantage) to state education agencies and the federal government (CALPADS, EdFacts). Lextures has no demographic data schema and no Title I reporting capability. Without this, Lextures cannot be used as an authoritative data system for state and federal reporting, limiting its adoption in Title I schools (which represent over 50% of US public schools). This data also powers at-risk early-warning systems (§9.2) and equity reporting.

## 2. Goals

- Store student demographic flags (free/reduced lunch, ELL, disability, race/ethnicity) sourced from SIS integration (§13.7) with strict access controls.
- Provide aggregate (anonymized) demographic reports for Title I compliance.
- Enable at-risk analysis (§9.2) that can be correlated with demographic factors.
- Ensure the strictest possible access controls: demographic data visible only to authorized school officials, never to peers or parents.

## 3. Non-Goals

- Individual-level demographic reporting to external parties (aggregate only from Lextures).
- SNAP/WIC program administration.
- Storing raw application forms or income documentation.
- Student discipline disparate-impact analysis (complex; deferred).

## 4. Personas & User Stories

- **As a district Title I coordinator**, I want Lextures to display aggregate data on economically disadvantaged student performance vs. non-disadvantaged students so that I can write the school's Title I plan.
- **As a school data analyst**, I want to run a report showing average quiz scores disaggregated by ELL status so that I can identify where ELL students need additional support.
- **As a district IT director**, I want demographic flags to sync automatically from our SIS (§13.7) so that we do not have to maintain this sensitive data in two places.
- **As a compliance officer**, I want all access to individual-level demographic data logged in the audit trail so that I can demonstrate appropriate access controls to auditors.

## 5. Functional Requirements

- **FR-1.** The system MUST store the following demographic flags per student user (sourced only from SIS import or admin entry; never self-reported): free_lunch, reduced_lunch, english_language_learner (ELL), disability_status (boolean — does not store specific disability), race_ethnicity (EDEN/CALPADS race codes: 1-Hispanic, 2-AmIndian, 3-Asian, 4-BlackAfrAm, 5-PacIsland, 6-White, 7-TwoOrMore), homeless_indicator, migrant_indicator.
- **FR-2.** Access to individual-level demographic data MUST be restricted to: `admin`, `district_admin`, `principal`, and `data_analyst` roles. Instructors MUST NOT see individual demographic flags (they may see aggregate class-level summaries — but not linked to a specific student).
- **FR-3.** All access to individual demographic records MUST be logged in the `user_audit` table with timestamp, accessor, and purpose code.
- **FR-4.** The system MUST provide aggregate demographic reports: (a) school-level breakdown by demographic category, (b) performance comparison between demographic subgroups (average grade, attendance rate, assessment pass rate), suitable for Title I plan narrative.
- **FR-5.** Demographic flags MUST be sourced from SIS (§13.7) where available; admin manual entry is the fallback. Flags MUST NOT be editable by instructors.
- **FR-6.** Free/reduced-lunch eligibility MUST NOT be visible to any user other than designated district/school officials (not parents, not students, not instructors). This is explicitly required by the Family Education Rights and Privacy Act and the National School Lunch Program confidentiality requirements (7 CFR §245.6).
- **FR-7.** The system SHOULD provide an "economic disadvantage" aggregate percentage per school (without identifying individuals) that can be used to qualify the school for Title I funding determination.

## 6. Non-Functional Requirements

- **Performance** — Demographic reports with aggregate queries over 10,000 students: < 3 s.
- **Security** — Demographic columns stored in a separate `student_demographics` table with row-level security (Postgres RLS) limiting access to authorized roles. Column-level encryption considered for highest-sensitivity fields (free/reduced lunch). Audit log on every SELECT of individual demographics. This is some of the most sensitive student data in the system.
- **Privacy & Compliance** — FERPA (20 U.S.C. §1232g): demographic data is an education record. National School Lunch Program confidentiality (7 CFR §245.6): free/reduced-lunch status is strictly confidential and must not be disclosed without parental consent. ESEA/ESSA Title I (20 U.S.C. §6301): required for federal funding eligibility determination. CALPADS demographic codes (California); EdFacts (federal). COPPA: demographic data for under-13 students. ADA: disability_status flag must not carry diagnostic detail (IEP/§12.10 handles that separately).
- **Accessibility** — Aggregate report pages must be WCAG 2.1 AA. Data tables must be proper ARIA grid (§12.1) with sortable columns.
- **Scalability** — One demographics row per student; 50,000-student district = 50,000 rows. Aggregate queries use pre-computed materialized views refreshed nightly.
- **Reliability** — Demographics are read-only once set; updates only via SIS sync or admin action. No student self-service.
- **Observability** — Audit access log is the primary observability mechanism. Alert: bulk demographic access (>100 records in 60 s) triggers security alert.
- **Maintainability** — Demographic table isolated from main `users` table via foreign key join; reducing blast radius of any data leak.
- **Internationalization** — Race/ethnicity code labels externalized; CALPADS code set is the reference for California, with extensibility for other states.
- **Backward compatibility** — New table; no existing schema affected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student's demographic record has `free_lunch = true`, *When* an instructor queries the student's profile via the API, *Then* the `student_demographics` fields are absent from the response (HTTP 200 with demographics omitted, not 403).
- **AC-2.** *Given* a district admin accesses a student's demographic record, *When* the access occurs, *Then* a row is written to `user_audit` with `action = 'demographics_view'`, `accessor_id`, and `student_id`.
- **AC-3.** *Given* a school has 200 students with `free_lunch = true` and 300 without, *When* the Title I aggregate report is generated, *Then* it shows "Economic disadvantage: 40% (200/500 students)" without identifying any individual student.
- **AC-4.** *Given* a security analyst queries the audit log for bulk access, *When* a user accesses more than 100 demographic records in 60 seconds, *Then* an alert is triggered to the security team.
- **AC-5.** *Given* a SIS sync imports 500 students with `lunch_code = 'F'` (Free), *When* the sync completes, *Then* all 500 students have `free_lunch = true` in `student_demographics` and the SIS sync log records "500 demographics updated."
- **AC-6.** *Given* a data analyst runs the performance-disaggregation report, *When* the report is requested for ELL vs. non-ELL quiz pass rates, *Then* the report shows aggregate pass rate for each subgroup with subgroup size, with any subgroup of < 10 students suppressed (n < 10 suppression for privacy).

## 8. Data Model

```sql
-- server/migrations/092_student_demographics.sql
CREATE TABLE student_demographics (
    student_id          UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    free_lunch          BOOLEAN,
    reduced_lunch       BOOLEAN,
    ell_status          BOOLEAN,
    disability_status   BOOLEAN,     -- does NOT store disability type (see §12.10 for accommodations)
    race_ethnicity_code TEXT,        -- EDEN/CALPADS codes: 1-7
    homeless_indicator  BOOLEAN,
    migrant_indicator   BOOLEAN,
    data_source         TEXT NOT NULL DEFAULT 'manual',  -- sis_sync | manual
    last_verified_at    TIMESTAMPTZ,
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- RLS policy (PostgreSQL Row Level Security)
ALTER TABLE student_demographics ENABLE ROW LEVEL SECURITY;
CREATE POLICY demo_access ON student_demographics
    USING (
        current_setting('app.user_role') IN ('admin', 'district_admin', 'principal', 'data_analyst')
    );
```

## 9. API Surface

- `GET /api/admin/students/:id/demographics` — individual demographics; restricted roles only.
- `PATCH /api/admin/students/:id/demographics` — manual update; admin/district_admin.
- `GET /api/admin/schools/:id/demographics/report` — aggregate Title I report; admin/district_admin.
- `GET /api/admin/schools/:id/demographics/disaggregated-performance` — performance by subgroup.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Free/reduced-lunch data exposed to instructors | L | H | RLS policy + application-layer authz + audit log; security penetration test |
| SIS sends incorrect lunch codes; data corrupted | M | M | Validate against known code set on import; flag anomalies in sync log |
| Demographic data used for discriminatory targeting | L | H | Limit report access to aggregate only; bias audit on §9.2 model |
| n<10 suppression not enforced, exposing small subgroups | M | H | Enforce in report generation code; unit test suppression logic |
