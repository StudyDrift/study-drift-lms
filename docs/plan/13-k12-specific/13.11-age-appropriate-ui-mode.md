## 1. Problem Statement

A 5-year-old kindergartner and a 17-year-old senior have radically different UI needs. Lextures currently presents the same dense, text-heavy, adult-oriented interface to all users. Young learners (K-2) need large text, large tap targets, icon-first navigation, minimal text menus, and friendly visual affordances. Elementary students (3-5) need a simplified but slightly more text-present interface. Secondary students can use the standard adult interface. Without age-appropriate UI modes, K-2 and 3-5 teachers will not adopt Lextures because the interface is inaccessible to their students and requires constant teacher mediation. Competing platforms (Seesaw for K-5, Google Classroom for secondary) have deeply age-differentiated interfaces.

## 2. Goals

- Implement three distinct UI modes: K-2 (ages 5-8), 3-5 (ages 8-11), and 6-12/Secondary (default adult).
- Mode is derived automatically from the student's `grade_level` (§13.6) with a teacher/admin override.
- K-2 mode: enlarged text (24px+), 48px+ tap targets, icon-first navigation, reduced menu complexity, emoji/avatar personalisation.
- 3-5 mode: slightly enlarged text (18px+), simplified navigation with icons and labels, reduced feature density.
- Secondary mode: current default interface (unchanged).
- Mode is applied only to the student-facing pages; instructor interface always uses Secondary mode.

## 3. Non-Goals

- A completely separate app for K-2 (single codebase with mode-switching).
- Age verification (uses grade level from SIS/teacher assignment, §13.6).
- Parent portal mode differentiation (parents use Secondary/adult mode).
- Custom themes or branding per age mode (§5.7 owns org branding).

## 4. Personas & User Stories

- **As a 1st grade teacher**, I want my students to see a big, colorful interface with large icons so that they can navigate to their assignments independently without needing me to click for them.
- **As a K-2 student**, I want to tap big pictures to open my activities so that I don't have to read small menu labels to find my work.
- **As a 4th grade student**, I want the interface to show words and icons together so that I understand where to go but can still read the labels.
- **As a teacher configuring a classroom**, I want to set the UI mode for my class to K-2 even if some students are age 9, because my special education class needs the simpler interface.

## 5. Functional Requirements

- **FR-1.** The system MUST automatically apply the appropriate UI mode based on the authenticated student's `grade_level` (§13.6): K-2 → K2 mode, 3-5 → Elementary mode, 6-12 → Standard mode.
- **FR-2.** Instructors and admins MUST be able to override the UI mode for a specific student via the accommodations settings or student profile, overriding the grade-level default.
- **FR-3.** K2 mode MUST implement: minimum font size 24px, minimum interactive element height 48px, icon + label navigation in the main nav, reduced module/assignment list to card view (not table), emoji reaction to completed activities.
- **FR-4.** Elementary mode MUST implement: minimum font size 18px, minimum touch target 44px, simplified navigation with icons visible alongside text labels, card-based module list.
- **FR-5.** UI mode MUST be applied before first paint via a CSS class on `<html>` (e.g., `ui-mode-k2`, `ui-mode-elementary`, `ui-mode-standard`), loaded from the user session token or preferences API.
- **FR-6.** The reading preferences panel (§12.6) MUST respect the active UI mode's minimum font size — reading preferences can only increase size above the mode minimum, not decrease below it.
- **FR-7.** All three modes MUST pass axe-core accessibility checks (WCAG 2.1 AA).

## 6. Non-Functional Requirements

- **Performance** — Mode CSS loaded in initial stylesheet; no additional JS for mode determination. Mode applied before first paint to avoid layout shift (CLS = 0).
- **Security** — UI mode is a display preference; it does not change access-control rules. Students in K2 mode still see only their authorized content.
- **Privacy & Compliance** — Grade level (source of mode) is FERPA-protected (§13.6). COPPA: K2-mode students are under 13; no additional data collected by the mode change. WCAG 2.1 AA (§12 series) must hold in all three modes. WCAG SC 1.4.4 (Text Resize): K2 base font 24px must not break layout when browser zoom is applied.
- **Accessibility** — K2 and Elementary modes must have LARGER touch targets than WCAG's minimum (44×44 px WCAG, 48×48 px K2 mode). Large text in K2 mode satisfies WCAG SC 1.4.3 large-text contrast threshold (3:1 instead of 4.5:1). All three modes pass axe-core. VoiceOver and TalkBack testing required for K2 mode (iPads are primary K-2 devices).
- **Scalability** — CSS-only mode system; no server rendering changes. Mode stored in user session.
- **Reliability** — If grade level is null (new user), default to Standard mode; no error.
- **Observability** — Metric: `ui_mode_distribution` (K2/Elementary/Standard percentage). Alert: none needed.
- **Maintainability** — Three CSS layers in `clients/web/src/styles/ui-modes/k2.css`, `elementary.css`. Tailwind utility overrides scoped to `.ui-mode-k2` parent class.
- **Internationalization** — Icon labels and navigation text fully i18n-ready.
- **Backward compatibility** — Standard mode = current interface; no regression for existing adult users.

## 7. Acceptance Criteria

- **AC-1.** *Given* a Grade 1 student logs in, *When* their profile has `grade_level = '1'`, *Then* the `<html>` element has class `ui-mode-k2` and all navigation items display icons at 48px height with text labels.
- **AC-2.** *Given* K2 mode is active, *When* a student views their course module list, *Then* modules render as cards (not rows/tables) with a large cover image and the module title in 24px+ font.
- **AC-3.** *Given* a teacher overrides a Grade 4 student's UI mode to K2 via the student profile settings, *When* that student logs in, *Then* they see K2 mode regardless of their grade level.
- **AC-4.** *Given* K2 mode is active, *When* axe-core runs on the student home page, *Then* zero violations are reported, and touch target sizes are all ≥ 48×48 px (verified by layout inspection).
- **AC-5.** *Given* a VoiceOver user on iPad navigates the K2 mode home screen, *When* they swipe through navigation items, *Then* VoiceOver reads each icon's label text correctly.
- **AC-6.** *Given* a student in K2 mode uses the reading preferences panel (§12.6) and tries to set font size smaller than the mode minimum, *When* they apply the change, *Then* the font size does not go below 24px.

## 8. Data Model

UI mode derived from `users.grade_level` (§13.6). Override stored in `user_reading_preferences`:

```sql
-- server/migrations/090_ui_mode_override.sql
ALTER TABLE user_reading_preferences
    ADD COLUMN ui_mode_override TEXT;  -- NULL | k2 | elementary | standard
```

## 9. API Surface

- `GET /api/me/reading-preferences` — extended to include `ui_mode_override` and `effective_ui_mode` (derived from grade level or override).
- `PATCH /api/admin/students/:id/ui-mode` — admin/teacher sets UI mode override.

## 10. UI / UX

- **New CSS files:** `clients/web/src/styles/ui-modes/k2.css`, `elementary.css`.
- **Modified:** `clients/web/src/main.tsx` — apply UI mode class on `<html>` at app init. `clients/web/src/components/layout/` — responsive layout adapts to mode class.
- **K2 design:** Large card navigation; colorful course illustrations; celebration animation on task completion (respects `prefers-reduced-motion`); emoji/sticker reward system (future §15.9).
- **User flows:** 1. K2 student logs in → sees big icons. 2. Teacher overrides a student's mode. 3. Grade level auto-updates from SIS sync → mode updates.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| K2 CSS overrides break existing adult UI on some pages | M | M | Strict CSS scope using `.ui-mode-k2` parent class; visual regression tests |
| Grade level not populated for existing students | H | M | Default to Standard mode; admin can bulk-set grade levels from SIS import |
| K2 large tap targets require layout redesign on complex pages | M | M | Scope K2 mode to student-facing pages only; instructor pages excluded |

## 15. Rollout Plan

- Feature flag: `ff_ui_mode` (default `false`; preview available for early-access K-2 schools).
- Sequencing: CSS layers → grade-level derivation → override API → pilot in one K-2 classroom.
- GA: All 6 ACs pass; UX testing with 5-year-old participants; axe-core clean in all modes.

## 16. Test Plan

- **Unit:** `effective_ui_mode` derivation logic (grade level → mode, override beats grade level).
- **Integration:** Mode applied before first paint; SIS sync updates mode.
- **End-to-end:** Playwright — login as Grade 1 student → verify `ui-mode-k2` class → navigate modules.
- **Accessibility:** axe-core in all three modes; VoiceOver on iPad for K2.
- **Performance:** CSS load adds < 5 KB; no CLS.
- **Manual:** UX testing with K-2 students (observed sessions with teacher present).
