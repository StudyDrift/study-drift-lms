## 1. Problem Statement

The majority of K-12 districts in the US provide students with Chromebooks and deploy web-content filtering software (GoGuardian, Securly, Cisco Umbrella, Lightspeed) to comply with the Children's Internet Protection Act (CIPA, 47 U.S.C. §254) and protect students from harmful online content. These filtering tools need to know (a) which user is on which device to apply age-appropriate policies, and (b) which learning applications are on the allowlist so they are never accidentally blocked. Lextures must make itself visible and friendly to content-filtering tools by publishing a well-known URL allowlist, implementing the GoGuardian and Securly extension APIs (where available), and providing district IT admins with documentation for manual allowlist configuration. Without this, Lextures will be randomly blocked for students behind a strict content filter, and district IT staff will receive support tickets every time a new Lextures feature loads a new CDN domain.

## 2. Goals

- Publish a documented URL allowlist for Lextures domains and CDN endpoints.
- Implement GoGuardian's and Securly's optional LMS integration hooks (activity classification, user context).
- Provide a district-configurable integration settings page for IT admins.
- Ensure Lextures URLs are never incorrectly categorized as "social" or "games" by filter classification.

## 3. Non-Goals

- Building a content filter ourselves.
- Integration with Cisco Umbrella, Lightspeed, Bark, or others in v1 (prioritize GoGuardian and Securly as the two largest).
- Monitoring or logging student browsing activity (filter vendors do that; not our role).
- Student device management (MDM scope).

## 4. Personas & User Stories

- **As a district IT director**, I want to see a published list of all Lextures URLs and CDN domains I need to allowlist so that I can add them to our GoGuardian policy in 10 minutes.
- **As a student on a school Chromebook**, I want Lextures to load without being blocked by our content filter so that I can access my coursework without IT intervention.
- **As a GoGuardian administrator**, I want Lextures to send the student's authenticated identity with each page view so that GoGuardian can apply the correct per-student policy.
- **As an IT admin**, I want to see in Lextures settings that the GoGuardian integration is active so that I know the user-context signal is being sent.

## 5. Functional Requirements

- **FR-1.** The system MUST publish a static, versioned URL allowlist document at `https://{tenant}.lextures.com/.well-known/content-filter-allowlist.json` listing all first-party domains, CDN hostnames, and API endpoints required for Lextures to function.
- **FR-2.** The system MUST include a `<meta name="lextures:user-role" content="{role}">` and `<meta name="lextures:org-id" content="{org_id}">` in the HTML `<head>` of student-facing pages, enabling content-filter browser extensions to read user context without API calls.
- **FR-3.** The system MUST support the GoGuardian Activity Classification API: when a student navigates to a Lextures page, emit an activity event with `{url, category: 'educational', title, student_id_hash}` to the GoGuardian endpoint if the district has configured a GoGuardian API key.
- **FR-4.** The system MUST support the Securly Classroom integration flag: include Lextures as a pre-approved "Educational Tool" in the Securly app catalog JSON served at the well-known endpoint.
- **FR-5.** A district admin MUST be able to configure GoGuardian API key and toggle Securly integration on/off via the admin settings page.
- **FR-6.** The allowlist document MUST be automatically updated when new CDN domains or third-party service endpoints are added to the Lextures infrastructure.

## 6. Non-Functional Requirements

- **Performance** — `/.well-known/content-filter-allowlist.json` must be served from CDN with < 50 ms TTFB; it is a static file updated on deploy.
- **Security** — GoGuardian API key stored in secrets manager (§17.17). Student ID transmitted to GoGuardian is a hash (SHA-256 of `student_id + secret_salt`), not the raw UUID. No PII in the allowlist document itself. The `<meta>` tags contain org_id and role, not name or email.
- **Privacy & Compliance** — CIPA (Children's Internet Protection Act, 47 U.S.C. §254): school districts must deploy content filtering; Lextures enables this by being allowlist-friendly. COPPA: activity events to GoGuardian for under-13 students must be disclosed in privacy policy (§10.2). FERPA: hashed student ID in GoGuardian events must be documented in the district's FERPA school official designation or DPA with GoGuardian.
- **Accessibility** — No UI changes to student-facing pages; `<meta>` tags are invisible. Admin settings page must be WCAG 2.1 AA.
- **Scalability** — Meta tags added to HTML template; zero runtime overhead. GoGuardian event emission is fire-and-forget (async, non-blocking).
- **Reliability** — If GoGuardian API is unreachable, events are dropped silently; Lextures functionality unaffected. Content filter integration is best-effort, not mission-critical.
- **Observability** — Metrics: `goguardian_events_sent_per_day`, `goguardian_api_errors`. Alert: GoGuardian error rate > 10%.
- **Maintainability** — Allowlist document generated from a config file `server/config/content-filter-allowlist.json`; updated by CI on deploy.
- **Internationalization** — Not applicable; this is technical integration.
- **Backward compatibility** — Additive only; no existing functionality changes.

## 7. Acceptance Criteria

- **AC-1.** *Given* a district IT admin opens `https://district.lextures.com/.well-known/content-filter-allowlist.json`, *When* the page loads, *Then* a JSON document is returned listing all first-party Lextures domains and required CDN hostnames.
- **AC-2.** *Given* a student is viewing a course page, *When* a GoGuardian extension reads the page's `<head>`, *Then* it finds `<meta name="lextures:user-role" content="student">` and `<meta name="lextures:org-id" content="{org_uuid}">`.
- **AC-3.** *Given* a district has configured a GoGuardian API key in admin settings, *When* a student navigates to a course module page, *Then* a GoGuardian activity event is emitted asynchronously with `category: 'educational'` and a hashed student ID.
- **AC-4.** *Given* the GoGuardian API is unreachable, *When* a student loads a page, *Then* the page loads normally with no error visible to the student; a background error is logged.
- **AC-5.** *Given* a new CDN domain is added to Lextures infrastructure, *When* the CI deploy runs, *Then* the allowlist document is automatically updated to include the new domain.
- **AC-6.** *Given* axe-core runs on the admin content-filter settings page, *When* the check executes, *Then* zero violations are reported.

## 8. Data Model

```sql
-- server/migrations/093_content_filter_integration.sql
CREATE TABLE content_filter_settings (
    org_id               UUID PRIMARY KEY REFERENCES organizations(id),
    goguardian_enabled   BOOLEAN NOT NULL DEFAULT false,
    goguardian_api_key_ref TEXT,  -- reference to secrets manager
    securly_enabled      BOOLEAN NOT NULL DEFAULT false,
    updated_at           TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 9. API Surface

- `GET /.well-known/content-filter-allowlist.json` — public; no auth.
- `GET/PATCH /api/admin/content-filter-settings` — admin only.
- GoGuardian events sent to GoGuardian's inbound API (external); no new Lextures route.

## 10. UI / UX

- **New page section:** `clients/web/src/pages/admin/ContentFilterSettings.tsx` — GoGuardian API key input, Securly toggle, allowlist download link.
- **Modified:** App HTML template (`index.html`) — add `<meta>` tags populated server-side via SSR or injected by a global React hook.
- **User flows:** 1. IT admin enables GoGuardian → enters API key → saves. 2. Admin downloads allowlist PDF/JSON for submission to filter tool.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| GoGuardian API changes without notice | M | L | Version-pin integration; monitor GoGuardian developer changelog |
| COPPA violation if student PII sent to GoGuardian without DPA | M | H | Use hashed ID only; require district to have GoGuardian DPA before enabling |
| Allowlist becomes stale as infrastructure changes | M | M | CI job auto-generates allowlist from infrastructure config on every deploy |

## 15. Rollout Plan

- Feature flag: `ff_content_filter_integration` (default `false`).
- Sequencing: Allowlist document → meta tags → GoGuardian event emission → Securly catalog → admin UI → flip flag.
- Pilot: One Chromebook district with GoGuardian; verify no blocked pages after allowlist submission.
- GA: All 6 ACs pass; IT admin walkthrough confirms allowlist covers all required domains.

## 16. Test Plan

- **Unit:** GoGuardian event payload construction; student ID hashing; allowlist JSON schema.
- **Integration:** GoGuardian API call with mock response; settings CRUD.
- **Security:** Hashed student ID not reversible; API key not in logs.
- **Accessibility:** axe-core on settings page.
- **Performance:** Allowlist served in < 50 ms from CDN.
- **Manual:** QA verifies `<meta>` tags present on course page; loads correctly behind GoGuardian test account.

## 17. Documentation & Training

- Admin docs: "Configuring GoGuardian and Securly integration" IT admin guide including allowlist document URL and setup instructions.
- Internal runbook: "Updating the content-filter allowlist" on infrastructure change.

## 18. Open Questions

1. Does GoGuardian require a formal partnership/app listing, or can any LMS use the activity API?
2. Should Lextures proactively submit to Securly's EdTech Index to be pre-approved in their filter?
3. Should the hashed student ID use a district-specific salt (to prevent cross-district correlation)?
