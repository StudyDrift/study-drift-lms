## 1. Problem Statement

Modern K-12 classroom management increasingly takes place through software. Many districts use dedicated hall-pass tools (SmartPass, e-HallPass) to track when and where students leave the classroom. LMS platforms like Schoology and Canvas have started incorporating lightweight hall-pass modules. Additionally, teachers use classroom-management "signals" (quiet cue, attention signal, timer, question submission) to orchestrate instruction without interrupting it. Lextures has none of these. While not a blocker, including these features makes Lextures a more complete daily classroom tool, increasing instructor engagement and reducing the need for third-party tools.

## 2. Goals

- Provide a digital hall pass that students can request and teachers can approve/deny from the LMS.
- Log hall-pass events (requested, approved, returned) with timestamps for school office visibility.
- Provide basic classroom-signal tools: attention timer, quiet signal, anonymous question queue.
- Keep this feature lightweight and optional; it must not clutter the core LMS experience.

## 3. Non-Goals

- Location tracking or GPS (privacy concern, especially for minors).
- Integration with third-party hall-pass systems (SmartPass, e-HallPass) in v1.
- Full classroom management suite (behavior tracking is §13.3).
- Bathroom occupancy tracking (hardware integration, out of scope).

## 4. Personas & User Stories

- **As a teacher**, I want students to request a hall pass digitally so that I can approve it without interrupting my instruction by speaking across the room.
- **As a school office assistant**, I want to see a live list of students currently out of class so that I can verify a student's authorized absence during a hall-pass period.
- **As a teacher**, I want to set a 5-minute countdown timer visible to the class on their screens so that students know how long they have to complete a warm-up activity.
- **As a student**, I want to submit an anonymous question to my teacher during a lesson so that I can ask for clarification without embarrassment.

## 5. Functional Requirements

- **FR-1.** Students MUST be able to request a hall pass from their LMS session, specifying destination (bathroom, office, library, nurse, other) and estimated duration.
- **FR-2.** Teachers MUST receive a pass request notification in their active session and be able to approve or deny with a single click; approved passes show in a "currently out" list.
- **FR-3.** Students MUST be marked as "returned" when they click "I'm back" or after the estimated time elapses (with a soft reminder to the teacher that a pass has exceeded its time).
- **FR-4.** A hall-pass log MUST record all pass events (request, approve, return) with timestamps, accessible to the teacher and school admin.
- **FR-5.** Teachers MUST be able to set a countdown timer visible on student screens, with a customizable duration (1–60 minutes) and a visual/text alert when time is up.
- **FR-6.** Students MUST be able to submit anonymous questions to the class question queue during a session; the teacher sees the queue and can address questions at their discretion.
- **FR-7.** All classroom-signal features MUST be accessible via a persistent widget available on any course page when a class session is "active" (teacher-toggled).

## 6. Non-Functional Requirements

- **Performance** — Hall pass request/approve: < 200 ms. Timer sync uses WebSocket or Server-Sent Events; drift < 1 s across all student sessions.
- **Security** — Pass requests are school-scoped; only students in the teacher's section can submit passes to that teacher. Anonymous question queue must be anonymous to classmates but not to teachers (moderate abuse potential).
- **Privacy & Compliance** — FERPA: hall-pass logs are education records (location information is sensitive for minors). COPPA: anonymous question queue must not collect PII for under-13 students. Pass data retained per district record-retention policy (not indefinitely).
- **Accessibility** — All pass/timer/question controls must be WCAG 2.1 AA; timer countdown announced to screen readers via live region at regular intervals (WCAG SC 4.1.3).
- **Scalability** — A class of 35 students; one teacher; WebSocket room per active class session. Low concurrent user count; minimal server load.
- **Reliability** — If WebSocket connection drops, students can fall back to page-reload polling for timer state.
- **Observability** — Metrics: `hall_passes_per_day`, `timer_sessions_per_day`. No detailed PII in metrics.
- **Maintainability** — Hall pass and classroom signals implemented as a lightweight `server/src/routes/classroom_signals.rs` module; not tightly coupled to course/gradebook.
- **Internationalization** — Destination options (bathroom, office, etc.) externalized for i18n.
- **Backward compatibility** — Feature is opt-in via teacher "Start class session" toggle.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student clicks "Request Hall Pass" and selects "Bathroom", *When* the teacher approves, *Then* the student's status shows "Out — Bathroom" in the teacher's "Currently Out" list with a timestamp.
- **AC-2.** *Given* a teacher sets a 5-minute countdown timer, *When* all student sessions are open for the same course, *Then* the timer shows the same remaining time (±2 s) on all student screens.
- **AC-3.** *Given* a student submits an anonymous question "Can you explain question 3 again?", *When* the teacher views the question queue, *Then* the question appears without the student's name.
- **AC-4.** *Given* a pass has been active for longer than the estimated duration, *When* the estimated time elapses, *Then* the teacher sees a soft alert: "Jordan's pass has exceeded 5 minutes."
- **AC-5.** *Given* a screen-reader user is on the course page with an active timer, *When* the timer reaches 1 minute remaining, *Then* a live region announces "1 minute remaining."
- **AC-6.** *Given* axe-core runs on the hall pass widget and classroom signals panel, *When* the check executes, *Then* zero Critical or Serious violations are reported.

## 8. Data Model

```sql
-- server/migrations/088_hall_pass.sql
CREATE TABLE hall_passes (
    id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    student_id     UUID NOT NULL REFERENCES users(id),
    section_id     UUID NOT NULL REFERENCES course_sections(id),
    destination    TEXT NOT NULL,
    estimated_mins INTEGER,
    status         TEXT NOT NULL DEFAULT 'requested',  -- requested | approved | returned | denied
    requested_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    approved_at    TIMESTAMPTZ,
    returned_at    TIMESTAMPTZ
);
CREATE TABLE anonymous_questions (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    course_id   UUID NOT NULL REFERENCES courses(id),
    session_id  UUID,  -- future: class session
    question    TEXT NOT NULL,
    upvotes     INTEGER NOT NULL DEFAULT 0,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 9. API Surface

- `POST /api/sections/:id/hall-passes` — request pass; student.
- `PATCH /api/hall-passes/:id` — approve/deny/return; teacher/student.
- `GET /api/sections/:id/hall-passes/active` — currently out list; teacher/admin.
- `POST /api/courses/:id/timer` — start timer; teacher (WebSocket broadcast).
- `POST /api/courses/:id/questions` — submit anonymous question; student.
- `GET /api/courses/:id/questions` — view queue; teacher.

## 10. UI / UX

- **New component:** `clients/web/src/components/classroom/ClassroomSignalsWidget.tsx` — floating panel, hidden until teacher activates a class session.
- **Student view:** Hall pass request button, timer display, question submit form.
- **Teacher view:** Pass approval queue, currently-out list, timer controls, question queue.
- **Accessibility:** Timer is `role="timer"` with `aria-live`; pass approval buttons have `aria-label`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Anonymous question queue abused for bullying | M | H | Teacher moderation queue; admin audit log; filter profanity |
| Timer WebSocket drift on slow connections | M | L | Client-side NTP sync with server timestamp; accept ±2 s tolerance |

## 15. Rollout Plan

- Feature flag: `ff_classroom_signals` (default `false`).
- Pilot: 3 classrooms at one school.
- GA: All 6 ACs pass; teacher usability feedback positive.

## 16. Test Plan

- **Unit:** Pass status machine transitions; anonymous question creation (no PII stored).
- **Integration:** Pass request/approve/return flow; timer WebSocket broadcast.
- **Security:** Student cannot approve their own pass; anonymous question not linked to user ID in response.
- **Accessibility:** axe-core on widget; live region for timer.
