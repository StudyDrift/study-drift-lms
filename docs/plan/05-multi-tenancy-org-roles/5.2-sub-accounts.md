## 1. Problem Statement

A K-12 district is not a flat list of courses — it is a hierarchy: district → schools → grade-levels or departments → courses. Similarly, a university is organized as university → colleges → departments → courses. Without a sub-account (or "unit") concept, a district IT director cannot delegate administration of individual schools to building principals, and a university cannot scope a department admin's visibility to their own department. This forces all administrative work through a single super-admin account and makes district-wide deployments operationally unmanageable.

## 2. Goals

- Model an `org_units` table that supports arbitrary-depth nesting within an organization.
- Allow courses and users to be assigned to a specific unit at any depth.
- Enable unit-scoped admin roles (e.g., a principal who administers one school but not others).
- Expose unit management in the admin API and a basic admin UI.
- Integrate unit scope into `authz.rs` so that role grants at a unit cascade to all descendant units.

## 3. Non-Goals

- Cross-organization unit sharing — not supported.
- More than 5 levels of nesting — deeper hierarchies defer to a recursive CTE approach post-MVP.
- SIS-driven automatic unit sync (OneRoster, Banner) — covered in 4.3 and 14.1.
- Section-level grouping within a course — covered in 5.4.

## 4. Personas & User Stories

- **As a district IT director**, I want to create schools under my district organization so that each principal only manages their own school's courses and users.
- **As a university registrar**, I want to nest departments under colleges so that department chairs see only their own section of the LMS.
- **As a principal**, I want to be an admin of my school's sub-account so that I can create courses and manage teachers without accessing other schools.
- **As an instructor**, I want my courses associated with the correct department so that department-level reports include my course data.
- **As a developer**, I want a closure table or adjacency list so that ancestor/descendant queries are efficient even for medium-depth hierarchies.

## 5. Functional Requirements

- **FR-1.** The system MUST provide an `org_units` table with `id`, `org_id`, `parent_unit_id` (nullable for root units), `name`, `unit_type` (`school`, `department`, `district`, `college`, `other`), `status`, `created_at`, `updated_at`.
- **FR-2.** The system MUST allow a course to be associated with exactly one `org_unit_id` (nullable, meaning it belongs directly to the root org).
- **FR-3.** The system MUST allow a user's role grant to be scoped to an `org_unit_id`, meaning the grant applies to that unit and all its descendants.
- **FR-4.** The system MUST enforce in `server/src/authz.rs` that a unit-scoped admin can only manage resources within their unit's subtree.
- **FR-5.** The system MUST provide REST endpoints for creating, listing, updating, and deleting org units, gated by org-admin or global-admin role.
- **FR-6.** The system SHOULD provide a `GET /api/orgs/:org_id/units/tree` endpoint returning the full nested unit hierarchy as JSON.
- **FR-7.** The system SHOULD propagate unit membership to analytics and reporting queries (9.5) so that unit-level aggregates are possible.
- **FR-8.** The system MAY store a `metadata` JSONB on each unit for district-specific fields (e.g., NCES school ID).

## 6. Non-Functional Requirements

- **Performance** — ancestor/descendant path queries must complete in ≤10 ms for hierarchies up to 500 units using a `ltree` or closure table strategy.
- **Security** — a unit-scoped admin must not be able to escalate to parent-unit or sibling-unit resources; test with privilege-escalation scenario.
- **Privacy & Compliance** — FERPA §99.31 "school official" concept maps to unit-scoped admin; unit boundary must appear in audit logs.
- **Accessibility** — unit tree UI must be keyboard-navigable (expand/collapse) and announce state changes via ARIA.
- **Scalability** — support ≥10,000 units per org without full-table scans; index on `(org_id, parent_unit_id)`.
- **Reliability** — deleting a unit with active courses must be blocked or cascade-suspended, not silently orphan rows.
- **Observability** — log `unit_id` on all scoped requests; add `unit_admin_actions` counter metric.
- **Maintainability** — unit hierarchy logic isolated in `server/src/services/org_units.rs`.
- **Internationalization** — unit `name` supports UTF-8; no locale restriction on unit types.
- **Backward compatibility** — `org_unit_id` on courses is nullable; existing courses unaffected until explicitly assigned.

## 7. Acceptance Criteria

- **AC-1.** *Given* an org admin, *When* `POST /api/orgs/:id/units` is called with `{"name":"Lincoln High","unit_type":"school"}`, *Then* the response is HTTP 201 and the unit appears under the org.
- **AC-2.** *Given* a unit "Lincoln High" with a child unit "Math Dept", *When* `GET /api/orgs/:id/units/tree` is requested, *Then* the response includes a nested structure with "Lincoln High" containing "Math Dept".
- **AC-3.** *Given* a user with a role grant scoped to "Lincoln High", *When* they call `GET /api/courses`, *Then* they see only courses assigned to "Lincoln High" or its descendants.
- **AC-4.** *Given* a unit-scoped admin, *When* they attempt to modify a course in a sibling unit, *Then* the system returns HTTP 403.
- **AC-5.** *Given* a unit with active courses, *When* a DELETE request is issued for that unit, *Then* the system returns HTTP 409 with a message listing the blocking resources.
- **AC-6.** *Given* a course is assigned to "Math Dept" (a child of "Lincoln High"), *When* a Lincoln High admin queries courses in their subtree, *Then* the Math Dept course is included in results.

## 8. Data Model

```sql
-- server/migrations/074_org_units.sql
CREATE TABLE org_units (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id         UUID NOT NULL REFERENCES organizations(id),
  parent_unit_id UUID REFERENCES org_units(id),
  name           TEXT NOT NULL,
  unit_type      TEXT NOT NULL DEFAULT 'other'
                   CHECK (unit_type IN ('district','school','college','department','other')),
  status         TEXT NOT NULL DEFAULT 'active'
                   CHECK (status IN ('active','archived')),
  metadata       JSONB NOT NULL DEFAULT '{}',
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_org_units_org_id        ON org_units(org_id);
CREATE INDEX idx_org_units_parent        ON org_units(parent_unit_id);

ALTER TABLE courses ADD COLUMN org_unit_id UUID REFERENCES org_units(id);
CREATE INDEX idx_courses_org_unit_id ON courses(org_unit_id);

-- Unit-scoped role grants extend server/migrations/008_rbac.sql
ALTER TABLE user_course_grants ADD COLUMN org_unit_id UUID REFERENCES org_units(id);
```

Backfill: `org_unit_id` is nullable; no backfill needed for existing courses.

## 9. API Surface

| Verb | Path | Auth scope | Description |
|---|---|---|---|
| POST | `/api/orgs/:id/units` | `org_admin` | Create unit |
| GET | `/api/orgs/:id/units` | `org_admin` | List flat units |
| GET | `/api/orgs/:id/units/tree` | `org_admin` | Nested tree JSON |
| PATCH | `/api/orgs/:id/units/:uid` | `org_admin` | Update unit |
| DELETE | `/api/orgs/:id/units/:uid` | `org_admin` | Delete (block if has courses) |

Request/response shapes follow org entity conventions (UUID ids, ISO timestamps).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Recursive CTE slow on deep trees | M | M | Use `ltree` extension or materialized path column; benchmark at 500 units |
| Privilege escalation via unit reparenting | M | H | Require global-admin to reparent; unit-admin cannot change own parent |
| UI tree becomes unwieldy for large districts | M | M | Virtual-scroll or paginate at 100+ siblings |
| Orphaned courses after unit deletion | M | H | FK constraint + application-level block with 409 response |

## 15. Rollout Plan

- Feature flag: `org_units_enabled` (default: `false`).
- Sequence: schema migration → API → unit tree UI → flip flag in staging → flag on production after QA.
- Dogfood: create 3 test schools in the dev org to validate tree rendering.
- GA criteria: unit-scoped admin demo passes; no regression in course list APIs.
- Rollback: `org_unit_id` nullable; disabling flag hides UI without breaking data.

## 16. Test Plan

- **Unit** — service functions: `get_subtree_ids`, `check_unit_ownership`.
- **Integration** — subtree role grant enforcement; blocking delete of unit with courses.
- **End-to-end** — Playwright: create school, assign course to school, log in as school admin, verify visibility scope.
- **Security** — sibling-unit access denied; parent-unit escalation denied.
- **Accessibility** — axe scan on tree component; keyboard traversal test.
- **Performance / load** — query benchmark with 500-unit tree.
- **Manual exploratory** — attempt re-parenting unit as a non-global-admin.
