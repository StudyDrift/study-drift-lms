## 1. Problem Statement

In higher education and large K-12 districts, a single course (e.g., "Introduction to Biology") is taught across many sections (001, 002, 003) that share syllabus and content but have separate rosters, separate gradebooks, and separate instructors. Currently, Lextures has only a course entity â€” there is no sections layer. Institutions that adopt Lextures today must create one course per section, which duplicates content and prevents any instructor from teaching the same course across multiple sections without managing separate copies. This is a direct enrollment-management and content-consistency blocker for any mid-size institution.

## 2. Goals

- Introduce a `course_sections` table that sits between a parent course and an enrollment.
- Preserve shared content (modules, assignments, syllabus) at the parent course level.
- Isolate per-section data: roster, gradebook, section instructor, meeting time/location.
- Provide APIs for section CRUD and section enrollment.
- Update the student and instructor views to show section context.

## 3. Non-Goals

- Cross-listing (merging sections into one shell) â€” covered in 5.5.
- Section-specific content overrides (blueprint-level control) â€” covered in 5.6.
- SIS-driven section sync â€” deferred to 14.1 and 4.3.
- Waitlist management â€” deferred.

## 4. Personas & User Stories

- **As a department coordinator**, I want to create three sections of "Bio 101" sharing the same modules and assignments so that students in each section see the same content with section-specific due dates.
- **As an instructor of Section 002**, I want my gradebook to show only my section's students so that I am not confused by another section's roster.
- **As a student**, I want to see which section I am enrolled in on my dashboard so that I know my meeting time and instructor.
- **As a registrar**, I want to move a student from Section 001 to Section 002 without re-enrolling them in the parent course so that their submission history is preserved.
- **As an admin**, I want to list all sections of "Bio 101" with their enrollment counts so that I can manage room assignments.

## 5. Functional Requirements

- **FR-1.** The system MUST introduce a `course_sections` table with `id`, `course_id`, `term_id`, `section_code`, `name`, `instructor_user_id`, `capacity`, `meeting_info` (JSONB), `status`, `created_at`, `updated_at`.
- **FR-2.** The system MUST migrate `enrollments` to include a nullable `section_id`; enrollments remain valid without a section for backward compatibility.
- **FR-3.** The system MUST scope the gradebook view (`CourseGradebook.tsx`) to show only students enrolled in the authenticated instructor's section(s).
- **FR-4.** The system MUST allow section-level due date overrides for assignments so that Section 001 and Section 002 can have different due dates on the same assignment.
- **FR-5.** The system MUST provide REST endpoints for section CRUD under a course (`/api/courses/:id/sections`).
- **FR-6.** The system MUST provide a section-transfer enrollment API (`PATCH /api/enrollments/:eid/section`) to move a student between sections.
- **FR-7.** The system SHOULD display the section name/code on the student's course card in the dashboard.
- **FR-8.** The system MAY support per-section meeting time displayed on the course calendar.

## 6. Non-Functional Requirements

- **Performance** â€” adding section join to gradebook query must not increase p95 query time beyond 20 ms for â‰¤500 students.
- **Security** â€” instructors MUST only see their own section's roster and grades; the section ownership check must be in `server/src/authz.rs`.
- **Privacy & Compliance** â€” FERPA: a student's section assignment is educational record; only school officials may access it.
- **Accessibility** â€” section selector must be a labeled combobox; keyboard accessible.
- **Scalability** â€” a course can have â‰¤200 sections in V1; index on `(course_id, status)`.
- **Reliability** â€” section transfer must be transactional (update enrollment + update gradebook reference atomically).
- **Observability** â€” log `section_id` on gradebook and enrollment requests.
- **Maintainability** â€” section logic in `server/src/services/sections.rs`; avoid scattering section joins across multiple route handlers.
- **Internationalization** â€” `section_code` and `name` are UTF-8 text; `meeting_info` JSONB stores locale-neutral data.
- **Backward compatibility** â€” `section_id` on enrollments nullable; existing single-section courses continue to function without change.

## 7. Acceptance Criteria

- **AC-1.** *Given* a course "Bio 101", *When* `POST /api/courses/:id/sections` is called with `{"section_code":"001","name":"Section 1","capacity":30}`, *Then* HTTP 201 is returned and the section is listed under the course.
- **AC-2.** *Given* students enrolled in Section 001 and Section 002 of "Bio 101", *When* the Section 001 instructor views the gradebook, *Then* only Section 001 students appear.
- **AC-3.** *Given* an assignment "Lab Report" with no section-specific override, *When* the due date is set at the course level, *Then* it applies to all sections.
- **AC-4.** *Given* Section 001 has a due-date override on "Lab Report", *When* a Section 001 student views the assignment, *Then* they see the overridden due date.
- **AC-5.** *Given* a student in Section 001, *When* `PATCH /api/enrollments/:eid/section` is called with `{"section_id":"<sec-002-id>"}`, *Then* the student's enrollment now points to Section 002 and their submission history is preserved.
- **AC-6.** *Given* a student dashboard, *When* the student is enrolled in "Bio 101 â€“ Section 002", *Then* the course card displays the section code "002".

## 8. Data Model

```sql
-- server/migrations/076_course_sections.sql
CREATE TABLE course_sections (
  id                 UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  course_id          UUID NOT NULL REFERENCES courses(id) ON DELETE CASCADE,
  term_id            UUID REFERENCES terms(id),
  section_code       TEXT NOT NULL,
  name               TEXT,
  instructor_user_id UUID REFERENCES users(id),
  capacity           INTEGER,
  meeting_info       JSONB NOT NULL DEFAULT '{}',
  status             TEXT NOT NULL DEFAULT 'active'
                       CHECK (status IN ('active','cancelled','archived')),
  created_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (course_id, section_code)
);

CREATE INDEX idx_course_sections_course ON course_sections(course_id);

ALTER TABLE course_enrollments ADD COLUMN section_id UUID REFERENCES course_sections(id);
CREATE INDEX idx_enrollments_section ON course_enrollments(section_id);

-- Section-level due date overrides
CREATE TABLE section_assignment_overrides (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  section_id     UUID NOT NULL REFERENCES course_sections(id) ON DELETE CASCADE,
  assignment_id  UUID NOT NULL REFERENCES module_assignments(id) ON DELETE CASCADE,
  due_at         TIMESTAMPTZ,
  available_from TIMESTAMPTZ,
  available_until TIMESTAMPTZ,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);
```
