## 1. Problem Statement

Every institution that adopts an LMS expects the platform to display its logo, colors, and URL — not the vendor's brand. Today Lextures has a single global UI with no per-tenant theming. A district IT director needs a subdomain (`greenvalley.lextures.io` or their own custom domain `lms.greenvalley.k12.us`) and the ability to upload their logo and set primary brand colors. Without this, Lextures appears unprofessional in institutional demos and is rejected at the procurement stage.

## 2. Goals

- Store per-org branding configuration (logo, primary/secondary color, favicon, custom CSS override).
- Support per-org subdomain routing (`{slug}.lextures.io`) and optional custom domain (`lms.district.edu`).
- Serve correct branding based on the incoming request hostname, before authentication.
- Allow org admins to manage their branding configuration in the admin UI.
- Apply branding to transactional emails (logo, colors) via the mail service.

## 3. Non-Goals

- White-labeling the mobile app binary — deferred to 7.1/7.4.
- Full custom CSS injection (security risk) — V1 provides a limited set of design tokens only.
- Multi-org branding for cross-listing — not supported.
- Custom email domain (SPF/DKIM for `no-reply@district.edu`) — V1 only changes the display name and logo; full custom email-from domain is separate infra work.

## 4. Personas & User Stories

- **As a district IT director**, I want users who visit `lms.greenvalley.k12.us` to see the Green Valley USD logo and colors so that students and parents recognize the platform as district-provided.
- **As an org admin**, I want to upload a logo and set a primary color via an admin settings page so that I do not need a developer to update my branding.
- **As a student**, I want to see my school's logo on the login page and in the navigation bar so that I trust the platform is my school's official tool.
- **As a developer**, I want the branding to be resolved by hostname before any React hydration so that the correct brand colors are in the initial HTML response.

## 5. Functional Requirements

- **FR-1.** The system MUST store a `org_branding` table with `org_id`, `logo_url`, `favicon_url`, `primary_color` (hex), `secondary_color` (hex), `custom_domain`, `custom_email_display_name`, `updated_at`.
- **FR-2.** The system MUST resolve the correct org (and its branding) from the incoming `Host` header on every request, supporting both `{slug}.lextures.io` subdomain routing and fully custom domain mapping.
- **FR-3.** The system MUST inject the resolved org branding as CSS variables in the HTML `<head>` (or in a `<meta>` tag read by the frontend) before React hydration.
- **FR-4.** The system MUST provide a REST API (`GET /api/orgs/:id/branding`, `PUT /api/orgs/:id/branding`) gated by `org_admin` role.
- **FR-5.** The system MUST allow org admins to upload a logo and favicon via the branding API; uploaded files are stored in object storage (8.1) and the URL is recorded.
- **FR-6.** The system SHOULD apply the org's logo and primary color to transactional email templates sent by `server/src/services/mail.rs`.
- **FR-7.** The system SHOULD validate hex color format and image MIME types (PNG, JPEG, SVG only) on upload.
- **FR-8.** The system MAY provide a preview of the branded login page before saving branding changes.

## 6. Non-Functional Requirements

- **Performance** — branding config must be cached in memory or Redis per hostname with TTL ≤60 s so that it does not add a DB round-trip to every request.
- **Security** — logo/favicon uploads must be scanned (or at least MIME-validated) before serving; custom-domain mapping must be admin-only to prevent subdomain hijacking.
- **Privacy & Compliance** — branding config is non-PII; no special handling.
- **Accessibility** — custom colors must be checked against WCAG 4.5:1 contrast ratio; admin UI warns if contrast fails.
- **Scalability** — hostname→org mapping cached; supports ≥10,000 orgs with O(1) lookup via hash map in process or Redis.
- **Reliability** — if branding lookup fails, fall back to default Lextures brand; never return 500 due to missing branding.
- **Observability** — log `org_slug` resolved from hostname on every request.
- **Maintainability** — branding resolution middleware in `server/src/http_auth.rs` or a dedicated `BrandingMiddleware`.
- **Internationalization** — branding values (logo, colors) are locale-neutral.
- **Backward compatibility** — requests to `app.lextures.io` continue to use the default brand; existing API clients unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* org "Greenvalley USD" with a configured custom domain `lms.greenvalley.k12.us`, *When* an HTTP request arrives with `Host: lms.greenvalley.k12.us`, *Then* the server returns the Greenvalley branded login page.
- **AC-2.** *Given* an org admin uploads a PNG logo via `PUT /api/orgs/:id/branding`, *When* they visit the org subdomain, *Then* the uploaded logo appears in the navigation bar.
- **AC-3.** *Given* an admin sets `primary_color` to `#003366`, *When* the frontend renders the branding, *Then* the CSS variable `--color-primary` equals `#003366` in the document.
- **AC-4.** *Given* the branding service is down or the record is missing, *When* a request arrives for any hostname, *Then* the default Lextures brand is served and no HTTP 500 is returned.
- **AC-5.** *Given* an admin sets a custom color with insufficient contrast ratio (<4.5:1 against white), *When* they save the branding, *Then* a warning is displayed: "This color may not meet WCAG AA contrast requirements."
- **AC-6.** *Given* an org with `custom_email_display_name = "Green Valley USD"`, *When* a password-reset email is sent, *Then* the email "From" display name is "Green Valley USD" and the logo appears in the email header.

## 8. Data Model

```sql
-- server/migrations/079_org_branding.sql
CREATE TABLE org_branding (
  org_id                   UUID PRIMARY KEY REFERENCES organizations(id),
  logo_url                 TEXT,
  favicon_url              TEXT,
  primary_color            TEXT DEFAULT '#4F46E5',
  secondary_color          TEXT DEFAULT '#7C3AED',
  custom_domain            TEXT UNIQUE,
  custom_email_display_name TEXT,
  updated_at               TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_org_branding_custom_domain ON org_branding(custom_domain);
```

Also: extend `organizations` table with a `subdomain` column (= `slug` by default).

## 9. API Surface

| Verb | Path | Auth scope | Description |
|---|---|---|---|
| GET | `/api/orgs/:id/branding` | `org_admin` | Fetch branding config |
| PUT | `/api/orgs/:id/branding` | `org_admin` | Update branding config |
| POST | `/api/orgs/:id/branding/logo` | `org_admin` | Upload logo (multipart) |
| POST | `/api/orgs/:id/branding/favicon` | `org_admin` | Upload favicon (multipart) |
| GET | `/api/resolve-branding` | public | Resolve branding by Host header (used by SSR/CDN) |

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Malicious SVG upload (XSS vector) | M | H | Sanitize SVG on upload; serve with `Content-Security-Policy: sandbox` |
| Custom domain DNS misconfiguration causes 404 | H | M | Provide setup guide; fallback to subdomain if custom domain unreachable |
| Insufficient color contrast breaks accessibility | H | M | Auto-check contrast; warn in UI before save |
| Branding cache stale after update | M | L | Cache TTL 60 s; admin can force-purge |

## 15. Rollout Plan

- Feature flag: `org_branding_enabled` (default: `false`).
- Sequence: DB migration → API → logo upload → middleware → UI → staging → production.
- Dogfood: internal dev org branded with "Lextures Dev" logo.
- GA criteria: branded login for custom domain works end-to-end; fallback brand tested.
- Rollback: disable flag; all styling falls back to default.

## 16. Test Plan

- **Unit** — hex color validation; contrast ratio calculator; MIME type check.
- **Integration** — hostname resolution to org; branding API CRUD.
- **End-to-end** — Playwright: upload logo, set color, visit subdomain, verify branded page.
- **Security** — SVG upload sanitization; non-admin cannot update branding.
- **Accessibility** — contrast ratio warning fires for failing colors; axe scan on branding settings page.
- **Performance / load** — branding lookup latency with cache hit ≤1 ms; cache miss ≤5 ms.
- **Manual exploratory** — visit custom domain before DNS propagates; verify graceful fallback.

## 18. Open Questions
