## 1. Problem Statement

K-12 districts mandate consistent curriculum: every section of "7th Grade Math" across all schools should have the same modules, learning outcomes, and assignments as the district-approved master course. Today, each teacher creates their course from scratch, leading to curriculum drift, missing standards alignments, and compliance gaps in districts with state-mandated content requirements. A blueprint / template system allows a curriculum director to author a master course and push updates to every child course across the district, while preserving teacher autonomy for local additions.

## 2. Goals

- Introduce a blueprint course concept: a special course that can push its content to linked child courses.
- Support initial "sync" (child course inherits all blueprint content on creation) and ongoing "push updates" (admin pushes delta changes to all children).
- Allow child courses to have local additions that are not overwritten by blueprint pushes.
- Provide an admin UI to manage blueprint associations and trigger pushes.
- Integrate with the existing `course_export_import` service where possible.

## 3. Non-Goals

- Two-way sync (child changes flowing back to blueprint) — not supported.
- File/media asset sync across blueprints — V1 syncs structural content only (modules, assignments, quizzes, outcomes); file attachments deferred.
- LTI-based publisher blueprint feeds — deferred.
- Blueprint nesting (blueprint of blueprints) — not supported in V1.

## 4. Personas & User Stories

- **As a district curriculum director**, I want to designate "District 7th Grade Math Master" as a blueprint so that when a new teacher creates their course, it starts fully populated with district-approved content.
- **As a curriculum director**, I want to push an updated standards alignment to all 45 child courses so that I do not have to log in to each course individually.
- **As a teacher with a child course**, I want to add my own local assignments without fear that the next blueprint push will delete them.
- **As an admin**, I want to see which courses are linked to which blueprints so that I can audit curriculum compliance.

## 5. Functional Requirements

- **FR-1.** The system MUST allow any course to be designated as a blueprint (`is_blueprint = true`) by an org-admin or curriculum-director role.
- **FR-2.** The system MUST allow child courses to be linked to a blueprint; a course can have at most one blueprint parent.
- **FR-3.** The system MUST support initial content sync: when a child is linked to a blueprint, all blueprint modules, assignments, and outcomes are copied to the child (as independent copies, not references).
- **FR-4.** The system MUST support push updates: an admin can push a diff (added/modified items) from the blueprint to all linked children; deleted items in the blueprint SHOULD be flagged in children but not auto-deleted.
- **FR-5.** The system MUST preserve child-only content (locally created modules/assignments) during a blueprint push.
- **FR-6.** The system MUST provide a `blueprint_sync_logs` table recording each push: who triggered it, when, how many children were affected, and any errors.
- **FR-7.** The system SHOULD provide a "locked" flag per blueprint item so that blueprint-locked items cannot be edited in the child course.
- **FR-8.** The system MAY allow the blueprint push to be scheduled (e.g., push every Sunday night) via the job queue (17.3).

## 6. Non-Functional Requirements

- **Performance** — pushing to 500 child courses must complete within 5 minutes; use background job (17.3) not a synchronous HTTP request.
- **Security** — only users with `blueprint_admin` role (a sub-role of `org_admin`) can designate blueprints or trigger pushes; child course instructors cannot modify locked blueprint items.
- **Privacy & Compliance** — blueprint content does not contain student data; no FERPA implications for the sync itself.
- **Accessibility** — blueprint indicator badge in course UI must have an accessible text alternative.
- **Scalability** — support ≥500 child courses per blueprint; async processing required.
- **Reliability** — push job must be transactional per-child (failure on one child does not abort others); failed children reported in sync log.
- **Observability** — sync job emits `blueprint_push_started`, `blueprint_push_completed`, `blueprint_push_child_error` log events; job duration metric.
- **Maintainability** — blueprint sync logic in `server/src/services/blueprint_sync.rs`; reuse `course_export_import.rs` where possible.
- **Internationalization** — all UI labels externalized; blueprint item `title`/`description` carry over as-is (no auto-translate at this stage).
- **Backward compatibility** — `is_blueprint` and `blueprint_parent_id` columns nullable/defaulting false; existing courses unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a course is designated as a blueprint, *When* a new child course is linked to it, *Then* all blueprint modules and assignments appear in the child course.
- **AC-2.** *Given* a teacher adds a local assignment to a child course, *When* the blueprint admin pushes an update, *Then* the local assignment remains untouched.
- **AC-3.** *Given* a blueprint push is triggered for 10 child courses, *When* the job completes, *Then* a `blueprint_sync_logs` entry records 10 children synced, 0 errors.
- **AC-4.** *Given* a blueprint item is marked as locked, *When* a teacher in a child course attempts to edit it, *Then* the edit controls are disabled and a tooltip explains "This item is managed by the district blueprint."
- **AC-5.** *Given* a blueprint push job is running, *When* one child course's sync fails due to a DB error, *Then* the remaining children continue processing and the failure is recorded in the sync log.
- **AC-6.** *Given* an admin views the blueprint management page, *When* they look at the child courses list, *Then* each course shows last-sync timestamp and sync status (success/error).

## 8. Data Model

```sql
-- server/migrations/078_blueprints.sql
ALTER TABLE courses
  ADD COLUMN is_blueprint         BOOLEAN NOT NULL DEFAULT false,
  ADD COLUMN blueprint_parent_id  UUID REFERENCES courses(id);

CREATE INDEX idx_courses_blueprint_parent ON courses(blueprint_parent_id);

-- Track which items originated from a blueprint (for lock enforcement)
ALTER TABLE course_structure_items
  ADD COLUMN blueprint_locked BOOLEAN NOT NULL DEFAULT false,
  ADD COLUMN blueprint_origin_id UUID; -- original item ID in blueprint course

CREATE TABLE blueprint_sync_logs (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  blueprint_id     UUID NOT NULL REFERENCES courses(id),
  triggered_by     UUID NOT NULL REFERENCES users(id),
  triggered_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  children_total   INTEGER NOT NULL DEFAULT 0,
  children_success INTEGER NOT NULL DEFAULT 0,
  children_error   INTEGER NOT NULL DEFAULT 0,
  log_detail       JSONB NOT NULL DEFAULT '[]'
);
```

## 9. API Surface

| Verb | Path | Auth scope | Description |
|---|---|---|---|
| PATCH | `/api/courses/:id` | `org_admin` | Set `is_blueprint=true` |
| POST | `/api/courses/:id/blueprint/children` | `org_admin` | Link child course to blueprint |
| DELETE | `/api/courses/:id/blueprint/children/:cid` | `org_admin` | Unlink child |
| POST | `/api/courses/:id/blueprint/push` | `blueprint_admin` | Trigger async push to all children |
| GET | `/api/courses/:id/blueprint/sync-logs` | `blueprint_admin` | List push history |

## 10. UI / UX

New tab in course settings: "Blueprint" — toggle blueprint status; list of linked child courses with last-sync status; "Push updates" button.
Blueprint push progress modal: list of child courses, real-time status via WebSocket or polling.
In child course module editor: locked items shown with a lock icon and tooltip.
Admin course list: blueprint badge on blueprint courses; "linked to blueprint" badge on child courses.

Empty state: "No child courses linked. Link existing courses to distribute this blueprint's content."
Mobile: blueprint settings accessible from course settings accordion.
Accessibility: lock icon has `aria-label="Blueprint-locked item – contact your district admin to modify"`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Blueprint push job times out on large districts | M | H | Process children in parallel batches; set job timeout to 10 min |
| Teachers circumvent lock via direct API call | M | H | Server-side lock enforcement in authz, not just UI disable |
| Circular blueprint reference (A→B→A) | L | H | Validate no cycles before linking; DB-level constraint |

## 15. Rollout Plan

- Feature flag: `blueprints_enabled` (default: `false`).
- Sequence: DB migration → services → API → UI → async push job → staging flag → production.
- Dogfood: one internal blueprint with 3 test child courses.
- GA criteria: push to 50 children completes in <30 seconds background; lock enforcement verified.
- Rollback: disable flag; schema columns nullable.

## 16. Test Plan

- **Unit** — blueprint content diff algorithm; lock enforcement function.
- **Integration** — async push job: 50 children, verify sync log; failure isolation test.
- **End-to-end** — Playwright: designate blueprint, link child, push update, verify child content.
- **Security** — teacher cannot edit locked item via API; non-admin cannot trigger push.
- **Accessibility** — lock icon accessible; blueprint badge accessible.
- **Performance / load** — push to 500 children via job queue; verify 5-min completion.
- **Manual exploratory** — local child content survives push; circular link attempt blocked.

## 17. Documentation & Training

- Help center: "District curriculum blueprints — keeping all courses aligned."

- API reference: blueprint endpoints.

## 18. Open Questions

1. Should file/media assets be included in V1 push, or strictly structural content?
2. Should blueprint lock be per-item or per-module (all items in a module are locked if module is locked)?
3. Should there be a "preview diff" before pushing, showing the changes that would be applied?
4. How should deleted blueprint items be handled in children — flag as "orphaned" or silently leave?
