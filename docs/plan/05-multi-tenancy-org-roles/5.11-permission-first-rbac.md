## 1. Problem Statement

Lextures already ships an RBAC schema — `"user".permissions`, `"user".app_roles`, `"user".rbac_role_permissions`, `"user".user_app_roles`, and the `scope:area:function:action` matcher in `server/internal/authz/authz.go` — but the code that *uses* this model is still riddled with hard-coded role-name string checks. Handlers compare `enrollment.role == "student"`, signup branches on `"Teacher"` vs `"Student"` vs `"Global Admin"`, SAML/OIDC/SCIM/Clever provisioning calls `rbac.AssignUserRoleByName(..., "Student")`, the web client gates UI on `getAccountType() === 'parent'`, and SQL `CHECK` constraints inline the list of legal role values. The consequence: an admin cannot rename "Teacher" to "Instructor", cannot define a new "Tutor" role that should be treated as staff, cannot disable the parent dashboard for a deployment that doesn't use that role, and cannot grant any baseline role the privilege of `viewAs=student` without code changes. Configurable roles in the database are an illusion as long as the application layer hard-codes the strings. This plan completes the role/permission decoupling started by 5.8–5.10 so that *every* runtime decision routes through a permission grant.

## 2. Goals

- Replace every hard-coded role-name check in `server/internal/` and `clients/web/src/` with a permission lookup against `/api/v1/me/permissions` or `rbac.UserHasPermission`.
- Introduce semantic permissions for behaviors that previously implied a role (e.g. `course:*:enrollments:view-as-student`, `course:*:gradebook:view-own-only`, `app:user:account-parent-dashboard`, `app:user:role-assign-baseline`).
- Make the set of legal `course.course_enrollments.role` and `"user".org_role_grants.role` values driven by their catalog tables rather than SQL `CHECK` constraints, so adding a role is a data change, not a schema migration.
- Keep the existing four baseline roles (`Student`, `Teacher`, `TA`, `Global Admin`, `Org Admin`, `Org Unit Admin`, `Org Viewer`, `Parent`) as *seed data only* — rename-safe, deletable, and authoritative only because they are the grantees of well-known permission strings.
- Provide a documented "role-to-permission" provisioning hook so SSO/SCIM/OneRoster/Clever map external roles into permissions (via a configurable baseline role assignment), not by string comparison in Go.
- Achieve a clean grep: `rg -n '"(student|teacher|instructor|ta|owner|designer|observer|auditor|librarian|parent)"' server/internal/` returns zero matches outside catalog seeds, migrations, and tests of the catalog itself.

## 3. Non-Goals

- Custom org-defined roles beyond the baseline set — deferred to 5.12.
- A visual permission-matrix editor (a usable JSON editor at `Settings → Roles & Permissions` exists today via `global:app:rbac:manage`); UX polish is out of scope.
- Renaming SQL columns called `role` (e.g. `course_enrollments.role`) — the column name stays; only the *semantics* of the value change (from enum-like to catalog-driven FK).
- Removing the `account_type` column on `"user".users` (`standard` / `parent`) — that distinction is structural (parents have no org-internal identity), not a role; we only stop branching UI on the literal `'parent'` string outside the auth bootstrap.
- Changing the `scope:area:function:action` matcher itself (5.8–5.9 already validated it).

## 4. Personas & User Stories

- **As a district IT director**, I want to rename "Teacher" to "Instructor" (or any locally-preferred term) without breaking the syllabus, gradebook, or enrollment flows, so that the platform matches my district's HR taxonomy.
- **As a deployment engineer at a university**, I want to grant `app:course:view-as-student` to the existing "Auditor" role so that program reviewers can use the student-view toggle without me forking the codebase.
- **As an integrations engineer**, I want a single config table that maps `OneRoster.role` and `Clever.role` strings to baseline role grants so that I can add a new vendor without writing Go.
- **As a security reviewer**, I want every authz decision in handlers and repos to be expressed as a permission check so that I can audit the privilege model from one place (`SELECT permission_string FROM permissions`) rather than `grep`ing for string literals.
- **As a frontend engineer**, I want to gate the parent-dashboard sidebar entry on a permission returned from `/me/permissions` so that I don't ship a third UI codepath for "what kind of user is this?".
- **As a Designer (course-level)**, I want my role's permission set to be editable by the org admin without a code deploy so that we can experiment with letting designers see ungraded submissions.

## 5. Functional Requirements

- **FR-1.** The system MUST emit new permissions covering every semantic that is currently expressed as a role-name string. Initial catalog (non-exhaustive — finalised in §8):
  - `course:*:enrollments:role-staff` — held by anyone whose course-enrollment role grants staff-equivalent course access (replaces `r == "teacher" || r == "instructor" || r == "owner"` in `enrollment.UserIsCourseStaff`).
  - `course:*:enrollments:role-student` — held by anyone whose enrollment role is the student-equivalent (replaces `UserHasEnrollmentRole(..., "student")` in `course_syllabus.go`, `my_grades.go`, `meperm.go`).
  - `course:*:enrollments:assign-staff` — required to set a target enrollment to any staff role (replaces inline `role == "teacher" || role == "instructor"` in `course_enrollments_http.go`).
  - `course:*:gradebook:view-own-only` — replaces the implicit "students see only their own grades" branch in `course_syllabus.go` and `my_grades.go`.
  - `app:user:account-parent-dashboard` — gates parent-dashboard redirect and the "Children" nav item (replaces `getAccountType() === 'parent'` in `side-nav-main-links.tsx`, `post-auth-redirect.ts`, `dashboard.tsx`).
  - `tenant:org:roles:bootstrap` — required to assign the very first `Global Admin` during signup (replaces the `BootstrapAdminEmail` magic check in `credentials.go`).
  - `app:user:role-assign-baseline` — required by SAML/OIDC/SCIM/Clever provisioning code paths before they may attach any baseline role to a freshly-created user.
- **FR-2.** The system MUST stop calling `rbac.AssignUserRoleByName` with hard-coded role names from non-bootstrap code paths. Provisioning callers (SAML, OIDC, SCIM, OneRoster, Clever) MUST instead resolve a baseline role via a new `provisioning_role_map` table keyed by `(provider, external_role)` → `app_roles.name`.
- **FR-3.** The system MUST replace the `CHECK (role IN (...))` constraints on `course.course_enrollments.role` and `"user".org_role_grants.role` with FK constraints to `course.enrollment_roles(role_key)` and a new `"user".org_role_keys(role_key)` catalog table, respectively. Adding a role becomes `INSERT INTO course.enrollment_roles`, not a migration to the CHECK constraint.
- **FR-4.** The system MUST extend the role catalogs with capability columns rather than encoding capability in the role string:
  - `course.enrollment_roles.is_staff BOOLEAN NOT NULL DEFAULT false`
  - `course.enrollment_roles.is_student_equivalent BOOLEAN NOT NULL DEFAULT false`
  - Helpers `UserIsCourseStaff` and `UserHasEnrollmentRole(role="student")` MUST consult these columns, not literal role strings.
- **FR-5.** The system MUST update `MyPermissions` (`server/internal/service/meperm/meperm.go`) so the response includes the FR-1 permissions whenever the underlying enrollment / role state warrants them. The handler MUST NOT branch on the string `"student"` / `"teacher"` — it MUST look up the viewer's role keys, join against the catalog, and emit the corresponding permission strings.
- **FR-6.** The frontend MUST consume only the `/me/permissions` set for role-derived UI decisions. Helpers `getAccountType()`, `isStudentEnrollment()`, and any literal `'parent' | 'student' | 'teacher'` comparisons MUST be replaced by `hasPermission('app:user:account-parent-dashboard')` etc. through the existing `usePermissions` hook (or its addition where missing).
- **FR-7.** The system MUST keep the `viewAs` query parameter (`?viewAs=student`) as a UX shortcut but MUST resolve it server-side to "does this caller hold `course:<code>:enrollments:role-student`" rather than the string compare in `parseViewAs`.
- **FR-8.** The system MUST seed permission grants in a single idempotent migration so that existing deployments come out the other side with no functional regression: every baseline role keeps the powers it has today.
- **FR-9.** The system SHOULD expose a developer-facing audit script (`server/internal/scripts/rbac_audit.go`) that fails CI when a hard-coded role-name string is reintroduced into `server/internal/**.go` or `clients/web/src/**.{ts,tsx}` outside the allowed exception list (catalog seeds, migrations, tests of the catalog).
- **FR-10.** The system SHOULD emit an audit-log event whenever a provisioning role map entry is added, removed, or changed (10.11 admin audit log).
- **FR-11.** The system MAY support per-org overrides of capability bits (e.g. one org wants `Designer` to be staff-equivalent for gradebook access). Out of scope for v1; the schema MUST allow it without a future migration (extra `org_id` FK nullable column reserved).

## 6. Non-Functional Requirements

- **Performance** — Role-key → permission expansion happens once per request inside the existing `MyPermissions` path; total added DB latency ≤ 1 ms (two indexed catalog reads, both under 30 rows). No N+1 queries on enrollment lookups.
- **Security** — Every authz decision routes through `authz.PermissionMatches` / `rbac.UserHasPermission`. The privilege-escalation matrix in 5.8 MUST be extended to cover the new permissions; tests MUST prove no path lets a non-staff enrollment receive `course:*:enrollments:role-staff`.
- **Privacy & Compliance** — No new PII surfaces. Catalog tables are organisation-agnostic until FR-11 lands.
- **Accessibility** — UI gating now driven by permissions; copy strings already live in i18n bundles.
- **Scalability** — Catalog tables are tiny (≤ 50 rows in practice). No partitioning needed.
- **Reliability** — Migration is idempotent and reversible (drop FK, restore CHECK constraint); seed inserts use `ON CONFLICT DO NOTHING`.
- **Observability** — Add a counter `rbac_role_string_compare_total` that the FR-9 audit script can also import from runtime to confirm zero hits in production (alert if > 0).
- **Maintainability** — Single source of truth for "what does this role mean" is the catalog tables; adding a role is a data change.
- **Internationalization** — `enrollment_roles.display_name` already exists and is the i18n key surface; no new strings hard-coded in Go.
- **Backward compatibility** — All existing permission strings remain. Existing course-grant rows continue to function. Renaming `Teacher` → `Instructor` in the catalog MUST NOT break any handler.

## 7. Acceptance Criteria

- **AC-1.** *Given* the migration has run, *When* an org admin renames the `Teacher` row in `"user".app_roles` to `Instructor` via the existing RBAC settings UI, *Then* signup flow, syllabus access, gradebook visibility, and `/me/permissions` continue to work for users granted that role — verified by a Playwright spec.
- **AC-2.** *Given* a user with an enrollment whose `role_key` has `is_staff = true` but is named `tutor` (not in any current `if r == "teacher"` branch), *When* they call `GET /api/v1/courses/{code}/gradebook`, *Then* HTTP 200 is returned — confirming the catalog-driven lookup replaced the string check.
- **AC-3.** *Given* a Clever provisioning callback delivers `role: "guardian"`, *When* the provisioning_role_map has an entry `('clever','guardian','Parent')`, *Then* the user is created with the `Parent` baseline role and `account_type = 'parent'`; no `"guardian"` literal appears in Go.
- **AC-4.** *Given* a Playwright test in `clients/web/e2e` toggles a parent user's `app:user:account-parent-dashboard` permission off via the RBAC settings UI, *When* the user reloads, *Then* the "Children" sidebar entry disappears and the parent dashboard redirect no longer fires.
- **AC-5.** *Given* the FR-9 audit script runs in CI, *When* a developer adds `if role == "student"` to a `.go` file outside the allow-list, *Then* CI fails with a clear pointer to this plan.
- **AC-6.** *Given* a non-admin user, *When* they attempt to mutate `provisioning_role_map` via the admin API, *Then* HTTP 403 is returned and an audit-log row is written.
- **AC-7.** *Given* the migration is rolled back (reverse SQL applied), *When* the existing test suite is run, *Then* every test still passes — proving the new layer is additive, not destructive.

## 8. Data Model

```sql
-- server/migrations/180_permission_first_rbac.sql

-- Catalog-driven org role keys (mirror of course.enrollment_roles for org grants).
CREATE TABLE "user".org_role_keys (
    role_key      TEXT PRIMARY KEY,
    display_name  TEXT NOT NULL,
    is_admin      BOOLEAN NOT NULL DEFAULT false,
    is_viewer     BOOLEAN NOT NULL DEFAULT false,
    sort_order    INTEGER NOT NULL DEFAULT 100
);

INSERT INTO "user".org_role_keys (role_key, display_name, is_admin, is_viewer, sort_order)
VALUES ('org_admin',      'Org Admin',      true,  true,  0),
       ('org_unit_admin', 'Org Unit Admin', true,  true,  10),
       ('org_viewer',     'Org Viewer',     false, true,  20);

-- Drop the legacy CHECK; replace with FK to the catalog.
ALTER TABLE "user".org_role_grants
    DROP CONSTRAINT IF EXISTS org_role_grants_role_check;
ALTER TABLE "user".org_role_grants
    ADD CONSTRAINT org_role_grants_role_fkey
    FOREIGN KEY (role) REFERENCES "user".org_role_keys(role_key);

-- Capability bits on the course enrollment-role catalog.
ALTER TABLE course.enrollment_roles
    ADD COLUMN IF NOT EXISTS is_staff              BOOLEAN NOT NULL DEFAULT false,
    ADD COLUMN IF NOT EXISTS is_student_equivalent BOOLEAN NOT NULL DEFAULT false,
    ADD COLUMN IF NOT EXISTS can_grade             BOOLEAN NOT NULL DEFAULT false,
    ADD COLUMN IF NOT EXISTS can_author_content    BOOLEAN NOT NULL DEFAULT false;

UPDATE course.enrollment_roles SET is_staff = true,              can_grade = true, can_author_content = true WHERE role_key IN ('owner','teacher','instructor');
UPDATE course.enrollment_roles SET is_staff = true,              can_grade = true                            WHERE role_key = 'ta';
UPDATE course.enrollment_roles SET is_staff = true,              can_author_content = true                   WHERE role_key = 'designer';
UPDATE course.enrollment_roles SET is_staff = true                                                           WHERE role_key IN ('observer','auditor','librarian');
UPDATE course.enrollment_roles SET is_student_equivalent = true                                              WHERE role_key = 'student';

ALTER TABLE course.course_enrollments
    DROP CONSTRAINT IF EXISTS course_enrollments_role_check;
ALTER TABLE course.course_enrollments
    ADD CONSTRAINT course_enrollments_role_fkey
    FOREIGN KEY (role) REFERENCES course.enrollment_roles(role_key);

-- Provisioning role map: external_provider+external_role → app_roles.name.
CREATE TABLE "user".provisioning_role_map (
    id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    provider          TEXT NOT NULL CHECK (provider IN ('saml','oidc','scim','oneroster','clever','classlink')),
    external_role     TEXT NOT NULL,
    app_role_id       UUID NOT NULL REFERENCES "user".app_roles(id) ON DELETE RESTRICT,
    account_type      TEXT NOT NULL DEFAULT 'standard' CHECK (account_type IN ('standard','parent')),
    created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (provider, lower(external_role))
);

-- Seed provisioning map from current hard-coded behaviour so deployments don't change semantics.
INSERT INTO "user".provisioning_role_map (provider, external_role, app_role_id, account_type)
SELECT 'oneroster', 'teacher',  r.id, 'standard' FROM "user".app_roles r WHERE r.name = 'Teacher'
UNION ALL SELECT 'oneroster', 'student',  r.id, 'standard' FROM "user".app_roles r WHERE r.name = 'Student'
UNION ALL SELECT 'oneroster', 'guardian', r.id, 'parent'   FROM "user".app_roles r WHERE r.name = 'Parent'
UNION ALL SELECT 'clever',    'teacher',  r.id, 'standard' FROM "user".app_roles r WHERE r.name = 'Teacher'
UNION ALL SELECT 'clever',    'student',  r.id, 'standard' FROM "user".app_roles r WHERE r.name = 'Student'
UNION ALL SELECT 'scim',      'teacher',  r.id, 'standard' FROM "user".app_roles r WHERE r.name = 'Teacher'
UNION ALL SELECT 'scim',      'student',  r.id, 'standard' FROM "user".app_roles r WHERE r.name = 'Student'
UNION ALL SELECT 'saml',      'teacher',  r.id, 'standard' FROM "user".app_roles r WHERE r.name = 'Teacher'
UNION ALL SELECT 'saml',      'student',  r.id, 'standard' FROM "user".app_roles r WHERE r.name = 'Student';

-- New permission strings (FR-1) — described, no role bindings yet.
INSERT INTO "user".permissions (permission_string, description) VALUES
  ('course:*:enrollments:role-staff',              'Caller holds a staff-equivalent enrollment in the course.'),
  ('course:*:enrollments:role-student',            'Caller holds a student-equivalent enrollment in the course.'),
  ('course:*:enrollments:assign-staff',            'May assign other users to staff-equivalent course roles.'),
  ('course:*:gradebook:view-own-only',             'Caller may view their own grades only (vs. full gradebook).'),
  ('app:user:account-parent-dashboard',            'Caller should see the parent dashboard and "Children" nav.'),
  ('app:user:role-assign-baseline',                'Service-side: provisioning code may attach a baseline role.'),
  ('tenant:org:roles:bootstrap',                   'May bootstrap the very first Global Admin for a fresh org.')
ON CONFLICT (permission_string) DO NOTHING;

-- Seed bindings: keep every baseline role's behaviour identical to today.
INSERT INTO "user".rbac_role_permissions (role_id, permission_id)
SELECT r.id, p.id
FROM "user".app_roles r JOIN "user".permissions p ON true
WHERE (r.name = 'Teacher' AND p.permission_string IN ('course:*:enrollments:assign-staff'))
   OR (r.name = 'Student' AND p.permission_string = 'course:*:gradebook:view-own-only')
   OR (r.name = 'Parent'  AND p.permission_string = 'app:user:account-parent-dashboard')
   OR (r.name = 'Global Admin' AND p.permission_string IN ('tenant:org:roles:bootstrap','app:user:role-assign-baseline'))
ON CONFLICT (role_id, permission_id) DO NOTHING;
```

Indexes:

```sql
CREATE INDEX IF NOT EXISTS idx_provisioning_role_map_lookup
    ON "user".provisioning_role_map (provider, lower(external_role));
```

Backfill: not required — capability bits seeded inline above; the catalog already contains every role currently referenced in code.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| A baseline-role rename in production breaks a forgotten string compare. | M | H | FR-9 audit script blocks reintroduction; ship the script *before* opening role-name editing in the admin UI. |
| `CHECK` → `FK` swap fails on a deployment that already contains an orphaned `role` value. | L | M | Pre-flight `SELECT DISTINCT role FROM course.course_enrollments WHERE role NOT IN (SELECT role_key FROM course.enrollment_roles)`; abort migration on non-empty result. |
| Provisioning_role_map entry is missing for a brand-new vendor and users are created with no role. | M | M | Provisioning code MUST fall back to `app_settings.default_provisioning_role_id`; alert when the fallback fires. |
| Capability bit mis-configuration grants a Student staff access. | L | H | Authz matrix test covers every `(role_key, permission)` pair after the migration. |
| Increased complexity in role lookups slows hot paths. | L | M | Capability bits are pre-joined in `MyPermissions`; benchmark before/after under 1k req/s. |

## 15. Rollout Plan

- Feature flag: `rbac_permission_first` (default: `false`).
- Sequence:
  1. Ship migration 180 (additive — capability bits + catalog FK swap + provisioning map + new permissions).
  2. Land server changes behind the flag; old code paths remain as fallback.
  3. Land client changes behind the same flag (the flag is exposed in `/me`).
  4. Run flag on staging for one week; compare `/me/permissions` outputs before/after for a sample of every role.
  5. Flip flag to `true` per-tenant; finally globally.
  6. Two releases later, delete the old code paths and the flag.
- Dogfood: internal Lextures org runs flag-on for two weeks.
- GA criteria: zero diff in `/me/permissions` between flag states for the seed catalog; FR-9 audit script green on `main`.
- Rollback: flip flag off; the new tables remain inert.

## 18. Open Questions

1. Should the `account_type = 'parent'` column on `"user".users` be folded into the role/permission model entirely, or stay as a structural flag for "this user has no org-internal identity"? Current proposal keeps it; revisit in 5.12.
2. Should provisioning-role-map entries be org-scoped from day one, or only globally? Current proposal: global; org scoping deferred to FR-11.
3. For `viewAs`, do we keep the string contract (`?viewAs=student`) forever, or migrate clients to a permission-string-based parameter? Current proposal: keep, since it's an external surface.
4. Do we need a per-permission *deny* row (negative grant) for cases like "Designer must NOT see grades even though they're staff", or is omitting the grant sufficient? Current proposal: omission is sufficient (additive model only).
