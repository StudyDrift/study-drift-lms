## 1. Problem Statement

K-12 districts are legally required under FERPA to provide parents with access to their minor child's educational records. Lextures has no parent account type, no child-linking mechanism, and no parent-facing view. A district cannot adopt Lextures as its primary LMS if parents cannot monitor their child's grades, assignments, and attendance. This feature introduces the `parent` role with read-only visibility into one or more linked students' course activity, grades, and teacher communications.

## 2. Goals

- Introduce a `parent` user type who can link to one or more student accounts (multi-child support).
- Provide a read-only parent view of each linked child's grades, assignments, and course content.
- Allow parents to communicate with teachers via the existing inbox/messaging system.
- Ensure strict data isolation: a parent sees only their linked child's data, never other students'.
- Support admin-managed linking (via CSV or OneRoster) as well as parent-initiated link requests.

## 3. Non-Goals

- Parent grade-change requests or submission re-attempts â€” parents are read-only.
- Parent attendance submission â€” handled in 13.2.
- Parent-teacher conference scheduling â€” handled in 13.12.
- Full Parent Portal dashboard with analytics â€” that is scope for 13.1; this plan covers the role, linking, and basic view.
- Unauthenticated parent access â€” parents must have a verified account.

## 4. Personas & User Stories

- **As a parent of two students**, I want to switch between my children's views so that I can monitor both Sarah's and Marcus's progress from one account.
- **As a parent**, I want to see Marcus's current grades and upcoming assignments so that I can help him prepare without asking the teacher every week.
- **As a parent**, I want to message Marcus's math teacher directly through the platform so that I do not have to use email.
- **As a district admin**, I want to bulk-import parent-student links from our SIS CSV export so that I do not manually link 10,000 families.
- **As a parent**, I want to receive notifications when a grade is posted or an assignment is past due so that I can follow up promptly.

## 5. Functional Requirements

- **FR-1.** The system MUST introduce a `parent_student_links` table: `id`, `parent_user_id`, `student_user_id`, `org_id`, `relationship` (`parent`, `guardian`, `other`), `status` (`active`, `pending`, `revoked`), `linked_by`, `linked_at`.
- **FR-2.** The system MUST restrict a parent user's data access to only the linked student's courses, grades, assignments, and feed entries â€” no other students' data may be returned.
- **FR-3.** The system MUST provide an admin API for bulk-linking parents to students (`POST /api/orgs/:id/parent-links/bulk`).
- **FR-4.** The system MUST provide a parent account creation flow (email invite from admin or self-registration with student ID verification) in the auth flow (`server/src/services/auth.rs`).
- **FR-5.** The system MUST provide a multi-child switcher UI so a parent with multiple linked children can toggle between child views.
- **FR-6.** The system MUST allow parents to send messages to their linked child's course instructors via the existing communication inbox (`server/src/routes/communication.rs`).
- **FR-7.** The system SHOULD display a "Parent view" read-only banner to make clear the parent cannot interact with course content (no submission, no quiz, no grade change).
- **FR-8.** The system SHOULD send parents configurable notifications (email or push) when a grade is posted or an assignment becomes past-due for their child.

## 6. Non-Functional Requirements

- **Performance** â€” parent data queries are read-only; child-scoped queries must execute in â‰¤50 ms p95.
- **Security** â€” a parent MUST never be able to access data for a student they are not linked to, even via direct URL manipulation; `parent_student_links` checked on every parent request in `authz.rs`. Parent-child links are sensitive; linking/unlinking requires admin or verified invitation flow.
- **Privacy & Compliance** â€” FERPA requires parents of under-18 students to have access to educational records; this feature implements the access mechanism. COPPA: parent accounts for under-13 students must go through the school-managed path only (no self-registration). Parent-visible data must not include other students' names or work.
- **Accessibility** â€” multi-child switcher must be keyboard-navigable; read-only banner must be announced by screen readers.
- **Scalability** â€” an org may have up to 2Ã— students in parent accounts; parent-student link table indexed by both user IDs.
- **Reliability** â€” revoking a link must take effect immediately (not cache-dependent).
- **Observability** â€” log `parent_user_id` and `student_user_id` on all parent-scoped requests; alert on unusual access volume.
- **Maintainability** â€” parent authorization guard in `server/src/authz.rs`; parent view routes in `server/src/routes/courses.rs` with parent-scoped query variant.
- **Internationalization** â€” UI labels externalized; relationship types displayed in user's locale.
- **Backward compatibility** â€” new `parent` account type does not interfere with existing student/instructor accounts; no schema changes to existing tables except adding `parent_student_links`.

## 7. Acceptance Criteria

- **AC-1.** *Given* a parent user linked to student Alice, *When* the parent calls `GET /api/parent/students/:alice_id/grades`, *Then* Alice's grades are returned.
- **AC-2.** *Given* a parent user linked only to Alice, *When* they call `GET /api/parent/students/:bob_id/grades` (Bob is not their child), *Then* HTTP 403 is returned.
- **AC-3.** *Given* a parent with two linked children Alice and Bob, *When* they open the dashboard, *Then* a child switcher shows both names and switching to Alice shows only Alice's courses.
- **AC-4.** *Given* a parent, *When* they attempt to submit an assignment on behalf of Alice via `POST /api/submissions`, *Then* HTTP 403 is returned.
- **AC-5.** *Given* an admin bulk-imports a parent-student CSV with 100 rows, *When* the import completes, *Then* 100 `parent_student_links` rows are created with status `active`.
- **AC-6.** *Given* a parent is linked to a student, *When* a grade is posted on an assignment, *Then* the parent receives an email notification (if enabled) within 5 minutes.

## 8. Data Model

```sql
-- server/migrations/082_parent_student_links.sql
CREATE TABLE parent_student_links (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id           UUID NOT NULL REFERENCES organizations(id),
  parent_user_id   UUID NOT NULL REFERENCES users(id),
  student_user_id  UUID NOT NULL REFERENCES users(id),
  relationship     TEXT NOT NULL DEFAULT 'parent'
                     CHECK (relationship IN ('parent','guardian','other')),
  status           TEXT NOT NULL DEFAULT 'active'
                     CHECK (status IN ('active','pending','revoked')),
  linked_by        UUID REFERENCES users(id),
  linked_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (parent_user_id, student_user_id)
);

CREATE INDEX idx_psl_parent  ON parent_student_links(parent_user_id);
CREATE INDEX idx_psl_student ON parent_student_links(student_user_id);

ALTER TABLE users ADD COLUMN account_type TEXT NOT NULL DEFAULT 'standard'
  CHECK (account_type IN ('standard','parent'));
```
