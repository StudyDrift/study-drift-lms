## 1. Problem Statement

Universities routinely offer the same course under two or more subject codes (e.g., HIST 301 and WMST 301 are the same class meeting together). Without cross-listing support, instructors must maintain two separate gradebooks and two separate content copies or resort to awkward manual workarounds. Cross-listing allows two or more course-section "shells" to share a single combined roster and a single unified gradebook while preserving their distinct catalog identities.

## 2. Goals

- Allow an admin to designate two or more course sections as cross-listed, merging their rosters and gradebook into a single combined view.
- Preserve each section's catalog identity (course code, department) while sharing one content workspace.
- Provide clear UI indication to students which cross-listed shell they are registered under.
- Expose the cross-list relationship in admin reporting.

## 3. Non-Goals

- Cross-org cross-listing — deferred; cross-listing is within a single org.
- Automatic SIS-driven cross-list sync — deferred to 14.1.
- Blueprint content propagation between cross-listed shells — covered in 5.6.
- Cross-term cross-listing (two sections in different terms sharing content) — not supported in V1.

## 4. Personas & User Stories

- **As a registrar**, I want to cross-list HIST 301 and WMST 301 so that the instructor teaches one class with a combined roster.
- **As an instructor**, I want a single gradebook showing all cross-listed students, regardless of which shell they enrolled through.
- **As a student enrolled in WMST 301**, I want to see content and announcements from the combined shell without knowing the internal plumbing.
- **As an admin**, I want to view which courses are cross-listed so that I can manage room capacity for the combined enrollment count.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a `cross_list_groups` table linking multiple `course_sections` into one logical teaching shell.
- **FR-2.** The system MUST designate one section as the "primary" shell within a cross-list group; all content authoring occurs on the primary.
- **FR-3.** The system MUST merge the rosters of all cross-listed sections for the combined gradebook view.
- **FR-4.** The system MUST allow the instructor to filter the gradebook by originating section within the cross-list group.
- **FR-5.** The system MUST prevent the same section from being in more than one cross-list group simultaneously.
- **FR-6.** The system SHOULD surface the combined enrollment count to the instructor and to admin reports.
- **FR-7.** The system MAY display a badge on the student's course card indicating the cross-listed partner code.

## 6. Non-Functional Requirements

- **Performance** — merged roster query must return in ≤30 ms for ≤500 combined students.
- **Security** — only org-admin or the primary section's instructor can manage cross-list group membership; secondary section instructors are read-only.
- **Privacy & Compliance** — students from different shells appearing in one gradebook is only permissible if both shells are within the same org (FERPA boundary).
- **Accessibility** — cross-list indicator in gradebook must be accessible to screen readers.
- **Scalability** — V1 limit of 10 sections per cross-list group.
- **Reliability** — removing a section from a cross-list group must not delete any grade data.
- **Observability** — log `cross_list_group_id` on gradebook requests.
- **Maintainability** — cross-list logic in `server/src/services/cross_listing.rs`.
- **Internationalization** — UI labels externalized.
- **Backward compatibility** — cross-listing is purely additive; existing sections unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* sections HIST-301-001 and WMST-301-001, *When* an admin creates a cross-list group with HIST-301-001 as primary, *Then* both sections appear linked in `GET /api/orgs/:id/cross-list-groups`.
- **AC-2.** *Given* a cross-list group, *When* the instructor views the gradebook, *Then* students from all member sections are shown in one table.
- **AC-3.** *Given* a cross-list group, *When* the instructor applies a section filter, *Then* only the selected section's students are visible.
- **AC-4.** *Given* section HIST-301-001 already in a cross-list group, *When* an admin tries to add it to a second group, *Then* HTTP 409 is returned.
- **AC-5.** *Given* a cross-listed group, *When* WMST-301-001 is removed from the group, *Then* its students' grades remain intact and the section operates as standalone.

## 8. Data Model

```sql
-- server/migrations/077_cross_listing.sql
CREATE TABLE cross_list_groups (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id     UUID NOT NULL REFERENCES organizations(id),
  name       TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE cross_list_members (
  group_id   UUID NOT NULL REFERENCES cross_list_groups(id) ON DELETE CASCADE,
  section_id UUID NOT NULL REFERENCES course_sections(id),
  is_primary BOOLEAN NOT NULL DEFAULT false,
  joined_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (group_id, section_id),
  UNIQUE (section_id) -- section can be in at most one group
);
```

## 9. API Surface

| Verb | Path | Auth scope | Description |
|---|---|---|---|
| POST | `/api/orgs/:id/cross-list-groups` | `org_admin` | Create cross-list group |
| POST | `/api/orgs/:id/cross-list-groups/:gid/members` | `org_admin` | Add section to group |
| DELETE | `/api/orgs/:id/cross-list-groups/:gid/members/:sid` | `org_admin` | Remove section from group |
| GET | `/api/orgs/:id/cross-list-groups` | `org_admin` | List groups |
| GET | `/api/courses/:id/gradebook?cross_list=true` | `instructor` | Merged cross-list gradebook |

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Cross-list group gradebook query is slow for large cohorts | M | M | Benchmark with 500-student merge; add covering index |
| Admin accidentally adds wrong section to group | M | M | Confirmation dialog with section details; undo within 24h |
| Grade visibility across departmental lines (FERPA edge) | L | H | Enforce same-org constraint; document in privacy policy |

## 15. Rollout Plan

- Feature flag: `cross_listing_enabled` (default: `false`).
- Sequence: DB migration → API → UI → staging flag → production flag.
- GA criteria: merged gradebook passes 500-student benchmark; section filter works.
- Rollback: disable flag; cross-list tables are additive.

## 16. Test Plan

- **Unit** — cross-list group membership uniqueness enforcement.
- **Integration** — merged roster query; section filter within combined view.
- **End-to-end** — Playwright: admin cross-lists two sections, instructor views merged gradebook.
- **Security** — secondary instructor cannot modify cross-list group.
- **Accessibility** — gradebook toggle accessible.
- **Performance / load** — 500-student merge query ≤30 ms.
- **Manual exploratory** — remove section from group, verify no grade data lost.
