## 1. Problem Statement

The CLI skeleton (`clients/cli/`) exists with a root command that only prints "Hello from Lextures CLI!" but has no configuration handling, flag parsing, or global plumbing. Power users, developers, and automation pipelines need a first-class CLI tool to script and integrate Lextures workflows. Without a proper foundation — config file loading, server URL resolution, API key injection, and structured output modes — no subsequent CLI command can be built reliably.

## 2. Goals

- Load configuration from `~/.lextures.yaml` with environment variable and flag overrides.
- Resolve server URL, API key, and output format globally for all subcommands.
- Provide a consistent error-handling and exit-code contract (`0` success, `1` user error, `2` server/unexpected error).
- Provide `--json` flag wiring so any subcommand can output machine-readable JSON.
- Ship `lextures version` and `lextures help` as working commands.

## 3. Non-Goals

- Authentication token management — covered in 21.2.
- Any resource-specific subcommands.
- Shell completion generation (can be added later with `cobra.GenBashCompletion`).

## 4. Personas & User Stories

- **As a DevOps engineer**, I want to set `LEXTURES_API_KEY` and `LEXTURES_SERVER` as environment variables so that I can run CLI commands in CI without a config file.
- **As a developer**, I want to keep multiple profiles in `~/.lextures.yaml` (staging, prod) and switch via `--config` so that I don't accidentally run destructive commands against production.
- **As an automation script author**, I want `--json` to make every command output valid JSON so that I can pipe results to `jq`.

## 5. Functional Requirements

- **FR-1.** The CLI MUST load configuration from `~/.lextures.yaml` by default; `--config <path>` MUST override the path.
- **FR-2.** Config values MUST be overridable via flags: `--api-key`, `--server`, `--json`.
- **FR-3.** Flag values MUST take precedence over config file values, which MUST take precedence over compiled-in defaults.
- **FR-4.** Environment variables `LEXTURES_API_KEY`, `LEXTURES_SERVER`, and `LEXTURES_JSON` MUST be honoured between config-file and flag layers in precedence.
- **FR-5.** The root command MUST expose `lextures version` printing the semver string and build commit.
- **FR-6.** When `--json` is set, error output MUST also be JSON: `{"error": "message", "code": N}`.
- **FR-7.** Exit codes MUST be consistent: `0` success, `1` bad input / usage error, `2` API / server error.
- **FR-8.** The config file MUST support a `profiles` map; `--profile <name>` selects a named profile.

## 6. Non-Functional Requirements

- **Performance** — CLI startup MUST complete in < 100 ms on a cold run (no lazy imports, minimal init).
- **Security** — API key MUST NOT be logged or printed in any non-`auth` command output. Config file SHOULD have `0600` permissions warned on if too permissive.
- **Reliability** — Missing or malformed config file MUST produce a clear error, not a panic.
- **Maintainability** — Config struct lives in `clients/cli/internal/config/`; shared HTTP client in `clients/cli/internal/client/`.
- **Backward compatibility** — Config file format versioned with a `version: 1` field; unknown keys ignored with a warning.

## 7. Acceptance Criteria

- **AC-1.** *Given* `~/.lextures.yaml` does not exist, *When* `lextures version` is run, *Then* it prints the version and exits `0` without error.
- **AC-2.** *Given* `LEXTURES_SERVER=https://staging.example.com` is set, *When* any command makes an API call, *Then* the request goes to `https://staging.example.com`.
- **AC-3.** *Given* `--json` is passed, *When* a command encounters an error, *Then* stderr contains `{"error":"…","code":N}` and exit code is non-zero.
- **AC-4.** *Given* a config file with `profiles.staging.server`, *When* `--profile staging` is passed, *Then* that server URL is used.
- **AC-5.** *Given* a config file is world-readable (`chmod 644`), *When* the CLI loads it, *Then* a warning is printed to stderr.

## 8. Data Model

No database changes. Config file schema:

```yaml
version: 1
server: https://app.lextures.com
api_key: <token>
json: false
profiles:
  staging:
    server: https://staging.lextures.com
    api_key: <staging-token>
```

## 9. API Surface

No new HTTP routes. The foundation wires the existing Lextures REST API base URL via config.

## 10. UI / UX

CLI only. Help output (`lextures --help`) MUST list all global flags with defaults. Error messages MUST be human-readable in non-JSON mode.

## 11. AI / ML Considerations

Not applicable.

## 12. Integration Points

- `clients/cli/cmd/root.go` — add global flag registration and config loading.
- `clients/cli/internal/config/` — new package for config struct + Viper wiring.
- `clients/cli/internal/client/` — new package for shared HTTP client (injects API key header, base URL).
- All subsequent CLI stories depend on `internal/client`.

## 13. Dependencies & Sequencing

- No upstream dependencies.
- Must ship before: 21.2, 21.3, 21.4, 21.5, 21.6, 21.7, 21.8.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Config precedence bugs cause wrong server targeted | M | H | Table-driven unit tests covering all precedence combinations |
| API key accidentally logged | L | H | Lint rule / grep in CI for logging the key field |

## 15. Rollout Plan

- Ships as part of `clients/cli/` directory; no server-side changes needed.
- Distributed as a standalone binary; no feature flag needed.

## 16. Test Plan

- **Unit** — Config loading precedence (flag > env > file > default); malformed YAML; missing file.
- **Integration** — `lextures version` exits 0; `--json` error formatting; `--profile` switching.
- **Manual** — Install binary locally; verify `~/.lextures.yaml` config loads correctly.

## 17. Documentation & Training

- `clients/cli/README.md` — installation, config file reference, environment variables.
- Help text on all flags (`cobra` `Long` fields).
