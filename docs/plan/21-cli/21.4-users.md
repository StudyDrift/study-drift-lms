## 1. Problem Statement

Provisioning students and instructors at the start of a term is one of the highest-friction admin tasks: the web UI requires creating each user individually. Admins at institutions with hundreds or thousands of enrollees need a CLI path to script user creation and enrollment from rosters (CSV, SIS exports). This story covers the individual commands; bulk import from file is a follow-on.

## 2. Goals

- `lextures users list` lists all users in the org.
- `lextures users create` provisions a new user account.
- `lextures users get <id>` fetches user details.
- `lextures users enroll` enrolls a user into a course with a role.
- All commands support `--json`.

## 3. Non-Goals

- Bulk user import from CSV (follow-on; can be scripted via `create` in a loop).
- User deactivation / deletion (sensitive; deferred to a separate story with audit log integration).
- Role management beyond the enroll-time role assignment.

## 4. Personas & User Stories

- **As an admin**, I want `lextures users create --email student@uni.edu --name "Jane Doe" --role student` to provision a student account so that I can automate semester onboarding.
- **As an admin**, I want `lextures users enroll --course CS101 --user jane@uni.edu --role student` to enroll a user so that I can script course enrollments from a SIS roster export.
- **As an instructor**, I want `lextures users list --json` to enumerate my enrolled students for use in scripts that email assignment reminders.

## 5. Functional Requirements

- **FR-1.** `lextures users list` MUST return all users visible to the authenticated user; supports `--org <id>`, `--role <role>`, `--limit`, `--page`.
- **FR-2.** `lextures users create` MUST accept `--email` (required), `--name` (required), and `--role` (optional, default `student`).
- **FR-3.** `lextures users get <id>` MUST accept both UUID and email address as `<id>`.
- **FR-4.** `lextures users enroll` MUST accept `--course <id>` (required), `--user <id|email>` (required), `--role <role>` (required: `student` | `instructor` | `ta`).
- **FR-5.** All commands MUST support `--json`.
- **FR-6.** `lextures users enroll` MUST be idempotent: re-enrolling an already-enrolled user MUST succeed (or print a clear warning) rather than error.

## 6. Non-Functional Requirements

- **Security** — Only admins and instructors (for their own courses) MUST be permitted to enroll; enforced server-side. CLI does not add client-side authz checks.
- **Privacy** — User list output MUST NOT include password hashes or internal tokens even in `--json` mode (server-side responsibility, CLI documents this).

## 7. Acceptance Criteria

- **AC-1.** *Given* `lextures users create --email x@y.com --name "X Y"`, *When* the command succeeds, *Then* the user ID is printed.
- **AC-2.** *Given* an existing email, *When* `lextures users create` is called again with the same email, *Then* the CLI prints a clear duplicate-user error and exits `1`.
- **AC-3.** *Given* `lextures users enroll --course <id> --user x@y.com --role student`, *When* the command succeeds, *Then* the user appears in the course roster.
- **AC-4.** *Given* `lextures users get x@y.com`, *When* the command runs, *Then* user details are returned (email lookup supported).
- **AC-5.** *Given* `lextures users enroll` called twice with the same params, *When* the second call runs, *Then* it exits `0` with a warning "already enrolled".

## 8. Data Model

No new database changes. Uses existing user and enrollment API endpoints.

## 9. API Surface

Uses existing endpoints:

```
GET    /api/users                ?org=&role=&limit=&page=
POST   /api/users                { email, name, role }
GET    /api/users/:id            (id = UUID or email)
POST   /api/courses/:course_id/enrollments   { user_id, role }
```

## 10. UI / UX

Non-JSON table: columns `ID`, `Name`, `Email`, `Role`, `Created`. Enroll success prints `"Enrolled <name> as <role> in course <id>"`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Accidental mass enrollment from script error | M | H | Dry-run flag `--dry-run` prints what would happen without API calls |
| Email vs UUID ambiguity in `--user` | L | M | Client tries UUID parse first, falls back to email lookup |
