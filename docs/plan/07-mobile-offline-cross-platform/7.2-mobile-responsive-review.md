## 1. Problem Statement

The Lextures web app was built desktop-first. Several critical flows are broken or severely degraded on mobile: the gradebook grid overflows horizontally with no touch-scroll, the dnd-kit drag-and-drop module editor is not touch-accessible, and the TipTap toolbar wraps unreadably on phones. No systematic mobile-breakpoint audit has been performed. Students who attempt to use Lextures on a phone encounter a frustrating experience. Fixing this is a prerequisite for both the native mobile app (7.1 — which uses WebViews) and for the PWA (7.3).

## 2. Goals

- Audit every existing page and component for mobile breakpoints (375 px, 390 px, 768 px widths).
- Fix the 10 highest-priority mobile regressions identified in the audit.
- Establish mobile design standards (spacing, tap targets, font sizes) in the Tailwind config.
- Add Playwright mobile-viewport regression tests for each fixed flow.
- Produce a comprehensive punch-list of remaining issues for V2.

## 3. Non-Goals

- Native mobile app development — covered in 7.1.
- Tablet-specific responsive layouts — deferred.
- PWA manifest / service worker — covered in 7.3.
- Redesigning the app visually — only fixing functional breakage and improving usability within existing design language.

## 4. Personas & User Stories

- **As a student on an iPhone**, I want to take a quiz without horizontal scrolling or cut-off buttons so that I can complete my work from my phone.
- **As an instructor on a tablet**, I want to view the gradebook without it being truncated so that I can enter grades while walking around the classroom.
- **As a student on Android**, I want to read announcements and reply in the course feed without zooming in to read text so that the experience feels native.
- **As a developer**, I want documented mobile standards in the design system so that new components are built mobile-first.

## 5. Functional Requirements

- **FR-1.** The system MUST fix the gradebook grid so that it scrolls horizontally within its container on screens <768 px rather than overflowing the viewport.
- **FR-2.** The system MUST replace or augment dnd-kit drag-and-drop in the module editor with touch-friendly reorder controls (up/down arrow buttons) on screens <768 px.
- **FR-3.** The system MUST ensure the TipTap editor toolbar collapses to an icon-only row on screens <640 px with a "more" overflow menu.
- **FR-4.** The system MUST ensure all interactive elements (buttons, links, toggles) have a minimum tap target of 44×44 CSS pixels per Apple HIG / WCAG 2.5.8.
- **FR-5.** The system MUST ensure the course navigation sidebar collapses to a bottom navigation bar or hamburger drawer on screens <768 px.
- **FR-6.** The system MUST verify that the quiz question layout is single-column and full-width on mobile, with answer choices stacked vertically.
- **FR-7.** The system SHOULD add `viewport` meta tag (`width=device-width, initial-scale=1`) if not already present in `index.html`.
- **FR-8.** The system SHOULD define standard Tailwind breakpoint tokens for mobile-first design and document them in the design system (Storybook or equivalent).

## 6. Non-Functional Requirements

- **Performance** — pages must achieve Lighthouse Mobile score ≥70 for FCP and LCP after fixes; no images loaded at desktop resolution on mobile (responsive `<img srcset>`).
- **Security** — no security implications; pure frontend changes.
- **Privacy & Compliance** — WCAG 2.5.8 (minimum target size) — part of WCAG 2.2 AA compliance required by 10.7.
- **Accessibility** — all fixes must not regress desktop a11y; each fixed component re-tested with axe.
- **Scalability** — N/A.
- **Reliability** — mobile fixes must not break existing desktop E2E Playwright tests.
- **Observability** — Core Web Vitals (FCP, LCP, CLS) tracked in CI for mobile viewport.
- **Maintainability** — use Tailwind responsive prefixes (`sm:`, `md:`, `lg:`); document mobile standards in `clients/web/CONTRIBUTING.md`.
- **Internationalization** — all fixes apply equally to RTL layouts (test with `dir="rtl"` on body).
- **Backward compatibility** — fixes must not break the desktop experience.

## 7. Acceptance Criteria

- **AC-1.** *Given* a 390 px wide viewport, *When* the gradebook page loads, *Then* the table scrolls horizontally within its container and the viewport does not overflow (no horizontal page scroll).
- **AC-2.** *Given* a touch device, *When* the instructor opens the module editor, *Then* drag handles are hidden and up/down arrow reorder buttons are visible per module item.
- **AC-3.** *Given* a 375 px wide viewport, *When* the TipTap editor toolbar is rendered, *Then* it fits within one row of icons plus an overflow "..." button; no icon is clipped or hidden behind the viewport edge.
- **AC-4.** *Given* any interactive element on any page, *When* its CSS dimensions are measured, *Then* both width and height are ≥44 px.
- **AC-5.** *Given* a 375 px wide viewport, *When* a quiz is open, *Then* each answer choice is displayed in a full-width stacked layout with no horizontal overflow.
- **AC-6.** *Given* the Lighthouse mobile audit runs in CI on the dashboard page, *When* the score is computed, *Then* it is ≥70.

## 10. UI / UX

Audit scope — pages to review and fix in priority order:
1. `clients/web/src/pages/lms/CourseGradebook.tsx` — horizontal-scroll fix.
2. `clients/web/src/pages/lms/CourseModules.tsx` — dnd-kit touch reorder.
3. `clients/web/src/components/editor/` — TipTap toolbar responsive overflow.
4. `clients/web/src/pages/lms/CourseModuleQuizPage.tsx` — quiz single-column layout.
5. `clients/web/src/components/layout/` — sidebar → bottom nav drawer.
6. `clients/web/src/pages/lms/CourseFeedPage.tsx` — compose box and feed cards.
7. `clients/web/src/pages/lms/Inbox.tsx` — message thread layout.
8. `clients/web/src/pages/lms/CourseGradingSettings.tsx` — settings form layout.
9. `clients/web/src/pages/lms/Dashboard.tsx` — course card grid responsive.
10. `clients/web/src/pages/lms/Settings.tsx` — settings nav + content pane.

Standards to document:
- Tap target minimum: 44×44 px.
- Body font minimum: 16 px on mobile.
- Touch-friendly spacing: ≥12 px between interactive elements.
- Responsive images: use `srcset` for all hero/banner images.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Mobile fix breaks desktop layout | M | M | Tailwind responsive prefix discipline; desktop E2E tests re-run after each fix |
| Touch reorder (dnd-kit) conflicts with scroll | H | M | Test on real devices (not emulator only); use pointer events API |
| Lighthouse score target unreachable due to third-party scripts | M | M | Audit third-party script load order; async-load non-critical scripts |

## 15. Rollout Plan

- No feature flag needed — fixes are progressive enhancements.
- Sequence: gradebook → module editor → TipTap toolbar → quiz → sidebar → remaining pages (priority order).
- Dogfood: team tests each fix on a real iPhone and Android device before merging.
- GA criteria: Lighthouse mobile ≥70; all 10 priority items fixed; Playwright mobile tests pass.
- Rollback: each fix is a discrete PR; revert individual PRs if regression detected.

## 16. Test Plan
