## 1. Problem Statement

Web push (6.3) covers browser-based push notifications, but students on iOS and Android native apps (7.1) require APNs (Apple Push Notification service) and FCM (Firebase Cloud Messaging) respectively. These are distinct protocols from Web Push, require native SDK integration in the mobile app, and require server-side libraries to compose and dispatch payloads. Without APNs/FCM, the native mobile apps (7.1) are silent — no grade alerts, no assignment reminders, no discussion notifications on students' phones.

## 2. Goals

- Implement server-side APNs delivery using the HTTP/2 APNs provider API with JWT authentication (p8 key).
- Implement server-side FCM delivery using the FCM v1 HTTP API with service account authentication.
- Reuse the `notification_preferences` table and event pipeline from 6.3.
- Store device push tokens in a unified `device_push_tokens` table shared between APNs and FCM.
- Handle token rotation, invalid-token cleanup, and delivery receipts.

## 3. Non-Goals

- Web Push (RFC 8030 / VAPID) — already covered in 6.3.
- Batch push (marketing campaigns) — transactional event-driven push only.
- Rich push media (images, video) in notification — V1 text-only with action URL.
- In-app message banners (different from OS-level push) — deferred.

## 4. Personas & User Stories

- **As a student with the Lextures iOS app**, I want to receive a push notification on my iPhone when a grade is posted so that I know immediately without opening the app.
- **As a student with the Lextures Android app**, I want a push notification badge on the app icon showing the count of unread notifications.
- **As a developer**, I want a single push-dispatch function that routes to APNs or FCM based on the device token type so that I do not maintain two code paths.
- **As an ops engineer**, I want stale device tokens automatically removed when APNs/FCM signals they are invalid so that we do not waste push quota on ghost devices.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a `device_push_tokens` table: `id`, `user_id`, `token`, `platform` (`apns`, `fcm`), `app_bundle_id`, `created_at`, `last_used_at`, `is_active`.
- **FR-2.** The system MUST provide `POST /api/me/device-tokens` (authenticated) for the mobile app to register an APNs or FCM device token on login; and `DELETE /api/me/device-tokens/:id` to deregister on logout.
- **FR-3.** The system MUST implement an APNs delivery adapter in `server/src/services/push/apns.rs` using the APNs HTTP/2 provider API with `.p8` key JWT authentication (no legacy binary protocol).
- **FR-4.** The system MUST implement an FCM delivery adapter in `server/src/services/push/fcm.rs` using the FCM v1 API (`https://fcm.googleapis.com/v1/projects/{project_id}/messages:send`) with OAuth2 service account credentials.
- **FR-5.** The system MUST route push notifications through the existing notification event pipeline from 6.3: events → check `notification_preferences.push_enabled` → enqueue job → dispatch to APNs or FCM based on `device_push_tokens.platform`.
- **FR-6.** The system MUST handle APNs error responses: on `BadDeviceToken` or `Unregistered`, mark the token `is_active = false`; on `TooManyRequests`, implement exponential back-off.
- **FR-7.** The system MUST handle FCM error responses: on `registration-token-not-registered`, mark token inactive; on quota errors, back-off.
- **FR-8.** The system SHOULD fan-out to all active device tokens for a user (they may have multiple devices).
- **FR-9.** The system SHOULD include a `data` payload in each push with `notification_type` and `action_url` for deep linking (7.1 FR-7).

## 6. Non-Functional Requirements

- **Performance** — APNs/FCM push job must complete within 5 s per device token under normal load; fan-out to 5 devices per user within 10 s.
- **Security** — APNs `.p8` key and FCM service account JSON stored in environment secrets (never in DB or source control); HTTPS-only communication with both providers. Push tokens are device-specific and user-authenticated; no cross-user token sharing possible.
- **Privacy & Compliance** — push payload must not contain raw grade values or student PII — only a notification type and a link back to the authenticated app. COPPA: under-13 iOS apps must not use IDFA; APNs does not require IDFA, so this is compliant. FERPA: notification text is generic ("You have a new grade"); full details only accessible post-authentication inside the app.
- **Accessibility** — APNs/FCM payloads include a `subtitle` and `body` for accessibility on lock screen; iOS notification includes `sound: default` to alert visually-impaired users using audio cues.
- **Scalability** — fan-out to ≥1,000 device tokens per event (e.g., a district-wide announcement); batched dispatch in job workers.
- **Reliability** — APNs/FCM jobs retry up to 3 times with exponential back-off; dead-letter queue for permanent failures; token cleanup job runs daily.
- **Observability** — log `push_job_id`, `platform`, `user_id`, `delivery_status` per dispatch; track APNs/FCM delivery success rate; alert on >5% failure rate sustained >5 min.
- **Maintainability** — `server/src/services/push/apns.rs`, `server/src/services/push/fcm.rs`, `server/src/services/push/dispatcher.rs` (routes to correct platform); unified `PushProvider` trait.
- **Internationalization** — push notification title/body text externalized in notification service; sent in English V1 (no per-device locale in push payload V1).
- **Backward compatibility** — web push subscriptions (6.3) stored in `push_subscriptions`; device push tokens in `device_push_tokens`; separate tables for clean separation.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student has registered an APNs token from their iPhone, *When* a grade is posted, *Then* an APNs push notification is delivered to the device within 5 s.
- **AC-2.** *Given* a student has registered an FCM token from their Android device, *When* an assignment is created, *Then* an FCM push notification is delivered.
- **AC-3.** *Given* an APNs response of `Unregistered` for a token, *When* the push job worker processes the response, *Then* the token is marked `is_active = false` and no further pushes are sent to it.
- **AC-4.** *Given* a user has 3 active device tokens (1 APNs, 2 FCM), *When* a notification event occurs, *Then* 3 push jobs are dispatched — one to APNs, two to FCM.
- **AC-5.** *Given* the APNs provider API returns rate-limit (429), *When* the push job retries, *Then* it waits 30 s before retry (exponential back-off applied) and succeeds on the retry.
- **AC-6.** *Given* a push notification is received on iOS, *When* the user taps it, *Then* the app deep-links to the specific course assignment referenced in the notification `data.action_url`.

## 8. Data Model

```sql
-- server/migrations/092_device_push_tokens.sql
CREATE TABLE device_push_tokens (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID NOT NULL REFERENCES users(id),
  token         TEXT NOT NULL,
  platform      TEXT NOT NULL CHECK (platform IN ('apns','fcm')),
  app_bundle_id TEXT,
  is_active     BOOLEAN NOT NULL DEFAULT true,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  last_used_at  TIMESTAMPTZ,
  UNIQUE (user_id, token)
);

CREATE INDEX idx_device_tokens_user_active ON device_push_tokens(user_id, is_active);
```

APNs `.p8` key stored in `APNS_P8_KEY` environment variable; FCM service account JSON in `FCM_SERVICE_ACCOUNT_JSON`.

## 9. API Surface

| Verb | Path | Auth scope | Description |
|---|---|---|---|
| POST | `/api/me/device-tokens` | authenticated | Register APNs or FCM token |
| DELETE | `/api/me/device-tokens/:id` | authenticated | Deregister token (on logout) |
| GET | `/api/me/device-tokens` | authenticated | List active tokens (for session management) |

Request body for POST:
```typescript
{ token: string; platform: "apns" | "fcm"; app_bundle_id?: string; }
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| APNs `.p8` key expires (10-year validity but needs rotation) | L | H | Document expiry date; alert 90 days before; rotation runbook |
| FCM project quota exceeded | M | M | Monitor daily send count; upgrade Firebase plan proactively |
| APNs delivery silent failure (no error, no delivery) | M | M | Use APNs feedback service + test delivery via Apple's notification console |
| Token accumulation (many inactive tokens) | H | M | Daily cleanup job marks inactive tokens; purge after 90 days |

## 15. Rollout Plan

- Feature flag: `native_push_enabled` (default: `false`).
- Sequence: DB migration → APNs adapter → FCM adapter → dispatcher → device token API → mobile app integration → job queue workers → flag.
- Dogfood: register dev team's iPhones and Android phones; verify grade notification delivery.
- GA criteria: APNs delivery confirmed on iPhone in staging; FCM delivery confirmed on Android; token cleanup job verified.
- Rollback: disable flag; `device_push_tokens` table preserved but unused.

## 16. Test Plan

- **Unit** — APNs JWT generation; FCM OAuth2 token fetch; dispatcher routing logic; invalid-token handling.
- **Integration** — mock APNs/FCM servers: verify payload structure; retry on 429; token deactivation on Unregistered.
- **End-to-end** — physical device test: iOS grade notification; Android discussion notification.
- **Security** — `.p8` key and FCM credentials not in source; API auth check on token registration.
- **Accessibility** — notification payload includes accessible title and body fields.
- **Performance / load** — fan-out to 1,000 device tokens within 10 s via job workers.
- **Manual exploratory** — logout deregisters token; same token re-registered after re-login; stale token cleanup.

## 18. Open Questions

1. Should we use a unified push abstraction library (e.g., `notify-rust`, or a Rust FCM crate) or implement both providers from scratch?
2. Should push notification text be personalized (include student name) or generic (only event type and course name)?
3. Should the daily token cleanup job also emit analytics on active device distribution (iOS vs Android ratio)?
4. Do we need a webhook from FCM for delivery receipts, or is failure-on-send sufficient?
