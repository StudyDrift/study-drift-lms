## 1. Problem Statement

K-12 students in low-bandwidth districts and self-learners on public transit need to access course content without a reliable internet connection. Lextures has no service worker, no Web App Manifest, and no offline data persistence. A student who loses connectivity while working on a lesson loses their progress, and students with poor home internet cannot access course materials at all. This feature transforms Lextures into a Progressive Web App (PWA) with offline-capable content viewing, draft-saving of in-progress work, and background sync.

## 2. Goals

- Implement a Web App Manifest so Lextures can be installed on the home screen (Android / iOS Safari / Chrome desktop).
- Register a Service Worker (Workbox-based) to cache the app shell and static assets.
- Enable offline access for previously-viewed course content using IndexedDB.
- Support background sync: quiz answers and discussion replies composed offline are sent when connectivity resumes.
- Show a clear offline indicator in the UI and degrade gracefully when features require connectivity.

## 3. Non-Goals

- Full feature parity offline (admin tools, file uploads, video streaming require connectivity).
- Building a custom service worker from scratch — use Workbox (Google) for cache strategy.
- Server-side offline capabilities — this is entirely a frontend feature.
- Peer-to-peer content sharing offline (e.g., via Bluetooth) — out of scope.

## 4. Personas & User Stories

- **As a student in a low-bandwidth district**, I want to download this week's reading materials at school and read them at home offline so that I am not blocked by poor home Wi-Fi.
- **As a self-learner on a train**, I want to start a quiz attempt offline and have my answers submitted automatically when I exit the tunnel so that I do not lose my work.
- **As a student**, I want to see which content items I have saved for offline access so that I can plan my study time.
- **As a developer**, I want the service worker to be generated by Workbox so that we get a battle-tested cache strategy without building it manually.

## 5. Functional Requirements

- **FR-1.** The system MUST add a Web App Manifest (`clients/web/public/manifest.json`) with `name`, `short_name`, `icons` (192 px, 512 px), `start_url`, `display: standalone`, `theme_color`, and `background_color`.
- **FR-2.** The system MUST register a Service Worker (`clients/web/public/sw.js`) generated by Workbox that implements: `CacheFirst` for static assets, `StaleWhileRevalidate` for API responses where safe, `NetworkFirst` for critical data (gradebook, current assignments).
- **FR-3.** The system MUST implement IndexedDB storage (using `idb` library or Dexie.js) to persist: course list, module item content, quiz question sets, and discussion thread posts for each course the student has recently visited.
- **FR-4.** The system MUST implement the Background Sync API (via Service Worker) to queue offline quiz answer submissions and discussion posts; these are synced when the browser regains connectivity.
- **FR-5.** The system MUST show an offline indicator banner ("You are offline — viewing cached content") when `navigator.onLine === false`.
- **FR-6.** The system MUST disable actions that require connectivity while offline (file upload, video play, live meeting join) with a clear disabled state and tooltip "Available when online."
- **FR-7.** The system SHOULD provide a "Save for offline" button on course content pages, triggering explicit caching of that item's content to IndexedDB.
- **FR-8.** The system SHOULD show a "Cached" indicator on content items available offline and "Syncing…" for items queued for sync.

## 6. Non-Functional Requirements

- **Performance** — service worker install and activation must complete within 5 s on first load; cached app shell must load offline within 1 s.
- **Security** — service worker intercepts all requests; must not cache authenticated API responses containing PII in a way accessible to other origins; use opaque responses with caution. Background sync payloads encrypted in IndexedDB.
- **Privacy & Compliance** — FERPA: cached educational data remains on the student's device; clear documentation that cached data can be cleared via browser settings. COPPA: no additional data collection from caching.
- **Accessibility** — offline banner uses `role="alert"` and `aria-live="assertive"`; "Save for offline" button has clear aria-label.
- **Scalability** — IndexedDB storage is per-browser; no server scaling concern. Background sync queue bounded to 50 pending items max.
- **Reliability** — background sync must retry failed submissions up to 5 times; after 5 failures display an error to the user. Service worker update flow must not leave stale clients indefinitely (skip-waiting + clients-claim).
- **Observability** — log `sw_cache_hit`, `sw_cache_miss`, `bg_sync_queued`, `bg_sync_sent` via analytics; alert if background sync failure rate >10%.
- **Maintainability** — Workbox config in `clients/web/vite.config.ts` (using `vite-plugin-pwa`); IndexedDB schema versioned in `clients/web/src/db/schema.ts`.
- **Internationalization** — offline banner text externalized; "Saved for offline" label i18n keyed.
- **Backward compatibility** — PWA features are additive; users without service worker support see the existing experience unchanged.

## 7. Acceptance Criteria

- **AC-1.** *Given* the app is loaded once online, *When* the network is disconnected and the app is reloaded, *Then* the app shell and the last-viewed dashboard load from cache within 1 s.
- **AC-2.** *Given* a student views a course content page online, *When* the network is disconnected and they navigate to that page again, *Then* the content is displayed from IndexedDB cache.
- **AC-3.** *Given* a student composes a discussion reply offline, *When* the network is restored, *Then* the reply is automatically submitted within 10 s via Background Sync.
- **AC-4.** *Given* the device is offline, *When* the student opens the app, *Then* a banner reads "You are offline — viewing cached content" and file upload buttons are disabled.
- **AC-5.** *Given* the Web App Manifest is served, *When* a Chrome user visits the site on Android, *Then* the browser displays a home-screen install prompt and the installed app launches in standalone mode without browser chrome.
- **AC-6.** *Given* a new service worker version is deployed, *When* the user refreshes the page, *Then* the new service worker activates and the user sees the updated app (not a stale cached version older than 24h).

## 8. Data Model

Client-side IndexedDB schema (`clients/web/src/db/schema.ts`):

```typescript
// Dexie.js schema
db.version(1).stores({
  courses:        '++id, org_id, name, updated_at',
  course_items:   '++id, course_id, type, updated_at',
  quiz_attempts:  '++id, quiz_id, status, created_at',   // offline draft attempts
  pending_syncs:  '++id, type, payload, created_at, attempts',
});
```

Server-side: no new tables. `Content-Type` and Cache-Control headers tuned on relevant API responses.

## 9. API Surface

No new endpoints. Changes to existing API:

- Add `Cache-Control: no-store` to authenticated gradebook and grade endpoints (must not be cached).
- Add `Cache-Control: max-age=300, stale-while-revalidate=600` to course content endpoints.
- Service worker intercepts `GET /api/courses/*` and returns cached response when offline.

New config in `clients/web/vite.config.ts`:
```typescript
VitePWA({
  registerType: 'autoUpdate',
  workbox: {
    runtimeCaching: [ /* cache strategies per URL pattern */ ]
  },
  manifest: { /* ... */ }
})
```

## 10. UI / UX

New components:
- `clients/web/src/components/OfflineBanner.tsx` — sticky top banner when `!navigator.onLine`.
- "Save for offline" icon button on `CourseModuleContentPage.tsx` and `CourseModuleQuizPage.tsx`.
- Sync status indicator in the notification bell: "3 items waiting to sync."

Offline states:
- Cached content page: renders normally, "Cached" badge in top-right corner.
- Quiz: allowed to continue offline draft; submit button says "Submit when online."
- Feed: cached posts shown; compose disabled with "Posting requires connection."
- Gradebook: last-cached data shown with "Last updated [time]" footer.

Install prompt: deferred to Workbox's `beforeinstallprompt` handler; shown after 2nd visit.
Mobile: offline banner stacks below top nav.
Accessibility: offline banner `role="alert"`; sync status announced via `aria-live`.

## 11. AI / ML Considerations

AI Tutor (6.9) and RAG notebook require connectivity; both are disabled and show "AI features require an internet connection" when offline.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Service worker caches stale authenticated responses with PII | M | H | `no-store` on grade/submission endpoints; test with DevTools cache inspection |
| Background sync not supported on iOS Safari | H | M | Graceful fallback: retry on reconnect via `online` event listener |
| Service worker update loop (users stuck on old version) | M | M | `skipWaiting` + `clientsClaim`; show "Update available — refresh" toast |
| IndexedDB storage quota exceeded | M | M | Evict oldest cached items when quota >80%; warn user |

## 15. Rollout Plan

- Feature flag: `pwa_enabled` (default: `false`).
- Sequence: Web App Manifest → Workbox cache strategies → IndexedDB schema → offline UI states → background sync → "Save for offline" button → flag.
- Dogfood: install PWA on team phones in dev environment.
- GA criteria: offline content viewing confirmed on real device; background sync confirmed on reconnect; Lighthouse PWA score ≥90.
- Rollback: service worker can be deregistered via JS; manifest has no side effects to remove.

## 16. Test Plan

- **Unit** — Dexie.js schema migrations; pending sync queue management.
- **Integration** — service worker cache strategy (mock fetch); background sync retry logic.
- **End-to-end** — Playwright with `offline` network condition: load app, go offline, navigate cached page; go online, verify pending sync submitted.
- **Security** — confirm no authenticated grade data in HTTP cache; DevTools cache inspection.
- **Accessibility** — offline banner announces correctly; disabled buttons have aria-disabled + tooltip.
- **Performance / load** — Lighthouse PWA audit: installable, offline fallback, ≥90 score.
- **Manual exploratory** — real device offline test on iPhone Safari; service worker update flow.

## 17. Documentation & Training

- Help center: "Using Lextures offline — installing as an app and saving content."
- Admin docs: no admin config needed; note that offline mode is browser-level.
- Internal runbook: service worker update release process; debugging cache issues.

## 18. Open Questions

1. Should we use Workbox's `GenerateSW` mode (automatic) or `InjectManifest` mode (custom logic)?
2. What is the maximum IndexedDB storage allocation per user — 50 MB? 200 MB?
3. Should quiz draft auto-save to IndexedDB every 30 s, or only on explicit "Save draft" action?
4. How do we handle offline access if the user's session JWT expires while offline?
