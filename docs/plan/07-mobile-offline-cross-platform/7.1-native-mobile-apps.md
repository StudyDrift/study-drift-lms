## 1. Problem Statement

K-12 students are mobile-first: Chromebook and phone are their primary devices. Self-learners study on commutes. Lextures has no native iOS or Android app — only a desktop-first React SPA. Without a mobile app, Lextures cannot be distributed through school-managed MDM profiles, cannot receive APNs/FCM push notifications, and cannot be discovered in the App Store or Google Play. This is a BLOCKER for the K-12 market and materially impacts self-learner retention.

## 2. Goals

- Build iOS and Android apps that wrap the Lextures web app using a cross-platform hybrid approach (React Native with a WebView bridge, or Capacitor wrapping the existing Vite/React app).
- Deliver the 5 most-used student flows natively: dashboard, course view, quiz taking, gradebook, and inbox.
- Enable APNs (iOS) and FCM (Android) push notifications (7.5).
- Support offline access for core content (dependent on 7.3 PWA/offline mode).
- Submit to App Store and Google Play (7.4).

## 3. Non-Goals

- Full feature parity with web on day one — V1 covers student-facing flows only; instructor admin and authoring remain web-only.
- Building completely from scratch with no web code reuse — the hybrid approach maximizes code reuse.
- Real-time video calling natively — virtual classroom (6.4) remains web-only via WebView.
- Tablet-specific layouts — phones first; iPad/Android tablet layouts deferred.

## 4. Personas & User Stories

- **As a K-12 student**, I want to take a quiz on my phone during study hall so that I do not need to find a laptop.
- **As a self-learner**, I want to open the Lextures app on my morning commute and continue where I left off so that I use idle time effectively.
- **As a student**, I want to receive a push notification on my phone when a grade is posted so that I do not have to constantly check the website.
- **As a district IT administrator**, I want to deploy the Lextures app via Apple School Manager so that students get it automatically on their district iPhones.
- **As a parent**, I want to install the Lextures app to monitor my child's progress from my phone.

## 5. Functional Requirements

- **FR-1.** The system MUST ship an iOS app (Swift/React Native/Capacitor wrapper) targeting iOS 16+ and an Android app targeting Android 9 (API 28)+.
- **FR-2.** The system MUST implement the following native screens: Splash/login, Dashboard (course list), Course home, Module viewer, Quiz taker, Gradebook (student view), Inbox, and Notification center.
- **FR-3.** The system MUST integrate APNs (iOS) and FCM (Android) push notification tokens via the backend push service (7.5).
- **FR-4.** The system MUST persist the user's auth JWT in the device's secure enclave (iOS Keychain, Android EncryptedSharedPreferences) — not local storage.
- **FR-5.** The system MUST support biometric authentication (Face ID / Fingerprint) for re-login after app backgrounding.
- **FR-6.** The system MUST support offline reading of previously downloaded course content (dependent on 7.3 offline mode / IndexedDB sync).
- **FR-7.** The system SHOULD support deep linking: tapping a push notification opens the relevant course/assignment screen directly.
- **FR-8.** The system MAY support org-specific MDM configuration via AppConfig XML (Apple DEP / Android for Work managed configuration).

## 6. Non-Functional Requirements

- **Performance** — app cold start to dashboard ≤3 s on mid-range device (iPhone SE 2nd gen, Pixel 5); quiz rendering ≤1 s.
- **Security** — JWT stored in native secure storage only; no plain-text tokens in preferences files; biometric auth gate; certificate pinning on API calls (configurable).
- **Privacy & Compliance** — COPPA: under-13 app usage requires school-managed distribution; app store age rating set appropriately (app contains educational content for all ages, school-approved). iOS App Privacy labels and Android Data Safety form must be accurate. FERPA: data transmitted only to Lextures servers; no third-party analytics SDK that sends student data to ad networks.
- **Accessibility** — VoiceOver (iOS) and TalkBack (Android) support on all native screens; dynamic text size support; tap targets ≥44×44 pt.
- **Scalability** — mobile app is a thin client; all scaling concerns are server-side.
- **Reliability** — app must gracefully handle network loss; previously loaded data shown from local cache; sync resumes on reconnect.
- **Observability** — integrate crash reporting (Sentry or Firebase Crashlytics) and performance monitoring; app version field on all API requests for diagnostics.
- **Maintainability** — React Native (Expo) or Capacitor chosen to maximize web code reuse; shared TypeScript business logic with the web frontend.
- **Internationalization** — UI strings reuse web i18n keys (react-i18next from 11.1); locale follows device setting.
- **Backward compatibility** — API is unchanged; mobile app is a new client type consuming existing REST APIs.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student installs the app on an iPhone, *When* they log in with their Lextures credentials, *Then* their course dashboard loads within 3 s and courses are displayed.
- **AC-2.** *Given* a student takes a quiz in the app, *When* they submit their answers, *Then* the grade is recorded identically to a web submission.
- **AC-3.** *Given* a student grants push notification permission on iOS, *When* a grade is posted, *Then* an APNs push notification arrives on the device.
- **AC-4.** *Given* the device is offline, *When* the student opens a previously cached course, *Then* the course content is displayed from local cache.
- **AC-5.** *Given* a deep link from a push notification, *When* the user taps it, *Then* the app opens directly to the relevant assignment page.
- **AC-6.** *Given* biometric authentication is enabled, *When* the app is foregrounded after 5 minutes of backgrounding, *Then* a Face ID / fingerprint prompt is shown before restoring the session.

## 9. API Surface

No new server endpoints required for V1. Mobile app consumes existing REST APIs. New in 7.5:

- `POST /api/me/device-tokens` — register APNs/FCM push token.
- `DELETE /api/me/device-tokens/:id` — deregister on logout.

Additional header for all mobile requests: `X-App-Version: 1.0.0`, `X-Platform: ios|android` for analytics and support.

## 10. UI / UX

Mobile-native screens (React Native with NativeWind or Capacitor with the responsive web app):

- **Splash screen:** org logo (from 5.7 branding), centered on white/dark background.
- **Login:** email/password + SSO button; biometric unlock if already logged in.
- **Dashboard:** course cards in vertical scroll; notification bell in header.
- **Course home:** tabs for Modules, Feed, Grades; sticky course header.
- **Quiz taker:** full-screen, one-question-at-a-time layout on mobile.
- **Gradebook:** card-based list of assignments and grades (not a grid on mobile).
- **Inbox:** conversation list + thread view.

Bottom navigation bar: Home, Courses, Notifications, Profile.
Accessibility: VoiceOver/TalkBack-compatible; semantic elements; skip-to-main-content.
Dark mode: follows system setting.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| App review rejection (App Store) | M | H | Follow Apple HIG; comply with App Store Review Guidelines 1.0 (privacy); have edu category ready |
| React Native performance on low-end Android | H | M | Benchmark on Pixel 3a; optimize list rendering with FlashList |
| JWT secure storage implementation vulnerability | M | H | Use Expo SecureStore (iOS Keychain wrapper); security review before shipping |
| App update lag (users stuck on old version) | M | M | Force-update prompt for security releases; semantic versioning |

## 15. Rollout Plan

- Feature flag: not applicable (app submission is the rollout).
- Sequence: choose framework (Expo / Capacitor) → core screens → biometric auth → push tokens → offline (7.3) → app store submission → internal testing → public release.
- Dogfood: TestFlight (iOS) and internal testing track (Android) for team and pilot schools.
- GA criteria: App Store and Play Store approval; 5 pilot school students confirm quiz and gradebook functionality.
- Rollback: pull app from stores if critical bug; server-side feature flags for specific features.

## 16. Test Plan

- **Unit** — secure token storage; JWT refresh logic.
- **Integration** — API calls from mobile client; push token registration.
- **End-to-end** — Detox (React Native) or Maestro: login, view course, take quiz, check grade.
- **Security** — JWT not in plaintext preferences; certificate pinning verified; biometric gate test.
- **Accessibility** — VoiceOver automated tests (iOS); TalkBack manual test for all screens.
- **Performance / load** — cold start benchmark on iPhone SE 2; Android Pixel 5; low-end Android.
- **Manual exploratory** — offline mode; deep link from push notification; org branding on login.
