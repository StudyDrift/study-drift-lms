## 1. Problem Statement

Instructors have no automated signal when a student begins to disengage or fail. In the current system, an instructor must manually review every student's gradebook row to find a struggling learner — this is impractical in courses with 50+ students. At-risk detection is a core feature of modern adaptive LMSs and a differentiator cited in procurement RFPs. Without it, Lextures has no early-intervention capability, a gap that directly leads to student failure and course dropout.

## 2. Goals

- Compute a nightly at-risk score per enrolled student per course using rule-based signals from engagement and grade data.
- Alert instructors via in-app notification and email when a student crosses a configurable risk threshold.
- Provide an "At-Risk Students" view in the course instructor panel for rapid intervention triage.
- Allow instructors to dismiss alerts, add notes, and mark students as "being supported."

## 3. Non-Goals

- Building a full ML model (initial release uses rule-based scoring; ML deferred to a follow-on).
- Automated outreach to students (alerting the instructor only; instructor decides how to respond).
- Parent/guardian alerts (depends on §5.10 parent portal).

## 4. Personas & User Stories

- **As an instructor of a 120-student lecture**, I want a daily digest of students who have missed two or more assignments so that I can focus my office hours on the highest-need students.
- **As an instructor**, I want to mark a student as "currently being supported" so that the alert does not keep re-triggering.
- **As a course coordinator**, I want to see which courses have the highest at-risk rates so that I can prioritize tutoring resources.
- **As a student**, I want proactive outreach from my instructor before I fail so that I can get help before it is too late.

## 5. Functional Requirements

- **FR-1.** The system MUST compute an at-risk score (0–100) for every active enrollment nightly using a weighted rule set including: missing assignments (weight 35 %), below-threshold quiz average (25 %), login inactivity > 7 days (25 %), and declining grade trend (15 %).
- **FR-2.** The system MUST create an `analytics.at_risk_alerts` record when a student's score exceeds `AT_RISK_THRESHOLD` (default 60) and no unresolved alert exists for that enrollment.
- **FR-3.** The system MUST send an email notification to the course instructor when a new at-risk alert is created (using email service, 6.2), with the student's name, current score, and primary risk factors.
- **FR-4.** Instructors MUST be able to view, dismiss, or snooze (7-day or 14-day) alerts for individual students.
- **FR-5.** Instructors MUST be able to add a note to an alert (e.g., "Student contacted — plans to submit by Friday").
- **FR-6.** The system MUST provide a `GET /courses/:id/at-risk` endpoint returning all active at-risk students sorted by score descending.
- **FR-7.** The at-risk scoring job MUST be idempotent: re-running it for the same date does not create duplicate alerts.
- **FR-8.** Admins MUST be able to configure the `AT_RISK_THRESHOLD` and individual signal weights per tenant.

## 6. Non-Functional Requirements

- **Performance** — Nightly scoring job completes within 30 min for a tenant with 10 000 active enrollments. Individual score recompute on-demand < 200 ms.
- **Security** — At-risk data accessible only to the course instructor, coordinator, and admin. Student MUST NOT see their own at-risk score. Alert notes treated as private instructor data (same rules as 9.1 instructor notes).
- **Privacy & Compliance** — FERPA: at-risk scores are education records; access logged. GDPR: at-risk data deleted on account erasure. Scoring logic must be documentable for institutional review.
- **Accessibility** — At-risk panel: color-coded severity uses icon + text labels (not color alone). Alert list keyboard-navigable. WCAG 2.1 AA.
- **Scalability** — Batch job processes enrollments in chunks of 500; parallelizable across job workers.
- **Reliability** — Job failure alerts admin via 17.7 observability; previous night's scores remain valid.
- **Observability** — Metrics: `at_risk_alerts_created_total`, `at_risk_alerts_resolved_total`, `at_risk_score_avg` (by course). Log student_id, course_id, score, top_risk_factor on each compute.
- **Maintainability** — Scoring engine in `server/src/services/at_risk_scoring.rs`; job in `server/src/workers/at_risk_job.rs`.
- **Internationalization** — Email templates and alert labels externalised.
- **Backward compatibility** — Feature is purely additive; no changes to grade or audit tables.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student has missed 3 assignments and not logged in for 10 days *When* the nightly job runs *Then* their at-risk score exceeds 60, an alert record is created, and an email is sent to the instructor within 15 minutes of job completion.
- **AC-2.** *Given* an instructor dismisses an alert *When* the nightly job next runs *Then* no new alert is created for that student until their score exceeds the threshold again on a subsequent date.
- **AC-3.** *Given* a student submits all missing work and logs in daily *When* the nightly job runs *Then* their score drops below the threshold, the existing alert is auto-resolved, and the instructor is notified.
- **AC-4.** *Given* a course has 200 students with 40 at-risk *When* the instructor opens the "At-Risk" tab *Then* 40 students are listed sorted by score descending with the top risk factor shown as a badge.
- **AC-5.** *Given* the nightly job runs twice on the same date (e.g., due to retry) *When* the second run completes *Then* no duplicate alert records are created.
- **AC-6.** *Given* a student's account is deleted under a GDPR erasure *When* the deletion job runs *Then* all at-risk alert records for that user are deleted.

## 8. Data Model

```sql
-- server/migrations/085_at_risk_alerts.sql
CREATE TABLE analytics.at_risk_scores (
  enrollment_id UUID NOT NULL,
  computed_date DATE NOT NULL,
  score         REAL NOT NULL CHECK (score BETWEEN 0 AND 100),
  missing_pct   REAL,
  quiz_avg      REAL,
  days_inactive INTEGER,
  grade_trend   REAL,
  PRIMARY KEY (enrollment_id, computed_date)
);

CREATE TYPE analytics.alert_status AS ENUM ('active', 'dismissed', 'snoozed', 'resolved');

CREATE TABLE analytics.at_risk_alerts (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  enrollment_id UUID NOT NULL,
  triggered_date DATE NOT NULL,
  score         REAL NOT NULL,
  status        analytics.alert_status NOT NULL DEFAULT 'active',
  snooze_until  DATE,
  notes         TEXT,
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  resolved_at   TIMESTAMPTZ,
  UNIQUE (enrollment_id, triggered_date)
);
CREATE INDEX ON analytics.at_risk_alerts (enrollment_id, status);
```

## 9. API Surface

| Method | Path | Auth | Description |
|---|---|---|---|
| `GET` | `/courses/:id/at-risk` | Instructor JWT | List active at-risk alerts sorted by score |
| `PATCH` | `/courses/:id/at-risk/:alert_id` | Instructor JWT | Update status (dismiss / snooze / resolve) or add note |
| `GET` | `/courses/:id/enrollments/:eid/at-risk-history` | Instructor JWT | Historical scores for one student |
| `POST` | `/admin/at-risk/run` | Admin JWT | Manually trigger scoring job for a course |
| `PUT` | `/admin/at-risk/config` | Admin JWT | Update threshold and signal weights |

## 10. UI / UX

- New "At-Risk" tab in the instructor course panel (sidebar or tabs on instructor dashboard).
- Alert list: student avatar, name, at-risk score (color-coded: orange 60–79, red 80+), primary risk factor badge (e.g., "5 missing", "Inactive 12d", "Failing avg"), action menu (Dismiss, Snooze, Add Note, View Progress).
- Dismissed/snoozed alerts shown in a collapsible "Resolved" section.
- Empty state: "No at-risk students — all students are on track."
- Instructor email: plain text + HTML template listing the student's name, score, 3 top risk factors, and a deep link to the student's progress page.
- Accessibility: score badge communicates level via icon + text, not color alone. Alert list `role="list"`. Action menu keyboard-navigable.
- i18n keys: `atRisk.title`, `atRisk.score`, `atRisk.missingWork`, `atRisk.inactive`, `atRisk.dismiss`, `atRisk.snooze`.

## 11. AI / ML Considerations

- Initial scoring: rule-based weighted sum (transparent, auditable, no model training required).
- Future iteration: logistic regression or gradient boosting trained on historical `(score, outcome=passed/failed)` pairs from `course_grades`.
- Eval metric for future ML model: precision/recall at 0.7 threshold; target recall ≥ 0.80 (miss no at-risk student) with precision ≥ 0.60.
- PII handling: model trained on anonymized enrollment IDs only; no PII in feature vectors.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| False positive alerts overwhelm instructor | M | M | Default threshold 60 is conservative; per-course tuning available |
| Scoring bias against students with legitimate excuses | M | H | Alert is a prompt for human review, never automated action; document this in UI |
| FERPA: at-risk score disclosed to wrong party | L | H | RBAC check on every endpoint; integration test for cross-role access |
| Nightly job runtime grows with tenant size | M | M | Chunk processing + parallel workers; alert if job > 60 min |

## 15. Rollout Plan

- **Flag:** `FEATURE_AT_RISK_ALERTS=true` (default false).
- **Sequencing:** scoring service + job → schema → API → email template → UI tab → enable flag.
- **GA criteria:** alert email delivered within 15 min of job completion; false-positive rate < 20 % validated on pilot cohort.
- **Rollback:** disable flag; score records remain in DB but no alerts fired; email suppressed.

## 16. Test Plan

- **Unit** — Scoring formula: each signal computes correctly; weighted sum; threshold check.
- **Integration** — Seeded DB: student with 3 missing assignments → score > 60 → alert created → email queued.
- **End-to-end** — Playwright: seed at-risk student data, trigger job, open At-Risk tab, dismiss alert, verify alert hidden.
- **Security** — Student cannot access at-risk endpoints (403); cross-course instructor access blocked.
- **Accessibility** — axe scan of alert list; color-blind safe palette verified.
- **Performance** — Scoring job on 10 000 enrollments completes < 30 min.
- **Manual exploratory** — Verify email template rendering; test snooze expiry; verify resolve on resubmission.

## 17. Documentation & Training

- Instructor docs: "Understanding at-risk alerts — what triggers them, how to respond, how to dismiss."
- Admin docs: configuring threshold and signal weights; monitoring job health.
- Privacy note: at-risk scores are private to instructors; add to institution data governance documentation.

## 18. Open Questions

2. Do we implement a student-facing "Your progress is falling behind — here's what to do" nudge alongside the instructor alert?
3. What is the right default threshold (60 out of 100) vs. making it percent-based per course?
