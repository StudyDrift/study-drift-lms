## 1. Problem Statement

Instructors have no consolidated view of an individual student's activity and performance across a course. The gradebook (`CourseGradebook.tsx`) shows aggregate scores per assignment, but there is no single page that answers "How is this student doing overall?" — covering assignment completion, quiz scores, engagement trends, and missing work. K-12 instructors and HE professors both cite per-student dashboards as a top-three LMS capability; its absence is a BLOCKER for any instructor-of-record relationship.

## 2. Goals

- Provide a per-student progress page accessible to instructors from the gradebook and enrollment roster.
- Aggregate assignment completion, quiz performance, module progress, and engagement metrics into a single view.
- Surface missing and late work prominently.
- Enable instructors to add private notes about a student's progress.

## 3. Non-Goals

- Parent-facing progress view (covered by §5.10 parent portal).
- AI-driven at-risk scoring (covered by 9.2).
- Standards-based mastery breakdown (covered by 9.3).
- Grade passback to SIS (covered by §14.5).

## 4. Personas & User Stories

- **As an instructor**, I want to see all of a student's assignment scores and submission dates on one page so that I can have an informed progress conversation.
- **As an instructor**, I want to see which modules a student has viewed and for how long so that I can identify whether a struggling student is even looking at the content.
- **As an instructor**, I want to add a private note ("Called parent 4/17 — student plans to submit late") so that I have a record for my own reference.
- **As a student**, I want to see my own progress dashboard so that I can identify which areas need more attention before the final exam.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a `GET /courses/:id/enrollments/:enrollment_id/progress` endpoint returning assignment completion rate, average quiz score, module view count, last-active timestamp, and missing-work list.
- **FR-2.** The progress page MUST display a timeline of student activity (submissions, quiz attempts, content views) sorted chronologically with day-level granularity.
- **FR-3.** The page MUST show a summary bar: modules completed %, assignments submitted %, average grade, and last-seen date.
- **FR-4.** The missing / late work section MUST list every item past due with days overdue and current grade status.
- **FR-5.** Instructors MUST be able to add, edit, and delete private notes on a student's progress page; notes MUST NOT be visible to the student.
- **FR-6.** Students MUST be able to view their own progress page (read-only, without private instructor notes).
- **FR-7.** The system SHOULD show a simple line chart of quiz score trend over time.
- **FR-8.** All data aggregated for the progress view MUST be refreshed at most every 5 minutes via a materialized view or cached query result.

## 6. Non-Functional Requirements

- **Performance** — Progress page loads < 1 s p95 for a course with 200 students and 50 assignments (served from materialized view or cache).
- **Security** — Instructors may view progress only for students enrolled in their course. Students may view only their own progress. Admin can view any. Enforced via existing RBAC (`server/src/authz.rs`).
- **Privacy & Compliance** — FERPA: progress data is an education record; access logged in `user_audit`. Private instructor notes not accessible to students or parents without instructor consent. GDPR: deletable on erasure request.
- **Accessibility** — Summary bar uses `role="status"` with screen-readable percentage text. Charts have text alternatives. Timeline list keyboard-navigable. WCAG 2.1 AA.
- **Scalability** — Materialized view refreshed every 5 min; no on-demand full-table scan per page view.
- **Reliability** — If materialized view refresh fails, serve stale data with "as of {timestamp}" label; alert on staleness > 30 min.
- **Observability** — Metrics: `progress_page_views_total` (by role). Log `course_id`, `student_id`, `viewer_role` on each access.
- **Maintainability** — Progress query in `server/src/repos/student_progress.rs` (new); route in `server/src/routes/reports.rs`.
- **Internationalization** — Date/time formatted per user locale. Percentage labels externalised.
- **Backward compatibility** — Gradebook and enrollment list unaffected; progress page is an additive link.

## 7. Acceptance Criteria

- **AC-1.** *Given* an instructor clicks a student's name in the gradebook *When* the progress page loads *Then* they see: completion %, average quiz score, last active date, and a list of missing items — all within 1 s.
- **AC-2.** *Given* a student has submitted 3 of 5 assignments *When* the progress page loads *Then* the summary bar shows "60 % assignments submitted" and the missing-work list shows the 2 overdue items.
- **AC-3.** *Given* an instructor adds a private note "Student has accommodation for extra time" *When* the student views their own progress page *Then* the note section is not visible.
- **AC-4.** *Given* a student views their own progress page *When* the page loads *Then* they see their own scores and activity but no other students' data.
- **AC-5.** *Given* the materialized view last refreshed 25 minutes ago *When* any user loads the progress page *Then* data is served with a "Last updated 25 min ago" label and a background refresh is triggered.
- **AC-6.** *Given* a quiz score trend chart is rendered *When* a screen-reader user navigates to the chart *Then* a text table alternative is available describing each data point.

## 8. Data Model

```sql
-- server/migrations/084_student_progress_view.sql
CREATE MATERIALIZED VIEW analytics.student_progress AS
SELECT
  e.id AS enrollment_id,
  e.course_id,
  e.user_id,
  COUNT(DISTINCT sa.id) FILTER (WHERE sa.submitted_at IS NOT NULL) AS assignments_submitted,
  COUNT(DISTINCT a.id) AS assignments_total,
  AVG(qa.score) AS avg_quiz_score,
  MAX(ua.created_at) AS last_active_at
FROM course.enrollments e
LEFT JOIN course.module_assignments a ON a.course_id = e.course_id
LEFT JOIN course.assignment_submissions sa ON sa.assignment_id = a.id AND sa.user_id = e.user_id
LEFT JOIN course.quiz_attempts qa ON qa.user_id = e.user_id
LEFT JOIN user_audit ua ON ua.user_id = e.user_id
GROUP BY e.id, e.course_id, e.user_id;

CREATE UNIQUE INDEX ON analytics.student_progress (enrollment_id);

CREATE TABLE analytics.instructor_notes (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  enrollment_id UUID NOT NULL,
  author_id     UUID NOT NULL REFERENCES user_account(id),
  note_text     TEXT NOT NULL,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON analytics.instructor_notes (enrollment_id);
```

## 10. UI / UX

- New page: `/lms/courses/:id/students/:eid/progress` — linked from gradebook row and enrollment roster.
- Layout: summary card (4 stats), then tabs: Overview | Activity Timeline | Assignments | Quizzes | Notes.
- Summary card: completion %, average grade badge (color-coded), last active relative time, missing items count.
- Activity timeline: virtualized list of events (icon + label + date). Filter by type (submissions, quizzes, views).
- Assignments tab: table with columns Item, Due, Submitted, Grade, Status (colored badge).
- Quizzes tab: table with attempt date, score, time spent; line chart of score over time.
- Notes tab (instructor only): note cards with create/edit/delete actions.
- Accessibility: summary stats in `<dl>` list; chart has `<caption>` and `<table>` fallback; all tabs follow ARIA tab pattern.
- Mobile: single-column stacked layout; summary stats in a 2×2 grid.
- i18n keys: `progress.submitted`, `progress.missing`, `progress.lastActive`, `progress.avgScore`, `progress.notes`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Materialized view refresh causes lock contention | M | M | Use `REFRESH MATERIALIZED VIEW CONCURRENTLY` |
| FERPA violation: showing one student's data to another | L | H | RBAC check on every request; integration test for cross-student access |
| Progress data stale if scheduled refresh job fails | M | M | Alert on staleness > 30 min; manual refresh API endpoint |

## 15. Rollout Plan

- **Flag:** `FEATURE_STUDENT_PROGRESS=true` (default false).
- **Sequencing:** materialized view migration → repos layer → API → UI → link from gradebook → enable flag.
- **GA criteria:** page loads < 1 s for 95th percentile course; instructor notes functional; FERPA access test passing.
- **Rollback:** disable flag; gradebook and enrollment list unchanged.

## 16. Test Plan

- **Unit** — Progress aggregation queries: correct counts for submitted/missing/avg.
- **Integration** — DB: materialized view refresh after grade insert; notes CRUD.
- **End-to-end** — Playwright: instructor opens progress page for a student, verifies summary stats match gradebook, adds note, refreshes — note persists.
- **Security** — Student A cannot access Student B's progress page (403); unenrolled instructor cannot access course (403).
- **Accessibility** — axe scan; chart text alternative rendered; tab panel keyboard navigation.
- **Performance** — k6: 100 concurrent progress page loads on a 1000-student course; p95 < 1 s.
- **Manual exploratory** — Verify missing-work list accuracy; verify activity timeline completeness.

## 17. Documentation & Training

- Instructor docs: "Viewing individual student progress — walkthrough, adding notes, interpreting the timeline."
- Student docs: "Your progress page — understanding your completion percentage and missing assignments."
- Admin docs: refreshing the progress materialized view manually; audit log access.

## 18. Open Questions

2. Should the activity timeline include content-view events from `user_audit`, or only gradeable events?
3. Do we implement week-over-week engagement delta (to support 9.2) in this view or separately?
