## 1. Problem Statement

Many HE institutions run institutional Learning Record Stores (LRS) and require their LMS to emit standards-based learning events so that institutional analytics, learning engineering teams, and accreditation bodies can consume activity data. Lextures emits nothing to external systems. Without IMS Caliper 1.2 or ADL xAPI 1.0.3 support, Lextures cannot integrate with any institutional analytics warehouse, cannot participate in the emerging learning engineering ecosystem, and fails a common HE IT procurement checklist item.

**Cross-link:** 9.6 xAPI ā†” Caliper integrations; also consumed by 8.12 (H5P xAPI receiver).

## 2. Goals

- Emit IMS Caliper Analytics 1.2 events for key learning interactions (course enrollment, content view, quiz attempt, assignment submission, grade).
- Emit ADL xAPI 1.0.3 statements to a configurable external LRS endpoint.
- Store all emitted events locally in a `analytics.xapi_statements` table as an internal LRS.
- Provide an admin UI to configure external LRS endpoints with credential management.

## 3. Non-Goals

- Building a full-featured LRS (SCORM Cloud, Learning Locker) ā€” internal store is append-only for querying, not a standards-compliant LRS server.
- Real-time streaming to third-party BI tools (batch export, deferred to 9.8).
- xAPI conformance certification (LRS receiver conformance is not in scope for the initial emitter).

## 4. Personas & User Stories

- **As an institutional learning analytics team member**, I want every quiz attempt to generate an xAPI statement so that my LRS can build adaptive recommendations.
- **As an IT administrator**, I want to configure Lextures to forward Caliper events to our institutional sensor network so that our student success platform receives data.
- **As a compliance officer**, I want all learning events stored immutably in the system so that I have an audit trail for accreditation.
- **As a developer integrating Lextures**, I want a documented list of all emitted Caliper profiles and xAPI verbs so that I can build conformant integrations.

## 5. Functional Requirements

- **FR-1.** The system MUST emit **IMS Caliper 1.2** events for: `SessionEvent.LoggedIn`, `NavigationEvent.NavigatedTo` (content views), `AssessmentItemEvent.Completed`, `GradeEvent.Graded`, `AssignmentSubmittedEvent`, `CourseEnrollmentEvent`.
- **FR-2.** The system MUST emit corresponding **ADL xAPI 1.0.3** statements using standard verbs: `launched`, `experienced`, `answered`, `passed`, `failed`, `completed`, `submitted`, `received`.
- **FR-3.** All events and statements MUST be stored in `analytics.xapi_statements` with the full JSON body and a unique statement_id (UUID per xAPI spec).
- **FR-4.** When an external LRS endpoint is configured (`POST /admin/lrs-config`), the system MUST forward statements asynchronously via the background job queue (17.3) using HTTP POST with Basic or OAuth 2.0 auth per xAPI spec Ā§7.
- **FR-5.** Statement forwarding MUST be retried up to 5 times with exponential back-off on network error; failed statements are moved to a dead-letter queue and surfaced in admin panel.
- **FR-6.** Administrators MUST be able to configure multiple LRS endpoints per tenant with per-endpoint enable/disable toggle.
- **FR-7.** An instructor MUST be able to preview the xAPI/Caliper statements generated by their course via a "View Event Log" page.
- **FR-8.** Personally identifiable information in actor fields MUST be hashed or anonymized when `LRS_ANONYMIZE_ACTORS=true` to support privacy-preserving LRS scenarios.

## 6. Non-Functional Requirements

- **Performance** ā€” Event emission to local DB MUST add < 5 ms to any user-facing request (async, non-blocking). LRS forwarding is async background job; no request latency impact.
- **Security** ā€” LRS credentials stored encrypted in secrets manager (17.17). xAPI actor `mbox` uses hashed email when anonymization is enabled. Statement immutability: no UPDATE on `xapi_statements`.
- **Privacy & Compliance** ā€” FERPA: event data transmitted to external LRS only with institutional data sharing agreement. GDPR: events include user_id; erasure replaces actor PII with pseudonymous hash. Consent toggle available per 10.17.
- **Accessibility** ā€” Admin config UI keyboard-navigable. Event log table accessible. WCAG 2.1 AA.
- **Scalability** ā€” Statement storage table partitioned by `created_at` (monthly partitions). Up to 1 M statements/day supported.
- **Reliability** ā€” Local DB write happens synchronously; LRS forwarding async. LRS outage does not affect primary user experience.
- **Observability** ā€” Metrics: `xapi_statements_emitted_total`, `caliper_events_emitted_total`, `lrs_forwarding_failed_total`. Log statement_id, verb, actor_hash, object_id.
- **Maintainability** ā€” Caliper emitter in `server/src/services/caliper.rs`; xAPI emitter in `server/src/services/xapi.rs`; shared event bus in `server/src/events/`.
- **Internationalization** ā€” Caliper/xAPI payload strings use English (standard per specs). UI labels externalised.
- **Backward compatibility** ā€” `user_audit` table unchanged; xAPI/Caliper is an additional event layer.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student completes a quiz attempt *When* the attempt is graded *Then* an xAPI statement with verb `passed` or `failed`, actor = hashed email, object = quiz IRI, result.score present, is stored in `analytics.xapi_statements` within 2 s.
- **AC-2.** *Given* an external LRS is configured *When* a statement is stored *Then* a background job forwards it to the LRS endpoint and logs the HTTP response.
- **AC-3.** *Given* the LRS endpoint returns 503 *When* the forwarding job retries 5 times *Then* the statement is moved to the dead-letter queue and an admin alert fires.
- **AC-4.** *Given* `LRS_ANONYMIZE_ACTORS=true` *When* a statement is generated for user alice@example.com *Then* the actor `mbox` field contains a SHA-256 hash of the email, not the plaintext address.
- **AC-5.** *Given* an instructor opens the Event Log for their course *When* the page loads *Then* all xAPI statements for the last 7 days are listed with verb, timestamp, and object title; clicking a row shows the full JSON body.
- **AC-6.** *Given* a Caliper `GradeEvent.Graded` is emitted *When* validated against the IMS Caliper 1.2 JSON-LD schema *Then* no validation errors are produced.

## 8. Data Model

```sql
-- server/migrations/089_xapi_statements.sql
CREATE TABLE analytics.xapi_statements (
  statement_id    UUID PRIMARY KEY,
  actor_hash      TEXT NOT NULL,
  verb_id         TEXT NOT NULL,
  object_id       TEXT NOT NULL,
  object_type     TEXT,
  result_score    REAL,
  result_success  BOOLEAN,
  context_course_id UUID REFERENCES course.courses(id),
  stored_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  full_json       JSONB NOT NULL
) PARTITION BY RANGE (stored_at);

CREATE TABLE analytics.xapi_statements_2026_04 PARTITION OF analytics.xapi_statements
  FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');

CREATE TABLE analytics.lrs_dead_letter (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  statement_id    UUID NOT NULL,
  lrs_endpoint_id UUID NOT NULL,
  last_error      TEXT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Caliper JSON-LD validation errors at institutional LRS | M | M | Validate emitted events against IMS Caliper schemas in CI |
| Statement volume causes Postgres table bloat | M | M | Partitioned table; automatic old-partition archival job |
| LRS credentials exposed in logs | M | H | Redact credential fields from all log lines; store only in secrets manager |
| GDPR erasure incompatible with xAPI statement immutability | M | H | Pseudonymize actor field on erasure (replace with hash); do not delete statement |

## 15. Rollout Plan
