## 1. Problem Statement

The `user_audit` table (migration 029) records page navigation events, but there are no derived engagement metrics: how long a student spent on a content page, what percentage of a video they watched, how frequently they log in, or how far they scrolled through a reading. These metrics are essential for at-risk scoring (9.2), instructor insights (9.10), and self-learner engagement loops. Without them, "adaptive" platform claims cannot be substantiated by behavioral data.

## 2. Goals

- Capture time-on-task (active session time per content item), login frequency, video watch percentage per video per student, and scroll depth on content pages.
- Aggregate these metrics into per-student, per-course summaries accessible to instructors and students.
- Store engagement events efficiently without overwhelming the Postgres audit table.
- Provide a visual engagement timeline on the student progress page (9.1).

## 3. Non-Goals

- Real-time analytics dashboards (initial release: aggregated daily; near-real-time in a follow-on).
- Screen recording or eye-tracking.
- External analytics provider integration (Amplitude, Mixpanel) — internal only.

## 4. Personas & User Stories

- **As an instructor**, I want to see that student X spent only 2 minutes on a 20-minute reading so that I understand why they did poorly on the related quiz.
- **As an instructor**, I want to see that 60 % of students stopped watching a lecture video at the 15-minute mark so that I know to restructure that section.
- **As a student**, I want to see my own weekly login frequency so that I can hold myself accountable to a study schedule.
- **As an at-risk algorithm** (9.2), I want the days-since-last-login signal to be accurate to the hour so that I can flag students who have gone silent.

## 5. Functional Requirements

- **FR-1.** The frontend MUST send a `heartbeat` event every 30 seconds while a user has a content page in the foreground (using Page Visibility API); the backend MUST accumulate these into `time_on_task_seconds` per `(user_id, content_item_id, date)`.
- **FR-2.** The video player (8.3) MUST emit `video.progress` events at 10-second intervals reporting `current_second`, `duration_second`, and `percent_watched`; backend stores the maximum `percent_watched` per `(user_id, video_object_id)`.
- **FR-3.** The frontend MUST report scroll depth (0–100 % of page height) when a user leaves a content page; backend stores max scroll depth per `(user_id, content_item_id)`.
- **FR-4.** The system MUST derive `login_frequency` as logins per 7-day rolling window from `user_audit` events with `action = 'login'`.
- **FR-5.** The system MUST provide a `GET /courses/:id/enrollments/:eid/engagement` endpoint returning: avg_time_on_task_per_session, logins_last_7_days, avg_video_watch_pct, avg_scroll_depth.
- **FR-6.** The system MUST aggregate a course-level "Video Drop-off" report showing the median watch-stop second across all students for each video.
- **FR-7.** Engagement events MUST be batched client-side and sent as a bulk POST every 60 seconds to avoid per-event HTTP overhead.

## 6. Non-Functional Requirements

- **Performance** — Batch event POST (up to 50 events) processes in < 100 ms server-side. Engagement summary query < 500 ms p95 (pre-aggregated).
- **Security** — Engagement events tied to authenticated user JWT. Bulk event endpoint rate-limited at 1 batch/minute/user. Cross-user access blocked.
- **Privacy & Compliance** — FERPA: engagement data is an education record. GDPR: deletable on erasure. No engagement data sold or shared externally. Disclosure in privacy policy that time-on-task and scroll depth are tracked.
- **Accessibility** — Heartbeat and scroll events are background JS — no UI impact. Engagement visualizations (charts) have text alternatives.
- **Scalability** — Engagement events are high-frequency; write to a dedicated `analytics.engagement_events` append-only table, then aggregate via nightly job into `analytics.engagement_summaries`. Partition `engagement_events` by month.
- **Reliability** — Client batches events in localStorage; if POST fails, retry on next interval. Lost events acceptable (stats are approximate).
- **Observability** — Metrics: `engagement_events_received_total`, `engagement_batch_size_avg`, `video_watch_pct_avg`. Log course_id, item_type on each batch.
- **Maintainability** — Engagement collector in `server/src/routes/engagement.rs` (new); aggregation job in `server/src/workers/engagement_aggregator.rs`.
- **Internationalization** — Duration formatting (minutes/seconds) locale-aware in UI.
- **Backward compatibility** — `user_audit` table unchanged; engagement is a parallel supplemental table.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student reads a content page for 3 minutes with the tab in focus *When* the engagement API receives 6 heartbeats *Then* `time_on_task_seconds` for that item accumulates to 180 s (±30 s).
- **AC-2.** *Given* a student watches 75 % of a video and closes the tab *When* the video player sends its final progress event *Then* `video_watch_pct` stored is 75 (not reset to 0 on next view if they re-watch from beginning and stop at 60 %).
- **AC-3.** *Given* a student has logged in 5 times in the last 7 days *When* the engagement summary endpoint is called *Then* `logins_last_7_days = 5`.
- **AC-4.** *Given* an instructor opens the "Video Drop-off" report for a 30-minute lecture *When* the report loads *Then* a histogram shows the distribution of watch-stop seconds and the median stop point is highlighted.
- **AC-5.** *Given* a student closes a content page after scrolling to 40 % *When* the scroll depth event is received *Then* the max stored scroll depth for that item is ≥ 40.
- **AC-6.** *Given* a student's account is deleted *When* the GDPR erasure job runs *Then* all rows in `analytics.engagement_events` for that user are deleted.

## 8. Data Model

```sql
-- server/migrations/090_engagement_events.sql
CREATE TABLE analytics.engagement_events (
  id              BIGSERIAL,
  user_id         UUID NOT NULL REFERENCES user_account(id) ON DELETE CASCADE,
  course_id       UUID REFERENCES course.courses(id),
  item_id         UUID,
  item_type       TEXT,        -- 'content_page' | 'video' | 'quiz' | 'assignment'
  event_type      TEXT NOT NULL,  -- 'heartbeat' | 'video_progress' | 'scroll_depth' | 'login'
  value           REAL,
  occurred_at     TIMESTAMPTZ NOT NULL DEFAULT now()
) PARTITION BY RANGE (occurred_at);

CREATE TABLE analytics.engagement_summaries (
  enrollment_id           UUID NOT NULL,
  summary_date            DATE NOT NULL,
  total_time_on_task_s    INTEGER NOT NULL DEFAULT 0,
  logins_last_7_days      INTEGER NOT NULL DEFAULT 0,
  avg_video_watch_pct     REAL,
  avg_scroll_depth        REAL,
  updated_at              TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (enrollment_id, summary_date)
);
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Heartbeat events flood Postgres | H | M | Write to time-series buffer (Redis sorted set) then flush to DB in bulk; or use partitioned append-only table |
| Page Visibility API unreliable on mobile | M | M | Use `visibilitychange` + `beforeunload` + periodic heartbeat; accept ±30 s accuracy |
| Engagement data used punitively against students | M | H | Never expose raw engagement to students as a "score"; frame as self-reflection tool |
| GDPR erasure of high-volume event table | M | M | `ON DELETE CASCADE` on user_id; partition-level deletion for old data |

## 15. Rollout Plan

- **Flag:** `FEATURE_ENGAGEMENT_TRACKING=true` (default false).
- **Sequencing:** engagement events table + partitions → batch receiver API → frontend tracker → aggregation job → UI widgets → enable flag.
- **GA criteria:** time-on-task accurate within ±60 s; video watch % stored; login frequency correct.
- **Rollback:** disable flag; tracker removed from frontend; event table preserved for future re-enable.

## 16. Test Plan

- **Unit** — Heartbeat accumulation arithmetic; scroll depth max tracking; login count derivation.
- **Integration** — Batch POST: 20 events stored correctly; aggregation job produces correct summary.
- **End-to-end** — Playwright: simulate 3 heartbeats on content page, verify time_on_task in API response.
- **Security** — Cross-user engagement access blocked; unauthenticated batch POST rejected.
- **Accessibility** — axe scan of engagement charts; text table alternatives present.
- **Performance** — 1 000 batch POSTs/s sustained without Postgres contention.
