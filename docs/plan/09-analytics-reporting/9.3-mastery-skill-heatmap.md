## 1. Problem Statement

There is no visual representation of what a student knows and does not know at the skill or concept level. Instructors cannot quickly identify that "15 of 30 students have not mastered quadratic equations" without manually reviewing individual quiz responses. Self-learners have no way to see their knowledge profile. The mastery heatmap connects the adaptive learning substrate (§1 concept graph) to a concrete instructor and student-facing analytics surface, turning abstract learner-model data into actionable insights.

**Cross-link:** 9.3 mastery heatmap depends on §1 concept graph (features 1.1 and 1.2).

## 2. Goals

- Render a heatmap of skill/concept mastery per student or per cohort, sourced from `learner_concept_states` (§1.1/1.2).
- Provide a class-wide aggregated heatmap showing which skills the cohort has mastered and which are weak.
- Allow instructors to drill from a weak-skill cell to the students struggling with that skill.
- Allow students to view their own mastery heatmap as a self-assessment tool.

## 3. Non-Goals

- Defining the concept graph itself (that is §1.2).
- Adaptive path re-routing based on heatmap data (that is §1.4).
- Standards alignment to Common Core / NGSS codes in the heatmap (deferred to §1.3).

## 4. Personas & User Stories

- **As an instructor**, I want to see a grid of students × skills colored by mastery level so that I can identify which skills need re-teaching before the midterm.
- **As a student**, I want a color-coded map of my own skill mastery so that I know which topics to study before the exam.
- **As an admin**, I want a course-level skill heatmap to identify systemic curriculum gaps across all sections so that I can recommend additional instructional support.
- **As a self-learner**, I want to see my mastery profile so that I can direct my study time to the lowest-mastery skills.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a `GET /courses/:id/analytics/mastery-heatmap` endpoint returning a 2D matrix of `enrollment_id × concept_id` with mastery scores (0.0–1.0) and confidence levels.
- **FR-2.** The heatmap MUST aggregate data from the `learner_concept_states` table (introduced by §1.1) for all students enrolled in the course.
- **FR-3.** The class heatmap MUST show a per-concept column with: mean mastery, % students mastered (≥ 0.8), % students at-risk (< 0.4), and a sparkline of mastery trend over the last 4 weeks.
- **FR-4.** Clicking a heatmap cell MUST navigate to a filtered student list showing students with mastery in that concept in the clicked range.
- **FR-5.** The student self-view MUST show only their own row, with a radar/spider chart alternative view for a different visual presentation of the same data.
- **FR-6.** The system MUST refresh heatmap data from `learner_concept_states` on-demand and on a scheduled 4-hour cadence.
- **FR-7.** Concepts with no assessment data for a student MUST display as "Not assessed" (grey) rather than zero mastery.

## 6. Non-Functional Requirements

- **Performance** — Heatmap API for a 200-student, 30-concept course loads < 800 ms p95 (denormalized cache).
- **Security** — Instructors see all students' rows; students see only their own. RBAC from `server/src/authz.rs`.
- **Privacy & Compliance** — FERPA: mastery data is a student education record. GDPR: deletable on erasure. Data in heatmap not shared externally without consent.
- **Accessibility** — Heatmap grid has full text alternative: `aria-label="Concept X mastery for Student Y: 0.75"`. Color scale must not rely solely on hue (add pattern or numerical label). WCAG 2.1 SC 1.4.1.
- **Scalability** — Materialized heatmap cache; do not compute per-request for large courses.
- **Reliability** — If `learner_concept_states` has no data (§1 not yet populated), display an empty-state message rather than a broken grid.
- **Observability** — Log heatmap view events with course_id and role. Metric: `heatmap_load_duration_ms`.
- **Maintainability** — Heatmap query in `server/src/repos/mastery_heatmap.rs` (new); UI in `clients/web/src/pages/lms/Reports.tsx` (new tab).
- **Internationalization** — Mastery level labels ("Mastered", "Developing", "Beginning", "Not assessed") externalised.
- **Backward compatibility** — Graceful empty state if §1 concept graph not deployed.

## 7. Acceptance Criteria

- **AC-1.** *Given* 20 students have concept state data for 10 concepts *When* the instructor opens the heatmap *Then* a 20×10 grid renders with cells colored green (mastered), amber (developing), red (at-risk), grey (not assessed) within 1 s.
- **AC-2.** *Given* an instructor clicks a red cell for "Quadratic Equations" *When* the drill-down opens *Then* a list of students with mastery < 0.4 in that concept is shown with their at-risk score.
- **AC-3.** *Given* a student opens their own mastery view *When* the page loads *Then* they see only their own row, and a radar chart option is available as an alternative view.
- **AC-4.** *Given* a screen-reader user navigates the heatmap table *When* they tab to a cell *Then* the screen reader announces "Quadratic Equations — Alice Johnson — Mastery 75 %".
- **AC-5.** *Given* §1 concept graph is not yet deployed and `learner_concept_states` is empty *When* the heatmap loads *Then* an empty-state panel shows "No skill data yet — this view requires adaptive quiz data."
- **AC-6.** *Given* the heatmap cache is older than 4 hours *When* the page loads *Then* a background refresh is triggered and "Refreshing…" is shown in the header.

## 8. Data Model

```sql
-- server/migrations/086_mastery_heatmap_cache.sql
-- Depends on §1.1/1.2 tables: concepts, learner_concept_states
CREATE MATERIALIZED VIEW analytics.mastery_heatmap AS
SELECT
  lcs.enrollment_id,
  lcs.concept_id,
  lcs.mastery_score,
  lcs.confidence,
  lcs.updated_at
FROM adaptive.learner_concept_states lcs
JOIN course.enrollments e ON e.id = lcs.enrollment_id;

CREATE UNIQUE INDEX ON analytics.mastery_heatmap (enrollment_id, concept_id);
CREATE INDEX ON analytics.mastery_heatmap (concept_id, mastery_score);
```

No new base tables: all data sourced from §1 learner model.

## 10. UI / UX

- New "Mastery Heatmap" tab on the course analytics page (accessible from `Reports.tsx`).
- Heatmap: sticky student-name column on left, concept names across top. Cells are colored squares with a numerical label (0–100 %). Tooltip on hover: full mastery score and last updated date.
- Color scale: green (≥ 80), yellow-green (60–79), amber (40–59), red (< 40), grey (not assessed). Pattern overlay for accessible mode.
- Class summary row at bottom: mean mastery per concept.
- Drill-down: slide-over panel listing affected students with a "Message" action linking to inbox.
- Student self-view: own row in context + toggle to radar chart (Chart.js or Recharts spider chart).
- Accessibility: table with `scope="col"` and `scope="row"` headers; cells have `aria-label`. Color-blind mode toggle replaces hue scale with numerical labels only.
- i18n keys: `mastery.heatmap`, `mastery.mastered`, `mastery.developing`, `mastery.notAssessed`, `mastery.refresh`.

## 11. AI / ML Considerations

Not applicable for rendering. Mastery scores are computed by the §1 learner model (Bayesian Knowledge Tracing or BKT). This feature only visualizes the output.

## 15. Rollout Plan
