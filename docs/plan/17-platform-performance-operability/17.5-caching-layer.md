## 1. Problem Statement

Every request to the Lextures API hits the database. Static assets, course content, and frequently-read collection endpoints have no caching layer — no HTTP `Cache-Control` headers, no Redis object cache, and no CDN for uploaded media. This results in unnecessary DB load, high latency for repeat content fetches, and unnecessarily high bandwidth costs for media files. At scale (semester start, large K-12 district rollout), the database will become the bottleneck within minutes.

## 2. Goals

- Add appropriate `Cache-Control`, `ETag`, and `Last-Modified` HTTP headers to all API endpoints.
- Implement a Redis object cache for frequently-read, rarely-changed data (course structure, enrollment lists, public course catalog).
- Configure DigitalOcean CDN (Spaces CDN) for uploaded media files (images, videos, PDFs).
- Implement Workbox/Service Worker caching for the React SPA assets (static JS/CSS bundles).
- Establish a cache invalidation strategy for each cached resource.

## 3. Non-Goals

- Application-level read-through cache for every DB query (too broad; only hot paths in scope).
- Distributed cache invalidation via CDC (Postgres logical replication) — overkill for this stage.
- Varnish or custom HTTP proxy caching.
- Edge caching of authenticated API responses (complex authorization + caching; partial pages only).

## 4. Personas & User Stories

- **As a student**, I want the course content page to load in under 1 second even when 500 students are viewing the same course simultaneously so that class time is not wasted waiting for the LMS.
- **As a self-learner browsing the public catalog**, I want the course listing to load fast even at peak time so that I am not deterred from signing up.
- **As a DevOps engineer**, I want uploaded media served from a CDN so that video playback does not saturate the app server's bandwidth.
- **As a frontend developer**, I want SPA assets to be cached by Service Workers so that repeat page loads are instant even on slow connections.

## 5. Functional Requirements

- **FR-1.** The system MUST set `Cache-Control: no-store` on all authenticated API responses containing user-specific data to prevent shared proxy caching.
- **FR-2.** The system MUST set `Cache-Control: public, max-age=3600, stale-while-revalidate=86400` with `ETag` on public course catalog and public course detail endpoints.
- **FR-3.** The system MUST implement a Redis cache for the following hot paths: course structure items (TTL 5 min), public course catalog page (TTL 10 min), enrollment roster per course (TTL 5 min), calendar feed per user (TTL 15 min, invalidated on assignment change).
- **FR-4.** The system MUST invalidate Redis cache keys when the underlying data changes: course structure updated → invalidate `course:{id}:structure`; enrollment added/removed → invalidate `course:{id}:enrollments`.
- **FR-5.** The system MUST configure DigitalOcean Spaces CDN for the `uploads` bucket so that all media file URLs use the CDN domain (not the app server).
- **FR-6.** The frontend build (`clients/web/`) MUST implement Workbox Service Worker with a cache-first strategy for static JS/CSS assets and a network-first strategy for API calls.
- **FR-7.** The system SHOULD set `Vary: Accept-Encoding` and enable Brotli/gzip compression on all API responses.
- **FR-8.** The system SHOULD implement `stale-while-revalidate` semantics in the Redis cache: serve the stale cached value while asynchronously refreshing it in the background.

## 6. Non-Functional Requirements

- **Performance** — Redis cache hit p95 latency ≤ 5 ms; cache-hit rate target > 80% for course structure endpoints during normal operation. CDN-served media requests should not touch the origin server.
- **Security** — `Cache-Control: no-store` on all JWT-gated routes. CDN origin authentication via signed URLs for private media (not world-readable).
- **Privacy & Compliance** — No PII cached in Redis without user-scoped cache key isolation. Cache keys scoped by `org_id` or `user_id` where data is org/user-specific.
- **Accessibility** — Service Worker cache must not return stale HTML that breaks the React app after a deployment; implement cache versioning.
- **Scalability** — Redis cache shared across all app instances (17.2); consistent cache behavior regardless of which instance handles the request.
- **Reliability** — Cache miss falls back to DB; never hard-fails. Redis unavailability treated as total cache miss, not an error response.
- **Observability** — Prometheus counters: `cache_hits_total{resource_type}`, `cache_misses_total{resource_type}`; gauge `cache_hit_rate` per resource; alert if hit rate drops below 50% sustained.
- **Maintainability** — Cache key naming convention documented: `{prefix}:{entity_type}:{id}:{variant}`; invalidation co-located with mutation in the same service function.
- **Internationalization** — Cache keys locale-aware where locale affects response content.
- **Backward compatibility** — Adding caching to an existing endpoint must not change the response shape or status codes.

## 7. Acceptance Criteria

- **AC-1.** *Given* 100 students load the same course structure simultaneously, *When* the Redis cache is populated after the first request, *Then* subsequent requests return within 10 ms (cache hit) and DB query count is ≤ 1 per 5-minute TTL window.
- **AC-2.** *Given* an instructor updates course structure, *When* the cache invalidation fires, *Then* the next request for that course structure returns the updated data (not stale).
- **AC-3.** *Given* a media file is stored in DO Spaces, *When* a student accesses the file URL, *Then* the response is served from the CDN edge (verified by `X-Cache: HIT` header) without touching the origin server.
- **AC-4.** *Given* the React SPA is loaded for the second time after initial caching by the Service Worker, *When* the network is offline, *Then* the static app shell loads from cache and displays an "offline" indicator.
- **AC-5.** *Given* an authenticated API endpoint returns user-specific data, *When* the response headers are inspected, *Then* `Cache-Control: no-store` is present.
- **AC-6.** *Given* Redis becomes unavailable, *When* a cached endpoint is called, *Then* the response is served from the DB with no error to the client (cache miss fallback).

## 8. Data Model

No DB schema changes. Redis cache key namespaces:

| Key Pattern | Content | TTL | Invalidated by |
|---|---|---|---|
| `cache:course:{id}:structure` | Serialized course items | 5 min | Course item create/update/delete |
| `cache:course:{id}:enrollments` | Enrollment list | 5 min | Enrollment change |
| `cache:catalog:page:{page}` | Public course catalog | 10 min | Course publish/unpublish |
| `cache:user:{id}:calendar` | iCal feed | 15 min | Assignment due date change |

## 9. API Surface

No new routes. Headers added to existing endpoints. Cache invalidation is internal (not an API concern).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Cache poisoning: wrong tenant's data served | M | H | Cache keys always include org/user scope; integration test cross-tenant isolation |
| Service Worker serves stale app after deployment | H | M | Cache versioning via build hash; `skipWaiting` on new SW version |
| Redis memory exhaustion | M | M | Set `maxmemory-policy allkeys-lru`; monitor memory usage; alert at 80% |
| Invalidation miss: stale data served post-mutation | M | M | Co-locate invalidation with mutation; integration test mutation → cache miss |

## 15. Rollout Plan

- **Feature flag:** `redis_cache_enabled` (OpenFeature, default `false`).
- **Phase 1:** HTTP headers (`Cache-Control`, `ETag`) on all endpoints — no Redis required; zero risk.
- **Phase 2:** Redis cache for hot paths; Service Worker for static assets.
- **Phase 3:** CDN for media files; stale-while-revalidate; GA.
- **Rollback:** Disable Redis cache flag; HTTP headers remain harmless; CDN configured as pass-through.

## 16. Test Plan

- **Unit** — Cache key generation; TTL calculation; ETag generation.
- **Integration** — Redis cache hit/miss behavior; invalidation on mutation; cross-tenant isolation.
- **End-to-end** — Playwright: load course twice; verify second load faster; service worker offline mode.
- **Security** — Cross-tenant cache isolation test; `Cache-Control: no-store` on authenticated routes.
- **Accessibility** — Offline indicator accessibility; service worker update notification accessible.
- **Performance / load** — k6: 500 VUs on course structure endpoint with cache enabled vs. disabled; compare DB query counts.
- **Manual exploratory** — QA: update course structure; verify stale cache served then invalidated; CDN hit header verification.

## 18. Open Questions

1. Should we use a read-through cache pattern (middleware intercepts all reads) or explicit cache calls in each service function?
2. Is a 5-minute TTL for course structure too aggressive (too much DB load on cache miss) or too conservative (stale data)?
3. Should the Service Worker use a network-first or cache-first strategy for API calls that are not cached (authenticated data)?
