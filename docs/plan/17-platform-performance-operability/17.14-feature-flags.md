## 1. Problem Statement

Lextures has no feature flag system. Every feature ships to 100% of users at once, making it impossible to dogfood, run A/B experiments, gradually roll out risky changes, or give beta access to specific organizations. The many "default `false`" flags referenced throughout this plan document are placeholders; without an actual flag system they are just environment variables. A proper feature flag system conforming to the OpenFeature spec enables safe, observable progressive rollouts with instant kill-switch capability.

## 2. Goals

- Implement an OpenFeature-compatible feature flag SDK in the Rust server and React frontend.
- Provide boolean, string, number, and JSON flag variants.
- Support targeting rules: flag enabled for specific org_ids, user_ids, percentage of users (by hash), or environment (staging/production).
- Store flag definitions in Postgres with a Redis cache for evaluation performance.
- Provide an admin UI for creating, enabling, disabling, and targeting flags without a code deployment.

## 3. Non-Goals

- A/B testing statistical analysis infrastructure (flags enable the experiment; analysis is separate).
- External feature flag services (LaunchDarkly, Split.io) — self-hosted is preferred for data sovereignty.
- Multi-variate testing beyond simple percentage rollout.
- Mobile SDK (deferred until native mobile app ships, 7.1).

## 4. Personas & User Stories

- **As a product manager**, I want to enable a new gradebook feature for 5% of organizations so that I can measure adoption and gather feedback before a full rollout.
- **As a developer**, I want to wrap a new AI feature in a flag so that I can deploy the code to production and enable it only for our internal test organization.
- **As an operations engineer**, I want to instantly disable a malfunctioning feature without a code deployment so that I can stop customer impact in under 30 seconds.
- **As an admin at a pilot organization**, I want to opt in to beta features through a settings page so that my team can test new capabilities early.

## 5. Functional Requirements

- **FR-1.** The system MUST implement an OpenFeature-spec–compatible evaluation API in Rust and TypeScript, with a shared flag definition store.
- **FR-2.** Flag evaluation MUST support the following context attributes: `org_id`, `user_id`, `user_role`, `environment` (staging/production).
- **FR-3.** The system MUST support targeting rules: (a) enabled for all, (b) enabled for a list of `org_id`s, (c) percentage rollout by `org_id` hash, (d) disabled for all.
- **FR-4.** Flag definitions MUST be stored in Postgres (`flags.definitions` table) and cached in Redis with a 30-second TTL; flag changes propagate to all app instances within 30 seconds.
- **FR-5.** The system MUST provide an admin UI at `/admin/flags` for creating, editing, and toggling flags.
- **FR-6.** The React frontend MUST evaluate flags via a `useFlag(flagKey)` hook backed by a lightweight SSE or polling endpoint, so that a flag change is reflected in the UI within 30 seconds without a page reload.
- **FR-7.** The system SHOULD log flag evaluation decisions to the audit log for compliance (which flags were evaluated for which user/org).
- **FR-8.** The system MAY support flag variations: string or JSON flags for configuration-style overrides (e.g., AI model selection per org).

## 6. Non-Functional Requirements

- **Performance** — Flag evaluation in Rust: p99 ≤ 1 ms (Redis cache hit). Frontend flag evaluation: p99 ≤ 5 ms (in-memory after initial fetch). Total flag load on Redis: < 1 KB per flag; < 100 active flags expected.
- **Security** — Flag definitions admin UI restricted to global admins. Flag evaluation results are not sensitive, but org-targeting lists are (contain customer org IDs); admin-only access.
- **Privacy & Compliance** — Flag evaluation log entries record `user_id`, `flag_key`, `result` — no PII. Audit trail for flag changes: who changed what, when.
- **Accessibility** — Admin flags page WCAG 2.1 AA.
- **Scalability** — Redis cache prevents per-request DB reads; 30-second TTL acceptable for flag freshness. Supports up to 1,000 active flags without performance degradation.
- **Reliability** — Redis unavailability falls back to DB evaluation; DB unavailability falls back to default flag values (fail-safe). No flag evaluation failure should cause a 500 error.
- **Observability** — Counter `feature_flag_evaluations_total{flag_key,result}`; useful for A/B experiment analysis.
- **Maintainability** — OpenFeature interface means the provider can be swapped to LaunchDarkly or Unleash later without changing calling code.
- **Internationalization** — Flag names and descriptions in English in admin UI; consumer-facing features controlled by flags use their own i18n.
- **Backward compatibility** — Adding a new flag does not affect existing flag evaluations; default values are always `false` for boolean flags unless explicitly set.

## 7. Acceptance Criteria

- **AC-1.** *Given* a flag `new_gradebook` is enabled for `org_id: abc-123`, *When* a user from `org_id: abc-123` evaluates the flag, *Then* `is_enabled(ctx)` returns `true`; for users from other orgs it returns `false`.
- **AC-2.** *Given* a flag is changed in the admin UI, *When* 30 seconds pass, *Then* all app instances and all active browser sessions reflect the new flag value without requiring a page reload or app restart.
- **AC-3.** *Given* a `percentage_rollout=10` targeting rule, *When* 1,000 different `org_id`s evaluate the flag, *Then* approximately 100 ± 20 receive `true` (stable: same org always gets the same result).
- **AC-4.** *Given* Redis is unavailable, *When* a flag is evaluated, *Then* the default flag value is returned and no error is thrown.
- **AC-5.** *Given* an admin disables a flag via the admin UI, *When* a user tries to access the guarded feature, *Then* the feature is inaccessible within 30 seconds.
- **AC-6.** *Given* a flag evaluation occurs, *When* the audit log is queried, *Then* a record exists with `flag_key`, `org_id`, `user_id`, `result`, and `evaluated_at`.

## 8. Data Model

```sql
-- server/migrations/085_feature_flags.sql
CREATE TABLE flags.definitions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    flag_key        TEXT NOT NULL UNIQUE,
    description     TEXT,
    flag_type       TEXT NOT NULL DEFAULT 'boolean',  -- boolean|string|number|json
    default_value   JSONB NOT NULL DEFAULT 'false',
    rules           JSONB NOT NULL DEFAULT '[]',
    -- rules: [{"type":"org_ids","value":["uuid1"]},{"type":"percentage","value":10}]
    enabled         BOOLEAN NOT NULL DEFAULT false,
    created_by      UUID REFERENCES user.users(id),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON flags.definitions (flag_key);
```

## 9. API Surface

| GET | `/admin/flags` | admin JWT | List all flags |

## 10. UI / UX

- **Admin → Platform → Feature Flags:** table of flag keys, descriptions, enabled status, last updated, edit button.
- Flag edit form: key, description, type, default value, targeting rules editor (JSON or visual builder).
- Percentage rollout rule: slider 0–100%.
- Org targeting rule: multi-select of org names.
- React `useFlag('flag_key')` hook: polls `/api/v1/flags/evaluate` on mount; re-evaluates every 30 s.
- Mobile: admin flags page responsive; org targeting scrollable.

## 11. AI / ML Considerations

Feature flags are used to progressively roll out AI features (16.7, 19.x). No AI within the flag evaluation itself.

## 12. Integration Points

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Flag evaluation hot path becomes a bottleneck | L | M | Redis cache; in-memory pre-fetch for active flags |
| Flag proliferation: hundreds of stale flags | M | M | Flag expiry date field; cleanup scheduled job |
| Percentage rollout not stable (user gets different result each time) | L | H | Deterministic hash of `org_id + flag_key`; same org always gets same result |
| Admin UI allows enabling a dangerous flag accidentally | M | M | Confirmation dialog for enabling flags in production; audit log |

## 15. Rollout Plan

- **Phase 1:** Backend flag evaluation engine; Postgres schema; Redis cache; admin CRUD UI.
- **Phase 2:** Frontend `useFlag` hook; SSE stream or polling; flag evaluation for existing "env var" flags migrated.
- **Phase 3:** Percentage rollout rules; org-targeting UI; GA.

## 16. Test Plan

- **Unit** — Flag evaluation rules: org_id match, percentage hash stability, default fallback.
- **Integration** — Flag change propagation via Redis; fallback on Redis miss.
- **End-to-end** — Playwright: admin creates flag; enables for org; user in org sees feature; user outside org does not.
- **Security** — Admin-only flag write access; org-targeting list not visible to non-admin.
- **Accessibility** — axe-core on admin flags page.
- **Performance / load** — k6: 10,000 flag evaluations/sec; p99 ≤ 1 ms with Redis cache.
- **Manual exploratory** — QA: disable flag live; verify feature disappears in browser within 30 s.
