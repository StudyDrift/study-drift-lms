## 1. Problem Statement

Lextures executes all work synchronously inside HTTP request handlers. Email sending, AI grading, video transcoding, webhook delivery, and PDF generation all block the request thread, causing slow response times and total failure if the downstream service is unavailable. Without a background job queue, adding any long-running or unreliable async operation requires ad-hoc goroutine spawning with no retry, no monitoring, and no dead-letter handling. This is a critical production-readiness gap cross-referenced in 6.2 (email), 8.3 (transcoding), 16.3 (webhooks), and the 90-day roadmap summary.

## 2. Goals

- Ship a reliable background job queue backed by either Postgres LISTEN/NOTIFY or Redis, with a clear architectural decision documented here.
- Support job types: email delivery (6.2), AI grading (3.x/19.3), webhook delivery (16.3), video transcoding trigger (8.3), PDF generation (9.8), calendar feed invalidation (16.5), payment fulfillment (16.8).
- Provide retry with exponential backoff, dead-letter queue, per-job visibility, and admin monitoring UI.
- Integrate with the observability stack (17.7) for queue depth, throughput, and latency metrics.
- Enable scheduled/cron jobs (17.4) as a thin layer on top of this infrastructure.

## 3. Architecture Decision: Postgres LISTEN/NOTIFY vs. Redis-backed Queue

Two viable options exist for a Rust-native job queue:

**Option A — Postgres LISTEN/NOTIFY (e.g., `pgmq`, `pg_boss`, or custom)**
- Pros: No new infrastructure dependency; jobs are ACID-durable in the same DB transaction as the triggering write; simpler ops; Postgres already required.
- Cons: Higher DB load at scale; LISTEN/NOTIFY has limited message size; requires careful polling if NOTIFY is missed; Postgres is a poor high-throughput queue.
- Suitable for: current scale (< 100 jobs/sec sustained), teams wanting minimal infra.

**Option B — Redis-backed (Faktory or `apalis-redis` or custom)**
- Pros: Redis is already required (17.2); sub-millisecond enqueue; proven throughput > 10,000 jobs/sec; sidekiq-rs / Faktory provide rich monitoring UIs.
- Cons: At-least-once delivery requires idempotency on job handlers; Redis data is non-durable without AOF/RDB persistence configured.
- Suitable for: high-throughput use cases (mass email, webhook fan-out).

**Recommendation:** Start with **Postgres-backed queue using `pgmq`** (a Postgres extension providing queue semantics with visibility timeout and dead-letter). Rationale: (1) Lextures already requires Postgres; (2) job volume at launch is modest; (3) jobs enqueued in the same transaction as the triggering write avoids the "sent email before DB commit" dual-write problem; (4) if scale demands it, migrate hot job types to a Redis-backed worker tier while keeping Postgres as the source of truth. Add `apalis` (Rust job framework that supports both Postgres and Redis backends) as the application-level abstraction so the backend is swappable.

## 4. Personas & User Stories

- **As a student**, I want to receive a confirmation email within 30 seconds of submitting an assignment so that I know it was received, even if the mail server is slow.
- **As an instructor**, I want AI grading of student submissions to happen asynchronously so that submitting a large assignment batch does not time out the HTTP request.
- **As a DevOps engineer**, I want to monitor the job queue depth and failure rate in Grafana so that I can detect processing backlogs before users are affected.
- **As a developer**, I want to define a new job type by implementing a single trait so that adding a new async task does not require understanding the queue infrastructure.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a `Job` trait in `server/src/jobs/` that each job type implements, with `execute(&self, ctx: &JobContext) -> Result<()>`.
- **FR-2.** The system MUST support Postgres-backed job storage via `pgmq` extension, with job rows in a `jobs` schema.
- **FR-3.** Each job MUST have: `id` (UUID), `job_type` (string), `payload` (JSONB), `status` (pending/running/completed/failed/dead), `attempts` (int), `max_attempts` (int), `scheduled_at` (timestamp), `run_at` (timestamp for delayed jobs), `error_log` (text).
- **FR-4.** The system MUST implement exponential backoff retry: attempt delays of 1 min, 5 min, 30 min, 2 h, 8 h. After `max_attempts`, move to dead-letter queue.
- **FR-5.** The system MUST run worker processes as Tokio tasks within the main Axum binary (not a separate process for MVP), with configurable concurrency per job type.
- **FR-6.** The system MUST support delayed job scheduling: `enqueue_at(job, scheduled_at)`.
- **FR-7.** The system MUST expose an admin UI endpoint listing queue depths, recent failures, and dead-letter jobs with re-drive capability.
- **FR-8.** The system SHOULD support job deduplication: `unique_key` field prevents the same logical job from being enqueued twice within a configurable window.
- **FR-9.** The system SHOULD support a priority queue: jobs have a priority (low/normal/high); workers process higher-priority jobs first.

## 6. Non-Functional Requirements

- **Performance** — Job enqueue latency ≤ 10 ms (single DB INSERT); worker poll interval ≤ 1 s; throughput: ≥ 500 jobs/sec with default 4-worker concurrency.
- **Security** — Job payloads stored in Postgres; access restricted to app DB user. Job payloads MUST NOT contain plaintext passwords, tokens, or PII beyond necessary identifiers (user_id, not email).
- **Privacy & Compliance** — FERPA: job payload containing student identifiers subject to same access controls as student data. Job logs (error messages) must not contain PII.
- **Accessibility** — Admin queue UI WCAG 2.1 AA.
- **Scalability** — Workers horizontally scalable: additional app instances automatically pick up jobs via Postgres row-level locking (`SELECT ... FOR UPDATE SKIP LOCKED`). No central coordinator needed.
- **Reliability** — Postgres-backed jobs survive app restart (durable). `SELECT ... FOR UPDATE SKIP LOCKED` prevents double processing. Visibility timeout handles worker crash without job loss.
- **Observability** — Prometheus gauges: `job_queue_depth{job_type}`, `job_processing_duration_seconds{job_type}`, `job_failures_total{job_type}`. Alert on dead-letter depth > 10 or queue depth > 1,000 for > 5 min.
- **Maintainability** — Each job type is one file in `server/src/jobs/`; registered in a central job registry. Adding a new job type does not require changes outside its own file and the registry.
- **Internationalization** — Job-generated user-facing content (email bodies) uses locale from job payload or user preferences.
- **Backward compatibility** — Existing synchronous email calls (if any exist) migrated to job enqueue; behavior unchanged from user perspective.

## 7. Acceptance Criteria

- **AC-1.** *Given* an instructor submits a large batch of quiz responses for AI grading, *When* the HTTP request completes, *Then* it returns within 200 ms and grading jobs are enqueued; grading results appear within 2 minutes.
- **AC-2.** *Given* the mail server is temporarily unreachable, *When* an email job fails, *Then* it is retried after 1 minute; after 5 failures, it moves to the dead-letter queue and an alert fires.
- **AC-3.** *Given* the app server restarts while a job is running, *When* the server restarts, *Then* the in-progress job's visibility timeout expires and it is re-picked by the next worker (no job loss).
- **AC-4.** *Given* two app instances are running, *When* a job is enqueued, *Then* exactly one instance processes it (no double processing, verified by `SELECT ... FOR UPDATE SKIP LOCKED`).
- **AC-5.** *Given* a dead-letter job exists in the admin UI, *When* an admin clicks "Re-drive," *Then* the job is re-enqueued with a fresh attempt count and `max_attempts` reset.
- **AC-6.** *Given* a job is enqueued with `scheduled_at` 10 minutes in the future, *When* 10 minutes pass, *Then* the worker picks it up and processes it; it is not processed before the scheduled time.

## 8. Data Model

```sql
-- server/migrations/081_job_queue.sql
CREATE SCHEMA IF NOT EXISTS jobs;

CREATE TABLE jobs.queue (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    job_type        TEXT NOT NULL,
    payload         JSONB NOT NULL,
    priority        SMALLINT NOT NULL DEFAULT 5,  -- 1=high, 10=low
    status          TEXT NOT NULL DEFAULT 'pending',
    attempts        SMALLINT NOT NULL DEFAULT 0,
    max_attempts    SMALLINT NOT NULL DEFAULT 5,
    unique_key      TEXT,
    scheduled_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    started_at      TIMESTAMPTZ,
    completed_at    TIMESTAMPTZ,
    error_log       TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX ON jobs.queue (unique_key) WHERE unique_key IS NOT NULL AND status = 'pending';
CREATE INDEX ON jobs.queue (status, priority, scheduled_at)
    WHERE status IN ('pending', 'failed');

CREATE TABLE jobs.dead_letters (
    id              UUID PRIMARY KEY,
    job_type        TEXT NOT NULL,
    payload         JSONB NOT NULL,
    error_log       TEXT,
    failed_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
    redriven        BOOLEAN NOT NULL DEFAULT false
);
```

## 11. AI / ML Considerations

AI grading jobs (`AiGradingJob`, `QuizGenerationJob`) will be the highest-priority job types. Each AI call should be a single job with idempotency key = `{submission_id}-{rubric_version}` to prevent duplicate AI calls on retry.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Postgres queue becomes bottleneck under high job volume | M | M | Monitor `job_queue_depth`; migrate hot job types to Redis (`apalis-redis`) if needed |
| Worker crash leaves job locked (visibility timeout not configured) | L | H | Implement visibility timeout + background lock-refresh task |
| Job payload contains PII, exposed in admin UI error logs | M | M | PII scrubber on error log before storage; payload access restricted to admin role |
| Duplicate job processing due to race condition | L | H | `SELECT ... FOR UPDATE SKIP LOCKED` prevents this; verified by concurrent integration test |

## 15. Rollout Plan

- **Feature flag:** `background_jobs_enabled` (OpenFeature, default `false`).
- **Phase 1:** Queue infrastructure + `EmailDeliveryJob`; migrate mail service.

## 18. Open Questions

1. Should we use the `apalis` crate (abstracts Postgres/Redis backend) or implement a thin custom layer on top of `pgmq`?
2. Is `pgmq` the right Postgres queue extension, or should we use raw `SELECT ... FOR UPDATE SKIP LOCKED` with our own schema?
3. Should job workers run in the same Axum binary (simpler) or a separate worker binary (better isolation, independent scaling)?
4. What is the job payload size limit, and how do we handle large payloads (e.g., AI grading of large submissions)?
