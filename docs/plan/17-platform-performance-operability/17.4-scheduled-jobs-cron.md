## 1. Problem Statement

Several LMS workflows require time-triggered execution: end-of-day grade posting, late-submission penalty sweeps, expired-token cleanup, API request log retention, and due-date reminder notifications. None of these exist today. Without a scheduler, these operations must be triggered manually or not at all, leading to data inconsistency (submissions marked as late only when the instructor manually refreshes) and growing table bloat from unretired expired data.

## 2. Goals

- Implement a scheduler (cron-like) as a thin layer on top of the background job queue (17.3).
- Support standard cron-expression scheduling for recurring jobs.
- Prevent multiple app instances from executing the same scheduled job simultaneously (distributed lock via Postgres or Redis).
- Provide a configuration-driven list of scheduled jobs with name, schedule, job type, and enabled flag.
- Expose scheduled job history (last run, status, next run) in the admin UI.

## 3. Non-Goals

- User-facing scheduled reports or reminders (those are triggered jobs from the queue, not a cron service).
- External cron services (no dependency on `cron.d`, `systemd` timers, or third-party cron SaaS).
- Dynamic schedule creation via the admin UI (configuration-driven is sufficient for MVP).

## 4. Personas & User Stories

- **As a platform engineer**, I want late-submission detection to run automatically at midnight every day so that instructors see accurate late-submission statuses without manual intervention.
- **As a DevOps engineer**, I want the scheduler to run on exactly one app instance at a time so that the same job is not processed twice when two instances are running.
- **As an admin**, I want to see the last-run time and status of each scheduled job so that I can verify the system is operating correctly.
- **As a developer**, I want to add a new scheduled job by adding a struct to a configuration list and implementing the `Job` trait so that I do not need to understand the scheduling infrastructure.

## 5. Functional Requirements

- **FR-1.** The system MUST implement a scheduler that reads a configuration-driven list of `ScheduledJob` definitions (name, cron expression, job type, payload template) and fires them on schedule.
- **FR-2.** The scheduler MUST use a distributed lock (Postgres advisory lock or Redis SETNX) so that only one app instance runs each scheduled job at a given trigger time.
- **FR-3.** Each scheduled job trigger MUST create a job entry in the `jobs.queue` table (17.3), making the actual execution subject to the same retry/dead-letter logic as regular jobs.
- **FR-4.** The system MUST include the following built-in scheduled jobs at launch:
  - `late_submission_sweep` — daily at 00:05 UTC; marks overdue submissions as late.
  - `expired_token_cleanup` — hourly; deletes expired/revoked API tokens from DB.
  - `request_log_retention` — daily at 03:00 UTC; deletes `api.request_log` rows older than 90 days.
  - `due_date_reminder` — daily at 08:00 local time (per-tenant); enqueues reminder notification jobs.
  - `inactive_integration_alert` — daily; flags webhook subscriptions/sync connections with no activity > 12 h.
- **FR-5.** The system MUST record each scheduled job trigger in a `jobs.schedule_history` table with: job_name, triggered_at, job_id (FK to queue), status.
- **FR-6.** The system SHOULD support enabling/disabling individual scheduled jobs via an admin endpoint without a code deployment.
- **FR-7.** The system MAY support per-tenant scheduled job overrides (e.g., different `due_date_reminder` time per timezone).

## 6. Non-Functional Requirements

- **Performance** — Scheduler tick overhead ≤ 1 ms; cron expression evaluation in-process. Scheduled job enqueue ≤ 10 ms.
- **Security** — Scheduler trigger is internal; no external HTTP trigger endpoint exposed without authentication. Distributed lock prevents split-brain execution.
- **Privacy & Compliance** — `request_log_retention` job enforces 90-day GDPR retention limit on API request logs.
- **Accessibility** — Admin schedule history page WCAG 2.1 AA.
- **Scalability** — Scheduler runs on all instances; distributed lock ensures only one fires per interval. Lock TTL equals the job's scheduled interval to prevent lock starvation.
- **Reliability** — If the scheduled trigger is missed (app restarted at trigger time), a "missed job" detection mechanism re-fires within the next tick. Lock TTL shorter than the scheduled interval.
- **Observability** — Prometheus gauge `scheduled_job_last_success_timestamp{job_name}`; alert if any job has not succeeded in 2× its schedule interval.
- **Maintainability** — Scheduled job list in `server/src/scheduler/config.rs`; each entry is a Rust struct, not a DB-stored cron expression (simpler, deployable via code change).
- **Internationalization** — Per-tenant timezone awareness for daily schedules via tenant settings.
- **Backward compatibility** — Scheduler is additive; no existing behavior changes.

## 7. Acceptance Criteria

- **AC-1.** *Given* a submission is past its due date at 00:05 UTC, *When* the `late_submission_sweep` job runs, *Then* the submission's `is_late` flag is set to `true`.
- **AC-2.** *Given* two app instances are running, *When* a scheduled job fires, *Then* exactly one instance acquires the distributed lock and enqueues the job; the other instance skips it.
- **AC-3.** *Given* an expired API token exists in the DB, *When* the `expired_token_cleanup` job runs, *Then* the token row is deleted from `auth.api_tokens`.
- **AC-4.** *Given* a scheduled job fails (job worker returns an error), *When* the next admin visits the schedule history page, *Then* the failed run is visible with status `failed` and an error summary.
- **AC-5.** *Given* the app restarts at exactly the scheduled trigger time for `late_submission_sweep`, *When* the app starts and the lock is not held, *Then* the job fires within the next scheduler tick (≤ 60 s).
- **AC-6.** *Given* the `request_log_retention` job runs, *When* rows older than 90 days exist in `api.request_log`, *Then* those rows are deleted and the table size decreases.

## 8. Data Model

```sql
-- server/migrations/082_scheduler.sql
CREATE TABLE jobs.schedule_history (
    id          BIGSERIAL PRIMARY KEY,
    job_name    TEXT NOT NULL,
    triggered_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    job_id      UUID REFERENCES jobs.queue(id),
    status      TEXT NOT NULL DEFAULT 'triggered',
    notes       TEXT
);
CREATE INDEX ON jobs.schedule_history (job_name, triggered_at DESC);

-- Distributed lock table (alternative to Redis SETNX)
CREATE TABLE jobs.schedule_locks (
    job_name    TEXT PRIMARY KEY,
    locked_by   TEXT NOT NULL,   -- instance identifier
    locked_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    expires_at  TIMESTAMPTZ NOT NULL
);
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Distributed lock starvation if lock TTL too long | L | M | TTL = 80% of schedule interval; monitor lock age |
| Missed job on app restart at trigger time | M | L | Last-run check on startup: if last trigger > interval, fire immediately |
| Late submission sweep runs during high-traffic window | L | M | Schedule at low-traffic time (00:05 UTC); configurable |
| Per-tenant timezone handling causes jobs to fire at wrong time | M | M | Use UTC internally; apply offset only for human-facing reminders |

## 15. Rollout Plan

- **Feature flag:** `scheduler_enabled` (OpenFeature, default `false`).
- **Phase 1:** Scheduler infrastructure + `expired_token_cleanup` + `request_log_retention`; internal testing.
- **Phase 2:** `late_submission_sweep`; `due_date_reminder`; admin UI.
- **Phase 3:** `inactive_integration_alert`; per-tenant timezone support; GA.
- **Rollback:** Disable feature flag; cleanup jobs become manual runbook tasks.

## 16. Test Plan

- **Unit** — Cron expression evaluation; distributed lock acquire/release; missed job detection.
- **Integration** — Two-instance test: verify only one fires per interval; test manual trigger endpoint.
- **End-to-end** — Create a submission past due; advance system clock (or trigger manually); verify `is_late` flag set.
- **Security** — Manual trigger endpoint requires admin auth; scheduler does not expose an unauthenticated HTTP endpoint.
- **Accessibility** — axe-core on admin scheduler page.
- **Performance / load** — Scheduler tick overhead benchmark; 100 concurrent late-submission sweep jobs.
- **Manual exploratory** — QA: disable a job; verify it does not fire; re-enable; verify it fires at next interval.

## 18. Open Questions

1. Should the cron schedule be stored in DB (allowing admin UI to change it without deploy) or in code (simpler, deploy-to-change)?
2. Should per-tenant scheduled jobs (like per-timezone reminders) be implemented as fan-out from the global scheduler or as separate per-tenant jobs?
3. Is Postgres advisory lock or the `schedule_locks` table approach better for the distributed lock? (Advisory locks are lighter but not visible in DB tooling.)
