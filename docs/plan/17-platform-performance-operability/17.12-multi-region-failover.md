## 1. Problem Statement

Lextures runs in a single DigitalOcean region. International university customers require data residency in the EU (or other regions) and cannot rely on a US-only infrastructure. Many US states (e.g., for K-12 state data) require in-state or domestic hosting. A single-region deployment also means a full region outage takes down the entire service with no automatic failover. Multi-region capability is required to meet enterprise SLAs and international regulatory compliance.

## 2. Goals

- Define and implement a primary/warm-standby architecture: one active region + one passive standby region with automated or manual failover capability.
- Use Postgres logical replication to keep a standby replica in a second region continuously synchronized.
- Define and document a region failover runbook achieving RTO ≤ 1 hour for a regional outage.
- Support per-tenant data residency configuration so that EU institutions run in the Frankfurt region and US institutions in the NYC/SFO region.
- Implement a Cloudflare Tunnel or equivalent for latency-based routing between regions.

## 3. Non-Goals

- Active-active multi-region (simultaneous writes to multiple regions) — extreme complexity for current stage; warm standby is the MVP.
- Multi-cloud (AWS + DigitalOcean) — DigitalOcean only for this plan.
- Sub-second RTO (that requires active-active; beyond current budget).
- Geo-distributed CDN for media (DO Spaces CDN handles that separately via 17.5).

## 4. Personas & User Stories

- **As a European university IT director**, I want student data to reside in the EU (Frankfurt region) so that we satisfy our GDPR data residency obligations without a contractual exception.
- **As a US K-12 district administrator**, I want the LMS to survive a full DigitalOcean NYC region outage and resume service within 1 hour so that students can continue during end-of-year testing.
- **As a DevOps engineer**, I want the failover runbook to be testable in staging so that I can confidently execute it under pressure during a real outage.
- **As a platform architect**, I want the app layer to be completely stateless (17.2) so that switching the load balancer to point to the standby region is sufficient to restore service.

## 5. Functional Requirements

- **FR-1.** The system MUST provision a warm-standby region (e.g., AMS3 for EU, SFO3 for US West) in `iac/production/` Terraform with: Postgres logical replication follower, Redis replica, DO Spaces cross-region sync, and app instances (stopped in standby mode).
- **FR-2.** The system MUST configure Postgres logical replication (Postgres 16+ `pg_logical`) from the primary region to the standby follower, with replication lag monitored in Prometheus (target: lag < 30 s).
- **FR-3.** The system MUST provide a manual failover runbook: (a) promote standby Postgres to primary (break replication), (b) start app instances in standby region, (c) update DNS to point to standby LB.
- **FR-4.** DNS failover MUST be implemented via a Cloudflare or DigitalOcean DNS record with TTL ≤ 60 s and health-check-based failover, so that DNS automatically points to the standby region's LB if the primary's health check fails for 3 consecutive checks.
- **FR-5.** The system MUST support per-tenant region assignment: new organizations can specify `region=eu` or `region=us` in their account settings; app routing directs their traffic to the appropriate regional primary.
- **FR-6.** The system MUST monitor replication lag and alert if lag > 60 s (approaching RPO threshold).
- **FR-7.** The system SHOULD perform a quarterly failover drill: simulate primary region outage, execute failover runbook, verify app functions in standby region, fail back.

## 6. Non-Functional Requirements

- **Performance** — Replication lag target: < 30 s steady state. Failover RTO: < 1 hour (manual) or < 5 min (DNS automatic after detection).
- **Security** — Replication user has minimal privileges (`REPLICATION` role only). Replication traffic encrypted with TLS. Standby region uses same secrets manager (17.17) with replicated secrets.
- **Privacy & Compliance** — EU tenant data stored in EU region; no cross-region data transfer except for shared org-level configs. Replication of EU data to US region explicitly prohibited in per-tenant config.
- **Accessibility** — N/A.
- **Scalability** — Standby region provisioned at 50% of primary capacity; scales up during failover as load increases.
- **Reliability** — Replication lag < 30 s means data loss on failover is bounded to ≤ 30 s (RPO). RTO ≤ 1 hour for manual failover.
- **Observability** — Prometheus gauge `postgres_replication_lag_seconds`; alert on lag > 60 s; Grafana dashboard "Replication Health."
- **Maintainability** — `iac/production/` split into `region-primary/` and `region-standby/` Terraform modules.
- **Internationalization** — Region selection UI in account settings uses locale-appropriate region names.
- **Backward compatibility** — Existing single-region deployments unaffected; multi-region is an opt-in IaC configuration.

## 7. Acceptance Criteria

- **AC-1.** *Given* the standby region Postgres follower is running, *When* a row is inserted in the primary DB, *Then* the same row is visible in the follower within 30 seconds.
- **AC-2.** *Given* the primary region health check fails for 3 consecutive Cloudflare health-check intervals, *When* DNS automatic failover triggers, *Then* the DNS record points to the standby LB within 3 minutes.
- **AC-3.** *Given* the manual failover runbook is executed, *When* all steps complete, *Then* the app is serving requests from the standby region within 1 hour with no data loss beyond the replication lag at the time of failure.
- **AC-4.** *Given* an EU organization is assigned `region=eu`, *When* the organization's data is stored, *Then* all Postgres writes go to the Frankfurt primary cluster and no personal data is replicated to the US region.
- **AC-5.** *Given* replication lag exceeds 60 seconds, *When* Prometheus evaluates the alert, *Then* an alert fires in Alertmanager and the on-call engineer is notified.
- **AC-6.** *Given* the quarterly failover drill is completed, *When* results are reviewed, *Then* RTO was achieved within the 1-hour target and all smoke tests passed.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Replication lag spike during high write load | M | M | Monitor lag; alert at 60 s; investigate with Postgres replication stats |
| Failover causes split-brain (both regions accepting writes) | L | H | Standby DB in read-only until manual promotion; strict runbook step ordering |
| Standby app instances not provisioned fast enough during failover | M | M | Pre-warm standby instances (1 stopped instance); scale from 1 to N after promotion |
| EU data replication accidentally configured to US | L | H | Terraform validation rule: EU org's standby must be in EU region |

## 15. Rollout Plan

- **Phase 1:** Postgres logical replication setup; replication monitoring; no failover capability yet.
- **Phase 2:** Standby region Terraform module; stopped app instances; DNS failover record.
- **Phase 3:** Failover runbook written and drilled; per-tenant region selection UI; GA for EU customers.
- **Rollback:** Disable standby region (stop replication, destroy standby resources in Terraform).

## 16. Test Plan

- **Unit** — Replication lag calculation; per-tenant region routing.
- **Integration** — Replication setup: insert rows; verify on follower; measure lag.
- **End-to-end** — Failover drill: stop primary region; execute runbook; verify app works from standby.
- **Security** — Replication TLS verified; EU-only data does not appear in US region DB.
- **Accessibility** — Region selection onboarding step axe-core.
- **Performance / load** — Replication lag under peak load (17.19 load test).
- **Manual exploratory** — QA: full failover runbook; DNS propagation timing; smoke tests in standby.

## 18. Open Questions

1. Should we use DigitalOcean's native Managed Database read replica in another region, or self-managed logical replication? DO managed is easier ops but may not support promotion.
2. Is a warm standby acceptable for the initial offering, or do institutional customers require active-active for their SLAs?
3. How does per-tenant region routing work for self-learner accounts without an org context?
