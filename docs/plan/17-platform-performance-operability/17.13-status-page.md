## 1. Problem Statement

When Lextures experiences downtime or degraded performance, there is no public status page for users, institutions, or IT administrators to consult. This leads to a flood of support tickets asking "is Lextures down?", wastes instructor and student time, and erodes trust. A publicly accessible status page — showing current operational status, active incidents, and historical uptime — is standard operating practice for any SaaS platform and is expected by institutional procurement and IT teams.

## 2. Goals

- Publish a public status page at `status.lextures.io` showing current service health.
- Display historical uptime for the past 90 days.
- Allow the operations team to post incident updates and resolve notices.
- Automatically transition service status to "degraded" or "down" when Prometheus health alerts fire (17.7).
- Notify subscribed users (email / RSS) when an incident is opened or resolved.

## 3. Non-Goals

- Building a custom status page from scratch (use Statuspage.io, Instatus, or an open-source equivalent).
- Real-user monitoring / synthetic transaction checks (deferred to later phase).
- Per-tenant status pages.

## 4. Personas & User Stories

- **As a student**, I want to check `status.lextures.io` when the app is slow so that I know immediately whether it's a known outage or a problem on my end.
- **As a K-12 IT director**, I want to subscribe to email notifications for incidents so that my staff are proactively informed rather than discovering outages from teachers.
- **As an operations engineer**, I want to post incident updates from a simple admin interface so that communication during an outage is fast and does not require code deployments.
- **As a university procurement officer**, I want to see the historical 90-day uptime percentage on the status page so that I can evaluate Lextures' reliability before signing a contract.

## 5. Functional Requirements

- **FR-1.** The system MUST publish a public status page at `status.lextures.io` using a hosted third-party service (Statuspage.io, Instatus, or self-hosted Cachet) with a custom domain.
- **FR-2.** The status page MUST display the following service components: API, Web App, Database, Job Queue, AI Services, Media/File Storage.
- **FR-3.** The status page MUST display historical uptime for 90 days per component.
- **FR-4.** Operations engineers MUST be able to create, update, and resolve incidents from the status page admin UI without a code deployment.
- **FR-5.** The system MUST automatically update component status via the Statuspage API when a Prometheus alert fires (Alertmanager webhook → Statuspage update).
- **FR-6.** The status page MUST offer email subscription and RSS/Atom feed for incident notifications.
- **FR-7.** The Lextures web app MUST display a banner when the status page reports an active incident, linking to the status page (polling Statuspage summary API every 5 minutes).
- **FR-8.** Scheduled maintenance windows MUST be postable in advance, triggering email notification to subscribers.

## 6. Non-Functional Requirements

- **Performance** — Status page must remain accessible even when Lextures itself is down (third-party hosted is the correct approach for this reason).
- **Security** — Status page admin access restricted to operations team (SSO or strong password). Statuspage API key stored in secrets manager (17.17).
- **Privacy & Compliance** — Email subscriptions collected on status page are for notification only; minimal data; covered by privacy policy.
- **Accessibility** — Status page must be WCAG 2.1 AA; chosen service (Statuspage/Instatus) must meet this bar.
- **Scalability** — Third-party hosted; no Lextures infrastructure scaling concern.
- **Reliability** — Third-party hosted status page survives Lextures region outage.
- **Observability** — Alertmanager webhook integration automates status updates from Prometheus alerts.
- **Maintainability** — Status page configuration (components, metrics) stored as code (Statuspage Terraform provider or equivalent) in `iac/production/`.
- **Internationalization** — Status page in English for MVP; Instatus supports multiple languages.
- **Backward compatibility** — Additive; no existing behavior changes.

## 7. Acceptance Criteria

- **AC-1.** *Given* `status.lextures.io` is visited by a user with no Lextures session, *When* the page loads, *Then* current operational status for all components is displayed with no authentication required.
- **AC-2.** *Given* a Prometheus alert fires for high error rate, *When* the Alertmanager webhook is processed, *Then* the status page automatically shows "Degraded" for the API component within 2 minutes.
- **AC-3.** *Given* a user subscribes to email notifications, *When* an incident is created, *Then* they receive an email notification within 5 minutes.
- **AC-4.** *Given* the Lextures web app polls the status summary API, *When* an active incident is detected, *Then* a banner appears at the top of the web app with a link to `status.lextures.io`.
- **AC-5.** *Given* the status page historical data shows 90 days, *When* a university procurement officer views it, *Then* per-component uptime percentages are displayed.
- **AC-6.** *Given* a maintenance window is scheduled 48 hours in advance, *When* subscribed users check their email, *Then* they receive a notification about the scheduled maintenance.

## 8. Data Model

No application DB schema changes. Statuspage/Instatus manages their own data. A thin Lextures integration stores only the Statuspage API key (in secrets manager) and the component ID mappings in a config file.

## 9. API Surface

| Method | Path | Auth | Notes |
|--------|------|------|-------|
| GET | `/api/v1/status-summary` | None | Proxies Statuspage summary; used by web app for banner |

## 10. UI / UX

- **Incident banner:** Top of every page in the React app; yellow/red background; dismissible per session; shows incident title + link to status page.
- **Status page:** Hosted at `status.lextures.io`; default Statuspage/Instatus template with Lextures branding (logo, colors).
- Mobile: banner visible on mobile; status page responsive (handled by hosting service).

## 11. AI / ML Considerations

Not applicable.

## 12. Integration Points

- **Internal:** `clients/web/src/` — incident banner component added to App shell.
- **Internal:** 17.7 (Alertmanager → Statuspage webhook integration).
- **External:** Statuspage.io or Instatus API.

## 13. Dependencies & Sequencing

- Must ship after: 17.7 (Alertmanager for automated status updates), 17.8 (health checks for status accuracy).
- Can ship before most other 17.x features as an independent, low-risk improvement.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Statuspage service has its own outage | L | M | Third-party SaaS with their own redundancy; acceptable risk |
| Alertmanager webhook fires too eagerly, causing false "Degraded" | M | M | Tune alert thresholds (17.7); require 5 min sustained before posting to status page |
| Web app banner polls status API too frequently, causing cost | L | L | 5-minute polling interval; cache response in Redux/React state |

## 15. Rollout Plan

- **Phase 1:** Set up Statuspage/Instatus; manually post initial "All systems operational."
- **Phase 2:** Alertmanager webhook integration for automated component updates.
- **Phase 3:** Web app incident banner; email subscriptions; scheduled maintenance posts; GA.
- **Rollback:** Disable Alertmanager webhook; remove web app banner; status page remains as manual tool.

## 16. Test Plan

- **Unit** — Alertmanager webhook handler; status proxy endpoint.
- **Integration** — Fire test Prometheus alert; verify status page component updates.
- **End-to-end** — Playwright: simulate active incident; verify banner appears in web app.
- **Security** — Status page admin access; API key in secrets manager.
- **Accessibility** — axe-core on incident banner component; status page accessibility audit against chosen service.
- **Performance / load** — Banner polling endpoint: 1,000 concurrent requests; verify p95 ≤ 100 ms.
- **Manual exploratory** — QA: post incident manually; verify email notification; dismiss banner.
