## 1. Problem Statement

Lextures emits structured `tracing` logs but has no metrics, no distributed traces, and no error reporting. When something goes wrong in production, the team has no way to understand latency distributions, error rates, queue depths, or which requests are slow. This makes debugging under load guesswork and makes on-call response reactive rather than proactive. SOC 2 Type II also requires evidence of monitoring and alerting. Establishing Prometheus + OpenTelemetry + Sentry is the foundational observability layer that every other operational improvement depends on.

## 2. Goals

- Export Prometheus metrics from the Axum server (`GET /metrics` endpoint).
- Instrument the application with OpenTelemetry traces, exported to a Jaeger/Tempo backend.
- Integrate Sentry for error/panic capture with source-mapped stack traces.
- Deploy Grafana with pre-built dashboards for: HTTP request rates/latency, DB pool, Redis, job queue, AI provider usage, business metrics (enrollments, grade submissions).
- Define alerting rules (Prometheus Alertmanager or Grafana Alerts) for actionable production alerts.

## 3. Non-Goals

- Full APM solution (DataDog, New Relic) — too expensive for current stage; Grafana stack is sufficient.
- Log aggregation platform (ELK) — structured JSON logs to stdout; collected by DigitalOcean Managed Logging or Loki.
- Synthetic monitoring / uptime robot (13.x status page is 17.13).
- Front-end real-user monitoring (RUM) — deferred to a later phase.

## 4. Personas & User Stories

- **As an on-call engineer**, I want a Grafana dashboard showing p95 API latency and error rate so that I can identify degradation within seconds of an alert firing.
- **As a developer**, I want distributed traces linking an HTTP request to its DB queries and AI calls so that I can pinpoint the slow step in a complex endpoint.
- **As a DevOps engineer**, I want Sentry to capture panics and unhandled errors with stack traces so that I learn about bugs before users file support tickets.
- **As a security officer**, I want all monitoring traffic to be internal to the VPC so that metrics endpoints are not accessible from the internet.
- **As a platform engineer**, I want job queue depth and dead-letter count in Grafana so that I can detect processing backlogs proactively.

## 5. Functional Requirements

- **FR-1.** The system MUST expose `GET /metrics` (Prometheus exposition format) on an internal port (not the public LB port), returning: HTTP request counters/histograms, DB pool utilization, Redis connection count, job queue depth by type, AI provider call counts/latency.
- **FR-2.** The system MUST instrument all HTTP handlers with OpenTelemetry spans (trace per request, child spans for DB queries and external HTTP calls) exported to an OTel collector.
- **FR-3.** The system MUST integrate Sentry Rust SDK for panic capture, `tracing` integration (ERROR-level events sent to Sentry), and performance transaction sampling (10% sample rate).
- **FR-4.** Sentry MUST be configured with separate DSNs for production and staging; source maps / debug symbols uploaded in the CI/CD pipeline.
- **FR-5.** The system MUST deploy Grafana (via `iac/production/` Terraform) with the following pre-built dashboards: (a) HTTP Overview, (b) Database & Cache, (c) Job Queue, (d) AI Provider Usage, (e) Business Metrics.
- **FR-6.** The system MUST configure Prometheus Alertmanager (or Grafana Alerts) with the following critical alerts: error_rate > 1% for 5 min, p95 latency > 1 s for 5 min, job dead-letter count > 10, DB pool utilization > 90%.
- **FR-7.** The system SHOULD emit `tracing` logs as structured JSON to stdout, collected by DigitalOcean's managed logging or a Loki sidecar.
- **FR-8.** The system SHOULD add a `trace_id` field to all HTTP response headers so that users can provide trace IDs when reporting issues.
- **FR-9.** The system MAY implement a `tracing` filter that redacts PII (email addresses, names) from log fields before emission (10.14).

## 6. Non-Functional Requirements

- **Performance** — Prometheus scrape endpoint returns in ≤ 50 ms. OTel span export is async and does not add to request latency. Sentry SDK async; no blocking on error reporting.
- **Security** — `/metrics` endpoint only accessible from within VPC; blocked at load balancer / firewall. Sentry DSN in secrets manager (17.17). OTel collector only accessible internally.
- **Privacy & Compliance** — No PII in metric labels or trace attributes. Sentry `before_send` hook scrubs email, name, and student ID from error events (10.14 / FERPA).
- **Accessibility** — Grafana is an operator tool; WCAG compliance for Grafana is handled by Grafana Labs.
- **Scalability** — Prometheus pull model; each app instance scrapes independently. Grafana / Prometheus deployed as separate services in production IaC.
- **Reliability** — Observability pipeline failure (OTel collector down, Sentry unreachable) must not affect app request handling. All instrumentation uses async, non-blocking exporters.
- **Observability** — Meta-monitoring: Prometheus monitors the Prometheus scrape health and Grafana availability.
- **Maintainability** — Metrics defined in `server/src/telemetry/metrics.rs`; all counters/histograms registered at startup via `prometheus` crate. Adding a new metric requires one line in the registry.
- **Internationalization** — N/A (operator tooling).
- **Backward compatibility** — Existing `tracing` calls remain unchanged; OTel bridge converts `tracing` spans to OTel spans transparently.

## 7. Acceptance Criteria

- **AC-1.** *Given* the app is running, *When* `GET /metrics` is called from within the VPC, *Then* a valid Prometheus exposition-format response is returned including `http_requests_total` and `http_request_duration_seconds` metrics.
- **AC-2.** *Given* a request to `/api/v1/courses` is processed, *When* the OTel trace is queried in Jaeger/Tempo, *Then* a trace exists with parent span `HTTP GET /api/v1/courses` and child spans for each DB query.
- **AC-3.** *Given* the app panics (simulated with a test endpoint), *When* Sentry is checked, *Then* an error event appears with the correct stack trace and no PII in the payload.
- **AC-4.** *Given* the Grafana "HTTP Overview" dashboard is loaded, *When* traffic is flowing, *Then* p50/p95/p99 latency time series are displayed with data points for the last 1 hour.
- **AC-5.** *Given* the job dead-letter count exceeds 10, *When* Prometheus evaluates the alert rule, *Then* an alert fires in Alertmanager and a Slack notification is sent within 5 minutes.
- **AC-6.** *Given* `GET /metrics` is called from a public internet IP (outside VPC), *When* the firewall rules are applied, *Then* the connection is refused.

## 8. Data Model

No DB schema changes. Prometheus metrics stored in Prometheus TSDB. OTel traces stored in Tempo/Jaeger. Sentry events stored in Sentry SaaS.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| High cardinality metric labels exhausting Prometheus memory | M | H | No user_id or course_id in metric labels; aggregate by route group |
| OTel exporter adds latency to hot paths | L | M | Async batch exporter; sample at 10% for performance traces |
| PII in Sentry error events (FERPA) | M | H | `before_send` hook with PII scrubber; test with synthetic PII payloads |
| Grafana instance becomes a security boundary gap | M | M | VPC-internal access only; SSO for Grafana access (admin SSO 4.2) |

## 15. Rollout Plan

- **Phase 1:** Prometheus metrics export + basic Grafana dashboard; Sentry integration.
- **Phase 2:** OTel traces; enriched dashboards; alerting rules.
- **Phase 3:** Business metrics dashboard; PII scrubber; log aggregation (Loki); GA.
- **Rollback:** Sentry/OTel are side-effect-only; disabling them does not affect app behavior.

## 16. Test Plan

- **Unit** — Metrics registry initialization; Sentry `before_send` PII scrubber.
- **Integration** — `/metrics` endpoint output validated against Prometheus text format; OTel trace emitted and readable.
- **End-to-end** — Playwright: trigger a known error; verify Sentry event received; verify Grafana metric visible.
- **Security** — `/metrics` unreachable from outside VPC; Sentry payload has no PII.
- **Accessibility** — Grafana is operator tooling; standard WCAG compliance verified for any Lextures-authored dashboard JSON.
- **Performance / load** — Benchmark `/metrics` endpoint latency; measure OTel async exporter overhead on p95 latency.
- **Manual exploratory** — QA: verify trace appears in Jaeger; Grafana dashboard loads correctly; alert fires on simulated threshold breach.

## 18. Open Questions

1. Should we use Grafana Cloud (managed) or self-host Grafana + Prometheus in production? Grafana Cloud's free tier may be sufficient initially.
2. What is the right OTel trace sampling rate — 10% for production, 100% for staging?
3. Should we use Loki for log aggregation alongside Prometheus for metrics, or rely on DigitalOcean's managed logging?
4. How do we handle Sentry quotas (events/month) at K-12 scale with many students triggering errors?
