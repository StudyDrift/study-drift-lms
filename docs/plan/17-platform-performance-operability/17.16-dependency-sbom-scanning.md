## 1. Problem Statement

Lextures has no automated dependency vulnerability scanning. A known CVE in a Rust crate (e.g., `rustls-webpki` — recently patched per `3bc3d59`) or npm package can linger undetected indefinitely. Without SBOM (Software Bill of Materials) generation, Lextures cannot respond to customer requests for a component inventory — increasingly required in US Federal and EU procurement. NIST SSDF and Executive Order 14028 both mandate SBOMs for software sold to US government-affiliated institutions.

## 2. Goals

- Enable GitHub Dependabot for automated dependency version PRs for Cargo and npm.
- Add `cargo-audit` to CI to block builds with known CVEs.
- Add `npm audit` to CI to block builds with high-severity npm vulnerabilities.
- Generate CycloneDX or SPDX SBOM files at release time and attach them to GitHub Releases.
- Publish a security disclosure timeline policy for dependency vulnerabilities (critical: 48 h; high: 7 days; medium: 30 days).

## 3. Non-Goals

- Runtime application security testing (DAST) — separate from static dependency scanning.
- Container image scanning (planned as part of 17.9 canary security gate).
- Custom CVE research (this plan uses existing advisory databases: RUSTSEC, npm security advisories).
- License compliance enforcement (useful but separate from vulnerability scanning).

## 4. Personas & User Stories

- **As a security engineer**, I want CI to fail if any dependency has a known critical CVE so that vulnerabilities are caught before production deploys.
- **As an enterprise procurement officer**, I want an SBOM attached to each Lextures release so that our security team can verify component inventory against our internal risk registry.
- **As a developer**, I want Dependabot to automatically open PRs for dependency updates so that I do not have to manually track upstream security releases.
- **As a DevOps engineer**, I want `cargo-audit` results in GitHub Security tab so that I have a central view of dependency vulnerabilities across the repo.

## 5. Functional Requirements

- **FR-1.** The repo MUST have a `.github/dependabot.yml` configuration enabling Dependabot for: `cargo` (daily, `server/`), `npm` (daily, `clients/web/`), `github-actions` (weekly).
- **FR-2.** CI MUST run `cargo audit --deny warnings` as a step in `.github/workflows/ci.yml`; a build with any known CVE in the RUSTSEC advisory database MUST fail.
- **FR-3.** CI MUST run `npm audit --audit-level=high` in `clients/web/`; a build with any high or critical npm vulnerability MUST fail.
- **FR-4.** The release workflow MUST generate a CycloneDX SBOM for the Rust server (`cyclonedx-rust`) and an SPDX SBOM for the npm dependencies, and attach both to the GitHub Release as assets.
- **FR-5.** The repo MUST configure GitHub Secret Scanning (already free on public repos; enable on private) to detect any accidental credential commits.
- **FR-6.** The repo MUST configure GitHub CodeQL analysis for Rust and TypeScript (via `actions/codeql-action`) to detect common code-level vulnerabilities beyond dependency CVEs.
- **FR-7.** A `SECURITY.md` at the repo root MUST document: vulnerability disclosure process, response SLAs (critical: 48 h, high: 7 d, medium: 30 d), and a disclosure email address.
- **FR-8.** The system SHOULD generate and store a merged SBOM (Rust + npm components) in CycloneDX JSON format at each release for customer download from the trust center (20.2).

## 6. Non-Functional Requirements

- **Performance** — `cargo audit` and `npm audit` CI steps should complete in < 60 s each.
- **Security** — SBOM files are public artifacts; they contain component names and versions but no secrets. Secret scanning catches any accidental credential exposure.
- **Privacy & Compliance** — NIST SSDF practice PS.3.2 (identify and confirm vulnerabilities) and PE.3.2 (verify component provenance) satisfied by this plan. CycloneDX is the NTIA minimum-element-compliant SBOM format.
- **Accessibility** — N/A.
- **Scalability** — Dependency scanning is a CI concern; no production scaling impact.
- **Reliability** — If the RUSTSEC advisory database is unavailable (network issue in CI), `cargo audit` should fail gracefully with a warning rather than blocking the build. Use `--no-fetch` with a cached advisory database for reliability.
- **Observability** — Dependabot PR volume tracked as a metric of dependency health. GitHub Security tab provides central vulnerability dashboard.
- **Maintainability** — `.github/dependabot.yml` is the single configuration file; `ci.yml` updated with audit steps.
- **Internationalization** — N/A.
- **Backward compatibility** — Adding audit steps to CI may reveal existing ignored vulnerabilities; triage and address before enabling as blocking step.

## 7. Acceptance Criteria

- **AC-1.** *Given* a known CVE-bearing crate is present in `Cargo.lock`, *When* CI runs `cargo audit --deny warnings`, *Then* the build fails with the CVE identifier and advisory link in the output.
- **AC-2.** *Given* Dependabot is configured, *When* a new version of a Cargo dependency with a security fix is released, *Then* a Dependabot PR is opened automatically within 24 hours.
- **AC-3.** *Given* a GitHub Release is created, *When* the release artifacts are listed, *Then* a `sbom-server.cdx.json` and `sbom-web.spdx.json` file are attached.
- **AC-4.** *Given* a developer accidentally commits an API key in a Rust file, *When* GitHub Secret Scanning processes the push, *Then* an alert appears in the GitHub Security tab and the developer is notified.
- **AC-5.** *Given* the `SECURITY.md` at the repo root, *When* a security researcher reads it, *Then* a clear disclosure email and response SLA are present.
- **AC-6.** *Given* `npm audit --audit-level=high` runs in CI, *When* a high-severity npm vulnerability exists in `node_modules`, *Then* the CI step fails with the package name and CVE.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Existing CVEs block CI immediately upon enabling | H | M | Run in warn-only mode first; triage and remediate before enabling as blocking |
| Dependabot floods repo with update PRs | M | L | Configure Dependabot with `open-pull-requests-limit: 5` and grouped updates |
| RUSTSEC database unavailable in CI (flaky) | M | L | Cache advisory DB in CI; use `--no-fetch` with cached copy |
| SBOM reveals internal component versions to attackers | L | L | SBOMs are standard industry practice; component names are not secrets |

## 15. Rollout Plan

- **Step 1 (day 1):** Add `.github/dependabot.yml`; enable GitHub Secret Scanning; add `SECURITY.md`.
- **Step 2 (week 1):** Add `cargo audit` and `npm audit` to CI in warn-only mode; review existing vulnerabilities.
- **Step 3 (week 2):** Remediate or suppress accepted vulnerabilities; flip to blocking mode.
- **Step 4 (release):** Add SBOM generation to release workflow; add CodeQL.

## 16. Test Plan

- **End-to-end** — Open a PR with a known CVE; verify CI fails; fix; verify CI passes.
- **Security** — Verify GitHub Secret Scanning fires on an accidental test credential commit.
- **Accessibility** — N/A.
- **Performance / load** — Measure `cargo audit` step duration; target < 60 s.
- **Manual exploratory** — QA: review GitHub Security tab; verify Dependabot PRs appear after configuration.

## 18. Open Questions

1. Should we use `cargo-deny` (which covers both CVEs and license compliance) instead of `cargo-audit`? `cargo-deny` is more powerful but requires more configuration.
2. Should Dependabot auto-merge minor/patch updates (with CI passing), or always require human review?
3. What is the right SBOM format for our customer base — CycloneDX JSON (most modern) or SPDX TV (more widely supported)?
