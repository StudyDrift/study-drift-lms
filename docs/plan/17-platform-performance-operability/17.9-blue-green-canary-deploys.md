## 1. Problem Statement

The current deployment model redeploys a single droplet by stopping the old container and starting a new one — causing seconds to minutes of downtime on every deploy. With institutional customers on live courses, any visible downtime during a deploy erodes trust and violates SLA expectations. Blue/green and canary deploy strategies eliminate deploy-time downtime and allow safe, observable rollouts of new versions with automated rollback on error-rate regression.

## 2. Goals

- Implement a rolling-restart deploy that achieves zero-downtime by draining each instance before terminating it.
- Support a blue/green swap pattern using the DigitalOcean Load Balancer to switch traffic between two app versions.
- Support a canary deploy: route 5–10% of traffic to the new version; auto-promote on success or auto-rollback on error-rate threshold breach.
- Integrate canary analysis with Prometheus metrics (17.7) to automate the promote/rollback decision.
- Update the CI/CD pipeline (`.github/workflows/deploy-demo.yml` → production equivalent) to execute the chosen deploy strategy.

## 3. Non-Goals

- Full Kubernetes deployment with Argo CD or Flux (deferred until k8s adoption).
- Feature flag–based canary at the application layer (that is 17.14).
- Multi-region failover (17.12).
- Database schema changes during zero-downtime deploy (17.10 covers migration strategy).

## 4. Personas & User Stories

- **As a DevOps engineer**, I want to deploy a new version by pushing to `main` without scheduling a maintenance window so that releases happen continuously without user disruption.
- **As a product manager**, I want a canary deploy that routes 5% of traffic to a new version for 10 minutes before full rollout so that I can catch regressions before all users are affected.
- **As an on-call engineer**, I want the canary analysis to automatically roll back to the old version if the error rate exceeds 1% so that I do not need to manually intervene at 2 AM.
- **As an institution**, I want deploys to cause zero service interruption so that students can continue working during course hours.

## 5. Functional Requirements

- **FR-1.** The system MUST support zero-downtime rolling restarts: new app instances are started and pass `/health/ready` before old instances receive a SIGTERM drain signal (17.8).
- **FR-2.** The CI/CD pipeline MUST implement a blue/green deploy step: provision a new "green" app instance set; shift 100% traffic to green via LB update; terminate "blue" instances on success.
- **FR-3.** The pipeline MUST implement a canary stage before full blue/green swap: route 5% of traffic to the new version for a configurable analysis window (default: 10 minutes).
- **FR-4.** During the canary window, the pipeline MUST monitor Prometheus metrics (`http_error_rate`, `http_request_duration_p95`) and automatically promote (shift to 100% new) if thresholds are met, or roll back (shift to 100% old) if thresholds are breached.
- **FR-5.** The LB traffic split (canary percentage) MUST be configurable per deploy via a pipeline input variable.
- **FR-6.** The pipeline MUST tag each deployed Docker image with the Git commit SHA and deploy timestamp for rollback traceability.
- **FR-7.** The pipeline MUST notify a Slack channel (16.6) or admin email on deploy start, canary promote, rollback, or failure.
- **FR-8.** The system SHOULD maintain at least 2 previous Docker image tags in the registry to enable emergency rollback without rebuilding.

## 6. Non-Functional Requirements

- **Performance** — Zero-downtime: p99 request success rate remains > 99.9% during a rolling deploy. Canary analysis window: 10 minutes (configurable).
- **Security** — Canary build images scanned for CVEs before promotion (17.16). New image SHA verified against CI artifact before LB switch.
- **Privacy & Compliance** — No user data processed differently during canary; same encryption, same auth middleware on both versions.
- **Accessibility** — N/A (deployment pipeline).
- **Scalability** — Strategy works with 2 to N app instances; LB traffic split works with any instance count.
- **Reliability** — Automatic rollback on error-rate breach ensures failures are contained to the canary window. Old instance teardown deferred until new instances are confirmed healthy.
- **Observability** — Deploy event annotation in Grafana dashboards (vertical line at deploy time); canary analysis results logged to Slack and pipeline artifacts.
- **Maintainability** — Deploy strategy encoded in `.github/workflows/deploy-production.yml`; not scattered across manual scripts.
- **Internationalization** — N/A.
- **Backward compatibility** — Both blue (old) and green (new) versions run against the same database during transition; migrations must be backward-compatible (17.10).

## 7. Acceptance Criteria

- **AC-1.** *Given* a deploy is triggered, *When* the rolling restart begins, *Then* zero HTTP 503 or 502 responses are observed by a k6 load test running during the deploy.
- **AC-2.** *Given* a canary deploy has routed 5% of traffic to the new version, *When* the new version returns HTTP 500 at > 1% rate for 3 consecutive minutes, *Then* traffic is automatically shifted back to 100% old version and the pipeline marks the deploy as failed.
- **AC-3.** *Given* a successful canary analysis (error rate < 0.5%, p95 latency within 10% of baseline), *When* the 10-minute window passes, *Then* traffic is automatically shifted to 100% new version.
- **AC-4.** *Given* a deploy completes, *When* Grafana is checked, *Then* a deploy annotation marker is visible at the correct timestamp on all dashboards.
- **AC-5.** *Given* a rollback is needed, *When* the pipeline triggers rollback to the previous image tag, *Then* the old version is running within 2 minutes and serving traffic.
- **AC-6.** *Given* the deploy pipeline requires LB traffic modification, *When* Terraform is run with the updated LB backend weights, *Then* the DigitalOcean LB reflects the new traffic split within 30 seconds.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Backward-incompatible DB migration breaks blue version during green canary | M | H | 17.10 expand/contract migration pattern required |
| Canary analysis false positive (network blip triggers rollback) | M | M | Require 3 consecutive failing windows before rollback; not 1 data point |
| LB traffic shift latency causes thundering herd on green instances | L | M | Gradual ramp: 5% → 25% → 100% with intermediate health checks |
| Canary analysis Prometheus query unavailable | L | M | Fallback to time-based canary (promote after 10 min with no alert) |

## 15. Rollout Plan

- **Phase 1:** Rolling restart (zero-downtime, no canary) — deploy to production immediately; validates graceful drain (17.8).
- **Phase 2:** Blue/green swap with manual promote; no automatic canary analysis.
- **Phase 3:** Automatic canary analysis + promote/rollback; full CI/CD integration; GA.
- **Rollback of rollout:** Revert pipeline to rolling restart phase if canary analysis integration is unstable.

## 16. Test Plan

- **Integration** — Staging: deploy new version with intentional error endpoint; verify rollback fires.
- **End-to-end** — Staging deploy with k6 load test running; verify zero 503s during rolling restart.
- **Security** — Verify image SHA checked before LB shift; no unsigned images promoted.
- **Accessibility** — N/A.
- **Performance / load** — k6 running during rolling restart; verify p99 success rate > 99.9%.
- **Manual exploratory** — QA: manual promote/rollback of pipeline; deploy with broken migration (verify rollback).

## 18. Open Questions
