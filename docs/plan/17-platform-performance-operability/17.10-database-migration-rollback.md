## 1. Problem Statement

The `server/migrations/` directory contains 72+ forward-only `.sql` migration files applied by `sqlx migrate run`. There is no documented down-migration strategy, no `down.sql` convention, and no tested rollback procedure. If a bad migration reaches production (e.g., a column dropped prematurely, a constraint too strict), the only recovery path today is a full database restore from backup — a slow, data-losing option. Blue/green deploys (17.9) require backward-compatible migrations (the old app version must keep running while the new one is being caned), which demands an explicit expand/contract migration pattern.

## 2. Goals

- Document and implement the expand/contract (also called backward-compatible) migration pattern for all schema changes going forward.
- Establish a convention for `down.sql` companion files for each migration that support rollback.
- Provide a tested runbook for rolling back a migration in production without full data restore.
- Integrate migration validation into CI so that non-backward-compatible changes (breaking column drops, renames) are caught before they reach production.
- Provide migration history tracking that survives app upgrades.

## 3. Non-Goals

- Automatic schema inference or ORM-based migration generation.
- Splitting the database into micro-services shards.
- Zero-downtime column type changes (those require multi-step expand/contract; documented in the pattern guide, not implemented automatically).
- Postgres logical replication-based blue/green DB failover (that is 17.12).

## 4. Personas & User Stories

- **As a backend developer**, I want a clear checklist for writing backward-compatible migrations so that my schema change does not break the old app version during a blue/green canary window.
- **As an on-call engineer**, I want a tested migration rollback runbook so that I can undo a bad migration in under 10 minutes without restoring a backup.
- **As a platform engineer**, I want CI to catch breaking migration patterns (DROP COLUMN, RENAME COLUMN) and require reviewer sign-off so that unsafe migrations never reach production unreviewed.

## 5. Functional Requirements

- **FR-1.** The repo MUST adopt the expand/contract migration pattern: (1) Expand — add new columns/tables while keeping old; (2) Migrate data — dual-write or backfill; (3) Contract — remove old columns only after all app instances use the new schema.
- **FR-2.** Each migration file MUST be accompanied by a `{NNN}_*.down.sql` rollback file in the same directory, even if the rollback is a no-op with a documented reason.
- **FR-3.** The CI pipeline MUST include a lint step that warns (not blocks) on migrations containing `DROP COLUMN`, `DROP TABLE`, `RENAME COLUMN`, or `ALTER COLUMN TYPE` without a corresponding expand step.
- **FR-4.** A runbook MUST be written and tested in staging for: (a) rolling back the most recent migration using `sqlx migrate revert` (where a down.sql exists), and (b) manually reversing a breaking migration using a corrective forward migration.
- **FR-5.** The `server/migrations/` README MUST document: naming convention, expand/contract pattern, required down.sql convention, and how to add a migration.
- **FR-6.** The system SHOULD track which migrations were applied on which deploy in the `_sqlx_migrations` table augmented with a `deploy_id` column, so incidents can be correlated with specific deploys.

## 6. Non-Functional Requirements

- **Performance** — Large backfill migrations MUST use batched UPDATE (1,000 rows/batch) to avoid long-running transactions that lock tables.
- **Security** — Migration files MUST NOT contain inline secrets (connection strings, API keys). Migrations run with the app's DB user, not a superuser.
- **Privacy & Compliance** — Migrations that delete PII columns must be accompanied by a documented data-erasure procedure for GDPR right-to-erasure compliance.
- **Accessibility** — N/A.
- **Scalability** — Migrations that add NOT NULL columns without a DEFAULT will fail on large tables (Postgres re-writes the table). Pattern guide must call this out.
- **Reliability** — `sqlx migrate run` is idempotent; running it multiple times is safe. `_sqlx_migrations` checksum validation catches file corruption.
- **Observability** — CI logs migration run output; production apply is logged to structured log with migration file names and durations.
- **Maintainability** — All current 72 migrations backfilled with stub `down.sql` files (even if they are `-- rollback not supported; restore from backup` comments) to establish the convention.
- **Internationalization** — N/A.
- **Backward compatibility** — This plan changes the development process, not the running application.

## 7. Acceptance Criteria

- **AC-1.** *Given* a new migration adds a column, *When* the PR is opened, *Then* a companion `down.sql` file exists; CI validates its presence.
- **AC-2.** *Given* a migration containing `DROP COLUMN` is submitted without a prior expand phase, *When* CI runs the lint check, *Then* a warning is emitted in the PR check requiring explicit reviewer approval.
- **AC-3.** *Given* a bad migration is applied to the staging database, *When* the rollback runbook is executed using `sqlx migrate revert`, *Then* the schema returns to its previous state and the app passes smoke tests.
- **AC-4.** *Given* a backward-compatible (expand) migration is applied, *When* the old app version and new app version both run against the database simultaneously (blue/green window), *Then* both versions operate without errors.
- **AC-5.** *Given* the `server/migrations/` directory, *When* a developer reads the README, *Then* the expand/contract pattern, naming convention, and down.sql requirement are clearly documented with examples.
- **AC-6.** *Given* a backfill migration runs on a table with 100,000 rows, *When* monitored in Postgres, *Then* no single transaction holds a lock for > 5 seconds.

## 8. Data Model

```sql
-- Augment _sqlx_migrations with deploy tracking
-- This is a corrective migration applied to existing migrations table:
-- server/migrations/073_migration_deploy_tracking.sql
ALTER TABLE _sqlx_migrations ADD COLUMN IF NOT EXISTS deploy_id TEXT;
```

Existing 72 `down.sql` stubs: batch-created in `server/migrations/` for each existing forward migration; most will be `-- Rollback not supported: restore from backup` comments, with a few having actual rollback SQL for recent additions.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Existing 72 migrations have no down.sql — large backfill task | H | L | Batch-create stub down.sql files; not all need actual SQL |
| Developer writes non-backward-compatible migration not caught by CI | M | H | CI lint + mandatory PR reviewer for destructive migration keywords |
| Large backfill migration locks table | M | H | Pattern guide requirement for batched UPDATE; CI flag on full-table UPDATE |
| `sqlx migrate revert` fails because down.sql was wrong | L | H | Test every real down.sql in staging before applying to production |

## 15. Rollout Plan

- **Week 1:** Write expand/contract pattern guide and down.sql convention; backfill stub down.sql for existing 72 migrations.
- **Week 2:** Add CI lint check; update migration README.
- **Week 3:** Test rollback runbook in staging with a real migration revert; document runbook.
- **Ongoing:** All future migrations follow new convention; enforced by CI and code review.

## 16. Test Plan

- **Unit** — CI lint script: test against migrations with/without DROP COLUMN; verify warning fires.
- **Integration** — `sqlx migrate run` + `sqlx migrate revert` cycle in CI against a fresh DB; verify idempotency.
- **End-to-end** — Staging: apply a test migration with a real `down.sql`; run app; revert; run app again; verify no errors.
- **Security** — Verify no credentials in migration files (grep in CI).
- **Accessibility** — N/A.
- **Performance / load** — Batched backfill test on 100k-row table; verify no lock > 5 s.
- **Manual exploratory** — QA: full rollback runbook execution in staging; timing measured.

## 17. Documentation & Training

- `server/migrations/README.md` — expand/contract pattern, naming convention, down.sql requirement.
- Internal runbook: "Rolling back a database migration in production."
- ADR: decision to use expand/contract over traditional up/down migrations.

## 18. Open Questions

1. Should `sqlx` CLI's `migrate revert` be used for rollbacks, or should we rely on corrective forward migrations only? (Corrective forward migrations are safer but require planning ahead.)
2. Should we add a `MIGRATE_LOCK` distributed lock to prevent two app instances from running `migrate run` simultaneously on startup?
3. What is the right batching strategy for backfill migrations — 1,000 rows/transaction or 10,000?
