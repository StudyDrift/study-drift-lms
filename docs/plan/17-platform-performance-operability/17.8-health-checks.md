## 1. Problem Statement

The existing `/health/ready` endpoint leaks internal error details (Postgres connection errors, stack traces) in its response body when the database is unavailable. This is both a security issue (information disclosure) and an operational gap: the endpoint does not distinguish between liveness (is the process alive?) and readiness (is it ready to serve traffic?), and it has no liveness endpoint at all. Load balancers and orchestration tools (Docker Compose healthcheck, future Kubernetes) need both probes defined correctly to enable safe rolling restarts and traffic draining.

## 2. Goals

- Implement a `/health/live` endpoint that returns 200 as long as the process is running (no external checks).
- Improve `/health/ready` to return a structured JSON health status with sub-component checks (DB, Redis, job queue) without leaking error details to unauthenticated callers.
- Ensure load balancer (17.1) uses `/health/ready` to remove unhealthy instances from rotation.
- Add `docker-compose.yml` healthcheck directives pointing to the correct endpoints.

## 3. Non-Goals

- Status page for end users (17.13).
- Deep synthetic transaction health checks ("can we enroll a user?") — operational overhead deferred.
- Kubernetes readiness gate annotations (out of scope until k8s adoption).

## 4. Personas & User Stories

- **As a load balancer**, I want to poll `/health/ready` every 5 seconds so that I can remove an instance from rotation the moment its database connection is broken.
- **As a DevOps engineer**, I want a liveness probe at `/health/live` so that Docker/orchestration restarts the container if the process is deadlocked but still responding.
- **As a security officer**, I want health endpoints to return safe, generic error messages to unauthenticated callers so that internal connection strings are not exposed.
- **As an on-call engineer**, I want an authenticated `/health/detailed` endpoint that shows per-component latencies so that I can diagnose which dependency is degraded.

## 5. Functional Requirements

- **FR-1.** The system MUST expose `GET /health/live` returning HTTP 200 `{"status":"ok"}` as long as the Axum process is accepting connections; no external dependency checks performed.
- **FR-2.** The system MUST expose `GET /health/ready` returning HTTP 200 if all critical dependencies (Postgres, Redis) are reachable, or HTTP 503 if any are not. Response body MUST NOT include connection strings, passwords, or internal error messages — only safe status indicators.
- **FR-3.** `GET /health/ready` response body MUST follow the format: `{"status":"ready"|"degraded"|"unhealthy", "checks":{"postgres":"ok"|"fail","redis":"ok"|"fail"}}`.
- **FR-4.** The system MUST expose `GET /health/detailed` behind admin authentication (JWT required), returning per-component latency measurements and safe error summaries.
- **FR-5.** The `/health/ready` check for Postgres MUST use a lightweight query (`SELECT 1`) with a 1-second timeout; the check for Redis MUST use `PING` with a 500 ms timeout.
- **FR-6.** The `docker-compose.yml` and `docker-compose.deploy.yml` MUST include `healthcheck` directives using `curl -f /health/live` and `curl -f /health/ready`.
- **FR-7.** The DigitalOcean Load Balancer health check (17.1) MUST point to `/health/ready` with a 5-second interval and 2-failure threshold.

## 6. Non-Functional Requirements

- **Performance** — `/health/live` response time < 5 ms. `/health/ready` response time < 200 ms (dominated by DB/Redis check latency).
- **Security** — `/health/live` and `/health/ready` are unauthenticated but return no sensitive data. `/health/detailed` requires admin JWT. No IP allowlist required for health probes (LB needs to reach them from its own IP).
- **Privacy & Compliance** — Health endpoints do not process user data. No PII in responses.
- **Accessibility** — N/A (machine-readable endpoints).
- **Scalability** — Health check overhead negligible; single lightweight query per probe interval.
- **Reliability** — Health probes must not share the same DB connection pool as request handlers; use a dedicated 1-connection pool for health checks to avoid false unhealthy on pool exhaustion.
- **Observability** — Prometheus counter `health_check_total{endpoint,status}`; alert if `ready` returns 503 for more than 30 seconds consecutively.
- **Maintainability** — All health check logic in `server/src/routes/health.rs` (already exists; refactor in place).
- **Internationalization** — N/A (JSON status codes, not localized messages).
- **Backward compatibility** — Existing `/health/ready` endpoint path preserved; response schema changes are backward-compatible (adds `checks` object to existing response).

## 7. Acceptance Criteria

- **AC-1.** *Given* the app is running and DB is reachable, *When* `GET /health/ready` is called, *Then* HTTP 200 `{"status":"ready","checks":{"postgres":"ok","redis":"ok"}}` is returned.
- **AC-2.** *Given* the database is unreachable, *When* `GET /health/ready` is called, *Then* HTTP 503 `{"status":"unhealthy","checks":{"postgres":"fail","redis":"ok"}}` is returned with no internal error messages.
- **AC-3.** *Given* the app is running, *When* `GET /health/live` is called, *Then* HTTP 200 `{"status":"ok"}` is returned in < 5 ms regardless of DB state.
- **AC-4.** *Given* an unauthenticated caller, *When* `GET /health/detailed` is called, *Then* HTTP 401 is returned.
- **AC-5.** *Given* the LB health check is configured to use `/health/ready`, *When* the DB goes down, *Then* the LB removes the instance from rotation within 15 seconds (3 consecutive failures × 5 s interval).
- **AC-6.** *Given* the `docker-compose.yml` healthcheck is configured, *When* `docker inspect` shows the container health, *Then* status is `healthy` when the app is running normally.

## 8. Data Model

No schema changes. Dedicated DB connection pool for health checks: 1 connection, separate from the main pool.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Health check uses main connection pool — false unhealthy under load | M | M | Dedicated 1-connection health pool |
| `/health/ready` too slow, causing LB flapping | L | M | 1 s Postgres timeout; 500 ms Redis timeout; p99 < 200 ms target |
| Existing `/health/ready` clients break on schema change | L | L | Additive JSON fields only; `status` field preserved |

## 15. Rollout Plan

- Straightforward refactor; no feature flag needed.
- Deploy to staging; verify LB health check passes; verify 503 on DB outage simulation.
- Deploy to production; update LB health check URL if changed.

## 16. Test Plan

- **Unit** — Postgres `SELECT 1` timeout; Redis PING; response schema serialization.
- **Integration** — Simulate DB down: verify 503 response, no error leakage.
- **End-to-end** — `docker-compose up`; verify `docker inspect` shows `healthy`.
- **Security** — Unauthenticated `/health/detailed` → 401; no internal info in `/health/ready` error.
- **Accessibility** — N/A.
- **Performance / load** — Measure `/health/ready` latency under DB load; ensure < 200 ms p95.
- **Manual exploratory** — QA: kill Redis; verify `checks.redis: fail`; restore Redis; verify `ready` again.
