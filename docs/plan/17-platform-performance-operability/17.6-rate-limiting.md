## 1. Problem Statement

Lextures has no rate limiting on any endpoint. `SECURITY.md` flags this explicitly. An attacker can perform credential-stuffing attacks against `/auth/login`, enumerate users, or exhaust the database with query floods — all at unlimited speed. The public API (16.1) cannot be safely exposed without rate limiting, as a single misbehaving API client could degrade service for all users. Rate limiting is also required for compliance with OWASP API Top 10 (API4: Unrestricted Resource Consumption).

## 2. Goals

- Implement per-IP rate limiting on authentication endpoints to prevent credential stuffing.
- Implement per-token rate limiting on the public API (16.1) to enforce API quotas.
- Implement global request rate limiting per IP to prevent resource exhaustion.
- Return standard `429 Too Many Requests` with `Retry-After` and `X-RateLimit-*` headers.
- Store rate-limit counters in Redis (17.2) so limits are enforced consistently across all app instances.

## 3. Non-Goals

- DDoS mitigation (that is a CDN/load-balancer concern, e.g., Cloudflare).
- Per-user rate limiting for internal routes (per-token limits cover the public API; internal routes are not publicly exposed).
- Adaptive rate limiting based on anomaly detection (phase 2).

## 4. Personas & User Stories

- **As a security engineer**, I want brute-force login attempts to be throttled after 10 failed attempts in 1 minute so that credential stuffing attacks are ineffective.
- **As a public API client developer**, I want to receive `429` with a `Retry-After` header when I exceed my quota so that my integration can back off gracefully.
- **As a student using the web app**, I want rapid clicking on the "Submit Quiz" button to not result in double submissions due to multiple request spikes.
- **As a DevOps engineer**, I want rate-limit counters stored in Redis so that limits are consistent across all app instances.

## 5. Functional Requirements

- **FR-1.** The system MUST enforce per-IP rate limits on authentication endpoints: `POST /auth/login`, `POST /auth/register`, `POST /auth/password-reset` — limit: 10 requests/min per IP, 50/hour.
- **FR-2.** The system MUST enforce per-token rate limits on all `/api/v1/` routes (16.1): configurable per token, default 1,000 requests/min.
- **FR-3.** The system MUST enforce a global per-IP limit on all routes: 300 requests/min per IP (to catch non-API flooding).
- **FR-4.** The system MUST return `HTTP 429 Too Many Requests` with `Retry-After: <seconds>` and `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` headers.
- **FR-5.** Rate-limit counters MUST be stored in Redis with sliding-window or token-bucket semantics; `INCR` + `EXPIRE` or sorted-set sliding window.
- **FR-6.** The system MUST allow per-token quota overrides: institutional API tokens can have custom limits stored in the token metadata (16.2).
- **FR-7.** The system SHOULD distinguish between human users (web app, session-cookie) and API clients (Bearer token) and apply different limit tiers.
- **FR-8.** The system SHOULD allow admin-configured IP allowlists (e.g., institutional IP ranges) that bypass IP-based rate limits.

## 6. Non-Functional Requirements

- **Performance** — Rate-limit check adds ≤ 2 ms per request (single Redis command). Use Redis pipeline where multiple limit checks needed per request.
- **Security** — IP extraction from `X-Forwarded-For` must account for trusted proxy headers only (configurable trusted proxy list); blind `X-Forwarded-For` trust enables IP spoofing bypass.
- **Privacy & Compliance** — IP addresses processed for rate limiting must not be logged with PII. Rate-limit Redis keys use `HMAC(IP, secret)` to prevent IP enumeration of rate-limit keys.
- **Accessibility** — `429` responses must be human-readable for browser users; redirect to a friendly "too many requests" page from the web app.
- **Scalability** — Redis-backed counters scale with the Redis cluster; no per-process state.
- **Reliability** — Redis unavailability falls back to "allow" (fail open) to prevent service disruption; logged as alert.
- **Observability** — Counter `rate_limit_exceeded_total{endpoint_group,limit_type}`; alert on spike (potential attack).
- **Maintainability** — Rate-limit configuration in `server/src/config/rate_limits.rs`; Axum tower middleware layer applied globally and per-route group.
- **Internationalization** — `429` error response body in RFC 7807 format; `detail` field locale-aware.
- **Backward compatibility** — No existing behavior changes for requests under the limit; only over-limit requests see new behavior.

## 7. Acceptance Criteria

- **AC-1.** *Given* an IP sends 11 login requests within 1 minute, *When* the 11th request arrives, *Then* the response is HTTP 429 with a `Retry-After` header and the login is not attempted.
- **AC-2.** *Given* a public API token with a 1,000 req/min limit, *When* the 1,001st request arrives in the same minute, *Then* the response is HTTP 429 with `X-RateLimit-Remaining: 0`.
- **AC-3.** *Given* two app instances are running and an IP sends 150 requests to each instance (300 total), *When* the 301st request arrives on either instance, *Then* the limit is enforced (shared Redis counter).
- **AC-4.** *Given* Redis is unavailable, *When* a rate-limited endpoint is called, *Then* the request is allowed through (fail open), and a metric `rate_limit_redis_miss_total` is incremented.
- **AC-5.** *Given* an admin-configured IP allowlist includes `203.0.113.0/24`, *When* a request from `203.0.113.5` exceeds the per-IP limit, *Then* the request is allowed through without a `429` response.
- **AC-6.** *Given* a `429` response is received by the React web app, *When* the UI displays the error, *Then* a user-friendly message is shown with a countdown until the limit resets.

## 8. Data Model

No DB tables. Redis key patterns:

| Key Pattern | Algorithm | TTL |
|---|---|---|
| `rl:ip:{ip_hash}:auth` | Token bucket / sliding window | 60 s |
| `rl:ip:{ip_hash}:global` | Sliding window | 60 s |
| `rl:token:{token_id}:api` | Token bucket | 60 s |

Using Redis sorted sets for sliding window: `ZADD rl:ip:{h}:auth {ts} {uuid}`, then `ZREMRANGEBYSCORE` + `ZCARD`.

## 9. API Surface

No new routes. Headers added to all responses:
- `X-RateLimit-Limit: 1000`
- `X-RateLimit-Remaining: 999`
- `X-RateLimit-Reset: 1713350400`

On `429`:
```jsonc
{
  "type": "https://lextures.io/errors/rate-limited",
  "title": "Too Many Requests",
  "status": 429,
  "detail": "Rate limit of 10 requests/min exceeded. Retry after 47 seconds.",
  "instance": "/auth/login"
}
```

## 10. UI / UX

- `429` on web app API calls: toast notification "Too many requests — please wait X seconds" using the `Retry-After` value.
- Login form: after 5 failed attempts, show "You've made too many attempts. Please wait before trying again." before the `429` threshold.
- Mobile: same behavior.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| IP spoofing bypasses rate limits | M | H | Validate `X-Forwarded-For` only from trusted proxy IPs; use `X-Real-IP` from LB |
| Legitimate batch clients hit API rate limit | M | M | Per-token quota overrides; burst allowance; clear `X-RateLimit-*` headers for clients |
| Redis unavailability disables rate limiting | M | M | Fail open + alert; document as acceptable short-term risk |
| Sliding window sorted set grows unbounded | L | M | `ZREMRANGEBYSCORE` cleanup on every write; set Redis key TTL |

## 15. Rollout Plan

- **Feature flag:** `rate_limiting_enabled` (OpenFeature, default `false`).
- **Phase 1:** Auth endpoint rate limiting (highest security impact); deploy in monitoring-only mode first.
- **Phase 2:** Global per-IP limit; per-token API limit.
- **Phase 3:** Admin IP allowlists; custom per-token quotas; GA.
- **Rollback:** Disable feature flag; Redis keys TTL out naturally.

## 16. Test Plan

- **Unit** — Token bucket decrement/refill; sliding window cleanup; `Retry-After` calculation.
- **Integration** — Exceed auth limit → verify 429; shared counter across two instances; fail-open on Redis unavailable.
- **End-to-end** — Playwright: 11 rapid login attempts; verify 429 response and UI toast.
- **Security** — IP spoofing attempt (`X-Forwarded-For: 1.2.3.4` from untrusted proxy); brute-force simulation with Hydra against staging.
- **Accessibility** — `429` toast component axe-core; focus management after rate-limit error.
- **Performance / load** — Redis rate-limit check latency benchmark: p99 ≤ 2 ms; overhead measured vs. baseline.
- **Manual exploratory** — QA: API token custom quota; admin allowlist; countdown UI behavior.

## 18. Open Questions

1. Should rate limiting use a sliding window (more accurate, more Redis commands) or a fixed window (simpler, bursty at window boundary)?
2. What is the right default global per-IP limit — 300 req/min may be too restrictive for power users with multiple browser tabs?
3. Should `429` responses count toward the rate limit (to prevent limit-enumeration via rapid 429 responses)?
4. Do we need CAPTCHA as an additional layer after repeated auth failures, or is `429` sufficient?
