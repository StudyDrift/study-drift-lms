## 1. Problem Statement

The `iac/production/` directory contains only a `.gitkeep` file. The current demo deployment (`iac/demo/`) is a single DigitalOcean droplet running `docker-compose` — a setup that cannot serve production traffic at scale, has no redundancy, and uses no managed database. Any institution relying on Lextures for live courses faces unacceptable downtime risk. Without codified production infrastructure, every deployment is manual and risky, and there is no path to SOC 2 or enterprise SLA compliance.

## 2. Goals

- Implement `iac/production/` as a Terraform configuration targeting DigitalOcean managed services (DB cluster, managed Redis, load balancer, spaces/object storage) as the initial cloud provider.
- Ensure the production environment is provisioned entirely from code with no manual console steps.
- Support environment separation: `production` and `staging` workspaces from the same Terraform modules.
- Integrate with secrets management (17.17) so no secrets are stored in Terraform state.
- Document the full provisioning runbook so a new engineer can bring up a production environment from scratch.

## 3. Non-Goals

- Multi-region active-active setup (17.12 — separate plan).
- Kubernetes migration (evaluate in 17.2; out of scope for this plan).
- AWS or GCP as primary provider (DigitalOcean is the current vendor; multi-cloud is 17.12).
- Custom Terraform providers.

## 4. Personas & User Stories

- **As a DevOps engineer**, I want to provision a production environment by running `terraform apply` so that I do not need manual console operations.
- **As a security officer**, I want all infrastructure defined in code and reviewed via pull requests so that infrastructure changes are auditable.
- **As a new team member**, I want a documented runbook so that I can understand and operate the production environment without tribal knowledge.
- **As an on-call engineer**, I want Terraform modules that are idempotent so that I can safely re-apply during an incident without causing additional outages.

## 5. Functional Requirements

- **FR-1.** The system MUST define all production infrastructure in `iac/production/` as Terraform HCL, referencing reusable modules where possible.
- **FR-2.** The production configuration MUST include: DigitalOcean Managed PostgreSQL cluster (primary + replica), Managed Redis cluster, App Platform or Droplet-based compute (behind DO Load Balancer), Spaces (S3-compatible object storage), and a domain/SSL configuration.
- **FR-3.** The Terraform state MUST be stored in a remote backend (DO Spaces bucket or Terraform Cloud) with state locking.
- **FR-4.** All secrets (DB passwords, API keys) MUST be sourced from a secrets manager (17.17), not Terraform variables or state files.
- **FR-5.** The configuration MUST support two Terraform workspaces: `staging` and `production`, with environment-specific sizing (staging: smaller DB/Redis instances).
- **FR-6.** The CI pipeline (`.github/workflows/`) MUST include a `terraform plan` step on PRs targeting `main` and a `terraform apply` step gated on approval for deployments.
- **FR-7.** The production Postgres cluster MUST have daily automated backups configured with 30-day retention (17.11).
- **FR-8.** Network access to the DB cluster MUST be restricted to app-tier IP ranges only; no public internet exposure to the DB.
- **FR-9.** The configuration SHOULD include a bastion/jump host pattern for emergency DB access.

## 6. Non-Functional Requirements

- **Performance** — Managed Postgres and Redis instances sized for ≥ 1,000 concurrent users at launch; instance type documented with scaling guidance.
- **Security** — All resources in a VPC; DB and Redis not publicly accessible. DO Firewall rules applied. TLS enforced on all inter-service communication. Secrets not in Terraform state (17.17 sourcing pattern).
- **Privacy & Compliance** — Data residency: Terraform workspace variables control the DO region; default to Frankfurt (EU) for EU customers, NYC/SFO for US. Encryption at rest enabled on Managed Postgres (DigitalOcean default).
- **Accessibility** — No UI; N/A.
- **Scalability** — Load balancer in front of app tier; horizontal scaling handled by 17.2. Postgres read replica configured from day one.
- **Reliability** — Managed Postgres high-availability (standby replica); automated failover. Managed Redis with replication. DO Load Balancer health checks calling `/health/ready`.
- **Observability** — DO monitoring alerts configured for CPU, memory, disk on all resources; forwarded to observability stack (17.7).
- **Maintainability** — Terraform modules organized: `iac/modules/{compute,database,networking,storage}`; production and staging reference the same modules with different variable inputs.
- **Internationalization** — N/A.
- **Backward compatibility** — Existing demo infrastructure (`iac/demo/`) remains unchanged; production is additive.

## 7. Acceptance Criteria

- **AC-1.** *Given* a fresh DigitalOcean account with credentials configured, *When* `terraform apply` is run in the `production` workspace, *Then* all resources provision successfully with zero manual console steps.
- **AC-2.** *Given* the production environment is provisioned, *When* the Lextures app server starts, *Then* it connects to the Managed Postgres cluster and Managed Redis without hardcoded connection strings in the app image.
- **AC-3.** *Given* a PR modifies `iac/production/`, *When* CI runs, *Then* a `terraform plan` output is posted as a PR comment and no `apply` occurs without manual approval.
- **AC-4.** *Given* the production Postgres cluster is configured, *When* DO's automated backup window passes, *Then* a backup appears in the DO console with a timestamp within the last 24 hours.
- **AC-5.** *Given* the DB security group is configured, *When* a connection attempt is made to the DB port from a non-app-tier IP, *Then* the connection is refused.
- **AC-6.** *Given* `staging` workspace is applied, *When* resource costs are estimated via `terraform plan`, *Then* staging instances are smaller/cheaper than production instances.

## 8. Data Model

Infrastructure state managed by Terraform (not DB migrations). No application DB schema changes.

Key Terraform resources:
```hcl
# iac/production/main.tf (structure)
module "networking" { source = "../modules/networking" }
module "database"   { source = "../modules/database"; ha = true }
module "redis"      { source = "../modules/redis" }
module "compute"    { source = "../modules/compute" }
module "storage"    { source = "../modules/storage" }
module "lb"         { source = "../modules/loadbalancer" }
```

## 9. API Surface

No application API changes. Infrastructure-level:
- DO Load Balancer health check calls `GET /health/ready` (17.8).
- Terraform outputs: `db_connection_string_secret_path`, `redis_url_secret_path`, `lb_ip`, `spaces_bucket_name`.

## 12. Integration Points

- **Internal:** `.github/workflows/ci.yml` — add `terraform plan` job; `.github/workflows/deploy-demo.yml` — create equivalent `deploy-production.yml`.
- **Internal:** `docker-compose.deploy.yml` — reference patterns for app configuration.
- **External:** DigitalOcean Terraform provider (`digitalocean/digitalocean`); Terraform Cloud or DO Spaces for remote state.
- **Internal:** 17.17 (secrets manager) — provisioned as part of this IaC; app references secret paths, not values.

## 13. Dependencies & Sequencing

- Must ship before: effectively everything in section 17 requires a production environment.
- Must ship after: 17.17 design finalized (to know how secrets are sourced in Terraform).
- Shared infra: secrets manager (17.17), observability (17.7).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Terraform state contains sensitive outputs | M | H | `sensitive = true` on all secret outputs; use 17.17 for secrets, not TF vars |
| Drift between Terraform state and actual resources | M | M | CI `terraform plan` on scheduled basis; drift detection alert |
| Cost overrun from production instance sizing | M | M | Cost estimate in `terraform plan` output; budget alert in DO |
| Managed Postgres failover causes downtime | L | H | Test failover in staging before production; connection pool retry logic in app |

## 15. Rollout Plan

- **Phase 1:** Networking and DB modules; staging workspace only.
- **Phase 2:** Full production workspace; load balancer; Redis; object storage.
- **Phase 3:** CI integration; `deploy-production.yml` pipeline; runbook documentation.
- **Rollback:** Terraform destroy (not recommended for production data — document snapshot-first destroy procedure).

## 16. Test Plan

- **Unit** — Terraform `validate` and `fmt` in CI.
- **Integration** — `terraform apply` in staging workspace against real DigitalOcean; verify connectivity from app container.
- **End-to-end** — Deploy full Lextures stack to production environment; run smoke tests.
- **Security** — Network policy audit: attempt DB connection from outside VPC; TLS certificate verification.
- **Accessibility** — N/A.
- **Performance / load** — Load test (17.19) against production-equivalent staging to validate instance sizing.
- **Manual exploratory** — QA: manual failover of Postgres primary; verify app reconnects; backup restore drill (17.11).

## 17. Documentation & Training

- Architecture diagram (network topology, data flow) in `iac/production/README.md`.

## 18. Open Questions

1. Should we migrate to DigitalOcean App Platform (managed compute) or remain on Droplets with a Load Balancer? App Platform simplifies operations but limits configuration.
2. Is DigitalOcean the right long-term provider, or should we plan for AWS/GCP migration in 17.12?
3. Should Terraform state be in DO Spaces or Terraform Cloud? (Terraform Cloud is free for small teams and has a better UI for plan approvals.)
4. How do we handle DB schema migrations during zero-downtime deploys (17.9, 17.10)?
