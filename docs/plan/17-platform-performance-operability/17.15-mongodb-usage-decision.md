## 1. Problem Statement

`docker-compose.yml` includes a MongoDB service container that is not referenced anywhere in the Rust server codebase. It consumes memory on development machines, adds an unexplained dependency to the developer setup, increases the Docker image attack surface, and creates confusion for new contributors ("do I need to understand MongoDB to work on this?"). A decision must be made: either identify a use case and commit to MongoDB, or remove it entirely.

## 2. Analysis: Commit or Remove?

**Arguments for keeping MongoDB:**
- Could be useful for storing schemaless event logs, activity streams, or AI conversation history (flexible document model).
- Some LMS data (AI tutor conversations, raw event streams) is a poor fit for Postgres's relational model.
- If a telemetry/analytics pipeline is planned (9.x), MongoDB's aggregation pipeline could serve as an intermediate store.

**Arguments for removing MongoDB:**
- The Rust ecosystem has first-class Postgres support (`sqlx`, `diesel`); MongoDB's official Rust driver is less mature.
- JSONB in Postgres already provides a document-store-like capability for the small flexible data use cases.
- Adding a second database technology doubles operational burden: backups (17.11), migrations (17.10), monitoring (17.7), and disaster recovery (17.12) must all be duplicated.
- SOC 2 / security audit surface is reduced by having fewer services.
- There is no existing code or feature that uses MongoDB; adding it would be speculative.

**Recommendation: Remove MongoDB from `docker-compose.yml` and `iac/` configurations.** The Lextures stack should commit to Postgres + Redis as the two data stores. If a genuine document-store use case emerges (AI conversation history, analytics events), revisit with a concrete use case and benchmark against Postgres JSONB first. Any future adoption should be a deliberate architectural decision backed by a new plan, not an unconsidered leftover service.

## 3. Goals

- Remove the MongoDB service from `docker-compose.yml`, `docker-compose.deploy.yml`, and `docker-compose.dev.yml`.
- Verify no Rust code, test, or migration references MongoDB.
- Document the removal decision in an Architecture Decision Record (ADR).
- If a legitimate MongoDB use case is identified during the review, create a new plan for it and remove this plan.

## 4. Non-Goals

- Evaluating MongoDB as a future option in detail (the ADR documents the decision; a new plan handles future adoption if needed).
- Migrating any data from MongoDB (there is none).
- Changes to Postgres or Redis configuration.

## 5. Functional Requirements

- **FR-1.** The system MUST remove the `mongo` service definition from `docker-compose.yml`, `docker-compose.deploy.yml`, and `docker-compose.dev.yml`.
- **FR-2.** The `iac/demo/` and `iac/production/` configurations MUST NOT reference MongoDB.
- **FR-3.** A code search MUST be performed for `mongodb`, `mongo`, `bson`, `mongo_uri` strings in the `server/` directory; any references MUST be removed or documented as planned features.
- **FR-4.** An ADR MUST be written at `docs/adr/0001-remove-mongodb.md` documenting the decision, rationale, and conditions under which MongoDB might be re-evaluated.
- **FR-5.** The `README.md` MUST be updated to remove any mention of MongoDB from the technology stack section.

## 6. Non-Functional Requirements

- **Performance** — Removing MongoDB from docker-compose reduces development machine RAM usage by ~200 MB.
- **Security** — Reduces attack surface; MongoDB's default auth (disabled in many `docker-compose` setups) is a security risk if left running.
- **Privacy & Compliance** — No data in MongoDB; no compliance action required.
- **Accessibility** — N/A.
- **Scalability** — No impact.
- **Reliability** — Developers no longer encounter spurious "MongoDB not connected" errors.
- **Observability** — Prometheus does not need a `mongodb_exporter`; simplifies 17.7 dashboards.
- **Maintainability** — Cleaner stack; new contributors do not need to understand why MongoDB is present.
- **Internationalization** — N/A.
- **Backward compatibility** — No production data stored; no users affected.

## 7. Acceptance Criteria

- **AC-1.** *Given* the updated `docker-compose.yml`, *When* `docker-compose up` is run, *Then* no MongoDB container starts and no application error occurs.
- **AC-2.** *Given* a full-text search of the `server/` directory for `mongodb` and `bson`, *When* the search completes, *Then* zero results are found.
- **AC-3.** *Given* the ADR at `docs/adr/0001-remove-mongodb.md`, *When* a new engineer reads it, *Then* they understand why MongoDB was removed and what conditions would prompt re-evaluation.
- **AC-4.** *Given* the CI pipeline runs, *When* tests execute, *Then* all tests pass without a running MongoDB instance.
- **AC-5.** *Given* `iac/demo/main.tf` is reviewed, *When* a search for `mongo` is performed, *Then* zero Terraform resources reference MongoDB.
