## 1. Problem Statement

Lextures has no documented backup policy, no automated database snapshots configured in IaC, and no tested restore procedure. This is cross-referenced as gap 10.15 (compliance). For any institutional deployment, data loss protection is a contractual requirement, and demonstrating a tested restore is standard in SOC 2 Type II audits. The current single-droplet setup means a disk failure or accidental DROP TABLE would result in permanent data loss.

## 2. Goals

- Configure automated daily Postgres backups (managed snapshots via DigitalOcean, plus `pg_dump` exports to object storage for a second copy).
- Configure automated file/object storage backups (DO Spaces cross-region replication or daily snapshot).
- Achieve a documented RPO (Recovery Point Objective) of ≤ 24 hours and RTO (Recovery Time Objective) of ≤ 2 hours for a full-database restore.
- Perform and document a quarterly restore drill in staging, verifying data integrity post-restore.
- Automate backup health monitoring: alert if a backup is not completed within the expected window.

## 3. Non-Goals

- Continuous (near-zero RPO) replication (that is 17.12 multi-region; Postgres logical replication).
- Application-level point-in-time recovery (PITR) of individual rows.
- Backup of Redis data (Redis is a cache; data loss acceptable at restart).

## 4. Personas & User Stories

- **As a university IT director**, I want to see documented RPO/RTO commitments in our contract so that I can assess risk for our production adoption of Lextures.
- **As a SOC 2 auditor**, I want evidence of automated backup runs and quarterly restore drill documentation so that I can verify the backup control is effective.
- **As an on-call engineer**, I want a tested runbook for restoring the Postgres database from backup within 2 hours so that I can recover from accidental data loss without guesswork.
- **As a DevOps engineer**, I want a Grafana alert that fires if today's backup has not completed by 03:00 UTC so that I catch backup failures before they accumulate into multi-day gaps.

## 5. Functional Requirements

- **FR-1.** The system MUST configure DigitalOcean Managed Postgres automated daily backups with 30-day retention via `iac/production/` Terraform.
- **FR-2.** The system MUST configure a daily `pg_dump` export (compressed, encrypted with AES-256) to a dedicated DO Spaces backup bucket, retained for 90 days, via the scheduled job system (17.4).
- **FR-3.** The system MUST configure DO Spaces versioning on the uploads bucket so that file deletions are recoverable for 30 days.
- **FR-4.** The system MUST alert (via 17.7 Alertmanager) if the daily `pg_dump` backup job has not completed successfully by 04:00 UTC.
- **FR-5.** The restore runbook MUST be written and tested, covering: (a) restore from DO Managed Postgres snapshot, (b) restore from `pg_dump` S3 export, (c) smoke test checklist post-restore.
- **FR-6.** The system MUST perform and document a quarterly restore drill: restore the most recent backup to a separate staging database, run the smoke test checklist, and archive the results.
- **FR-7.** Backup encryption keys MUST be stored in the secrets manager (17.17), not in the backup files or IaC configuration.
- **FR-8.** The system SHOULD encrypt the DO Managed Postgres connection to the backup export job using TLS and verify the backup file checksum (SHA-256) after each export.

## 6. Non-Functional Requirements

- **Performance** — `pg_dump` export of a 50 GB database must complete within 2 hours without impacting production query performance (use `pg_dump --no-password --lock-wait-timeout=0`; run against read replica if available).
- **Security** — Backup files AES-256 encrypted; encryption key in secrets manager (17.17). Backup bucket access restricted to backup job service account; no public access. Backup log entries do not contain connection strings.
- **Privacy & Compliance** — Backup files contain all user data; same FERPA/GDPR data handling obligations apply. Backup retention periods documented in DPA. GDPR right-to-erasure: erasure from backup files is not required for 90-day old backups per GDPR recital 65 (reasonable); documented in privacy policy.
- **Accessibility** — N/A.
- **Scalability** — Backup process does not block production traffic; runs on read replica where available.
- **Reliability** — Two independent backup copies (DO Managed + `pg_dump` to Spaces) provides defense-in-depth; single backup failure does not lose all recovery options.
- **Observability** — Prometheus gauge `backup_last_success_timestamp_seconds{type=managed|pgdump}`; alert if value is > 25 hours old.
- **Maintainability** — Backup configuration in `iac/production/` Terraform; backup job in `server/src/jobs/` (BackupExportJob); restore runbook in `docs/runbooks/`.
- **Internationalization** — N/A.
- **Backward compatibility** — Backup process is additive; no existing behavior changes.

## 7. Acceptance Criteria

- **AC-1.** *Given* the production IaC is applied, *When* 24 hours pass, *Then* a DO Managed Postgres snapshot appears in the DO console with a timestamp within the last 24 hours.
- **AC-2.** *Given* the daily backup job runs, *When* it completes, *Then* a `.dump.gz.enc` file appears in the backup Spaces bucket with a SHA-256 checksum file alongside it.
- **AC-3.** *Given* the daily backup job fails to complete by 04:00 UTC, *When* Prometheus evaluates the alert, *Then* an alert fires and a notification is sent to the on-call channel.
- **AC-4.** *Given* the restore runbook is executed against the latest `pg_dump` backup, *When* the restored database is connected to a staging app, *Then* the smoke test checklist passes (login, course load, grade read) within 2 hours.
- **AC-5.** *Given* a file is accidentally deleted from the uploads Spaces bucket, *When* the Spaces versioning policy is checked, *Then* the deleted file can be recovered from a previous version.
- **AC-6.** *Given* the quarterly restore drill documentation is reviewed, *When* it is submitted to the SOC 2 auditor, *Then* it contains: backup source, restore timestamp, smoke test results, and engineer sign-off.

## 8. Data Model

```sql
-- server/migrations/083_backup_log.sql
CREATE TABLE ops.backup_log (
    id              BIGSERIAL PRIMARY KEY,
    backup_type     TEXT NOT NULL,      -- 'pgdump', 'managed_snapshot'
    started_at      TIMESTAMPTZ NOT NULL,
    completed_at    TIMESTAMPTZ,
    status          TEXT NOT NULL,      -- 'running', 'success', 'failed'
    file_path       TEXT,               -- Spaces object key (not URL)
    file_size_bytes BIGINT,
    checksum_sha256 TEXT,
    error_message   TEXT
);
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Backup job runs on primary, impacting prod queries | M | M | Run against read replica; `--lock-wait-timeout=0` |
| Backup encryption key lost | L | H | Key in secrets manager with rotation plan; backup of key in separate secure store |
| Backup file corruption not detected | L | H | SHA-256 checksum verified post-export; alert on mismatch |
| Restore takes > 2 hours (RTO breach) | M | M | Pre-warm restore environment; parallel restore with `pg_restore -j 4` |

## 15. Rollout Plan

- **Phase 1:** Enable DO Managed Postgres daily backups in `iac/production/`; verify in DO console.
- **Phase 2:** `pg_dump` scheduled job + backup Spaces bucket + encryption.
- **Phase 3:** Monitoring alert; first restore drill in staging; runbook finalized.
- **Rollback:** Backups are additive; disabling them is the "rollback."

## 18. Open Questions

1. Should we use Postgres logical replication to a standby (near-zero RPO) rather than nightly `pg_dump`? That is 17.12; for now, daily backup is the baseline.
2. Should the backup job run against the primary or the read replica? (Read replica preferred but requires one to be provisioned — 17.1 decision.)
3. What is the right backup retention policy for SOC 2: 30 days (DO managed) + 90 days (pg_dump)? Or do we need 1-year retention for some regulations?
