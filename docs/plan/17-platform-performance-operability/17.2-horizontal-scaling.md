## 1. Problem Statement

The current deployment is a single DigitalOcean droplet running `docker-compose`. There is no load balancer, no shared state layer, and no guarantee that the Axum application is stateless. Any per-process in-memory state (caches, rate-limit counters, upload progress) breaks if a second instance is deployed. A single-node deployment cannot survive rolling deploys without downtime, cannot absorb traffic spikes, and fails all enterprise SLA requirements for Lextures to be adopted by K-12 districts and universities.

## 2. Goals

- Audit and eliminate all in-process mutable state from the Axum server.
- Deploy a DigitalOcean Load Balancer (or equivalent) in front of two or more app instances.
- Introduce managed Redis for: session/JWT revocation lists, rate-limit counters, API token validation cache, feed caches, and general short-TTL key-value caching.
- Ensure the updated `docker-compose.deploy.yml` and `iac/production/` Terraform support multi-instance deployments.
- Demonstrate zero-downtime rolling restart with two app instances behind the load balancer.

## 3. Non-Goals

- Kubernetes or container orchestration (evaluate post-MVP; out of scope).
- Database connection pooling beyond PgBouncer (covered separately if needed).
- Application-level caching strategy (17.5 covers HTTP/CDN caching; this plan covers Redis as the shared memory layer).
- Multi-region active-active (17.12).

## 4. Personas & User Stories

- **As a DevOps engineer**, I want to scale the app tier from one to three instances without any downtime so that I can absorb traffic spikes during semester start.
- **As an on-call engineer**, I want to restart a single app instance without dropping in-flight requests so that routine maintenance does not require a maintenance window.
- **As an architect**, I want all per-process caches to be backed by Redis so that adding a second app node does not cause cache incoherence.
- **As a security engineer**, I want JWT revocation to propagate to all nodes within 30 seconds so that a compromised session cannot outlive revocation on any node.

## 5. Functional Requirements

- **FR-1.** The system MUST route all HTTP traffic through a DigitalOcean Load Balancer with health-check-based instance removal (calling `GET /health/ready` — see 17.8).
- **FR-2.** The Axum application MUST be stateless: no Axum `Extension` or global state that is mutated per-request and not backed by an external store.
- **FR-3.** The system MUST introduce a shared Redis instance (managed, via 17.1 IaC) accessible to all app nodes with the following namespaced uses: `rate:*` (17.6), `token:*` (16.2), `feed:*` (16.5), `session:*` (JWT blocklist), `jobs:*` (17.3).
- **FR-4.** JWT revocation MUST be enforced via a Redis blocklist: on logout or token revocation, the JWT's `jti` claim is added to Redis with TTL equal to the token's remaining lifetime.
- **FR-5.** Upload progress tracking (if any per-process state exists) MUST be migrated to Redis or removed.
- **FR-6.** The `docker-compose.deploy.yml` MUST be updated to support running multiple `api` service replicas behind a reverse proxy (Caddy or nginx) for local multi-instance testing.
- **FR-7.** Connection pool settings (sqlx `max_connections`) MUST be tuned for multi-instance deployment to avoid exceeding Postgres connection limits.
- **FR-8.** The system SHOULD implement graceful shutdown: on SIGTERM, the app drains in-flight requests within a 30-second window before exiting.

## 6. Non-Functional Requirements

- **Performance** — Load balancer adds ≤ 2 ms latency overhead. Redis connection pool: minimum 5, maximum 20 connections per app instance. Redis reads p99 ≤ 2 ms.
- **Security** — Redis connection uses TLS (managed Redis requires TLS on DigitalOcean). Redis not publicly accessible; VPC-only. No secrets in Redis values (only opaque tokens/session IDs).
- **Privacy & Compliance** — JWT blocklist entries in Redis do not contain PII — only the `jti` UUID and expiry.
- **Accessibility** — N/A (infrastructure change).
- **Scalability** — Tested at 3 instances; architecture supports up to 10 without design changes. Postgres max connections = instances × pool_size; size pool accordingly (e.g., 3 instances × 20 connections = 60; Postgres limit 100 → use PgBouncer if needed).
- **Reliability** — Load balancer health check removes unhealthy instance from rotation within 15 seconds of first failed check. App graceful shutdown (FR-8) prevents in-flight request drops during rolling restart.
- **Observability** — Prometheus gauge `app_instances_healthy`; Redis metrics via `redis_exporter`; LB request distribution metric.
- **Maintainability** — Statelessness enforced by code review policy and automated test: CI test starts two app instances against shared Redis and runs integration suite.
- **Internationalization** — N/A.
- **Backward compatibility** — Single-instance `docker-compose.yml` (development) unchanged; multi-instance is a deployment-time concern.

## 7. Acceptance Criteria

- **AC-1.** *Given* two app instances are running behind the load balancer, *When* one instance is stopped, *Then* the load balancer routes all traffic to the remaining instance within 15 seconds, with no 5xx errors on in-flight requests.
- **AC-2.** *Given* a user logs out (JWT revocation), *When* the revoked JWT is used on a different app instance within 30 seconds, *Then* the second instance returns HTTP 401 (blocklist checked from Redis).
- **AC-3.** *Given* a rate-limit counter is incremented on instance A, *When* a request arrives on instance B for the same token, *Then* instance B reads the same counter from Redis and enforces the limit correctly.
- **AC-4.** *Given* the app receives SIGTERM, *When* in-flight requests are still processing, *Then* those requests complete before the process exits (max 30 s drain window).
- **AC-5.** *Given* the Postgres connection pool is sized at 20 per instance and 3 instances are running, *When* all 60 connections are active, *Then* the Postgres server is not overloaded and queries queue rather than fail.
- **AC-6.** *Given* `docker-compose.deploy.yml` is used with `--scale api=2`, *When* the stack starts, *Then* both API instances register as healthy with the reverse proxy.

## 8. Data Model

No application schema changes. Redis key namespaces:

| Namespace | Content | TTL |
|-----------|---------|-----|
| `session:jti:{jti}` | `"revoked"` | JWT remaining lifetime |
| `token:{prefix}` | Serialized token metadata | 15 min |
| `rate:{scope}:{key}` | Request counter | Window duration |
| `feed:{user_id}` | Cached iCal feed | 15 min |
| `jobs:queue:*` | Job queue (17.3) | Varies |

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Hidden stateful global discovered during audit | M | H | Comprehensive code search for `static mut`, `once_cell` mutable state; CI test with two instances |
| Redis connection pool exhaustion | M | M | Pool size tuning; monitor `redis_connected_clients`; alert at 80% utilization |
| Postgres connection limit exceeded with multiple instances | M | H | PgBouncer sidecar if per-instance pools × instances > Postgres max_connections |
| Rolling restart drops requests if drain window too short | L | M | Tune SIGTERM drain window (30 s default); load balancer pre-drain health check |

## 15. Rollout Plan

- **Phase 1:** Redis integration + JWT blocklist; statelessness audit and fixes; single-instance deployment (no visible change to users).
- **Phase 2:** Two-instance staging deployment behind load balancer; integration test suite run.
- **Phase 3:** Production two-instance deployment; monitor for 1 week; enable autoscaling on traffic signals.
- **Rollback:** Return to single-instance by removing load balancer; Redis remains and is used by single instance.

## 16. Test Plan

- **Unit** — JWT blocklist Redis integration; graceful shutdown signal handling; connection pool configuration.
- **Integration** — Two-instance local stack: verify rate limits shared; verify revocation propagates; verify no 5xx during rolling restart.
- **End-to-end** — Playwright: complete user journey (login → course → quiz) with two-instance stack; simulate instance restart mid-session.
- **Security** — Redis connection TLS verified; Redis not reachable from outside VPC.
- **Accessibility** — N/A.
- **Performance / load** — k6: 500 VUs against 2-instance stack; compare latency distribution with single-instance baseline; verify LB distributes traffic approximately equally.
- **Manual exploratory** — QA: kill one instance mid-request; observe LB failover; verify graceful drain.
