## 1. Problem Statement

There is currently no mechanism to prevent students from seeing their grades immediately after they are entered. Instructors need to enter all grades before releasing any, to avoid disclosing scores to some students while others have yet to submit. Universities frequently hold grades until a class-wide release, and K-12 instructors want to review scores holistically before students see them. The absence of a hold-and-release workflow forces instructors to use external spreadsheets or avoid entering grades in Lextures until they are ready — directly undermining the gradebook's utility.

## 2. Goals

- Introduce a per-assignment posting policy: "automatic" (release as entered) or "manual" (hold until instructor posts).
- Allow instructors to release grades for an entire assignment with a single action.
- Allow selective release (post grades for specific students, e.g., those who submitted on time).
- Optionally schedule automatic release at a future date/time.
- Integrate with blind grading (3.3): revealing identities does not automatically post grades.
- Log all post/hold events in the grade-change audit trail (3.10).

## 3. Non-Goals

- Holding grades after posting (once posted, students see them immediately).
- Per-student hold without a per-assignment context (addressed differently by excused/exempt 3.12).
- Automatic grade release based on quiz submission (quiz scores remain immediate for now — separate policy).

## 4. Personas & User Stories

- **As an instructor**, I want to enter all essay grades before any student can see them so that students who submitted late do not have an unfair advantage knowing classmates' scores.
- **As an instructor**, I want to schedule automatic grade release at 5 PM Friday so that I do not need to be online at that moment.
- **As a student**, I want a clear indicator "Grades not yet released" in My Grades so that I know my score exists but is pending, rather than assuming I have not been graded.
- **As an admin**, I want to know when grades for an assignment were posted and by whom, to satisfy audit requests.

## 5. Functional Requirements

- **FR-1.** The system MUST support two posting policy modes per assignment: `automatic` (default, existing behaviour) and `manual` (hold until instructor releases).
- **FR-2.** In `manual` mode, entered grades MUST be stored and visible in the instructor gradebook but MUST NOT be visible to students until explicitly posted.
- **FR-3.** The system MUST provide a "Post grades" action per assignment that makes all held grades visible to students simultaneously.
- **FR-4.** The system MUST provide a "Post grades for selected students" action for partial release.
- **FR-5.** The system MUST allow scheduling automatic release at a specific `release_at` timestamp per assignment; a background job (17.3) executes the release.
- **FR-6.** The student "My Grades" view MUST show a "Grades pending" placeholder for assignments in manual mode where grades have been entered but not posted.
- **FR-7.** The system MUST record a `grades_posted` audit event (3.10) capturing `assignment_id`, `posted_by_user_id`, `posted_at`, and count of grades released.
- **FR-8.** The system MUST integrate with 3.4 moderated grading: if an assignment uses moderated grading, the "Post grades" action MUST be blocked until all submissions are reconciled.

## 6. Non-Functional Requirements

- **Performance** — Bulk post for an assignment (up to 500 students) completes in ≤ 2 s. Scheduled release job runs within 30 s of the `release_at` time.
- **Security** — Post action restricted to instructor/owner role enforced in `server/src/authz.rs`. Students cannot query held grade values via the API.
- **Privacy & Compliance** — Held grades are FERPA records. Access controls must prevent any student-accessible API endpoint from returning a held grade value. Post events logged per FERPA audit requirements.
- **Accessibility** — "Post grades" button and schedule modal keyboard accessible. WCAG 2.1 AA.
- **Scalability** — Scheduled release jobs use the 17.3 job queue; no polling required by the server process.
- **Reliability** — Scheduled release is idempotent; if the job runs twice, no double-post occurs (check `posted_at IS NULL` before posting).
- **Observability** — Log `grades_posted` with assignment_id, count. Metric: `grade_post.count`, `grade_post.scheduled_release_lag_seconds`.
- **Maintainability** — Posting logic in `server/src/services/grading/posting.rs`; route in `server/src/routes/grading.rs`.
- **Internationalization** — "Grades pending" and "Post grades" labels i18n-keyed.
- **Backward compatibility** — All existing assignments default to `automatic`; no change to student visibility for existing grades.

## 7. Acceptance Criteria

- **AC-1.** *Given* an assignment with `posting_policy = 'manual'`, *when* an instructor enters a score for a student, *then* that student's My Grades view shows "Grades pending" — not the score.
- **AC-2.** *Given* manual mode with grades entered for 30 students, *when* the instructor clicks "Post grades," *then* all 30 students immediately see their scores and a `grades_posted` audit event is recorded.
- **AC-3.** *Given* a scheduled release at `release_at = 2026-05-01T17:00:00Z`, *when* that time passes and the background job runs, *then* grades are posted automatically and logged.
- **AC-4.** *Given* an assignment with moderated grading and unreconciled submissions, *when* the instructor clicks "Post grades," *then* the system returns an error: "X submissions still need reconciliation."
- **AC-5.** *Given* the instructor calls the gradebook API while grades are in manual hold, *when* a student's JWT is used, *then* the API returns `null` or omits the score field for held grades.
- **AC-6.** *Given* "Post grades for 5 selected students," *when* the action completes, *then* only those 5 students see their scores; the remaining students still see "Grades pending."

## 8. Data Model

```sql
-- Migration: server/migrations/080_grade_posting_policies.sql

ALTER TABLE module_assignments
    ADD COLUMN posting_policy TEXT NOT NULL DEFAULT 'automatic'
        CHECK (posting_policy IN ('automatic','manual')),
    ADD COLUMN release_at TIMESTAMPTZ;

ALTER TABLE course_grades
    ADD COLUMN posted_at TIMESTAMPTZ;
-- NULL = held; NOT NULL = posted and visible to student

CREATE INDEX idx_course_grades_posted ON course_grades(assignment_id)
    WHERE posted_at IS NULL;
```

## 9. API Surface

| Verb | Path | Auth | Description |
|---|---|---|---|
| POST | `/api/courses/:cid/assignments/:aid/post-grades` | instructor | Post all held grades |
| POST | `/api/courses/:cid/assignments/:aid/post-grades/select` | instructor | Post grades for specific student IDs |
| DELETE | `/api/courses/:cid/assignments/:aid/post-grades` | instructor | Retract posted grades (return to held) |
| PATCH | `/api/courses/:cid/assignments/:aid` | instructor | Set `posting_policy` and `release_at` |

Student-facing grade endpoints filter: `WHERE posted_at IS NOT NULL OR posting_policy = 'automatic'`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Student sees grade via direct API call with held status | H | H | Server-side filter on `posted_at IS NOT NULL` before returning grade to student scope |
| Scheduled release job delayed or lost | M | M | Job queue persists jobs; monitor `grade_post.scheduled_release_lag_seconds` alert |
| Instructor accidentally retracts posted grades | L | H | Confirmation modal; audit event; retraction requires explicit intent |

## 15. Rollout Plan

- Feature flag: `GRADE_POSTING_POLICIES_ENABLED` (default: `true` — low risk, existing behaviour is `automatic`).
- Schema migration → all existing `course_grades` rows backfilled with `posted_at = created_at`.
- New assignment creation UI shows posting policy toggle.
- GA criteria: student-facing grade endpoint integration test confirms no held grade leakage.

## 16. Test Plan

- **Unit** — `post_grades()` service function; idempotency check; scheduled job logic.
- **Integration** — Grade API with student JWT returns null for held grade; instructor JWT returns score.
- **End-to-end** — Playwright: set manual policy → enter grade → student sees pending → instructor posts → student sees score.
- **Security** — Verify student cannot access held grade via any API path; authz matrix.
- **Accessibility** — axe-core on gradebook with lock icons and pending state.
- **Manual exploratory** — Schedule release; verify timing within 30 s; test retraction.
