## 1. Problem Statement

Lextures generates quiz questions via LLM but has no taxonomy of concepts, skills, or prerequisites underpinning those questions. This means the system cannot determine whether a student who struggles with "quadratic equations" is actually missing the prerequisite "factoring polynomials," nor can it propose a coherent remediation path. Every other adaptive feature (learner model, spaced repetition, IRT calibration, recommendations) requires a concept graph as its substrate. Without it the product cannot credibly claim adaptive learning behaviour.

## 2. Goals

- Provide a persistent, queryable directed acyclic graph (DAG) of concepts with prerequisite edges.
- Allow instructors and admins to author and curate concepts via UI and API.
- Allow questions (quiz items) to be tagged to one or more concepts.
- Expose a stable internal Rust API for graph traversal (ancestors, descendants, nearest gap).
- Support import of well-known taxonomies (Bloom's levels, Common Core domain labels) as concept metadata, enabling 1.3.

## 3. Non-Goals

- Full standards-alignment import pipeline (covered in 1.3).
- Automatic concept extraction from course content (future AI feature, not in this plan).
- Multi-tenant concept graphs at the district/institution level (covered in 5.1–5.2); initial scope is a global shared graph with course-level overrides.
- Student-facing concept browser UI (deferred; instructors only in v1).

## 4. Personas & User Stories

- **As a course author**, I want to tag each quiz question with one or more concepts so the system can track student mastery per concept.
- **As an instructor**, I want to define that "solving linear equations" is a prerequisite of "solving quadratic equations" so the adaptive engine never assigns the latter before mastery of the former.
- **As a platform admin**, I want to import a Bloom's taxonomy scaffold so instructors can align new concepts to cognitive levels without re-inventing the wheel.
- **As a self-learner**, I want the system to automatically navigate me from my current knowledge gaps backward through prerequisites to find the right starting point.
- **As a developer**, I want a Rust trait for graph traversal so that recommendation and adaptive-quiz services can query concept relationships without raw SQL.

## 5. Functional Requirements

- **FR-1.** The system MUST store concepts in a relational table with `id`, `slug`, `name`, `description`, `bloom_level` (enum), and optional `parent_concept_id` for lightweight hierarchical grouping.
- **FR-2.** The system MUST store prerequisite edges in a separate `concept_prerequisites` join table (directed: concept A requires concept B) to represent a DAG separately from the hierarchy.
- **FR-3.** The system MUST enforce DAG integrity: inserting a prerequisite edge that would create a cycle MUST be rejected with a descriptive error.
- **FR-4.** The system MUST provide a `concept_question_tags` table linking `question_id` to `concept_id` (many-to-many) so questions can be tagged at creation time and retroactively.
- **FR-5.** The system MUST expose CRUD REST endpoints for concepts and prerequisite edges (`/api/v1/concepts`, `/api/v1/concepts/{id}/prerequisites`), accessible to instructors and admins.
- **FR-6.** The system MUST expose a `GET /api/v1/concepts/{id}/ancestors` and `GET /api/v1/concepts/{id}/descendants` endpoint for graph traversal.
- **FR-7.** The system SHOULD expose a `GET /api/v1/concepts/search?q=` full-text search endpoint for use in the question-tagging UI.
- **FR-8.** The system SHOULD provide a seed migration containing a minimal starter taxonomy (e.g., Bloom's six levels as top-level concepts) that installations can extend.

## 6. Non-Functional Requirements

- **Performance** — Concept CRUD p95 ≤ 100 ms. Ancestor/descendant traversal for graphs up to depth 10 and 500 nodes MUST complete in ≤ 200 ms using a recursive CTE.
- **Security** — Concept creation and edge management require `instructor` or `admin` role. Read endpoints are accessible to authenticated users. Cycle-detection runs server-side; client cannot bypass it.
- **Privacy & Compliance** — Concept graph contains no PII; standard schema-level access controls apply. FERPA/GDPR obligations fall on tables that reference `user_id`, not on the concept graph itself.
- **Accessibility** — Concept tagging UI (e.g., multi-select tag input in question editor) MUST be keyboard navigable and screen-reader announced per WCAG 2.1 AA.
- **Scalability** — Initial deployment target: 10 k concepts, 50 k prerequisite edges. Recursive CTEs sufficient at this scale; consider materialized path (ltree) if graph exceeds 100 k nodes.
- **Reliability** — DAG cycle check MUST be atomic with edge insertion; use a Postgres constraint trigger or application-level check within a serializable transaction.
- **Observability** — Log concept creation/deletion events to the audit log. Expose Prometheus counter `concept_graph.edge_count` updated on write.
- **Maintainability** — Graph traversal logic in `server/src/services/concept_graph.rs`; route handlers in `server/src/routes/concepts.rs`. No business logic in handlers.
- **Internationalization** — `name` and `description` fields support Unicode. Locale-specific concept names deferred to a `concept_translations` table (v2).
- **Backward compatibility** — Adds new tables only; existing `module_quizzes.questions_json` remains unchanged. `concept_question_tags` rows are additive.

## 7. Acceptance Criteria

- **AC-1.** *Given* an admin sends `POST /api/v1/concepts` with a valid name and bloom_level, *when* the request succeeds, *then* a concept row is returned with a UUID, slug auto-derived from name, and a 201 status.
- **AC-2.** *Given* concept A has prerequisite B which has prerequisite C, *when* `GET /api/v1/concepts/A/ancestors` is called, *then* the response contains both B and C.
- **AC-3.** *Given* concepts A → B (A requires B) exist, *when* an attempt is made to add edge B → A, *then* the API returns HTTP 422 with a cycle-detected error.
- **AC-4.** *Given* a question is tagged to concept X via `concept_question_tags`, *when* a quiz including that question is graded, *then* the learner model update (1.1) references concept X.
- **AC-5.** *Given* a full-text search query `q=linear`, *when* `GET /api/v1/concepts/search?q=linear` is called, *then* concepts whose name or description contains "linear" are returned in relevance order.
- **AC-6.** *Given* the seed migration has run, *when* `GET /api/v1/concepts?parent=bloom` is called, *then* six Bloom's-level concepts are returned.

## 8. Data Model

```sql
-- Migration: server/migrations/0029_concept_graph.sql

CREATE TYPE course.bloom_level AS ENUM (
    'remember', 'understand', 'apply', 'analyze', 'evaluate', 'create'
);

CREATE TABLE course.concepts (
    id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    slug              TEXT NOT NULL UNIQUE,
    name              TEXT NOT NULL,
    description       TEXT,
    bloom_level       course.bloom_level,
    parent_concept_id UUID REFERENCES course.concepts(id) ON DELETE SET NULL,
    created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_concepts_parent ON course.concepts (parent_concept_id);
CREATE INDEX idx_concepts_name_fts ON course.concepts USING gin(to_tsvector('english', name || ' ' || COALESCE(description, '')));

CREATE TABLE course.concept_prerequisites (
    concept_id      UUID NOT NULL REFERENCES course.concepts(id) ON DELETE CASCADE,
    prerequisite_id UUID NOT NULL REFERENCES course.concepts(id) ON DELETE CASCADE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (concept_id, prerequisite_id),
    CHECK (concept_id <> prerequisite_id)
);

CREATE INDEX idx_cp_prerequisite ON course.concept_prerequisites (prerequisite_id);

-- Junction table: questions <-> concepts (question_id is UUID matching questions in bank, 2.1)
CREATE TABLE course.concept_question_tags (
    concept_id  UUID NOT NULL REFERENCES course.concepts(id) ON DELETE CASCADE,
    question_id UUID NOT NULL,   -- FK added when question bank (2.1) ships
    PRIMARY KEY (concept_id, question_id)
);
```

Backfill: none required. New questions tagged forward; existing `questions_json` items can be retroactively tagged via admin bulk-tag tool (v2).

## 9. API Surface

```
POST   /api/v1/concepts                          Auth: instructor|admin
GET    /api/v1/concepts?parent=&bloom=&q=        Auth: authenticated
GET    /api/v1/concepts/{id}                     Auth: authenticated
PUT    /api/v1/concepts/{id}                     Auth: instructor|admin
DELETE /api/v1/concepts/{id}                     Auth: admin

POST   /api/v1/concepts/{id}/prerequisites       Auth: instructor|admin
  Body: { prerequisite_id: string }
DELETE /api/v1/concepts/{id}/prerequisites/{pid} Auth: instructor|admin

GET    /api/v1/concepts/{id}/ancestors           Auth: authenticated
GET    /api/v1/concepts/{id}/descendants         Auth: authenticated

GET    /api/v1/concepts/search?q=               Auth: authenticated
```

Ancestor/descendant response: `{ nodes: Concept[], edges: [ConceptId, ConceptId][] }`.

## 10. UI / UX

**Concept tag picker** (inline in question editor, `clients/web/src/components/quiz/`):
- Typeahead multi-select backed by `/api/v1/concepts/search`.
- Each selected tag shows a chip with concept name and Bloom's level badge.
- Keyboard: arrow keys to navigate suggestions, Enter to select, Backspace to remove chip.
- ARIA: `role="combobox"`, `aria-expanded`, `aria-activedescendant`.

**Concept management page** (`/admin/concepts`, new page):
- Tree view of concept hierarchy.
- Prerequisite graph visualisation (dagre-d3 or React Flow, lightweight).
- Create / edit / delete modals.
- Import CSV button (admin only).

Empty state: "No concepts yet — add your first concept or import a taxonomy."
Error state: "Cycle detected — this prerequisite would create a circular dependency."

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Cycle detection race condition under concurrent inserts | L | H | Use serializable transaction isolation for edge insertion |
| Concept graph becomes too large for recursive CTEs | L | M | Add ltree extension and materialised path column as migration if node count exceeds 50 k |
| Instructors don't tag questions, leaving mastery tracking empty | M | H | Auto-suggest top-3 concept tags via LLM during question creation (v2 enhancement) |
| Taxonomy disagreements between instructors | M | L | Make concept graph per-course-scoped in v2 (currently global with course override flag reserved) |

## 15. Rollout Plan

- **Feature flag:** `concept_graph_enabled` (default: `false`).
- **Migration sequencing:** deploy `0029_concept_graph.sql` → seed Bloom's taxonomy → deploy UI behind flag → enable for beta instructors → GA.
- **Pilot cohort:** two test courses; instructors manually tag 50 questions.
- **GA criteria:** at least 500 questions tagged across pilot courses; search latency p95 ≤ 100 ms.
- **Rollback:** disable flag; tables are additive, rollback is dropping tables (only safe before production data exists).

## 16. Test Plan

- **Unit** — cycle detection function (pure Rust graph traversal test); slug generation from name (Unicode edge cases).
- **Integration** — create concept, add prerequisite, fetch ancestors; attempt cycle → expect 422; FTS search for known term.
- **End-to-end** — Playwright: instructor opens question editor, searches for concept, selects tag, saves question, verifies tag persists on reload.
- **Security** — student cannot create/edit/delete concepts (403); instructor can; admin can delete.
- **Accessibility** — axe scan on concept tag picker; keyboard navigation test.
- **Performance / load** — recursive CTE on a 500-node graph completes in ≤ 200 ms (measure via Postgres `EXPLAIN ANALYZE`).
- **Manual exploratory** — import Bloom's seed, verify six top-level nodes; manually build a 5-level prerequisite chain and verify ancestor traversal.

## 18. Open Questions

1. Should the concept graph be global (shared across all courses/orgs) or per-course? Global is simpler for v1 but may cause concept-name conflicts across subjects.
2. Do we ship a richer default taxonomy (Common Core Math domains, NGSS disciplinary core ideas) in the seed migration, or leave that to 1.3?
3. What is the UI for the prerequisite graph — a simple list of "requires" chips, or a full visual DAG editor?
4. Should `concept_question_tags` reference the question bank (2.1) or the inline `questions_json` array index? Decision affects how deeply 2.1 must ship before tagging is useful.
