## 1. Problem Statement

Currently, quiz questions in Lextures are stored as a JSONB blob (`questions_json`) inside `module_quizzes`. There is no standalone question entity: questions cannot be shared across quizzes, versioned, tagged, searched, or assembled into random-pull pools. Every quiz is a closed silo. This architecture blocks every advanced assessment feature: spaced repetition, IRT calibration, QTI import, adaptive item selection, and reusable item pools. Building a normalised question bank is the foundational prerequisite for the entire assessment-and-authoring feature group.

## 2. Goals

- Extract questions from `questions_json` blobs into a first-class `questions` table with structured metadata.
- Support item pool semantics: a quiz can reference a pool of N questions and randomly draw M from it at delivery time.
- Enable tagging, searching, and filtering questions by type, concept, standard, difficulty, and author.
- Provide a question bank management UI (browse, create, edit, retire questions).
- Maintain backward compatibility with existing quizzes using the legacy JSONB format during migration.

## 3. Non-Goals

- QTI import (covered in 2.13).
- Full question versioning history (covered in 2.6).
- IRT parameter storage (covered in 1.6 — added as columns on the questions table).
- Detailed question types beyond the existing MC/TF/short-answer in v1 bank migration (new types covered in 2.2).
- Per-question access control beyond course-scoped instructor ownership.

## 4. Personas & User Stories

- **As an instructor**, I want to reuse a question I wrote for Quiz 1 in Quiz 3 without copy-pasting, so changes propagate automatically.
- **As a course author**, I want to define a pool of 50 algebra questions and have the system randomly select 10 for each student's quiz attempt, reducing cheating.
- **As an instructor**, I want to search my question bank by concept tag and filter by difficulty to assemble a balanced final exam.
- **As a platform admin**, I want to bulk-import questions from a CSV or QTI file into the shared bank.
- **As a developer**, I want a stable `question_id` foreign key that all other features (hints, misconceptions, SRS, IRT) can reference.

## 5. Functional Requirements

- **FR-1.** The system MUST create a `course.questions` table with structured fields: `id`, `course_id` (owner scope), `question_type`, `stem` (rich text), `options` (JSONB array), `correct_answer`, `explanation`, `metadata` (JSONB), `status` (`draft`/`active`/`retired`).
- **FR-2.** The system MUST support item pool quiz delivery: a quiz may reference a `question_pool_id` instead of a fixed question list; at attempt start, M questions are randomly sampled from the pool's active questions.
- **FR-3.** The system MUST provide CRUD REST endpoints for questions (`/api/v1/courses/{id}/questions`) and pools (`/api/v1/courses/{id}/question-pools`), accessible to instructors and admins.
- **FR-4.** The system MUST provide a search endpoint `GET /api/v1/courses/{id}/questions?q=&type=&concept_id=&difficulty=&status=` with full-text search on stem.
- **FR-5.** The system MUST migrate existing `questions_json` blobs to rows in `course.questions`, associating them with the originating quiz and marking them `source = 'legacy_json'`.
- **FR-6.** The system MUST maintain a `quiz_question_refs` table linking quizzes to questions (either fixed-list or pool-reference), replacing the `questions_json` field.
- **FR-7.** The system SHOULD support cross-course sharing: an admin can mark a question as `shared = true`, making it visible to all instructors in the same organisation (org-scoped in v1).
- **FR-8.** The system MAY provide a bulk import endpoint accepting a JSON array or CSV of questions.

## 6. Non-Functional Requirements

- **Performance** — Question search p95 ≤ 150 ms for banks up to 10 k questions using GIN full-text index. Pool sampling at attempt start MUST complete in ≤ 50 ms.
- **Security** — Questions belong to a course; instructors can only view/edit questions in courses they own or co-instruct. Students cannot access question bank endpoints. Shared questions are read-only to non-owners.
- **Privacy & Compliance** — Question content may contain student examples or cultural references; not PII by default. FERPA applies to linked attempt data, not question text.
- **Accessibility** — Question bank UI (browse/edit) MUST be keyboard navigable; question editor uses existing TipTap component which MUST be WCAG 2.1 AA audited (see 12.1).
- **Scalability** — Target: 100 k questions per installation, 10 k per course. GIN FTS index; paginated API (cursor-based, not offset for large banks).
- **Reliability** — Pool sampling is transactional (sampled question IDs written to `attempt_question_selections` at attempt start; same questions served on page reload). Legacy JSON fallback during migration window.
- **Observability** — Counter `question_bank.questions_created`, `question_bank.pool_samples`. Log migration progress per quiz.
- **Maintainability** — Question service in `server/src/services/question_bank.rs`; migration script in `server/migrations/0040_question_bank.sql` + a data migration Rust binary.
- **Internationalization** — Question stem and options are UTF-8 rich text; locale variants stored as separate rows with `locale` field (v2).
- **Backward compatibility** — Legacy `questions_json` column remains and is populated until the migration flag is flipped; old quiz delivery code path falls back to JSONB if `quiz_question_refs` is empty for a quiz.

## 7. Acceptance Criteria

- **AC-1.** *Given* an instructor creates a question via `POST /api/v1/courses/{id}/questions`, *when* the request succeeds, *then* the question is returned with a UUID and `status = 'draft'`.
- **AC-2.** *Given* a quiz references a pool of 50 questions with `sample_n = 10`, *when* two students start the quiz, *then* each receives a different random selection of 10 questions (sampled independently).
- **AC-3.** *Given* the legacy migration runs on an existing course with `questions_json`, *when* complete, *then* all questions exist as rows in `course.questions` and the quiz delivers the same questions as before.
- **AC-4.** *Given* a question is tagged to concept X, *when* `GET /api/v1/courses/{id}/questions?concept_id=X` is called, *then* only questions tagged to X are returned.
- **AC-5.** *Given* a student starts a quiz with a pool reference, *when* they reload the quiz mid-attempt, *then* they see the same 10 questions (pool sample is stable for the attempt).
- **AC-6.** *Given* a question is marked `status = 'retired'`, *when* a pool samples questions, *then* retired questions are excluded from the sample.

## 8. Data Model

```sql
-- Migration: server/migrations/0040_question_bank.sql

CREATE TYPE course.question_type AS ENUM (
    'mc_single', 'mc_multiple', 'true_false', 'short_answer', 'numeric',
    'matching', 'ordering', 'hotspot', 'formula', 'code', 'file_upload',
    'audio_response', 'video_response'
);

CREATE TYPE course.question_status AS ENUM ('draft', 'active', 'retired');

CREATE TABLE course.questions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    course_id       UUID NOT NULL REFERENCES course.courses(id) ON DELETE CASCADE,
    question_type   course.question_type NOT NULL DEFAULT 'mc_single',
    stem            TEXT NOT NULL,
    options         JSONB,
    correct_answer  JSONB,
    explanation     TEXT,
    points          NUMERIC(6,2) NOT NULL DEFAULT 1.0,
    status          course.question_status NOT NULL DEFAULT 'draft',
    shared          BOOLEAN NOT NULL DEFAULT FALSE,
    source          TEXT NOT NULL DEFAULT 'authored',  -- 'authored'|'ai'|'qti'|'legacy_json'
    metadata        JSONB NOT NULL DEFAULT '{}',
    irt_a           NUMERIC(6,4),
    irt_b           NUMERIC(6,4),
    irt_status      TEXT NOT NULL DEFAULT 'uncalibrated',
    created_by      UUID REFERENCES "user".users(id),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_q_course ON course.questions (course_id, status);
CREATE INDEX idx_q_stem_fts ON course.questions USING gin(to_tsvector('english', stem));
CREATE INDEX idx_q_type ON course.questions (course_id, question_type);

CREATE TABLE course.question_pools (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    course_id   UUID NOT NULL REFERENCES course.courses(id),
    name        TEXT NOT NULL,
    description TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE course.question_pool_members (
    pool_id     UUID NOT NULL REFERENCES course.question_pools(id) ON DELETE CASCADE,
    question_id UUID NOT NULL REFERENCES course.questions(id) ON DELETE CASCADE,
    PRIMARY KEY (pool_id, question_id)
);

CREATE TABLE course.quiz_question_refs (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    quiz_id     UUID NOT NULL REFERENCES course.module_quizzes(id) ON DELETE CASCADE,
    question_id UUID REFERENCES course.questions(id),
    pool_id     UUID REFERENCES course.question_pools(id),
    sample_n    INTEGER,
    position    SMALLINT,
    CHECK ((question_id IS NOT NULL) != (pool_id IS NOT NULL))
);

CREATE TABLE course.attempt_question_selections (
    attempt_id  UUID NOT NULL,
    question_id UUID NOT NULL REFERENCES course.questions(id),
    position    SMALLINT NOT NULL,
    PRIMARY KEY (attempt_id, question_id)
);
```

- Table view: stem (truncated), type badge, concept tags, difficulty (from IRT), status, last edited.

- Bulk actions: retire, move to pool, export CSV.

- Rich text stem with MathJax support (2.3).
- Option rows with correct-answer radio/checkbox.
- Concept tag picker (1.2).
- Misconception tagger per distractor (1.10).
- Hint builder (1.9).
- Save as draft / publish toggle.

**Quiz builder** (updated):

- "Add from bank" button searches question bank.
- "Add pool" button creates a random-pull pool reference.
- Pool shows as a single row with sample_n spinner.

## 14. Risks & Mitigations


| Risk                                                                           | Likelihood | Impact | Mitigation                                                                                   |
| ------------------------------------------------------------------------------ | ---------- | ------ | -------------------------------------------------------------------------------------------- |
| Legacy JSONB migration corrupts question data                                  | M          | H      | Run migration in read-only mode first, diff output, require human approval before write      |
| Pool sampling produces duplicate questions across attempts of the same student | L          | M      | Seed random sampler with (user_id, attempt_id) for deterministic but user-unique selection   |
| Question bank grows too large for FTS index                                    | L          | M      | Add Postgres FTS index partition by course_id; evaluate Meilisearch if bank exceeds 1 M rows |
| Instructors confused by draft vs. active status                                | M          | L      | Show clear status badges; prevent draft questions from appearing in student-facing quizzes   |


## 15. Rollout Plan

- **Feature flag:** `question_bank_enabled` (default: `false`).
- **Migration sequencing:** deploy schema → run legacy JSONB migration (offline, data-migration binary) → deploy question bank API → enable UI → GA.
- **Pilot cohort:** 3 courses with 50+ questions each.
- **GA criteria:** all existing quiz questions migrated; pool sampling working; search latency p95 ≤ 150 ms.
- **Rollback:** disable flag; quiz delivery falls back to legacy JSONB path; no data loss.

## 16. Test Plan

- **Unit** — Pool sampling (deterministic given seed; excludes retired; honours sample_n); legacy JSONB parser.
- **Integration** — Create question → add to pool → create quiz referencing pool → start attempt → verify 10 questions selected and stable on reload.
- **End-to-end** — Playwright: instructor opens question bank, creates MC question, tags concept, adds to pool, creates quiz from pool, student takes quiz, verifies 10 unique questions.
- **Security** — Student cannot access question bank API (403); instructor of another course cannot access this course's bank.
- **Accessibility** — axe scan on question bank page; keyboard navigation through table; question editor form labels.
- **Performance / load** — k6: 200 concurrent attempt-starts with pool sampling; p95 ≤ 50 ms.
- **Manual exploratory** — Verify retired question excluded from pool; verify legacy quiz still works after migration; verify cross-course shared question visible to other instructors.
