## 1. Problem Statement

Courses in Lextures store free-form `available_from` and `available_until` timestamps, but there is no shared "Spring 2026" term object that multiple courses can reference. This means an admin cannot quickly archive all Fall 2025 courses, a student cannot filter their dashboard by term, and a registrar cannot tie final grades to a grading period for SIS export. Every K-12 district and university models time in explicit terms (semesters, quarters, trimesters, grading periods), and this absence is a direct obstacle to institutional adoption.

## 2. Goals

- Introduce an org-scoped `terms` table representing academic years, semesters, or grading periods.
- Allow courses to be linked to a term, inheriting term-level date boundaries.
- Provide APIs for term CRUD, used by admins to set up the academic calendar.
- Surface term selector in course creation UI and student dashboard filter.
- Integrate with RFC 5545 iCalendar export (16.5) so term dates appear in calendar feeds.

## 3. Non-Goals

- Complex multi-layer grading-period nesting (e.g., marking period within a quarter within a semester) — V1 supports a single level.
- SIS-driven term sync (Banner, PowerSchool) — deferred to 14.1.
- Holiday / no-class day modeling — deferred to 14.6.
- Per-section term override — handled in 5.4.

## 4. Personas & User Stories

- **As a district registrar**, I want to create "Fall 2026" as a term so that I can assign all fall courses to it and archive them en masse at year-end.
- **As a student**, I want to filter my course dashboard by term so that I see only my current semester's courses.
- **As an instructor**, I want to select a term when creating a course so that it inherits the correct start/end dates automatically.
- **As an admin**, I want to list all courses in a given term so that I can generate enrollment reports for that period.
- **As a self-learner**, I want terms to be optional so that self-paced courses have no artificial time boundary.

## 5. Functional Requirements

- **FR-1.** The system MUST provide a `terms` table with `id`, `org_id`, `name`, `term_type` (`semester`, `quarter`, `trimester`, `year`, `grading_period`, `custom`), `start_date`, `end_date`, `status` (`upcoming`, `active`, `completed`, `archived`), `created_at`, `updated_at`.
- **FR-2.** The system MUST allow `courses.term_id` (nullable) to reference a term; when set, the course's effective availability window defaults to the term's `start_date`/`end_date` unless overridden.
- **FR-3.** The system MUST provide admin REST endpoints for term CRUD scoped to the authenticated user's org.
- **FR-4.** The system MUST allow filtering `GET /api/courses` by `term_id` as a query parameter.
- **FR-5.** The system SHOULD automatically transition term `status` from `upcoming` → `active` → `completed` via a scheduled job based on current date.
- **FR-6.** The system SHOULD expose term dates as iCalendar VEVENT entries in the course calendar feed (RFC 5545 iCalendar).
- **FR-7.** The system MAY support a `grading_periods` child table for intra-term marking periods (Q1, Q2 within a semester) for K-12 report card support (deferred to 13.4).

## 6. Non-Functional Requirements

- **Performance** — term lookup is a small table (≤100 rows per org in typical use); full-table scan acceptable; still add index on `(org_id, status)`.
- **Security** — term management is gated to `org_admin` role; students and instructors have read-only access to term metadata.
- **Privacy & Compliance** — term dates are not PII; no special handling required. FERPA implication: term scoping is used in grade-export workflows.
- **Accessibility** — term selector in UI must be a labelled `<select>` or combobox with keyboard support.
- **Scalability** — typical org has <100 terms; no special scaling concerns.
- **Reliability** — term status transitions via scheduled job must be idempotent.
- **Observability** — log `term_id` on course list requests; alert if scheduled status-transition job fails.
- **Maintainability** — term service isolated in `server/src/services/terms.rs`.
- **Internationalization** — term names are free text (UTF-8); dates stored as `DATE` (no time component) in Postgres, displayed in user's locale.
- **Backward compatibility** — `term_id` nullable; existing courses unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* an org admin, *When* `POST /api/orgs/:id/terms` is called with `{"name":"Fall 2026","term_type":"semester","start_date":"2026-08-25","end_date":"2026-12-15"}`, *Then* HTTP 201 is returned with the term object.
- **AC-2.** *Given* a course linked to "Fall 2026", *When* `GET /api/courses/:id` is called, *Then* the response includes a `term` object with `name: "Fall 2026"` and the date range.
- **AC-3.** *Given* a student, *When* `GET /api/courses?term_id=<fall-2026-id>` is called, *Then* only courses in that term are returned.
- **AC-4.** *Given* a term whose `end_date` has passed, *When* the scheduled job runs, *Then* the term's `status` transitions to `completed`.
- **AC-5.** *Given* a completed term, *When* its iCal feed is exported, *Then* the VCALENDAR output includes a VEVENT for the term's date range with correct DTSTART and DTEND per RFC 5545 iCalendar.
- **AC-6.** *Given* a self-paced course with no `term_id`, *When* it is listed in the student dashboard, *Then* it appears in an "Ongoing / Self-paced" section separate from term-grouped courses.

## 8. Data Model

```sql
-- server/migrations/075_terms.sql
CREATE TABLE terms (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id     UUID NOT NULL REFERENCES organizations(id),
  name       TEXT NOT NULL,
  term_type  TEXT NOT NULL DEFAULT 'semester'
               CHECK (term_type IN ('semester','quarter','trimester','year','grading_period','custom')),
  start_date DATE NOT NULL,
  end_date   DATE NOT NULL,
  status     TEXT NOT NULL DEFAULT 'upcoming'
               CHECK (status IN ('upcoming','active','completed','archived')),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT terms_end_after_start CHECK (end_date > start_date)
);

CREATE INDEX idx_terms_org_status ON terms(org_id, status);

ALTER TABLE courses ADD COLUMN term_id UUID REFERENCES terms(id);
CREATE INDEX idx_courses_term_id ON courses(term_id);
```

No backfill required; `term_id` nullable.

## 9. API Surface

| Verb | Path | Auth scope | Description |
|---|---|---|---|
| POST | `/api/orgs/:id/terms` | `org_admin` | Create term |
| GET | `/api/orgs/:id/terms` | authenticated | List terms for org |
| PATCH | `/api/orgs/:id/terms/:tid` | `org_admin` | Update term |
| DELETE | `/api/orgs/:id/terms/:tid` | `org_admin` | Archive term (no courses) |
| GET | `/api/courses?term_id=` | authenticated | Filter courses by term |

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Instructors set course dates outside term bounds | H | L | Warning in UI (not a hard block) |
| Scheduled status transition job fails silently | M | M | Alert on job failure; manual re-run endpoint |
| Conflicting date ranges for overlapping terms | L | L | Validation warning (not block) — some schools legitimately overlap terms |

## 15. Rollout Plan

- Feature flag: `terms_enabled` (default: `false`).
- Sequence: DB migration → API → UI → scheduled job → flag on staging → flag on production.
- Dogfood: create Fall/Spring 2026 terms in dev org.
- GA criteria: course filter by term works; status transitions verified in staging over 24h.
- Rollback: disable flag; `term_id` nullable so no data breakage.

## 16. Test Plan

- **Unit** — term date validation; status transition logic.

- **End-to-end** — Playwright: admin creates term, assigns course, student filters dashboard by term.
- **Security** — student cannot create/modify terms; org-admin cannot modify another org's terms.
- **Accessibility** — date input fields labeled; term filter keyboard accessible.
