## 1. Problem Statement

The current enrollment model supports only three course-level roles: owner, instructor, and student (with a TA flag in `server/migrations/057_enrollments_catalog_teacher_ta.sql`). Real institutions require a richer set: a Teaching Assistant who can grade but not edit course structure; a Designer who can author content but not see student grades; an Observer/Auditor who can view everything read-only (peer reviewers, deans, accreditors); a Librarian who manages course files without seeing grades. Without these roles, instructors cannot delegate work safely and accreditation reviews cannot be conducted in-platform.

## 2. Goals

- Define and implement five new course-level roles: `ta`, `designer`, `observer`, `auditor`, `librarian`.
- Map each role to a capability set in `server/src/authz.rs`.
- Provide enrollment UI flow for assigning non-student, non-instructor roles.
- Emit role-change audit events.
- Maintain backward compatibility with the existing `catalog_teacher_ta` flag.

## 3. Non-Goals

- Custom role definitions per course or org (allow-list of capabilities) — deferred.
- Org-level versions of these roles (e.g., district-wide librarian) — those map to 5.8 `org_viewer`.
- Parent/Guardian role — covered in 5.10.

## 4. Personas & User Stories

- **As an instructor**, I want to assign a PhD student as a TA so that they can grade submissions and post announcements without being able to delete modules.
- **As a curriculum developer (Designer)**, I want access to build and modify course content without seeing student grades or personal data.
- **As a dean (Observer)**, I want read-only access to a course's content, discussions, and aggregate grade data (not individual identities) for program review.
- **As a librarian**, I want to upload and manage course reserve files without touching the gradebook or student roster.
- **As an accreditor (Auditor)**, I want read-only access to all course artifacts including submissions and grade history to conduct an audit.

## 5. Functional Requirements

- **FR-1.** The system MUST define course-role permissions as follows: TA (grade, post announcements, view roster, not edit structure); Designer (edit content/structure, not view grades/roster); Observer (read everything except PII-level individual grades); Auditor (read everything including individual grades, no write); Librarian (manage course files only).
- **FR-2.** The system MUST store these roles in the existing enrollment / grants model with a `role` column value matching one of the new enum values.
- **FR-3.** The system MUST enforce each role's capabilities in `server/src/authz.rs` with unit-tested permission matrix.
- **FR-4.** The system MUST provide an enrollment API endpoint to add a user to a course with a non-student, non-instructor role.
- **FR-5.** The system MUST prevent a TA from assigning themselves or others to the instructor/owner role.
- **FR-6.** The system SHOULD migrate the existing `catalog_teacher_ta` boolean flag to the new `ta` role enum value.
- **FR-7.** The system SHOULD surface each user's course role in the enrollments list UI with a role badge.
- **FR-8.** The system MAY allow instructors to customize the capability set for the TA role only (toggle grade/announce/feedback permissions).

## 6. Non-Functional Requirements

- **Performance** — role lookup is part of the request-level authz cache; no additional DB queries per request.
- **Security** — designers must not see any student PII or grade data; authz must be tested for each role against each resource type. Observer must see only aggregate, not individual, grade breakdowns unless Auditor.
- **Privacy & Compliance** — FERPA: Observer role limited to aggregate data; Auditor role requires "legitimate educational interest" justification — admin must confirm when assigning.
- **Accessibility** — role badge in enrollments table uses `aria-label`; role assignment modal is keyboard accessible.
- **Scalability** — role permission lookup cached per request; no additional load concern.
- **Reliability** — role migration from `catalog_teacher_ta` must be idempotent.
- **Observability** — role assignments and revocations logged to audit trail.
- **Maintainability** — permissions defined as a capability matrix constant in `server/src/authz.rs`; adding a new role requires only one code location to update.
- **Internationalization** — role display names externalized.
- **Backward compatibility** — migration of `catalog_teacher_ta` flag to `ta` role; existing instructor/student enrollments unaffected.

## 7. Acceptance Criteria

- **AC-1.** *Given* a user enrolled as `designer`, *When* they call `GET /api/courses/:id/gradebook`, *Then* HTTP 403 is returned.
- **AC-2.** *Given* a user enrolled as `ta`, *When* they submit a grade via `POST /api/submissions/:id/grade`, *Then* the grade is accepted and returns HTTP 200.
- **AC-3.** *Given* a user enrolled as `observer`, *When* they call `GET /api/courses/:id/content`, *Then* content is returned; when they call `DELETE /api/modules/:id`, *Then* HTTP 403 is returned.
- **AC-4.** *Given* a user enrolled as `librarian`, *When* they upload a file via `POST /api/courses/:id/files`, *Then* the upload succeeds; when they access `GET /api/courses/:id/gradebook`, *Then* HTTP 403 is returned.
- **AC-5.** *Given* the migration runs on a DB with `catalog_teacher_ta = true` enrollments, *When* inspected after migration, *Then* those rows have `role = 'ta'`.
- **AC-6.** *Given* a TA, *When* they attempt to add another user as `instructor` via the enrollment API, *Then* HTTP 403 is returned.

## 8. Data Model

```sql
-- server/migrations/081_extended_course_roles.sql
-- Extend the role enum used in course_enrollments / user_course_grants
ALTER TABLE course_enrollments
  ADD COLUMN course_role TEXT DEFAULT 'student'
    CHECK (course_role IN ('instructor','owner','ta','designer','observer','auditor','librarian','student'));

-- Migrate existing TA flag
UPDATE course_enrollments
   SET course_role = 'ta'
 WHERE catalog_teacher_ta = true
   AND course_role = 'instructor';
```

The capability matrix is defined in `server/src/authz.rs` as a const lookup table, not in the database.

## 9. API Surface

| Verb | Path | Auth scope | Description |
|---|---|---|---|
| POST | `/api/courses/:id/enrollments` | `instructor` / `org_admin` | Add user with specified `course_role` |
| PATCH | `/api/courses/:id/enrollments/:eid` | `instructor` / `org_admin` | Change role on existing enrollment |
| GET | `/api/courses/:id/enrollments` | `instructor` / `ta` | List enrollments with roles |

Request body for POST: `{ user_id: string; course_role: "ta"|"designer"|"observer"|"auditor"|"librarian" }`

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Designer accidentally sees student data due to misconfigured permission | M | H | Automated authz matrix integration test for all role/resource combinations |
| FERPA violation: observer sees individual grades | M | H | Observer role explicitly excludes individual grade columns in gradebook query |
| Role migration breaks existing TA enrollments | M | M | Idempotent migration; test on copy of prod data |

## 15. Rollout Plan

- Feature flag: `extended_course_roles_enabled` (default: `false`).
- Sequence: DB migration (additive `course_role` column) → capability matrix → authz update → UI → TA migration → flag.
- Dogfood: assign internal user as Designer on test course.
- GA criteria: all authz matrix tests pass; TA migration verified.
- Rollback: disable flag; `course_role` column nullable falls back to existing logic.

## 16. Test Plan

- **Unit** — capability matrix: each role × each resource type returns correct allow/deny.
- **Integration** — enrollment API with new roles; migration idempotency.
- **End-to-end** — Playwright: assign TA, log in as TA, grade submission; verify Designer cannot access gradebook.
- **Security** — authz matrix: all 5 new roles × 10 resource types = 50 permission checks automated.
- **Accessibility** — role badge and assignment modal accessible.
- **Performance / load** — no additional DB queries per role check; benchmark confirms.
- **Manual exploratory** — observer attempts to see individual student grade; auditor can view full history.
