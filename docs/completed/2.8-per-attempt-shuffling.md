# 2.8 — Per-Attempt Question Shuffling & Answer Shuffling

**Status: implemented.**

## Implementation summary

- Quiz-level toggles were already present as `module_quizzes.shuffle_questions` and `shuffle_choices` (API: `shuffleQuestions` / `shuffleChoices`). At attempt start, the server now applies both when materializing `attempt_question_selections`, using `rand::rng()` for option permutations (Fisher–Yates via `shuffle_indices` in `server/src/services/quiz_attempt.rs`).
- New `course.attempt_option_orders` stores a `SMALLINT[]` permutation per question and attempt: `option_order[i]` is the **authored** choice index shown at display position `i`. Rows are written in the same transaction as selections; `correct_choice_index` on delivered `QuizQuestion` is remapped so grading stays correct.
- Learner delivery with an in-progress attempt now always resolves questions from `attempt_question_selections` whenever the bank has refs and selections exist (not only for pool quizzes). `usesServerQuestionSampling` is set true in that case so `QuizStudentTakePanel` skips client-side shuffle (stable across reloads).
- Per-question override: `course.questions.shuffle_choices_override` (`NULL` = inherit, `false` = lock authored option order). Editable on **Course → Question bank** for MC / multi-MC / true–false (`clients/web/src/pages/lms/CourseQuestionBankPage.tsx`).
- Migration: `server/migrations/082_per_attempt_shuffling.sql`.

## 1. Problem Statement

Without question and answer shuffling, students sitting near each other can trivially share answers ("Question 3, answer B") because every student sees the same question order and the same option ordering. This is one of the most basic academic integrity measures in any quiz platform. The absence of shuffling in the `delivery` settings means Lextures cannot credibly serve in-person or online exams in K-12 or HE contexts where even rudimentary cheating deterrence is expected.

## 2. Goals

- Support per-attempt question shuffling (randomise question order for each attempt).
- Support per-attempt answer-option shuffling (randomise option order for MC and multi-select questions).
- Make both toggleable per quiz and per question (some questions, like ordered lists, should not have their options shuffled).
- Persist the shuffled order server-side so it is stable on page reload.

## 3. Non-Goals

- Shuffling for question types where order is semantically meaningful (ordering/sequence questions, matching questions — option shuffle disabled for these types automatically).
- True anti-cheating exam security (covered by 2.10 lockdown mode, 14.9 proctoring).
- Group-level shuffle seeds (same shuffle for all students in a section) — individual randomisation only.

## 4. Personas & User Stories

- **As an instructor**, I want to enable "shuffle questions" so each student gets a different ordering of the same 20 questions, making copying harder.
- **As an instructor**, I want to enable "shuffle answers" so the correct option appears in a different position for each student's attempt.
- **As a student**, I want the question and option order to remain stable if I reload the page mid-attempt, so I do not lose my place.
- **As an instructor**, I want to disable answer shuffling on specific questions (e.g., questions where "All of the above" or "None of the above" is the last option and should stay last).

## 5. Functional Requirements

- **FR-1.** The system MUST add `shuffle_questions` (boolean, default false) and `shuffle_answers` (boolean, default false) to the quiz delivery configuration (`module_quizzes`).
- **FR-2.** The system MUST add `shuffle_answers` (boolean, default true when parent quiz setting is true) per question in the question bank, allowing per-question override.
- **FR-3.** The system MUST compute the shuffled order at attempt start and persist it in `attempt_question_selections` (question order column) and a new `attempt_option_orders` table (per-question option permutation).
- **FR-4.** The system MUST serve questions and options in the persisted shuffled order for every subsequent request within the same attempt.
- **FR-5.** The system MUST NOT shuffle option order for question types `ordering`, `matching`, `hotspot`, or any type where option order is semantically determined.
- **FR-6.** The system MUST store the canonical correct answer in database-independent form (by `option_id`, not by position index) so grading is unaffected by shuffling.

## 6. Non-Functional Requirements

- **Performance** — Shuffle computation at attempt start is in-memory (Fischer-Yates O(n)); no additional DB queries. Persisting option order adds at most one INSERT per question in the attempt.
- **Security** — Shuffled order is generated server-side with a cryptographically random seed (`rand::rng()` in Rust). Clients cannot observe or predict the order before attempt start.
- **Privacy & Compliance** — Shuffled order is an operational detail; standard attempt-record privacy applies (FERPA).
- **Accessibility** — Screen reader users MUST receive questions and options in a logical order even when shuffled; the visual order and DOM order MUST match the persisted shuffled order (no CSS reordering tricks).
- **Scalability** — No scaling concern; shuffle is O(n) per attempt.
- **Reliability** — Shuffle persisted atomically with attempt-start transaction; page reload always returns same order.
- **Observability** — Log shuffle settings applied per attempt at DEBUG level. No special metrics needed.
- **Maintainability** — Shuffle logic in `server/src/services/quiz_attempt.rs` as pure helpers (`shuffle_indices`, `apply_choice_display_order`).
- **Internationalization** — No i18n implications; shuffle is order-based.
- **Backward compatibility** — Default is `shuffle_questions = false, shuffle_answers = false`; existing quizzes unchanged.

## 7. Acceptance Criteria

- **AC-1.** *Given* a quiz with `shuffle_questions = true`, *when* two students start the quiz, *then* at least 90 % of the time their question orders differ (probabilistic; test with 50 attempt pairs).
- **AC-2.** *Given* a student reloads the quiz page mid-attempt, *when* the questions load, *then* the order is identical to before the reload.
- **AC-3.** *Given* a quiz with `shuffle_answers = true`, *when* two students start the quiz, *then* at least one MC question has options in different order for the two students.
- **AC-4.** *Given* a question has `shuffle_answers = false` override, *when* answers are shuffled for the quiz, *then* that specific question's options appear in authored order.
- **AC-5.** *Given* an ordering-type question is in a shuffled quiz, *when* the student sees the question, *then* the ordering-question's draggable items appear in the authored order (not shuffled).
- **AC-6.** *Given* options are shuffled and a student selects the correct answer, *when* the quiz is graded, *then* the grade is correct regardless of the option's shuffled position.

## 8. Data Model

Implemented as `server/migrations/082_per_attempt_shuffling.sql`:

- `course.questions.shuffle_choices_override` — optional per-question lock (false = authored order when quiz shuffles choices).
- `course.attempt_option_orders` — `option_order SMALLINT[]` (authored index at each display position).

Quiz-level columns were already present from `044_quiz_comprehensive_settings.sql` (`shuffle_questions`, `shuffle_choices`).

## 9. API Surface

No new endpoints. Shuffled question order is in `attempt_question_selections.position`; option order is applied when building module quiz payloads for learners with an attempt id. Question bank REST includes `shuffleChoicesOverride` on create/update/get.

## 10. UI / UX
