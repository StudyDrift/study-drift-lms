## 1. Problem Statement

Instructors reviewing student assignment submissions today have no way to highlight text, add margin comments, or draw on uploaded PDFs or documents inside the browser. This forces them to download files, annotate externally, and re-upload — a workflow that destroys turnaround time and is the #1 stated reason instructors choose Canvas/Gradescope over Lextures. Without SpeedGrader-equivalent inline annotation, Lextures cannot win a single HE or K-12 RFP where grading workflow is evaluated.

## 2. Goals

- Deliver an in-browser annotation layer on top of PDF and common document submissions (PDF, DOCX converted to PDF, images).
- Persist annotation data (highlights, freehand strokes, text comments, point annotations) server-side, linked to the submission.
- Allow instructors to navigate forward/backward through the submission queue without leaving the grader UI.
- Expose annotation data via API so third-party tools and export flows (3.11 CSV, 3.14 originality) can reference comments.
- Achieve WCAG 2.1 AA for the annotation toolbar and comment sidebar.

## 3. Non-Goals

- Real-time collaborative multi-grader annotation (deferred to 3.4 Moderated Grading).
- Annotation of video or audio submissions (deferred to 3.2).
- DOCX native annotation (DOCX is converted server-side to PDF before rendering).
- Annotation of non-document submissions (code text, links, quizzes).

## 4. Personas & User Stories

- **As an instructor**, I want to highlight a sentence in a student's PDF essay and attach a comment so that the student receives precise, contextual feedback.
- **As an instructor**, I want to press "next student" and jump immediately to the next ungraded submission without navigating away so that I can grade a full class in one session.
- **As a student**, I want to open my returned submission and see the instructor's highlights and comments alongside my score so that I understand exactly what to improve.
- **As a TA**, I want to add annotations that are visible to the instructor-of-record before grades are released so that moderated grading remains consistent.
- **As an admin**, I want annotation data retained for the same period as submissions so that grade appeals can reference original evidence.

## 5. Functional Requirements

- **FR-1.** The system MUST render uploaded PDF submissions inside the browser using PDF.js without requiring any plugin download.
- **FR-2.** The system MUST provide annotation tools: text highlight (with colour picker ≥4 colours), freehand draw, text comment box, and point/pin comment.
- **FR-3.** The system MUST persist each annotation as a record linked to `(submission_id, annotator_user_id)`, storing page number, bounding coordinates, tool type, colour, and text body.
- **FR-4.** The system MUST display existing annotations when a grader or the student reopens the submission view.
- **FR-5.** The system MUST provide a "submission navigator" (previous/next student, filter by graded/ungraded) within the grader panel.
- **FR-6.** The system MUST support exporting a submission as a flattened PDF with annotations burned in, downloadable by instructor or student.
- **FR-7.** The system SHOULD convert DOCX/PPTX/ODT to PDF server-side (via LibreOffice headless or equivalent) before rendering.
- **FR-8.** The system SHOULD render inline images (PNG, JPG, WEBP) via a simple pan/zoom viewer with the same annotation overlay.
- **FR-9.** The system MAY support rubber-stamp comments (reusable text macros) per instructor.

## 6. Non-Functional Requirements

- **Performance** — Initial PDF render (first page) MUST appear within 2 s for files ≤ 10 MB on a 10 Mbps connection. Annotation save round-trip p95 ≤ 300 ms.
- **Security** — Submission files MUST be served via pre-signed URLs (once 8.1 object storage lands) or guarded byte-range endpoints requiring `instructor` or `owner` role for the course. Annotation records scoped by `course_id` enforced at the DB query layer in `server/src/authz.rs`.
- **Privacy & Compliance** — Annotation data is FERPA-protected student education record data. Annotations MUST be included in any FERPA data export. PII in annotation text MUST be handled under the same retention policy as grades (see 10.1).
- **Accessibility** — Annotation toolbar keyboard-navigable (Tab/Enter/Escape); ARIA labels on all tool buttons; comment text inputs properly labelled; colour choices supplemented by icon/pattern for colour-blind users (WCAG 2.1 AA).
- **Scalability** — Annotation storage modelled as append-only rows; expect up to 500 annotations per submission, 10 k submissions per deployment. Index on `submission_id`.
- **Reliability** — Annotation saves MUST be idempotent (upsert by `(submission_id, annotator_user_id, client_annotation_id)`). Offline drafts buffered in IndexedDB and synced on reconnect.
- **Observability** — Emit `annotation.created`, `annotation.deleted`, `submission.opened_for_grading` structured log events. Track annotation save latency histogram in metrics.
- **Maintainability** — PDF.js integration isolated in `clients/web/src/components/annotation/` module; server annotation routes under `server/src/routes/annotations.rs`.
- **Internationalization** — All toolbar labels and comment placeholders use i18n keys; RTL text in annotation comment boxes supported.
- **Backward compatibility** — Existing submissions with no annotations render normally; migration is additive only.

## 7. Acceptance Criteria

- **AC-1.** *Given* a student has submitted a PDF assignment, *when* an instructor opens the submission in the grader panel, *then* the PDF renders in-browser with all pages scrollable within 3 s.
- **AC-2.** *Given* the grader panel is open, *when* the instructor highlights text on page 2 and types a comment, *then* the annotation appears immediately and is persisted so that a page reload shows it unchanged.
- **AC-3.** *Given* an instructor has annotated a submission, *when* the student opens the returned assignment, *then* all annotations are visible with the instructor's comment text readable.
- **AC-4.** *Given* there are 30 submissions for an assignment, *when* the instructor clicks "next student" 29 times, *then* all submissions are navigable in sequence without a full page reload between transitions.
- **AC-5.** *Given* an annotated submission, *when* the instructor clicks "Download annotated PDF," *then* a flattened PDF containing all annotations is returned within 10 s for a ≤ 5 MB source file.
- **AC-6.** *Given* a user without `instructor` or `owner` role in the course, *when* they request the annotation list API endpoint for any submission, *then* the server returns HTTP 403.
- **AC-7.** *Given* the annotation toolbar, *when* a keyboard user tabs through all controls, *then* every button is focusable and activatable without a mouse, and no focus trap occurs.

## 8. Data Model

```sql
-- Migration: server/migrations/073_submission_annotations.sql

CREATE TABLE submission_annotations (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    submission_id   UUID NOT NULL REFERENCES module_assignments(id) ON DELETE CASCADE,
    annotator_id    UUID NOT NULL REFERENCES users(id),
    client_id       TEXT NOT NULL,               -- idempotency key from client
    page            INT NOT NULL DEFAULT 1,
    tool_type       TEXT NOT NULL CHECK (tool_type IN ('highlight','draw','text','pin')),
    colour          TEXT NOT NULL DEFAULT '#FFFF00',
    coords_json     JSONB NOT NULL,              -- {x1,y1,x2,y2} or stroke points
    body            TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at      TIMESTAMPTZ,
    UNIQUE (submission_id, client_id)
);

CREATE INDEX idx_sub_annotations_submission ON submission_annotations(submission_id)
    WHERE deleted_at IS NULL;
```

Backfill: no backfill needed — additive table.

## 9. API Surface

| Verb | Path | Auth | Description |
|---|---|---|---|
| GET | `/api/courses/:cid/assignments/:aid/submissions/:sid/annotations` | instructor/owner | List annotations for a submission |
| POST | `/api/courses/:cid/assignments/:aid/submissions/:sid/annotations` | instructor/owner | Create annotation |
| PATCH | `/api/courses/:cid/assignments/:aid/submissions/:sid/annotations/:id` | annotator only | Update comment text |
| DELETE | `/api/courses/:cid/assignments/:aid/submissions/:sid/annotations/:id` | annotator or owner | Soft-delete annotation |
| GET | `/api/courses/:cid/assignments/:aid/submissions/:sid/annotated-pdf` | instructor/student-self | Download flattened PDF |

Request body (POST/PATCH):
```ts
{ clientId: string; page: number; toolType: 'highlight'|'draw'|'text'|'pin';
  colour: string; coordsJson: object; body?: string }
```

Rate limit: 60 annotation writes per minute per user.

## 10. UI / UX

New components under `clients/web/src/components/annotation/`:
- `AnnotationViewer.tsx` — PDF.js canvas layer + overlay SVG.
- `AnnotationToolbar.tsx` — floating toolbar with tool selector.
- `AnnotationCommentPanel.tsx` — right-hand sidebar listing all comments.
- `SubmissionNavigator.tsx` — prev/next/filter bar at top of grader.

Modified pages:
- `clients/web/src/pages/lms/CourseModuleAssignmentPage.tsx` — embed grader panel.

Key flows:
1. Instructor opens assignment → sees submission list → clicks "Grade" → grader panel opens.
2. Instructor selects highlight tool → drags across text → comment popover appears → types → saves.
3. Instructor clicks "Next student" → navigator loads next submission, retaining scroll position.
4. Student clicks "View feedback" → read-only annotation overlay shown.

Empty state: "No annotations yet — use the toolbar above to add feedback."
Loading state: skeleton shimmer while PDF pages render.
Error state: "Could not load submission. Retry?" with reload button.
Mobile: annotation toolbar collapses to bottom sheet; pan-only mode on small screens (<768 px).
ARIA: `role="toolbar"` on toolbar container, `aria-label` per tool button, `role="dialog"` on comment popover.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| DOCX-to-PDF fidelity varies | M | M | Use Gotenberg (LibreOffice) with acceptance tests on a corpus of sample docs |
| Large PDF files (50+ pages) slow render | M | H | Lazy-load pages; stream byte ranges; set max file size 50 MB |
| Annotation coordinate system differs across zoom levels | H | M | Normalise coords to PDF points (not pixels); recalculate on render |
| LibreOffice sidecar adds ops complexity | M | M | Use Gotenberg Docker image in docker-compose; document in IaC runbook |

## 15. Rollout Plan

- Feature flag: `ANNOTATION_ENABLED` (default: `false`).
- Phase 1: schema migration → annotation API with flag-guarded routes.
- Phase 2: PDF viewer + annotation toolbar behind flag.
- Phase 3: dogfood with 2–3 internal courses; collect latency data.
- Phase 4: enable for pilot institution; gather instructor NPS.
- GA criteria: p95 save latency < 300 ms, zero data-loss incidents in 2-week pilot, WCAG audit pass.
- Rollback: disable flag; annotation table rows are retained but UI hidden.

## 16. Test Plan

- **Unit** — annotation coordinate normalisation logic, PDF export service, authz checks in `annotations.rs`.
- **Integration** — CRUD API with real DB; verify UNIQUE constraint on `(submission_id, client_id)`; verify 403 for non-instructor.
- **End-to-end** — Playwright: upload PDF → open grader → add highlight comment → reload → verify annotation persists; student opens feedback → sees annotation.
- **Security** — Authz matrix: student cannot POST/DELETE on other students' submissions; TA can annotate but not delete instructor annotations.
- **Accessibility** — axe-core in Playwright; manual VoiceOver + NVDA run on toolbar and comment panel.
- **Performance / load** — k6 script: 50 concurrent instructors annotating simultaneously; pass criterion p95 < 300 ms save latency.
- **Manual exploratory** — QA checklist: all tool types, colour picker, undo, PDF with Arabic/CJK text, 100-page PDF.
