## 1. Problem Statement

Adaptive feedback in Lextures is entirely post-attempt: a student submits a quiz, receives a score, and possibly a short AI-generated explanation. There is no progressive hint system during an attempt, and no worked examples are accessible mid-question. Research on cognitive apprenticeship and scaffolding (Wood, Bruner & Ross 1976; Sweller's Cognitive Load Theory) shows that graduated hints and worked examples during problem-solving are significantly more effective than feedback given only at the end. Without hint scaffolding, struggling learners either guess or abandon the question entirely, which degrades both engagement and mastery signal quality.

## 2. Goals

- Implement a multi-level hint system per question: up to 5 hint levels, each revealing progressively more guidance.
- Allow course authors to write static hints and/or configure AI-generated adaptive hints.
- Track hint usage (which hints were accessed, at what attempt number) to feed mastery quality signals back to 1.1.
- Provide worked-example overlays for STEM-style questions (step-by-step solution, accessible after N failed attempts or after submission).
- Respect "lockdown" mode (2.10) — hints disabled in proctored exams.

## 3. Non-Goals

- Full AI tutoring conversation during assessment (covered in 6.9 AI tutor).
- Misconception-targeted hints (covered in 1.10 — hints in this plan are concept-level, not error-specific).
- Hint generation for code-execution questions (covered in 2.4, complex enough to be separate).
- Hint usage tracked for grading penalties (configurable, but penalty logic is in grading rules, not this plan).

## 4. Personas & User Stories

- **As a K-12 student**, I want to request a hint when I'm stuck on a math problem so I can make progress without being told the answer outright.
- **As a self-learner**, I want to see a worked example after I've tried and failed a question twice so I can understand the method before retrying.
- **As a course author**, I want to write up to 3 hints per question that progressively narrow in on the answer, and optionally link a worked-example video.
- **As an instructor**, I want to see how often students use hints per question so I can identify which questions need better scaffolding or clearer writing.
- **As a platform developer**, I want hint requests and responses to be server-side so hint content cannot be extracted from the client before a student requests it.

## 5. Functional Requirements

- **FR-1.** The system MUST support attaching up to 5 ordered hint levels to any question in the question bank, each with a `body` (rich text / markdown) and optional `media_url`.
- **FR-2.** The system MUST expose a `POST /api/v1/quiz-attempts/{id}/hints` endpoint that returns the next unrevealed hint for a given question within the active attempt, one at a time.
- **FR-3.** The system MUST record each hint request in a `hint_requests` log (attempt_id, question_id, hint_level, requested_at) for analytics.
- **FR-4.** The system MUST NOT return hint bodies in the initial question delivery payload — hints are fetched on-demand only, so their content cannot be scraped from the DOM before use.
- **FR-5.** The system MUST support AI-generated hints as a fallback when no static hints are authored: on hint request, call the AI service to generate a concept-level nudge (not the answer) using the question stem and concept tags.
- **FR-6.** The system MUST support worked examples attached to questions: a structured document (`worked_example_body`, `steps: [{number, explanation, expression}]`) shown in an overlay.
- **FR-7.** The system MUST respect the quiz delivery mode: if `hints_disabled: true` is set on the quiz (lockdown mode, 2.10), the hint endpoint MUST return HTTP 403.
- **FR-8.** The system SHOULD allow course authors to configure a penalty per hint level (e.g., -5 % per hint) that is subtracted from the question score at grading time.
- **FR-9.** The system SHOULD integrate hint usage count into the mastery update in 1.1: a correct answer with 3 hints used contributes less mastery credit than one with 0 hints.

## 6. Non-Functional Requirements

- **Performance** — Static hint fetch p95 ≤ 50 ms. AI-generated hint p95 ≤ 2 000 ms (acceptable for on-demand hint, not in quiz flow critical path). Worked example fetch p95 ≤ 100 ms.
- **Security** — Hint content is only revealed on authenticated request within an active attempt. The next hint level cannot be fetched before the previous one is revealed. Instructor-only access to hint analytics.
- **Privacy & Compliance** — Hint request logs contain (user_id, question_id) — educational record (FERPA). AI hint generation MUST redact student-name and PII before sending to LLM (see 10.17).
- **Accessibility** — Hint overlay MUST trap focus and be dismissible via Escape. Worked-example steps MUST be ordered list elements for screen readers. Math in hints rendered via MathJax/KaTeX (2.3).
- **Scalability** — Hint content is small (< 1 KB per level); cacheable at CDN. AI hint calls are async; do not block quiz state machine.
- **Reliability** — If AI hint service is unavailable, fall back to a generic "Try re-reading the question stem and listing what you know" response; never return an error to the student mid-quiz.
- **Observability** — Prometheus counter `hints.requested` (labeled by type: static/ai), `hints.ai_fallback`. Log AI hint calls at INFO. Alert if AI hint error rate > 5 %.
- **Maintainability** — Hint logic in `server/src/services/hint_service.rs`. Route in `server/src/routes/quiz_attempts.rs`. No hint content in route handlers.
- **Internationalization** — Static hint bodies stored as rich text; locale-specific variants stored as separate hint rows with `locale` field. AI hints generated in the learner's locale.
- **Backward compatibility** — Hint tables are additive. Questions without hints work identically to today.

## 7. Acceptance Criteria

- **AC-1.** *Given* a question has 3 static hints, *when* a student calls the hint endpoint for the first time, *then* only hint level 1 body is returned (not levels 2 or 3).
- **AC-2.** *Given* a student has revealed hint level 2, *when* they call the hint endpoint again, *then* hint level 3 is returned; a subsequent call returns `{ no_more_hints: true }`.
- **AC-3.** *Given* a quiz has `hints_disabled: true`, *when* the hint endpoint is called, *then* the response is HTTP 403.
- **AC-4.** *Given* a question has no static hints authored, *when* a student requests a hint, *then* an AI-generated nudge is returned within 3 seconds and no answer is revealed.
- **AC-5.** *Given* a student uses 2 hints and a hint penalty of 5 % per hint is configured, *when* the quiz is graded, *then* the question score is reduced by 10 %.
- **AC-6.** *Given* an instructor views hint analytics for a question, *when* the report loads, *then* they see hint request counts per level, and the percentage of students who used each level.

## 8. Data Model

```sql
-- Migration: server/migrations/0038_hints.sql

CREATE TABLE course.question_hints (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    question_id UUID NOT NULL,   -- FK to questions table (2.1)
    level       SMALLINT NOT NULL CHECK (level BETWEEN 1 AND 5),
    body        TEXT NOT NULL,
    media_url   TEXT,
    locale      TEXT NOT NULL DEFAULT 'en',
    penalty_pct NUMERIC(4,1) NOT NULL DEFAULT 0.0,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (question_id, level, locale)
);

CREATE TABLE course.question_worked_examples (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    question_id UUID NOT NULL,
    title       TEXT,
    body        TEXT,
    steps       JSONB NOT NULL DEFAULT '[]',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (question_id)
);

CREATE TABLE course.hint_requests (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    attempt_id  UUID NOT NULL,
    question_id UUID NOT NULL,
    hint_level  SMALLINT NOT NULL,
    hint_type   TEXT NOT NULL DEFAULT 'static',  -- 'static' | 'ai'
    requested_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (requested_at);
```

## 9. API Surface

```
POST /api/v1/quiz-attempts/{attempt_id}/questions/{question_id}/hint
  Auth: own user (student with active attempt)
  Response 200: { level: number, body: string, media_url?: string, no_more_hints?: bool }
  Response 403: { error: "hints_disabled" }

GET  /api/v1/quiz-attempts/{attempt_id}/questions/{question_id}/worked-example
  Auth: own user (only after N failed attempts OR after attempt submission)
  Response 200: { title, body, steps: [{number, explanation, expression}] }

GET  /api/v1/questions/{question_id}/hints          Auth: instructor|admin  (author view)
POST /api/v1/questions/{question_id}/hints          Auth: instructor|admin
PUT  /api/v1/questions/{question_id}/hints/{id}     Auth: instructor|admin
DELETE /api/v1/questions/{question_id}/hints/{id}   Auth: instructor|admin

GET  /api/v1/questions/{question_id}/hint-analytics Auth: instructor|admin
  Response: { levels: [{level, request_count, pct_users}] }
```

## 10. UI / UX

**In-quiz hint button** (in `clients/web/src/components/quiz/`):
- "Get a hint" button below each question (hidden when `hints_disabled`).
- On click: hint panel slides in below question stem, showing hint 1.
- "Another hint" button appears if more levels exist.
- Hint penalty notice shown before first reveal: "Using this hint will reduce your score by 5 %."
- ARIA: `role="dialog"`, `aria-modal="false"`, `aria-live="polite"` for hint text.

**Worked example overlay**:
- "Show worked example" link (appears after 2 incorrect attempts or post-submission).
- Modal dialog with step-by-step accordion (each step expandable).
- Math rendered via KaTeX (2.3).

**Author hint editor** (question bank editor, 2.1):
- "Add hint" button adds a hint level row with rich text editor.
- Drag to reorder hint levels.
- Penalty field per level.
- Worked example tab with step builder.

Mobile: hint panel renders below question, full width, with a close button; worked example is a full-screen bottom sheet.

## 11. AI / ML Considerations

AI-generated hints call an LLM endpoint (via the existing OpenRouter abstraction) with a prompt constructed from:
1. Question stem (stripped of identifying student data).
2. Concept tags (from 1.2) as context.
3. System instruction: "Provide a pedagogically appropriate hint that guides the student without revealing the answer. Hint level {N} of 5: be {more/less} specific."

PII redaction (10.17) MUST strip student name and any personally identifying submission content before the prompt is sent. Cost budget: ≤ $0.005 per AI hint request; use a fast, low-cost model (e.g., GPT-4o-mini class).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| AI hints reveal the answer despite prompt constraints | M | H | Add post-generation answer-leakage check; fall back to generic hint on detection |
| Instructors don't author hints, leaving all hints AI-generated | H | M | Auto-generate 3 static hints at question creation time using AI; instructor can edit |
| Hint penalty discourages help-seeking | M | M | Default penalty to 0; make it opt-in; show research rationale in UI |
| Hint content cached at CDN leaks between users | L | H | Hints fetched per-attempt, not statically cached; include attempt_id in cache key |

## 15. Rollout Plan

- **Feature flag:** `hint_scaffolding_enabled` (default: `false`).
- **Migration sequencing:** deploy migration → ship static hint authoring → ship student hint UI → enable AI hint fallback → GA.
- **Pilot cohort:** 2 K-12 math courses; instructors author hints for 20 questions.
- **GA criteria:** hint request rate > 5 % of quiz questions attempted (indicates discoverability); AI hint error rate < 5 %.
- **Rollback:** disable flag; hint endpoint returns 404; existing quiz behaviour unchanged.

## 16. Test Plan

- **Unit** — Hint-level sequential enforcement; AI prompt builder (PII redaction); penalty calculation.
- **Integration** — Create hint levels → student requests hints in order → verify sequential enforcement; verify penalty applied at grading.
- **End-to-end** — Playwright: student takes quiz, clicks "Get a hint," sees hint 1, clicks again, sees hint 2, verifies hint 3 available; verify worked example shown post-submission.
- **Security** — Unauthenticated request to hint endpoint returns 401; hints from another student's attempt are inaccessible.
- **Accessibility** — axe scan on hint panel; Escape key closes overlay; worked example steps read as ordered list by screen reader.
- **Performance / load** — 500 concurrent hint requests (static) p95 ≤ 50 ms; AI hint generation p99 ≤ 3 000 ms.
- **Manual exploratory** — Verify `hints_disabled` blocks hint button in UI and returns 403 from API; verify AI hint does not contain the answer for a known question.

## 18. Open Questions

1. Should hint levels be globally sequential (level 1 always first) or should authors be able to configure branching hints based on wrong-answer content (requires 1.10)?
2. Should worked examples be available before submission, or only after N failed attempts or post-submission?
3. Is there a maximum number of hint requests per question, per student, per attempt — or unlimited?
4. Should AI hint generation happen synchronously (blocking the student for up to 2 s) or should there be a loading state with streaming?

## 19. Implementation status (shipped)

Core behaviour is implemented with these practical deltas from the plan sketch above:

- **Course flag** `hint_scaffolding_enabled` on `course.courses` (default off). Learner hint and worked-example endpoints return **404** when the feature is disabled for the course.
- **Schema** — `course.question_hints`, `course.question_worked_examples`, and `course.hint_requests` ship in migration `095_hints_scaffolding.sql`; `hint_requests` is a normal table (not range-partitioned).
- **Routes** — Learner hint and worked-example live on the existing module-quiz attempt paths in `server/src/routes/courses.rs` (not a separate `quiz_attempts` router). Instructor hint CRUD, worked-example upsert, and hint analytics live under the course-scoped question bank routes in `server/src/routes/question_bank.rs`.
- **Service** — `server/src/services/hint_service.rs`: ordered static hints, AI fallback, answer-leak heuristics, generic fallback, worked-example unlock (after submission or after two hint requests on the question), per-level grading penalties summed at submit time, and mastery dampening from hint use counts.
- **Web** — Course features toggle; in-quiz hint and worked-example UI in `quiz-student-take-panel.tsx` for bank questions identified by UUID (non-bank ids do not call hint APIs).
- **Follow-ups** — Prometheus metrics from NFR §6 were not added (only structured logging on the AI path). Rich question-bank editor UI for hints/worked examples can layer on the existing APIs.
