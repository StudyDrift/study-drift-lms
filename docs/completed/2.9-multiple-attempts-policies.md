## 1. Problem Statement

The quiz model has no policy for how multiple attempts are resolved into a single gradebook score. Some instructors want the highest score (incentivises retakes as learning), some want the latest attempt (penalises guessing strategies), some want the average (measures consistency), and some want the first attempt (simulates exam conditions). This is described as THIN in `MISSING_FEATURES.md` — the schema may record multiple attempts but there is no policy engine to select which one counts. Without this, instructors who allow retakes have no control over the grading outcome, and the gradebook cannot reliably aggregate scores for courses with retake policies.

## 2. Goals

- Add a `retake_policy` field to quiz configuration with four modes: `highest`, `latest`, `average`, `first`.
- Add a `max_attempts` field (integer, null = unlimited) to control how many attempts are allowed.
- Implement the policy engine in the grading service so the gradebook always reflects the policy-selected score.
- Allow per-student attempt-count overrides as part of accommodations (2.11).
- Surface the applicable policy and current attempt count in the student quiz UI.

## 3. Non-Goals

- Complex adaptive retry policies (e.g., "unlock new questions on retry") — covered in 1.4 and 1.9.
- Prerequisite-gated retakes (e.g., "must review lesson before retry") — a future enhancement.
- Per-question retake tracking (attempt policy applies at the whole-quiz level).

## 4. Personas & User Stories

- **As an instructor**, I want to set "highest score counts" so students are incentivised to retake quizzes until they master the material.
- **As an HE instructor**, I want "first attempt counts" on my midterm, with a maximum of 1 attempt, to prevent repeated guessing.
- **As a self-learner**, I want "average score" on practice quizzes so I cannot just get lucky on one attempt.
- **As a student**, I want to see "Attempt 2 of 3 allowed" in the quiz UI so I know how many retakes I have left.
- **As an instructor with a struggling student**, I want to grant an extra attempt override without changing the limit for the whole class.

## 5. Functional Requirements

- **FR-1.** The system MUST add `max_attempts` (integer, nullable = unlimited) and `retake_policy` (`highest` | `latest` | `average` | `first`) to the quiz configuration, with sensible defaults (`max_attempts = 1, policy = 'latest'`).
- **FR-2.** The system MUST enforce `max_attempts` at attempt-start time: if the student has reached the limit, the API MUST return HTTP 403 with `"max_attempts_reached"`.
- **FR-3.** The system MUST apply the retake policy when computing the gradebook score: `highest` = max score across all attempts; `latest` = most recent submitted attempt score; `average` = mean of all submitted scores; `first` = score of first submitted attempt.
- **FR-4.** The system MUST expose `GET /api/v1/quiz-attempts?quiz_id=&user_id=` returning all attempts for a (quiz, student) pair with scores, so instructors can see the full attempt history.
- **FR-5.** The system MUST update the gradebook entry to the policy-selected score after each new attempt is submitted.
- **FR-6.** The system MUST show in the student quiz view: current attempt number, max attempts allowed (or "Unlimited"), and which attempt counts ("Highest score counts").
- **FR-7.** The system SHOULD allow per-enrollment attempt-count overrides (extra attempt) written to `enrollment_quiz_overrides` table, checked before enforcing `max_attempts`.

## 6. Non-Functional Requirements

- **Performance** — Policy computation (selecting from N attempt scores) MUST complete in ≤ 10 ms. Gradebook update after attempt submission MUST be atomic with the attempt submission transaction.
- **Security** — `max_attempts` enforcement is server-side; client cannot start a new attempt by calling the API directly if the limit is reached. Override is instructor/admin-only.
- **Privacy & Compliance** — All attempt scores are FERPA-protected educational records. Gradebook score shown to student reflects the policy, not all raw attempt scores (student sees "Your score: 85 %" not all attempt scores by default, unless instructor enables).
- **Accessibility** — Attempt counter in quiz UI MUST be an accessible text element, not just a graphic. Policy label MUST be readable by screen readers.
- **Scalability** — Policy computation is O(N) where N = attempt count per student; N ≤ max_attempts (bounded). No scaling concern.
- **Reliability** — Gradebook update is part of the grading transaction; if grading fails, gradebook is not updated (consistent).
- **Observability** — Counter `quiz_attempts.blocked_max_attempts`. Log policy applied per gradebook update.
- **Maintainability** — Policy selection logic in `server/src/services/quiz_grading.rs` as a `RetakePolicy` enum. No policy logic in route handlers.
- **Internationalization** — Policy names use i18n keys. Attempt counter "Attempt X of Y" is a formatted i18n string.
- **Backward compatibility** — Existing quizzes default to `max_attempts = null, retake_policy = 'latest'` via migration backfill, preserving existing behaviour.

## 7. Acceptance Criteria

- **AC-1.** *Given* a quiz with `max_attempts = 2` and a student has already submitted 2 attempts, *when* they try to start a third, *then* the API returns HTTP 403 with `"max_attempts_reached"`.
- **AC-2.** *Given* `retake_policy = 'highest'` and a student scores 60 % on attempt 1 and 80 % on attempt 2, *when* the gradebook score is computed, *then* it shows 80 %.
- **AC-3.** *Given* `retake_policy = 'average'` and scores of 60 % and 80 %, *when* the gradebook score is computed, *then* it shows 70 %.
- **AC-4.** *Given* `retake_policy = 'first'` and scores of 60 % and 80 %, *when* the gradebook score is computed, *then* it shows 60 %.
- **AC-5.** *Given* a student with a per-enrollment extra-attempt override of +1, *when* they have used the base `max_attempts` attempts, *then* one more attempt is permitted.
- **AC-6.** *Given* the student UI for a quiz with `max_attempts = 3`, *when* the student is on attempt 2, *then* they see "Attempt 2 of 3" and "Highest score counts" (or whichever policy is set).

## 8. Data Model

```sql
-- Migration: server/migrations/0047_retake_policy.sql

CREATE TYPE course.retake_policy AS ENUM ('highest', 'latest', 'average', 'first');

ALTER TABLE course.module_quizzes
    ADD COLUMN max_attempts   INTEGER CHECK (max_attempts > 0),   -- NULL = unlimited
    ADD COLUMN retake_policy  course.retake_policy NOT NULL DEFAULT 'latest';

CREATE TABLE course.enrollment_quiz_overrides (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    enrollment_id       UUID NOT NULL REFERENCES course.enrollments(id) ON DELETE CASCADE,
    quiz_id             UUID NOT NULL REFERENCES course.module_quizzes(id) ON DELETE CASCADE,
    extra_attempts      INTEGER NOT NULL DEFAULT 0,
    time_multiplier     NUMERIC(4,2),   -- for accommodations (2.11)
    created_by          UUID NOT NULL REFERENCES "user".users(id),
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (enrollment_id, quiz_id)
);
```

## 9. API Surface

```

  Body addition: { max_attempts?: number, retake_policy: string }

-- Attempt history for instructor:
GET /api/v1/quizzes/{qid}/attempts?user_id=   Auth: instructor|admin
  Response: { attempts: AttemptSummary[], policy_score: number }

-- Override for individual student:
POST /api/v1/enrollments/{id}/quiz-overrides  Auth: instructor|admin
  Body: { quiz_id, extra_attempts, time_multiplier? }
DELETE /api/v1/enrollments/{id}/quiz-overrides/{qid}

-- Student attempt info (existing attempt response extended):
GET /api/v1/quiz-attempts/{id}
  Response addition: { attempt_number, max_attempts, remaining_attempts, retake_policy }
```

## 10. UI / UX

**Instructor gradebook**:
- Hover on a score cell → tooltip shows all attempt scores and which one counts.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Average policy counts abandoned (0 %) attempts | M | M | Exclude unsubmitted attempts from average; only submitted attempts count |
| Instructor changes policy after students have taken attempts | M | M | Warn instructor in UI "Changing this policy will retroactively affect existing gradebook scores"; log the change |
| Extra-attempt override race condition (instructor grants while student is starting) | L | L | Attempt-start transaction checks override atomically |

## 15. Rollout Plan

- **Feature flag:** not needed (defaults preserve existing behaviour).
- **Migration sequencing:** add columns with defaults → backfill existing quizzes with `max_attempts = null, retake_policy = 'latest'` → deploy.
- **Pilot cohort:** any instructor with retake quizzes.
- **GA criteria:** all four policies compute correct gradebook score; `max_attempts` enforcement verified.
- **Rollback:** revert `retake_policy` column to nullable; treat null as 'latest'; no behaviour change.

## 16. Test Plan

- **Unit** — All four policy computations; `max_attempts` enforcement with override.
- **Integration** — Submit 2 attempts with different scores → verify each policy returns correct gradebook score; attempt limit → 403 on third attempt.
- **End-to-end** — Playwright: instructor sets highest-score policy, student takes twice, verify gradebook shows higher score.
- **Security** — Student cannot modify max_attempts or retake_policy; override only writable by instructor.
- **Accessibility** — Attempt counter text readable by screen reader; policy label announced.
- **Performance** — Policy computation for 10 attempts: ≤ 10 ms.
- **Manual exploratory** — Verify average excludes abandoned attempts; verify policy change warning displayed to instructor.
