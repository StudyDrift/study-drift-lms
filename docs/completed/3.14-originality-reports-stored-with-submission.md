## 1. Problem Statement

Even when originality detection (3.5) runs, the detection metadata (similarity score, AI probability, provider report URL, snapshot of the flagged submission text) is not persistently linked to the submission record for long-term access. If a student later disputes a finding months after the course ends, or an academic integrity committee requires documentary evidence, the report should be retrievable alongside the submission rather than depending on an active third-party API session. This plan specifies the storage and retrieval layer for originality reports, building on the detection infrastructure defined in 3.5.

## 2. Goals

- Persist the full originality report response (similarity percentage, AI probability, provider JSON, report embed token) in object storage and link to the submission.
- Allow instructors and academic integrity officers to retrieve any historical originality report by submission ID, even after the detection provider's link expires.
- Store a text snapshot of the submission at the time of detection (immutable) for evidentiary purposes.
- Include originality report data in FERPA DSAR exports.
- Support retention policy alignment with submission file retention.

## 3. Non-Goals

- Re-running detection on archived submissions (on-demand re-check is a separate action).
- Building an academic integrity case-management module.
- Storing raw HTML of provider reports (embed token + structured JSON sufficient).

## 4. Personas & User Stories

- **As an academic integrity officer**, I want to open a submission from two semesters ago and see the originality report that was generated at the time, even if the Turnitin report link has expired.
- **As an instructor**, I want to access the originality report stored with a submission any time I open the grader panel, not just immediately after detection runs.
- **As a student**, I want to see my originality report whenever I view my submission feedback, so that I understand how my work was evaluated.
- **As a compliance officer**, I want to include originality report data in a FERPA data export for a student so that the export is complete.

## 5. Functional Requirements

- **FR-1.** The system MUST store the complete provider API response JSON for each detection run in object storage, keyed by `(submission_id, provider, detection_run_id)`.
- **FR-2.** The system MUST capture a plain-text snapshot of the submitted content at the time of detection and store it immutably in object storage.
- **FR-3.** The `originality_reports` table (created in 3.5) MUST include a `report_storage_key` column pointing to the stored JSON in object storage.
- **FR-4.** The system MUST provide an API endpoint that retrieves a stored report JSON (or a rendered summary) by `submission_id`, accessible to instructors indefinitely and to students per the course visibility policy.
- **FR-5.** When a Turnitin/Copyleaks embed token expires, the system MUST fall back to showing the stored similarity percentage and AI probability from the stored JSON rather than an error.
- **FR-6.** Originality report records MUST be included in FERPA DSAR exports when the report is associated with the requesting student.
- **FR-7.** The system MUST apply the same retention schedule as submission files to stored report JSONs; deletion is coordinated with submission deletion.

## 6. Non-Functional Requirements

- **Performance** — Retrieving a stored report from object storage p95 ≤ 500 ms (pre-signed URL cached 15 min). Detection provider JSON stored asynchronously after job completion; no impact on submission path.
- **Security** — Object storage keys for report JSONs are non-guessable (UUID-based); access gated behind API auth. Text snapshots contain student PII — stored under the same access controls as submission files.
- **Privacy & Compliance** — Text snapshots are FERPA education records. GDPR right-to-erasure does not apply to records retained for academic integrity legal obligations; document retention basis in privacy policy. Reports deleted on submission deletion cascade.
- **Accessibility** — Stored report summary UI (fallback view) meets WCAG 2.1 AA.
- **Scalability** — Object storage scales linearly; average report JSON < 50 KB; text snapshot < 100 KB. No performance concern at expected volume.
- **Reliability** — Storage write occurs in the background job after detection completes; on write failure, job retried (17.3). Detection result still stored in `originality_reports` table even if object storage write fails.
- **Observability** — Log `originality_report.stored` with `submission_id`, `provider`, `storage_key`. Alert on storage write failure rate.
- **Maintainability** — Storage logic in `server/src/services/originality/storage.rs`; called from the detection job completion handler.
- **Internationalization** — Fallback summary view uses i18n-keyed labels.
- **Backward compatibility** — Existing `originality_reports` rows (3.5) extended with `report_storage_key` column; NULL for records before this migration.

## 7. Acceptance Criteria

- **AC-1.** *Given* a detection job completes for a submission, *when* the job finishes, *then* the provider JSON and text snapshot are stored in object storage and `report_storage_key` is set on the `originality_reports` row.
- **AC-2.** *Given* a stored report exists, *when* the instructor opens the grader panel 6 months later, *then* the similarity score and AI probability are shown correctly even if the provider embed URL has expired.
- **AC-3.** *Given* a stored report and its embed token has expired, *when* the instructor clicks "View full report," *then* a fallback summary view shows the stored score, timestamp, and provider name.
- **AC-4.** *Given* a FERPA DSAR export is run for student S, *when* the export includes submission data, *then* originality report metadata (provider, similarity_pct, ai_probability, report date) is included for each submission where a report exists.
- **AC-5.** *Given* a submission is deleted, *when* the deletion cascade runs, *then* the associated `originality_reports` rows are deleted and the object storage keys are enqueued for cleanup.
- **AC-6.** *Given* the object storage write fails after detection, *when* the job retries, *then* the detection data in the DB is not overwritten; the storage write is retried independently.

## 8. Data Model

```sql
-- Migration: server/migrations/085_originality_report_storage.sql
-- Extends the originality_reports table created in 3.5 (migration 077)

ALTER TABLE originality_reports
    ADD COLUMN report_storage_key TEXT,    -- object storage key for provider JSON
    ADD COLUMN snapshot_storage_key TEXT;  -- object storage key for text snapshot
```

No new tables. Object storage bucket: `originality-reports/` prefix.
Retention tag applied to objects matching submission retention policy.

## 9. API Surface

Extended (from 3.5):
- `GET /api/courses/:cid/assignments/:aid/submissions/:sid/originality` — now includes `report_storage_key` (not returned to client; resolved server-side to pre-signed URL or summary).
- `GET /api/courses/:cid/assignments/:aid/submissions/:sid/originality/summary` — returns stored JSON summary (similarity_pct, ai_probability, provider, timestamp) from object storage fallback.

DSAR export endpoint extended:
- `GET /api/admin/users/:uid/dsar-export` — includes originality report summaries in submission data.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Object storage costs grow unexpectedly | L | M | Reports are small (< 100 KB each); minimal cost; lifecycle rule to archive after 2 years |
| Text snapshot captures PII inappropriately | M | H | Apply same access controls as submission; include in DSAR scope documentation |
| Provider JSON schema changes between versions | M | M | Store raw JSON; parse on read; version-tag the stored JSON |

- **End-to-end** — (Combined with 3.5 E2E): grader panel shows stored summary after embed token simulated-expired.
- **Security** — Object storage key not exposed to client; access via pre-signed URL with expiry.

## 18. Open Questions

1. Retention period for stored report JSONs: should it match submission file retention (typically term + 7 years in HE) or a shorter period?
2. Should text snapshots be stored in encrypted form separate from the normal object storage bucket?
3. Should we also store a rendered HTML snapshot of the submission at submission time for non-text file types?
