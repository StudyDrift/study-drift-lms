## 1. Problem Statement

The current JWT implementation in `server/src/jwt.rs` issues 72-hour tokens with no refresh mechanism and no server-side revocation. When a user changes their password, is deprovisioned via SCIM, or is terminated by an admin, their existing 72-hour JWT remains valid until expiry — creating a significant security window. There is no way to log out of all devices, revoke a stolen token, or even implement "remember me" vs. "session-only" semantics. This is a foundational security gap that must be fixed before building any of the higher-order auth features (SSO, MFA, session management).

## 2. Goals

- Replace the single long-lived JWT with a short-lived access token (15 min) and a long-lived opaque refresh token.
- Implement server-side refresh token storage with revocation support.
- Provide a `/auth/refresh` endpoint to exchange a valid refresh token for a new access token.
- Implement logout that revokes the refresh token (client-side and server-side).
- Allow admins to revoke all refresh tokens for a specific user (used by SCIM deprovisioning, password change, account suspension).

## 3. Non-Goals

- Sliding expiry or absolute session timeout configuration per user (follow-on to 4.9).
- Refresh token rotation with grace period (can be added; start with simple revoke-on-use).
- JWT signing key rotation (separate operational concern; document as runbook).

## 4. Personas & User Stories

- **As a security administrator**, I want to terminate all active sessions for a compromised account with a single admin action so that the threat is contained immediately.
- **As a user**, I want my "stay logged in" session to last 30 days without re-entering credentials so that routine visits to Lextures are seamless.
- **As a user on a shared computer**, I want to log out and know that my session cannot be resumed from that browser so that my data is protected.
- **As a student**, I want to be automatically re-authenticated when my access token expires without seeing a login prompt during active use so that my workflow is uninterrupted.

## 5. Functional Requirements

- **FR-1.** The system MUST issue two tokens on successful authentication: a short-lived JWT access token (15 min TTL) and an opaque refresh token (30-day TTL by default).
- **FR-2.** The system MUST provide `POST /auth/refresh` accepting a refresh token and returning a new access token (and optionally a new refresh token with rotation).
- **FR-3.** The system MUST store refresh tokens server-side as hashed values (SHA-256), keyed by user_id with `expires_at` and `revoked_at`.
- **FR-4.** The system MUST revoke a refresh token on: explicit logout, password change, admin revoke-all action.
- **FR-5.** The system MUST provide `POST /auth/logout` that revokes the refresh token and clears the client-side access token.
- **FR-6.** The system MUST provide an admin endpoint to revoke all active refresh tokens for a user (used by SCIM deprovisioning at 4.5).
- **FR-7.** The system SHOULD implement refresh token rotation: issue a new refresh token on each use and revoke the old one.
- **FR-8.** The system MUST reject access tokens presented after the user's `token_invalidated_at` timestamp (covers password-change revocation without full token reissue).

## 6. Non-Functional Requirements

- **Performance** — `/auth/refresh` p95 ≤ 150 ms (DB lookup + JWT sign). Refresh token lookup by hash uses indexed column.
- **Security** — Refresh tokens: 256-bit cryptographic random; stored as SHA-256 hash. Access tokens: short-lived (15 min) signed JWT; `jti` claim for future revocation. `token_invalidated_at` field on `users` table: any JWT issued before this timestamp is rejected. HTTPS-only; refresh token transmitted in `HttpOnly`, `Secure`, `SameSite=Strict` cookie (alternative: response body + client storage with XSS mitigations).
- **Privacy & Compliance** — Refresh token metadata (user_id, device hint, IP at creation) is session data. GDPR: sessions deletable on right-to-erasure (delete all refresh tokens for user). FERPA: session records not education records per se, but session hijacking could expose education records.
- **Accessibility** — Automatic token refresh is transparent to users; no UI element required. WCAG 2.1 AA for any related settings page.
- **Scalability** — Refresh token table indexed by `token_hash` and `user_id`. Expired rows cleaned by scheduled job. Expect ≤ 5 active refresh tokens per user.
- **Reliability** — If `/auth/refresh` fails (DB unavailable), client must re-authenticate; access token expiry is the fallback.
- **Observability** — Metrics: `auth.refresh_success_total`, `auth.refresh_failure_total` (categorised by reason: expired/revoked/not_found). Alert on spike in refresh failures.
- **Maintainability** — Refresh token CRUD in `server/src/services/auth/refresh_tokens.rs`; JWT issuance updated in `server/src/jwt.rs`. HTTP auth middleware updated in `server/src/http_auth.rs` to check `token_invalidated_at`.
- **Internationalization** — No user-facing strings needed for token mechanics.
- **Backward compatibility** — Existing 72-h JWTs continue to work until their expiry during transition; new logins issue short-lived tokens. `token_invalidated_at` defaults to NULL (no existing tokens revoked on migration).

## 7. Acceptance Criteria

- **AC-1.** *Given* a successful login, *when* the response is received, *then* it contains both a short-lived `access_token` (exp = 15 min) and a `refresh_token`.
- **AC-2.** *Given* a valid refresh token, *when* `POST /auth/refresh` is called, *then* a new access token is returned.
- **AC-3.** *Given* a user calls `POST /auth/logout`, *when* the logout completes, *then* subsequent calls to `/auth/refresh` with the old refresh token return HTTP 401.
- **AC-4.** *Given* an admin revokes all sessions for user U, *when* U's next API call uses their existing access token after it expires, *then* `/auth/refresh` returns HTTP 401 and they must log in again.
- **AC-5.** *Given* a user changes their password, *when* the password change is saved, *then* `token_invalidated_at` is set to NOW() and any JWT issued before that time is rejected on next request.
- **AC-6.** *Given* a refresh token used once (with rotation enabled), *when* the same refresh token is presented again, *then* HTTP 401 is returned (rotation: old token revoked on first use).

## 8. Data Model

```sql
-- Migration: server/migrations/093_refresh_tokens.sql

CREATE TABLE refresh_tokens (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash      TEXT NOT NULL UNIQUE,   -- SHA-256(plaintext), hex
    expires_at      TIMESTAMPTZ NOT NULL,
    revoked_at      TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    user_agent      TEXT,
    ip_address      INET
);

CREATE INDEX idx_refresh_token_hash ON refresh_tokens(token_hash)
    WHERE revoked_at IS NULL;

ALTER TABLE users
    ADD COLUMN token_invalidated_at TIMESTAMPTZ;
```

JWT access token claims updated to include `iat` (issued-at); middleware checks `iat < users.token_invalidated_at`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| 15-min access token too short for slow API consumers | M | M | Clients that cache access tokens get < 15 min; document expiry; client-side refresh before calls |
| Refresh token rotation causes race conditions on concurrent requests | M | M | Grace window (30 s): accept prior token within grace period after rotation |
| `token_invalidated_at` check adds latency on every API call | M | L | Cache user record with short TTL (30 s) in Redis; or store invalidation in JWT claim on re-issue |

## 15. Rollout Plan

- No feature flag — this is a foundational fix.
- Phase 1: add `refresh_tokens` table + `/auth/refresh` + `/auth/logout`.
- Phase 2: update `http_auth.rs` to check `token_invalidated_at`.
- Phase 3: shorten existing JWT TTL from 72h to 15 min.
- Phase 3 requires client to implement transparent refresh; coordinate with frontend.
- GA criteria: all auth integration tests updated for short-lived tokens; existing E2E tests pass.

## 16. Test Plan

- **Unit** — Token hash storage; `token_invalidated_at` check; refresh endpoint logic; rotation.
- **Integration** — Login → refresh → logout → refresh-rejected. Admin revoke → refresh-rejected. Password change → old JWT rejected.
- **End-to-end** — Playwright: access token expires → auto-refresh → user stays logged in. Logout → refresh → redirect to login.
- **Security** — Replay of revoked refresh token. Race condition with rotation grace period. Brute force of refresh token (256-bit entropy — infeasible, but test rate limiting).
- **Performance** — k6: `/auth/refresh` endpoint, 500 concurrent requests, p95 ≤ 150 ms.
