## 1. Problem Statement

Users have no visibility into their active login sessions. A student who notices suspicious activity on their account, or an instructor who left a device at a conference, cannot see or terminate sessions from their account settings. This is a basic security hygiene feature expected by most modern web applications and is often listed in institutional security policies. The `refresh_tokens` table introduced by 4.8 provides the complete data model; this feature is purely the UI + targeted revocation API on top of it.

## 2. Goals

- Display a list of active sessions (refresh tokens) to the user in their account settings, showing device/browser, approximate location (IP-derived), last active time, and authentication method.
- Allow users to terminate individual sessions or all other sessions ("sign out everywhere else").
- Show MFA method used per session.
- Show sessions created by SCIM provisioning or SSO separately from password-based sessions.
- Surface the "current session" prominently so users can distinguish it.

## 3. Non-Goals

- Admin-level session surveillance across all users (admin revoke is via 4.5/4.8 admin endpoint).
- Real-time push notification when a new session is created (can be added later via 6.3).
- IP geolocation mapping with map visualization — text location (city/country from MaxMind or ip-api) is sufficient.

## 4. Personas & User Stories

- **As a student**, I want to see all devices where I am logged into Lextures so that I can tell if someone else has access to my account.
- **As an instructor**, I want to click "Sign out everywhere else" so that when I change my password after a suspected breach, I know all old sessions are terminated.
- **As a student on a shared school computer**, I want to sign out of only that specific device session from my phone so that my account is secure without affecting my other devices.
- **As a security-conscious HE user**, I want to see that I used MFA to authenticate on each session so that I can verify my account is properly secured.

## 5. Functional Requirements

- **FR-1.** The system MUST list all active (non-revoked, non-expired) refresh tokens for the authenticated user, including: `created_at`, `last_used_at` (from refresh calls), `user_agent` (parsed to device/browser label), approximate location (IP geolocation), auth method (password/SAML/OIDC/magic-link/passkey/TOTP), and whether it is the current session.
- **FR-2.** The system MUST allow users to revoke individual sessions by `refresh_token.id`.
- **FR-3.** The system MUST provide "Sign out all other sessions" — revokes all refresh tokens for the user except the current one.
- **FR-4.** The system MUST identify the "current session" by matching the refresh token ID from the current request context.
- **FR-5.** The session list MUST update in near-real-time (≤ 30 s lag) to reflect new logins.
- **FR-6.** The system SHOULD display a human-readable device label (e.g., "Chrome on macOS" or "Safari on iPhone") parsed from the `user_agent` string.
- **FR-7.** The system SHOULD display approximate location as "City, Country" derived from the session's IP address using a local geolocation database.

## 6. Non-Functional Requirements

- **Performance** — Session list endpoint returns all active sessions ≤ 200 ms (indexed by `user_id`). Revoke action ≤ 100 ms.
- **Security** — Users can only see and revoke their own sessions; admin revoke covered by 4.8. Current session cannot be revoked via the list (force logout requires password re-confirmation).
- **Privacy & Compliance** — IP addresses stored in `refresh_tokens.ip_address` are PII. Geolocation used only for display; not sold or shared. IP addresses included in GDPR DSAR exports. MaxMind DB used locally (no third-party API call per session list view).
- **Accessibility** — Session list is a table with `scope="row"` headers; revoke button per row; "Sign out all other sessions" button with confirmation dialog. WCAG 2.1 AA.
- **Scalability** — Expect ≤ 10 active sessions per user; minimal DB load.
- **Reliability** — Revocation uses the same path as 4.8 logout; idempotent.
- **Observability** — Log `session.revoked_by_user` with `session_id`, `user_id`. Metric: `session_management.revocations_total`.
- **Maintainability** — Session list endpoint in `server/src/routes/me.rs`; reuses `refresh_tokens` table from 4.8.
- **Internationalization** — Device labels, location strings, and date formats i18n/l10n-aware.
- **Backward compatibility** — `user_agent` and `ip_address` columns added in 4.8 migration; sessions before 4.8 show "Unknown device."

## 7. Acceptance Criteria

- **AC-1.** *Given* a user logged in on 3 devices, *when* they open Account Settings → Sessions, *then* they see 3 sessions with device labels, locations, and one marked "Current session."
- **AC-2.** *Given* a user clicks "Sign out" on a specific session, *when* confirmed, *then* that session's refresh token is revoked and the session disappears from the list.
- **AC-3.** *Given* a user clicks "Sign out everywhere else," *when* confirmed, *then* all sessions except the current one are revoked; subsequent API calls from those devices return 401.
- **AC-4.** *Given* a session created via TOTP MFA, *when* viewed in the session list, *then* it shows "Authenticated with TOTP" as the auth method.
- **AC-5.** *Given* a user attempts to revoke their current session via the session list API, *when* the request is made, *then* the server returns HTTP 422 "Cannot revoke the current session from this UI."

## 8. Data Model

Extends `refresh_tokens` table from 4.8 migration:
```sql
-- Migration: server/migrations/094_refresh_token_session_metadata.sql
-- (Adds to the table created in 093_refresh_tokens.sql)

ALTER TABLE refresh_tokens
    ADD COLUMN last_refreshed_at TIMESTAMPTZ,
    ADD COLUMN auth_method TEXT,    -- 'password','saml','oidc','magic_link','passkey','totp'
    ADD COLUMN location_city TEXT,
    ADD COLUMN location_country TEXT;
```

IP geolocation resolved once at session creation using MaxMind GeoLite2-City DB (offline lookup).

## 9. API Surface

| Verb | Path | Auth | Description |
|---|---|---|---|
| GET | `/api/me/sessions` | authenticated | List active sessions |
| DELETE | `/api/me/sessions/:id` | authenticated (owns session) | Revoke specific session |
| DELETE | `/api/me/sessions` | authenticated | Revoke all sessions except current |

`GET /api/me/sessions` response:
```ts
{ sessions: Array<{
    id: string; createdAt: string; lastUsedAt: string;
    deviceLabel: string; location: string;
    authMethod: string; isCurrent: boolean;
}> }
```

## 10. UI / UX

New section in account settings: `clients/web/src/pages/lms/AccountSettingsPage.tsx` or a dedicated `clients/web/src/pages/Sessions.tsx`.
- Table: "Device," "Location," "Signed in," "Last active," "Method," "Action."
- Current session row highlighted (badge: "This device").
- "Sign out" button per non-current row.
- "Sign out everywhere else" button at the top.
- Confirmation dialog before bulk revocation.

ARIA: table `aria-label="Active sessions"`. Revoke button per row `aria-label="Sign out session on {device}"`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| GeoIP accuracy is low (VPN/TOR users) | H | L | Display "Location approximate" disclaimer; this is for orientation, not enforcement |
| User accidentally revokes current session | M | M | UI prevents revoking current session directly; "sign out everywhere else" preserves current |

## 15. Rollout Plan

- Feature flag: `SESSION_MANAGEMENT_UI_ENABLED` (default: `false`).
- Requires 4.8 to be in production first.
- Phase 1: list + per-session revoke.
- Phase 2: geolocation + device label parsing.
- GA criteria: AC-1 through AC-5 pass; axe-core clean.

## 16. Test Plan

- **Unit** — User-agent parsing; geolocation lookup; "current session" detection.
- **Integration** — Session list scoped to current user only; revoke updates DB and rejects next refresh.
- **End-to-end** — Playwright: log in on two sessions → open settings → revoke one → verify other still active.
- **Security** — User cannot revoke another user's session via ID guessing (UUID + authz check).
- **Accessibility** — axe-core on sessions table; keyboard navigation; confirmation dialog focus trap.
