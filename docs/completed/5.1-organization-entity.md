## 1. Problem Statement

Lextures has no concept of an owning organization above a course. Every installation is effectively a single, flat tenant, which means two universities sharing one deployment would see each other's data and there is no boundary for a district to manage its own schools, users, and branding. Without an `organizations` table as the root tenant entity, no institutional sales deal can close: every K-12 district and university IT department asks "where is our data isolated?" before signing. Introducing a first-class organization entity unlocks the entire multi-tenancy model and is the prerequisite for all downstream org-hierarchy, role, and branding work.

## 2. Goals

- Introduce a durable `organizations` top-level table that acts as the root tenant boundary in the database.
- Scope all existing resources (users, courses, files, settings) to an organization via foreign-key.
- Provide CRUD admin APIs so super-admins can provision, update, and soft-delete organizations.
- Ensure all authorization checks in `server/src/authz.rs` enforce the org boundary.
- Ship a zero-downtime migration that places all existing data in a default "seed" organization.

## 3. Non-Goals

- Sub-account nesting (schools, departments) — covered in 5.2.
- Per-tenant custom domains and branding — covered in 5.7.
- SAML/OIDC per-tenant IdP federation — covered in 4.1/4.2.
- Billing or seat management — out of scope for this plan.
- Cross-org content sharing or federation — not addressed here.

## 4. Personas & User Stories

- **As a SaaS super-admin**, I want to provision a new organization with a name and slug so that a district can start using Lextures under its own isolated tenant.
- **As a district IT director**, I want all my users, courses, and data scoped to my organization so that I can be assured other tenants cannot read our records.
- **As an instructor**, I want to belong to one (or more) organizations so that I see only the courses and people relevant to my institution.
- **As a developer**, I want every DB query to include an `org_id` predicate so that row-level isolation is impossible to forget.
- **As a compliance officer**, I want a per-org data-residency flag so that we can document where tenant data lives for GDPR/FERPA purposes.

## 5. Functional Requirements

- **FR-1.** The system MUST have an `organizations` table with at minimum: `id` (UUID), `slug` (unique), `name`, `status` (active/suspended/deleted), `created_at`, `updated_at`.
- **FR-2.** The system MUST associate every `users` row, every `courses` row, and every resource derived from them (enrollments, submissions, grades) with an `org_id` foreign key.
- **FR-3.** The system MUST enforce in `server/src/authz.rs` that a user can only access resources whose `org_id` matches the org the user is a member of (or a super-admin override).
- **FR-4.** The system MUST provide a super-admin REST API (`POST /api/admin/orgs`, `GET /api/admin/orgs`, `PATCH /api/admin/orgs/:id`, `DELETE /api/admin/orgs/:id`) gated by the `global_admin` role.
- **FR-5.** The system MUST execute a database migration that creates the `organizations` table, inserts a default seed org, and backfills `org_id` on all existing rows without downtime.
- **FR-6.** The system SHOULD support a `max_users` and `max_courses` quota field per organization for future seat-management.
- **FR-7.** The system SHOULD emit an audit log event for every org create, update, suspend, or delete action.
- **FR-8.** The system MAY store a `metadata` JSONB column on `organizations` for extensible per-tenant key/value configuration.

## 6. Non-Functional Requirements

- **Performance** — org-scoped queries MUST add ≤2 ms p95 overhead vs. unscoped equivalents; all `org_id` columns MUST be covered by a B-tree index.
- **Security** — no cross-tenant data leak is acceptable; authz middleware must be tested with an explicit cross-tenant attack scenario. Super-admin endpoints must require the `global_admin` role (see `server/src/routes/rbac.rs`).
- **Privacy & Compliance** — org isolation is the primary FERPA boundary; a per-org `data_region` enum field enables GDPR residency documentation.
- **Accessibility** — the super-admin org management UI must meet WCAG 2.1 AA.
- **Scalability** — the schema must support ≥10,000 organizations without table partitioning at MVP; evaluate Postgres row-level security (RLS) as a future hardening layer.
- **Reliability** — the backfill migration must be idempotent and safe to re-run; application must start successfully even when migration is mid-flight on a read replica.
- **Observability** — add `org_id` and `org_slug` as structured log fields on every request in `server/src/main.rs` tracing span.
- **Maintainability** — `org_id` must be threaded through all existing route handlers before the feature flag is enabled; lint rule or test matrix to catch missing org scope.
- **Internationalization** — org `name` field stored in UTF-8; no locale constraints.
- **Backward compatibility** — single-org deployments continue to work with zero config change; the default seed org acts as a transparent wrapper.

## 7. Acceptance Criteria

- **AC-1.** *Given* a super-admin auth token, *When* `POST /api/admin/orgs` is called with `{"name":"Riverdale USD","slug":"riverdale-usd"}`, *Then* the response is HTTP 201 with a body containing a UUID `id` and the provided fields.
- **AC-2.** *Given* two organizations Org-A and Org-B, and a user belonging to Org-A, *When* that user calls `GET /api/courses` authenticated, *Then* only courses with `org_id = Org-A` are returned and no Org-B courses appear.
- **AC-3.** *Given* the migration `073_organizations.sql` has been applied to a database with existing rows, *When* the schema is inspected, *Then* every `users` and `courses` row has a non-null `org_id` matching the seed org.
- **AC-4.** *Given* a cross-tenant exploit attempt (JWT from Org-A, requesting course in Org-B), *When* the auth middleware evaluates the request, *Then* HTTP 403 is returned and no Org-B data is exposed.
- **AC-5.** *Given* a global admin suspends an org, *When* any user from that org attempts to authenticate, *Then* the system returns HTTP 403 with error code `org_suspended`.
- **AC-6.** *Given* a new org is provisioned, *When* the audit log is queried, *Then* a record with `action=org_created`, `actor_id`, and `org_id` is present.

## 8. Data Model

New tables/columns:

```sql
-- server/migrations/073_organizations.sql
CREATE TABLE organizations (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slug        TEXT NOT NULL UNIQUE,
  name        TEXT NOT NULL,
  status      TEXT NOT NULL DEFAULT 'active'
                CHECK (status IN ('active','suspended','deleted')),
  max_users   INTEGER,
  max_courses INTEGER,
  data_region TEXT NOT NULL DEFAULT 'us-east-1',
  metadata    JSONB NOT NULL DEFAULT '{}',
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

ALTER TABLE users   ADD COLUMN org_id UUID REFERENCES organizations(id);
ALTER TABLE courses ADD COLUMN org_id UUID REFERENCES organizations(id);
-- Repeat for enrollments, course_structure_items, module_assignments, etc.

CREATE INDEX idx_users_org_id   ON users(org_id);
CREATE INDEX idx_courses_org_id ON courses(org_id);
```

Backfill strategy:
1. Insert seed org with well-known UUID.
2. `UPDATE users SET org_id = '<seed_uuid>' WHERE org_id IS NULL;`
3. Same for courses and all child tables.
4. Add `NOT NULL` constraint in a subsequent migration after backfill is verified.

Migration file convention: `server/migrations/NNN_*.sql`.

## 9. API Surface

| Verb | Path | Auth scope | Description |
|---|---|---|---|
| POST | `/api/admin/orgs` | `global_admin` | Create org |
| GET | `/api/admin/orgs` | `global_admin` | List orgs (paginated) |
| GET | `/api/admin/orgs/:id` | `global_admin` | Get org detail |
| PATCH | `/api/admin/orgs/:id` | `global_admin` | Update name/status/quota |
| DELETE | `/api/admin/orgs/:id` | `global_admin` | Soft-delete (sets status=deleted) |

Request body for POST/PATCH:
```typescript
{ name: string; slug?: string; max_users?: number; max_courses?: number; data_region?: string; metadata?: Record<string,unknown> }
```

Response:
```typescript
{ id: string; slug: string; name: string; status: string; max_users: number|null; data_region: string; created_at: string }
```

Rate limit: 60 writes/minute per admin token. All routes must be documented in OpenAPI spec at `server/src/routes/mod.rs`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Backfill migration causes downtime on large DB | M | H | Run backfill in batches of 10k rows; use `NOT NULL` as a deferred constraint |
| Developer forgets `org_id` filter in new routes | H | H | Add integration test asserting cross-tenant leak returns 403; CI gate |
| Slug collision across orgs | L | M | DB unique constraint + `LOWER(slug)` normalization |
| Super-admin endpoint exposed to non-admins | M | H | Role check in middleware + automated authz matrix test |

## 15. Rollout Plan

- Feature flag: `multi_tenancy_enabled` (default: `false`).
- Sequence: schema migration → backfill → deploy code behind flag → enable on staging → verify → enable on production.
- Dogfood: single internal org created in staging to test isolation.
- GA criteria: all cross-tenant tests pass; zero regressions in existing E2E suite.
- Rollback: set `multi_tenancy_enabled=false`; data model is additive (nullable columns until NOT NULL applied).

## 16. Test Plan

- **Unit** — `authz.rs` org-boundary guard; org CRUD service functions.
- **Integration** — cross-tenant API scenarios (expect 403); seed-org backfill idempotency.
- **End-to-end** — Playwright: admin creates org, invites user, user logs in, can only see org-scoped courses.
- **Security** — authz matrix: global_admin can manage all orgs; org_admin can see only own org; student cannot call admin endpoints.
- **Accessibility** — axe scan on Organizations admin page.
- **Performance / load** — k6 test: 1,000 concurrent users across 50 orgs; verify ≤2 ms overhead.
- **Manual exploratory** — attempt to read Org-B data as Org-A user via direct API calls.

## 18. Open Questions

1. Should orgs support a hierarchy (parent org → child org) now, or defer to 5.2 (sub-accounts)?
2. Should `org_id` be embedded in the JWT or looked up per-request from the DB for security revocability?
3. What is the data-deletion policy when an org is soft-deleted — hard purge after N days or retain indefinitely?
4. Do we implement Postgres Row-Level Security as a defense-in-depth layer, or rely solely on application-level checks?
