## 1. Problem Statement

Self-learners enrolling in a course without an instructor have no mechanism to determine where they should start. There is no "where should this learner begin" flow. Learners who enrol in a calculus course may already know everything in the first three modules or may lack the algebra prerequisites entirely. Without a placement assessment, learners either waste time or encounter frustrating difficulty spikes. K-12 deployments similarly require beginning-of-year diagnostic assessments to place students in the appropriate instructional track. This gap is a primary abandonment driver for the self-learner market.

## 2. Goals

- Implement a diagnostic assessment type: a short adaptive quiz (CAT mode from 1.6) that estimates a learner's ability profile across the prerequisite concept space of a course.
- On completion, route the learner to the appropriate starting module via the adaptive path engine (1.4).
- Allow instructors and course authors to configure diagnostic coverage (which concepts, how many items, stopping rule).
- Persist diagnostic results as the initial seeding of the learner model (1.1) so all downstream adaptive features benefit immediately.
- Surface a placement summary to the learner showing what they know and what they will learn.

## 3. Non-Goals

- Formal computer-adaptive diagnostic for high-stakes credentialing (not a licensure exam; formative only).
- Automated course generation from diagnostic results (covered in 1.8).
- Prerequisite gap analysis spanning multiple courses (multi-course learning path, v2).
- Standards-based diagnostic reporting to districts (depends on 3.7 and 9.5).

## 4. Personas & User Stories

- **As a self-learner**, I want to take a short placement quiz when I enrol in a new course so the system shows me where to start instead of making me guess.
- **As an instructor**, I want to configure a diagnostic at the course or unit level specifying which concepts it should probe, so students are placed at the right lesson.
- **As a K-12 student**, I want to take a beginning-of-year diagnostic that reveals my strengths and gaps so my teacher can personalise my learning plan.
- **As an instructor**, I want to see each student's diagnostic score profile after the assessment period so I can plan differentiated instruction.
- **As a course author**, I want the diagnostic to automatically use questions from the question bank (2.1) tagged to the relevant concepts, without having to build a separate quiz.

## 5. Functional Requirements

- **FR-1.** The system MUST support a `diagnostic` quiz type in the quiz delivery model, distinct from `formative` and `summative`.
- **FR-2.** The system MUST allow course authors to configure a diagnostic with: target concept list (from 1.2), maximum item count (default 20), CAT stopping rule (SE ≤ 0.3 or max items reached), and placement threshold per concept (mastery ≥ X → skip module Y).
- **FR-3.** The system MUST integrate CAT item selection (from 1.6) as the default item selection mode for diagnostic assessments, falling back to concept-balanced random selection when IRT parameters are not yet calibrated.
- **FR-4.** The system MUST write diagnostic results to `learner_concept_states` (1.1) upon completion, seeding mastery values from the θ estimates.
- **FR-5.** The system MUST trigger the adaptive path engine (1.4) after diagnostic completion to compute and present the recommended starting module.
- **FR-6.** The system MUST present a placement summary page to the learner after the diagnostic: concepts assessed, proficiency level per concept (Beginner / Developing / Proficient / Advanced), and recommended starting point.
- **FR-7.** The system SHOULD allow learners to bypass the diagnostic ("skip and start from the beginning") with one click.
- **FR-8.** The system SHOULD allow re-taking the diagnostic at instructor discretion (configurable: once per enrollment / once per term / always).
- **FR-9.** The system MAY align diagnostic concept coverage to specific standards (from 1.3) for K-12 beginning-of-year placement reporting.

## 6. Non-Functional Requirements

- **Performance** — Diagnostic quiz next-item selection MUST respond in p95 ≤ 100 ms. Placement summary page MUST render in ≤ 500 ms after submission.
- **Security** — Diagnostic attempts are per-enrollment; learners cannot access each other's results. Placement configuration is instructor/admin-only.
- **Privacy & Compliance** — Diagnostic results are educational record (FERPA). Parent/guardian access governed by 5.10 parent portal plans. Diagnostic data included in DSAR exports and erasure. COPPA: under-13 diagnostic results must not be used for any purpose other than in-product placement.
- **Accessibility** — Diagnostic UI inherits all accessibility requirements from the quiz-taking UI. Placement summary MUST use color-independent indicators (not color alone for proficiency levels). WCAG 2.1 AA required.
- **Scalability** — Diagnostic quizzes are short (≤ 30 items); no special scaling concerns beyond general quiz infrastructure.
- **Reliability** — If CAT item selection fails mid-session, the diagnostic MUST continue with random selection rather than blocking the learner. Partial diagnostic results can still seed the learner model.
- **Observability** — Log diagnostic completion events with concept coverage, θ estimates, and placement decision. Prometheus counter `diagnostic.completed`, `diagnostic.bypassed`.
- **Maintainability** — Diagnostic orchestration in `server/src/services/diagnostic.rs`; reuses `adaptive_quiz_ai.rs` and `irt.rs` services. No duplicate quiz logic.
- **Internationalization** — Proficiency level labels (Beginner / Developing / Proficient / Advanced) are i18n keys. Standards descriptions use locale from 1.3.
- **Backward compatibility** — Existing quiz types (`formative`, `summative`) are unaffected. Diagnostic type is additive.

## 7. Acceptance Criteria

- **AC-1.** *Given* a course has a diagnostic configured for concepts A, B, C, *when* a new learner enrolls, *then* the diagnostic is presented automatically before the first module (or after learner dismisses the skip prompt).
- **AC-2.** *Given* a learner completes the diagnostic with high proficiency on concepts A and B but low on C, *when* the placement summary loads, *then* A and B show "Proficient" and C shows "Developing", and the recommended start is the module teaching C.
- **AC-3.** *Given* a learner clicks "Skip diagnostic", *when* they proceed, *then* they are routed to the first module in linear order and no mastery values are seeded.
- **AC-4.** *Given* IRT parameters are not calibrated for any items in the concept pool, *when* the diagnostic runs, *then* items are selected using concept-balanced random selection and a warning is logged.
- **AC-5.** *Given* a K-12 instructor reviews diagnostic results, *when* they open the class diagnostic report, *then* they see a per-student, per-concept proficiency grid.
- **AC-6.** *Given* the diagnostic is configured for single attempt only, *when* a learner attempts to retake it, *then* the UI shows their previous results and blocks a new attempt.

## 8. Data Model

```sql
-- Migration: server/migrations/0036_diagnostic.sql

CREATE TYPE course.diagnostic_stopping_rule AS ENUM ('max_items', 'se_threshold', 'both');

CREATE TABLE course.course_diagnostics (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    course_id        UUID NOT NULL REFERENCES course.courses(id) ON DELETE CASCADE,
    concept_ids      UUID[] NOT NULL,
    max_items        INTEGER NOT NULL DEFAULT 20,
    stopping_rule    course.diagnostic_stopping_rule NOT NULL DEFAULT 'both',
    se_threshold     NUMERIC(4,3) NOT NULL DEFAULT 0.3,
    retake_policy    TEXT NOT NULL DEFAULT 'once',   -- 'once' | 'per_term' | 'always'
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (course_id)
);

CREATE TABLE course.diagnostic_attempts (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    diagnostic_id   UUID NOT NULL REFERENCES course.course_diagnostics(id),
    enrollment_id   UUID NOT NULL REFERENCES course.enrollments(id),
    started_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at    TIMESTAMPTZ,
    bypassed        BOOLEAN NOT NULL DEFAULT FALSE,
    placement_item_id UUID REFERENCES course.course_structure_items(id),
    theta_summary   JSONB,   -- {concept_id: theta} at completion
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (diagnostic_id, enrollment_id)   -- one attempt per enrollment (overridable)
);
```

## 10. UI / UX

**Diagnostic entry screen** (first screen after enrollment, new page `/courses/{id}/diagnostic`):
- Explains what the diagnostic is and how long it takes (estimated via `max_items * avg_seconds_per_item`).
- Two CTAs: "Start placement assessment" and "Skip — start from the beginning."
- Accessible: skip link for screen-reader users; no auto-advance without user action.

**Diagnostic quiz UI** — inherits from `CourseModuleQuizPage.tsx` with:
- Progress bar showing items completed / estimated total (adapts dynamically as CAT stops early).
- No back navigation (diagnostic integrity).
- Timer optional (inherits from 2.7 time limits feature).

**Placement summary page**:
- Concept proficiency cards (4-level icon + label, not color-only).
- "Your recommended starting point: Module 3 — Solving Equations."
- "Continue to Module 3" CTA.

## 11. AI / ML Considerations

CAT item selection from 1.6 is the primary algorithm. The θ-to-proficiency-level mapping (Beginner/Developing/Proficient/Advanced) is a configurable cut-score table (default cut points: θ < -1.0 = Beginner, -1.0–0 = Developing, 0–1.0 = Proficient, > 1.0 = Advanced). Course authors can override cut scores.

No external LLM calls in diagnostic delivery. Future: LLM-generated personalised feedback after diagnostic ("Based on your results, here's a study plan").

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Insufficient calibrated items for CAT on new courses | H | M | Fallback to random selection; log warning; show "calibration in progress" to instructor |
| Learners find diagnostic anxiety-inducing | M | M | Frame as "placement" not "test"; show no pass/fail messaging; make skip easy |
| Diagnostic placement is wrong (over/under-placed) | M | M | Allow learner to manually change starting module; collect feedback event |
| Retake policy circumvented via new enrollment | L | L | Tie retake policy to user_id + course_id, not enrollment_id |

## 15. Rollout Plan

- **Feature flag:** `diagnostic_assessments_enabled` (default: `false`, per-course toggle).
- **Migration sequencing:** deploy migration → add diagnostic configuration UI → deploy diagnostic delivery flow → enable for pilot.
- **Pilot cohort:** two self-learner courses with 50 + enrollments each.
- **GA criteria:** 100 diagnostic completions; placement routing correct (verified by manual spot-check with pilot users); p95 ≤ 100 ms on next-item selection.
- **Rollback:** disable flag; enrollments see no diagnostic; existing learner state unaffected.

## 16. Test Plan

- **Unit** — θ-to-proficiency-level cut-score mapping; CAT stopping rule (SE < threshold); bypass flow.
- **Integration** — Enroll user in diagnostic-enabled course → start diagnostic → respond to 10 items → assert mastery seeded in `learner_concept_states`.
- **End-to-end** — Playwright: new enrollment lands on diagnostic page, completes 5 items, views placement summary, clicks "Continue to Module 3."
- **Security** — Learner cannot start another learner's diagnostic; instructor can view class results.
- **Accessibility** — Proficiency level icons have text alternatives; no color-only indicators; keyboard navigation through diagnostic quiz identical to standard quiz.
- **Performance / load** — 200 concurrent diagnostic sessions; next-item p95 ≤ 100 ms.
- **Manual exploratory** — Verify bypass flow seeds no mastery; verify retake block; verify CAT early stopping when SE < threshold.

## 18. Open Questions

1. Should a course be allowed to have multiple diagnostics (e.g., one per unit) or only one at the course level?
2. What is the default set of concepts for a diagnostic when the instructor has not configured it explicitly — all concepts in the course, or only prerequisites?
3. Should diagnostic results be visible to parents/guardians (K-12 parent portal, 5.10)?
4. How do we handle a diagnostic where the question bank has fewer items than `max_items` — reduce max, or show an error to the author?
