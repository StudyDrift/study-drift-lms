## 1. Problem Statement

Content authors and instructional designers who maintain large question banks need CLI tools to automate question import, export, and organization — especially when migrating from legacy LMS platforms using QTI packages. The web UI is unsuitable for batch operations on thousands of questions. This story covers `questions list`, `questions create`, and `questions import`.

## 2. Goals

- `lextures questions list` lists questions in a bank with filtering.
- `lextures questions create` creates a single question from a JSON or YAML spec.
- `lextures questions import --file <qti-path>` imports a QTI question package.
- All commands support `--json`.

## 3. Non-Goals

- Question editing (use `create` to create a new version).
- Quiz assembly from the question bank (web UI only for v1).
- QTI export (follow-on).
- AI-generated question authoring (separate AI story).

## 4. Personas & User Stories

- **As a content designer migrating from Blackboard**, I want `lextures questions import --file export.zip` to bulk-import our question banks without manual re-entry.
- **As an instructor building a custom question bank**, I want `lextures questions create --bank mybank --type multiple-choice --content ./q1.json` to add questions programmatically from a script.
- **As an admin auditing question banks**, I want `lextures questions list --bank <id> --json` to enumerate all questions for compliance review.

## 5. Functional Requirements

- **FR-1.** `lextures questions list` MUST return questions; supports `--bank <id>`, `--type <type>`, `--limit`, `--page`.
- **FR-2.** `lextures questions create` MUST accept `--bank <id>` (required), `--type <type>` (required: `multiple-choice` | `true-false` | `short-answer` | `essay`), `--content <json-string|@file>` (required).
- **FR-3.** `lextures questions import --file <path>` MUST accept `.zip` QTI packages; MUST print import progress and a summary of created/skipped/failed question counts on completion.
- **FR-4.** `--content @filename` syntax MUST read content from a file (following curl convention).
- **FR-5.** All commands MUST support `--json`.

## 6. Non-Functional Requirements

- **Performance** — QTI import of 500-question packages MUST complete in < 2 minutes; progress reported.
- **Reliability** — Import MUST be transactional per question (failure on one question MUST not block others); summary MUST report per-question errors.
- **Security** — Uploaded QTI files MUST be validated for zip-bomb and path-traversal before extraction server-side.

## 7. Acceptance Criteria

- **AC-1.** *Given* `lextures questions list --bank <id> --json`, *When* it runs, *Then* stdout is a valid JSON array.
- **AC-2.** *Given* `--content @q1.json` where `q1.json` is a valid question spec, *When* `questions create` runs, *Then* the question ID is printed.
- **AC-3.** *Given* a valid QTI 2.1 zip file, *When* `questions import --file qti.zip` runs, *Then* a summary `"Imported 47/50 questions (3 skipped)"` is printed.
- **AC-4.** *Given* a malformed QTI file, *When* `questions import` runs, *Then* the CLI prints a parse error per malformed question and exits non-zero if all questions failed.

## 8. Data Model

No new CLI-side data model. Server-side QTI parsing is a server concern.

## 9. API Surface

```
GET    /api/question-banks/:bank_id/questions     ?type=&limit=&page=
POST   /api/question-banks/:bank_id/questions     { type, content }
POST   /api/question-banks/:bank_id/import        multipart/form-data { file }
```

## 10. UI / UX

`questions list` table: `ID`, `Type`, `Preview` (first 60 chars of question text). Import shows a progress bar and final summary table of results.

## 11. AI / ML Considerations

Not applicable for this story. AI question generation is a separate feature.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| QTI parsing complexity | H | M | Use an existing Go QTI library if available; otherwise implement a basic subset (MCQ, T/F) and iterate |
| Server import endpoint not implemented | M | H | Ship `list` and `create` first; gate `import` on server endpoint |

## 15. Rollout Plan

- `list` and `create` ship with the main CLI binary.
- `import` ships when the server-side QTI endpoint is ready.

## 16. Test Plan

- **Unit** — `--content @file` reading; question type enum validation.
- **Integration** — `list`, `create` against test server; `import` with a small QTI fixture.
- **Manual** — Import a real QTI export from a common LMS against staging.
