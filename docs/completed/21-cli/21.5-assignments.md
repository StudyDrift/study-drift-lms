## 1. Problem Statement

Instructors managing assignments across many courses — setting up problem sets, distributing rubrics, accepting student submissions — currently must use the web UI for every action. Students submitting work from terminal-based development environments (e.g., coding assignments) have no CLI path. This story covers the full assignment lifecycle: list, create, get, and submit.

## 2. Goals

- `lextures assignments list --course <id>` enumerates assignments for a course.
- `lextures assignments create` creates an assignment with title and optional point value.
- `lextures assignments get <id>` fetches assignment details including due date and rubric summary.
- `lextures assignments submit --assignment <id> --file <path>` uploads a file submission.
- All commands support `--json`.

## 3. Non-Goals

- Rubric authoring via CLI (too complex for v1; use web UI).
- Submission grading (covered in 21.6).
- Quiz/exam assignment types (handled by question bank, 21.7).
- Multi-file submissions (single file per submission in v1; can be a zip).

## 4. Personas & User Stories

- **As an instructor**, I want `lextures assignments create --course CS101 --title "HW1" --points 100 --due 2027-09-15` so that I can script assignment creation from a course template.
- **As a student working in a terminal**, I want `lextures assignments submit --assignment hw1 --file ./submission.zip` so that I can submit my work without leaving my development environment.
- **As an admin auditing a course**, I want `lextures assignments list --course CS101 --json` so that I can export the assignment list for reporting.

## 5. Functional Requirements

- **FR-1.** `lextures assignments list --course <id>` MUST return all assignments in the course; supports `--limit`, `--page`.
- **FR-2.** `lextures assignments create` MUST accept `--course <id>` (required), `--title <title>` (required), `--points <n>` (optional), `--due <ISO8601>` (optional).
- **FR-3.** `lextures assignments get <id>` MUST print title, course, points, due date, submission count (for instructors), and submission status (for students).
- **FR-4.** `lextures assignments submit` MUST accept `--assignment <id>` (required) and `--file <path>` (required); file MUST be uploaded via multipart form.
- **FR-5.** `lextures assignments submit` MUST print a submission receipt (submission ID, timestamp) on success.
- **FR-6.** `lextures assignments submit` MUST validate the file path exists locally before attempting upload and exit `1` with a clear error if not.
- **FR-7.** All commands MUST support `--json`.

## 6. Non-Functional Requirements

- **Performance** — File upload MUST show a progress bar for files > 1 MB.
- **Reliability** — Upload MUST retry once on transient network failure. Large files MUST stream rather than buffer entirely in memory.
- **Security** — File upload path MUST be validated (no path traversal); server enforces allowed MIME types.
- **Accessibility** — Progress output MUST be suppressible with `--quiet` for scripted use.

## 7. Acceptance Criteria

- **AC-1.** *Given* a valid course ID, *When* `lextures assignments list --course <id> --json` runs, *Then* stdout is a valid JSON array.
- **AC-2.** *Given* `lextures assignments create --course <id> --title "HW1" --points 100`, *When* the command succeeds, *Then* the assignment ID is printed.
- **AC-3.** *Given* `--file /nonexistent/path`, *When* `lextures assignments submit` runs, *Then* the CLI exits `1` with "File not found: /nonexistent/path" before making any API call.
- **AC-4.** *Given* a 5 MB file, *When* `lextures assignments submit` runs, *Then* a progress bar is displayed.
- **AC-5.** *Given* a successful submit, *When* the command completes, *Then* the submission ID and timestamp are printed (human) or `{"submission_id":"…","submitted_at":"…"}` (JSON mode).

## 8. Data Model

No new database changes. Uses existing assignment and submission endpoints.

## 9. API Surface

Uses existing endpoints:

```
GET    /api/courses/:course_id/assignments        ?limit=&page=
POST   /api/courses/:course_id/assignments        { title, points, due_at }
GET    /api/assignments/:id
POST   /api/assignments/:id/submissions           multipart/form-data { file }
```

## 10. UI / UX

Progress bar via `github.com/schollz/progressbar/v3` or similar. Non-JSON list: columns `ID`, `Title`, `Points`, `Due`, `Submissions`. Submit prints a receipt box on success.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Large file upload OOM | M | M | Stream upload via `io.Reader`; do not buffer |
| Submission endpoint not yet multipart-capable | M | H | Audit server endpoint; may need server-side change |
| Student submits wrong file | M | M | Print file path and size in confirmation before upload; `--dry-run` shows what would be uploaded |

## 15. Rollout Plan

- Shipped as part of CLI binary. File upload depends on server multipart support being confirmed.

## 16. Test Plan
