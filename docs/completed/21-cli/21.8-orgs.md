## 1. Problem Statement

Platform administrators managing multi-tenant Lextures deployments need a CLI path to provision and list organizations without accessing the database or web admin UI. This is particularly important for SaaS operators onboarding new institutional customers and for infrastructure automation that provisions orgs as part of a customer onboarding pipeline.

## 2. Goals

- `lextures orgs list` returns all organizations visible to the authenticated super-admin.
- `lextures orgs create --name <name>` provisions a new organization.
- `lextures orgs get <id>` fetches org details.
- All commands support `--json`.

## 3. Non-Goals

- Org deletion (high blast radius; requires a separate process with data retention review).
- Org-level billing or subscription management.
- User management within an org (covered in 21.4 with `--org` flag).

## 4. Personas & User Stories

- **As a SaaS operator**, I want `lextures orgs create --name "Westside School District"` so that I can provision a new institutional org from my onboarding script without accessing the database.
- **As a platform admin**, I want `lextures orgs list --json` to enumerate all orgs for a billing reconciliation script.

## 5. Functional Requirements

- **FR-1.** `lextures orgs list` MUST return all orgs; requires super-admin role; supports `--limit`, `--page`.
- **FR-2.** `lextures orgs create` MUST accept `--name` (required); MUST return the new org's ID and slug.
- **FR-3.** `lextures orgs get <id>` MUST accept org ID or slug; returns name, created date, user count, course count.
- **FR-4.** All commands MUST support `--json`.
- **FR-5.** Non-super-admin users calling `orgs list` MUST receive a `403` from the server; the CLI MUST surface it as `"Permission denied: super-admin role required"` and exit `2`.

## 6. Non-Functional Requirements

- **Security** — All org management endpoints MUST require super-admin scope; enforced server-side.
- **Auditability** — Org creation MUST be logged server-side (existing audit log).

## 7. Acceptance Criteria

- **AC-1.** *Given* a super-admin token, *When* `lextures orgs list --json` runs, *Then* stdout is a valid JSON array of org objects.
- **AC-2.** *Given* a non-admin token, *When* `lextures orgs list` runs, *Then* exit code is `2` and stderr contains "Permission denied".
- **AC-3.** *Given* `lextures orgs create --name "Test Org"`, *When* it runs, *Then* the org ID and slug are printed.
- **AC-4.** *Given* `lextures orgs get <id>`, *When* it runs, *Then* name, user count, and course count are printed.

## 8. Data Model

No new database changes. Uses existing org management endpoints.

## 9. API Surface

```
GET    /api/orgs              ?limit=&page=
POST   /api/orgs              { name }
GET    /api/orgs/:id
```

## 10. UI / UX

Non-JSON `orgs list` table: `ID`, `Name`, `Slug`, `Users`, `Courses`, `Created`. Create prints `"Created org <name> (id: <id>, slug: <slug>)"`.
