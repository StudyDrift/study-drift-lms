## 1. Problem Statement

All Lextures API endpoints require authentication. Currently there is no mechanism in the CLI to obtain or store a session token. Users running automation or scripting workflows cannot authenticate without manually copying tokens from the web UI. `lextures auth login` must support interactive browser-based OAuth and/or API key flows, and persist the token securely so subsequent commands work without re-authenticating.

## 2. Goals

- `lextures auth login` performs interactive login and stores the resulting token.
- `lextures auth logout` clears the stored token.
- `lextures auth status` shows the current authenticated identity and token expiry.
- Tokens are stored securely (OS keychain preferred, encrypted file fallback).
- API key mode (`--api-key` flag / env var) works without any login step.

## 3. Non-Goals

- Web UI authentication flows.
- SSO / SAML integration for enterprise (future).
- Multi-account simultaneous sessions (profiles in 21.1 cover switching, not concurrent).

## 4. Personas & User Stories

- **As an instructor**, I want to run `lextures auth login` and be sent to the browser to log in so that I don't have to manage tokens manually.
- **As a DevOps engineer**, I want to set `LEXTURES_API_KEY` in my CI environment so that the CLI authenticates without interactive login.
- **As a security-conscious admin**, I want the CLI to store my token in the OS keychain so that it is not written in plaintext to disk.
- **As an instructor switching institutions**, I want `lextures auth logout` to fully clear my session so that I don't accidentally act as the wrong org.

## 5. Functional Requirements

- **FR-1.** `lextures auth login` MUST initiate a device-flow or browser-redirect OAuth flow against the Lextures server's `/oauth/device` or `/oauth/authorize` endpoint.
- **FR-2.** On successful login, the CLI MUST store the access token and refresh token securely (OS keychain via `github.com/zalando/go-keyring`, fallback to `~/.lextures-token` with `0600` permissions).
- **FR-3.** `lextures auth logout` MUST delete the stored token from keychain/file and print a confirmation.
- **FR-4.** `lextures auth status` MUST print the authenticated user's email, org, token expiry, and storage backend in use.
- **FR-5.** When `--api-key` or `LEXTURES_API_KEY` is set, the CLI MUST use it directly and skip keychain lookup.
- **FR-6.** When the stored access token is expired and a refresh token is available, the CLI MUST silently refresh before executing the command.
- **FR-7.** When no token exists and no `--api-key` is set, the CLI MUST print an actionable error: `"Not authenticated. Run 'lextures auth login'."` and exit `1`.

## 6. Non-Functional Requirements

- **Security** — Tokens MUST NOT appear in shell history, process list, or log output. Keychain storage preferred. File fallback MUST warn if permissions exceed `0600`.
- **Reliability** — Token refresh MUST be retried once on transient failure before surfacing the error to the user.
- **Maintainability** — Auth logic isolated in `clients/cli/internal/auth/`; HTTP client in 21.1 injects the token header automatically.

## 7. Acceptance Criteria

- **AC-1.** *Given* the user is not logged in, *When* they run any resource command, *Then* they see `"Not authenticated. Run 'lextures auth login'."` and exit code `1`.
- **AC-2.** *Given* a valid token is stored in the keychain, *When* `lextures auth status` is run, *Then* it prints the user's email and token expiry without prompting for credentials.
- **AC-3.** *Given* `LEXTURES_API_KEY=abc` is set, *When* any command runs, *Then* the HTTP request contains `Authorization: Bearer abc`.
- **AC-4.** *Given* a stored access token has expired and a refresh token exists, *When* the user runs any command, *Then* the token is refreshed silently and the command succeeds.
- **AC-5.** *Given* the user runs `lextures auth logout`, *When* the command completes, *Then* the keychain/file entry is deleted and subsequent `lextures auth status` shows "Not authenticated".

## 8. Data Model

No server-side database changes. Token storage:

- **Primary** — OS keychain service `lextures`, account `<profile-name>` (JSON payload with `access_token`, `refresh_token`, `expiry`).
- **Fallback** — `~/.lextures-token` (JSON, `0600`).

## 9. API Surface

New server endpoints required (if not already present):

```
POST /api/oauth/device/code     -> { device_code, user_code, verification_uri, interval }
POST /api/oauth/device/token    -> { access_token, refresh_token, expires_in }
POST /api/oauth/token/refresh   -> { access_token, expires_in }
GET  /api/auth/me               -> { id, email, name, org_id }
```

## 10. UI / UX

CLI only. `auth login` prints verification URI and user code for device flow, then polls until the user completes login in the browser. Progress shown with a spinner.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| OS keychain unavailable in headless CI | H | M | Document API key env var as the CI path; keychain failure falls back to file automatically |
| Device flow not yet implemented on server | M | H | Coordinate with server team; ship with API key only first, add device flow in a follow-up |
| Token refresh race condition in parallel scripts | L | M | Use file lock around refresh write |

## 15. Rollout Plan

- API key path available immediately; device flow gated on server endpoint availability.
